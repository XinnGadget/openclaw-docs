# Гигиена транскриптов (исправления для провайдеров)

В этом документе описаны **исправления, специфичные для провайдера**, которые применяются к транскриптам перед запуском (формированием контекста модели). Это корректировки **в оперативной памяти**, необходимые для соответствия строгим требованиям провайдера. Эти шаги по "гигиене" **не** перезаписывают сохранённый на диске транскрипт в формате JSONL; однако отдельный проход по исправлению файлов сессии может перезаписать некорректные JSONL-файлы, удалив недопустимые строки до загрузки сессии. При выполнении исправления исходный файл резервируется рядом с файлом сессии.

Область применения включает:

- Очистка идентификаторов вызовов инструментов
- Проверка входных данных вызовов инструментов
- Исправление сопоставления результатов инструментов
- Проверка/упорядочивание ходов
- Очистка сигнатур мыслей
- Очистка полезных данных изображений
- Маркировка происхождения пользовательского ввода (для промтов, маршрутизируемых между сессиями)

Если вам нужны подробности о хранении транскриптов, см.:

- [/reference/session-management-compaction](/reference/session-management-compaction)

---

## Где это выполняется

Вся "гигиена" транскриптов централизована во встроенном раннере:

- Выбор политики: `src/agents/transcript-policy.ts`
- Применение очистки/исправления: `sanitizeSessionHistory` в `src/agents/pi-embedded-runner/google.ts`

Политика использует `provider`, `modelApi` и `modelId`, чтобы определить, какие действия применить.

Отдельно от "гигиены" транскриптов файлы сессии исправляются (при необходимости) перед загрузкой:

- `repairSessionFileIfNeeded` в `src/agents/session-file-repair.ts`
- Вызывается из `run/attempt.ts` и `compact.ts` (встроенный раннер)

## Глобальное правило: очистка изображений

Полезные данные изображений всегда очищаются, чтобы предотвратить отклонение со стороны провайдера из-за ограничений по размеру (уменьшение масштаба/пересжатие изображений в формате base64, превышающих допустимый размер).

Это также помогает контролировать нагрузку на токены, вызванную изображениями, для моделей с поддержкой зрения. Меньшие максимальные размеры обычно снижают использование токенов; большие размеры сохраняют детализацию.

Реализация:

- `sanitizeSessionMessagesImages` в `src/agents/pi-embedded-helpers/images.ts`
- `sanitizeContentBlocksImages` в `src/agents/tool-images.ts`
- Максимальный размер стороны изображения настраивается через `agents.defaults.imageMaxDimensionPx` (по умолчанию: `1200`).

---

## Глобальное правило: некорректные вызовы инструментов

Блоки вызовов инструментов ассистента, в которых отсутствуют и `input`, и `arguments`, удаляются до формирования контекста модели. Это предотвращает отклонения со стороны провайдера из-за частично сохранённых вызовов инструментов (например, после сбоя из-за ограничения скорости).

Реализация:

- `sanitizeToolCallInputs` в `src/agents/session-transcript-repair.ts`
- Применяется в `sanitizeSessionHistory` в `src/agents/pi-embedded-runner/google.ts`

---

## Глобальное правило: происхождение ввода между сессиями

Когда агент отправляет промт в другую сессию через `sessions_send` (включая шаги ответа/объявления между агентами), OpenClaw сохраняет созданный ход пользователя со следующим:

- `message.provenance.kind = "inter_session"`

Эти метаданные записываются при добавлении в транскрипцию и не меняют роль (`role: "user"` остаётся для совместимости с провайдером). Читатели транскриптов могут использовать их, чтобы не рассматривать маршрутизируемые внутренние промты как инструкции, созданные конечным пользователем.

Во время перестроения контекста OpenClaw также добавляет короткий маркер `[Inter-session message]` к этим ходам пользователя в оперативной памяти, чтобы модель могла отличать их от внешних инструкций конечного пользователя.

## Матрица провайдеров (текущее поведение)

**OpenAI / OpenAI Codex**

- Только очистка изображений.
- Удаление сиротских сигнатур рассуждений (отдельные элементы рассуждений без следующего за ними блока содержимого) для транскриптов OpenAI Responses/Codex.
- Нет очистки идентификаторов вызовов инструментов.
- Нет исправления сопоставления результатов инструментов.
- Нет проверки или переупорядочивания ходов.
- Нет синтетических результатов инструментов.
- Нет удаления сигнатур мыслей.

**Google (Generative AI / Gemini CLI / Antigravity)**

- Очистка идентификаторов вызовов инструментов: строго буквенно-цифровые.
- Исправление сопоставления результатов инструментов и синтетические результаты инструментов.
- Проверка ходов (чередование ходов в стиле Gemini).
- Исправление порядка ходов для Google (добавление небольшого начального хода пользователя, если история начинается с хода ассистента).
- Antigravity Claude: нормализация сигнатур рассуждений; удаление блоков рассуждений без подписи.

**Anthropic / Minimax (совместимые с Anthropic)**

- Исправление сопоставления результатов инструментов и синтетические результаты инструментов.
- Проверка ходов (объединение последовательных ходов пользователя для соблюдения строгого чередования).

**Mistral (включая обнаружение на основе model-id)**

- Очистка идентификаторов вызовов инструментов: strict9 (буквенно-цифровые, длина 9).

**OpenRouter Gemini**

- Очистка сигнатур мыслей: удаление значений `thought_signature`, не являющихся base64 (сохраняются только base64).

**Всё остальное**

- Только очистка изображений.

## Историческое поведение (до 2026.1.22)

До выпуска 2026.1.22 OpenClaw применял несколько уровней "гигиены" транскриптов:

- Расширение **transcript-sanitize** запускалось при каждом построении контекста и могло:
  - Исправлять сопоставление использования инструментов и результатов.
  - Очищать идентификаторы вызовов инструментов (включая нестрогий режим, сохраняющий `_`/`-`).
- Раннер также выполнял очистку, специфичную для провайдера, что приводило к дублированию работы.
- Дополнительные изменения происходили вне политики провайдера, включая:
  - Удаление тегов `<final>` из текста ассистента перед сохранением.
  - Удаление пустых ходов ассистента с ошибками.
  - Обрезка содержимого ассистента после вызовов инструментов.

Эта сложность приводила к регрессиям между провайдерами (в частности, к проблемам с сопоставлением `call_id|fc_id` в `openai-responses`). В версии 2026.1.22 расширение было удалено, логика была централизована в раннере, а для OpenAI было введено правило **без изменений**, кроме очистки изображений.
