# Песочница (Sandboxing)

OpenClaw может запускать **инструменты внутри бэкендов песочницы**, чтобы уменьшить радиус потенциального ущерба.
Это **необязательно** и регулируется настройками (`agents.defaults.sandbox` или `agents.list[].sandbox`). Если песочница отключена, инструменты выполняются на хосте.
Шлюз (Gateway) остаётся на хосте; при включённой песочнице выполнение инструментов происходит в изолированной среде.

Это не идеальная граница безопасности, но она существенно ограничивает доступ к файловой системе и процессам, если модель совершает ошибочные действия.

## Что помещается в песочницу

- Выполнение инструментов (`exec`, `read`, `write`, `edit`, `apply_patch`, `process` и т. д.).
- Опциональный браузер в песочнице (`agents.defaults.sandbox.browser`):
  - По умолчанию браузер в песочнице запускается автоматически (обеспечивает доступность CDP), когда это требуется инструменту браузера.
    Настраивается через `agents.defaults.sandbox.browser.autoStart` и `agents.defaults.sandbox.browser.autoStartTimeoutMs`.
  - По умолчанию контейнеры браузера в песочнице используют выделенную сеть Docker (`openclaw-sandbox-browser`) вместо глобальной сети `bridge`.
    Настраивается через `agents.defaults.sandbox.browser.network`.
  - Опциональный параметр `agents.defaults.sandbox.browser.cdpSourceRange` ограничивает входящий трафик CDP на границе контейнера с помощью списка разрешённых CIDR (например, `172.21.0.1/32`).
  - Доступ наблюдателя noVNC по умолчанию защищён паролем; OpenClaw выдаёт краткосрочный URL-токен, который открывает локальную загрузочную страницу и запускает noVNC с паролем во фрагменте URL (не в логах запросов/заголовков).
  - Параметр `agents.defaults.sandbox.browser.allowHostControl` позволяет сессиям в песочнице явно нацеливаться на браузер хоста.
  - Опциональные списки разрешённых адресов управляют `target: "custom"`: `allowedControlUrls`, `allowedControlHosts`, `allowedControlPorts`.

Не помещается в песочницу:

- Сам процесс шлюза (Gateway).
- Любые инструменты, явно разрешённые для выполнения вне песочницы (например, `tools.elevated`).
  - **Elevated exec обходит песочницу и использует заданный путь выхода (`gateway` по умолчанию или `node`, если цель exec — `node`).**
  - Если песочница отключена, `tools.elevated` не меняет способ выполнения (уже на хосте). См. [Elevated Mode](/tools/elevated).

## Режимы

Параметр `agents.defaults.sandbox.mode` определяет, **когда** используется песочница:

- `"off"`: песочница не используется.
- `"non-main"`: песочница применяется только для **неосновных** сессий (значение по умолчанию, если вы хотите, чтобы обычные чаты выполнялись на хосте).
- `"all"`: все сессии выполняются в песочнице.
  Примечание: `"non-main"` основано на `session.mainKey` (по умолчанию `"main"`), а не на идентификаторе агента.
  Сессии групп/каналов используют свои собственные ключи, поэтому они считаются неосновными и помещаются в песочницу.

## Область действия

Параметр `agents.defaults.sandbox.scope` определяет, **сколько контейнеров** будет создано:

- `"agent"` (по умолчанию): один контейнер на агента.
- `"session"`: один контейнер на сессию.
- `"shared"`: один контейнер, используемый всеми сессиями в песочнице.

## Бэкенд

Параметр `agents.defaults.sandbox.backend` определяет, **какая среда выполнения** предоставляет песочницу:

- `"docker"` (по умолчанию): локальная среда выполнения песочницы на базе Docker.
- `"ssh"`: общая удалённая среда выполнения песочницы на базе SSH.
- `"openshell"`: среда выполнения песочницы на базе OpenShell.

Настройки, специфичные для SSH, находятся в `agents.defaults.sandbox.ssh`.
Настройки, специфичные для OpenShell, находятся в `plugins.entries.openshell.config`.

### Выбор бэкенда

| | Docker | SSH | OpenShell |
| --- | --- | --- | --- |
| **Где выполняется** | Локальный контейнер | Любой хост, доступный по SSH | Песочница, управляемая OpenShell |
| **Настройка** | `scripts/sandbox-setup.sh` | SSH-ключ + целевой хост | Включён плагин OpenShell |
| **Модель рабочей области** | Примонтирование или копирование | Удалённая каноническая (однократная инициализация) | `mirror` или `remote` |
| **Управление сетью** | `docker.network` (по умолчанию: нет) | Зависит от удалённого хоста | Зависит от OpenShell |
| **Песочница для браузера** | Поддерживается | Не поддерживается | Пока не поддерживается |
| **Примонтированные директории** | `docker.binds` | Н/Д | Н/Д |
| **Лучше всего подходит для** | Локальная разработка, полная изоляция | Перенос нагрузки на удалённый хост | Управляемые удалённые песочницы с опциональной двусторонней синхронизацией |

### Бэкенд SSH

Используйте `backend: "ssh"`, если хотите, чтобы OpenClaw помещал `exec`, инструменты работы с файлами и чтение медиа в песочницу на произвольном хосте, доступном по SSH.

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "all",
        backend: "ssh",
        scope: "session",
        workspaceAccess: "rw",
        ssh: {
          target: "user@gateway-host:22",
          workspaceRoot: "/tmp/openclaw-sandboxes",
          strictHostKeyChecking: true,
          updateHostKeys: true,
          identityFile: "~/.ssh/id_ed25519",
          certificateFile: "~/.ssh/id_ed25519-cert.pub",
          knownHostsFile: "~/.ssh/known_hosts",
          // Или используйте SecretRefs / встроенные данные вместо локальных файлов:
          // identityData: { source: "env", provider: "default", id: "SSH_IDENTITY" },
          // certificateData: { source: "env", provider: "default", id: "SSH_CERTIFICATE" },
          // knownHostsData: { source: "env", provider: "default", id: "SSH_KNOWN_HOSTS" },
        },
      },
    },
  },
}
```

Как это работает:

- OpenClaw создаёт удалённую корневую директорию для каждой области действия в `sandbox.ssh.workspaceRoot`.
- При первом использовании после создания или воссоздания OpenClaw однократно инициализирует удалённую рабочую область из локальной.
- После этого `exec`, `read`, `write`, `edit`, `apply_patch`, чтение медиа в запросах и размещение входящих медиа выполняются напрямую в удалённой рабочей области через SSH.
- OpenClaw не синхронизирует изменения из удалённой области обратно в локальную автоматически.

Материалы для аутентификации:

- `identityFile`, `certificateFile`, `knownHostsFile`: используйте существующие локальные файлы и передайте их через конфигурацию OpenSSH.
- `identityData`, `certificateData`, `knownHostsData`: используйте встроенные строки или SecretRefs. OpenClaw обрабатывает их через обычный снимок состояния секретов, записывает во временные файлы с правами `0600` и удаляет их по завершении SSH-сессии.
- Если для одного и того же элемента заданы и `*File`, и `*Data`, для этой SSH-сессии приоритет имеет `*Data`.

Это модель **удалённой канонической** рабочей области. Удалённая рабочая область SSH становится реальным состоянием песочницы после начальной инициализации.

Важные последствия:

- Изменения на хосте, выполненные вне OpenClaw после этапа инициализации, не будут видны удалённо до воссоздания песочницы.
- `openclaw sandbox recreate` удаляет удалённую корневую директорию для каждой области действия и повторно инициализирует её из локальной при следующем использовании.
- Песочница для браузера не поддерживается в бэкенде SSH.
- Настройки `sandbox.docker.*` не применяются к бэкенду SSH.

### Бэкенд OpenShell

Используйте `backend: "openshell"`, если хотите, чтобы OpenClaw помещал инструменты в песочницу в удалённой среде, управляемой OpenShell. Полное руководство по настройке, справочник по конфигурации и сравнение режимов рабочей области см. на специальной [странице OpenShell](/gateway/openshell).
