# Узлы

**Узел** — это сопутствующее устройство (macOS/iOS/Android/без интерфейса), которое подключается к **WebSocket** шлюза (тот же порт, что и для операторов) с параметром `role: "node"` и предоставляет набор команд (например, `canvas.*`, `camera.*`, `device.*`, `notifications.*`, `system.*`) через `node.invoke`. Детали протокола: [Протокол шлюза](/gateway/protocol).

Устаревший транспорт: [Протокол Bridge](/gateway/bridge-protocol) (TCP JSONL; только для исторических целей в текущих узлах).

macOS также может работать в **режиме узла**: приложение в строке меню подключается к WS-серверу шлюза и предоставляет свои локальные команды canvas/камеры как узел (таким образом, команды `openclaw nodes …` работают с этим Mac).

Примечания:

- Узлы являются **периферийными устройствами**, а не шлюзами. Они не запускают службу шлюза.
- Сообщения Telegram/WhatsApp и т. д. поступают на **шлюз**, а не на узлы.
- Руководство по устранению неполадок: [/nodes/troubleshooting](/nodes/troubleshooting)

## Сопряжение и статус

**Узлы WS используют сопряжение устройств.** Узлы предоставляют идентификатор устройства во время `connect`; шлюз создаёт запрос на сопряжение устройства для `role: node`. Одобрите через CLI устройств (или через пользовательский интерфейс).

Быстрый CLI:

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw devices reject <requestId>
openclaw nodes status
openclaw nodes describe --node <idOrNameOrIp>
```

Если узел повторяет попытку подключения с изменёнными данными аутентификации (роль/области/открытый ключ), предыдущий ожидающий запрос заменяется и создаётся новый `requestId`. Перед одобрением повторно выполните `openclaw devices list`.

Примечания:

- `nodes status` помечает узел как **сопряжённый**, если его роль в сопряжении устройства включает `node`.
- Запись о сопряжении устройства — это надёжный контракт с утверждённой ролью. Ротация токенов остаётся в рамках этого контракта; она не может преобразовать сопряжённый узел в другую роль, которую не одобрили при сопряжении.
- `node.pair.*` (CLI: `openclaw nodes pending/approve/reject/rename`) — это отдельное хранилище сопряжения узлов, управляемое шлюзом; оно **не** регулирует рукопожатие `connect` WS.
- Область одобрения соответствует объявленным командам в ожидающем запросе:
  - запрос без команд: `operator.pairing`
  - команды узла без выполнения: `operator.pairing` + `operator.write`
  - `system.run` / `system.run.prepare` / `system.which`: `operator.pairing` + `operator.admin`

## Удалённый хост узла (system.run)

Используйте **хост узла**, когда ваш шлюз работает на одном компьютере, а вы хотите, чтобы команды выполнялись на другом. Модель по-прежнему взаимодействует со **шлюзом**; шлюз перенаправляет вызовы `exec` на **хост узла**, когда выбран `host=node`.

### Что где выполняется

- **Хост шлюза**: получает сообщения, запускает модель, маршрутизирует вызовы инструментов.
- **Хост узла**: выполняет `system.run`/`system.which` на компьютере узла.
- **Одобрения**: применяются на хосте узла через `~/.openclaw/exec-approvals.json`.

Примечание об одобрении:

- Выполнения узла с одобрением привязываются к точному контексту запроса.
- Для прямого выполнения файлов оболочки/среды выполнения OpenClaw также старается привязать один конкретный локальный файл и отклоняет выполнение, если этот файл изменяется до выполнения.
- Если OpenClaw не может определить ровно один конкретный локальный файл для команды интерпретатора/среды выполнения, выполнение с одобрением отклоняется вместо имитации полного покрытия среды выполнения. Используйте песочницу, отдельные хосты или явный доверенный список разрешений/полный рабочий процесс для более широкой семантики интерпретатора.

### Запуск хоста узла (в переднем плане)

На компьютере узла:

```bash
openclaw node run --host <gateway-host> --port 18789 --display-name "Build Node"
```

### Удалённый шлюз через SSH-туннель (привязка к loopback)

Если шлюз привязан к loopback (`gateway.bind=loopback`, по умолчанию в локальном режиме), удалённые хосты узлов не могут подключиться напрямую. Создайте SSH-туннель и укажите хосту узла локальный конец туннеля.

Пример (хост узла → хост шлюза):

```bash
# Терминал A (оставить запущенным): перенаправить локальный 18790 → шлюз 127.0.0.1:18789
ssh -N -L 18790:127.0.0.1:18789 user@gateway-host

# Терминал B: экспортировать токен шлюза и подключиться через туннель
export OPENCLAW_GATEWAY_TOKEN="<gateway-token>"
openclaw node run --host 127.0.0.1 --port 18790 --display-name "Build Node"
```

Примечания:

- `openclaw node run` поддерживает аутентификацию по токену или паролю.
- Предпочтение отдаётся переменным окружения: `OPENCLAW_GATEWAY_TOKEN` / `OPENCLAW_GATEWAY_PASSWORD`.
- Резервный вариант конфигурации — `gateway.auth.token` / `gateway.auth.password`.
- В локальном режиме хост узла намеренно игнорирует `gateway.remote.token` / `gateway.remote.password`.
- В удалённом режиме `gateway.remote.token` / `gateway.remote.password` учитываются в соответствии с правилами приоритета для удалённых подключений.
- Если активные локальные `gateway.auth.*` SecretRefs настроены, но не разрешены, аутентификация хоста узла завершается ошибкой.
- Разрешение аутентификации хоста узла учитывает только переменные окружения `OPENCLAW_GATEWAY_*`.

### Запуск хоста узла (как службы)

```bash
openclaw node install --host <gateway-host> --port 18789 --display-name "Build Node"
openclaw node restart
```

### Сопряжение и присвоение имени

На хосте шлюза:

```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw nodes status
```

Если узел повторяет попытку подключения с изменёнными данными аутентификации, повторно выполните `openclaw devices list` и одобрите текущий `requestId`.

Варианты присвоения имени:

- `--display-name` в `openclaw node run` / `openclaw node install` (сохраняется в `~/.openclaw/node.json` на узле).
- `openclaw nodes rename --node <id|name|ip> --name "Build Node"` (переопределение на шлюзе).

### Добавление команд в список разрешённых

Одобрения выполнения **привязаны к хосту узла**. Добавьте записи в список разрешённых с шлюза:

```bash
openclaw approvals allowlist add --node <id|name|ip> "/usr/bin/uname"
openclaw approvals allowlist add --node <id|name|ip> "/usr/bin/sw_vers"
```

Одобрения хранятся на хосте узла в `~/.openclaw/exec-approvals.json`.

### Указание выполнения на узле

Настройте значения по умолчанию (конфигурация шлюза):

```bash
openclaw config set tools.exec.host node
openclaw config set tools.exec.security allowlist
openclaw config set tools.exec.node "<id-or-name>"
```

Или для каждой сессии:

```
/exec host=node security=allowlist node=<id-or-name>
```

После настройки любой вызов `exec` с `host=node` будет выполняться на хосте узла (с учётом списка разрешённых/одобрений узла).

`host=auto` не будет неявно выбирать узел самостоятельно, но явный запрос `host=node` для каждого вызова разрешён из `auto`. Если вы хотите, чтобы выполнение на узле было значением по умолчанию для сессии, установите `tools.exec.host=node` или явно укажите `/exec host=node ...`.
