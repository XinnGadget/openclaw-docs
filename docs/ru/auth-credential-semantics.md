# Семантика учётных данных аутентификации

В этом документе определена стандартная семантика проверки пригодности и разрешения учётных данных, используемая в следующих компонентах:

- `resolveAuthProfileOrder`
- `resolveApiKeyForProfile`
- `models status --probe`
- `doctor-auth`

Цель состоит в том, чтобы обеспечить согласованность поведения во время выбора и во время выполнения.

## Стабильные коды причин зондирования

- `ok`
- `excluded_by_auth_order`
- `missing_credential`
- `invalid_expires`
- `expired`
- `unresolved_ref`
- `no_model`

## Учётные данные токена

Учётные данные токена (`type: "token"`) поддерживают встроенные значения `token` и/или `tokenRef`.

### Правила проверки пригодности

1. Профиль токена считается непригодным, если отсутствуют и `token`, и `tokenRef`.
2. Поле `expires` является необязательным.
3. Если поле `expires` присутствует, оно должно представлять собой конечное число, большее `0`.
4. Если значение `expires` некорректно (`NaN`, `0`, отрицательное число, неконечное число или неверный тип), профиль считается непригодным с кодом `invalid_expires`.
5. Если значение `expires` относится к прошлому, профиль считается непригодным с кодом `expired`.
6. `tokenRef` не обходит проверку `expires`.

### Правила разрешения

1. Семантика резолвера соответствует семантике проверки пригодности для `expires`.
2. Для пригодных профилей токен может быть разрешён из встроенного значения или `tokenRef`.
3. Неразрешимые ссылки приводят к появлению `unresolved_ref` в выводе команды `models status --probe`.

## Явная фильтрация по порядку аутентификации

- Если для провайдера задано значение `auth.order.<provider>` или переопределение порядка в хранилище аутентификации, команда `models status --probe` проверяет только те идентификаторы профилей, которые остаются в разрешённом порядке аутентификации для этого провайдера.
- Сохранённый профиль для этого провайдера, который исключён из явного порядка, не будет автоматически проверен позже. В выводе зондирования он будет отмечен кодом `reasonCode: excluded_by_auth_order` и деталью `Excluded by auth.order for this provider.`

## Разрешение целей зондирования

- Цели зондирования могут поступать из профилей аутентификации, учётных данных окружения или файла `models.json`.
- Если у провайдера есть учётные данные, но OpenClaw не может разрешить кандидата модели для зондирования, команда `models status --probe` выводит `status: no_model` с кодом `reasonCode: no_model`.

## Защита политики SecretRef для OAuth

- Ввод SecretRef предназначен только для статических учётных данных.
- Если учётные данные профиля имеют тип `type: "oauth"`, объекты SecretRef не поддерживаются для материала учётных данных этого профиля.
- Если `auth.profiles.<id>.mode` имеет значение `"oauth"`, ввод `keyRef`/`tokenRef`, поддерживаемый SecretRef, для этого профиля отклоняется.
- Нарушения приводят к критическим ошибкам на этапах разрешения аутентификации при запуске или перезагрузке.

## Совместимость с устаревшими сообщениями

Для совместимости со скриптами первая строка сообщений об ошибках зондирования остаётся неизменной:

`Auth profile credentials are missing or expired.`

Подробные сведения, удобные для восприятия человеком, и стабильные коды причин могут быть добавлены в последующих строках.
