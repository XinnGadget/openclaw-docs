# Присутствие (Presence)

"Присутствие" (presence) в OpenClaw — это облегчённое представление (с максимально возможной точностью) следующего:

- самого **шлюза (Gateway)**;
- **клиентов, подключённых к шлюзу** (приложение для macOS, WebChat, CLI и т. д.).

Присутствие используется главным образом для отображения вкладки **Instances** в приложении для macOS и для обеспечения быстрого визуального контроля со стороны оператора.

## Поля присутствия (что отображается)

Записи присутствия — это структурированные объекты, содержащие такие поля, как:

- `instanceId` (необязательно, но настоятельно рекомендуется): стабильный идентификатор клиента (обычно `connect.client.instanceId`);
- `host`: понятное человеку имя хоста;
- `ip`: IP-адрес (с максимально возможной точностью);
- `version`: строка с версией клиента;
- `deviceFamily` / `modelIdentifier`: подсказки о аппаратном обеспечении;
- `mode`: `ui`, `webchat`, `cli`, `backend`, `probe`, `test`, `node` и т. д.;
- `lastInputSeconds`: "количество секунд с момента последнего ввода пользователя" (если известно);
- `reason`: `self`, `connect`, `node-connected`, `periodic` и т. д.;
- `ts`: отметка времени последнего обновления (в миллисекундах с начала эпохи).

## Источники данных (откуда берётся присутствие)

Записи присутствия формируются из нескольких источников и **объединяются**.

### 1) Запись о самом шлюзе

Шлюз всегда создаёт запись "о себе" при запуске, чтобы в пользовательских интерфейсах отображался хост шлюза ещё до подключения каких-либо клиентов.

### 2) Подключение через WebSocket

Каждый WS-клиент начинает с запроса `connect`. После успешного рукопожатия шлюз обновляет (upserts) запись присутствия для этого соединения.

#### Почему разовые команды CLI не отображаются

CLI часто подключается для выполнения коротких разовых команд. Чтобы не засорять список Instances, запись присутствия для `client.mode === "cli"` **не создаётся**.

### 3) Маячки `system-event`

Клиенты могут отправлять более подробные периодические маячки через метод `system-event`. Приложение для macOS использует этот механизм для передачи имени хоста, IP-адреса и значения `lastInputSeconds`.

### 4) Подключение узла (role: node)

Когда узел подключается к шлюзу через WebSocket с `role: node`, шлюз обновляет (upserts) запись присутствия для этого узла (по тому же принципу, что и для других WS-клиентов).

## Правила объединения и устранения дубликатов (почему важен `instanceId`)

Записи присутствия хранятся в единой карте в оперативной памяти:

- Записи индексируются по **ключу присутствия**.
- Лучший ключ — стабильный `instanceId` (из `connect.client.instanceId`), который сохраняется после перезапусков.
- Ключи не чувствительны к регистру.

Если клиент переподключается без стабильного `instanceId`, он может отображаться как **дублирующаяся** строка.

## Время жизни (TTL) и ограниченный размер

Присутствие намеренно является временным:

- **TTL:** записи старше 5 минут удаляются;
- **Максимальное количество записей:** 200 (сначала удаляются самые старые).

Это позволяет поддерживать актуальность списка и избегать неограниченного роста потребления памяти.

## Особенность работы с удалёнными подключениями/туннелями (IP-адреса loopback)

Когда клиент подключается через SSH-туннель или перенаправление локального порта, шлюз может видеть удалённый адрес как `127.0.0.1`. Чтобы не перезаписывать корректный IP-адрес, указанный клиентом, loopback-адреса удалённых подключений игнорируются.

## Потребители данных

### Вкладка Instances в macOS

Приложение для macOS отображает результаты вызова `system-presence` и добавляет небольшой индикатор состояния (Active/Idle/Stale) на основе давности последнего обновления.

## Советы по отладке

- Чтобы увидеть необработанный список, вызовите `system-presence` для шлюза.
- Если вы видите дубликаты:
  - убедитесь, что клиенты отправляют стабильный `client.instanceId` при рукопожатии;
  - убедитесь, что периодические маячки используют тот же `instanceId`;
  - проверьте, отсутствует ли `instanceId` в записи, созданной на основе подключения (в этом случае дубликаты ожидаемы).
