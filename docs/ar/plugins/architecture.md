## نموذج الإمكانات العام

تمثل الإمكانات نموذج **Plugins الأصلية** العام داخل OpenClaw. يسجل كل
Plugin أصلي في OpenClaw نفسه ضمن نوع واحد أو أكثر من أنواع الإمكانات:

| الإمكانية             | طريقة التسجيل                                  | Plugins نموذجية                       |
| --------------------- | ---------------------------------------------- | ------------------------------------- |
| الاستدلال النصي       | `api.registerProvider(...)`                    | `openai`، `anthropic`                 |
| خلفية استدلال CLI     | `api.registerCliBackend(...)`                  | `openai`، `anthropic`                 |
| الكلام                | `api.registerSpeechProvider(...)`              | `elevenlabs`، `microsoft`             |
| النسخ الفوري          | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                            |
| الصوت الفوري          | `api.registerRealtimeVoiceProvider(...)`       | `openai`                              |
| فهم الوسائط           | `api.registerMediaUnderstandingProvider(...)`  | `openai`، `google`                    |
| توليد الصور           | `api.registerImageGenerationProvider(...)`     | `openai`، `google`، `fal`، `minimax`  |
| توليد الموسيقى        | `api.registerMusicGenerationProvider(...)`     | `google`، `minimax`                   |
| توليد الفيديو         | `api.registerVideoGenerationProvider(...)`     | `qwen`                                |
| جلب الويب             | `api.registerWebFetchProvider(...)`            | `firecrawl`                           |
| البحث على الويب       | `api.registerWebSearchProvider(...)`           | `google`                              |
| القناة / المراسلة     | `api.registerChannel(...)`                     | `msteams`، `matrix`                   |

أي Plugin لا يسجل أي إمكانات ولكنه يوفّر مرابط أو أدوات أو
خدمات يُعد Plugin **قديمًا يعتمد على المرابط فقط**. ولا يزال هذا النمط مدعومًا بالكامل.

### موقف التوافق الخارجي

تم اعتماد نموذج الإمكانات في النواة ويُستخدم اليوم من قبل
Plugins المجمّعة/الأصلية، لكن توافق Plugins الخارجية لا يزال يحتاج إلى معيار
أكثر صرامة من "إن كان مُصدّرًا فهو إذًا ثابت."

الإرشادات الحالية:

- **Plugins الخارجية الحالية:** حافظ على عمل التكاملات المعتمدة على المرابط؛
  وتعامل مع هذا باعتباره خط الأساس للتوافق
- **Plugins المجمّعة/الأصلية الجديدة:** فضّل التسجيل الصريح للإمكانات بدلًا من
  التوغلات الخاصة بالمورّد أو التصاميم الجديدة المعتمدة على المرابط فقط
- **Plugins الخارجية التي تعتمد تسجيل الإمكانات:** هذا مسموح، لكن اعتبر أسطح
  المساعدة الخاصة بالإمكانات قابلة للتطور ما لم تذكر الوثائق صراحة أن العقد ثابت

القاعدة العملية:

- واجهات API الخاصة بتسجيل الإمكانات هي الاتجاه المقصود
- تظل المرابط القديمة المسار الأكثر أمانًا لتجنب الكسر بالنسبة إلى Plugins الخارجية أثناء
  الانتقال
- ليست كل المسارات الفرعية المصدّرة متساوية؛ فضّل العقد الضيق الموثّق، لا
  صادرات المساعدة العرضية

### أشكال Plugin

يصنف OpenClaw كل Plugin محمّل إلى شكل معين بناءً على سلوك التسجيل
الفعلي الخاص به (وليس فقط البيانات الوصفية الثابتة):

- **plain-capability** -- يسجل نوع إمكانية واحدًا فقط بالضبط (على سبيل المثال
  Plugin موفّر فقط مثل `mistral`)
- **hybrid-capability** -- يسجل عدة أنواع من الإمكانات (على سبيل المثال
  `openai` يملك الاستدلال النصي والكلام وفهم الوسائط وتوليد
  الصور)
- **hook-only** -- يسجل المرابط فقط (مطبوعة أو مخصصة)، من دون إمكانات
  أو أدوات أو أوامر أو خدمات
- **non-capability** -- يسجل أدوات أو أوامر أو خدمات أو مسارات لكن من دون
  إمكانات

استخدم `openclaw plugins inspect <id>` لرؤية شكل Plugin وتفصيل
الإمكانات الخاصة به. راجع [مرجع CLI](/cli/plugins#inspect) للتفاصيل.

### المرابط القديمة

لا يزال المرابط `before_agent_start` مدعومًا باعتباره مسار توافق لـ
Plugins المعتمدة على المرابط فقط. لا تزال Plugins حقيقية قديمة تعتمد عليه.

الاتجاه:

- حافظ على عمله
- وثّقه باعتباره قديمًا
- فضّل `before_model_resolve` لأعمال تجاوز النموذج/الموفّر
- فضّل `before_prompt_build` لأعمال تعديل الموجّه
- أزله فقط بعد انخفاض الاستخدام الفعلي وإثبات تغطية التركيبات الاختبارية لأمان الترحيل

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى
إحدى هذه التسميات:

| الإشارة                   | المعنى                                                       |
| ------------------------- | ------------------------------------------------------------ |
| **config valid**          | يُحلَّل الإعداد بشكل صحيح وتُحل Plugins                      |
| **compatibility advisory** | يستخدم Plugin نمطًا مدعومًا لكنه أقدم (مثل `hook-only`)    |
| **legacy warning**        | يستخدم Plugin المرابط `before_agent_start`، وهو مهمل        |
| **hard error**            | الإعداد غير صالح أو فشل تحميل Plugin                         |

لن يؤدي `hook-only` ولا `before_agent_start` إلى كسر Plugin لديك اليوم --
فـ `hook-only` إرشادي فقط، و`before_agent_start` لا يؤدي إلا إلى تحذير. كما
تظهر هذه الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يتكون نظام Plugins في OpenClaw من أربع طبقات:

1. **البيان + الاكتشاف**
   يعثر OpenClaw على Plugins المرشحة من المسارات المضبوطة، وجذور مساحات العمل،
   وجذور الإضافات العامة، والإضافات المجمعة. ويقرأ الاكتشاف أولًا
   بيانات `openclaw.plugin.json` الخاصة بـ Plugins الأصلية إضافةً إلى
   بيانات الحزم المدعومة.
2. **التمكين + التحقق**
   تقرر النواة ما إذا كان Plugin المكتشف ممكّنًا أو معطّلًا أو محجوبًا أو
   محددًا لفتحة حصرية مثل الذاكرة.
3. **التحميل وقت التشغيل**
   تُحمّل Plugins OpenClaw الأصلية داخل العملية عبر jiti وتُسجل
   الإمكانات في سجل مركزي. وتُطبع الحزم المتوافقة إلى سجلات في
   السجل من دون استيراد كود وقت التشغيل.
4. **استهلاك السطح**
   يقرأ باقي OpenClaw السجل لعرض الأدوات، والقنوات، وإعداد
   الموفّر، والمرابط، ومسارات HTTP، وأوامر CLI، والخدمات.

وبالنسبة إلى CLI الخاصة بـ Plugin على وجه التحديد، ينقسم اكتشاف الأمر الجذري إلى مرحلتين:

- تأتي البيانات الوصفية وقت التحليل من `registerCli(..., { descriptors: [...] })`
- يمكن أن تظل وحدة CLI الحقيقية الخاصة بـ Plugin كسولة وتُسجل عند أول استدعاء

يحافظ ذلك على كود CLI المملوك لـ Plugin داخل Plugin مع السماح لـ OpenClaw
بحجز أسماء الأوامر الجذرية قبل التحليل.

الحد التصميمي المهم:

- يجب أن يعمل الاكتشاف + التحقق من الإعداد اعتمادًا على **بيانات البيان/المخطط الوصفية**
  من دون تنفيذ كود Plugin
- يأتي سلوك وقت التشغيل الأصلي من مسار `register(api)` الخاص بوحدة Plugin

يتيح هذا الفصل لـ OpenClaw التحقق من الإعداد، وشرح Plugins المفقودة/المعطّلة، و
بناء تلميحات الواجهة/المخطط قبل تنشيط وقت التشغيل الكامل.

### Plugins القنوات وأداة الرسائل المشتركة

لا تحتاج Plugins القنوات إلى تسجيل أداة إرسال/تعديل/تفاعل منفصلة لأفعال
الدردشة العادية. يحتفظ OpenClaw بأداة `message` مشتركة واحدة في النواة،
وتملك Plugins القنوات الاكتشاف والتنفيذ الخاصين بالقناة خلفها.

الحد الحالي هو:

- تملك النواة مضيف أداة `message` المشتركة، وربط الموجّه، ودفتر
  الجلسة/الخيط، وتوزيع التنفيذ
- تملك Plugins القنوات اكتشاف الأفعال المقيّدة، واكتشاف الإمكانات، وأي أجزاء
  مخطط خاصة بالقناة
- تملك Plugins القنوات قواعد محادثة الجلسة الخاصة بالموفّر، مثل
  كيفية ترميز معرفات المحادثة لمعرفات الخيوط أو وراثتها من المحادثات الأصلية
- تنفذ Plugins القنوات الفعل النهائي عبر مهايئ الأفعال الخاص بها

بالنسبة إلى Plugins القنوات، يكون سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. يسمح
استدعاء الاكتشاف الموحّد هذا لـ Plugin بإرجاع أفعاله المرئية، وإمكاناته،
ومساهمات المخطط معًا حتى لا تنحرف هذه الأجزاء عن بعضها.

تمرر النواة نطاق وقت التشغيل إلى خطوة الاكتشاف هذه. وتشمل الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` الوارد والموثوق

وهذا مهم بالنسبة إلى Plugins الحساسة للسياق. يمكن للقناة إخفاء أفعال
الرسائل أو إظهارها بناءً على الحساب النشط، أو الغرفة/الخيط/الرسالة الحالية، أو
هوية الطالب الموثوق بها، من دون ترميز تفرعات خاصة بالقناة داخل أداة
`message` الأساسية في النواة.

ولهذا السبب لا تزال تغييرات التوجيه في المشغل المضمّن من عمل Plugin: فالمشغل
مسؤول عن تمرير هوية الدردشة/الجلسة الحالية إلى حد الاكتشاف الخاص بـ Plugin
حتى تعرض أداة `message` المشتركة السطح الصحيح المملوك للقناة في
الدور الحالي.

وبالنسبة إلى مساعدات التنفيذ المملوكة للقناة، يجب على Plugins المجمعة أن تحتفظ
بوقت تشغيل التنفيذ داخل وحدات الإضافة الخاصة بها. لم تعد النواة تملك أوقات تشغيل
أفعال الرسائل لـ Discord وSlack وTelegram وWhatsApp ضمن `src/agents/tools`.
نحن لا ننشر مسارات فرعية منفصلة `plugin-sdk/*-action-runtime`، ويجب على
Plugins المجمعة استيراد كود وقت التشغيل المحلي الخاص بها مباشرة من
الوحدات المملوكة للإضافة.

وينطبق الحد نفسه على أسطح SDK المسماة باسم الموفّر عمومًا: يجب ألا
تستورد النواة حاويات تسهيل خاصة بالقنوات لـ Slack أو Discord أو Signal أو
WhatsApp أو إضافات مشابهة. إذا احتاجت النواة إلى سلوك ما، فإما أن تستهلك
الحاوية `api.ts` / `runtime-api.ts` الخاصة بـ Plugin المجمّع نفسه، أو
ترقّي الحاجة إلى إمكانية عامة ضيقة في SDK المشتركة.

وبالنسبة إلى الاستطلاعات على وجه التحديد، يوجد مساران للتنفيذ:

- `outbound.sendPoll` هو خط الأساس المشترك للقنوات التي تناسب نموذج
  الاستطلاع الشائع
- `actions.handleAction("poll")` هو المسار المفضل لدلالات الاستطلاع
  الخاصة بالقناة أو لمعاملات الاستطلاع الإضافية

تؤجل النواة الآن تحليل الاستطلاعات المشتركة إلى ما بعد رفض توزيع استطلاعات Plugin
للفعل، حتى تتمكن معالجات الاستطلاعات المملوكة لـ Plugin من قبول حقول استطلاع
خاصة بالقناة من دون أن يحجبها محلل الاستطلاع العام أولًا.

راجع [مسار التحميل](#load-pipeline) للتسلسل الكامل لبدء التشغيل.

## نموذج ملكية الإمكانات

يتعامل OpenClaw مع Plugin الأصلية باعتبارها حد الملكية الخاص بـ **شركة**
أو **ميزة**، لا باعتبارها مجموعة عشوائية من التكاملات غير المترابطة.

وهذا يعني:

- ينبغي عادةً أن تملك Plugin الشركة كل الأسطح الخاصة بتلك الشركة داخل OpenClaw
- ينبغي عادةً أن تملك Plugin الميزة سطح الميزة الكامل الذي تقدمه
- ينبغي أن تستهلك القنوات إمكانات النواة المشتركة بدلًا من إعادة تنفيذ
  سلوك الموفّر بصورة مخصصة

أمثلة:

- تملك Plugin المجمعة `openai` سلوك موفّر نماذج OpenAI وسلوك
  OpenAI للكلام + الصوت الفوري + فهم الوسائط + توليد الصور
- تملك Plugin المجمعة `elevenlabs` سلوك الكلام الخاص بـ ElevenLabs
- تملك Plugin المجمعة `microsoft` سلوك الكلام الخاص بـ Microsoft
- تملك Plugin المجمعة `google` سلوك موفّر نماذج Google إضافة إلى
  سلوك Google لفهم الوسائط + توليد الصور + البحث على الويب
- تملك Plugin المجمعة `firecrawl` سلوك جلب الويب الخاص بـ Firecrawl
- تملك Plugins المجمعة `minimax` و`mistral` و`moonshot` و`zai`
  واجهاتها الخلفية الخاصة بفهم الوسائط
- تملك Plugin المجمعة `qwen` سلوك موفّر النص الخاص بـ Qwen إضافة إلى
  فهم الوسائط وسلوك توليد الفيديو
- تُعد Plugin `voice-call` Plugin ميزة: فهي تملك نقل المكالمات، والأدوات،
  وCLI، والمسارات، وجسر تدفق الوسائط الخاص بـ Twilio، لكنها تستهلك
  إمكانات الكلام المشتركة إضافةً إلى النسخ الفوري والصوت الفوري بدلًا من
  استيراد Plugins المورّدين مباشرة

الحالة النهائية المقصودة هي:

- تعيش OpenAI داخل Plugin واحدة حتى إن كانت تمتد عبر نماذج النص والكلام والصور و
  الفيديو مستقبلًا
- يمكن لمورّد آخر أن يفعل الشيء نفسه ضمن نطاقه الخاص
- لا تهتم القنوات بأي Plugin مورّد تملك الموفّر؛ فهي تستهلك
  عقد الإمكانية المشتركة الذي تكشفه النواة

وهذا هو التمييز الأساسي:

- **plugin** = حد الملكية
- **capability** = عقد النواة الذي يمكن لعدة Plugins تنفيذه أو استهلاكه

لذلك إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فليس السؤال الأول هو
"أي موفّر يجب أن يضمّن معالجة الفيديو بشكل صلب؟" بل السؤال الأول هو "ما هو
عقد إمكانية الفيديو في النواة؟" وبمجرد وجود هذا العقد، يمكن Plugins المورّدين
أن تسجل ضمنه ويمكن Plugins القنوات/الميزات أن تستهلكه.

إذا لم تكن الإمكانية موجودة بعد، فعادةً تكون الخطوة الصحيحة هي:

1. تعريف الإمكانية المفقودة في النواة
2. كشفها عبر واجهة API الخاصة بـ Plugin/وقت التشغيل بطريقة مطبوعة
3. توصيل القنوات/الميزات بهذه الإمكانية
4. ترك Plugins المورّدين تسجل التنفيذات

يحافظ ذلك على وضوح الملكية مع تجنب سلوك في النواة يعتمد على
مورّد واحد أو مسار كود خاص بـ Plugin واحدة.

### طبقات الإمكانات

استخدم هذا النموذج الذهني عند تحديد موضع الكود:

- **طبقة الإمكانات في النواة**: التنسيق المشترك، والسياسات، والبدائل الاحتياطية، وقواعد
  دمج الإعدادات، ودلالات التسليم، والعقود المطبوعة
- **طبقة Plugin الخاصة بالمورّد**: واجهات API الخاصة بالمورّد، والمصادقة، وفهارس النماذج، وتوليف
  الكلام، وتوليد الصور، وواجهات الفيديو الخلفية المستقبلية، ونقاط نهاية الاستخدام
- **طبقة Plugin الخاصة بالقناة/الميزة**: تكامل Slack/Discord/voice-call/إلخ
  الذي يستهلك إمكانات النواة ويعرضها على سطح ما

على سبيل المثال، يتبع TTS هذا الشكل:

- تملك النواة سياسة TTS وقت الرد، وترتيب البدائل الاحتياطية، والتفضيلات، وتسليم القناة
- تملك `openai` و`elevenlabs` و`microsoft` تنفيذات التوليف
- تستهلك `voice-call` مساعد وقت تشغيل TTS الخاص بالاتصال الهاتفي

ويجب تفضيل هذا النمط نفسه للإمكانات المستقبلية.

### مثال على Plugin شركة متعددة الإمكانات

يجب أن تبدو Plugin الشركة متماسكة من الخارج. إذا كان OpenClaw يملك عقودًا
مشتركة للنماذج، والكلام، والنسخ الفوري، والصوت الفوري، وفهم الوسائط،
وتوليد الصور، وتوليد الفيديو، وجلب الويب، والبحث على الويب،
فيمكن لمورّد أن يملك جميع أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

المهم ليس أسماء المساعدات الدقيقة. المهم هو الشكل:

- Plugin واحدة تملك سطح المورّد
- لا تزال النواة تملك عقود الإمكانات
- تستهلك Plugins القنوات والميزات مساعدات `api.runtime.*`، لا كود المورّد
- يمكن لاختبارات العقد أن تؤكد أن Plugin سجلت الإمكانات التي
  تدّعي أنها تملكها

### مثال على الإمكانية: فهم الفيديو

يعامل OpenClaw بالفعل فهم الصور/الصوت/الفيديو باعتباره
إمكانية مشتركة واحدة. وينطبق نموذج الملكية نفسه هنا:

1. تعرّف النواة عقد media-understanding
2. تسجل Plugins المورّدين `describeImage` و`transcribeAudio` و
   `describeVideo` بحسب الاقتضاء
3. تستهلك Plugins القنوات والميزات السلوك المشترك في النواة بدلًا من
   التوصيل مباشرة إلى كود المورّد

ويمنع ذلك تضمين افتراضات الفيديو الخاصة بأحد الموفّرين داخل النواة. إذ تملك Plugin
سطح المورّد؛ بينما تملك النواة عقد الإمكانية وسلوك البدائل الاحتياطية.

ويستخدم توليد الفيديو بالفعل التسلسل نفسه: إذ تملك النواة
عقد الإمكانية المطبوعة ومساعد وقت التشغيل، وتسجل Plugins المورّدين
تنفيذات `api.registerVideoGenerationProvider(...)` ضمنه.

هل تحتاج إلى قائمة تحقق عملية للطرح؟ راجع
[دليل الإمكانات](/ar/plugins/architecture).

## العقود وفرضها

سطح API الخاص بـ Plugin مطبوع ومركزي عمدًا في
`OpenClawPluginApi`. يعرّف هذا العقد نقاط التسجيل المدعومة و
مساعدات وقت التشغيل التي يمكن لـ Plugin الاعتماد عليها.

لماذا هذا مهم:

- يحصل مؤلفو Plugins على معيار داخلي ثابت واحد
- يمكن للنواة رفض الملكية المكررة مثل تسجيل Pluginين لمعرف الموفّر نفسه
- يمكن لبدء التشغيل إظهار تشخيصات عملية لعمليات التسجيل غير الصحيحة
- يمكن لاختبارات العقد فرض ملكية Plugins المجمعة ومنع الانحراف الصامت

هناك طبقتان من الفرض:

1. **فرض التسجيل وقت التشغيل**
   يتحقق سجل Plugins من عمليات التسجيل أثناء تحميل Plugins. أمثلة:
   تؤدي معرفات الموفّرين المكررة، ومعرفات موفّري الكلام المكررة، وعمليات
   التسجيل غير الصحيحة إلى تشخيصات Plugin بدلًا من سلوك غير معرّف.
2. **اختبارات العقد**
   تُلتقط Plugins المجمعة في سجلات العقود أثناء تشغيل الاختبارات حتى
   يتمكن OpenClaw من تأكيد الملكية صراحةً. ويُستخدم هذا اليوم مع
   موفّري النماذج، وموفّري الكلام، وموفّري البحث على الويب، وملكية التسجيل المجمّع.

والأثر العملي هو أن OpenClaw يعرف مسبقًا أي Plugin تملك أي
سطح. وهذا يتيح للنواة والقنوات التركيب بسلاسة لأن الملكية
معلنة ومطبوعة وقابلة للاختبار بدلًا من أن تكون ضمنية.

### ما الذي ينتمي إلى عقد

العقود الجيدة الخاصة بـ Plugin تكون:

- مطبوعة
- صغيرة
- خاصة بالإمكانات
- مملوكة للنواة
- قابلة لإعادة الاستخدام بواسطة عدة Plugins
- قابلة للاستهلاك بواسطة القنوات/الميزات من دون معرفة بالمورّد

العقود السيئة الخاصة بـ Plugin تكون:

- سياسات خاصة بالمورّد مخفية داخل النواة
- منافذ هروب خاصة بـ Plugin واحدة تتجاوز السجل
- كود قناة يصل مباشرة إلى تنفيذ خاص بمورّد
- كائنات وقت تشغيل مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

عند الشك، ارفع مستوى التجريد: عرّف الإمكانية أولًا، ثم دع
Plugins تتصل بها.

## نموذج التنفيذ

تعمل Plugins OpenClaw الأصلية **داخل العملية** مع Gateway. وهي ليست
معزولة. وPlugin أصلية محمّلة لديها حد الثقة نفسه على مستوى العملية مثل
كود النواة.

النتائج المترتبة:

- يمكن لـ Plugin أصلية تسجيل أدوات، ومعالجات شبكة، ومرابط، وخدمات
- يمكن لخلل في Plugin أصلية أن يعطّل Gateway أو يزعزع استقراره
- تعادل Plugin أصلية خبيثة تنفيذ كود عشوائي داخل
  عملية OpenClaw

وتكون الحزم المتوافقة أكثر أمانًا افتراضيًا لأن OpenClaw يتعامل معها حاليًا
بوصفها حزم بيانات وصفية/محتوى. وفي الإصدارات الحالية، يعني هذا في الغالب
Skills المجمّعة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة بالنسبة إلى Plugins غير المجمعة. وتعامل
مع Plugins مساحة العمل بوصفها كودًا لوقت التطوير، لا إعدادات افتراضية للإنتاج.

وبالنسبة إلى أسماء حزم مساحة العمل المجمعة، أبقِ معرف Plugin مثبتًا في
اسم npm: `@openclaw/<id>` افتراضيًا، أو لاحقة مطبوعة معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding` عندما
تكشف الحزمة عمدًا عن دور Plugin أضيق.

ملاحظة ثقة مهمة:

- `plugins.allow` يثق في **معرفات Plugins**، لا في مصدر المنشأ.
- إن كانت Plugin مساحة عمل تحمل المعرف نفسه الذي تحمله Plugin مجمعة، فإنها
  تظلل النسخة المجمعة عمدًا عندما تكون Plugin مساحة العمل تلك ممكّنة/ضمن قائمة السماح.
- هذا سلوك طبيعي ومفيد للتطوير المحلي، واختبار التصحيحات، والإصلاحات العاجلة.

## حد التصدير

يصدّر OpenClaw الإمكانات، لا وسائل الراحة التنفيذية.

أبقِ تسجيل الإمكانات عامًا. وقلّص صادرات المساعدات غير التعاقدية:

- المسارات الفرعية الخاصة بمساعدات Plugin المجمعة
- المسارات الفرعية لسباكة وقت التشغيل غير المقصودة كواجهة API عامة
- مساعدات الراحة الخاصة بالمورّد
- مساعدات الإعداد/التهيئة التي تُعد تفاصيل تنفيذ

ولا تزال بعض المسارات الفرعية الخاصة بمساعدات Plugins المجمعة موجودة في خريطة
تصدير SDK المولدة لأجل التوافق وصيانة Plugins المجمعة. وتشمل الأمثلة الحالية
`plugin-sdk/feishu` و`plugin-sdk/feishu-setup` و`plugin-sdk/zalo`
و`plugin-sdk/zalo-setup` وعدة حدود `plugin-sdk/matrix*`. تعامل مع هذه
باعتبارها صادرات محفوظة لتفاصيل التنفيذ، لا باعتبارها النمط الموصى به في SDK
لـ Plugins الخارجية الجديدة.

## مسار التحميل

عند بدء التشغيل، ينفّذ OpenClaw تقريبًا ما يلي:

1. اكتشاف جذور Plugins المرشحة
2. قراءة بيانات Plugins الأصلية أو الحزم المتوافقة وبيانات الحزم الوصفية
3. رفض المرشحين غير الآمنين
4. تطبيع إعداد Plugin (`plugins.enabled` و`allow` و`deny` و`entries`،
   و`slots`، و`load.paths`)
5. تحديد التمكين لكل مرشح
6. تحميل الوحدات الأصلية الممكّنة عبر jiti
7. استدعاء مرابط `register(api)` الأصلية (أو `activate(api)` — وهو اسم مستعار قديم) وجمع عمليات التسجيل في سجل Plugin
8. كشف السجل إلى الأوامر وأسطح وقت التشغيل

<Note>
`activate` هو اسم مستعار قديم لـ `register` — إذ يحلّ المحمّل أيّهما موجود (`def.register ?? def.activate`) ويستدعيه في النقطة نفسها. تستخدم جميع Plugins المجمعة `register`؛ وفضّل `register` في Plugins الجديدة.
</Note>

تحدث بوابات الأمان **قبل** تنفيذ وقت التشغيل. ويُحظر المرشحون
عندما يخرج الإدخال عن جذر Plugin، أو يكون المسار قابلاً للكتابة من الجميع، أو
تبدو ملكية المسار مشبوهة بالنسبة إلى Plugins غير المجمعة.

### السلوك المعتمد على البيان أولًا

يُعد البيان مصدر الحقيقة لطائرة التحكم. ويستخدمه OpenClaw من أجل:

- تحديد Plugin
- اكتشاف القنوات/Skills/مخطط الإعداد المعلنة أو إمكانات الحزمة
- التحقق من `plugins.entries.<id>.config`
- تعزيز تسميات/عناصر نائبة في Control UI
- عرض بيانات التثبيت/الفهرس الوصفية
- الحفاظ على واصفات تنشيط وتهيئة منخفضة الكلفة من دون تحميل وقت تشغيل Plugin

بالنسبة إلى Plugins الأصلية، تكون وحدة وقت التشغيل هي جزء طائرة البيانات. فهي تسجل
السلوك الفعلي مثل المرابط، والأدوات، والأوامر، أو تدفقات الموفّر.

تظل كتل `activation` و`setup` الاختيارية في البيان ضمن طائرة التحكم.
فهي واصفات بيانات وصفية فقط لتخطيط التنشيط واكتشاف الإعداد؛
ولا تستبدل التسجيل في وقت التشغيل، أو `register(...)`، أو `setupEntry`.
ويستخدم أوائل مستهلكي التنشيط الحي الآن تلميحات الأوامر والقنوات والموفّرين في البيان
لتضييق تحميل Plugins قبل تجسيد السجل على نطاق أوسع:

- يضيّق تحميل CLI إلى Plugins التي تملك الأمر الأساسي المطلوب
- يضيّق حل إعداد القناة/Plugin إلى Plugins التي تملك
  معرف القناة المطلوب
- يضيّق حل إعداد/وقت تشغيل الموفّر الصريح إلى Plugins التي تملك
  معرف الموفّر المطلوب

يفضّل اكتشاف الإعداد الآن المعرفات المملوكة للواصفات مثل `setup.providers` و
`setup.cliBackends` لتضييق Plugins المرشحة قبل أن يعود إلى
`setup-api` بالنسبة إلى Plugins التي لا تزال تحتاج إلى مرابط وقت تشغيل أثناء الإعداد. وإذا ادّعت
أكثر من Plugin مكتشفة واحدة ملكية معرف موفّر إعداد أو خلفية CLI بعد
تطبيعه، يرفض بحث الإعداد المالك الملتبس بدلًا من الاعتماد على ترتيب الاكتشاف.

### ما الذي يخزنه المحمّل مؤقتًا

يحتفظ OpenClaw بمخازن مؤقتة قصيرة داخل العملية من أجل:

- نتائج الاكتشاف
- بيانات سجل البيان
- سجلات Plugins المحمّلة

تقلل هذه المخازن المؤقتة من زيادات بدء التشغيل المفاجئة ومن كلفة الأوامر المتكررة. ويمكن
التفكير فيها بأمان على أنها مخازن مؤقتة قصيرة العمر للأداء، لا وسائل تخزين دائم.

ملاحظة أداء:

- اضبط `OPENCLAW_DISABLE_PLUGIN_DISCOVERY_CACHE=1` أو
  `OPENCLAW_DISABLE_PLUGIN_MANIFEST_CACHE=1` لتعطيل هذه المخازن المؤقتة.
- اضبط نوافذ المخزن المؤقت عبر `OPENCLAW_PLUGIN_DISCOVERY_CACHE_MS` و
  `OPENCLAW_PLUGIN_MANIFEST_CACHE_MS`.

## نموذج السجل

لا تعدّل Plugins المحمّلة مباشرةً متغيرات النواة العامة العشوائية. بل تسجل في
سجل مركزي لـ Plugin.

يتتبع السجل:

- سجلات Plugin (الهوية، والمصدر، والمنشأ، والحالة، والتشخيصات)
- الأدوات
- المرابط القديمة والمرابط المطبوعة
- القنوات
- الموفّرون
- معالجات Gateway RPC
- مسارات HTTP
- مسجّلات CLI
- الخدمات الخلفية
- الأوامر المملوكة لـ Plugin

ثم تقرأ ميزات النواة من هذا السجل بدلًا من التحدث إلى وحدات Plugin
مباشرة. وهذا يُبقي التحميل أحادي الاتجاه:

- وحدة Plugin -> التسجيل في السجل
- وقت تشغيل النواة -> استهلاك السجل

هذا الفصل مهم لقابلية الصيانة. فهو يعني أن معظم أسطح النواة تحتاج فقط
إلى نقطة تكامل واحدة: "اقرأ السجل"، وليس "أضف حالة خاصة لكل وحدة Plugin".

## نداءات ربط المحادثة

يمكن لـ Plugins التي تربط محادثة أن تتفاعل عند حسم الموافقة.

استخدم `api.onConversationBindingResolved(...)` لتلقي نداء بعد الموافقة
على طلب الربط أو رفضه:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // A binding now exists for this plugin + conversation.
        console.log(event.binding?.conversationId);
        return;
      }

      // The request was denied; clear any local pending state.
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

حقول حمولة النداء:

- `status`: `"approved"` أو `"denied"`
- `decision`: `"allow-once"` أو `"allow-always"` أو `"deny"`
- `binding`: الربط المحسوم للطلبات الموافق عليها
- `request`: ملخص الطلب الأصلي، وتلميح الفك، ومعرف المرسل، و
  البيانات الوصفية للمحادثة

هذا النداء مخصص للإشعار فقط. فهو لا يغير من يُسمح له بربط
محادثة، ويعمل بعد انتهاء معالجة الموافقة في النواة.

## مرابط وقت تشغيل الموفّر

أصبح لدى Plugins الموفّر الآن طبقتان:

- بيانات البيان الوصفية: `providerAuthEnvVars` لبحث مصادقة الموفّر عبر البيئة
  منخفض الكلفة قبل تحميل وقت التشغيل، و`providerAuthAliases` لمتغيرات الموفّر
  التي تشترك في المصادقة، و`channelEnvVars` لبحث القناة/الإعداد عبر البيئة
  منخفض الكلفة قبل تحميل وقت التشغيل، بالإضافة إلى `providerAuthChoices`
  لتسميات الاختيار الخاصة بالتهيئة/المصادقة وبيانات أعلام CLI الوصفية منخفضة الكلفة قبل تحميل وقت التشغيل
- مرابط وقت الإعداد: `catalog` / `discovery` القديم بالإضافة إلى `applyConfigDefaults`
- مرابط وقت التشغيل: `normalizeModelId` و`normalizeTransport`،
  و`normalizeConfig`،
  و`applyNativeStreamingUsageCompat` و`resolveConfigApiKey`،
  و`resolveSyntheticAuth` و`resolveExternalAuthProfiles`،
  و`shouldDeferSyntheticProfileAuth`،
  و`resolveDynamicModel` و`prepareDynamicModel` و`normalizeResolvedModel`،
  و`contributeResolvedModelCompat` و`capabilities`،
  و`normalizeToolSchemas` و`inspectToolSchemas`،
  و`resolveReasoningOutputMode` و`prepareExtraParams` و`createStreamFn`،
  و`wrapStreamFn` و`resolveTransportTurnState`،
  و`resolveWebSocketSessionPolicy` و`formatApiKey` و`refreshOAuth`،
  و`buildAuthDoctorHint` و`matchesContextOverflowError`،
  و`classifyFailoverReason` و`isCacheTtlEligible`،
  و`buildMissingAuthMessage` و`suppressBuiltInModel` و`augmentModelCatalog`،
  و`isBinaryThinking` و`supportsXHighThinking`،
  و`resolveDefaultThinkingLevel` و`isModernModelRef` و`prepareRuntimeAuth`،
  و`resolveUsageAuth` و`fetchUsageSnapshot` و`createEmbeddingProvider`،
  و`buildReplayPolicy`،
  و`sanitizeReplayHistory` و`validateReplayTurns` و`onModelSelected`

لا يزال OpenClaw يملك حلقة الوكيل العامة، والتحويل الاحتياطي، ومعالجة
التفريغ، وسياسة الأدوات. وتمثل هذه المرابط سطح التوسعة الخاص بالسلوك
الخاص بالموفّر من دون الحاجة إلى نقل استدلال مخصص كامل.

استخدم `providerAuthEnvVars` في البيان عندما يكون لدى الموفّر بيانات اعتماد
قائمة على البيئة يجب أن تراها مسارات المصادقة/الحالة/منتقي النماذج العامة من دون تحميل
وقت تشغيل Plugin. واستخدم `providerAuthAliases` في البيان عندما يجب على
معرف موفّر واحد إعادة استخدام متغيرات البيئة وملفات تعريف المصادقة
والمصادقة المعتمدة على الإعداد وخيار تهيئة مفتاح API الخاص
بمعرف موفّر آخر. واستخدم `providerAuthChoices` في البيان عندما ينبغي لأسطح
CLI الخاصة بالتهيئة/اختيار المصادقة أن تعرف معرف اختيار الموفّر، وتسميات المجموعات،
وتوصيل المصادقة البسيط أحادي العلم من دون تحميل وقت تشغيل الموفّر. وأبقِ
`envVars` في وقت تشغيل الموفّر لتلميحات موجّهة للمشغل مثل تسميات
التهيئة أو متغيرات إعداد `client-id`/`client-secret` الخاصة بـ OAuth.

استخدم `channelEnvVars` في البيان عندما تكون لدى القناة مصادقة أو إعداد
يعتمد على البيئة ويجب أن تراه آليات الرجوع العامة إلى بيئة الصدفة، أو فحوصات
الإعداد/الحالة، أو مطالبات الإعداد من دون تحميل وقت تشغيل القناة.

### ترتيب المرابط واستخدامها

بالنسبة إلى Plugins النماذج/الموفّرين، يستدعي OpenClaw المرابط بهذا الترتيب التقريبي.
يمثل العمود "متى يُستخدم" دليل القرار السريع.

| 1   | `catalog`                          | نشر إعداد الموفّر في `models.providers` أثناء توليد `models.json`                                               | عندما يملك الموفّر فهرسًا أو إعدادات افتراضية أساسية لـ URL                                                                                 |

يقوم `normalizeModelId` و`normalizeTransport` و`normalizeConfig` أولًا بفحص
Plugin الموفّر المطابق، ثم ينتقل إلى Plugins الموفّرين الأخرى القادرة على استخدام المرابط
حتى يغيّر أحدها فعلًا معرف النموذج أو النقل/الإعداد. وهذا يُبقي
الطبقات الوسيطة للأسماء المستعارة/التوافق الخاصة بالموفّر عاملة من دون أن يضطر المستدعي إلى معرفة
أي Plugin مجمّعة تملك إعادة الكتابة. وإذا لم يقم أي مرابط موفّر بإعادة كتابة
إدخال إعداد مدعوم من عائلة Google، فسيظل مطبّع إعداد Google المجمّع
يطبّق تنظيف التوافق هذا.

إذا كان الموفّر يحتاج إلى بروتوكول wire مخصص بالكامل أو منفّذ طلبات مخصص،
فهذه فئة مختلفة من التوسعة. هذه المرابط مخصصة لسلوك الموفّر
الذي لا يزال يعمل ضمن حلقة الاستدلال العادية في OpenClaw.

### مثال على موفّر

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

### أمثلة مدمجة

- يستخدم Anthropic المرابط `resolveDynamicModel` و`capabilities` و`buildAuthDoctorHint`،
  و`resolveUsageAuth` و`fetchUsageSnapshot` و`isCacheTtlEligible`،
  و`resolveDefaultThinkingLevel` و`applyConfigDefaults` و`isModernModelRef`،
  و`wrapStreamFn` لأنه يملك التوافق المستقبلي لـ Claude 4.6،
  وتلميحات عائلة الموفّر، وإرشادات إصلاح المصادقة، وتكامل نقطة نهاية الاستخدام،
  وأهلية ذاكرة التخزين المؤقت للموجّه، والإعدادات الافتراضية المعتمدة على المصادقة، وسياسة
  التفكير الافتراضي/التكيفي لـ Claude، وتشكيل البث الخاص بـ Anthropic من أجل
  رؤوس beta، و`/fast` / `serviceTier`، و`context1m`.
- تبقى مساعدات البث الخاصة بـ Claude في Anthropic حاليًا ضمن الحد العام
  `api.ts` / `contract-api.ts` الخاص بـ Plugin المجمّعة نفسها. إذ يصدّر سطح
  الحزمة هذا `wrapAnthropicProviderStream` و`resolveAnthropicBetas`،
  و`resolveAnthropicFastMode` و`resolveAnthropicServiceTier`، إضافة إلى
  بُناة أغلفة Anthropic الأدنى مستوى بدلًا من توسيع SDK العامة حول
  قواعد رؤوس beta الخاصة بموفّر واحد.
- تستخدم OpenAI المرابط `resolveDynamicModel` و`normalizeResolvedModel` و
  `capabilities` بالإضافة إلى `buildMissingAuthMessage` و`suppressBuiltInModel`،
  و`augmentModelCatalog` و`supportsXHighThinking` و`isModernModelRef`
  لأنها تملك التوافق المستقبلي لـ GPT-5.4، وتطبيع
  `openai-completions` -> `openai-responses` المباشر الخاص بـ OpenAI،
  وتلميحات المصادقة الواعية بـ Codex، وإخفاء Spark، وصفوف قائمة OpenAI الاصطناعية،
  وسياسة التفكير/النموذج الحي لـ GPT-5؛ بينما تملك عائلة البث
  `openai-responses-defaults` أغلفة OpenAI Responses الأصلية المشتركة من أجل
  رؤوس الإسناد، و`/fast`/`serviceTier`، وإسهاب النص، وبحث الويب الأصلي لـ Codex،
  وتشكيل حمولة توافق الاستدلال، وإدارة سياق Responses.
- تستخدم OpenRouter المرابط `catalog` بالإضافة إلى `resolveDynamicModel` و
  `prepareDynamicModel` لأن الموفّر تمريري وقد يكشف معرفات نماذج جديدة قبل
  تحديثات الفهرس الثابت في OpenClaw؛ كما يستخدم
  `capabilities` و`wrapStreamFn` و`isCacheTtlEligible` لإبقاء
  رؤوس الطلب الخاصة بالموفّر، وبيانات التوجيه الوصفية، ورقع الاستدلال، وسياسة
  ذاكرة التخزين المؤقت للموجّه خارج النواة. وتأتي سياسة إعادة التشغيل الخاصة به من
  عائلة `passthrough-gemini`، بينما تملك عائلة البث `openrouter-thinking`
  حقن الاستدلال الوكيل وتخطي النماذج غير المدعومة / `auto`.
- يستخدم GitHub Copilot المرابط `catalog` و`auth` و`resolveDynamicModel` و
  `capabilities` بالإضافة إلى `prepareRuntimeAuth` و`fetchUsageSnapshot` لأنه
  يحتاج إلى تسجيل دخول الجهاز المملوك للموفّر، وسلوك الرجوع إلى النماذج،
  وخصائص تفريغ Claude، وتبادل رمز GitHub إلى رمز Copilot، ونقطة نهاية
  استخدام مملوكة للموفّر.
- يستخدم OpenAI Codex المرابط `catalog` و`resolveDynamicModel`،
  و`normalizeResolvedModel` و`refreshOAuth` و`augmentModelCatalog` بالإضافة
  إلى `prepareExtraParams` و`resolveUsageAuth` و`fetchUsageSnapshot` لأنه
  لا يزال يعمل على عمليات نقل OpenAI الأساسية لكنه يملك

## مساعدات وقت التشغيل

يمكن لـ Plugins الوصول إلى بعض مساعدات النواة المحددة عبر `api.runtime`. بالنسبة إلى TTS:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

ملاحظات:

- يعيد `textToSpeech` حمولة خرج TTS العادية في النواة لأسطح الملفات/الملاحظات الصوتية.
- يستخدم إعداد `messages.tts` الأساسي واختيار الموفّر.
- يعيد مخزنًا مؤقتًا لصوت PCM + معدل العينة. ويجب على Plugins إعادة أخذ العينات/الترميز للمورّدين.
- `listVoices` اختياري لكل موفّر. استخدمه لمنتقيات الأصوات المملوكة للمورّد أو تدفقات الإعداد.
- يمكن أن تتضمن قوائم الأصوات بيانات وصفية أغنى مثل اللغة والمنطقة، والجنس، ووسوم الشخصية لأجل منتقيات واعية بالموفّر.
- تدعم OpenAI وElevenLabs الاتصال الهاتفي اليوم. ولا تدعم Microsoft ذلك.

يمكن لـ Plugins أيضًا تسجيل موفّري الكلام عبر `api.registerSpeechProvider(...)`.

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

ملاحظات:

- أبقِ سياسة TTS، والبدائل الاحتياطية، وتسليم الردود في النواة.
- استخدم موفّري الكلام لسلوك التوليف المملوك للمورّد.
- يُطبَّع إدخال Microsoft `edge` القديم إلى معرف الموفّر `microsoft`.
- نموذج الملكية المفضل موجه نحو الشركة: يمكن لـ Plugin مورّد واحدة أن تملك
  موفّري النص والكلام والصور والوسائط المستقبلية مع إضافة OpenClaw لتلك
  العقود الخاصة بالإمكانات.

بالنسبة إلى فهم الصور/الصوت/الفيديو، تسجل Plugins موفّرًا مطبوعًا واحدًا لـ
media-understanding بدلًا من حقيبة مفتاح/قيمة عامة:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

ملاحظات:

- أبقِ التنسيق، والبدائل الاحتياطية، والإعداد، وتوصيل القنوات في النواة.
- أبقِ سلوك المورّد في Plugin الموفّر.
- يجب أن يبقى التوسع الإضافي مطبوعًا: أساليب اختيارية جديدة، وحقول نتائج
  اختيارية جديدة، وإمكانات اختيارية جديدة.
- يتبع توليد الفيديو النمط نفسه بالفعل:
  - تملك النواة عقد الإمكانية ومساعد وقت التشغيل
  - تسجل Plugins المورّدين `api.registerVideoGenerationProvider(...)`
  - تستهلك Plugins الميزات/القنوات `api.runtime.videoGeneration.*`

وبالنسبة إلى مساعدات وقت تشغيل media-understanding، يمكن لـ Plugins استدعاء:

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});

const video = await api.runtime.mediaUnderstanding.describeVideoFile({
  filePath: "/tmp/inbound-video.mp4",
  cfg: api.config,
});
```

وبالنسبة إلى نسخ الصوت، يمكن لـ Plugins استخدام وقت تشغيل
media-understanding أو الاسم المستعار الأقدم STT:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // Optional when MIME cannot be inferred reliably:
  mime: "audio/ogg",
});
```

ملاحظات:

- يُعد `api.runtime.mediaUnderstanding.*` السطح المشترك المفضل من أجل
  فهم الصور/الصوت/الفيديو.
- يستخدم إعداد الصوت الخاص بـ media-understanding في النواة (`tools.media.audio`) وترتيب البدائل الاحتياطية الخاص بالموفّر.
- يعيد `{ text: undefined }` عندما لا يتم إنتاج مخرجات نسخ (على سبيل المثال عند تخطي الإدخال أو عدم دعمه).
- يبقى `api.runtime.stt.transcribeAudioFile(...)` اسمًا مستعارًا للتوافق.

يمكن لـ Plugins أيضًا إطلاق تشغيلات وكلاء فرعيين في الخلفية عبر `api.runtime.subagent`:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

ملاحظات:

- `provider` و`model` هما تجاوزان اختياريان لكل تشغيل، وليسا تغييرات دائمة على الجلسة.
- لا يلتزم OpenClaw بحقول التجاوز هذه إلا للمستدعين الموثوقين.
- بالنسبة إلى تشغيلات البدائل الاحتياطية المملوكة لـ Plugin، يجب على المشغلين تفعيل ذلك صراحةً عبر `plugins.entries.<id>.subagent.allowModelOverride: true`.
- استخدم `plugins.entries.<id>.subagent.allowedModels` لتقييد Plugins الموثوقة بأهداف `provider/model` القياسية المحددة، أو `"*"` للسماح بأي هدف صراحةً.
- تظل تشغيلات الوكلاء الفرعيين الخاصة بـ Plugins غير الموثوقة تعمل، لكن طلبات التجاوز تُرفض بدلًا من الرجوع الاحتياطي بصمت.

وبالنسبة إلى البحث على الويب، يمكن لـ Plugins استهلاك مساعد وقت التشغيل المشترك بدلًا من
الوصول إلى توصيل أداة الوكيل:

```ts
const providers = api.runtime.webSearch.listProviders({
  config: api.config,
});

const result = await api.runtime.webSearch.search({
  config: api.config,
  args: {
    query: "OpenClaw plugin runtime helpers",
    count: 5,
  },
});
```

كما يمكن لـ Plugins أيضًا تسجيل موفّري البحث على الويب عبر
`api.registerWebSearchProvider(...)`.

ملاحظات:

- أبقِ اختيار الموفّر، وحل بيانات الاعتماد، ودلالات الطلب المشتركة في النواة.
- استخدم موفّري البحث على الويب لعمليات النقل الخاصة بالبحث والمملوكة للمورّد.
- يُعد `api.runtime.webSearch.*` السطح المشترك المفضل بالنسبة إلى Plugins الميزات/القنوات التي تحتاج إلى سلوك البحث من دون الاعتماد على غلاف أداة الوكيل.

### `api.runtime.imageGeneration`

```ts
const result = await api.runtime.imageGeneration.generate({
  config: api.config,
  args: { prompt: "A friendly lobster mascot", size: "1024x1024" },

const providers = api.runtime.imageGeneration.listProviders({
```

## مسارات Gateway HTTP

يمكن لـ Plugins كشف نقاط نهاية HTTP عبر `api.registerHttpRoute(...)`.

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

حقول المسار:

- `path`: مسار route ضمن خادم Gateway HTTP.
- `auth`: مطلوب. استخدم `"gateway"` لطلب مصادقة Gateway العادية، أو `"plugin"` للمصادقة التي تديرها Plugin/التحقق من Webhook.
- `match`: اختياري. `"exact"` (الافتراضي) أو `"prefix"`.
- `replaceExisting`: اختياري. يتيح لـ Plugin نفسها استبدال تسجيل مسارها الحالي.
- `handler`: أعد `true` عندما يكون المسار قد عالج الطلب.

ملاحظات:

- تمت إزالة `api.registerHttpHandler(...)` وسيتسبب في خطأ تحميل Plugin. استخدم `api.registerHttpRoute(...)` بدلًا منه.
- يجب على مسارات Plugin التصريح عن `auth` صراحةً.
- تُرفض تعارضات `path + match` المطابقة إلا إذا كان `replaceExisting: true`، ولا يمكن لـ Plugin واحدة استبدال مسار Plugin أخرى.
- تُرفض المسارات المتداخلة ذات مستويات `auth` المختلفة. أبقِ سلاسل السقوط `exact`/`prefix` ضمن مستوى auth نفسه فقط.
- لا تتلقى مسارات `auth: "plugin"` تلقائيًا نطاقات وقت تشغيل المشغل. فهي مخصصة لـ Webhook الذي تديره Plugin/التحقق من التوقيع، وليست لاستدعاءات مساعدات Gateway ذات الامتيازات.
- تعمل مسارات `auth: "gateway"` داخل نطاق وقت تشغيل طلب Gateway، لكن هذا النطاق محافظ عمدًا:
  - تبقي مصادقة bearer ذات السر المشترك (`gateway.auth.mode = "token"` / `"password"`) نطاقات وقت تشغيل مسار Plugin مثبتة على `operator.write`، حتى لو أرسل المستدعي `x-openclaw-scopes`
  - تحترم أوضاع HTTP الموثوقة والحاملة للهوية (على سبيل المثال `trusted-proxy` أو `gateway.auth.mode = "none"` على ingress خاص) قيمة `x-openclaw-scopes` فقط عندما يكون الرأس موجودًا صراحةً
  - إذا كان `x-openclaw-scopes` غائبًا في طلبات مسار Plugin الحاملة للهوية تلك، يعود نطاق وقت التشغيل إلى `operator.write`
- القاعدة العملية: لا تفترض أن مسار Plugin ذي مصادقة gateway هو سطح إدارة ضمني. إذا كان مسارك يحتاج إلى سلوك خاص بالمشرف فقط، فاطلب وضع مصادقة حاملًا للهوية ووثّق عقد الرأس `x-openclaw-scopes` الصريح.

- تظل الأجزاء الداخلية للإضافات المجمعة خاصة. يجب على Plugins الخارجية استخدام

  `<plugin-package-root>/api.js` هو حاوية المساعدات/الأنواع،
  و`<plugin-package-root>/runtime-api.js` هو حاوية وقت التشغيل فقط،
  و`<plugin-package-root>/index.js` هو إدخال Plugin المجمعة،
  و`<plugin-package-root>/setup-entry.js` هو إدخال Plugin للإعداد.
- أمثلة موفّرين مجمّعين حاليين:
  - يستخدم Anthropic `api.js` / `contract-api.js` لمساعدات بث Claude مثل
    `wrapAnthropicProviderStream`، ومساعدات رؤوس beta، وتحليل `service_tier`.
  - تستخدم OpenAI `api.js` لبناة الموفّر، ومساعدات النموذج الافتراضي، وبناة
    موفّر الوقت الحقيقي.
  - تستخدم OpenRouter `api.js` لباني الموفّر الخاص بها إضافة إلى مساعدات
    الإعداد/التهيئة، بينما يمكن لـ `register.runtime.js` أن يعيد تصدير مساعدات
    `plugin-sdk/provider-stream` العامة للاستخدام المحلي في المستودع.
- تفضّل نقاط الإدخال العامة المحمّلة عبر الواجهة Snapshot إعداد وقت التشغيل النشط
  عندما يكون موجودًا، ثم تعود إلى ملف الإعداد المحلول على القرص عندما
  لا يكون OpenClaw قد بدأ بعد في تقديم Snapshot وقت تشغيل.
- تظل البدائيات العامة المشتركة هي عقد SDK العام المفضل. ولا تزال توجد
  مجموعة توافق صغيرة ومحفوظة من حدود المساعدات الموسومة بأسماء القنوات المجمعة. تعامل
  مع تلك باعتبارها حدود صيانة/توافق خاصة بالحزم المجمعة، لا أهداف استيراد جديدة
  لأطراف ثالثة؛ ويجب أن تُضاف العقود الجديدة العابرة للقنوات إلى
  المسارات الفرعية العامة `plugin-sdk/*` أو إلى الحاويات المحلية لـ Plugin مثل `api.js` /
  `runtime-api.js`.

ملاحظة التوافق:

- تجنب حاوية الجذر `openclaw/plugin-sdk` في الكود الجديد.
- فضّل أولًا البدائيات الثابتة الضيقة. فالمسارات الفرعية الأحدث الخاصة بالإعداد/الاقتران/الرد/
  الملاحظات/العقد/الوارد/الخيوط/الأوامر/secret-input/Webhook/infra/
  allowlist/status/message-tool هي العقد المقصود لأعمال Plugins الجديدة
  المجمعة والخارجية.
  ينتمي تحليل/مطابقة الهدف إلى `openclaw/plugin-sdk/channel-targets`.
  وتنتمي بوابات أفعال الرسائل ومساعدات معرف رسالة التفاعل إلى
  `openclaw/plugin-sdk/channel-actions`.
- لا تُعد حاويات المساعدات الخاصة بالإضافات المجمعة والمحددة افتراضيًا ثابتة. إذا كانت
  المساعدة مطلوبة فقط لإضافة مجمعة، فأبقها خلف الحد المحلي
  `api.js` أو `runtime-api.js` الخاص بتلك الإضافة بدلًا من ترقيتها إلى
  `openclaw/plugin-sdk/<extension>`.
- يجب أن تكون حدود المساعدات المشتركة الجديدة عامة، لا موسومة باسم قناة. ينتمي تحليل
  الأهداف المشتركة إلى `openclaw/plugin-sdk/channel-targets`؛ أما الأجزاء الداخلية
  الخاصة بالقناة فتبقى خلف الحد المحلي `api.js` أو `runtime-api.js`
  المملوك لـ Plugin.
- توجد مسارات فرعية خاصة بالإمكانات مثل `image-generation`،
  و`media-understanding`، و`speech` لأن Plugins المجمعة/الأصلية تستخدمها
  اليوم. ووجودها لا يعني بحد ذاته أن كل مساعدة مُصدّرة هي
  عقدًا خارجيًا مجمّدًا على المدى الطويل.

## مخططات أداة الرسائل

يجب أن تملك Plugins مساهمات المخطط الخاصة بـ `describeMessageTool(...)`
الخاصة بالقناة. أبقِ الحقول الخاصة بالموفّر داخل Plugin، لا في النواة المشتركة.

وبالنسبة إلى أجزاء المخطط المحمولة المشتركة، أعد استخدام المساعدات العامة المصدّرة عبر
`openclaw/plugin-sdk/channel-actions`:

- `createMessageToolButtonsSchema()` لحمولات بأسلوب شبكة الأزرار
- `createMessageToolCardSchema()` لحمولات البطاقات المنظمة

إذا كان شكل المخطط لا يكون منطقيًا إلا لموفّر واحد، فعرّفه في
مصدر Plugin نفسه بدلًا من ترقيته إلى SDK المشتركة.

## حل هدف القناة

يجب أن تملك Plugins القنوات دلالات الأهداف الخاصة بالقناة. أبقِ
مضيف الإرسال الخارجي عامًا واستخدم سطح مهايئ المراسلة لقواعد الموفّر:

- يقرر `messaging.inferTargetChatType({ to })` ما إذا كان الهدف المطبّع
  يجب أن يُعامل كـ `direct` أو `group` أو `channel` قبل البحث في الدليل.
- يخبر `messaging.targetResolver.looksLikeId(raw, normalized)` النواة بما إذا كان
  ينبغي لإدخال ما أن يتجاوز مباشرة إلى حل شبيه بالمعرف بدلًا من البحث في الدليل.
- يُعد `messaging.targetResolver.resolveTarget(...)` بديل Plugin الاحتياطي عندما
  تحتاج النواة إلى حل نهائي مملوك للموفّر بعد التطبيع أو بعد
  عدم العثور في الدليل.
- يملك `messaging.resolveOutboundSessionRoute(...)` إنشاء مسار الجلسة الخاص بالموفّر
  بمجرد حل الهدف.

التقسيم الموصى به:

- استخدم `inferTargetChatType` لقرارات التصنيف التي يجب أن تحدث قبل
  البحث في الأقران/المجموعات.
- استخدم `looksLikeId` لفحوصات "عامل هذا كمعرف هدف صريح/أصلي".
- استخدم `resolveTarget` كبديل تطبيع خاص بالموفّر، لا
  للبحث الواسع في الدليل.
- أبقِ المعرفات الأصلية الخاصة بالموفّر مثل معرفات الدردشة، ومعرفات الخيوط، وJIDs،
  والمعالجات، ومعرفات الغرف داخل قيم `target` أو ضمن معاملات خاصة بالموفّر، لا
  في حقول SDK عامة.

## الأدلة المعتمدة على الإعداد

يجب على Plugins التي تستمد إدخالات الدليل من الإعداد أن تُبقي هذا المنطق داخل
Plugin وأن تعيد استخدام المساعدات المشتركة من
`openclaw/plugin-sdk/directory-runtime`.

استخدم هذا عندما تحتاج القناة إلى أقران/مجموعات معتمدة على الإعداد مثل:

- أقران الرسائل المباشرة المدفوعة بقائمة السماح
- خرائط القنوات/المجموعات المكوّنة
- بدائل الدليل الثابتة ذات النطاق الخاص بالحساب

لا تتعامل المساعدات المشتركة في `directory-runtime` إلا مع العمليات العامة:

- تصفية الاستعلام
- تطبيق الحد
- مساعدات إزالة التكرار/التطبيع
- بناء `ChannelDirectoryEntry[]`

يجب أن يبقى فحص الحساب الخاص بالقناة وتطبيع المعرفات داخل
تنفيذ Plugin.

## فهارس الموفّرين

يمكن لـ Plugins الموفّر تعريف فهارس النماذج للاستدلال باستخدام
`registerProvider({ catalog: { run(...) { ... } } })`.

يعيد `catalog.run(...)` البنية نفسها التي يكتبها OpenClaw في
`models.providers`:

- `{ provider }` لإدخال موفّر واحد
- `{ providers }` لعدة إدخالات موفّر

استخدم `catalog` عندما تملك Plugin معرفات نماذج خاصة بالموفّر، أو إعدادات افتراضية لـ base URL، أو بيانات وصفية للنماذج محكومة بالمصادقة.

يتحكم `catalog.order` في توقيت دمج فهرس Plugin نسبةً إلى
الموفّرين الضمنيين المدمجين في OpenClaw:

- `simple`: موفّرون عاديون معتمدون على مفتاح API أو البيئة
- `profile`: موفّرون يظهرون عند وجود ملفات تعريف للمصادقة
- `paired`: موفّرون يركّبون عدة إدخالات موفّر مترابطة
- `late`: التمرير الأخير، بعد الموفّرين الضمنيين الآخرين

يفوز الموفّرون اللاحقون عند تعارض المفاتيح، لذلك يمكن لـ Plugins أن
تتجاوز عمدًا إدخال موفّر مدمجًا يحمل معرف الموفّر نفسه.

التوافق:

- لا يزال `discovery` يعمل كاسم مستعار قديم
- إذا تم تسجيل `catalog` و`discovery` معًا، يستخدم OpenClaw `catalog`

## فحص القنوات للقراءة فقط

إذا كانت Plugin لديك تسجل قناة، فافضّل تنفيذ
`plugin.config.inspectAccount(cfg, accountId)` إلى جانب `resolveAccount(...)`.

السبب:

- يمثل `resolveAccount(...)` مسار وقت التشغيل. ويُسمح له بأن يفترض أن بيانات الاعتماد
  قد جُسدت بالكامل ويمكنه الفشل بسرعة عند غياب الأسرار المطلوبة.
- لا ينبغي لمسارات الأوامر المخصصة للقراءة فقط مثل `openclaw status` و`openclaw status --all`،
  و`openclaw channels status`، و`openclaw channels resolve`، وتدفقات
  إصلاح doctor/config أن تحتاج إلى تجسيد بيانات اعتماد وقت التشغيل فقط من أجل
  وصف الإعداد.

السلوك الموصى به لـ `inspectAccount(...)`:

- أعد فقط حالة حساب وصفية.
- حافظ على `enabled` و`configured`.
- ضمّن حقول مصدر/حالة بيانات الاعتماد عند الصلة، مثل:
  - `tokenSource`، `tokenStatus`
  - `botTokenSource`، `botTokenStatus`
  - `appTokenSource`، `appTokenStatus`
  - `signingSecretSource`، `signingSecretStatus`
- لا تحتاج إلى إرجاع قيم الرموز الخام فقط للإبلاغ عن
  التوفر للقراءة فقط. يكفي إرجاع `tokenStatus: "available"` (مع حقل المصدر المطابق)
  لأوامر على نمط الحالة.
- استخدم `configured_unavailable` عندما تكون بيانات الاعتماد مضبوطة عبر SecretRef لكن
  غير متاحة في مسار الأمر الحالي.

وهذا يتيح للأوامر المخصصة للقراءة فقط الإبلاغ عن "مضبوط لكنه غير متاح في مسار الأمر هذا"
بدلًا من التعطل أو الإبلاغ بشكل خاطئ أن الحساب غير مضبوط.

## حزم Package

يمكن أن يتضمن دليل Plugin ملف `package.json` مع `openclaw.extensions`:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

يصبح كل إدخال Plugin. وإذا سردت الحزمة عدة extensions، فإن معرف Plugin
يصبح `name/<fileBase>`.

إذا كانت Plugin لديك تستورد تبعيات npm، فثبّتها في ذلك الدليل حتى
يصبح `node_modules` متاحًا (`npm install` / `pnpm install`).

حاجز أمان: يجب أن يبقى كل إدخال في `openclaw.extensions` داخل دليل Plugin
بعد حل الروابط الرمزية. وتُرفض الإدخالات التي تخرج من دليل الحزمة.

ملاحظة أمان: يقوم `openclaw plugins install` بتثبيت تبعيات Plugin باستخدام
`npm install --omit=dev --ignore-scripts` (من دون نصوص دورة حياة، ومن دون تبعيات تطوير وقت التشغيل). أبقِ أشجار تبعيات Plugin
"JavaScript/TypeScript خالصة" وتجنب الحزم التي تتطلب عمليات بناء `postinstall`.

اختياري: يمكن أن يشير `openclaw.setupEntry` إلى وحدة خفيفة مخصصة للإعداد فقط.
عندما يحتاج OpenClaw إلى أسطح إعداد لقناة Plugin معطّلة، أو
عندما تكون قناة Plugin ممكّنة لكنها لا تزال غير مضبوطة، فإنه يحمّل `setupEntry`
بدلًا من الإدخال الكامل لـ Plugin. ويبقي ذلك بدء التشغيل والإعداد أخف
عندما يكون إدخال Plugin الرئيسي أيضًا يوصل أدوات أو مرابط أو كودًا آخر
خاصًا بوقت التشغيل فقط.

اختياري: يمكن أن يفعّل
`openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
لقناة Plugin المسار نفسه `setupEntry` خلال
مرحلة بدء التشغيل السابقة للاستماع في Gateway، حتى عندما تكون القناة مضبوطة بالفعل.

استخدم هذا فقط عندما يغطي `setupEntry` سطح بدء التشغيل بالكامل الذي يجب أن يوجد
قبل أن تبدأ Gateway بالاستماع. وعمليًا، هذا يعني أن إدخال الإعداد
يجب أن يسجل كل إمكانية مملوكة للقناة يعتمد عليها بدء التشغيل، مثل:

- تسجيل القناة نفسها
- أي مسارات HTTP يجب أن تكون متاحة قبل أن تبدأ Gateway بالاستماع
- أي أساليب Gateway أو أدوات أو خدمات يجب أن توجد خلال النافذة نفسها

إذا كان الإدخال الكامل لا يزال يملك أي إمكانية مطلوبة لبدء التشغيل، فلا تفعّل
هذا العلم. وأبقِ Plugin على السلوك الافتراضي ودع OpenClaw يحمّل الإدخال
الكامل أثناء بدء التشغيل.

يمكن للقنوات المجمعة أيضًا نشر مساعدات سطح عقد مخصصة للإعداد فقط يمكن للنواة
الرجوع إليها قبل تحميل وقت تشغيل القناة الكامل. ويكون سطح ترقية الإعداد الحالي:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

تستخدم النواة هذا السطح عندما تحتاج إلى ترقية إعداد قناة قديم ذي حساب واحد
إلى `channels.<id>.accounts.*` من دون تحميل الإدخال الكامل لـ Plugin.
ويمثل Matrix المثال المجمّع الحالي: فهو ينقل فقط مفاتيح المصادقة/bootstrap
إلى حساب مُرقّى مسمى عندما تكون الحسابات المسماة موجودة بالفعل، ويمكنه
الحفاظ على مفتاح حساب افتراضي مضبوط وغير قياسي بدلًا من إنشاء
`accounts.default` دائمًا.

تحافظ مهايئات رقع الإعداد هذه على كسل اكتشاف سطح العقد المجمّع. إذ يبقى وقت
الاستيراد خفيفًا؛ ويتم تحميل سطح الترقية فقط عند أول استخدام بدلًا من إعادة
الدخول في بدء تشغيل القناة المجمعة عند استيراد الوحدة.

وعندما تتضمن أسطح بدء التشغيل تلك أساليب Gateway RPC، فأبقها ضمن
بادئة خاصة بـ Plugin. فمساحات أسماء إدارة النواة (`config.*`،
`exec.approvals.*`، `wizard.*`، `update.*`) تبقى محفوظة وتُحل دائمًا
إلى `operator.admin`، حتى لو طلبت Plugin نطاقًا أضيق.

مثال:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }
  }
}
```

### البيانات الوصفية لفهرس القناة

يمكن لـ Plugins القنوات الإعلان عن بيانات الإعداد/الاكتشاف الوصفية عبر `openclaw.channel` و
تلميحات التثبيت عبر `openclaw.install`. وهذا يُبقي بيانات فهرس النواة خالية من البيانات.

مثال:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (self-hosted)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "Self-hosted chat via Nextcloud Talk webhook bots.",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

حقول `openclaw.channel` المفيدة إلى جانب المثال الأدنى:

- `detailLabel`: تسمية ثانوية لأسطح الفهرس/الحالة الأكثر غنى
- `docsLabel`: تجاوز نص الرابط الخاص بالوثائق
- `preferOver`: معرفات Plugin/قناة الأقل أولوية التي يجب أن يتفوق عليها هذا الإدخال في الفهرس
- `selectionDocsPrefix` و`selectionDocsOmitLabel` و`selectionExtras`: عناصر تحكم في النص لسطح الاختيار
- `markdownCapable`: يميز القناة على أنها قادرة على Markdown لقرارات التنسيق الصادر
- `exposure.configured`: إخفاء القناة من أسطح عرض القنوات المضبوطة عندما تضبط على `false`
- `exposure.setup`: إخفاء القناة من منتقيات الإعداد/التهيئة التفاعلية عندما تضبط على `false`
- `exposure.docs`: وسم القناة على أنها داخلية/خاصة لأسطح التنقل في الوثائق
- `showConfigured` / `showInSetup`: أسماء مستعارة قديمة ما تزال مقبولة للتوافق؛ وفضّل `exposure`
- `quickstartAllowFrom`: ضم القناة إلى تدفق `allowFrom` القياسي في البدء السريع
- `forceAccountBinding`: طلب ربط صريح للحساب حتى عند وجود حساب واحد فقط
- `preferSessionLookupForAnnounceTarget`: تفضيل البحث في الجلسة عند حل أهداف الإشعار

يمكن لـ OpenClaw أيضًا دمج **فهارس قنوات خارجية** (على سبيل المثال، تصدير سجل
MPM). ضع ملف JSON في أحد المواقع التالية:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

أو وجّه `OPENCLAW_PLUGIN_CATALOG_PATHS` (أو `OPENCLAW_MPM_CATALOG_PATHS`) إلى
ملف JSON واحد أو أكثر (مفصولة بفواصل/فواصل منقوطة/محددة بـ `PATH`). يجب أن
يحتوي كل ملف على `{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }`. كما يقبل المحلل
`"packages"` أو `"plugins"` بوصفها أسماء مستعارة قديمة للمفتاح `"entries"`.

## Plugins محرّك السياق

تملك Plugins محرّك السياق تنسيق سياق الجلسة لعمليات الإدخال والتجميع و
Compaction. سجّلها من Plugin لديك باستخدام
`api.registerContextEngine(id, factory)`، ثم حدّد المحرّك النشط عبر
`plugins.slots.contextEngine`.

استخدم هذا عندما تحتاج Plugin لديك إلى استبدال مسار السياق الافتراضي أو توسيعه
بدلًا من مجرد إضافة بحث في الذاكرة أو مرابط.

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("lossless-claw", () => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

إذا كان محرّكك **لا** يملك خوارزمية Compaction، فأبقِ `compact()`
منفّذًا وفوّضه صراحةً:

```ts
import {
  buildMemorySystemPromptAddition,
  delegateCompactionToRuntime,
} from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", () => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact(params) {
      return await delegateCompactionToRuntime(params);
    },
  }));
}
```

## إضافة إمكانية جديدة

عندما تحتاج Plugin إلى سلوك لا يناسب API الحالية، فلا تتجاوز
نظام Plugins عبر توغل خاص. أضف الإمكانية المفقودة.

التسلسل الموصى به:

1. تعريف عقد النواة
   قرر السلوك المشترك الذي يجب أن تملكه النواة: السياسة، والبديل الاحتياطي، ودمج الإعداد،
   ودورة الحياة، والدلالات المواجهة للقناة، وشكل مساعد وقت التشغيل.
2. إضافة أسطح Plugin/وقت التشغيل المطبوعة للتسجيل
   وسّع `OpenClawPluginApi` و/أو `api.runtime` بأصغر
   سطح إمكانية مطبوع ومفيد.
3. توصيل النواة + مستهلكي القنوات/الميزات
   يجب أن تستهلك Plugins القنوات والميزات الإمكانية الجديدة عبر النواة،
   لا عن طريق استيراد تنفيذ مورّد مباشرة.
4. تسجيل تنفيذات المورّد
   عندها تسجّل Plugins المورّدين واجهاتها الخلفية ضمن الإمكانية.
5. إضافة تغطية للعقود
   أضف اختبارات حتى تبقى الملكية وشكل التسجيل صريحين بمرور الوقت.

بهذه الطريقة يبقى OpenClaw صاحب رأي من دون أن يصبح مضمّنًا بشكل صلب في
رؤية موفّر واحد للعالم. راجع [دليل الإمكانات](/ar/plugins/architecture)
للحصول على قائمة ملفات عملية ومثال مطبّق.

### قائمة التحقق الخاصة بالإمكانات

عندما تضيف إمكانية جديدة، فعادةً يجب أن يطال التنفيذ
هذه الأسطح معًا:

- أنواع عقد النواة في `src/<capability>/types.ts`
- مساعد المشغّل/وقت التشغيل في النواة في `src/<capability>/runtime.ts`
- سطح تسجيل API الخاصة بـ Plugin في `src/plugins/types.ts`
- توصيل سجل Plugin في `src/plugins/registry.ts`
- كشف وقت تشغيل Plugin في `src/plugins/runtime/*` عندما تحتاج
  Plugins الميزات/القنوات إلى استهلاكه
- مساعدات الالتقاط/الاختبار في `src/test-utils/plugin-registration.ts`
- تأكيدات الملكية/العقد في `src/plugins/contracts/registry.ts`
- وثائق المشغل/Plugin في `docs/`

إذا كان أحد هذه الأسطح مفقودًا، فغالبًا ما تكون هذه علامة على أن الإمكانية
لم تُدمج بالكامل بعد.

### قالب الإمكانية

النمط الأدنى:

```ts
// core contract
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// plugin API
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// shared runtime helper for feature/channel plugins
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

نمط اختبار العقد:

```ts
expect(findVideoGenerationProviderIdsForPlugin("openai")).toEqual(["openai"]);
```

وهذا يُبقي القاعدة بسيطة:

- تملك النواة عقد الإمكانية + التنسيق
- تملك Plugins المورّدين تنفيذات المورّد
- تستهلك Plugins الميزات/القنوات مساعدات وقت التشغيل
- تُبقي اختبارات العقد الملكية صريحة
