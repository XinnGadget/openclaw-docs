# Управление сеансами

OpenClaw организует диалоги в **сеансы**. Каждое сообщение направляется в сеанс в зависимости от того, откуда оно поступило — из личных сообщений (DM), групповых чатов, заданий cron и т. д.

## Как маршрутизируются сообщения

| Источник | Поведение |
| --------------- | ------------------------- |
| Личные сообщения (DM) | По умолчанию используется общий сеанс |
| Групповые чаты | Изолированы для каждой группы |
| Комнаты/каналы | Изолированы для каждой комнаты |
| Задания cron | Новый сеанс при каждом запуске |
| Вебхуки | Изолированы для каждого хука |

## Изоляция личных сообщений (DM)

По умолчанию все личные сообщения используют один сеанс для обеспечения непрерывности диалога. Это подходит для однопользовательских конфигураций.

<Warning>
Если несколько человек могут отправлять сообщения вашему агенту, включите изоляцию DM. Без неё все пользователи будут использовать один и тот же контекст диалога — личные сообщения Алисы будут видны Бобу.
</Warning>

**Решение:**

```json5
{
  session: {
    dmScope: "per-channel-peer", // изоляция по каналу + отправителю
  },
}
```

Другие варианты:

- `main` (по умолчанию) — все DM используют один сеанс.
- `per-peer` — изоляция по отправителю (для всех каналов).
- `per-channel-peer` — изоляция по каналу + отправителю (рекомендуется).
- `per-account-channel-peer` — изоляция по учётной записи + каналу + отправителю.

<Tip>
Если один и тот же человек связывается с вами через несколько каналов, используйте `session.identityLinks`, чтобы связать его идентичности — тогда он будет использовать один сеанс.
</Tip>

Проверьте свою конфигурацию с помощью команды `openclaw security audit`.

## Жизненный цикл сеанса

Сеансы повторно используются до истечения срока их действия:

- **Ежедневный сброс** (по умолчанию) — новый сеанс в 4:00 по местному времени на хосте шлюза.
- **Сброс при бездействии** (опционально) — новый сеанс после периода неактивности. Задайте параметр `session.reset.idleMinutes`.
- **Ручной сброс** — введите `/new` или `/reset` в чате. Команда `/new <model>` также позволяет сменить модель.

Если настроены и ежедневный сброс, и сброс при бездействии, приоритет отдаётся тому, который истекает первым.

## Где хранится состояние

Всё состояние сеанса принадлежит **шлюзу**. Клиентские интерфейсы запрашивают данные сеанса у шлюза.

- **Хранилище:** `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- **Транскрипты:** `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`

## Обслуживание сеансов

OpenClaw автоматически ограничивает объём хранилища сеансов с течением времени. По умолчанию он работает в режиме `warn` (сообщает о том, что будет удалено). Чтобы включить автоматическую очистку, задайте для `session.maintenance.mode` значение `"enforce"`:

```json5
{
  session: {
    maintenance: {
      mode: "enforce",
      pruneAfter: "30d",
      maxEntries: 500,
    },
  },
}
```

Предварительный просмотр: `openclaw sessions cleanup --dry-run`.

## Просмотр сеансов

- `openclaw status` — путь к хранилищу сеансов и недавняя активность.
- `openclaw sessions --json` — все сеансы (фильтрация с помощью `--active <minutes>`).
- `/status` в чате — использование контекста, модель и переключатели.
- `/context list` — содержимое системного промпта.
