# Миграция OpenClaw на новую машину

Это руководство описывает перенос шлюза OpenClaw на новую машину без повторного прохождения процедуры onboarding.

## Что переносится

При копировании **каталога состояния** (`~/.openclaw/` по умолчанию) и вашего **рабочего пространства** сохраняются:

- **Конфигурация** — `openclaw.json` и все настройки шлюза
- **Аутентификация** — профили аутентификации для каждого агента `auth-profiles.json` (API-ключи + OAuth), а также состояние каналов/провайдеров в каталоге `credentials/`
- **Сессии** — история диалогов и состояние агента
- **Состояние каналов** — вход в WhatsApp, сессия Telegram и т. д.
- **Файлы рабочего пространства** — `MEMORY.md`, `USER.md`, навыки и промты

<Tip>
Выполните команду `openclaw status` на старой машине, чтобы подтвердить путь к каталогу состояния.
Для пользовательских профилей используется `~/.openclaw-<profile>/` или путь, заданный через `OPENCLAW_STATE_DIR`.
</Tip>

## Шаги по миграции

<Steps>
  <Step title="Остановите шлюз и создайте резервную копию">
    На **старой** машине остановите шлюз, чтобы файлы не изменялись во время копирования, затем создайте архив:

    ```bash
    openclaw gateway stop
    cd ~
    tar -czf openclaw-state.tgz .openclaw
    ```

    Если вы используете несколько профилей (например, `~/.openclaw-work`), создайте архив для каждого из них отдельно.
  </Step>

  <Step title="Установите OpenClaw на новой машине">
    [Установите](/install) CLI (и Node, если необходимо) на новой машине.
    Допустимо, если в процессе onboarding будет создан новый каталог `~/.openclaw/` — вы перезапишете его на следующем шаге.
  </Step>

  <Step title="Скопируйте каталог состояния и рабочее пространство">
    Передайте архив с помощью `scp`, `rsync -a` или внешнего накопителя, затем распакуйте:

    ```bash
    cd ~
    tar -xzf openclaw-state.tgz
    ```

    Убедитесь, что скрытые каталоги были включены, а права владения файлами соответствуют пользователю, который будет запускать шлюз.
  </Step>

  <Step title="Запустите doctor и проверьте">
    На новой машине запустите [Doctor](/gateway/doctor), чтобы применить миграции конфигурации и восстановить сервисы:

    ```bash
    openclaw doctor
    openclaw gateway restart
    openclaw status
    ```
  </Step>
</Steps>

## Распространённые ошибки

<AccordionGroup>
  <Accordion title="Несоответствие профиля или каталога состояния">
    Если старый шлюз использовал параметр `--profile` или переменную `OPENCLAW_STATE_DIR`, а новый — нет,
    каналы будут отображаться как вышедшие из системы, а сессии окажутся пустыми.
    Запустите шлюз с **тем же** профилем или каталогом состояния, который вы перенесли, затем повторно выполните `openclaw doctor`.
  </Accordion>

  <Accordion title="Копирование только openclaw.json">
    Одного файла конфигурации недостаточно. Профили аутентификации модели находятся в каталоге
    `agents/<agentId>/agent/auth-profiles.json`, а состояние каналов/провайдеров — в каталоге `credentials/`.
    Всегда переносите **весь** каталог состояния.
  </Accordion>

  <Accordion title="Права доступа и владение">
    Если вы скопировали файлы от имени пользователя root или сменили пользователя, шлюз может не суметь прочитать учётные данные.
    Убедитесь, что каталог состояния и рабочее пространство принадлежат пользователю, запускающему шлюз.
  </Accordion>

  <Accordion title="Удаленный режим">
    Если ваш интерфейс указывает на **удалённый** шлюз, сессии и рабочее пространство принадлежат удалённому хосту.
    Переносите сам хост шлюза, а не свой локальный ноутбук. См. [FAQ](/help/faq#where-things-live-on-disk).
  </Accordion>

  <Accordion title="Секретные данные в резервных копиях">
    Каталог состояния содержит профили аутентификации, учётные данные каналов и другое состояние провайдера.
    Храните резервные копии в зашифрованном виде, избегайте небезопасных каналов передачи и меняйте ключи, если есть подозрение на их компрометацию.
  </Accordion>
</AccordionGroup>

## Контрольный список проверки

На новой машине убедитесь, что:

- [ ] `openclaw status` показывает, что шлюз запущен
- [ ] Каналы по-прежнему подключены (повторное сопряжение не требуется)
- [ ] Панель управления открывается и отображает существующие сессии
- [ ] Файлы рабочего пространства (память, конфигурации) присутствуют
