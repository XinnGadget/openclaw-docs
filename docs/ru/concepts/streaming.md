# Потоковая передача + разбиение на чанки

В OpenClaw есть два отдельных уровня потоковой передачи:

- **Потоковая передача блоками (каналы):** отправка завершённых **блоков** по мере того, как их формирует ассистент. Это обычные сообщения канала (не дельты токенов).
- **Потоковая передача предварительного просмотра (Telegram/Discord/Slack):** обновление временного **сообщения предварительного просмотра** в процессе генерации.

На данный момент **настоящей потоковой передачи дельт токенов** в сообщения канала не существует. Потоковая передача предварительного просмотра основана на сообщениях (отправка + редактирование/добавление).

## Потоковая передача блоками (сообщения канала)

При потоковой передаче блоками вывод ассистента отправляется крупными чанками по мере его доступности.

```
Вывод модели
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ chunker отправляет блоки по мере роста буфера
       └─ (blockStreamingBreak=message_end)
            └─ chunker сбрасывает буфер при message_end
                   └─ отправка в канал (ответы блоками)
```

Легенда:

- `text_delta/events`: события потока модели (могут быть редкими для моделей без потоковой передачи).
- `chunker`: `EmbeddedBlockChunker`, применяющий минимальные/максимальные границы + предпочтение разбиения.
- `channel send`: фактические исходящие сообщения (ответы блоками).

**Элементы управления:**

- `agents.defaults.blockStreamingDefault`: `"on"`/`"off"` (по умолчанию — `"off"`).
- Переопределения для каналов: `*.blockStreaming` (и варианты для отдельных аккаунтов) для принудительного включения `"on"`/`"off"` для конкретного канала.
- `agents.defaults.blockStreamingBreak`: `"text_end"` или `"message_end"`.
- `agents.defaults.blockStreamingChunk`: `{ minChars, maxChars, breakPreference? }`.
- `agents.defaults.blockStreamingCoalesce`: `{ minChars?, maxChars?, idleMs? }` (объединение потоковых блоков перед отправкой).
- Жёсткий лимит для канала: `*.textChunkLimit` (например, `channels.whatsapp.textChunkLimit`).
- Режим разбиения на чанки для канала: `*.chunkMode` (`length` — по умолчанию, `newline` — разбиение по пустым строкам (границы абзацев) перед разбиением по длине).
- Мягкий лимит для Discord: `channels.discord.maxLinesPerMessage` (по умолчанию — 17) — разбиение длинных ответов, чтобы избежать обрезки интерфейса.

**Семантика границ:**

- `text_end`: отправка блоков по мере их формирования chunker; сброс при каждом `text_end`.
- `message_end`: ожидание завершения сообщения ассистента, затем сброс буферизованного вывода.

При `message_end` всё ещё используется chunker, если буферизованный текст превышает `maxChars`, поэтому в конце может быть отправлено несколько чанков.

## Алгоритм разбиения на чанки (нижняя/верхняя границы)

Разбиение на блоки реализовано с помощью `EmbeddedBlockChunker`:

- **Нижняя граница:** не отправлять, пока буфер >= `minChars` (если не принудительно).
- **Верхняя граница:** предпочтительно разбиение до `maxChars`; если принудительно — разбиение на `maxChars`.
- **Предпочтение разбиения:** `paragraph` → `newline` → `sentence` → `whitespace` → жёсткое разбиение.
- **Блоки кода:** никогда не разбивать внутри блоков; при принудительном разбиении на `maxChars` закрыть и заново открыть блок, чтобы сохранить корректность Markdown.

Значение `maxChars` ограничивается `textChunkLimit` канала, поэтому превысить лимиты для конкретного канала невозможно.

## Объединение (слияние потоковых блоков)

Когда потоковая передача блоками включена, OpenClaw может **объединять последовательные блоки-чанки** перед их отправкой. Это позволяет избежать "спама однострочными сообщениями", сохраняя при этом постепенный вывод.

- Объединение ожидает **промежутков бездействия** (`idleMs`) перед сбросом буфера.
- Буферы ограничены `maxChars` и сбрасываются, если их размер превышен.
- `minChars` предотвращает отправку крошечных фрагментов до накопления достаточного объёма текста (финальный сброс всегда отправляет оставшийся текст).
- Соединитель определяется на основе `blockStreamingChunk.breakPreference` (`paragraph` → `\n\n`, `newline` → `\n`, `sentence` → пробел).
- Переопределения для каналов доступны через `*.blockStreamingCoalesce` (включая конфигурации для отдельных аккаунтов).
- Значение `minChars` по умолчанию для объединения увеличено до 1500 для Signal/Slack/Discord, если не переопределено.

## Человекоподобная пауза между блоками

Когда потоковая передача блоками включена, можно добавить **рандомизированную паузу** между ответами блоками (после первого блока). Это делает ответы с несколькими "пузырями" более естественными.

- Конфигурация: `agents.defaults.humanDelay` (переопределение для отдельного агента через `agents.list[].humanDelay`).
- Режимы: `off` (по умолчанию), `natural` (800–2500 мс), `custom` (`minMs`/`maxMs`).
- Применяется только к **ответам блоками**, не к финальным ответам или сводкам инструментов.

## "Передавать чанки или всё сразу"

Это соответствует:

- **Передача чанков:** `blockStreamingDefault: "on"` + `blockStreamingBreak: "text_end"` (отправка по мере формирования). Для каналов, кроме Telegram, также требуется `*.blockStreaming: true`.
- **Передача всего в конце:** `blockStreamingBreak: "message_end"` (сброс один раз, возможно, несколькими чанками, если сообщение очень длинное).
- **Отсутствие потоковой передачи блоками:** `blockStreamingDefault: "off"` (только финальный ответ).

**Примечание для каналов:** потоковая передача блоками **отключена**, если `*.blockStreaming` явно не установлено в `true`. Каналы могут передавать предварительный просмотр в реальном времени (`channels.<channel>.streaming`) без ответов блоками.

Напоминание о расположении конфигурации: значения по умолчанию `blockStreaming*` находятся в `agents.defaults`, а не в корневой конфигурации.

## Режимы потоковой передачи предварительного просмотра

Ключевой параметр: `channels.<channel>.streaming`

Режимы:

- `off`: отключение потоковой передачи предварительного просмотра.
- `partial`: единый предварительный просмотр, который заменяется на последний текст.
- `block`: обновление предварительного просмотра шагами с разбиением на чанки/добавлением.
- `progress`: предварительный просмотр прогресса/статуса во время генерации, финальный ответ по завершении.

### Сопоставление каналов

| Канал | `off` | `partial` | `block` | `progress` |
| -------- | ----- | --------- | ------- | ----------------- |
| Telegram | ✅ | ✅ | ✅ | сопоставляется с `partial` |
| Discord | ✅ | ✅ | ✅ | сопоставляется с `partial` |
| Slack | ✅ | ✅ | ✅ | ✅ |

Только для Slack:

- `channels.slack.streaming.nativeTransport` включает вызовы API потоковой передачи Slack при `channels.slack.streaming.mode="partial"` (по умолчанию — `true`).
- Потоковая передача Slack и статус потока ассистента требуют цели потока ответов; в прямых сообщениях верхнего уровня предварительный просмотр в стиле потока не отображается.

Миграция устаревших ключей:

- Telegram: `streamMode` + булево `streaming` автоматически мигрируют в перечисление `streaming`.
- Discord: `streamMode` + булево `streaming` автоматически мигрируют в перечисление `streaming`.
- Slack: `streamMode` автоматически мигрирует в `streaming.mode`; булево `streaming` автоматически мигрирует в `streaming.mode` плюс `streaming.nativeTransport`; устаревший `nativeStreaming` автоматически мигрирует в `streaming.nativeTransport`.

### Поведение во время выполнения

Telegram:

- Использует `sendMessage` + `editMessageText` для обновления предварительного просмотра в прямых сообщениях и группах/темах.
- Потоковая передача предварительного просмотра пропускается, когда явно включена потоковая передача блоками в Telegram (чтобы избежать двойной потоковой передачи).
- `/reasoning stream` может записывать рассуждения в предварительный просмотр.

Discord:

- Использует отправку + редактирование сообщений предварительного просмотра.
- Режим `block` использует разбиение на черновые чанки (`draftChunk`).
- Потоковая передача предварительного просмотра пропускается, когда явно включена потоковая передача блоками в Discord.
