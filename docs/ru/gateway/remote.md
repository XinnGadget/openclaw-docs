# Удалённый доступ (SSH, туннели и tailnets)

Этот репозиторий поддерживает "удалённый доступ через SSH", сохраняя единственный шлюз (master) запущенным на выделенном хосте (настольный компьютер/сервер) и подключая к нему клиенты.

- Для **операторов (вас / приложения для macOS)**: туннелирование через SSH — универсальный запасной вариант.
- Для **узлов (iOS/Android и будущих устройств)**: подключение к шлюзу через **WebSocket** (LAN/tailnet или SSH-туннель — по необходимости).

## Основная идея

- WebSocket шлюза привязывается к **loopback** на настроенном порту (по умолчанию — 18789).
- Для удалённого использования вы перенаправляете этот порт loopback через SSH (или используете tailnet/VPN и обходитесь без туннелирования).

## Распространённые настройки VPN/tailnet (где находится агент)

Рассматривайте **хост шлюза** как "место, где находится агент". Он управляет сессиями, профилями аутентификации, каналами и состоянием.
Ваш ноутбук/настольный компьютер (и узлы) подключаются к этому хосту.

### 1) Постоянно работающий шлюз в вашем tailnet (VPS или домашний сервер)

Запустите шлюз на постоянном хосте и подключайтесь к нему через **Tailscale** или SSH.

- **Оптимальный пользовательский опыт:** оставьте `gateway.bind: "loopback"` и используйте **Tailscale Serve** для интерфейса управления (Control UI).
- **Запасной вариант:** оставьте loopback + SSH-туннель с любого компьютера, которому нужен доступ.
- **Примеры:** [exe.dev](/install/exe-dev) (простая виртуальная машина) или [Hetzner](/install/hetzner) (производственный VPS).

Это идеальный вариант, если ваш ноутбук часто переходит в спящий режим, но вы хотите, чтобы агент работал постоянно.

### 2) Домашний настольный компьютер запускает шлюз, ноутбук — удалённое управление

Ноутбук **не** запускает агент. Он подключается удалённо:

- Используйте режим **Remote over SSH** в приложении для macOS (Настройки → Общие → "OpenClaw runs").
- Приложение открывает и управляет туннелем, поэтому WebChat + проверки работоспособности работают "из коробки".

Инструкция: [удалённый доступ для macOS](/platforms/mac/remote).

### 3) Ноутбук запускает шлюз, удалённый доступ с других машин

Оставьте шлюз локальным, но обеспечьте безопасный доступ к нему:

- SSH-туннель к ноутбуку с других машин либо
- Tailscale Serve для интерфейса управления (Control UI) и оставьте шлюз только для loopback.

Руководство: [Tailscale](/gateway/tailscale) и [обзор веб-интерфейса](/web).

## Поток команд (что и где выполняется)

Один сервис шлюза управляет состоянием и каналами. Узлы — периферийные устройства.

Пример потока (Telegram → узел):

- Сообщение из Telegram поступает на **шлюз**.
- Шлюз запускает **агент** и решает, нужно ли вызвать инструмент узла.
- Шлюз обращается к **узлу** через WebSocket шлюза (`node.*` RPC).
- Узел возвращает результат; шлюз отправляет ответ обратно в Telegram.

Примечания:

- **Узлы не запускают сервис шлюза.** На одном хосте должен работать только один шлюз, если вы намеренно не запускаете изолированные профили (см. [Несколько шлюзов](/gateway/multiple-gateways)).
- Режим "node mode" в приложении для macOS — это просто клиент узла через WebSocket шлюза.

## SSH-туннель (CLI + инструменты)

Создайте локальный туннель к удалённому Gateway WS:

```bash
ssh -N -L 18789:127.0.0.1:18789 user@host
```

При активном туннеле:

- Команды `openclaw health` и `openclaw status --deep` теперь обращаются к удалённому шлюзу через `ws://127.0.0.1:18789`.
- Команды `openclaw gateway status`, `openclaw gateway health`, `openclaw gateway probe` и `openclaw gateway call` также могут обращаться к перенаправленному URL через параметр `--url`, если это необходимо.

Примечание: замените `18789` на ваш настроенные `gateway.port` (или `--port`/`OPENCLAW_GATEWAY_PORT`).
Примечание: при передаче параметра `--url` CLI не обращается к настройкам или учётным данным из окружения.
Укажите параметры `--token` или `--password` явно. Отсутствие явных учётных данных приведёт к ошибке.

## Настройки удалённого доступа по умолчанию для CLI

Вы можете сохранить удалённый целевой объект, чтобы команды CLI использовали его по умолчанию:

```json5
{
  gateway: {
    mode: "remote",
    remote: {
      url: "ws://127.0.0.1:18789",
      token: "your-token",
    },
  },
}
```

Если шлюз работает только в режиме loopback, оставьте URL как `ws://127.0.0.1:18789` и сначала откройте SSH-туннель.

## Приоритет учётных данных

Разрешение учётных данных шлюза следует единому контракту для путей вызова/проверки/статуса и мониторинга одобрения выполнения в Discord. Узел-хост использует тот же базовый контракт с одним исключением для локального режима (он намеренно игнорирует `gateway.remote.*`):

- Явные учётные данные (`--token`, `--password` или `gatewayToken` в инструменте) всегда имеют приоритет на путях вызова, которые принимают явную аутентификацию.
- Безопасность при переопределении URL:
  - Переопределения URL в CLI (`--url`) никогда не используют неявные настройки/учётные данные из окружения.
  - Переопределения URL в окружении (`OPENCLAW_GATEWAY_URL`) могут использовать только учётные данные из окружения (`OPENCLAW_GATEWAY_TOKEN` / `OPENCLAW_GATEWAY_PASSWORD`).
- Настройки по умолчанию для локального режима:
  - токен: `OPENCLAW_GATEWAY_TOKEN` -> `gateway.auth.token` -> `gateway.remote.token` (запасной вариант для удалённого доступа применяется только тогда, когда локальный ввод токена аутентификации не задан);
  - пароль: `OPENCLAW_GATEWAY_PASSWORD` -> `gateway.auth.password` -> `gateway.remote.password` (запасной вариант для удалённого доступа применяется только тогда, когда локальный ввод пароля аутентификации не задан).
- Настройки по умолчанию для удалённого режима:
  - токен: `gateway.remote.token` -> `OPENCLAW_GATEWAY_TOKEN` -> `gateway.auth.token`;
  - пароль: `OPENCLAW_GATEWAY_PASSWORD` -> `gateway.remote.password` -> `gateway.auth.password`.
- Исключение для локального режима узла-хоста: `gateway.remote.token` / `gateway.remote.password` игнорируются.
- Проверки токена для удалённой проверки/статуса по умолчанию строги: они используют только `gateway.remote.token` (без запасного варианта для локального токена) при работе в удалённом режиме.
- Переопределения окружения для шлюза используют только `OPENCLAW_GATEWAY_*`.

## Чат-интерфейс через SSH

WebChat больше не использует отдельный HTTP-порт. Интерфейс чата SwiftUI подключается напрямую к WebSocket шлюза.

- Перенаправьте порт `18789` через SSH (см. выше), затем подключите клиенты к `ws://127.0.0.1:18789`.
- В macOS предпочтительнее использовать режим "Remote over SSH" в приложении, который автоматически управляет туннелем.

## Правила безопасности (удалённый доступ/VPN)

Краткая версия: **держите шлюз только в режиме loopback**, если вы не уверены, что вам нужна привязка.

- **Loopback + SSH/Tailscale Serve** — самый безопасный вариант по умолчанию (без публичного доступа).
- Открытый текст `ws://` по умолчанию работает только в режиме loopback. Для доверенных частных сетей установите `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` в клиентском процессе в качестве экстренного решения.
- Привязки **не в режиме loopback** (`lan`/`tailnet`/`custom` или `auto`, когда loopback недоступен) должны использовать аутентификацию шлюза: токен, пароль или обратный прокси с учётом идентификационных данных и `gateway.auth.mode: "trusted-proxy"`.
