# Тестирование

В OpenClaw есть три набора тестов Vitest (юнит-/интеграционные, e2e, live) и небольшой набор раннеров Docker.

Этот документ — руководство по тому, как мы проводим тестирование:

- Что покрывает каждый набор тестов (и что намеренно не покрывает)
- Какие команды запускать для типовых рабочих процессов (локально, перед отправкой кода, отладка)
- Как live-тесты находят учётные данные и выбирают модели/провайдеров
- Как добавлять регрессионные тесты для реальных проблем с моделями/провайдерами

## Быстрый старт

В большинстве случаев:

- Полный проход (требуется перед отправкой кода): `pnpm build && pnpm check && pnpm test`
- Более быстрый запуск полного набора тестов локально на мощном компьютере: `pnpm test:max`
- Прямой цикл наблюдения Vitest: `pnpm test:watch`
- Прямой запуск тестов для конкретного файла теперь также поддерживает пути расширений/каналов: `pnpm test extensions/discord/src/monitor/message-handler.preflight.test.ts`
- Сайт QA на базе Docker: `pnpm qa:lab:up`

Когда вы вносите изменения в тесты или хотите дополнительно убедиться в корректности:

- Проверка покрытия: `pnpm test:coverage`
- Набор e2e-тестов: `pnpm test:e2e`

При отладке реальных провайдеров/моделей (требуются реальные учётные данные):

- Набор live-тестов (модели + проверка инструментов/образов шлюза): `pnpm test:live`
- Запуск одного live-теста в тихом режиме: `pnpm test:live -- src/agents/models.profiles.live.test.ts`

Совет: если вам нужен только один неудачный кейс, лучше сузить область live-тестов с помощью переменных окружения allowlist, описанных ниже.

## Наборы тестов (что и где выполняется)

Представьте наборы тестов как "постепенное увеличение реалистичности" (и возрастание нестабильности/стоимости):

### Юнит-/интеграционные тесты (по умолчанию)

- Команда: `pnpm test`
- Конфигурация: десять последовательных запусков шардов (`vitest.full-*.config.ts`) для существующих ограниченных проектов Vitest
- Файлы: инвентаризация юнит-тестов в `src/**/*.test.ts`, `packages/**/*.test.ts`, `test/**/*.test.ts`, а также разрешённые тесты узла `ui`, покрываемые `vitest.unit.config.ts`
- Область применения:
  - Чисто юнит-тесты
  - Интеграционные тесты в рамках процесса (аутентификация шлюза, маршрутизация, инструменты, парсинг, конфигурация)
  - Детерминированные регрессионные тесты для известных ошибок
- Ожидания:
  - Запуск в CI
  - Не требуются реальные ключи
  - Должны выполняться быстро и стабильно
- Примечания по проектам:
  - Нецелевой `pnpm test` теперь запускает одиннадцать меньших конфигураций шардов (`core-unit-src`, `core-unit-security`, `core-unit-ui`, `core-unit-support`, `core-support-boundary`, `core-contracts`, `core-bundled`, `core-runtime`, `agentic`, `auto-reply`, `extensions`) вместо одного большого процесса корневого проекта. Это снижает пиковое потребление оперативной памяти на загруженных машинах и предотвращает "голодание" несвязанных наборов тестов из-за `auto-reply`/`extensions`.
  - `pnpm test --watch` по-прежнему использует собственный граф проектов корневого `vitest.config.ts`, поскольку многошардный цикл наблюдения непрактичен.
  - `pnpm test`, `pnpm test:watch` и `pnpm test:perf:imports` сначала направляют явные цели файла/каталога через ограниченные каналы, поэтому `pnpm test extensions/discord/src/monitor/message-handler.preflight.test.ts` позволяет избежать затрат на полный запуск корневого проекта.
  - `pnpm test:changed` расширяет изменённые пути git до тех же ограниченных каналов, если разница затрагивает только маршрутизируемые исходные/тестовые файлы; изменения конфигурации/настройки по-прежнему приводят к перезапуску всего корневого проекта.
  - Некоторые тесты `plugin-sdk` и `commands` также направляются через выделенные лёгкие каналы, которые пропускают `test/setup-openclaw-runtime.ts`; файлы с состоянием/тяжёлой средой выполнения остаются в существующих каналах.
  - Некоторые вспомогательные исходные файлы `plugin-sdk` и `commands` также сопоставляют изменения в режиме с явными сопутствующими тестами в этих лёгких каналах, поэтому изменения вспомогательных файлов не приводят к перезапуску полного тяжёлого набора тестов для этого каталога.
  - Для `auto-reply` теперь есть три выделенных пула: вспомогательные функции верхнего уровня, интеграционные тесты верхнего уровня `reply.*` и поддерево `src/auto-reply/reply/**`. Это позволяет исключить самые тяжёлые тесты для ответов из дешёвых тестов статуса/фрагментов/токенов.
- Примечание о встроенном раннере:
  - При изменении входных данных для обнаружения инструментов сообщений или контекста выполнения сжатия сохраняйте оба уровня покрытия.
  - Добавляйте сфокусированные регрессионные тесты для чистых границ маршрутизации/нормализации.
  - Также следите за работоспособностью наборов интеграционных тестов встроенного раннера: `src/agents/pi-embedded-runner/compact.hooks.test.ts`, `src/agents/pi-embedded-runner/run.overflow-compaction.test.ts` и `src/agents/pi-embedded-runner/run.overflow-compaction.loop.test.ts`.
  - Эти наборы проверяют, что ограниченные идентификаторы и поведение сжатия по-прежнему проходят через реальные пути `run.ts` / `compact.ts`; тесты только для вспомогательных функций не заменяют эти интеграционные пути.
- Примечание о пуле:
  - Базовая конфигурация Vitest теперь по умолчанию использует `threads`.
  - Общая конфигурация Vitest также фиксирует `isolate: false` и использует неизолированный раннер для корневых проектов, e2e и live-конфигураций.
  - Корневой канал UI сохраняет свою настройку `jsdom` и оптимизатор, но теперь также работает на общем неизолированном раннере.
  - Каждый шард `pnpm test` наследует те же значения по умолчанию `threads` + `isolate: false` из общей конфигурации Vitest.
  - Общий лаунчер `scripts/run-vitest.mjs` теперь также добавляет `--no-maglev` для дочерних процессов Node Vitest по умолчанию, чтобы снизить нагрузку на компиляцию V8 во время больших локальных запусков. Установите `OPENCLAW_VITEST_ENABLE_MAGLEV=1`, если вам нужно сравнить с поведением стандартной V8.
- Примечание о быстрой локальной итерации:
  - `pnpm test:changed` направляется через ограниченные каналы, когда изменённые пути чётко соответствуют меньшему набору тестов.
  - `pnpm test:max` и `pnpm test:changed:max` сохраняют то же поведение маршрутизации, но с более высоким лимитом рабочих процессов.
  - Автомасштабирование локальных рабочих процессов теперь намеренно консервативно и также снижается, когда средняя нагрузка на хост уже высока, поэтому несколько одновременных запусков Vitest по умолчанию наносят меньше ущерба.
  - Базовая конфигурация Vitest помечает файлы проектов/конфигураций как `forceRerunTriggers`, поэтому перезапуски в изменённом режиме остаются корректными при изменении тестовой проводки.
  - Конфигурация сохраняет `OPENCLAW_VITEST_FS_MODULE_CACHE` включённым на поддерживаемых хостах; установите `OPENCLAW_VITEST_FS_MODULE_CACHE_PATH=/abs/path`, если вам нужно одно явное местоположение кэша для прямого профилирования.
- Примечание о профилировании производительности:
  - `pnpm test:perf:imports` включает отчётность о длительности импорта Vitest, а также вывод разбивки по импорту.
  - `pnpm test:perf:imports:changed` ограничивает представление профилирования файлами, изменёнными с момента `origin/main`.
- `pnpm test:perf:changed:bench -- --ref <git-ref>` сравнивает маршрутизируемый `test:changed` с нативным корневым путём проекта для этого зафиксированного изменения и выводит время выполнения и максимальное потребление оперативной памяти на macOS.
- `pnpm test:perf:changed:bench -- --worktree` проводит бенчмаркинг текущего "грязного" дерева, направляя список изменённых файлов через `scripts/test-projects.mjs` и корневую конфигурацию Vitest.
- `pnpm test:perf:profile:main` записывает профиль ЦП основного потока для запуска и накладных расходов на преобразование Vitest/Vite.
- `pnpm test:perf:profile:runner` записывает профили ЦП и памяти раннера для набора юнит-тестов с отключённым параллелизмом файлов.

### E2E (проверка шлюза)
