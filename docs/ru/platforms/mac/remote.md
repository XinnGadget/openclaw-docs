# Удалённый OpenClaw (macOS ⇄ удалённый хост)

Этот процесс позволяет приложению на macOS выступать в качестве полноценного пульта дистанционного управления для шлюза OpenClaw, работающего на другом хосте (настольном компьютере/сервере). Это функция приложения **Remote over SSH** (удалённый запуск). Все функции — проверки работоспособности, переадресация Voice Wake и Web Chat — используют одну и ту же удалённую конфигурацию SSH из раздела _Настройки → Общие_.

## Режимы

- **Локальный (этот Mac)**: всё выполняется на ноутбуке. SSH не используется.
- **Удалённый через SSH (по умолчанию)**: команды OpenClaw выполняются на удалённом хосте. Приложение на Mac устанавливает SSH‑соединение с параметром `-o BatchMode`, используя выбранный идентификатор/ключ и локальную переадресацию портов.
- **Удалённый прямой (ws/wss)**: туннель SSH не используется. Приложение на Mac подключается к URL шлюза напрямую (например, через Tailscale Serve или публичный HTTPS‑обратный прокси).

## Удалённые транспортные механизмы

Удалённый режим поддерживает два транспортных механизма:

- **Туннель SSH** (по умолчанию): используется команда `ssh -N -L ...` для переадресации порта шлюза на localhost. Шлюз будет видеть IP узла как `127.0.0.1`, поскольку туннель является loopback.
- **Прямой (ws/wss)**: подключение напрямую к URL шлюза. Шлюз видит реальный IP клиента.

## Предварительные требования на удалённом хосте

1. Установите Node + pnpm и соберите/установите OpenClaw CLI (`pnpm install && pnpm build && pnpm link --global`).
2. Убедитесь, что `openclaw` находится в PATH для неинтерактивных оболочек (при необходимости создайте символическую ссылку в `/usr/local/bin` или `/opt/homebrew/bin`).
3. Настройте SSH с аутентификацией по ключу. Для стабильного доступа вне локальной сети рекомендуется использовать IP‑адреса **Tailscale**.

## Настройка приложения на macOS

1. Откройте _Настройки → Общие_.
2. В разделе **OpenClaw runs** выберите **Remote over SSH** и задайте:
   - **Транспорт**: **Туннель SSH** или **Прямой (ws/wss)**.
   - **Цель SSH**: `user@host` (опционально `:port`).
     - Если шлюз находится в той же локальной сети и анонсирует Bonjour, выберите его из обнаруженного списка, чтобы автоматически заполнить это поле.
   - **URL шлюза** (только для прямого режима): `wss://gateway.example.ts.net` (или `ws://...` для локальной сети).
   - **Файл идентификатора** (расширенные настройки): путь к вашему ключу.
   - **Корень проекта** (расширенные настройки): удалённый путь к рабочей копии, используемый для команд.
   - **Путь к CLI** (расширенные настройки): необязательный путь к исполняемому файлу/бинарнику `openclaw` (заполняется автоматически, если анонсирован).
3. Нажмите **Test remote**. Успех означает, что удалённая команда `openclaw status --json` выполняется корректно. Ошибки обычно связаны с проблемами PATH/CLI; код выхода 127 означает, что CLI не найден на удалённой машине.
4. Проверки работоспособности и Web Chat теперь будут автоматически выполняться через этот SSH‑туннель.

## Web Chat

- **Туннель SSH**: Web Chat подключается к шлюзу через переадресованный порт управления WebSocket (по умолчанию 18789).
- **Прямой (ws/wss)**: Web Chat подключается напрямую к настроенному URL шлюза.
- Отдельного HTTP‑сервера WebChat больше нет.

## Разрешения

- На удалённом хосте требуются те же разрешения TCC, что и на локальном (Автоматизация, Доступность, Запись экрана, Микрофон, Распознавание речи, Уведомления). Запустите процесс настройки на этой машине, чтобы предоставить их один раз.
- Узлы анонсируют своё состояние разрешений через `node.list` / `node.describe`, чтобы агенты знали, какие возможности доступны.

## Примечания по безопасности

- Предпочитайте привязки loopback на удалённом хосте и подключайтесь через SSH или Tailscale.
- Туннелирование SSH использует строгую проверку ключа хоста; сначала доверьте ключ хоста, чтобы он появился в `~/.ssh/known_hosts`.
- Если вы привязываете шлюз к интерфейсу, не являющемуся loopback, требуйте действительной аутентификации шлюза: токена, пароля или обратного прокси с учётом идентификатора с `gateway.auth.mode: "trusted-proxy"`.
- См. [Безопасность](/gateway/security) и [Tailscale](/gateway/tailscale).

## Процесс входа в WhatsApp (удалённый)

- Выполните `openclaw channels login --verbose` **на удалённом хосте**. Отсканируйте QR‑код с помощью WhatsApp на своём телефоне.
- Повторно выполните вход на этом хосте, если срок действия аутентификации истечёт. Проверка работоспособности выявит проблемы со ссылкой.

## Устранение неполадок

- **exit 127 / не найдено**: `openclaw` не находится в PATH для неинтерактивных оболочек. Добавьте его в `/etc/paths`, в файл конфигурации вашей оболочки или создайте символическую ссылку в `/usr/local/bin`/`/opt/homebrew/bin`.
- **Сбой проверки работоспособности**: проверьте доступность SSH, PATH и то, что Baileys вошёл в систему (`openclaw status --json`).
- **Web Chat зависает**: убедитесь, что шлюз запущен на удалённом хосте и переадресованный порт соответствует порту WS шлюза; пользовательскому интерфейсу требуется работоспособное соединение WS.
- **IP узла отображается как 127.0.0.1**: это ожидаемо при использовании туннеля SSH. Переключите **Транспорт** на **Прямой (ws/wss)**, если хотите, чтобы шлюз видел реальный IP клиента.
- **Voice Wake**: фразы-триггеры переадресовываются автоматически в удалённом режиме; отдельный переадресатор не требуется.

## Звуки уведомлений

Выберите звуки для каждого уведомления с помощью скриптов с `openclaw` и `node.invoke`, например:

```bash
openclaw nodes notify --node <id> --title "Ping" --body "Remote gateway ready" --sound Glass
```

В приложении больше нет глобального переключателя "звука по умолчанию"; вызывающие стороны выбирают звук (или отсутствие звука) для каждого запроса.
