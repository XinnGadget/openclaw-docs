# Doctor

`openclaw doctor` — это инструмент для восстановления и миграции в OpenClaw. Он исправляет устаревшие конфигурации и состояния, проверяет работоспособность и предоставляет практические шаги по восстановлению.

## Быстрый старт

```bash
openclaw doctor
```

### Без графического интерфейса / автоматизация

```bash
openclaw doctor --yes
```

Принимает значения по умолчанию без запросов (включая шаги по перезапуску/восстановлению сервисов/песочницы, если применимо).

```bash
openclaw doctor --repair
```

Применяет рекомендуемые исправления без запросов (исправления + перезапуски там, где это безопасно).

```bash
openclaw doctor --repair --force
```

Применяет также агрессивные исправления (перезаписывает пользовательские конфигурации супервизора).

```bash
openclaw doctor --non-interactive
```

Запускает без запросов и применяет только безопасные миграции (нормализация конфигурации + перемещение состояний на диске). Пропускает действия по перезапуску/сервису/песочнице, требующие подтверждения человека.
Миграции устаревшего состояния выполняются автоматически при обнаружении.

```bash
openclaw doctor --deep
```

Проверяет системные сервисы на наличие дополнительных установок шлюза (launchd/systemd/schtasks).

Если вы хотите просмотреть изменения перед записью, сначала откройте файл конфигурации:

```bash
cat ~/.openclaw/openclaw.json
```

## Что делает инструмент (кратко)

- Опциональное предварительное обновление для установок из git (только в интерактивном режиме).
- Проверка актуальности протокола UI (перестраивает Control UI, если схема протокола новее).
- Проверка работоспособности + запрос на перезапуск.
- Сводка статуса навыков (подходящие/отсутствующие/заблокированные) и статуса плагинов.
- Нормализация конфигурации для устаревших значений.
- Миграция конфигурации Talk из устаревших плоских полей `talk.*` в `talk.provider` + `talk.providers.<provider>`.
- Проверка миграции браузера для устаревших конфигураций расширения Chrome и готовности Chrome MCP.
- Предупреждения о переопределении провайдера OpenCode (`models.providers.opencode` / `models.providers.opencode-go`).
- Предупреждения о затенении OAuth для OpenAI Codex (`models.providers.openai-codex`).
- Проверка предварительных условий TLS для профилей OAuth OpenAI Codex.
- Миграция устаревшего состояния на диске (сессии/каталог агента/аутентификация WhatsApp).
- Миграция устаревших ключей манифеста плагина (`speechProviders`, `realtimeTranscriptionProviders`, `realtimeVoiceProviders`, `mediaUnderstandingProviders`, `imageGenerationProviders`, `videoGenerationProviders`, `webFetchProviders`, `webSearchProviders` → `contracts`).
- Миграция устаревшего хранилища cron (`jobId`, `schedule.cron`, поля доставки/полезной нагрузки верхнего уровня, `provider` в полезной нагрузке, простые задания вебхука с `notify: true`).
- Проверка файлов блокировки сеансов и очистка устаревших блокировок.
- Проверка целостности состояния и разрешений (сеансы, транскрипты, каталог состояния).
- Проверка разрешений файла конфигурации (chmod 600) при локальном запуске.
- Проверка работоспособности аутентификации модели: проверяет истечение срока действия OAuth, может обновить истекающие токены и сообщает о состояниях охлаждения/отключения профиля аутентификации.
- Обнаружение дополнительного каталога рабочей области (`~/openclaw`).
- Восстановление образа песочницы, когда песочница включена.
- Миграция устаревших сервисов и обнаружение дополнительных шлюзов.
- Миграция устаревшего состояния канала Matrix (в режиме `--fix` / `--repair`).
- Проверка времени выполнения шлюза (сервис установлен, но не запущен; кэшированная метка launchd).
- Предупреждения о статусе канала (проверяются из работающего шлюза).
- Аудит конфигурации супервизора (launchd/systemd/schtasks) с возможностью восстановления.
- Проверка лучших практик времени выполнения шлюза (Node против Bun, пути менеджера версий).
- Диагностика конфликтов портов шлюза (по умолчанию `18789`).
- Предупреждения о безопасности для открытых политик DM.
- Проверка аутентификации шлюза для режима локального токена (предлагает генерацию токена, если источник токена отсутствует; не перезаписывает конфигурации SecretRef токена).
- Проверка lingering для systemd в Linux.
- Проверка размера загрузочных файлов рабочей области (предупреждения об усечении/приближении к лимиту для файлов контекста).
- Проверка статуса завершения оболочки и автоматическая установка/обновление.
- Проверка готовности провайдера встраивания поиска в памяти (локальная модель, удалённый API-ключ или бинарный файл QMD).
- Проверка установки из исходного кода (несоответствие рабочей области pnpm, отсутствие ресурсов UI, отсутствие бинарного файла tsx).
- Запись обновлённой конфигурации + метаданных мастера.

## Backfill и сброс в Dreams UI

Сцена Dreams в Control UI включает действия **Backfill**, **Reset** и **Clear Grounded** для рабочего процесса основанного на сновидениях. Эти действия используют методы RPC в стиле doctor шлюза, но они **не** являются частью CLI-инструмента `openclaw doctor` для восстановления/миграции.

Что они делают:

- **Backfill** сканирует исторические файлы `memory/YYYY-MM-DD.md` в активной рабочей области, запускает проход дневника REM с учётом контекста и записывает обратимые записи backfill в `DREAMS.md`.
- **Reset** удаляет только помеченные записи дневника backfill из `DREAMS.md`.
- **Clear Grounded** удаляет только временные записи с учётом контекста, которые поступили из исторического воспроизведения и ещё не накопили живого воспоминания или ежедневной поддержки.

Что они **не** делают сами по себе:

- они не редактируют `MEMORY.md`;
- они не запускают полные миграции doctor;
- они не помещают автоматически кандидатов с учётом контекста в хранилище краткосрочного продвижения, если вы явно не запустите путь CLI с помеченными кандидатами.

Если вы хотите, чтобы историческое воспроизведение с учётом контекста влияло на обычный канал глубокого продвижения, используйте поток CLI:

```bash
openclaw memory rem-backfill --path ./memory --stage-short-term
```

Это помещает долговечных кандидатов с учётом контекста в хранилище краткосрочных сновидений, сохраняя `DREAMS.md` в качестве поверхности для просмотра.

## Подробное описание поведения и обоснования

### 0) Опциональное обновление (установки из git)

Если это проверка из git и doctor работает в интерактивном режиме, он предлагает обновить (получить/перебазировать/собрать) перед запуском doctor.

### 1) Нормализация конфигурации

Если конфигурация содержит устаревшие формы значений (например, `messages.ackReaction` без переопределения для конкретного канала), doctor нормализует их в соответствии с текущей схемой.

Это включает устаревшие плоские поля Talk. Текущая публичная конфигурация Talk — `talk.provider` + `talk.providers.<provider>`. Doctor переписывает старые формы `talk.voiceId` / `talk.voiceAliases` / `talk.modelId` / `talk.outputFormat` / `talk.apiKey` в карту провайдера.

### 2) Миграция устаревших ключей конфигурации

Когда конфигурация содержит устаревшие ключи, другие команды отказываются выполняться и просят запустить `openclaw doctor`.

Doctor выполнит:

- объяснит, какие устаревшие ключи были найдены;
- покажет применённую миграцию;
- перепишет `~/.openclaw/openclaw.json` с обновлённой схемой.

Шлюз также автоматически запускает миграции doctor при запуске, когда обнаруживает устаревший формат конфигурации, поэтому устаревшие конфигурации исправляются без ручного вмешательства. Миграции хранилища заданий cron выполняются с помощью `openclaw doctor --fix`.

Текущие миграции:

- `routing.allowFrom` → `channels.whatsapp.allowFrom`;
- `routing.groupChat.requireMention` → `channels.whatsapp/telegram/imessage.groups."*".requireMention`;
- `routing.groupChat.historyLimit` → `messages.groupChat.historyLimit`;
- `routing.groupChat.mentionPatterns` → `messages.groupChat.mentionPatterns`;
- `routing.queue` → `messages.queue`;
- `routing.bindings` → ключи верхнего уровня `bindings`;
- `routing.agents`/`routing.defaultAgentId` → `agents.list` + `agents.list[].default`;
- устаревшие `talk.voiceId`/`talk.voiceAliases`/`talk.modelId`/`talk.outputFormat`/`talk.apiKey` → `talk.provider` + `talk.providers.<provider>`;
- `routing.agentToAgent` → `tools.agentToAgent`;
- `routing.transcribeAudio` → `tools.media.audio.models`;
- `messages.tts.<provider>` (`openai`/`elevenlabs`/`microsoft`/`edge`) → `messages.tts.providers.<provider>`;
- `channels.discord.voice.tts.<provider>` (`openai`/`elevenlabs`/`microsoft`/`edge`) → `channels.discord.voice.tts.providers.<provider>`;
