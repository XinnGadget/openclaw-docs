# Кэширование промтов

Кэширование промтов означает, что провайдер модели может повторно использовать неизменные префиксы промтов (обычно инструкции системы/разработчика и другой стабильный контекст) в разных итерациях, вместо того чтобы обрабатывать их заново каждый раз. OpenClaw приводит использование провайдера к операциям `cacheRead` и `cacheWrite`, если вышестоящий API напрямую предоставляет эти счётчики.

Панели состояния также могут восстанавливать счётчики кэша из самого свежего журнала использования транскрипции, если в снимке текущей сессии они отсутствуют. Благодаря этому `/status` может продолжать отображать строку кэша после частичной потери метаданных сессии. Существующие ненулевые значения кэша в текущей сессии по-прежнему имеют приоритет над значениями, восстановленными из транскрипции.

Почему это важно: снижение затрат на токены, более быстрые ответы и более предсказуемая производительность для длительно работающих сессий. Без кэширования повторные промты каждый раз полностью оплачиваются, даже если большая часть входных данных не изменилась.

На этой странице описаны все параметры, связанные с кэшем, которые влияют на повторное использование промтов и затраты на токены.

## Основные параметры

### `cacheRetention` (глобальное значение по умолчанию, для модели и для агента)

Задайте сохранение кэша как глобальное значение по умолчанию для всех моделей:

```yaml
agents:
  defaults:
    params:
      cacheRetention: "long" # none | short | long
```

Переопределите для конкретной модели:

```yaml
agents:
  defaults:
    models:
      "anthropic/claude-opus-4-6":
        params:
          cacheRetention: "short" # none | short | long
```

Переопределение для агента:

```yaml
agents:
  list:
    - id: "alerts"
      params:
        cacheRetention: "none"
```

Порядок слияния конфигураций:

1. `agents.defaults.params` (глобальное значение по умолчанию — применяется ко всем моделям)
2. `agents.defaults.models["provider/model"].params` (переопределение для конкретной модели)
3. `agents.list[].params` (соответствующий идентификатор агента; переопределение по ключу)

### `contextPruning.mode: "cache-ttl"`

Удаляет старый контекст результатов инструментов после истечения срока действия кэша (TTL), чтобы запросы после периода бездействия не перезаписывали чрезмерно большой объём истории.

```yaml
agents:
  defaults:
    contextPruning:
      mode: "cache-ttl"
      ttl: "1h"
```

Полный порядок действий см. в разделе [Обрезка сессий](/concepts/session-pruning).

### Heartbeat для поддержания активности

Heartbeat может поддерживать активность окон кэша и сокращать количество повторных записей в кэш после периодов бездействия.

```yaml
agents:
  defaults:
    heartbeat:
      every: "55m"
```

Для агента можно задать собственный heartbeat в `agents.list[].heartbeat`.

## Поведение провайдеров

### Anthropic (прямой API)

- Поддержка `cacheRetention`.
- При использовании профилей аутентификации с API-ключом Anthropic OpenClaw задаёт `cacheRetention: "short"` для ссылок на модели Anthropic, если значение не задано.
- В ответах Anthropic в формате Messages предоставляются значения `cache_read_input_tokens` и `cache_creation_input_tokens`, поэтому OpenClaw может отображать `cacheRead` и `cacheWrite`.
- Для прямых запросов к Anthropic `cacheRetention: "short"` соответствует кэшу с временем жизни 5 минут, а `cacheRetention: "long"` увеличивает время жизни до 1 часа только на хостах `api.anthropic.com`.

### OpenAI (прямой API)

- Кэширование промтов выполняется автоматически в поддерживаемых последних моделях. OpenClaw не нужно вставлять маркеры кэша на уровне блоков.
- OpenClaw использует `prompt_cache_key` для обеспечения стабильного маршрутизации кэша между итерациями и `prompt_cache_retention: "24h"` только при выборе `cacheRetention: "long"` на прямых хостах OpenAI.
- В ответах OpenAI количество токенов промта в кэше предоставляется через `usage.prompt_tokens_details.cached_tokens` (или `input_tokens_details.cached_tokens` в событиях API Responses). OpenClaw сопоставляет это значение с `cacheRead`.
- OpenAI не предоставляет отдельный счётчик токенов для записи в кэш, поэтому `cacheWrite` остаётся равным `0` на путях OpenAI, даже если провайдер обновляет кэш.
- OpenAI возвращает полезные заголовки для трассировки и ограничения скорости, такие как `x-request-id`, `openai-processing-ms` и `x-ratelimit-*`, но учёт попаданий в кэш должен осуществляться на основе данных в полезной нагрузке, а не заголовков.
- На практике OpenAI часто ведёт себя как кэш начального префикса, а не как повторное использование полной истории в стиле Anthropic. Стабильные длинные префиксы текста могут достигать плато в `4864` токена в кэше в текущих тестах, в то время как транскрипции с большим количеством инструментов или в стиле MCP часто достигают плато около `4608` токенов в кэше даже при точных повторениях.

### Anthropic Vertex

- Модели Anthropic на Vertex AI (`anthropic-vertex/*`) поддерживают `cacheRetention` так же, как и прямой API Anthropic.
- `cacheRetention: "long"` соответствует реальному времени жизни кэша промтов в 1 час на конечных точках Vertex AI.
- Значение сохранения кэша по умолчанию для `anthropic-vertex` соответствует значениям по умолчанию для прямого API Anthropic.
- Запросы к Vertex маршрутизируются через механизм формирования кэша с учётом границ, поэтому повторное использование кэша соответствует тому, что фактически получает провайдер.

### Amazon Bedrock

- Ссылки на модели Anthropic Claude (`amazon-bedrock/*anthropic.claude*`) поддерживают явную передачу `cacheRetention`.
- Для моделей Bedrock, не относящихся к Anthropic, `cacheRetention` принудительно устанавливается в `"none"` во время выполнения.

### Модели Anthropic в OpenRouter

Для ссылок на модели `openrouter/anthropic/*` OpenClaw внедряет `cache_control` Anthropic в блоки промтов системы/разработчика, чтобы улучшить повторное использование кэша промтов, только если запрос по-прежнему направлен на проверенный маршрут OpenRouter (`openrouter` на его конечной точке по умолчанию или любой URL провайдера/базы, который разрешается в `openrouter.ai`).

Если вы перенаправите модель на произвольный URL-прокси, совместимый с OpenAI, OpenClaw прекратит внедрение маркеров кэша Anthropic, специфичных для OpenRouter.

### Другие провайдеры

Если провайдер не поддерживает этот режим кэширования, `cacheRetention` не оказывает никакого эффекта.

### Прямой API Google Gemini

- Прямой транспорт Gemini (`api: "google-generative-ai"`) сообщает о попаданиях в кэш через вышестоящий `cachedContentTokenCount`; OpenClaw сопоставляет это значение с `cacheRead`.
- Когда `cacheRetention` задано для прямой модели Gemini, OpenClaw автоматически создаёт, повторно использует и обновляет ресурсы `cachedContents` для системных промтов в запусках Google AI Studio. Это означает, что вам больше не нужно вручную создавать дескриптор кэшированного содержимого.
- Вы по-прежнему можете передать уже существующий дескриптор кэшированного содержимого Gemini через `params.cachedContent` (или устаревший `params.cached_content`) в настроенной модели.
- Это отличается от кэширования префиксов промтов в Anthropic/OpenAI. Для Gemini OpenClaw управляет ресурсом `cachedContents`, специфичным для провайдера, вместо того чтобы внедрять маркеры кэша в запрос.
