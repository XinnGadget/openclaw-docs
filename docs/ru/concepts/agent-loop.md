# Агентский цикл (OpenClaw)

Агентский цикл — это полный "реальный" запуск агента: приём данных → сборка контекста → вывод модели → выполнение инструментов → потоковая передача ответов → сохранение состояния. Это основной путь, который преобразует сообщение в действия и итоговый ответ, сохраняя при этом согласованность состояния сессии.

В OpenClaw цикл представляет собой единичный сериализованный запуск в рамках сессии, который генерирует события жизненного цикла и потоковые события по мере того, как модель размышляет, вызывает инструменты и передаёт выходные данные. В этом документе объясняется, как организован этот полноценный цикл от начала до конца.

## Точки входа

- Gateway RPC: `agent` и `agent.wait`.
- CLI: команда `agent`.

## Как это работает (общий обзор)

1. RPC `agent` проверяет параметры, определяет сессию (sessionKey/sessionId), сохраняет метаданные сессии и немедленно возвращает `{ runId, acceptedAt }`.
2. `agentCommand` запускает агента:
   - определяет модель и значения по умолчанию для режима "размышления"/подробного вывода;
   - загружает снимок навыков;
   - вызывает `runEmbeddedPiAgent` (среда выполнения pi-agent-core);
   - генерирует событие **завершения/ошибки жизненного цикла**, если встроенный цикл не сгенерировал его.
3. `runEmbeddedPiAgent`:
   - сериализует запуски с помощью очередей для каждой сессии и глобальной очереди;
   - определяет модель и профиль аутентификации, создаёт сессию pi;
   - подписывается на события pi и передаёт дельты ассистента/инструментов;
   - контролирует время выполнения — прерывает запуск, если время превышено;
   - возвращает полезные данные и метаданные об использовании.
4. `subscribeEmbeddedPiSession` связывает события pi-agent-core с потоком `agent` в OpenClaw:
   - события инструментов ⇒ `stream: "tool"`;
   - дельты ассистента ⇒ `stream: "assistant"`;
   - события жизненного цикла ⇒ `stream: "lifecycle"` (`phase: "start" | "end" | "error"`).
5. `agent.wait` использует `waitForAgentRun`:
   - ожидает события **завершения/ошибки жизненного цикла** для `runId`;
   - возвращает `{ status: ok|error|timeout, startedAt, endedAt, error? }`.

## Очереди и параллелизм

- Запуски сериализуются по ключу сессии (полоса сессии) и, при необходимости, через глобальную полосу.
- Это предотвращает конфликты инструментов/сессий и обеспечивает согласованность истории сессии.
- Каналы обмена сообщениями могут выбирать режимы очереди (collect/steer/followup), которые используются в этой системе полос.
  См. [Command Queue](/concepts/queue).

## Подготовка сессии и рабочей области

- Рабочая область определяется и создаётся; для изолированных запусков может быть выполнено перенаправление в корневую директорию изолированной рабочей области.
- Навыки загружаются (или повторно используются из снимка) и внедряются в окружение и промпт.
- Файлы начальной загрузки/контекста определяются и внедряются в отчёт о системном промпте.
- Получает блокировку записи сессии; `SessionManager` открывается и подготавливается перед началом потоковой передачи.

## Сборка промпта и системный промпт

- Системный промпт формируется из базового промпта OpenClaw, промпта навыков, контекста начальной загрузки и переопределений для каждого запуска.
- Применяются ограничения, специфичные для модели, и зарезервированные токены для сжатия.
- См. [System prompt](/concepts/system-prompt), чтобы узнать, что видит модель.

## Точки перехвата (где можно вмешаться)

В OpenClaw есть две системы перехвата:

- **Внутренние хуки** (хуки шлюза): скрипты, управляемые событиями, для команд и событий жизненного цикла.
- **Хуки плагинов**: точки расширения внутри жизненного цикла агента/инструмента и конвейера шлюза.

### Внутренние хуки (хуки шлюза)

- **`agent:bootstrap`**: выполняется во время сборки файлов начальной загрузки до окончательного формирования системного промпта.
  Используйте этот хук для добавления/удаления файлов контекста начальной загрузки.
- **Хуки команд**: `/new`, `/reset`, `/stop` и другие события команд (см. документацию по хукам).

См. [Hooks](/automation/hooks) для настройки и примеров.

### Хуки плагинов (жизненный цикл агента и шлюза)

Эти хуки выполняются внутри агентского цикла или конвейера шлюза:

- **`before_model_resolve`**: выполняется до начала сессии (без `messages`), чтобы детерминированно переопределить провайдера/модель до определения модели.
- **`before_prompt_build`**: выполняется после загрузки сессии (с `messages`), чтобы внедрить `prependContext`, `systemPrompt`, `prependSystemContext` или `appendSystemContext` до отправки промпта. Используйте `prependContext` для динамического текста на каждом шаге и поля системного контекста для стабильных указаний, которые должны находиться в пространстве системного промпта.
- **`before_agent_start`**: хук для обеспечения обратной совместимости, который может выполняться на любом из этапов; предпочтительнее использовать явные хуки, указанные выше.
- **`before_agent_reply`**: выполняется после встроенных действий и перед вызовом LLM, позволяя плагину захватить ход и вернуть синтетический ответ или полностью отключить ход.
- **`agent_end`**: анализ окончательного списка сообщений и метаданных выполнения после завершения.
- **`before_compaction` / `after_compaction`**: наблюдение за циклами сжатия или их аннотирование.
- **`before_tool_call` / `after_tool_call`**: перехват параметров/результатов инструментов.
- **`before_install`**: анализ результатов встроенного сканирования и, при необходимости, блокировка установки навыков или плагинов.
- **`tool_result_persist`**: синхронная трансформация результатов инструментов перед их записью в транскрипт сессии.
- **`message_received` / `message_sending` / `message_sent`**: хуки для входящих и исходящих сообщений.
- **`session_start` / `session_end`**: границы жизненного цикла сессии.
- **`gateway_start` / `gateway_stop`**: события жизненного цикла шлюза.

Правила принятия решений для хуков исходящих сообщений/инструментов:

- `before_tool_call`: `{ block: true }` — окончательное действие, останавливающее обработчики с более низким приоритетом.
- `before_tool_call`: `{ block: false }` — действие без эффекта, не отменяющее предыдущую блокировку.
- `before_install`: `{ block: true }` — окончательное действие, останавливающее обработчики с более низким приоритетом.
- `before_install`: `{ block: false }` — действие без эффекта, не отменяющее предыдущую блокировку.
- `message_sending`: `{ cancel: true }` — окончательное действие, останавливающее обработчики с более низким приоритетом.
- `message_sending`: `{ cancel: false }` — действие без эффекта, не отменяющее предыдущий отказ.

См. [Plugin hooks](/plugins/architecture#provider-runtime-hooks) для API хуков и деталей регистрации.

## Потоковая передача и частичные ответы

- Дельты ассистента передаются из pi-agent-core и генерируются как события `assistant`.
- Потоковая передача блоков может выдавать частичные ответы либо при `text_end`, либо при `message_end`.
- Потоковая передача рассуждений может быть реализована как отдельный поток или как ответы блоков.
- См. [Streaming](/concepts/streaming) для информации о разбиении на фрагменты и поведении ответов блоков.

## Выполнение инструментов и инструменты обмена сообщениями

- События начала/обновления/завершения инструмента передаются в потоке `tool`.
- Результаты инструментов проверяются на размер и наличие изображений перед записью/передачей.
- Отправка сообщений через инструменты отслеживается для предотвращения дублирования подтверждений ассистента.

## Формирование ответа и подавление

- Окончательные полезные данные формируются из:
  - текста ассистента (и необязательных рассуждений);
  - встроенных сводок инструментов (при подробном выводе и разрешении);
  - текста ошибки ассистента при ошибке модели.
- Токен молчания `NO_REPLY` / `no_reply` фильтруется из исходящих полезных данных.
- Дубликаты инструментов обмена сообщениями удаляются из окончательного списка полезных данных.
- Если не осталось отображаемых полезных данных и инструмент выдал ошибку, генерируется резервный ответ об ошибке инструмента (если инструмент обмена сообщениями ещё не отправил пользователю видимый ответ).

## Сжатие и повторные попытки

- Автоматическое сжатие генерирует события потока `compaction` и может инициировать повторную попытку.
- При повторной попытке буферы в памяти и сводки инструментов сбрасываются, чтобы избежать дублирования вывода.
- См. [Compaction](/concepts/compaction) для информации о конвейере сжатия.

## Потоки событий (на данный момент)

- `lifecycle`: генерируется `subscribeEmbeddedPiSession` (и как запасной вариант — `agentCommand`);
- `assistant`: потоковые дельты из pi-agent-core;
- `tool
