# Nostr

**Статус:** опциональный встроенный плагин (по умолчанию отключён до настройки).

Nostr — децентрализованный протокол для социальных сетей. Этот канал позволяет OpenClaw получать зашифрованные прямые сообщения (DM) через NIP-04 и отвечать на них.

## Встроенный плагин

В текущих версиях OpenClaw плагин Nostr включён в поставку, поэтому для стандартных сборок отдельная установка не требуется.

### Устаревшие/кастомные установки

- При первичной настройке (`openclaw onboard`) и с помощью команды `openclaw channels add` плагин Nostr по-прежнему доступен из общего каталога каналов.
- Если в вашей сборке отсутствует встроенный плагин Nostr, установите его вручную:

```bash
openclaw plugins install @openclaw/nostr
```

Для локальной проверки (рабочие процессы разработки):

```bash
openclaw plugins install --link <путь-к-локальному-плагину-nostr>
```

После установки или включения плагинов перезапустите Gateway.

### Неинтерактивная настройка

```bash
openclaw channels add --channel nostr --private-key "$NOSTR_PRIVATE_KEY"
openclaw channels add --channel nostr --private-key "$NOSTR_PRIVATE_KEY" --relay-urls "wss://relay.damus.io,wss://relay.primal.net"
```

Используйте параметр `--use-env`, чтобы хранить `NOSTR_PRIVATE_KEY` в окружении, а не сохранять ключ в конфигурации.

## Быстрая настройка

1. Сгенерируйте пару ключей Nostr (если необходимо):

```bash
# Используя nak
nak key generate
```

2. Добавьте в конфигурацию:

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
    },
  },
}
```

3. Экспортируйте ключ:

```bash
export NOSTR_PRIVATE_KEY="nsec1..."
```

4. Перезапустите Gateway.

## Справочная информация по конфигурации

| Ключ | Тип | Значение по умолчанию | Описание |
| --- | --- | --- | --- |
| `privateKey` | строка | обязательно | Приватный ключ в формате `nsec` или шестнадцатеричном |
| `relays` | массив строк | `['wss://relay.damus.io', 'wss://nos.lol']` | URLs ретрансляторов (WebSocket) |
| `dmPolicy` | строка | `pairing` | Политика доступа к DM |
| `allowFrom` | массив строк | `[]` | Разрешённые публичные ключи отправителей |
| `enabled` | булево | `true` | Включение/отключение канала |
| `name` | строка | — | Отображаемое имя |
| `profile` | объект | — | Метаданные профиля NIP-01 |

## Метаданные профиля

Данные профиля публикуются как событие NIP-01 `kind:0`. Вы можете управлять ими через интерфейс управления (Channels → Nostr → Profile) или задать напрямую в конфигурации.

Пример:

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      profile: {
        name: "openclaw",
        displayName: "OpenClaw",
        about: "Персональный ассистент-бот для DM",
        picture: "https://example.com/avatar.png",
        banner: "https://example.com/banner.png",
        website: "https://example.com",
        nip05: "openclaw@example.com",
        lud16: "openclaw@example.com",
      },
    },
  },
}
```

Примечания:
- URLs профиля должны использовать `https://`.
- При импорте из ретрансляторов поля объединяются, а локальные переопределения сохраняются.

## Управление доступом

### Политики DM

- **pairing** (по умолчанию): неизвестные отправители получают код сопряжения.
- **allowlist**: только публичные ключи из `allowFrom` могут отправлять DM.
- **open**: открытые входящие DM (требуется `allowFrom: ["*"]`).
- **disabled**: игнорировать входящие DM.

Примечания по реализации:
- Подписи входящих событий проверяются до применения политики отправителя и расшифровки NIP-04, поэтому поддельные события отклоняются на раннем этапе.
- Ответы на сопряжение отправляются без обработки исходного тела DM.
- Входящие DM ограничены по частоте, а слишком большие полезные нагрузки отбрасываются до расшифровки.

### Пример списка разрешённых

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      dmPolicy: "allowlist",
      allowFrom: ["npub1abc...", "npub1xyz..."],
    },
  },
}
```

## Форматы ключей

Поддерживаемые форматы:
- **Приватный ключ:** `nsec...` или 64-символьный шестнадцатеричный код.
- **Публичные ключи (`allowFrom`):** `npub...` или шестнадцатеричный код.

## Ретрансляторы

Значения по умолчанию: `relay.damus.io` и `nos.lol`.

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      relays: ["wss://relay.damus.io", "wss://relay.primal.net", "wss://nostr.wine"],
    },
  },
}
```

Советы:
- Используйте 2–3 ретранслятора для резервирования.
- Избегайте слишком большого количества ретрансляторов (задержка, дублирование).
- Платные ретрансляторы могут повысить надёжность.
- Для тестирования подойдут локальные ретрансляторы (`ws://localhost:7777`).

## Поддержка протоколов

| NIP | Статус | Описание |
| --- | --- | --- |
| NIP-01 | Поддерживается | Базовый формат событий + метаданные профиля |
| NIP-04 | Поддерживается | Зашифрованные DM (`kind:4`) |
| NIP-17 | Планируется | DM в "подарочной упаковке" |
| NIP-44 | Планируется | Версионированное шифрование |

## Тестирование

### Локальный ретранслятор

```bash
# Запустить strfry
docker run -p 7777:7777 ghcr.io/hoytech/strfry
```

```json5
{
  channels: {
    nostr: {
      privateKey: "${NOSTR_PRIVATE_KEY}",
      relays: ["ws://localhost:7777"],
    },
  },
}
```

### Ручное тестирование

1. Запишите публичный ключ бота (npub) из логов.
2. Откройте клиент Nostr (Damus, Amethyst и т. д.).
3. Отправьте DM на публичный ключ бота.
4. Проверьте ответ.

## Устранение неполадок

### Не получаются сообщения

- Проверьте, что приватный ключ действителен.
- Убедитесь, что URLs ретрансляторов доступны и используют `wss://` (или `ws://` для локальных).
- Проверьте, что `enabled` не имеет значения `false`.
- Ознакомьтесь с логами Gateway на предмет ошибок подключения к ретрансляторам.

### Не отправляются ответы

- Проверьте, что ретранслятор принимает записи.
- Убедитесь в наличии исходящего соединения.
- Следите за лимитами частоты ретранслятора.

### Дублирование ответов

- Это ожидаемо при использовании нескольких ретрансляторов.
- Сообщения дедуплицируются по ID события; только первая доставка запускает ответ.

## Безопасность

- Никогда не коммитите приватные ключи.
- Используйте переменные окружения для ключей.
- Для ботов в продакшене рассмотрите использование `allowlist`.
- Подписи проверяются до применения политики отправителя, а политика отправителя применяется до расшифровки, поэтому поддельные события отклоняются на раннем этапе, а неизвестные отправители не могут заставить выполнять полную криптографическую работу.

## Ограничения (MVP)

- Только прямые сообщения (без групповых чатов).
- Нет вложений мультимедиа.
- Поддерживается только NIP-04 (планируется поддержка NIP-17 с "подарочной упаковкой").
