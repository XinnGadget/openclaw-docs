# Проверки работоспособности (CLI)

Краткое руководство по проверке подключения канала без догадок.

## Быстрые проверки

- `openclaw status` — краткая локальная сводка: доступность/режим шлюза, подсказка об обновлении, возраст аутентификации связанного канала, сессии и недавняя активность.
- `openclaw status --all` — полная локальная диагностика (только для чтения, с цветовой маркировкой, безопасно для вставки при отладке).
- `openclaw status --deep` — запрашивает у работающего шлюза текущую проверку работоспособности (`health` с `probe:true`), включая проверки каналов для каждого аккаунта, если это поддерживается.
- `openclaw health` — запрашивает у работающего шлюза снимок состояния работоспособности (только через WS; без прямых сокетов каналов из CLI).
- `openclaw health --verbose` — принудительно выполняет текущую проверку работоспособности и выводит детали подключения шлюза.
- `openclaw health --json` — вывод снимка состояния работоспособности в машиночитаемом формате.
- Отправьте `/status` как отдельное сообщение в WhatsApp/WebChat, чтобы получить ответ о состоянии без вызова агента.
- Логи: отслеживайте `/tmp/openclaw/openclaw-*.log` и фильтруйте по `web-heartbeat`, `web-reconnect`, `web-auto-reply`, `web-inbound`.

## Глубокая диагностика

- Учётные данные на диске: `ls -l ~/.openclaw/credentials/whatsapp/<accountId>/creds.json` (время модификации должно быть актуальным).
- Хранилище сессий: `ls -l ~/.openclaw/agents/<agentId>/sessions/sessions.json` (путь можно переопределить в конфигурации). Количество и недавние получатели отображаются через `status`.
- Процесс перепривязки: `openclaw channels logout && openclaw channels login --verbose`, если в логах появляются коды состояния 409–515 или `loggedOut` (примечание: процесс входа через QR-код автоматически перезапускается один раз для состояния 515 после сопряжения).

## Конфигурация монитора работоспособности

- `gateway.channelHealthCheckMinutes`: как часто шлюз проверяет работоспособность канала. По умолчанию: `5`. Установите `0`, чтобы отключить перезапуски монитора работоспособности глобально.
- `gateway.channelStaleEventThresholdMinutes`: как долго подключённый канал может оставаться бездействующим, прежде чем монитор работоспособности посчитает его устаревшим и перезапустит. По умолчанию: `30`. Это значение должно быть больше или равно `gateway.channelHealthCheckMinutes`.
- `gateway.channelMaxRestartsPerHour`: ограничение на количество перезапусков монитора работоспособности для канала/аккаунта в течение одного часа. По умолчанию: `10`.
- `channels.<provider>.healthMonitor.enabled`: отключение перезапусков монитора работоспособности для конкретного канала при сохранении глобального мониторинга.
- `channels.<provider>.accounts.<accountId>.healthMonitor.enabled`: переопределение для нескольких аккаунтов, которое имеет приоритет над настройкой на уровне канала.
- Эти переопределения для отдельных каналов применяются к встроенным мониторам каналов, которые поддерживают их на данный момент: Discord, Google Chat, iMessage, Microsoft Teams, Signal, Slack, Telegram и WhatsApp.

## Если что-то не работает

- `logged out` или статус 409–515 → перепривяжите с помощью `openclaw channels logout`, затем `openclaw channels login`.
- Шлюз недоступен → запустите его: `openclaw gateway --port 18789` (используйте `--force`, если порт занят).
- Нет входящих сообщений → убедитесь, что привязанный телефон онлайн и отправитель разрешён (`channels.whatsapp.allowFrom`); для групповых чатов проверьте соответствие списка разрешённых и правил упоминания (`channels.whatsapp.groups`, `agents.list[].groupChat.mentionPatterns`).

## Специальная команда "health"

`openclaw health` запрашивает у работающего шлюза снимок состояния работоспособности (без прямых сокетов каналов из CLI). По умолчанию может возвращать свежий кэшированный снимок состояния шлюза; шлюз затем обновляет этот кэш в фоновом режиме. `openclaw health --verbose` вместо этого принудительно выполняет текущую проверку. Команда сообщает о связанных учётных данных/возрасте аутентификации, когда это доступно, сводках проверок для каждого канала, сводке хранилища сессий и длительности проверки. Она возвращает ненулевой код, если шлюз недоступен или проверка завершилась ошибкой/истекло время ожидания.

Опции:

- `--json`: вывод в машиночитаемом формате JSON
- `--timeout <ms>`: переопределение времени ожидания проверки по умолчанию (10 секунд)
- `--verbose`: принудительное выполнение текущей проверки и вывод деталей подключения шлюза
- `--debug`: псевдоним для `--verbose`

Снимок состояния работоспособности включает: `ok` (булево значение), `ts` (метка времени), `durationMs` (время проверки), статус для каждого канала, доступность агента и сводку хранилища сессий.
