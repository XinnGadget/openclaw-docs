# Блокировка шлюза

## Зачем это нужно

- Гарантировать, что на одном хосте на базовом порту работает только один экземпляр шлюза; дополнительные шлюзы должны использовать изолированные профили и уникальные порты.
- Переживать сбои/SIGKILL без оставления устаревших файлов блокировки.
- Быстро выдавать чёткую ошибку, если управляющий порт уже занят.

## Механизм работы

- Шлюз привязывает слушатель WebSocket (по умолчанию `ws://127.0.0.1:18789`) сразу при запуске, используя эксклюзивный TCP-слушатель.
- Если привязка завершается ошибкой `EADDRINUSE`, при запуске выбрасывается исключение `GatewayLockError("другой экземпляр шлюза уже прослушивает ws://127.0.0.1:<порт>")`.
- ОС автоматически освобождает слушатель при завершении любого процесса, включая сбои и SIGKILL — отдельный файл блокировки или этап очистки не требуются.
- При завершении работы шлюз закрывает сервер WebSocket и базовый HTTP-сервер, чтобы оперативно освободить порт.

## Возможные ошибки

- Если порт занят другим процессом, при запуске выбрасывается `GatewayLockError("другой экземпляр шлюза уже прослушивает ws://127.0.0.1:<порт>")`.
- Другие ошибки привязки выводятся как `GatewayLockError("не удалось привязать сокет шлюза на ws://127.0.0.1:<порт>: …")`.

## Операционные примечания

- Если порт занят _другим_ процессом, ошибка будет той же; освободите порт или выберите другой с помощью команды `openclaw gateway --port <порт>`.
- Приложение для macOS по-прежнему поддерживает собственную облегчённую защиту по PID перед запуском шлюза; блокировка во время выполнения обеспечивается привязкой WebSocket.
