# Сообщения

На этой странице описано, как OpenClaw обрабатывает входящие сообщения, сессии, постановку в очередь, потоковую передачу и видимость рассуждений.

## Поток сообщений (общий обзор)

```
Входящее сообщение
  -> маршрутизация/привязки -> ключ сессии
  -> очередь (если выполняется запуск)
  -> запуск агента (потоковая передача + инструменты)
  -> исходящие ответы (ограничения канала + разбиение на части)
```

Основные параметры находятся в конфигурации:

- `messages.*` — для префиксов, постановки в очередь и поведения групп.
- `agents.defaults.*` — для настроек потоковой передачи блоков и разбиения на части по умолчанию.
- Переопределения для каналов (`channels.whatsapp.*`, `channels.telegram.*` и т. д.) — для ограничений и переключения потоковой передачи.

См. [Конфигурация](/gateway/configuration) для полной схемы.

## Удаление дубликатов входящих сообщений

Каналы могут повторно доставлять одно и то же сообщение после переподключений. OpenClaw хранит краткосрочный кэш, индексированный по каналу/учётной записи/участнику/сессии/идентификатору сообщения, чтобы повторные доставки не запускали новый запуск агента.

## Подавление частых сообщений (debouncing)

Быстрые последовательные сообщения от **одного отправителя** могут быть объединены в один ход агента с помощью параметра `messages.inbound`. Подавление применяется для каждого канала и беседы и использует самое последнее сообщение для организации цепочки ответов/идентификаторов.

Конфигурация (глобальное значение по умолчанию + переопределения для отдельных каналов):

```json5
{
  messages: {
    inbound: {
      debounceMs: 2000,
      byChannel: {
        whatsapp: 5000,
        slack: 1500,
        discord: 1500,
      },
    },
  },
}
```

Примечания:

- Подавление применяется только к **текстовым** сообщениям; медиафайлы и вложения отправляются немедленно.
- Управляющие команды обходят подавление, оставаясь отдельными.

## Сессии и устройства

Сессии управляются шлюзом, а не клиентами.

- Прямые чаты объединяются в основной ключ сессии агента.
- Группы/каналы получают собственные ключи сессии.
- Хранилище сессий и транскрипты находятся на хосте шлюза.

Несколько устройств/каналов могут соответствовать одной сессии, но история не полностью синхронизируется с каждым клиентом. Рекомендация: используйте одно основное устройство для длительных бесед, чтобы избежать расхождения контекста. Пользовательский интерфейс управления (Control UI) и текстовый интерфейс (TUI) всегда показывают транскрипты сессии, поддерживаемые шлюзом, — они являются первоисточником.

Подробности: [Управление сессиями](/concepts/session).

## Тела входящих сообщений и контекст истории

OpenClaw разделяет **тело промпта** и **тело команды**:

- `Body`: текст промпта, отправляемый агенту. Может включать конверты канала и необязательные обёртки истории.
- `CommandBody`: необработанный текст пользователя для разбора директив/команд.
- `RawBody`: устаревший псевдоним для `CommandBody` (сохранён для совместимости).

Когда канал предоставляет историю, используется общая обёртка:

- `[Сообщения чата с момента вашего последнего ответа — для контекста]`
- `[Текущее сообщение — ответьте на него]`

Для **непрямых чатов** (группы/каналы/комнаты) **тело текущего сообщения** дополняется меткой отправителя (в том же стиле, что и записи истории). Это обеспечивает согласованность сообщений в реальном времени и поставленных в очередь/исторических сообщений в промпте агента.

Буферы истории являются **только ожидающими**: они включают сообщения группы, которые **не** запускали выполнение (например, сообщения с упоминанием), и **исключают** сообщения, уже присутствующие в транскрипте сессии.

Удаление директив применяется только к разделу **текущего сообщения**, поэтому история остаётся нетронутой. Каналы, которые оборачивают историю, должны устанавливать `CommandBody` (или `RawBody`) в исходный текст сообщения и сохранять `Body` как объединённый промпт.

Буферы истории настраиваются через `messages.groupChat.historyLimit` (глобальное значение по умолчанию) и переопределения для отдельных каналов, такие как `channels.slack.historyLimit` или `channels.telegram.accounts.<id>.historyLimit` (установите `0`, чтобы отключить).

## Постановка в очередь и последующие действия

Если выполнение уже активно, входящие сообщения могут быть поставлены в очередь, направлены в текущее выполнение или собраны для последующего хода.

- Настройка через `messages.queue` (и `messages.queue.byChannel`).
- Режимы: `interrupt`, `steer`, `followup`, `collect`, а также варианты с бэклогом.

Подробности: [Постановка в очередь](/concepts/queue).

## Потоковая передача, разбиение на части и пакетирование

Потоковая передача блоков отправляет частичные ответы по мере того, как модель генерирует текстовые блоки. Разбиение на части учитывает ограничения текста канала и избегает разделения кода в ограждённых блоках.

Ключевые настройки:

- `agents.defaults.blockStreamingDefault` (`on|off`, по умолчанию `off`)
- `agents.defaults.blockStreamingBreak` (`text_end|message_end`)
- `agents.defaults.blockStreamingChunk` (`minChars|maxChars|breakPreference`)
- `agents.defaults.blockStreamingCoalesce` (пакетирование на основе простоя)
- `agents.defaults.humanDelay` (пауза, имитирующая человеческое поведение, между ответами в блоках)
- Переопределения для каналов: `*.blockStreaming` и `*.blockStreamingCoalesce` (для каналов, кроме Telegram, требуется явное указание `*.blockStreaming: true`)

Подробности: [Потоковая передача + разбиение на части](/concepts/streaming).

## Видимость рассуждений и токены

OpenClaw может показывать или скрывать рассуждения модели:

- `/reasoning on|off|stream` управляет видимостью.
- Содержание рассуждений по-прежнему учитывается в использовании токенов, когда оно генерируется моделью.
- Telegram поддерживает потоковую передачу рассуждений в черновик.

Подробности: [Директивы мышления + рассуждений](/tools/thinking) и [Использование токенов](/reference/token-use).

## Префиксы, организация цепочки ответов и ответы

Форматирование исходящих сообщений централизовано в `messages`:

- `messages.responsePrefix`, `channels.<channel>.responsePrefix` и `channels.<channel>.accounts.<id>.responsePrefix` (каскад исходящих префиксов), а также `channels.whatsapp.messagePrefix` (входящий префикс для WhatsApp)
- Организация цепочки ответов через `replyToMode` и значения по умолчанию для отдельных каналов

Подробности: [Конфигурация](/gateway/configuration-reference#messages) и документация по каналам.
