# Сопряжение, управляемое шлюзом (вариант B)

При сопряжении, управляемом шлюзом, **шлюз** является единственным источником достоверной информации о том, каким узлам разрешено подключаться. Интерфейсы пользователя (приложение для macOS, будущие клиенты) — лишь фронтенды, которые утверждают или отклоняют ожидающие запросы.

**Важно:** узлы WS используют **сопряжение устройств** (роль `node`) во время `connect`. `node.pair.*` — это отдельный магазин сопряжения, который **не** регулирует рукопожатие WS. Этот поток используют только клиенты, явно вызывающие `node.pair.*`.

## Основные понятия

- **Ожидающий запрос**: узел запросил подключение; требуется утверждение.
- **Сопряжённый узел**: утверждённый узел с выданным токеном аутентификации.
- **Транспорт**: конечная точка Gateway WS пересылает запросы, но не принимает решений о допуске в систему. (Поддержка устаревшего моста TCP удалена.)

## Как работает сопряжение

1. Узел подключается к Gateway WS и запрашивает сопряжение.
2. Шлюз сохраняет **ожидающий запрос** и генерирует событие `node.pair.requested`.
3. Вы утверждаете или отклоняете запрос (через CLI или UI).
4. При утверждении шлюз выдаёт **новый токен** (токены обновляются при повторном сопряжении).
5. Узел переподключается с использованием токена и теперь считается "сопряжённым".

Ожидающие запросы автоматически истекают через **5 минут**.

## Рабочий процесс CLI (подходит для безголового режима)

```bash
openclaw nodes pending
openclaw nodes approve <requestId>
openclaw nodes reject <requestId>
openclaw nodes status
openclaw nodes rename --node <id|name|ip> --name "Living Room iPad"
```

Команда `nodes status` отображает сопряжённые/подключённые узлы и их возможности.

## Интерфейс API (протокол шлюза)

События:

- `node.pair.requested` — генерируется при создании нового ожидающего запроса.
- `node.pair.resolved` — генерируется, когда запрос утверждён/отклонён/истек.

Методы:

- `node.pair.request` — создать новый или повторно использовать существующий ожидающий запрос.
- `node.pair.list` — вывести список ожидающих и сопряжённых узлов (`operator.pairing`).
- `node.pair.approve` — утвердить ожидающий запрос (выдаёт токен).
- `node.pair.reject` — отклонить ожидающий запрос.
- `node.pair.verify` — проверить `{ nodeId, token }`.

Примечания:

- `node.pair.request` является идемпотентным для каждого узла: повторные вызовы возвращают один и тот же ожидающий запрос.
- Повторные запросы для одного и того же ожидающего узла также обновляют сохранённые метаданные узла и последнюю версию списка разрешённых команд (для видимости оператору).
- Утверждение **всегда** генерирует новый токен; токен никогда не возвращается из `node.pair.request`.
- Запросы могут включать параметр `silent: true` в качестве подсказки для потоков автоматического утверждения.
- `node.pair.approve` использует объявленные команды из ожидающего запроса для применения дополнительных областей утверждения:
  - запрос без команд: `operator.pairing`;
  - запрос с командами, не требующими выполнения: `operator.pairing` + `operator.write`;
  - запрос с `system.run` / `system.run.prepare` / `system.which`: `operator.pairing` + `operator.admin`.

Важно:

- Сопряжение узлов — это процесс установления доверия/идентификации плюс выдача токена.
- Оно **не** закрепляет актуальный набор команд узла для каждого узла.
- Актуальные команды узла определяются тем, что узел объявляет при подключении после применения глобальной политики команд узлов шлюза (`gateway.nodes.allowCommands` / `denyCommands`).
- Политика разрешения/запроса `system.run` для каждого узла хранится на узле в `exec.approvals.node.*`, а не в записи о сопряжении.

## Управление командами узлов (с 2026.03.31)

<Warning>
**Критическое изменение:** начиная с версии `2026.03.31`, команды узлов отключены до утверждения сопряжения узла. Одного сопряжения устройства уже недостаточно для предоставления объявленных команд узла.
</Warning>

Когда узел подключается впервые, запрос на сопряжение отправляется автоматически. Пока запрос на сопряжение не утверждён, все ожидающие команды узла фильтруются и не выполняются. После установления доверия через утверждение сопряжения объявленные команды узла становятся доступными в соответствии с обычной политикой команд.

Это означает:

- Узлы, которые ранее полагались только на сопряжение устройств для предоставления команд, теперь должны завершить сопряжение узла.
- Команды, поставленные в очередь до утверждения сопряжения, удаляются, а не откладываются.

## Границы доверия для событий узла (с 2026.03.31)

<Warning>
**Критическое изменение:** выполнения, инициированные узлом, теперь ограничены сокращённой доверенной областью.
</Warning>

Сводные данные, инициированные узлом, и связанные с ними события сеанса ограничены предполагаемой доверенной областью. Потоки, управляемые уведомлениями или инициированные узлом, которые ранее полагались на более широкий доступ к инструментам хоста или сеанса, могут потребовать корректировки. Это усиление безопасности гарантирует, что события узла не смогут получить доступ к инструментам уровня хоста за пределами того, что разрешено границей доверия узла.

## Автоматическое утверждение (приложение для macOS)

Приложение для macOS может при необходимости попытаться выполнить **тихое утверждение**, если:

- запрос помечен как `silent`;
- приложение может проверить подключение SSH к хосту шлюза с использованием того же пользователя.

Если тихое утверждение не удаётся, выполняется возврат к обычному диалогу "Утвердить/Отклонить".

## Хранение данных (локальное, приватное)

Состояние сопряжения хранится в каталоге состояния шлюза (по умолчанию `~/.openclaw`):

- `~/.openclaw/nodes/paired.json`;
- `~/.openclaw/nodes/pending.json`.

Если вы переопределите `OPENCLAW_STATE_DIR`, папка `nodes/` переместится вместе с ним.

Примечания по безопасности:

- Токены — это секретные данные; относитесь к `paired.json` как к конфиденциальной информации.
- Для обновления токена требуется повторное утверждение (или удаление записи об узле).

## Поведение транспорта

- Транспорт **не имеет состояния**; он не хранит информацию о членстве.
- Если шлюз недоступен или сопряжение отключено, узлы не могут выполнить сопряжение.
- Если шлюз находится в удалённом режиме, сопряжение всё равно выполняется относительно хранилища удалённого шлюза.
