## Публичная модель возможностей

Возможности — это публичная модель **нативного плагина** внутри OpenClaw. Каждый нативный плагин OpenClaw регистрируется для одной или нескольких типов возможностей:

| Возможность | Метод регистрации | Примеры плагинов |
| --- | --- | --- |
| Вывод текста | `api.registerProvider(...)` | `openai`, `anthropic` |
| Бэкенд вывода CLI | `api.registerCliBackend(...)` | `openai`, `anthropic` |
| Речь | `api.registerSpeechProvider(...)` | `elevenlabs`, `microsoft` |
| Транскрипция в реальном времени | `api.registerRealtimeTranscriptionProvider(...)` | `openai` |
| Голос в реальном времени | `api.registerRealtimeVoiceProvider(...)` | `openai` |
| Понимание медиа | `api.registerMediaUnderstandingProvider(...)` | `openai`, `google` |
| Генерация изображений | `api.registerImageGenerationProvider(...)` | `openai`, `google`, `fal`, `minimax` |
| Генерация музыки | `api.registerMusicGenerationProvider(...)` | `google`, `minimax` |
| Генерация видео | `api.registerVideoGenerationProvider(...)` | `qwen` |
| Получение веб-данных | `api.registerWebFetchProvider(...)` | `firecrawl` |
| Веб-поиск | `api.registerWebSearchProvider(...)` | `google` |
| Канал / обмен сообщениями | `api.registerChannel(...)` | `msteams`, `matrix` |

Плагин, который не регистрирует возможности, но предоставляет хуки, инструменты или сервисы, называется **устаревшим плагином только с хуками**. Этот шаблон по-прежнему полностью поддерживается.

### Позиция по внешней совместимости

Модель возможностей реализована в ядре и используется встроенными/нативными плагинами, но для совместимости внешних плагинов требуются более строгие критерии, чем "экспортировано — значит заморожено".

Текущие рекомендации:

- **существующие внешние плагины:** обеспечить работу интеграций на основе хуков; рассматривать это как базовый уровень совместимости;
- **новые встроенные/нативные плагины:** отдавать предпочтение явной регистрации возможностей вместо специфичных для поставщика решений или новых дизайнов только с хуками;
- **внешние плагины, принимающие регистрацию возможностей:** разрешено, но рассматривать вспомогательные интерфейсы, специфичные для возможностей, как развивающиеся, если в документации явно не указано, что контракт стабилен.

Практическое правило:

- API регистрации возможностей — это предполагаемое направление развития;
- устаревшие хуки остаются самым безопасным путём без сбоев для внешних плагинов во время перехода;
- экспортированные вспомогательные подпути не равнозначны; предпочтительнее узкий задокументированный контракт, а не случайные вспомогательные экспорты.

### Формы плагинов

OpenClaw классифицирует каждый загруженный плагин по форме на основе его фактического поведения при регистрации (а не только статических метаданных):

- **plain-capability** — регистрирует ровно один тип возможности (например, плагин только провайдера, такой как `mistral`);
- **hybrid-capability** — регистрирует несколько типов возможностей (например, `openai` владеет выводом текста, речью, пониманием медиа и генерацией изображений);
- **hook-only** — регистрирует только хуки (типизированные или пользовательские), без возможностей, инструментов, команд или сервисов;
- **non-capability** — регистрирует инструменты, команды, сервисы или маршруты, но не возможности.

Используйте `openclaw plugins inspect <id>`, чтобы увидеть форму плагина и разбивку по возможностям. Подробнее см. в [справочнике по CLI](/cli/plugins#inspect).

### Устаревшие хуки

Хук `before_agent_start` по-прежнему поддерживается как путь совместимости для плагинов только с хуками. Устаревшие реальные плагины по-прежнему зависят от него.

Направление развития:

- сохранить его работоспособность;
- задокументировать как устаревший;
- предпочитать `before_model_resolve` для работы с переопределением модели/провайдера;
- предпочитать `before_prompt_build` для работы с изменением промпта;
- удалить только после снижения реального использования и проверки безопасности миграции с помощью тестовых покрытий.

### Сигналы совместимости

При запуске `openclaw doctor` или `openclaw plugins inspect <id>` вы можете увидеть один из следующих сигналов:

| Сигнал | Значение |
| --- | --- |
| **config valid** | Конфигурация успешно проанализирована, плагины разрешены |
| **compatibility advisory** | Плагин использует поддерживаемый, но устаревший шаблон (например, `hook-only`) |
| **legacy warning** | Плагин использует `before_agent_start`, который устарел |
| **hard error** | Конфигурация недействительна или плагин не удалось загрузить |

Ни `hook-only`, ни `before_agent_start` не нарушат работу вашего плагина сегодня — `hook-only` является рекомендацией, а `before_agent_start` вызывает только предупреждение. Эти сигналы также отображаются в `openclaw status --all` и `openclaw plugins doctor`.

## Обзор архитектуры

Система плагинов OpenClaw состоит из четырёх уровней:

1. **Манифест + обнаружение**
   OpenClaw находит потенциальные плагины из настроенных путей, корневых каталогов рабочих пространств, глобальных корневых каталогов расширений и встроенных расширений. Сначала считываются нативные манифесты `openclaw.plugin.json` и поддерживаемые манифесты пакетов.
2. **Включение + валидация**
   Ядро определяет, включён ли обнаруженный плагин, отключён, заблокирован или выбран для эксклюзивного слота, например, памяти.
3. **Загрузка во время выполнения**
   Нативные плагины OpenClaw загружаются внутри процесса через jiti и регистрируют возможности в центральном реестре. Совместимые пакеты нормализуются в записи реестра без импорта кода выполнения.
4. **Использование поверхностей**
   Остальная часть OpenClaw считывает реестр для предоставления инструментов, каналов, настройки провайдера, хуков, HTTP-маршрутов, команд CLI и сервисов.

Для CLI плагинов конкретно обнаружение корневых команд разделено на два этапа:

- метаданные времени парсинга поступают из `registerCli(..., { descriptors: [...] })`;
- реальный модуль CLI плагина может оставаться ленивым и регистрироваться при первом вызове.

Это позволяет коду CLI, принадлежащему плагину, оставаться внутри плагина, при этом OpenClaw может зарезервировать имена корневых команд до парсинга.

Важное разграничение дизайна:

- обнаружение + валидация конфигурации должны работать на основе **метаданных манифеста/схемы** без выполнения кода плагина;
- нативное поведение во время выполнения исходит из пути `register(api)` модуля плагина.

Такое разделение позволяет OpenClaw валидировать конфигурацию, объяснять отсутствующие/отключённые плагины и создавать подсказки UI/схемы до активации полного времени выполнения.

### Плагины каналов и общий инструмент сообщений

Плагины каналов не должны регистрировать отдельный инструмент отправки/редактирования/реакции для обычных действий чата. OpenClaw поддерживает один общий инструмент `message` в ядре, а плагины каналов владеют обнаружением и выполнением, специфичным для канала, за ним.

Текущее разграничение:

- ядро владеет общим хостом инструмента `message`, настройкой промпта, учётом сессий/потоков и диспетчеризацией выполнения;
- плагины каналов владеют обнаружением действий, специфичным для канала, обнаружением возможностей и любыми фрагментами схемы, специфичными для канала;
- плагины каналов владеют грамматикой разговоров провайдера, например, как идентификаторы разговоров кодируют идентификаторы потоков или наследуют от родительских разговоров;
- плагины каналов выполняют окончательное действие через свой адаптер действий.

Для плагинов каналов поверхность SDK — `ChannelMessageActionAdapter.describeMessageTool(...)`. Этот унифицированный вызов обнаружения позволяет плагину вернуть свои видимые действия, возможности и вклад в схему вместе, чтобы эти части не расходились.

- `currentChannelId`;
- `currentThreadTs`;
- `currentMessageId`;
