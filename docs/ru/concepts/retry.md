# Политика повторных попыток

## Цели

- Выполнять повторные попытки для каждого HTTP-запроса, а не для многоэтапного потока.
- Сохранять порядок, повторяя только текущий этап.
- Избегать дублирования неидемпотентных операций.

## Значения по умолчанию

- Количество попыток: 3
- Максимальный предел задержки: 30 000 мс
- Джиттер: 0,1 (10 процентов)
- Значения по умолчанию для провайдеров:
  - Минимальная задержка для Telegram: 400 мс
  - Минимальная задержка для Discord: 500 мс

## Поведение

### Discord

- Повторные попытки выполняются только при ошибках, связанных с ограничением частоты запросов (HTTP 429).
- Используется параметр Discord `retry_after`, если он доступен; в противном случае применяется экспоненциальная отсрочка.

### Telegram

- Повторные попытки выполняются при временных ошибках (429, тайм-аут, разрыв/сброс/закрытие соединения, временная недоступность).
- Используется `retry_after`, если он доступен; в противном случае применяется экспоненциальная отсрочка.
- Ошибки разбора Markdown не повторяются — вместо этого используется обычный текст.

## Конфигурация

Задайте политику повторных попыток для каждого провайдера в файле `~/.openclaw/openclaw.json`:

```json5
{
  channels: {
    telegram: {
      retry: {
        attempts: 3,
        minDelayMs: 400,
        maxDelayMs: 30000,
        jitter: 0.1,
      },
    },
    discord: {
      retry: {
        attempts: 3,
        minDelayMs: 500,
        maxDelayMs: 30000,
        jitter: 0.1,
      },
    },
  },
}
```

## Примечания

- Повторные попытки применяются для каждого запроса (отправка сообщения, загрузка медиа, реакция, опрос, стикер).
- В составных потоках завершённые этапы не повторяются.
