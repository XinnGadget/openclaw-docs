# Флаги диагностики

Флаги диагностики позволяют включать целевые журналы отладки, не активируя подробного логирования во всей системе. Флаги подключаются по запросу и не оказывают никакого влияния, пока подсистема их не проверит.

## Как это работает

- Флаги представляют собой строки (без учёта регистра).
- Вы можете включить флаги в конфигурации или через переопределение переменной окружения.
- Поддерживаются подстановочные знаки:
  - `telegram.*` соответствует `telegram.http`
  - `*` включает все флаги

## Включение через конфигурацию

```json
{
  "diagnostics": {
    "flags": ["telegram.http"]
  }
}
```

Несколько флагов:

```json
{
  "diagnostics": {
    "flags": ["telegram.http", "gateway.*"]
  }
}
```

После изменения флагов перезапустите шлюз.

## Переопределение через переменную окружения (однократно)

```bash
OPENCLAW_DIAGNOSTICS=telegram.http,telegram.payload
```

Отключить все флаги:

```bash
OPENCLAW_DIAGNOSTICS=0
```

## Куда попадают журналы

Флаги выводят журналы в стандартный файл журнала диагностики. По умолчанию:

```
/tmp/openclaw/openclaw-YYYY-MM-DD.log
```

Если вы задали параметр `logging.file`, используйте указанный путь. Журналы представлены в формате JSONL (один JSON-объект на строку). Редактирование по-прежнему применяется в соответствии с параметром `logging.redactSensitive`.

## Извлечение журналов

Выберите самый свежий файл журнала:

```bash
ls -t /tmp/openclaw/openclaw-*.log | head -n 1
```

Отфильтруйте журналы диагностики Telegram HTTP:

```bash
rg "telegram http error" /tmp/openclaw/openclaw-*.log
```

Или отслеживайте в реальном времени при воспроизведении проблемы:

```bash
tail -f /tmp/openclaw/openclaw-$(date +%F).log | rg "telegram http error"
```

Для удалённых шлюзов также можно использовать команду `openclaw logs --follow` (см. [/cli/logs](/cli/logs)).

## Примечания

- Если параметр `logging.level` задан выше, чем `warn`, эти журналы могут быть подавлены. Значение по умолчанию `info` подходит.
- Флаги безопасно оставлять включёнными: они влияют только на объём журналов для конкретной подсистемы.
- Используйте [/logging](/logging), чтобы изменить места назначения журналов, уровни и редактирование.
