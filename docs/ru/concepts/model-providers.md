# Провайдеры моделей

На этой странице рассматривается **провайдеры LLM/моделей** (не каналы чата вроде WhatsApp/Telegram).
Правила выбора моделей см. в разделе [/concepts/models](/concepts/models).

## Краткие правила

- Ссылки на модели используют формат `провайдер/модель` (пример: `opencode/claude-opus-4-6`).
- Если вы задаёте `agents.defaults.models`, это становится списком разрешённых моделей.
- Помощники CLI: `openclaw onboard`, `openclaw models list`, `openclaw models set <провайдер/модель>`.
- Правила резервного времени выполнения, зонды охлаждения и сохранение переопределений сессии описаны в [/concepts/model-failover](/concepts/model-failover).
- `models.providers.*.models[].contextWindow` — собственные метаданные модели;
  `models.providers.*.models[].contextTokens` — эффективное ограничение времени выполнения.
- Плагины провайдеров могут внедрять каталоги моделей через `registerProvider({ catalog })`;
  OpenClaw объединяет этот вывод в `models.providers` перед записью в `models.json`.
- Манифесты провайдеров могут объявлять `providerAuthEnvVars` и
  `providerAuthAliases`, чтобы общие зонды аутентификации на основе переменных окружения и варианты провайдеров
  не требовали загрузки времени выполнения плагина. Оставшаяся основная карта переменных окружения теперь предназначена только для провайдеров без плагинов/основных провайдеров и для некоторых случаев с общим приоритетом, таких как использование API-ключа Anthropic при подключении.
- Плагины провайдеров также могут управлять поведением времени выполнения провайдера через
  `normalizeModelId`, `normalizeTransport`, `normalizeConfig`,
  `applyNativeStreamingUsageCompat`, `resolveConfigApiKey`,
  `resolveSyntheticAuth`, `shouldDeferSyntheticProfileAuth`,
  `resolveDynamicModel`, `prepareDynamicModel`,
  `normalizeResolvedModel`, `contributeResolvedModelCompat`,
  `capabilities`, `normalizeToolSchemas`,
  `inspectToolSchemas`, `resolveReasoningOutputMode`,
  `prepareExtraParams`, `createStreamFn`, `wrapStreamFn`,
  `resolveTransportTurnState`, `resolveWebSocketSessionPolicy`,
  `createEmbeddingProvider`, `formatApiKey`, `refreshOAuth`,
  `buildAuthDoctorHint`,
  `matchesContextOverflowError`, `classifyFailoverReason`,
  `isCacheTtlEligible`, `buildMissingAuthMessage`, `suppressBuiltInModel`,
  `augmentModelCatalog`, `isBinaryThinking`, `supportsXHighThinking`,
  `resolveDefaultThinkingLevel`, `applyConfigDefaults`, `isModernModelRef`,
  `prepareRuntimeAuth`, `resolveUsageAuth`, `fetchUsageSnapshot` и
  `onModelSelected`.
- Примечание: `capabilities` времени выполнения провайдера — это общие метаданные раннера (семейство провайдеров, особенности транскрипции/инструментов, подсказки по транспорту/кэшированию). Это не то же самое, что [общедоступная модель возможностей](/plugins/architecture#public-capability-model), которая описывает, что регистрирует плагин (текстовый вывод, речь и т. д.).

## Поведение провайдера, управляемое плагином

Плагины провайдеров теперь могут управлять большей частью логики, специфичной для провайдера, в то время как OpenClaw сохраняет общий цикл вывода.

Типичное разделение:

- `auth[].run` / `auth[].runNonInteractive`: провайдер управляет потоками подключения/входа в систему для `openclaw onboard`, `openclaw models auth` и настройки без интерфейса;
- `wizard.setup` / `wizard.modelPicker`: провайдер управляет метками выбора аутентификации, устаревшими псевдонимами, подсказками списка разрешённых моделей при подключении и записями настройки в мастере подключения/выборе модели;
- `catalog`: провайдер появляется в `models.providers`;
- `normalizeModelId`: провайдер нормализует устаревшие/предварительные идентификаторы моделей перед поиском или канонизацией;
- `normalizeTransport`: провайдер нормализует семейство транспорта `api` / `baseUrl` перед общим сборкой модели; OpenClaw сначала проверяет соответствующий провайдер, затем другие плагины провайдеров с хуками, пока один из них фактически не изменит транспорт;
- `normalizeConfig`: провайдер нормализует конфигурацию `models.providers.<id>` перед её использованием во время выполнения; OpenClaw сначала проверяет соответствующий провайдер, затем другие плагины провайдеров с хуками, пока один из них фактически не изменит конфигурацию. Если ни один хук провайдера не перезаписывает конфигурацию, встроенные помощники семейства Google всё равно нормализуют поддерживаемые записи провайдеров Google.
- `applyNativeStreamingUsageCompat`: провайдер применяет переписывание совместимости нативного потокового использования для конфигурационных провайдеров на основе конечных точек;
- `resolveConfigApiKey`: провайдер разрешает аутентификацию по маркерам окружения для конфигурационных провайдеров без принудительной полной загрузки аутентификации во время выполнения. `amazon-bedrock` также имеет встроенный резолвер маркеров окружения AWS, даже несмотря на то, что аутентификация времени выполнения Bedrock использует цепочку SDK AWS по умолчанию;
- `resolveSyntheticAuth`: провайдер может предоставлять доступность локальной/самостоятельной или другой аутентификации на основе конфигурации без сохранения секретов в виде открытого текста;
- `shouldDeferSyntheticProfileAuth`: провайдер может отмечать сохранённые синтетические заполнители профилей как имеющие более низкий приоритет по сравнению с аутентификацией на основе окружения/конфигурации;
- `resolveDynamicModel`: провайдер принимает идентификаторы моделей, которых ещё нет в локальном статическом каталоге;
- `prepareDynamicModel`: провайдеру требуется обновление метаданных перед повторной попыткой динамического разрешения;
- `normalizeResolvedModel`: провайдеру требуются переписывание транспорта или базового URL;
- `contributeResolvedModelCompat`: провайдер вносит флаги совместимости для своих моделей поставщиков, даже если они поступают через другой совместимый транспорт;
- `capabilities`: провайдер публикует особенности транскрипции/инструментов/семейства провайдеров;
- `normalizeToolSchemas`: провайдер очищает схемы инструментов перед тем, как встроенный раннер их увидит;
- `inspectToolSchemas`: провайдер отображает предупреждения о схемах, специфичные для транспорта, после нормализации;
- `resolveReasoningOutputMode`: провайдер выбирает между нативным и помеченным контрактами вывода рассуждений;
- `prepareExtraParams`: провайдер устанавливает значения по умолчанию или нормализует параметры запроса для каждой модели;
- `createStreamFn`: провайдер заменяет обычный путь потока полностью пользовательским транспортом;
- `wrapStreamFn`: провайдер применяет обёртки заголовков/тела запроса/совместимости модели;
- `resolveTransportTurnState`: провайдер предоставляет заголовки или метаданные транспорта для каждого хода;
- `resolveWebSocketSessionPolicy`: провайдер предоставляет заголовки нативного WebSocket-сеанса или политику охлаждения сеанса;
- `createEmbeddingProvider`: провайдер управляет поведением встраивания памяти, когда оно принадлежит плагину провайдера, а не основному коммутатору встраивания;
- `formatApiKey`: провайдер форматирует сохранённые профили аутентификации в строку `apiKey` времени выполнения, ожидаемую транспортом;
- `refreshOAuth`: провайдер управляет обновлением OAuth, когда общих обновляющих элементов `pi-ai` недостаточно;
- `buildAuthDoctorHint`: провайдер добавляет рекомендации по устранению неполадок при сбое обновления OAuth;
- `matchesContextOverflowError`: провайдер распознаёт ошибки переполнения окна контекста, специфичные для провайдера, которые могут быть пропущены общими эвристиками;
- `classifyFailoverReason`: провайдер сопоставляет ошибки транспорта/API, специфичные для провайдера, с причинами переключения, такими как ограничение скорости или перегрузка;
- `isCacheTtlEligible`: провайдер определяет, какие идентификаторы моделей вышестоящего уровня поддерживают TTL кэша подсказок;
- `buildMissingAuthMessage`: провайдер заменяет общую ошибку хранилища аутентификации подсказкой по восстановлению, специфичной для провайдера;
- `suppressBuiltInModel`: провайдер скрывает устаревшие строки вышестоящего уровня и может возвращать ошибку, принадлежащую поставщику, при сбоях прямого разрешения;
- `augmentModelCatalog`: провайдер добавляет синтетические/окончательные строки каталога после обнаружения и слияния конфигурации;
- `isBinaryThinking`: провайдер управляет бинарным UX включения/выключения рассуждений;
- `supportsXHighThinking`: провайдер включает модели в `xhigh`;
- `resolveDefaultThinkingLevel`: провайдер управляет политикой `/think` по умолчанию для семейства моделей;
- `applyConfigDefaults`: провайдер применяет глобальные значения по умолчанию, специфичные для провайдера, во время материализации конфигурации на основе режима аутентификации, окружения или семейства моделей;
- `isModernModelRef`: провайдер управляет сопоставлением предпочтительных моделей в режиме реального времени/пробного запуска;
- `prepareRuntimeAuth`: провайдер преобразует настроенные учётные данные в краткосрочный токен времени выполнения;
- `resolveUsageAuth`: провайдер разрешает учётные данные использования/квоты для `/usage` и связанных поверхностей состояния/отчётности;
- `fetchUsageSnapshot`: провайдер управляет получением/анализом конечной точки использования, в то время как ядро по-прежнему управляет оболочкой сводки и форматированием;
- `onModelSelected`:
