# API OpenResponses (HTTP)

Gateway в OpenClaw может предоставлять конечную точку `POST /v1/responses`, совместимую с OpenResponses.

Эта конечная точка **по умолчанию отключена**. Сначала необходимо включить её в конфигурации.

- `POST /v1/responses`
- Тот же порт, что и у Gateway (мультиплексирование WS + HTTP): `http://<gateway-host>:<port>/v1/responses`

Под капотом запросы выполняются как обычный запуск агента Gateway (тот же путь кода, что и для `openclaw agent`), поэтому маршрутизация, разрешения и конфигурация соответствуют вашему Gateway.

## Аутентификация, безопасность и маршрутизация

Поведение в работе соответствует [OpenAI Chat Completions](/gateway/openai-http-api):

- используйте соответствующий путь аутентификации HTTP Gateway:
  - аутентификация по общему секрету (`gateway.auth.mode="token"` или `"password"`): `Authorization: Bearer <token-or-password>`
  - аутентификация через доверенный прокси (`gateway.auth.mode="trusted-proxy"`): заголовки прокси с идентификацией от настроенного доверенного источника прокси (не loopback)
  - открытая аутентификация для частного входа (`gateway.auth.mode="none"`): заголовок аутентификации отсутствует
- рассматривайте конечную точку как полный доступ оператора для экземпляра Gateway
- для режимов аутентификации по общему секрету (`token` и `password`) игнорируйте более узкие значения `x-openclaw-scopes`, указанные в токене, и восстановите стандартные настройки полного доступа оператора
- для режимов HTTP с доверенной идентификацией (например, аутентификация через доверенный прокси или `gateway.auth.mode="none"`) учитывайте `x-openclaw-scopes`, если заголовок присутствует, в противном случае используйте стандартный набор областей доступа оператора
- выбирайте агентов с помощью `model: "openclaw"`, `model: "openclaw/default"`, `model: "openclaw/<agentId>"` или `x-openclaw-agent-id`
- используйте `x-openclaw-model`, если хотите переопределить бэкенд-модель выбранного агента
- используйте `x-openclaw-session-key` для явной маршрутизации сессии
- используйте `x-openclaw-message-channel`, если вам нужен нестандартный контекст синтетического входящего канала

Матрица аутентификации:

- `gateway.auth.mode="token"` или `"password"` + `Authorization: Bearer ...`
  - подтверждает владение общим секретом оператора Gateway
  - игнорирует более узкие `x-openclaw-scopes`
  - восстанавливает полный стандартный набор областей доступа оператора:
    `operator.admin`, `operator.approvals`, `operator.pairing`,
    `operator.read`, `operator.talk.secrets`, `operator.write`
  - рассматривает повороты чата на этой конечной точке как повороты от отправителя-владельца
- режимы HTTP с доверенной идентификацией (например, аутентификация через доверенный прокси или `gateway.auth.mode="none"` для частного входа)
  - учитывают `x-openclaw-scopes`, если заголовок присутствует
  - возвращаются к стандартному набору областей доступа оператора, если заголовок отсутствует
  - теряют семантику владельца только в том случае, если вызывающий явно сужает области доступа и исключает `operator.admin`

Включите или отключите эту конечную точку с помощью `gateway.http.endpoints.responses.enabled`.

Та же поверхность совместимости включает:

- `GET /v1/models`
- `GET /v1/models/{id}`
- `POST /v1/embeddings`
- `POST /v1/chat/completions`

Для канонического объяснения того, как модели, нацеленные на агентов, `openclaw/default`, передача эмбеддингов и переопределение бэкенд-модели сочетаются друг с другом, см. [OpenAI Chat Completions](/gateway/openai-http-api#agent-first-model-contract) и [Список моделей и маршрутизация агентов](/gateway/openai-http-api#model-list-and-agent-routing).

## Поведение сессии

По умолчанию конечная точка **не сохраняет состояние между запросами** (новый ключ сессии генерируется при каждом вызове).

Если запрос включает строку `user` из OpenResponses, Gateway выводит стабильный ключ сессии на её основе, поэтому повторные вызовы могут использовать одну и ту же сессию агента.

## Форма запроса (поддерживается)

Запрос соответствует API OpenResponses с входными данными на основе элементов. Текущая поддержка:

- `input`: строка или массив объектов элементов.
- `instructions`: объединяются с системным промтом.
- `tools`: определения клиентских инструментов (функциональные инструменты).
- `tool_choice`: фильтрация или требование клиентских инструментов.
- `stream`: включает потоковую передачу SSE.
- `max_output_tokens`: максимально возможное ограничение вывода (зависит от провайдера).
- `user`: стабильная маршрутизация сессии.

Принимаются, но **в настоящее время игнорируются**:

- `max_tool_calls`
- `reasoning`
- `metadata`
- `store`
- `truncation`

Поддерживается:

- `previous_response_id`: OpenClaw повторно использует предыдущую сессию ответа, если запрос остаётся в пределах той же области агента/пользователя/запрошенной сессии.

## Элементы (вход)

### `message`

Роли: `system`, `developer`, `user`, `assistant`.

- `system` и `developer` добавляются к системному промту.
- Самый последний элемент `user` или `function_call_output` становится "текущим сообщением".
- Более ранние сообщения `user`/`assistant` включаются в историю для контекста.

### `function_call_output` (инструменты на основе поворотов)

Отправьте результаты инструмента обратно модели:

```json
{
  "type": "function_call_output",
  "call_id": "call_123",
  "output": "{\"temperature\": \"72F\"}"
}
```

### `reasoning` и `item_reference`

Принимаются для совместимости со схемой, но игнорируются при построении промта.

## Инструменты (функциональные инструменты на стороне клиента)

Предоставьте инструменты с помощью `tools: [{ type: "function", function: { name, description?, parameters? } }]`.

Если агент решает вызвать инструмент, ответ возвращает элемент вывода `function_call`. Затем вы отправляете дополнительный запрос с `function_call_output`, чтобы продолжить поворот.

## Изображения (`input_image`)

Поддерживаются источники в формате base64 или URL:

```json
{
  "type": "input_image",
  "source": { "type": "url", "url": "https://example.com/image.png" }
}
```

Допустимые типы MIME (на данный момент): `image/jpeg`, `image/png`, `image/gif`, `image/webp`, `image/heic`, `image/heif`.
Максимальный размер (на данный момент): 10 МБ.

## Файлы (`input_file`)

Поддерживаются источники в формате base64 или URL:

```json
{
  "type": "input_file",
  "source": {
    "type": "base64",
    "media_type": "text/plain",
    "data": "SGVsbG8gV29ybGQh",
    "filename": "hello.txt"
  }
}
```

Допустимые типы MIME (на данный момент): `text/plain`, `text/markdown`, `text/html`, `text/csv`,
`application/json`, `application/pdf`.

Максимальный размер (на данный момент): 5 МБ.

Текущее поведение:

- Содержимое файла декодируется и добавляется в **системный промт**, а не в сообщение пользователя, поэтому оно остаётся временным (не сохраняется в истории сессии).
- Декодированный текст файла оборачивается как **ненадёжный внешний контент** перед добавлением, поэтому байты файла рассматриваются как данные, а не как доверенные инструкции.
- Вставленный блок использует явные маркеры границ, такие как
  `<<<EXTERNAL_UNTRUSTED_CONTENT id="...">>>` /
  `<<<END_EXTERNAL_UNTRUSTED_CONTENT id="...">>>`, и включает строку метаданных `Source: External`.
- Этот путь ввода файлов намеренно пропускает длинный баннер `SECURITY NOTICE:`, чтобы сохранить бюджет промта; маркеры границ и метаданные остаются на месте.
- PDF-файлы сначала анализируются на наличие текста. Если текста мало, первые страницы растеризуются в изображения и передаются модели, а вставленный блок файла использует заполнитель `[PDF content rendered to images]`.

Для анализа PDF используется устаревшая сборка `pdfjs-dist`, совместимая с Node (без воркеров). Современная сборка PDF.js ожидает воркеров браузера/глобальных объектов DOM
