# Побочные вопросы BTW

Команда `/btw` позволяет быстро задать побочный вопрос о **текущей сессии**, не добавляя этот вопрос в обычную историю диалога.

Реализация основана на поведении `/btw` в Claude Code, но адаптирована для Gateway в OpenClaw и архитектуры с поддержкой нескольких каналов.

## Что делает команда

Когда вы отправляете:

```text
/btw что изменилось?
```

OpenClaw выполняет следующие действия:

1. создаёт снимок контекста текущей сессии;
2. запускает отдельный вызов модели **без инструментов** (tool-less);
3. отвечает только на побочный вопрос;
4. не затрагивает основной процесс;
5. **не** записывает вопрос и ответ BTW в историю сессии;
6. передаёт ответ в виде **оперативного побочного результата** (live side result), а не обычного сообщения ассистента.

Важная концептуальная модель:

- тот же контекст сессии;
- отдельный одноразовый побочный запрос;
- отсутствие вызовов инструментов;
- отсутствие загрязнения будущего контекста;
- отсутствие сохранения в транскрипте.

## Чего команда не делает

`/btw` **не**:

- создаёт новую долговременную сессию;
- продолжает незавершённую основную задачу;
- запускает инструменты или циклы вызовов инструментов агента;
- записывает данные вопроса/ответа BTW в историю транскриптов;
- отображается в `chat.history`;
- сохраняется после перезагрузки.

Команда намеренно **временная** (ephemeral).

## Как работает контекст

BTW использует текущую сессию **только в качестве фонового контекста**.

Если основной процесс активен, OpenClaw создаёт снимок текущего состояния сообщений и включает выполняющийся основной промпт в качестве фонового контекста, при этом явно указывая модели:

- отвечать только на побочный вопрос;
- не возобновлять и не завершать незавершённую основную задачу;
- не выдавать вызовы инструментов или псевдовызовы инструментов.

Это позволяет изолировать BTW от основного процесса, сохраняя при этом осведомлённость о сути сессии.

## Модель доставки

BTW **не** доставляется как обычное сообщение ассистента в транскрипте.

На уровне протокола Gateway:

- обычный чат с ассистентом использует событие `chat`;
- BTW использует событие `chat.side_result`.

Такое разделение намеренно. Если бы BTW использовал обычный путь события `chat`, клиенты обрабатывали бы его как обычную историю диалога.

Поскольку BTW использует отдельное оперативное событие и не воспроизводится из `chat.history`, он исчезает после перезагрузки.

## Поведение на интерфейсах

### TUI

В TUI ответ BTW отображается встраиваемо в представлении текущей сессии, но остаётся временным:

- визуально отличается от обычного ответа ассистента;
- может быть закрыт с помощью `Enter` или `Esc`;
- не воспроизводится при перезагрузке.

### Внешние каналы

В таких каналах, как Telegram, WhatsApp и Discord, ответ BTW доставляется в виде чётко помеченного разового ответа, поскольку в этих интерфейсах нет концепции локального временного наложения.

Ответ по-прежнему обрабатывается как побочный результат, а не как обычная история сессии.

### Control UI / веб

Gateway корректно передаёт BTW как `chat.side_result`, и BTW не включается в `chat.history`, поэтому условия сохранения данных уже соблюдены для веб-версии.

Текущему Control UI всё ещё требуется специальный обработчик `chat.side_result`, чтобы отображать BTW в режиме реального времени в браузере. Пока эта поддержка на стороне клиента не реализована, BTW — это функция уровня Gateway с полным набором возможностей для TUI и внешних каналов, но ещё не полностью реализованная в браузере с точки зрения UX.

## Когда использовать BTW

Используйте `/btw`, если вам нужно:

- быстро уточнить что-то по текущей работе;
- получить фактический побочный ответ, пока длительный процесс ещё выполняется;
- получить временный ответ, который не должен стать частью будущего контекста сессии.

Примеры:

```text
/btw какой файл мы редактируем?
/btw что означает эта ошибка?
/btw сформулируй текущую задачу в одном предложении
/btw сколько будет 17 × 19?
```

## Когда не использовать BTW

Не используйте `/btw`, если хотите, чтобы ответ стал частью будущего рабочего контекста сессии.

В этом случае задавайте вопрос обычным образом в основной сессии, а не через BTW.
