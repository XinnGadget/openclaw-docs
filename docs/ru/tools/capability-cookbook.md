# Добавление возможностей

<Info>
  Это **руководство для участников** для разработчиков ядра OpenClaw. Если вы
создаёте внешний плагин, ознакомьтесь вместо этого с разделом [Создание плагинов](/plugins/building-plugins).
</Info>

Используйте это руководство, когда в OpenClaw требуется новая область применения — например, генерация изображений, генерация видео или какая-либо будущая функциональность, поддерживаемая поставщиком.

Правило:

- плагин = граница ответственности;
- возможность = общий контракт ядра.

Это означает, что не следует начинать с прямого подключения поставщика к каналу или инструменту. Начните с определения возможности.

## Когда создавать возможность

Создавайте новую возможность, если выполняются все следующие условия:

1. Её потенциально может реализовать более одного поставщика.
2. Каналы, инструменты или функциональные плагины должны использовать её, не учитывая конкретного поставщика.
3. Ядро должно управлять поведением при откате, политиками, конфигурацией или доставкой.

Если работа относится только к поставщику и общего контракта ещё нет, остановитесь и сначала определите контракт.

## Стандартная последовательность действий

1. Определите типизированный контракт ядра.
2. Добавьте регистрацию плагина для этого контракта.
3. Добавьте общий вспомогательный компонент среды выполнения.
4. Подключите один реальный плагин поставщика в качестве подтверждения концепции.
5. Переведите потребителей функций/каналов на использование вспомогательного компонента среды выполнения.
6. Добавьте тесты контракта.
7. Документируйте конфигурацию для оператора и модель ответственности.

## Что и где размещается

Ядро:

- типы запросов и ответов;
- реестр поставщиков + разрешение;
- поведение при откате;
- схема конфигурации, а также распространяемые метаданные документации `title` / `description` для вложенных объектов, узлов с подстановкой, элементов массивов и композиционных узлов;
- поверхность вспомогательного компонента среды выполнения.

Плагин поставщика:

- вызовы API поставщика;
- обработка аутентификации поставщика;
- нормализация запросов с учётом особенностей поставщика;
- регистрация реализации возможности.

Функциональный плагин / плагин канала:

- вызывает `api.runtime.*` или соответствующий вспомогательный компонент `plugin-sdk/*-runtime`;
- никогда не вызывает реализацию поставщика напрямую.

## Контрольный список файлов

При добавлении новой возможности, как правило, затрагиваются следующие области:

- `src/<capability>/types.ts`
- `src/<capability>/...registry/runtime.ts`
- `src/plugins/types.ts`
- `src/plugins/registry.ts`
- `src/plugins/captured-registration.ts`
- `src/plugins/contracts/registry.ts`
- `src/plugins/runtime/types-core.ts`
- `src/plugins/runtime/index.ts`
- `src/plugin-sdk/<capability>.ts`
- `src/plugin-sdk/<capability>-runtime.ts`
- один или несколько пакетов встроенных плагинов;
- конфигурация / документация / тесты.

## Пример: генерация изображений

Генерация изображений соответствует стандартной схеме:

1. Ядро определяет `ImageGenerationProvider`.
2. Ядро предоставляет `registerImageGenerationProvider(...)`.
3. Ядро предоставляет `runtime.imageGeneration.generate(...)`.
4. Плагины `openai`, `google`, `fal` и `minimax` регистрируют реализации, поддерживаемые поставщиками.
5. Будущие поставщики могут зарегистрировать тот же контракт без изменения каналов/инструментов.

Ключ конфигурации отделён от маршрутизации анализа зрения:

- `agents.defaults.imageModel` = анализ изображений;
- `agents.defaults.imageGenerationModel` = генерация изображений.

Сохраняйте их раздельно, чтобы поведение при откате и политики оставались явными.

## Контрольный список для проверки

Перед выпуском новой возможности убедитесь в следующем:

- ни один канал/инструмент не импортирует код поставщика напрямую;
- вспомогательный компонент среды выполнения является общим путём;
- по крайней мере один тест контракта подтверждает встроенную ответственность;
- в документации по конфигурации указан новый ключ модели/конфигурации;
- в документации плагина разъяснена граница ответственности.

Если в запросе на включение (PR) пропущен уровень возможностей и поведение поставщика жёстко закодировано в канале/инструменте, верните его и сначала определите контракт.
