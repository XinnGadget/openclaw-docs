# Приложение для iOS (узел)

Доступность: внутренний предварительный просмотр. Приложение для iOS пока не распространяется публично.

## Что оно делает

- Подключается к шлюзу через WebSocket (локальная сеть или tailnet).
- Предоставляет возможности узла: холст (Canvas), снимок экрана, захват изображения с камеры, определение местоположения, режим разговора (Talk mode), голосовое пробуждение (Voice wake).
- Принимает команды `node.invoke` и отправляет события о состоянии узла.

## Требования

- Шлюз, запущенный на другом устройстве (macOS, Linux или Windows через WSL2).
- Путь в сети:
  - Та же локальная сеть через Bonjour, **или**
  - Tailnet через unicast DNS-SD (пример домена: `openclaw.internal.`), **или**
  - Ручное указание хоста и порта (запасной вариант).

## Быстрый старт (сопряжение + подключение)

1. Запустите шлюз:

```bash
openclaw gateway --port 18789
```

2. В приложении для iOS откройте "Настройки" и выберите обнаруженный шлюз (или включите "Ручной хост" и введите хост и порт).

3. Подтвердите запрос на сопряжение на хосте шлюза:

```bash
openclaw devices list
openclaw devices approve <requestId>
```

Если приложение повторяет попытку сопряжения с изменёнными данными аутентификации (роль/области/открытый ключ), предыдущий ожидающий запрос отменяется и создаётся новый `requestId`. Перед подтверждением выполните команду `openclaw devices list` ещё раз.

4. Проверьте подключение:

```bash
openclaw nodes status
openclaw gateway call node.list --params "{}"
```

## Push через релей для официальных сборок

Официальные сборки приложения для iOS используют внешний релей push вместо передачи необработанного токена APNs шлюзу.

Требования со стороны шлюза:

```json5
{
  gateway: {
    push: {
      apns: {
        relay: {
          baseUrl: "https://relay.example.com",
        },
      },
    },
  },
}
```

Как работает процесс:

- Приложение для iOS регистрируется у релея с использованием App Attest и квитанции приложения.
- Релей возвращает непрозрачный дескриптор релея, а также грант на отправку, ограниченный областью регистрации.
- Приложение для iOS получает идентификатор сопряжённого шлюза и включает его в регистрацию у релея, так что регистрация через релей делегируется этому конкретному шлюзу.
- Приложение передаёт регистрацию через релей сопряжённому шлюзу с помощью `push.apns.register`.
- Шлюз использует сохранённый дескриптор релея для `push.test`, пробуждения в фоновом режиме и уведомлений о пробуждении.
- Базовый URL релея шлюза должен совпадать с URL релея, встроенным в официальную сборку/сборку TestFlight для iOS.
- Если приложение позже подключается к другому шлюзу или к сборке с другим базовым URL релея, оно обновляет регистрацию у релея вместо повторного использования старой привязки.

Что шлюзу **не** требуется для этого пути:

- Токен релея для всего развёртывания.
- Прямой ключ APNs для официальных сборок/сборок TestFlight с поддержкой релея.

Ожидаемый процесс для оператора:

1. Установите официальную сборку/сборку TestFlight для iOS.
2. Задайте `gateway.push.apns.relay.baseUrl` на шлюзе.
3. Сопрягите приложение со шлюзом и дождитесь завершения подключения.
4. Приложение автоматически публикует `push.apns.register` после получения токена APNs, подключения сеанса оператора и успешной регистрации у релея.
5. После этого `push.test`, пробуждения при переподключении и уведомления о пробуждении могут использовать сохранённую регистрацию через релей.

Примечание о совместимости:

- `OPENCLAW_APNS_RELAY_BASE_URL` по-прежнему работает как временное переопределение переменной окружения для шлюза.

## Процесс аутентификации и установления доверия

Релей существует для обеспечения двух ограничений, которые прямое использование APNs на шлюзе не может обеспечить для официальных сборок iOS:

- Только подлинные сборки OpenClaw для iOS, распространяемые через Apple, могут использовать размещённый релей.
- Шлюз может отправлять push-уведомления через релей только для устройств iOS, сопряжённых с этим конкретным шлюзом.

Пошагово:

1. `Приложение для iOS -> шлюз`
   - Приложение сначала сопрягается со шлюзом через обычный процесс аутентификации шлюза.
   - Это даёт приложению аутентифицированный сеанс узла, а также аутентифицированный сеанс оператора.
   - Сеанс оператора используется для вызова `gateway.identity.get`.

2. `Приложение для iOS -> релей`
   - Приложение вызывает конечные точки регистрации у релея через HTTPS.
   - Регистрация включает доказательство App Attest и квитанцию приложения.
   - Релей проверяет идентификатор пакета, доказательство App Attest и квитанцию Apple, требуя официального/производственного пути распространения.
   - Именно это блокирует использование размещённого релея локальными сборками Xcode/для разработки. Локальная сборка может быть подписана, но она не удовлетворяет требованиям официального доказательства распространения Apple, ожидаемого релеем.

3. `Делегирование идентичности шлюза`
   - Перед регистрацией у релея приложение получает идентификатор сопряжённого шлюза из `gateway.identity.get`.
   - Приложение включает этот идентификатор шлюза в полезную нагрузку регистрации у релея.
   - Релей возвращает дескриптор релея и грант на отправку, ограниченный областью регистрации, которые делегированы этому идентификатору шлюза.

4. `Шлюз -> релей`
   - Шлюз сохраняет дескриптор релея и грант на отправку из `push.apns.register`.
   - При `push.test`, пробуждениях при переподключении и уведомлениях о пробуждении шлюз подписывает запрос на отправку с использованием собственной идентичности устройства.
   - Релей проверяет как сохранённый грант на отправку, так и подпись шлюза на соответствие делегированному идентификатору шлюза из регистрации.
   - Другой шлюз не может повторно использовать эту сохранённую регистрацию, даже если он каким-то образом получит дескриптор.

5. `Релей -> APNs`
   - Релею принадлежат производственные учётные данные APNs и необработанный токен APNs для официальной сборки.
   - Шлюз никогда не хранит необработанный токен APNs для официальных сборок с поддержкой релея.
   - Релей отправляет окончательное push-уведомление в APNs от имени сопряжённого шлюза.

Почему была создана такая архитектура:

- Чтобы не хранить производственные учётные данные APNs на пользовательских шлюзах.
- Чтобы избежать хранения необработанных токенов APNs официальных сборок на шлюзе.
- Чтобы разрешить использование размещённого релея только для официальных сборок/сборок TestFlight OpenClaw.
- Чтобы предотвратить отправку уведомлений о пробуждении одним шлюзом на устройства iOS, принадлежащие другому шлюзу.

Локальные/ручные сборки остаются на прямом использовании APNs. Если вы тестируете такие сборки без релея, шлюзу по-прежнему требуются прямые учётные данные APNs:

```bash
export OPENCLAW_APNS_TEAM_ID="TEAMID"
export OPENCLAW_APNS_KEY_ID="KEYID"
export OPENCLAW_APNS_PRIVATE_KEY_P8="$(cat /path/to/AuthKey_KEYID.p8)"
```

Это переменные окружения времени выполнения для хоста шлюза, а не настройки Fastlane. Файл `apps/ios/fastlane/.env` хранит только данные аутентификации App Store Connect / TestFlight, такие как `ASC_KEY_ID` и `ASC_ISSUER_ID`; он не настраивает прямую доставку APNs для локальных сборок iOS.

Рекомендуемое хранилище для хоста шлюза:

```bash
mkdir -p ~/.openclaw/credentials/apns
chmod 700 ~/.openclaw/credentials/apns
mv /path/to/AuthKey_KEYID.p8 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
chmod 600 ~/.openclaw/credentials/apns/AuthKey_KEYID.p8
export OPENCLAW_APNS_PRIVATE_KEY_PATH="$HOME/.openclaw/credentials/apns/AuthKey_KEYID.p8"
```

Не фиксируйте файл `.p8` и не помещайте его в каталог проверки репозитория.

## Пути обнаружения

### Bonjour (локальная сеть)

Приложение для iOS просматривает `_openclaw-gw._tcp` в `local.` и, при настройке, в том же домене обнаружения DNS-SD для широкой зоны. Шлюзы в той же локальной сети отображаются автоматически из `local.`; для обнаружения в разных сетях можно использовать настроенный домен для широкой зоны без изменения типа маяка.
