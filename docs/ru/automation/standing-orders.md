# Постоянные распоряжения

Постоянные распоряжения предоставляют вашему агенту **постоянные полномочия** для выполнения определённых программ. Вместо того чтобы каждый раз давать отдельные инструкции по задачам, вы определяете программы с чётким объёмом работ, триггерами и правилами эскалации — и агент выполняет их автономно в рамках этих границ.

Это разница между тем, чтобы каждую пятницу говорить помощнику "отправь еженедельный отчёт", и предоставлением постоянных полномочий: "Ты отвечаешь за еженедельный отчёт. Составляй его каждую пятницу, отправляй и обращайся за помощью только в случае, если что-то выглядит неправильно".

## Зачем нужны постоянные распоряжения?

**Без постоянных распоряжений:**

- Вам нужно давать агенту инструкции для каждой задачи
- Агент бездействует между запросами
- Рутинная работа забывается или откладывается
- Вы становитесь узким местом

**С постоянными распоряжениями:**

- Агент выполняет задачи автономно в заданных границах
- Рутинная работа выполняется по расписанию без дополнительных указаний
- Вы участвуете только в случае исключений и для утверждения действий
- Агент продуктивно использует время простоя

## Как это работает

Постоянные распоряжения определяются в файлах вашего [рабочего пространства агента](/concepts/agent-workspace). Рекомендуется включать их непосредственно в файл `AGENTS.md` (который автоматически внедряется в каждую сессию), чтобы агент всегда имел к ним доступ. Для более крупных конфигураций вы можете разместить их в отдельном файле, например `standing-orders.md`, и ссылаться на него из `AGENTS.md`.

Каждая программа определяет:

1. **Объём работ** — что агенту разрешено делать
2. **Триггеры** — когда выполнять (по расписанию, событию или условию)
3. **Точки утверждения** — что требует одобрения человека перед выполнением
4. **Правила эскалации** — когда нужно остановиться и обратиться за помощью

Агент загружает эти инструкции при каждой сессии через файлы начальной загрузки рабочего пространства (см. [Рабочее пространство агента](/concepts/agent-workspace) для полного списка автоматически внедряемых файлов) и выполняет их, используя в сочетании с [cron-заданиями](/automation/cron-jobs) для соблюдения временных рамок.

<Tip>
Поместите постоянные распоряжения в `AGENTS.md`, чтобы гарантировать их загрузку при каждой сессии. Начальная загрузка рабочего пространства автоматически внедряет `AGENTS.md`, `SOUL.md`, `TOOLS.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`, `BOOTSTRAP.md` и `MEMORY.md` — но не произвольные файлы в подкаталогах.
</Tip>

## Структура постоянного распоряжения

```markdown
## Программа: Еженедельный отчёт о статусе

**Полномочия:** Собирать данные, формировать отчёт, отправлять заинтересованным сторонам
**Триггер:** Каждую пятницу в 16:00 (обеспечивается через cron-задание)
**Точка утверждения:** Не требуется для стандартных отчётов. Отмечать аномалии для проверки человеком.
**Эскалация:** Если источник данных недоступен или показатели выглядят необычно (>2σ от нормы)

### Шаги выполнения

1. Получить показатели из настроенных источников
2. Сравнить с показателями прошлой недели и целевыми значениями
3. Сформировать отчёт в Reports/weekly/YYYY-MM-DD.md
4. Отправить сводку через настроенное средство связи
5. Записать завершение в Agent/Logs/

### Что НЕ делать

- Не отправлять отчёты внешним сторонам
- Не изменять исходные данные
- Не пропускать отправку, если показатели плохие — сообщать точно
```

## Постоянные распоряжения + cron-задания

Постоянные распоряжения определяют, **что** агенту разрешено делать. [Cron-задания](/automation/cron-jobs) определяют, **когда** это происходит. Они работают вместе:

```
Постоянное распоряжение: "Ты отвечаешь за ежедневную сортировку входящих сообщений"
    ↓
Cron-задание (ежедневно в 8:00): "Выполнить сортировку входящих сообщений согласно постоянным распоряжениям"
    ↓
Агент: Читает постоянные распоряжения → выполняет шаги → сообщает о результатах
```

Запрос cron-задания должен ссылаться на постоянное распоряжение, а не дублировать его:

```bash
openclaw cron add \
  --name daily-inbox-triage \
  --cron "0 8 * * 1-5" \
  --tz America/New_York \
  --timeout-seconds 300 \
  --announce \
  --channel bluebubbles \
  --to "+1XXXXXXXXXX" \
  --message "Выполнить ежедневную сортировку входящих сообщений согласно постоянным распоряжениям. Проверить почту на наличие новых оповещений. Разобрать, классифицировать и сохранить каждый элемент. Сообщить сводку владельцу. Эскалировать неизвестные элементы."
```

## Примеры

### Пример 1: Контент и социальные сети (еженедельный цикл)

```markdown
## Программа: Контент и социальные сети

**Полномочия:** Составлять контент, планировать публикации, формировать отчёты об вовлечённости
**Точка утверждения:** Все публикации требуют проверки владельцем в течение первых 30 дней, затем постоянное утверждение
**Триггер:** Еженедельный цикл (понедельник — проверка → середина недели — черновики → пятница — краткий отчёт)

### Еженедельный цикл

- **Понедельник:** Проверить показатели платформ и вовлечённость аудитории
- **Вторник–четверг:** Составить публикации для соцсетей, создать контент для блога
- **Пятница:** Сформировать еженедельный маркетинговый отчёт → отправить владельцу

### Правила для контента

- Стиль должен соответствовать бренду (см. SOUL.md или руководство по стилю бренда)
- Никогда не указывать, что контент создан ИИ, в публичном контенте
- Включать показатели, когда они доступны
- Сосредоточиться на ценности для аудитории, а не на саморекламе
```

### Пример 2: Финансовые операции (срабатывание по событию)

```markdown
## Программа: Обработка финансовых данных

**Полномочия:** Обрабатывать данные о транзакциях, формировать отчёты, отправлять сводки
**Точка утверждения:** Не требуется для анализа. Рекомендации требуют утверждения владельцем.
**Триггер:** Обнаружение нового файла с данными ИЛИ ежемесячный цикл по расписанию

### Когда поступают новые данные

1. Обнаружить новый файл в указанной входной директории
2. Разобрать и классифицировать все транзакции
3. Сравнить с целевыми показателями бюджета
4. Отметить: необычные элементы, превышение порогов, новые регулярные платежи
5. Сформировать отчёт в указанной выходной директории
6. Отправить сводку владельцу через настроенное средство связи

### Правила эскалации

- Один элемент > 500 долларов: немедленное оповещение
- Категория превышает бюджет на 20 %: отметить в отчёте
- Нераспознанная транзакция: запросить у владельца классификацию
- Ошибка обработки после 2 попыток: сообщить об ошибке, не делать предположений
```

### Пример 3: Мониторинг и оповещения (непрерывный режим)

```markdown
## Программа: Мониторинг системы

**Полномочия:** Проверять работоспособность системы, перезапускать сервисы, отправлять оповещения
**Точка утверждения:** Перезапускать сервисы автоматически. Эскалировать, если перезапуск не удаётся дважды.
**Триггер:** При каждом цикле проверки работоспособности

### Проверки

- Ответные сигналы конечных точек работоспособности сервисов
- Объём дискового пространства выше порогового значения
- Незавершённые задачи не устарели (>24 часов)
- Каналы доставки работают

### Матрица ответов

| Условие | Действие | Эскалация? |
| --- | --- | --- |
| Сервис не работает | Перезапустить автоматически | Только если перезапуск не удаётся дважды |
| Объём дискового пространства < 10 % | Оповестить владельца | Да |
| Устаревшая задача > 24 часов | Напомнить владельцу | Нет |
| Канал недоступен | Записать и повторить попытку в следующем цикле | Если канал недоступен > 2 часов |
```

## Паттерн "Выполнить-Проверить-Сообщить"

Постоянные распоряжения работают лучше всего в сочетании со строгой дисциплиной выполнения. Каждая задача в постоянном распоряжении должна следовать этому циклу:

1. **Выполнить** — Сделать реальную работу (а не просто подтвердить получение инструкции)
2. **Проверить** — Убедиться, что результат верен (файл существует, сообщение доставлено, данные разобраны)
3. **Сообщить** — Сообщить владельцу, что было сделано и что проверено

```markdown
### Правила выполнения

- Каждая задача следует паттерну "Выполнить-Проверить-Сообщить". Без исключений.
- "Я сделаю это" — это не выполнение. Сделайте это, затем сообщите.
- "Сделано" без проверки недопустимо. Докажите это.
- Если выполнение не удаётся: повторите попытку один раз с изменённым подходом.
- Если всё ещё не удаётся: сообщите об ошибке с диагностикой. Никогда не терпите неудачу молча.
- Никогда не повторяйте попытки бесконечно — максимум 3 попытки, затем эскалация.
```

Этот паттерн предотвращает наиболее распространённый сбой агента: подтверждение получения задачи без её выполнения.
