# Интерфейс управления (браузер)

Интерфейс управления — это небольшое одностраничное приложение на **Vite + Lit**, которое предоставляется Gateway:

- по умолчанию: `http://<host>:18789/`
- необязательный префикс: задайте `gateway.controlUi.basePath` (например, `/openclaw`)

Он взаимодействует **напрямую с WebSocket Gateway** на том же порту.

## Быстрый запуск (локально)

Если Gateway запущен на том же компьютере, откройте:

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (или [http://localhost:18789/](http://localhost:18789/))

Если страница не загружается, сначала запустите Gateway: `openclaw gateway`.

Аутентификация выполняется во время рукопожатия WebSocket с использованием:

- `connect.params.auth.token`
- `connect.params.auth.password`
- заголовков идентификации Tailscale Serve, если `gateway.auth.allowTailscale: true`
- заголовков идентификации доверенного прокси, если `gateway.auth.mode: "trusted-proxy"`

Панель настроек панели управления сохраняет токен для текущей сессии вкладки браузера и выбранного URL Gateway; пароли не сохраняются. При первом подключении обычно генерируется токен Gateway для аутентификации с общим секретом, но также возможна аутентификация по паролю, если `gateway.auth.mode` имеет значение `"password"`.

## Сопряжение устройства (первое подключение)

Когда вы подключаетесь к интерфейсу управления с нового браузера или устройства, Gateway требует **однократного подтверждения сопряжения** — даже если вы находитесь в том же Tailnet с `gateway.auth.allowTailscale: true`. Это мера безопасности для предотвращения несанкционированного доступа.

**Что вы увидите:** "отключено (1008): требуется сопряжение"

**Чтобы подтвердить устройство:**

```bash
# Вывести список ожидающих запросов
openclaw devices list

# Подтвердить по ID запроса
openclaw devices approve <requestId>
```

Если браузер повторяет попытку сопряжения с изменёнными данными аутентификации (роль/области/открытый ключ), предыдущий ожидающий запрос отменяется и создаётся новый `requestId`. Перед подтверждением выполните команду `openclaw devices list` ещё раз.

После подтверждения устройство запоминается и не потребует повторного подтверждения, пока вы не отозвете его с помощью `openclaw devices revoke --device <id> --role <role>`. См. [CLI устройств](/cli/devices) для ротации и отзыва токенов.

**Примечания:**

- Прямые локальные подключения через loopback (`127.0.0.1` / `localhost`) подтверждаются автоматически.
- Подключения через браузер к Tailnet и LAN всё равно требуют явного подтверждения, даже если они выполняются с того же компьютера.
- Каждый профиль браузера генерирует уникальный ID устройства, поэтому при смене браузера или очистке данных браузера потребуется повторное сопряжение.

## Поддержка языков

Интерфейс управления может локализоваться при первой загрузке на основе локали вашего браузера. Чтобы изменить её позже, откройте **Обзор -> Доступ к Gateway -> Язык**. Выбор локали находится в карточке "Доступ к Gateway", а не в разделе "Внешний вид".

- Поддерживаемые локали: `en`, `zh-CN`, `zh-TW`, `pt-BR`, `de`, `es`, `ja-JP`, `ko`, `fr`, `tr`, `uk`, `id`, `pl`
- Переводы на языки, отличные от английского, загружаются в браузере по требованию.
- Выбранная локаль сохраняется в хранилище браузера и используется при последующих посещениях.
- Отсутствующие ключи перевода возвращаются к английскому языку.

## Что можно делать (на данный момент)

- Общаться с моделью через Gateway WS (`chat.history`, `chat.send`, `chat.abort`, `chat.inject`)
- Потоковая передача вызовов инструментов + карточки вывода инструментов в чате в реальном времени (события агента)
- Каналы: встроенные и связанные/внешние плагины, статус каналов, вход через QR-код и конфигурация для каждого канала (`channels.status`, `web.login.*`, `config.patch`)
- Экземпляры: список присутствия + обновление (`system-presence`)
- Сессии: список + переопределение модели/мышления/быстрого/подробного/рассуждения для каждой сессии (`sessions.list`, `sessions.patch`)
- "Сны": статус "сна", включение/выключение, чтение "Дневника снов" (`doctor.memory.status`, `doctor.memory.dreamDiary`, `config.patch`)
- Задания cron: список/добавление/редактирование/запуск/включение/выключение + история запусков (`cron.*`)
- Навыки: статус, включение/выключение, установка, обновление API-ключей (`skills.*`)
- Узлы: список + возможности (`node.list`)
- Подтверждения exec: редактирование списков разрешённых Gateway или узлов + политика запросов для `exec host=gateway/node` (`exec.approvals.*`)
- Конфигурация: просмотр/редактирование `~/.openclaw/openclaw.json` (`config.get`, `config.set`)
- Конфигурация: применение + перезапуск с проверкой (`config.apply`) и активация последней активной сессии
- Записи конфигурации включают защиту с базовым хэшем, чтобы предотвратить перезапись при одновременных изменениях
- Записи конфигурации (`config.set`/`config.apply`/`config.patch`) также предварительно проверяют разрешение активных SecretRef для ссылок в отправляемой полезной нагрузке конфигурации; неразрешённые активные отправленные ссылки отклоняются до записи
- Схема конфигурации + рендеринг формы (`config.schema` / `config.schema.lookup`, включая заголовок поля / описание, подсказки UI, сводки для непосредственных дочерних элементов, метаданные документации для вложенных объектов/подстановочных знаков/массивов/композиций, а также схемы плагинов и каналов, если они доступны); Редактор необработанного JSON доступен только тогда, когда снимок допускает безопасное двустороннее преобразование
- Если снимок не может безопасно пройти двустороннее преобразование необработанного текста, интерфейс управления принудительно переводит в режим формы и отключает режим необработанного текста для этого снимка
- Структурированные значения объектов SecretRef отображаются только для чтения в текстовых полях формы, чтобы предотвратить случайное повреждение объекта при преобразовании в строку
- Отладка: снимки состояния/работоспособности/моделей + журнал событий + ручные вызовы RPC (`status`, `health`, `models.list`)
- Журналы: потоковая передача файлов журналов Gateway с фильтрацией/экспортом (`logs.tail`)
- Обновление: запуск обновления пакета/git + перезапуск (`update.run`) с отчётом о перезапуске

Примечания к панели заданий cron:

- Для изолированных заданий доставка по умолчанию — сводка анонса. Вы можете отключить её, если хотите выполнять только внутренние запуски.
- Поля канала/цели появляются, когда выбран анонс.
- Режим вебхука использует `delivery.mode = "webhook"` с `delivery.to`, установленным на допустимый URL вебхука HTTP(S).
- Для заданий основной сессии доступны режимы доставки вебхука и "нет".
- Расширенные элементы управления редактированием включают удаление после запуска, очистку переопределения агента, параметры точного/сдвинутого cron, переопределение модели/мышления агента и переключатели доставки с наилучшим усилием.
- Проверка формы выполняется встроенно с ошибками на уровне полей; недопустимые значения отключают кнопку сохранения до исправления.
- Задайте `cron.webhookToken`, чтобы отправить выделенный токен носителя; если он опущен, вебхук отправляется без заголовка аутентификации.
- Устаревший запасной вариант: сохранённые устаревшие задания с `notify: true` всё ещё могут использовать `cron.webhook` до миграции.

## Поведение чата

- `chat.send` **не блокирует**: он сразу подтверждает с `{ runId, status: "started" }`, а ответ передаётся через события `chat`.
- Повторная отправка с тем же `idempotencyKey` возвращает `{ status: "in_flight" }` во время выполнения и `{ status: "ok" }` после завершения.
- Ответы `chat.history` ограничены по размеру для безопасности UI. Если записи транскрипции слишком велики, Gateway может обрезать длинные текстовые поля, опустить тяжёлые блоки метаданных и заменить oversized сообщения заполнителем (`[chat.history omitted: message too large]`).
- `chat.history` также удаляет теги директив, отображаемые только для отображения, из видимого текста помощника (например, `[[reply_to_*]]` и `[[audio_as_voice]]`), XML-полезные нагрузки вызовов инструментов в виде обычного текста (включая `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` и усечённые блоки вызовов инструментов), а также просочившиеся токены управления моделью ASCII/полной ширины и опускает записи помощника, весь видимый текст которых — это только точный тихий токен `NO_REPLY` / `no_reply`.
