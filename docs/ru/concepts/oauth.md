# OAuth

OpenClaw поддерживает "аутентификацию по подписке" через OAuth для провайдеров, которые её предоставляют (в частности, **OpenAI Codex (ChatGPT OAuth)**). Для Anthropic сейчас существует следующее разделение:

- **Ключ API Anthropic**: стандартная тарификация через API Anthropic
- **Anthropic Claude CLI / аутентификация по подписке внутри OpenClaw**: сотрудники Anthropic сообщили, что такое использование снова разрешено

OpenAI Codex OAuth явно поддерживается для использования во внешних инструментах, таких как OpenClaw. На этой странице объясняется:

Для использования Anthropic в продакшене рекомендуется более безопасный путь — аутентификация через ключ API.

- как работает **обмен токенами** в OAuth (PKCE);
- где **хранятся** токены (и почему);
- как работать с **несколькими учётными записями** (профили + переопределения для каждой сессии).

OpenClaw также поддерживает **плагины провайдеров**, которые реализуют собственные потоки OAuth или работы с ключами API. Запустите их с помощью команды:

```bash
openclaw models auth login --provider <id>
```

## "Приёмник токенов" (зачем он нужен)

Провайдеры OAuth обычно выдают **новый токен обновления** во время процессов входа в систему или обновления. Некоторые провайдеры (или клиенты OAuth) могут аннулировать старые токены обновления, когда для того же пользователя/приложения выдаётся новый.

Практический симптом:

- вы входите в систему через OpenClaw _и_ через Claude Code / Codex CLI → позже одна из систем случайно "выходит" из системы

Чтобы снизить вероятность такой ситуации, OpenClaw рассматривает `auth-profiles.json` как **"приёмник токенов"**:

- среда выполнения считывает учётные данные из **одного места**;
- мы можем хранить несколько профилей и маршрутизировать их детерминированно;
- когда учётные данные повторно используются из внешнего CLI (например, Codex CLI), OpenClaw копирует их с указанием источника и повторно считывает этот внешний источник вместо того, чтобы обновлять сам токен обновления.

## Хранение (где находятся токены)

Секретные данные хранятся **для каждого агента**:

- Профили аутентификации (OAuth + ключи API + необязательные ссылки на уровне значений): `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`
- Файл для обеспечения обратной совместимости: `~/.openclaw/agents/<agentId>/agent/auth.json` (статические записи `api_key` удаляются при обнаружении)

Файл только для импорта устаревших данных (всё ещё поддерживается, но не является основным хранилищем):

- `~/.openclaw/credentials/oauth.json` (импортируется в `auth-profiles.json` при первом использовании)

Все вышеперечисленные пути также учитывают переменную `$OPENCLAW_STATE_DIR` (переопределение каталога состояния). Полная справка: [/gateway/configuration](/gateway/configuration-reference#auth-storage)

Информацию о статических ссылках на секреты и поведении активации моментальных снимков во время выполнения см. в разделе [Управление секретами](/gateway/secrets).

## Миграция Anthropic Claude CLI

OpenClaw снова поддерживает повторное использование Anthropic Claude CLI. Если у вас уже есть локальный вход в Claude на хосте, процесс настройки/конфигурации может напрямую использовать его.

## Обмен OAuth (как работает вход в систему)

Интерактивные потоки входа в систему в OpenClaw реализованы в `@mariozechner/pi-ai` и интегрированы в мастера/команды.

### Токен настройки Anthropic

Схема потока:

1. запустите токен настройки Anthropic или вставьте токен из OpenClaw;
2. OpenClaw сохраняет полученные учётные данные Anthropic в профиле аутентификации;
3. выбор модели остаётся на `anthropic/...`;
4. существующие профили аутентификации Anthropic остаются доступными для отката/управления порядком.

### OpenAI Codex (ChatGPT OAuth)

OpenAI Codex OAuth явно поддерживается для использования вне Codex CLI, включая рабочие процессы OpenClaw.

Схема потока (PKCE):

1. сгенерируйте верификатор/запрос PKCE + случайное значение `state`;
2. откройте `https://auth.openai.com/oauth/authorize?...`;
3. попытайтесь перехватить обратный вызов на `http://127.0.0.1:1455/auth/callback`;
4. если обратный вызов не может быть привязан (или вы работаете удалённо/без графического интерфейса), вставьте URL перенаправления/код;
5. выполните обмен на `https://auth.openai.com/oauth/token`;
6. извлеките `accountId` из токена доступа и сохраните `{ access, refresh, expires, accountId }`.

Путь мастера: `openclaw onboard` → выбор аутентификации `openai-codex`.

## Обновление + истечение срока действия

Профили хранят отметку времени `expires`.

Во время выполнения:

- если `expires` находится в будущем → используйте сохранённый токен доступа;
- если срок действия истёк → обновите (под блокировкой файла) и перезапишите сохранённые учётные данные;
- исключение: повторно используемые учётные данные внешнего CLI управляются извне; OpenClaw повторно считывает хранилище аутентификации CLI и никогда не использует скопированный токен обновления самостоятельно.

Процесс обновления выполняется автоматически; как правило, вам не нужно управлять токенами вручную.

## Несколько учётных записей (профили) + маршрутизация

Существуют два шаблона:

### 1) Предпочтительный: отдельные агенты

Если вы хотите, чтобы "личное" и "рабочее" никогда не пересекались, используйте изолированные агенты (отдельные сессии + учётные данные + рабочее пространство):

```bash
openclaw agents add work
openclaw agents add personal
```

Затем настройте аутентификацию для каждого агента (с помощью мастера) и маршрутизируйте чаты к нужному агенту.

### 2) Расширенный: несколько профилей в одном агенте

`auth-profiles.json` поддерживает несколько идентификаторов профилей для одного провайдера.

Выберите, какой профиль будет использоваться:

- глобально через порядок конфигурации (`auth.order`);
- для каждой сессии через `/model ...@<profileId>`.

Пример (переопределение для сессии):

- `/model Opus@anthropic:work`

Как узнать, какие идентификаторы профилей существуют:

- `openclaw channels list --json` (показывает `auth[]`)
