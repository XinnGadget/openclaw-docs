# Управление сессиями и компакция (глубокое погружение)

В этом документе объясняется, как OpenClaw управляет сессиями от начала до конца:

- **Маршрутизация сессий** (как входящие сообщения сопоставляются с `sessionKey`)
- **Хранилище сессий** (`sessions.json`) и что оно отслеживает
- **Сохранение транскриптов** (`*.jsonl`) и их структура
- **Очистка транскриптов** (исправления, специфичные для провайдера, перед запусками)
- **Ограничения контекста** (окно контекста против отслеживаемых токенов)
- **Компакция** (ручная + автокомпакция) и где можно подключить предварительную работу перед компакцией
- **Тихая уборка** (например, запись в память, которая не должна приводить к выводу, видимому пользователю)

Если вам сначала нужен обзор более высокого уровня, начните с:

## Источник истины: шлюз

OpenClaw спроектирован вокруг единого **процесса шлюза (Gateway)**, который управляет состоянием сессий.

- Пользовательские интерфейсы (приложение для macOS, веб-интерфейс управления, TUI) должны запрашивать у шлюза списки сессий и количество токенов.
- В удалённом режиме файлы сессий находятся на удалённом хосте; проверка локальных файлов на Mac не отразит то, что использует шлюз.

---

## Два уровня сохранения данных

OpenClaw сохраняет сессии на двух уровнях:

1. **Хранилище сессий (`sessions.json`)**
   - Карта "ключ-значение": `sessionKey -> SessionEntry`
   - Небольшой размер, изменяемый, безопасно редактировать (или удалять записи)
   - Отслеживает метаданные сессии (текущий идентификатор сессии, последнее действие, переключатели, счётчики токенов и т. д.)

2. **Транскрипт (`<sessionId>.jsonl`)**
   - Транскрипт с возможностью только добавления, имеющий древовидную структуру (записи имеют `id` + `parentId`)
   - Хранит фактическую беседу, вызовы инструментов и сводки компакции
   - Используется для восстановления контекста модели для будущих шагов

---

## Расположение файлов на диске

Для каждого агента на хосте шлюза:

- Хранилище: `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- Транскрипты: `~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl`
  - Сессии темы Telegram: `.../<sessionId>-topic-<threadId>.jsonl`

OpenClaw определяет эти пути через `src/config/sessions.ts`.

## Обслуживание хранилища и управление диском

Сохранение сессий имеет автоматические средства обслуживания (`session.maintenance`) для `sessions.json` и артефактов транскриптов:

- `mode`: `warn` (по умолчанию) или `enforce`
- `pruneAfter`: порог возраста устаревших записей (по умолчанию `30d`)
- `maxEntries`: ограничение количества записей в `sessions.json` (по умолчанию `500`)
- `rotateBytes`: ротация `sessions.json` при превышении размера (по умолчанию `10mb`)
- `resetArchiveRetention`: срок хранения архивов транскриптов `*.reset.<timestamp>` (по умолчанию: такой же, как `pruneAfter`; `false` отключает очистку)
- `maxDiskBytes`: необязательный лимит дискового пространства для директории сессий
- `highWaterBytes`: необязательный целевой объём после очистки (по умолчанию `80%` от `maxDiskBytes`)

Порядок принудительной очистки дискового пространства (`mode: "enforce"`):

1. Сначала удаляются самые старые архивные или сиротские артефакты транскриптов.
2. Если объём по-прежнему превышает целевой, удаляются самые старые записи сессий и соответствующие файлы транскриптов.
3. Процесс продолжается до тех пор, пока использование не станет равным или ниже `highWaterBytes`.

В режиме `mode: "warn"` OpenClaw сообщает о потенциальных удалениях, но не изменяет хранилище/файлы.

Запуск обслуживания по требованию:

```bash
openclaw sessions cleanup --dry-run
openclaw sessions cleanup --enforce
```

## Сессии cron и журналы запусков

Изолированные запуски cron также создают записи сессий/транскриптов, и для них предусмотрены отдельные средства управления хранением:

- `cron.sessionRetention` (по умолчанию `24h`) удаляет старые изолированные сессии запусков cron из хранилища сессий (`false` отключает).
- `cron.runLog.maxBytes` + `cron.runLog.keepLines` удаляют файлы `~/.openclaw/cron/runs/<jobId>.jsonl` (по умолчанию: `2_000_000` байт и `2000` строк).

## Ключи сессий (`sessionKey`)

`sessionKey` определяет, в каком "контейнере беседы" вы находитесь (маршрутизация + изоляция).

Распространённые шаблоны:

- Основной/прямой чат (для каждого агента): `agent:<agentId>:<mainKey>` (по умолчанию `main`)
- Группа: `agent:<agentId>:<channel>:group:<id>`
- Комната/канал (Discord/Slack): `agent:<agentId>:<channel>:channel:<id>` или `...:room:<id>`
- Cron: `cron:<job.id>`
- Webhook: `hook:<uuid>` (если не переопределено)

Канонические правила описаны в [/concepts/session](/concepts/session).

---

## Идентификаторы сессий (`sessionId`)

Каждый `sessionKey` указывает на текущий `sessionId` (файл транскрипта, продолжающий беседу).

Основные правила:

- **Сброс** (`/new`, `/reset`) создаёт новый `sessionId` для этого `sessionKey`.
- **Ежедневный сброс** (по умолчанию в 4:00 утра по местному времени на хосте шлюза) создаёт новый `sessionId` при следующем сообщении после границы сброса.
- **Истечение времени бездействия** (`session.reset.idleMinutes` или устаревший `session.idleMinutes`) создаёт новый `sessionId`, когда сообщение поступает после истечения периода бездействия. Если настроены и ежедневный сброс, и истечение времени бездействия, приоритет имеет то, что истекает первым.
- **Защита от разветвления родительского потока** (`session.parentForkMaxTokens`, по умолчанию `100000`) пропускает разветвление родительского транскрипта, если родительская сессия уже слишком велика; новый поток начинается с чистого листа. Установите `0`, чтобы отключить.

Деталь реализации: решение принимается в `initSessionState()` в `src/auto-reply/reply/session.ts`.

## Схема хранилища сессий (`sessions.json`)

Тип значения в хранилище — `SessionEntry` в `src/config/sessions.ts`.

Ключевые поля (не исчерпывающий список):

- `sessionId`: текущий идентификатор транскрипта (имя файла выводится из этого значения, если не задано `sessionFile`)
- `updatedAt`: отметка времени последнего действия
- `sessionFile`: необязательный явный путь к транскрипту
- `chatType`: `direct | group | room` (помогает интерфейсам и политике отправки)
- `provider`, `subject`, `room`, `space`, `displayName`: метаданные для маркировки групп/каналов
- Переключатели:
  - `thinkingLevel`, `verboseLevel`, `reasoningLevel`, `elevatedLevel`
  - `sendPolicy` (переопределение для сессии)
- Выбор модели:
  - `providerOverride`, `modelOverride`, `authProfileOverride`
- Счётчики токенов (приблизительные / зависят от провайдера):
  - `inputTokens`, `outputTokens`, `totalTokens`, `contextTokens`
- `compactionCount`: сколько раз автокомпакция была выполнена для этого ключа сессии
- `memoryFlushAt`: отметка времени последнего сброса памяти перед компакцией
- `memoryFlushCompactionCount`: количество компакций на момент последнего сброса

Хранилище безопасно редактировать, но шлюз имеет приоритет: он может перезаписать или восстановить записи во время работы сессий.
