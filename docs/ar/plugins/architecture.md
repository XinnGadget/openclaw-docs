## نموذج القدرات العام

القدرات هي نموذج **الإضافات الأصلية** العام داخل OpenClaw. كل
إضافة OpenClaw أصلية تسجل نفسها مقابل نوع قدرة واحد أو أكثر:

| القدرة | طريقة التسجيل | أمثلة على الإضافات |
| ---------------------- | ------------------------------------------------ | ------------------------------------ |
| استدلال النص | `api.registerProvider(...)` | `openai`, `anthropic` |
| واجهة خلفية للاستدلال في CLI | `api.registerCliBackend(...)` | `openai`, `anthropic` |
| الكلام | `api.registerSpeechProvider(...)` | `elevenlabs`, `microsoft` |
| النسخ الفوري | `api.registerRealtimeTranscriptionProvider(...)` | `openai` |
| الصوت الفوري | `api.registerRealtimeVoiceProvider(...)` | `openai` |
| فهم الوسائط | `api.registerMediaUnderstandingProvider(...)` | `openai`, `google` |
| توليد الصور | `api.registerImageGenerationProvider(...)` | `openai`, `google`, `fal`, `minimax` |
| توليد الموسيقى | `api.registerMusicGenerationProvider(...)` | `google`, `minimax` |
| توليد الفيديو | `api.registerVideoGenerationProvider(...)` | `qwen` |
| جلب الويب | `api.registerWebFetchProvider(...)` | `firecrawl` |
| البحث في الويب | `api.registerWebSearchProvider(...)` | `google` |
| القنوات / المراسلة | `api.registerChannel(...)` | `msteams`, `matrix` |

الإضافة التي تسجل صفرًا من القدرات لكنها توفر خطافات أو أدوات أو
خدمات هي إضافة **قديمة تعتمد على الخطافات فقط**. لا يزال هذا النمط
مدعومًا بالكامل.

### موقف التوافق الخارجي

تم اعتماد نموذج القدرات في النواة ويُستخدم اليوم من قبل الإضافات
المجمعة/الأصلية، لكن توافق الإضافات الخارجية لا يزال يحتاج إلى معيار
أشد من مجرد "تم تصديره، إذن فهو ثابت".

الإرشادات الحالية:

- **الإضافات الخارجية الحالية:** حافظ على عمل التكاملات المعتمدة على الخطافات؛ واعتبر
  هذا خط الأساس للتوافق
- **الإضافات الجديدة المجمعة/الأصلية:** فضّل تسجيل القدرات الصريح بدلًا من
  التوغل الخاص بالمورد أو التصميمات الجديدة المعتمدة على الخطافات فقط
- **الإضافات الخارجية التي تتبنى تسجيل القدرات:** هذا مسموح، لكن تعامل مع
  الأسطح المساعدة الخاصة بالقدرات على أنها تتطور ما لم توضح الوثائق
  صراحةً أن العقد مستقر

القاعدة العملية:

- واجهات تسجيل القدرات هي الاتجاه المقصود
- تظل الخطافات القديمة المسار الأكثر أمانًا لتجنب الكسر للإضافات الخارجية خلال
  مرحلة الانتقال
- ليست كل المسارات الفرعية المساعدة المصدّرة متساوية؛ فضّل العقد الضيق الموثّق،
  لا الصادرات المساعدة العرضية

### أشكال الإضافات

يصنف OpenClaw كل إضافة محملة إلى شكل بناءً على سلوك التسجيل الفعلي الخاص بها
(وليس فقط على البيانات الوصفية الثابتة):

- **plain-capability** -- يسجل نوع قدرة واحدًا فقط (مثل
  إضافة مزود فقط مثل `mistral`)
- **hybrid-capability** -- يسجل عدة أنواع من القدرات (مثل
  `openai` الذي يملك استدلال النص والكلام وفهم الوسائط وتوليد
  الصور)
- **hook-only** -- يسجل خطافات فقط (مكتوبة الأنواع أو مخصصة)، من دون قدرات
  أو أدوات أو أوامر أو خدمات
- **non-capability** -- يسجل أدوات أو أوامر أو خدمات أو مسارات، ولكن من دون
  قدرات

استخدم `openclaw plugins inspect <id>` لرؤية شكل الإضافة وتفصيل
القدرات الخاصة بها. راجع [مرجع CLI](/cli/plugins#inspect) للتفاصيل.

### الخطافات القديمة

لا يزال الخطاف `before_agent_start` مدعومًا كمسار توافق للإضافات
المعتمدة على الخطافات فقط. لا تزال الإضافات القديمة المستخدمة فعليًا تعتمد عليه.

الاتجاه:

- الاستمرار في دعمه
- توثيقه على أنه قديم
- تفضيل `before_model_resolve` لأعمال تجاوز النموذج/المزود
- تفضيل `before_prompt_build` لأعمال تعديل الموجّه
- عدم إزالته إلا بعد انخفاض الاستخدام الفعلي وإثبات أمان الترحيل عبر تغطية التركيبات الاختبارية

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`, قد ترى
إحدى هذه التسميات:

| الإشارة | المعنى |
| -------------------------- | ------------------------------------------------------------ |
| **config valid** | يتم تحليل الإعدادات بشكل صحيح ويتم حل الإضافات |
| **compatibility advisory** | تستخدم الإضافة نمطًا مدعومًا ولكنه أقدم (مثل `hook-only`) |
| **legacy warning** | تستخدم الإضافة `before_agent_start`، وهو مهمل |
| **hard error** | الإعدادات غير صالحة أو فشل تحميل الإضافة |

لن يؤدي `hook-only` ولا `before_agent_start` إلى كسر إضافتك اليوم --
فـ `hook-only` مجرد توصية، و`before_agent_start` لا يؤدي إلا إلى تحذير. تظهر هذه
الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يتكون نظام الإضافات في OpenClaw من أربع طبقات:

1. **البيان + الاكتشاف**
   يعثر OpenClaw على الإضافات المرشحة من المسارات المهيأة وجذور مساحة العمل
   والجذور العامة للإضافات والامتدادات المجمعة. يقرأ الاكتشاف بيانات
   `openclaw.plugin.json` الأصلية بالإضافة إلى بيانات الحزم المدعومة أولًا.
2. **التمكين + التحقق**
   تقرر النواة ما إذا كانت الإضافة المكتشفة مُمكّنة أو معطلة أو محظورة أو
   محددة لفتحة حصرية مثل الذاكرة.
3. **التحميل وقت التشغيل**
   تُحمَّل إضافات OpenClaw الأصلية داخل العملية عبر jiti وتُسجَّل
   القدرات في سجل مركزي. وتُطبَّع الحزم المتوافقة إلى سجلات في
   السجل من دون استيراد شيفرة وقت التشغيل.
4. **استهلاك الأسطح**
   يقرأ باقي OpenClaw السجل لإظهار الأدوات والقنوات وإعداد المزود
   والخطافات ومسارات HTTP وأوامر CLI والخدمات.

وبالنسبة إلى CLI الخاص بالإضافات تحديدًا، يُقسَّم اكتشاف الأمر الجذر إلى مرحلتين:

- تأتي بيانات التحليل الوصفية من `registerCli(..., { descriptors: [...] })`
- يمكن أن تبقى وحدة CLI الحقيقية الخاصة بالإضافة كسولة وتُسجَّل عند أول استدعاء

هذا يبقي شيفرة CLI المملوكة للإضافة داخل الإضافة مع السماح لـ OpenClaw في
الوقت نفسه بحجز أسماء الأوامر الجذرية قبل التحليل.

الحد التصميمي المهم:

- يجب أن يعمل الاكتشاف والتحقق من الإعدادات انطلاقًا من **بيانات البيان/المخطط**
  من دون تنفيذ شيفرة الإضافة
- يأتي سلوك وقت التشغيل الأصلي من مسار `register(api)` في وحدة الإضافة

يسمح هذا الفصل لـ OpenClaw بالتحقق من الإعدادات وشرح الإضافات
المفقودة/المعطلة وبناء تلميحات الواجهة/المخطط قبل تنشيط وقت التشغيل الكامل.

### إضافات القنوات وأداة الرسائل المشتركة

لا تحتاج إضافات القنوات إلى تسجيل أداة إرسال/تعديل/تفاعل منفصلة من أجل
إجراءات الدردشة العادية. يحتفظ OpenClaw بأداة `message` مشتركة واحدة في
النواة، بينما تملك إضافات القنوات الاكتشاف والتنفيذ الخاصين بالقناة خلفها.

الحد الحالي هو:

- النواة تملك مضيف أداة `message` المشتركة، وربط الموجّه، ومسك الدفاتر الخاصة
  بالجلسات/المحادثات، وتوزيع التنفيذ
- إضافات القنوات تملك اكتشاف الإجراءات المقيّدة، واكتشاف القدرات،
  وأي أجزاء من المخطط خاصة بالقناة
- إضافات القنوات تملك قواعد المحادثة الخاصة بالمزود للجلسة، مثل
  كيفية ترميز معرفات المحادثات لمعرفات الخيوط أو وراثتها من المحادثات الأم
- تنفذ إضافات القنوات الإجراء النهائي عبر محوّل الإجراءات الخاص بها

بالنسبة إلى إضافات القنوات، سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. تسمح مكالمة
الاكتشاف الموحدة هذه للإضافة بإرجاع إجراءاتها المرئية وقدراتها ومساهماتها
في المخطط معًا حتى لا تنحرف هذه الأجزاء عن بعضها.

تمرر النواة النطاق وقت التشغيل إلى خطوة الاكتشاف هذه. وتشمل الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- `requesterSenderId` الوارد الموثوق

وهذا مهم للإضافات الحساسة للسياق. يمكن للقناة إخفاء أو إظهار
إجراءات الرسائل بناءً على الحساب النشط أو الغرفة/الخيط/الرسالة الحالية أو
هوية الطالب الموثوقة، من دون ترميز فروع خاصة بالقناة داخل
أداة `message` الأساسية.

لهذا السبب تظل تغييرات التوجيه في المشغل المضمّن عملًا خاصًا بالإضافة: فالمشغل
مسؤول عن تمرير هوية الدردشة/الجلسة الحالية إلى حد الاكتشاف الخاص بالإضافة
حتى تكشف أداة `message` المشتركة السطح الصحيح المملوك للقناة للدور الحالي.

أما بالنسبة إلى مساعدات التنفيذ المملوكة للقنوات، فيجب على الإضافات المجمعة أن تُبقي
وقت تشغيل التنفيذ داخل وحدات الامتداد الخاصة بها. لم تعد النواة تملك أوقات تشغيل
إجراءات رسائل Discord أو Slack أو Telegram أو WhatsApp ضمن `src/agents/tools`.
ولا ننشر مسارات فرعية منفصلة من `plugin-sdk/*-action-runtime`، ويجب على
الإضافات المجمعة استيراد شيفرة وقت التشغيل المحلية الخاصة بها مباشرةً من
وحداتها المملوكة للامتداد.

وينطبق الحد نفسه عمومًا على الوصلات المسماة بحسب المزود في SDK: لا ينبغي
للنواة أن تستورد حاويات تسهيلية خاصة بقنوات Slack أو Discord أو Signal أو
WhatsApp أو امتدادات مشابهة. وإذا احتاجت النواة إلى سلوك ما، فإما أن تستهلك
حاوية `api.ts` / `runtime-api.ts` الخاصة بالإضافة المجمعة نفسها أو ترفع الحاجة
إلى قدرة عامة ضيقة في SDK المشترك.

وبالنسبة إلى الاستطلاعات تحديدًا، هناك مساران للتنفيذ:

- `outbound.sendPoll` هو خط الأساس المشترك للقنوات التي تناسب نموذج
  الاستطلاع العام
- `actions.handleAction("poll")` هو المسار المفضل لدلالات الاستطلاع الخاصة بالقناة
  أو لمعلمات الاستطلاع الإضافية

تؤجل النواة الآن تحليل الاستطلاع المشترك إلى ما بعد رفض توزيع استطلاع الإضافة
للإجراء، بحيث يمكن لمعالجات الاستطلاع المملوكة للإضافة قبول حقول
استطلاع خاصة بالقناة من دون أن يسبقها محلل الاستطلاع العام ويمنعها.

راجع [مسار التحميل](#load-pipeline) للتسلسل الكامل عند بدء التشغيل.

## نموذج ملكية القدرات

يتعامل OpenClaw مع الإضافة الأصلية على أنها حد ملكية لـ **شركة** أو
**ميزة**، لا مجرد حاوية لتكاملات غير مترابطة.

هذا يعني:

- يجب أن تملك إضافة الشركة عادةً كل الأسطح المواجهة لـ OpenClaw الخاصة بتلك الشركة
- يجب أن تملك إضافة الميزة عادةً سطح الميزة الكامل الذي تقدمه
- يجب أن تستهلك القنوات القدرات الأساسية المشتركة بدلًا من إعادة تنفيذ
  سلوك المزود بصورة مخصصة

أمثلة:

- الإضافة المجمعة `openai` تملك سلوك مزود نماذج OpenAI وكذلك سلوك
  OpenAI للكلام + الصوت الفوري + فهم الوسائط + توليد الصور
- الإضافة المجمعة `elevenlabs` تملك سلوك الكلام الخاص بـ ElevenLabs
- الإضافة المجمعة `microsoft` تملك سلوك الكلام الخاص بـ Microsoft
- الإضافة المجمعة `google` تملك سلوك مزود نماذج Google بالإضافة إلى
  سلوك Google لفهم الوسائط + توليد الصور + البحث في الويب
- الإضافة المجمعة `firecrawl` تملك سلوك جلب الويب الخاص بـ Firecrawl
- الإضافات المجمعة `minimax` و`mistral` و`moonshot` و`zai` تملك
  الواجهات الخلفية الخاصة بها لفهم الوسائط
- الإضافة `voice-call` هي إضافة ميزات: فهي تملك نقل المكالمات والأدوات
  وCLI والمسارات وجسر تدفق الوسائط من Twilio، لكنها تستهلك
  القدرات المشتركة للكلام + النسخ الفوري + الصوت الفوري بدلًا من
  استيراد إضافات المورد مباشرة

الحالة النهائية المقصودة هي:

- يعيش OpenAI في إضافة واحدة حتى لو امتد إلى نماذج النص والكلام والصور
  والفيديو مستقبلًا
- ويمكن لمورد آخر أن يفعل الشيء نفسه لمساحته الخاصة
- ولا تهتم القنوات بأي إضافة مورد تملك المزود؛ بل تستهلك عقد
  القدرة المشتركة التي تكشفها النواة

وهذا هو التمييز الأساسي:

- **plugin** = حد الملكية
- **capability** = عقد النواة الذي يمكن لعدة إضافات تنفيذه أو استهلاكه

لذلك إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فالسؤال الأول ليس
"أي مزود يجب أن يرمّز التعامل مع الفيديو؟" بل السؤال الأول هو: "ما
هو عقد قدرة الفيديو الأساسية؟" وبمجرد وجود هذا العقد، تستطيع إضافات المورد
التسجيل مقابله ويمكن لإضافات القنوات/الميزات استهلاكه.

إذا لم تكن القدرة موجودة بعد، فالخطوة الصحيحة غالبًا هي:

1. تعريف القدرة المفقودة في النواة
2. كشفها عبر واجهة برمجة الإضافة/وقت التشغيل بطريقة مكتوبة الأنواع
3. توصيل القنوات/الميزات بهذه القدرة
4. ترك إضافات المورد تسجل التنفيذات

يحافظ ذلك على وضوح الملكية ويجنب سلوك النواة الذي يعتمد على
مورد واحد أو مسار شيفرة خاص بإضافة واحدة.

### طبقات القدرات

استخدم هذا النموذج الذهني عند اتخاذ قرار بشأن موضع الشيفرة:

- **طبقة القدرة الأساسية**: التنسيق المشترك والسياسات وآليات التراجع وقواعد
  دمج الإعدادات ودلالات التسليم والعقود المكتوبة الأنواع
- **طبقة إضافة المورد**: واجهات API الخاصة بالمورد والمصادقة وكتالوجات النماذج
  وتوليف الكلام وتوليد الصور وواجهات الفيديو الخلفية مستقبلًا ونقاط استخدام الخدمة
- **طبقة إضافة القناة/الميزة**: تكاملات Slack/Discord/voice-call/etc.
  التي تستهلك القدرات الأساسية وتعرضها على سطح معين

على سبيل المثال، يتبع TTS هذا الشكل:

- تملك النواة سياسة TTS وقت الرد وترتيب التراجع والتفضيلات وتسليم القنوات
- وتملك `openai` و`elevenlabs` و`microsoft` تنفيذات التوليف
- وتستهلك `voice-call` مساعد وقت تشغيل TTS للهاتف

ويجب تفضيل هذا النمط نفسه للقدرات المستقبلية.

### مثال على إضافة شركة متعددة القدرات

يجب أن تبدو إضافة الشركة متماسكة من الخارج. إذا كان لدى OpenClaw
عقود مشتركة للنماذج والكلام والنسخ الفوري والصوت الفوري وفهم
الوسائط وتوليد الصور وتوليد الفيديو وجلب الويب والبحث في الويب،
فيمكن لمورد ما أن يملك كل أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // خطافات وقت التشغيل/كتالوج النماذج/المصادقة
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // إعدادات الكلام الخاصة بالمورد — نفّذ واجهة SpeechProviderPlugin مباشرةً
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // منطق بيانات الاعتماد + الجلب
      }),
    );
  },
};

export default plugin;
```

المهم ليس أسماء المساعدات الدقيقة. بل الشكل:

- إضافة واحدة تملك سطح المورد
- النواة تظل هي مالكة عقود القدرات
- القنوات وإضافات الميزات تستهلك مساعدات `api.runtime.*`، لا شيفرة المورد
- يمكن لاختبارات العقود التحقق من أن الإضافة سجلت القدرات التي
  تدّعي امتلاكها

### مثال على القدرة: فهم الفيديو

يعامل OpenClaw بالفعل فهم الصور/الصوت/الفيديو على أنه قدرة مشتركة واحدة.
وينطبق نموذج الملكية نفسه هنا:

1. تعرّف النواة عقد فهم الوسائط
2. تسجل إضافات المورد `describeImage` و`transcribeAudio` و
   `describeVideo` بحسب الاقتضاء
3. تستهلك القنوات وإضافات الميزات السلوك الأساسي المشترك بدلًا من
   الربط المباشر مع شيفرة المورد

وهذا يمنع ترسيخ افتراضات مورد واحد بشأن الفيديو داخل النواة. فالإضافة
تملك سطح المورد؛ والنواة تملك عقد القدرة وسلوك التراجع.

ويتبع توليد الفيديو التسلسل نفسه بالفعل: فالنواة تملك عقد
القدرة المكتوب الأنواع ومساعد وقت التشغيل، وتسجل إضافات المورد
تنفيذات `api.registerVideoGenerationProvider(...)` مقابله.

هل تحتاج إلى قائمة تحقق عملية للطرح؟ راجع
[دفتر وصفات القدرات](/ar/plugins/architecture).

## العقود والإنفاذ

سطح واجهة برمجة الإضافة مكتوب الأنواع ومركزي عمدًا في
`OpenClawPluginApi`. ويعرّف هذا العقد نقاط التسجيل المدعومة
ومساعدات وقت التشغيل التي يمكن أن تعتمد عليها الإضافة.

سبب أهمية ذلك:

- يحصل مؤلفو الإضافات على معيار داخلي ثابت واحد
- تستطيع النواة رفض الملكية المكررة، مثل تسجيل إضافتين للمعرف نفسه
  للمزود
- يمكن لبدء التشغيل إظهار تشخيصات عملية للتسجيلات غير الصحيحة
- يمكن لاختبارات العقود فرض ملكية الإضافات المجمعة ومنع الانحراف الصامت

هناك طبقتان من الإنفاذ:

1. **إنفاذ التسجيل وقت التشغيل**
   يتحقق سجل الإضافات من التسجيلات أثناء تحميل الإضافات. أمثلة:
   معرفات المزود المكررة، ومعرفات مزود الكلام المكررة، والتسجيلات
   غير الصحيحة تؤدي إلى تشخيصات للإضافات بدلًا من سلوك غير معرّف.
2. **اختبارات العقود**
   تُلتقط الإضافات المجمعة في سجلات العقود أثناء تشغيل الاختبارات حتى
   يتمكن OpenClaw من تأكيد الملكية بشكل صريح. ويُستخدم هذا اليوم مع
   مزودي النماذج، ومزودي الكلام، ومزودي البحث في الويب، وملكية التسجيل المجمعة.

الأثر العملي هو أن OpenClaw يعرف مسبقًا أي إضافة تملك أي
سطح. ويسمح ذلك للنواة والقنوات بالتأليف بسلاسة لأن الملكية
معلنة ومكتوبة الأنواع وقابلة للاختبار بدل أن تكون ضمنية.

### ما الذي يجب أن يكون ضمن العقد

عقود الإضافات الجيدة تكون:

- مكتوبة الأنواع
- صغيرة
- خاصة بالقدرات
- مملوكة للنواة
- قابلة لإعادة الاستخدام من عدة إضافات
- قابلة للاستهلاك من القنوات/الميزات من دون معرفة المورد

أما عقود الإضافات السيئة فهي:

- سياسة خاصة بالمورد مخفية داخل النواة
- منافذ هروب لمرة واحدة تتجاوز السجل
- شيفرة قناة تصل مباشرة إلى تنفيذ خاص بمورد
- كائنات وقت تشغيل مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

عند الشك، ارفع مستوى التجريد: عرّف القدرة أولًا، ثم دع الإضافات
تتصل بها.

## نموذج التنفيذ

تعمل إضافات OpenClaw الأصلية **داخل العملية** مع Gateway. وهي غير
معزولة. وتملك الإضافة الأصلية المحمّلة حد الثقة على مستوى العملية نفسه
الخاص بشيفرة النواة.

الآثار المترتبة:

- يمكن للإضافة الأصلية تسجيل أدوات ومعالجات شبكة وخطافات وخدمات
- يمكن لخلل في إضافة أصلية أن يعطل البوابة أو يزعزع استقرارها
- الإضافة الأصلية الخبيثة تعادل تنفيذ شيفرة عشوائية داخل عملية
  OpenClaw

تكون الحزم المتوافقة أكثر أمانًا افتراضيًا لأن OpenClaw يتعامل معها حاليًا
على أنها حزم بيانات وصفية/محتوى. وفي الإصدارات الحالية، يعني ذلك غالبًا
Skills المجمعة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة للإضافات غير المجمعة.
وتعامل مع إضافات مساحة العمل على أنها شيفرة وقت تطوير، لا إعدادات
إنتاجية افتراضية.

وبالنسبة إلى أسماء حزم مساحة العمل المجمعة، اجعل معرف الإضافة مرتكزًا في اسم npm:
`@openclaw/<id>` افتراضيًا، أو لاحقة نوعية معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding` عندما
تكشف الحزمة عمدًا دور إضافة أضيق.

ملاحظة ثقة مهمة:

- `plugins.allow` يثق في **معرفات الإضافات**، لا في مصدرها.
- إضافة مساحة عمل تحمل المعرف نفسه لإضافة مجمعة ستُظلّل عمدًا
  النسخة المجمعة عندما تكون إضافة مساحة العمل مُمكّنة/ضمن قائمة السماح.
- وهذا سلوك طبيعي ومفيد للتطوير المحلي واختبار التصحيحات والإصلاحات العاجلة.

## حد التصدير

يصدّر OpenClaw القدرات، لا تسهيلات التنفيذ.

أبقِ تسجيل القدرات عامًا. وقلّم صادرات المساعدات غير التعاقدية:

- المسارات الفرعية الخاصة بمساعدات الإضافات المجمعة
- المسارات الفرعية الخاصة بتوصيلات وقت التشغيل غير المقصودة كواجهة API عامة
- المساعدات التسهيلية الخاصة بالمورد
- مساعدات الإعداد/التهيئة التي تُعد تفاصيل تنفيذ

لا تزال بعض المسارات الفرعية المساعدة للإضافات المجمعة موجودة في خريطة
صادرات SDK المولدة من أجل التوافق وصيانة الإضافات المجمعة. ومن الأمثلة الحالية
`plugin-sdk/feishu`, `plugin-sdk/feishu-setup`, `plugin-sdk/zalo`,
`plugin-sdk/zalo-setup`, والعديد من وصلات `plugin-sdk/matrix*`. تعامل مع هذه
المسارات على أنها صادرات محجوزة تمثل تفاصيل تنفيذ، لا النمط الموصى به في SDK
للإضافات الخارجية الجديدة.

## مسار التحميل

عند بدء التشغيل، يقوم OpenClaw تقريبًا بما يلي:

1. اكتشاف جذور الإضافات المرشحة
2. قراءة بيانات الإضافات الأصلية أو الحزم المتوافقة وبيانات الحزم الوصفية
3. رفض المرشحين غير الآمنين
4. تطبيع إعدادات الإضافات (`plugins.enabled`, `allow`, `deny`, `entries`,
   `slots`, `load.paths`)
5. تقرير التمكين لكل مرشح
6. تحميل الوحدات الأصلية المفعلة عبر jiti
7. استدعاء خطافات `register(api)` الأصلية (أو `activate(api)` — وهو اسم مستعار قديم) وجمع التسجيلات في سجل الإضافات
8. كشف السجل إلى الأوامر/أسطح وقت التشغيل

<Note>
`activate` اسم مستعار قديم لـ `register` — يقوم المحمل بحل أيٍّ منهما موجود (`def.register ?? def.activate`) ويستدعيه في النقطة نفسها. تستخدم جميع الإضافات المجمعة `register`; ويفضَّل استخدام `register` في الإضافات الجديدة.
</Note>

تحدث بوابات الأمان **قبل** تنفيذ وقت التشغيل. ويُحظر المرشحون
عندما يفلت الإدخال من جذر الإضافة، أو يكون المسار قابلاً للكتابة من الجميع،
أو تبدو ملكية المسار مشبوهة بالنسبة إلى الإضافات غير المجمعة.

### سلوك البيان أولًا

البيان هو مصدر الحقيقة في مستوى التحكم. يستخدمه OpenClaw من أجل:

- تحديد الإضافة
- اكتشاف القنوات/Skills/مخطط الإعدادات أو قدرات الحزمة المعلنة
- التحقق من `plugins.entries.<id>.config`
- تعزيز تسميات/عناصر نائبة في Control UI
- إظهار بيانات التثبيت/الكتالوج الوصفية

أما بالنسبة إلى الإضافات الأصلية، فتمثل وحدة وقت التشغيل جزء مستوى البيانات. فهي
تسجل السلوك الفعلي مثل الخطافات أو الأدوات أو الأوامر أو تدفقات المزود.

### ما الذي يخزنه المحمّل مؤقتًا

يحتفظ OpenClaw بمخابئ قصيرة العمر داخل العملية لما يلي:

- نتائج الاكتشاف
- بيانات سجل البيان
- سجلات الإضافات المحملة

تقلل هذه المخابئ من الاندفاعات عند بدء التشغيل ومن كلفة الأوامر المتكررة. ومن
الآمن اعتبارها مخابئ أداء قصيرة العمر، لا طبقة استمرارية.

ملاحظة أداء:

- اضبط `OPENCLAW_DISABLE_PLUGIN_DISCOVERY_CACHE=1` أو
  `OPENCLAW_DISABLE_PLUGIN_MANIFEST_CACHE=1` لتعطيل هذه المخابئ.
- اضبط نوافذ المخابئ باستخدام `OPENCLAW_PLUGIN_DISCOVERY_CACHE_MS` و
  `OPENCLAW_PLUGIN_MANIFEST_CACHE_MS`.

## نموذج السجل

لا تقوم الإضافات المحملة بتعديل متغيرات النواة العامة العشوائية مباشرةً. بل
تسجّل نفسها في سجل إضافات مركزي.

يتتبع السجل ما يلي:

- سجلات الإضافات (الهوية، المصدر، الأصل، الحالة، التشخيصات)
- الأدوات
- الخطافات القديمة والخطافات المكتوبة الأنواع
- القنوات
- المزودون
- معالجات Gateway RPC
- مسارات HTTP
- مسجلو CLI
- الخدمات الخلفية
- الأوامر المملوكة للإضافات

ثم تقرأ ميزات النواة من هذا السجل بدلًا من التحدث إلى وحدات الإضافات
مباشرةً. وهذا يبقي التحميل في اتجاه واحد:

- وحدة الإضافة -> التسجيل في السجل
- وقت تشغيل النواة -> استهلاك السجل

ويهم هذا الفصل من أجل سهولة الصيانة. فهو يعني أن معظم أسطح النواة تحتاج
إلى نقطة تكامل واحدة فقط: "اقرأ السجل"، لا "أنشئ حالة خاصة لكل
وحدة إضافة".

## استدعاءات ربط المحادثات

يمكن للإضافات التي تربط محادثة أن تتفاعل عند حسم موافقة ما.

استخدم `api.onConversationBindingResolved(...)` لتلقي استدعاء
بعد الموافقة على طلب الربط أو رفضه:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // يوجد الآن ربط لهذه الإضافة + المحادثة.
        console.log(event.binding?.conversationId);
        return;
      }

      // تم رفض الطلب؛ امسح أي حالة محلية معلقة.
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

حقول حمولة الاستدعاء:

- `status`: `"approved"` أو `"denied"`
- `decision`: `"allow-once"` أو `"allow-always"` أو `"deny"`
- `binding`: الربط المحسوم للطلبات الموافق عليها
- `request`: ملخص الطلب الأصلي، وتلميح الفصل، ومعرف المرسِل، وبيانات
  المحادثة الوصفية

هذا الاستدعاء مخصص للإشعارات فقط. ولا يغيّر من يملك صلاحية ربط
المحادثة، ويعمل بعد انتهاء معالجة الموافقة الأساسية.

## خطافات وقت تشغيل المزود

أصبحت إضافات المزود الآن ذات طبقتين:

- بيانات البيان الوصفية: `providerAuthEnvVars` للبحث الرخيص عن مصادقة
  المزود في متغيرات البيئة قبل تحميل وقت التشغيل، و`providerAuthAliases`
  لمتغيرات المزود التي تشترك في المصادقة، و`channelEnvVars` للبحث
  الرخيص عن إعداد/مصادقة القناة عبر البيئة قبل تحميل وقت
  التشغيل، بالإضافة إلى `providerAuthChoices` لتسميات اختيار المصادقة/التهيئة
  الرخيصة وبيانات أعلام CLI الوصفية قبل تحميل وقت التشغيل
- خطافات وقت الإعداد: `catalog` / الاسم القديم `discovery` بالإضافة إلى `applyConfigDefaults`
- خطافات وقت التشغيل: `normalizeModelId`, `normalizeTransport`,
  `normalizeConfig`,
  `applyNativeStreamingUsageCompat`, `resolveConfigApiKey`,
  `resolveSyntheticAuth`, `resolveExternalAuthProfiles`,
  `shouldDeferSyntheticProfileAuth`,
  `resolveDynamicModel`, `prepareDynamicModel`, `normalizeResolvedModel`,
  `contributeResolvedModelCompat`, `capabilities`,
  `normalizeToolSchemas`, `inspectToolSchemas`,
  `resolveReasoningOutputMode`, `prepareExtraParams`, `createStreamFn`,
  `wrapStreamFn`, `resolveTransportTurnState`,
  `resolveWebSocketSessionPolicy`, `formatApiKey`, `refreshOAuth`,
  `buildAuthDoctorHint`, `matchesContextOverflowError`,
  `classifyFailoverReason`, `isCacheTtlEligible`,
  `buildMissingAuthMessage`, `suppressBuiltInModel`, `augmentModelCatalog`,
  `isBinaryThinking`, `supportsXHighThinking`,
  `resolveDefaultThinkingLevel`, `isModernModelRef`, `prepareRuntimeAuth`,
  `resolveUsageAuth`, `fetchUsageSnapshot`, `createEmbeddingProvider`,
  `buildReplayPolicy`,
  `sanitizeReplayHistory`, `validateReplayTurns`, `onModelSelected`

لا يزال OpenClaw يملك حلقة الوكيل العامة وآلية التراجع ومعالجة النصوص
وسياست الأدوات. وتمثل هذه الخطافات سطح الامتداد للسلوك الخاص بالمزود من دون
الحاجة إلى ناقل استدلال مخصص كامل.

استخدم `providerAuthEnvVars` في البيان عندما يملك المزود بيانات اعتماد
معتمدة على البيئة يجب أن تراها مسارات المصادقة/الحالة/منتقي النماذج العامة
من دون تحميل وقت تشغيل الإضافة. واستخدم `providerAuthAliases` في البيان عندما
يجب أن يعيد معرّف مزود واحد استخدام متغيرات البيئة وملفات تعريف المصادقة
والمصادقة المدعومة بالإعدادات وخيار تهيئة مفتاح API الخاص بمعرف مزود آخر. واستخدم
`providerAuthChoices` في البيان عندما تحتاج أسطح CLI الخاصة بالتهيئة/اختيار المصادقة
إلى معرفة معرف خيار المزود وتسميات المجموعة الخاصة به وربط المصادقة البسيط
المعتمد على علم واحد من دون تحميل وقت تشغيل المزود. واحتفظ بمتغيرات البيئة
الخاصة بوقت تشغيل المزود `envVars` من أجل التلميحات الموجهة للمشغل مثل
تسميات التهيئة أو متغيرات إعداد `client-id`/`client-secret` الخاصة بـ OAuth.

استخدم `channelEnvVars` في البيان عندما تكون للقناة مصادقة أو إعداد
مدفوع بالبيئة يجب أن تراه آليات التراجع العامة إلى shell-env أو فحوصات
الإعدادات/الحالة أو موجّهات الإعداد من دون تحميل وقت تشغيل القناة.

### ترتيب الخطافات واستخدامها

بالنسبة إلى إضافات النماذج/المزودين، يستدعي OpenClaw الخطافات بهذا الترتيب التقريبي.
ويمثل عمود "متى يُستخدم" دليل القرار السريع.

| #   | الخطاف | ما الذي يفعله | متى يُستخدم |
| --- | --------------------------------- | -------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | `catalog` | نشر إعدادات المزود في `models.providers` أثناء توليد `models.json` | عندما يملك المزود كتالوجًا أو إعدادات افتراضية لـ base URL |
| 2   | `applyConfigDefaults` | تطبيق الإعدادات الافتراضية العامة المملوكة للمزود أثناء مادية الإعدادات | عندما تعتمد الإعدادات الافتراضية على وضع المصادقة أو البيئة أو دلالات عائلة نماذج المزود |
| --  | _(البحث المدمج عن النموذج)_ | يحاول OpenClaw أولًا مسار السجل/الكتالوج العادي | _(ليس خطاف إضافة)_ |

يتحقق `normalizeModelId` و`normalizeTransport` و`normalizeConfig` أولًا من
إضافة المزود المطابقة، ثم ينتقل إلى إضافات مزود أخرى قادرة على الخطافات
حتى يغيّر أحدها فعليًا معرف النموذج أو الناقل/الإعدادات. وهذا يحافظ
على عمل الجسور المستعارة/المتوافقة من دون أن يحتاج المستدعي إلى معرفة
أي إضافة مجمعة تملك إعادة الكتابة. وإذا لم تُعد أي خطافة مزود كتابة
إدخال إعداد مدعوم من Google-family، فسيظل مطبّع إعدادات Google
المجمّع يطبّق تنظيف التوافق هذا.

إذا احتاج المزود إلى بروتوكول نقل مخصص كامل أو منفذ طلبات مخصص،
فهذه فئة مختلفة من الامتدادات. هذه الخطافات مخصصة لسلوك المزود
الذي لا يزال يعمل على حلقة الاستدلال العادية في OpenClaw.

### مثال على المزود

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

### أمثلة مدمجة

- تستخدم Anthropic `resolveDynamicModel` و`capabilities` و`buildAuthDoctorHint`,
  و`resolveUsageAuth`, و`fetchUsageSnapshot`, و`isCacheTtlEligible`,
  و`resolveDefaultThinkingLevel`, و`applyConfigDefaults`, و`isModernModelRef`,
  و`wrapStreamFn` لأنها تملك توافق Claude 4.6 الأمامي،
  وتلميحات عائلة المزود، وإرشادات إصلاح المصادقة، وتكامل نقطة
  الاستخدام، وأهلية ذاكرة التخزين المؤقت للموجّه، والإعدادات الافتراضية
  المعتمدة على المصادقة، وسياسة التفكير الافتراضية/التكيفية لـ Claude،
  وتشكيل البث الخاص بـ Anthropic لرؤوس beta و`/fast` / `serviceTier`
  و`context1m`.
- تبقى مساعدات بث Anthropic الخاصة بـ Claude ضمن الوصلة العامة
  `api.ts` / `contract-api.ts` الخاصة بالإضافة المجمعة في الوقت الحالي. وتصدّر
  هذه الحزمة `wrapAnthropicProviderStream`, و`resolveAnthropicBetas`,
  و`resolveAnthropicFastMode`, و`resolveAnthropicServiceTier`, وبناة
  أغلفة Anthropic منخفضة المستوى بدلًا من توسيع SDK العام حول قواعد
  رؤوس beta الخاصة بمزود واحد.
- تستخدم OpenAI `resolveDynamicModel`, و`normalizeResolvedModel`, و
  `capabilities` بالإضافة إلى `buildMissingAuthMessage`, و`suppressBuiltInModel`,
  و`augmentModelCatalog`, و`supportsXHighThinking`, و`isModernModelRef`
  لأنها تملك توافق GPT-5.4 الأمامي، والتطبيع المباشر لـ OpenAI
  من `openai-completions` إلى `openai-responses`, وتلميحات المصادقة
  الواعية بـ Codex، وكبح Spark، وصفوف OpenAI الاصطناعية، وسياسة
  التفكير/النموذج الحي لـ GPT-5؛ وتملك عائلة البث `openai-responses-defaults`
  أغلفة OpenAI Responses الأصلية المشتركة لرؤوس الإسناد،
  و`/fast`/`serviceTier`, وtext verbosity, وCodex web search الأصلي،
  وتشكيل الحمولات المتوافقة مع الاستدلال، وإدارة سياق Responses.
- تستخدم OpenRouter `catalog` بالإضافة إلى `resolveDynamicModel` و
  `prepareDynamicModel` لأن المزود يمرر البيانات وقد يكشف معرفات
  نماذج جديدة قبل تحديث الكتالوج الثابت في OpenClaw؛ كما يستخدم
  `capabilities`, و`wrapStreamFn`, و`isCacheTtlEligible` لإبقاء
  الرؤوس الخاصة بالمزود، وبيانات التوجيه الوصفية، وتصحيحات الاستدلال،
  وسياسة ذاكرة التخزين المؤقت خارج النواة. تأتي سياسة replay الخاصة به من
  عائلة `passthrough-gemini`, بينما تملك عائلة البث `openrouter-thinking`

- تستخدم GitHub Copilot `catalog`, و`auth`, و`resolveDynamicModel`, و

  لا يزال يعمل على نواقل OpenAI الأساسية لكنه يملك تطبيع

## مساعدات وقت التشغيل

يمكن للإضافات الوصول إلى مساعدين مختارين من النواة عبر `api.runtime`. بالنسبة إلى TTS:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

ملاحظات:

- يعيد `textToSpeech` حمولة خرج TTS الأساسية العادية لأسطح الملفات/الملاحظات الصوتية.
- يستخدم إعداد `messages.tts` الأساسي واختيار المزود.
- يعيد مخزن صوت PCM المؤقت + معدل العينة. ويجب على الإضافات إعادة أخذ العينات/الترميز من أجل المزودين.
- `listVoices` اختياري لكل مزود. استخدمه لمنتقيات الأصوات أو تدفقات الإعداد المملوكة للمورد.
- قد تتضمن قوائم الأصوات بيانات وصفية أغنى مثل اللغة والمنطقة والنوع وعلامات الشخصية لمنتقيات تراعي المزود.
- يدعم OpenAI وElevenLabs الاتصالات الهاتفية اليوم. ولا يدعم Microsoft ذلك.

يمكن للإضافات أيضًا تسجيل مزودي الكلام عبر `api.registerSpeechProvider(...)`.

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

ملاحظات:

- أبقِ سياسة TTS وآلية التراجع وتسليم الردود في النواة.
- استخدم مزودي الكلام لسلوك التوليف المملوك للمورد.
- يُطبَّع الإدخال القديم `edge` الخاص بـ Microsoft إلى معرّف المزود `microsoft`.
- نموذج الملكية المفضل موجَّه حسب الشركة: يمكن لإضافة مورد واحدة أن تملك
  مزودي النص والكلام والصور والوسائط المستقبلية مع إضافة OpenClaw
  لعقود تلك القدرات.

أما بالنسبة إلى فهم الصور/الصوت/الفيديو، فتسجل الإضافات مزودًا واحدًا مكتوب
الأنواع لفهم الوسائط بدل حقيبة عامة من المفاتيح/القيم:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

ملاحظات:

- أبقِ التنسيق وآلية التراجع والإعدادات وربط القنوات في النواة.
- أبقِ سلوك المورد في إضافة المزود.
- يجب أن يبقى التوسع الإضافي مكتوب الأنواع: أساليب اختيارية جديدة، وحقول
  نتائج اختيارية جديدة، وقدرات اختيارية جديدة.
- يتبع توليد الفيديو النمط نفسه بالفعل:
  - تملك النواة عقد القدرة ومساعد وقت التشغيل
  - تسجل إضافات المورد `api.registerVideoGenerationProvider(...)`
  - تستهلك إضافات الميزات/القنوات `api.runtime.videoGeneration.*`

بالنسبة إلى مساعدات وقت التشغيل الخاصة بفهم الوسائط، يمكن للإضافات استدعاء:

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});

const video = await api.runtime.mediaUnderstanding.describeVideoFile({
  filePath: "/tmp/inbound-video.mp4",
  cfg: api.config,
});
```

وبالنسبة إلى نسخ الصوت، يمكن للإضافات استخدام وقت تشغيل فهم الوسائط أو
الاسم المستعار الأقدم لـ STT:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // اختياري عندما لا يمكن استنتاج MIME بشكل موثوق:
  mime: "audio/ogg",
});
```

ملاحظات:

- `api.runtime.mediaUnderstanding.*` هو السطح المشترك المفضل
  لفهم الصور/الصوت/الفيديو.
- يستخدم إعداد الصوت الأساسي لفهم الوسائط (`tools.media.audio`) وترتيب التراجع بين المزودين.
- يعيد `{ text: undefined }` عندما لا ينتج أي خرج نسخ (على سبيل المثال، عندما يكون الإدخال متخطيًا/غير مدعوم).
- لا يزال `api.runtime.stt.transcribeAudioFile(...)` اسمًا مستعارًا للتوافق.

يمكن للإضافات أيضًا تشغيل عمليات وكيل فرعي في الخلفية عبر `api.runtime.subagent`:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

ملاحظات:

- `provider` و`model` تجاوزات اختيارية لكل تشغيل، وليسا تغييرات دائمة للجلسة.
- لا يلتزم OpenClaw بحقول التجاوز هذه إلا للمتصلين الموثوقين.
- بالنسبة إلى التشغيلات الاحتياطية المملوكة للإضافة، يجب على المشغلين تفعيل
  `plugins.entries.<id>.subagent.allowModelOverride: true`.
- استخدم `plugins.entries.<id>.subagent.allowedModels` لتقييد الإضافات الموثوقة
  إلى أهداف `provider/model` معيارية محددة، أو `"*"` للسماح الصريح بأي هدف.
- تظل تشغيلات الوكلاء الفرعيين للإضافات غير الموثوقة تعمل، لكن تُرفض طلبات التجاوز بدل أن تتراجع بصمت.

بالنسبة إلى البحث في الويب، يمكن للإضافات استهلاك مساعد وقت التشغيل المشترك بدلًا من
الوصول مباشرة إلى ربط أداة الوكيل:

```ts
const providers = api.runtime.webSearch.listProviders({
  config: api.config,
});

const result = await api.runtime.webSearch.search({
  config: api.config,
  args: {
    query: "OpenClaw plugin runtime helpers",
    count: 5,
  },
});
```

يمكن للإضافات أيضًا تسجيل مزودي البحث في الويب عبر
`api.registerWebSearchProvider(...)`.

ملاحظات:

- أبقِ اختيار المزود وحل بيانات الاعتماد ودلالات الطلب المشتركة في النواة.
- استخدم مزودي البحث في الويب لنواقل البحث الخاصة بالمورد.
- `api.runtime.webSearch.*` هو السطح المشترك المفضل لإضافات الميزات/القنوات التي تحتاج إلى سلوك بحث من دون الاعتماد على مغلف أداة الوكيل.

```ts
const result = await api.runtime.imageGeneration.generate({
```

## مسارات Gateway HTTP

يمكن للإضافات كشف نقاط نهاية HTTP عبر `api.registerHttpRoute(...)`.

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

حقول المسار:

- `path`: مسار الطريق ضمن خادم HTTP الخاص بـ Gateway.
- `auth`: مطلوب. استخدم `"gateway"` لطلب مصادقة البوابة العادية، أو `"plugin"` للمصادقة/التحقق الخاص بالإضافة أو webhook.
- `match`: اختياري. `"exact"` (الافتراضي) أو `"prefix"`.
- `replaceExisting`: اختياري. يسمح للإضافة نفسها باستبدال تسجيل مسار سابق خاص بها.
- `handler`: أرجع `true` عندما يكون المسار قد عالج الطلب.

ملاحظات:

- تمت إزالة `api.registerHttpHandler(...)` وسيؤدي استخدامها إلى خطأ عند تحميل الإضافة. استخدم `api.registerHttpRoute(...)` بدلًا منها.
- يجب أن تعلن مسارات الإضافات عن `auth` صراحة.
- تُرفض تعارضات `path + match` الدقيقة إلا إذا كانت `replaceExisting: true`، ولا يمكن لإضافة أن تستبدل مسارًا لإضافة أخرى.
- تُرفض المسارات المتداخلة التي تملك مستويات `auth` مختلفة. أبقِ سلاسل السقوط `exact`/`prefix` على مستوى المصادقة نفسه فقط.
- لا تتلقى مسارات `auth: "plugin"` نطاقات وقت تشغيل المشغّل تلقائيًا. فهي مخصصة لـ webhooks/التحقق من التوقيع الذي تديره الإضافة، لا لاستدعاءات المساعدات المميزة في Gateway.
- تعمل مسارات `auth: "gateway"` داخل نطاق وقت تشغيل طلب Gateway، لكن هذا النطاق محافظ عمدًا:
  - مصادقة bearer بالسر المشترك (`gateway.auth.mode = "token"` / `"password"`) تُبقي نطاقات وقت تشغيل مسارات الإضافات مثبتة على `operator.write`, حتى لو أرسل المستدعي `x-openclaw-scopes`
  - أوضاع HTTP الموثوقة التي تحمل هوية (مثل `trusted-proxy` أو `gateway.auth.mode = "none"` على ingress خاص) لا تكرم `x-openclaw-scopes` إلا عندما يكون الرأس حاضرًا صراحة
  - إذا كان `x-openclaw-scopes` غائبًا في طلبات مسارات الإضافات الحاملة للهوية، يتراجع نطاق وقت التشغيل إلى `operator.write`
- القاعدة العملية: لا تفترض أن مسار إضافة بمصادقة البوابة هو سطح إدارة ضمني. وإذا كان مسارك يحتاج إلى سلوك إداري فقط، فاطلب وضع مصادقة يحمل هوية ووثّق عقد الرأس الصريح `x-openclaw-scopes`.

  - تستخدم OpenRouter `api.js` لباني المزود الخاص بها بالإضافة إلى مساعدات
    التهيئة/الإعدادات، بينما يمكن لـ `register.runtime.js` أن يعيد تصدير
    مساعدات `plugin-sdk/provider-stream` العامة للاستخدام المحلي داخل المستودع.
- تفضّل نقاط الإدخال العامة المحملة عبر الواجهة لقطة إعدادات وقت التشغيل
  النشطة عندما تكون موجودة، ثم تتراجع إلى ملف الإعدادات المحلول على
  القرص عندما لا يكون OpenClaw قد بدأ بعد في تقديم لقطة وقت تشغيل.
- تظل الأساسيات العامة المشتركة هي عقد SDK العام المفضل. ولا تزال هناك
  مجموعة توافق صغيرة محجوزة من الوصلات المساعدة ذات العلامة الخاصة بالقنوات المجمعة.
  تعامل مع هذه الوصلات على أنها مسارات صيانة/توافق للإضافات المجمعة، لا أهداف
  استيراد جديدة لأطراف خارجية؛ ويجب أن تهبط العقود الجديدة العابرة للقنوات
  على المسارات العامة `plugin-sdk/*` أو على حاويات الإضافة المحلية `api.js` /
  `runtime-api.js`.

ملاحظة التوافق:

- تجنب الحاوية الجذرية `openclaw/plugin-sdk` في الشيفرة الجديدة.
- فضّل أولًا الأساسيات المستقرة الضيقة. فالمسارات الفرعية الأحدث الخاصة بالإعداد/الاقتران/الرد/
  والملاحظات/العقد/الوارد/الخيوط/الأوامر/الإدخال السري/webhook/البنية/
  وقائمة السماح/الحالة/أداة الرسائل هي العقد المقصود لعمل الإضافات الجديدة
  المجمعة والخارجية.
  ويجب أن يكون التحليل/المطابقة الخاصة بالأهداف على `openclaw/plugin-sdk/channel-targets`.
  وتجب أن تكون بوابات إجراءات الرسائل ومساعدات معرفات رسائل التفاعل على
  `openclaw/plugin-sdk/channel-actions`.
- لا تكون الحاويات المساعدة الخاصة بالامتدادات المجمعة مستقرة افتراضيًا. وإذا
  كانت المساعدة مطلوبة فقط لامتداد مجمع، فابقها خلف وصلة
  `api.js` أو `runtime-api.js` المحلية الخاصة به بدل ترقيتها إلى
  `openclaw/plugin-sdk/<extension>`.
- يجب أن تكون وصلات المساعدات المشتركة الجديدة عامة لا ممهورة باسم قناة.
  فتحليل الأهداف المشتركة ينتمي إلى `openclaw/plugin-sdk/channel-targets`;
  أما الأجزاء الداخلية الخاصة بالقناة فتبقى خلف `api.js` أو `runtime-api.js`
  المحلية الخاصة بالإضافة المالكة.
- توجد مسارات فرعية خاصة بالقدرات مثل `image-generation`,
  و`media-understanding`, و`speech` لأن الإضافات المجمعة/الأصلية تستخدمها
  اليوم. لكن وجودها لا يعني بحد ذاته أن كل مساعد مُصدَّر يمثل
  عقدًا خارجيًا طويل الأمد ومجمّدًا.

## مخططات أداة الرسائل

يجب أن تملك الإضافات المساهمات الخاصة بالقناة في مخطط
`describeMessageTool(...)`. أبقِ الحقول الخاصة بالمزود داخل الإضافة،
لا في النواة المشتركة.

وبالنسبة إلى أجزاء المخطط المحمولة المشتركة، أعد استخدام المساعدات العامة المصدَّرة عبر
`openclaw/plugin-sdk/channel-actions`:

- `createMessageToolButtonsSchema()` لحمولات على نمط شبكة الأزرار
- `createMessageToolCardSchema()` لحمولات البطاقات المنظمة

إذا كان شكل المخطط منطقيًا لمزود واحد فقط، فعرّفه في مصدر تلك الإضافة
الخاص بدل ترقيته إلى SDK المشترك.

## حل أهداف القنوات

يجب أن تملك إضافات القنوات دلالات الأهداف الخاصة بالقناة. أبقِ مضيف
الإرسال الخارجي عامًا واستخدم سطح محول المراسلة لقواعد المزود:

- `messaging.inferTargetChatType({ to })` يقرر ما إذا كان يجب التعامل مع
  هدف مُطبَّع على أنه `direct` أو `group` أو `channel` قبل البحث في الدليل.
- `messaging.targetResolver.looksLikeId(raw, normalized)` يخبر النواة ما إذا كان
  يجب أن يتجاوز الإدخال مباشرةً إلى حل شبيه بالمعرف بدل البحث في الدليل.
- `messaging.targetResolver.resolveTarget(...)` هو تراجع الإضافة عندما
  تحتاج النواة إلى حل نهائي يملكه المزود بعد التطبيع أو بعد عدم العثور عليه في الدليل.
- `messaging.resolveOutboundSessionRoute(...)` يملك إنشاء مسار الجلسة الصادر الخاص بالمزود بمجرد حل الهدف.

التقسيم الموصى به:

- استخدم `inferTargetChatType` لقرارات الفئة التي يجب أن تحدث قبل
  البحث عن الأقران/المجموعات.
- استخدم `looksLikeId` للتحقق من "اعتبر هذا معرف هدف صريحًا/أصليًا".
- استخدم `resolveTarget` للتراجع الخاص بالمزود في التطبيع، لا للبحث الواسع في الدليل.
- أبقِ المعرفات الأصلية للمزود مثل معرفات الدردشة، ومعرفات الخيوط، وJIDs، والمعالِجات، ومعرفات الغرف
  داخل قيم `target` أو في معلمات خاصة بالمزود، لا في حقول SDK عامة.

## الأدلة المدعومة بالإعدادات

يجب على الإضافات التي تستمد إدخالات الدليل من الإعدادات أن تُبقي هذا المنطق داخل
الإضافة وأن تعيد استخدام المساعدات المشتركة من
`openclaw/plugin-sdk/directory-runtime`.

استخدم ذلك عندما تحتاج القناة إلى أقران/مجموعات مدعومة بالإعدادات مثل:

- أقران الرسائل المباشرة المعتمدين على قائمة السماح
- خرائط القنوات/المجموعات المضبوطة
- تراجعات الدليل الثابتة المرتبطة بالحساب

المساعدات المشتركة في `directory-runtime` تعالج العمليات العامة فقط:

- تصفية الاستعلام
- تطبيق الحدود
- إزالة التكرار/التطبيع
- بناء `ChannelDirectoryEntry[]`

أما فحص الحسابات الخاص بالقناة وتطبيع المعرفات فيجب أن يبقيا داخل تنفيذ الإضافة.

## كتالوجات المزودين

يمكن لإضافات المزودين تعريف كتالوجات نماذج للاستدلال باستخدام
`registerProvider({ catalog: { run(...) { ... } } })`.

تعيد `catalog.run(...)` الشكل نفسه الذي يكتبه OpenClaw في
`models.providers`:

- `{ provider }` لإدخال مزود واحد
- `{ providers }` لعدة إدخالات مزود

استخدم `catalog` عندما تملك الإضافة معرفات النماذج الخاصة بالمزود أو
إعدادات base URL الافتراضية أو البيانات الوصفية للنماذج المحكومة بالمصادقة.

يتحكم `catalog.order` في وقت دمج كتالوج الإضافة نسبة إلى المزودين
الضمنيين المدمجين في OpenClaw:

- `simple`: مزودون عاديون يعتمدون على مفتاح API أو البيئة
- `profile`: مزودون يظهرون عند وجود ملفات تعريف المصادقة
- `paired`: مزودون يركّبون عدة إدخالات مزود مترابطة
- `late`: المرور الأخير، بعد المزودين الضمنيين الآخرين

يفوز المزود اللاحق عند تعارض المفتاح، وبالتالي يمكن للإضافات عمدًا تجاوز
إدخال مزود مدمج يحمل معرّف المزود نفسه.

التوافق:

- لا يزال `discovery` يعمل كاسم مستعار قديم
- إذا سُجِّل كل من `catalog` و`discovery`، فسيستخدم OpenClaw `catalog`

## فحص القنوات للقراءة فقط

إذا كانت إضافتك تسجل قناة، ففضّل تنفيذ
`plugin.config.inspectAccount(cfg, accountId)` إلى جانب `resolveAccount(...)`.

السبب:

- يمثل `resolveAccount(...)` مسار وقت التشغيل. ويُسمح له بافتراض أن بيانات
  الاعتماد قد تمّت ماديتها بالكامل، ويمكنه الفشل سريعًا عند غياب الأسرار المطلوبة.
- يجب ألا تحتاج مسارات الأوامر المخصصة للقراءة فقط مثل `openclaw status`,
  و`openclaw status --all`, و`openclaw channels status`, و`openclaw channels resolve`,
  وتدفقات إصلاح doctor/config إلى مادية بيانات اعتماد وقت التشغيل
  فقط لوصف الإعدادات.

السلوك الموصى به لـ `inspectAccount(...)`:

- أعد فقط حالة حساب وصفية.
- حافظ على `enabled` و`configured`.
- ضمّن حقول مصدر/حالة بيانات الاعتماد عند الحاجة، مثل:
  - `tokenSource`, `tokenStatus`
  - `botTokenSource`, `botTokenStatus`
  - `appTokenSource`, `appTokenStatus`
  - `signingSecretSource`, `signingSecretStatus`
- لا تحتاج إلى إرجاع قيم الرموز الخام فقط للإبلاغ عن التوفر للقراءة فقط.
  يكفي إرجاع `tokenStatus: "available"` (وحقل المصدر المطابق).
- استخدم `configured_unavailable` عندما تكون بيانات الاعتماد مضبوطة عبر SecretRef ولكن
  غير متاحة في مسار الأمر الحالي.

وهذا يسمح للأوامر المخصصة للقراءة فقط بالإبلاغ عن "مهيأ لكنه غير متاح في مسار
الأمر هذا" بدل التعطل أو الإبلاغ خطأً عن أن الحساب غير مهيأ.

## حزم Package packs

يمكن أن يضم دليل الإضافة `package.json` مع `openclaw.extensions`:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

يصبح كل إدخال إضافة. وإذا أدرجت الحزمة عدة امتدادات، يصبح معرّف الإضافة
`name/<fileBase>`.

إذا كانت إضافتك تستورد تبعيات npm، فثبّتها في ذلك الدليل بحيث يكون
`node_modules` متاحًا (`npm install` / `pnpm install`).

حاجز أمان: يجب أن يبقى كل إدخال في `openclaw.extensions` داخل دليل الإضافة
بعد حل الروابط الرمزية. وتُرفض الإدخالات التي تفلت من دليل الحزمة.

ملاحظة أمان: يثبت `openclaw plugins install` تبعيات الإضافات باستخدام
`npm install --omit=dev --ignore-scripts` (من دون نصوص lifecycle، ومن دون تبعيات تطوير وقت التشغيل). حافظ على أشجار
تبعيات الإضافة "JS/TS خالصة" وتجنب الحزم التي تتطلب عمليات بناء `postinstall`.

اختياري: يمكن أن يشير `openclaw.setupEntry` إلى وحدة إعداد خفيفة فقط.
عندما يحتاج OpenClaw إلى أسطح إعداد لإضافة قناة معطلة، أو
عندما تكون إضافة القناة مفعلة لكنها لا تزال غير مهيأة، فإنه يحمّل `setupEntry`
بدل مدخل الإضافة الكامل. ويُبقي ذلك بدء التشغيل والإعداد أخف
عندما يكون مدخل الإضافة الرئيسي لديك يربط أيضًا أدوات أو خطافات أو
شيفرة وقت تشغيل فقط أخرى.

اختياري: يمكن لـ `openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
إدخال إضافة القناة في المسار نفسه لـ `setupEntry` أثناء مرحلة بدء التشغيل قبل
الاستماع في البوابة، حتى عندما تكون القناة مهيأة بالفعل.

استخدم هذا فقط عندما يغطي `setupEntry` بالكامل سطح بدء التشغيل الذي يجب أن يوجد
قبل أن تبدأ البوابة في الاستماع. وعمليًا، يعني ذلك أن مدخل الإعداد
يجب أن يسجل كل قدرة تملكها القناة ويعتمد عليها بدء التشغيل، مثل:

- تسجيل القناة نفسها
- أي مسارات HTTP يجب أن تكون متاحة قبل أن تبدأ البوابة في الاستماع
- أي أساليب أو أدوات أو خدمات Gateway يجب أن توجد خلال النافذة نفسها

إذا كان المدخل الكامل لا يزال يملك أي قدرة مطلوبة عند بدء التشغيل، فلا
تفعّل هذا العلم. أبقِ الإضافة على السلوك الافتراضي ودع OpenClaw يحمّل
المدخل الكامل أثناء بدء التشغيل.

يمكن للقنوات المجمعة أيضًا نشر مساعدات لسطح العقد خاصة بالإعداد فقط يمكن للنواة
استشارتها قبل تحميل وقت تشغيل القناة الكامل. وسطح ترقية الإعداد الحالي هو:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

تستخدم النواة هذا السطح عندما تحتاج إلى ترقية إعداد قناة قديم أحادي الحساب
إلى `channels.<id>.accounts.*` من دون تحميل مدخل الإضافة الكامل.
وتمثل Matrix المثال المجمع الحالي: فهي تنقل فقط مفاتيح
المصادقة/التمهيد إلى حساب مسمى مُرقّى عندما تكون الحسابات المسمّاة موجودة
بالفعل، ويمكنها الحفاظ على مفتاح حساب افتراضي غير قياسي مضبوط بدل إنشاء
`accounts.default` دائمًا.

تحافظ محولات رقع الإعداد هذه على كسل اكتشاف سطح العقد المجمعة. فيبقى
وقت الاستيراد خفيفًا؛ ولا يُحمَّل سطح الترقية إلا عند أول استخدام بدل
إعادة إدخال بدء تشغيل القناة المجمعة عند استيراد الوحدة.

وعندما تتضمن أسطح بدء التشغيل هذه أساليب Gateway RPC، فأبقها تحت
بادئة خاصة بالإضافة. تظل مساحات أسماء إدارة النواة (`config.*`,
`exec.approvals.*`, `wizard.*`, `update.*`) محجوزة وتُحل دائمًا إلى
`operator.admin`, حتى لو طلبت الإضافة نطاقًا أضيق.

مثال:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }
  }
}
```

### بيانات كتالوج القنوات الوصفية

يمكن لإضافات القنوات الإعلان عن بيانات وصفية للإعداد/الاكتشاف عبر `openclaw.channel` و
تلميحات التثبيت عبر `openclaw.install`. وهذا يبقي بيانات الكتالوج الأساسية خالية من البيانات.

مثال:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (self-hosted)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "Self-hosted chat via Nextcloud Talk webhook bots.",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

حقول `openclaw.channel` المفيدة إلى جانب المثال الأدنى:

- `detailLabel`: تسمية ثانوية لأسطح الكتالوج/الحالة الأكثر ثراءً
- `docsLabel`: تجاوز نص الرابط الخاص بالوثائق
- `preferOver`: معرّفات إضافات/قنوات أقل أولوية يجب أن تتقدم هذه المدخلة في الكتالوج عليها
- `selectionDocsPrefix`, و`selectionDocsOmitLabel`, و`selectionExtras`: عناصر تحكم في نسخة سطح الاختيار
- `markdownCapable`: يحدد القناة على أنها قادرة على Markdown لقرارات التنسيق الصادر
- `exposure.configured`: إخفاء القناة من أسطح قائمة القنوات المهيأة عند تعيينها إلى `false`
- `exposure.setup`: إخفاء القناة من منتقيات الإعداد/التهيئة التفاعلية عند تعيينها إلى `false`
- `exposure.docs`: تحديد القناة على أنها داخلية/خاصة لأسطح التنقل في الوثائق
- `showConfigured` / `showInSetup`: أسماء مستعارة قديمة لا تزال مقبولة للتوافق؛ ويفضل `exposure`
- `quickstartAllowFrom`: إدخال القناة في تدفق `allowFrom` القياسي للبدء السريع
- `forceAccountBinding`: فرض ربط حساب صريح حتى عند وجود حساب واحد فقط
- `preferSessionLookupForAnnounceTarget`: تفضيل بحث الجلسة عند حل أهداف الإعلانات

يمكن لـ OpenClaw أيضًا دمج **كتالوجات قنوات خارجية** (مثل تصدير سجل
MPM). أسقط ملف JSON في أحد المواقع التالية:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

أو وجّه `OPENCLAW_PLUGIN_CATALOG_PATHS` (أو `OPENCLAW_MPM_CATALOG_PATHS`) إلى
ملف JSON واحد أو أكثر (مفصولًا بفواصل/فواصل منقوطة/`PATH`). ويجب أن يحتوي كل ملف
على `{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }`. كما يقبل
المحلل `"packages"` أو `"plugins"` كأسماء مستعارة قديمة للمفتاح `"entries"`.

## إضافات محرك السياق

تملك إضافات محرك السياق تنسيق سياق الجلسة من أجل الإدخال، والتجميع،
والضغط. سجّلها من إضافتك باستخدام
`api.registerContextEngine(id, factory)`, ثم اختر المحرك النشط باستخدام
`plugins.slots.contextEngine`.

استخدم هذا عندما تحتاج إضافتك إلى استبدال مسار السياق الافتراضي أو توسيعه
بدل مجرد إضافة بحث في الذاكرة أو خطافات.

```ts
import { buildMemorySystemPromptAddition } from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("lossless-claw", () => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages, availableTools, citationsMode }) {
      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

إذا كان محركك **لا** يملك خوارزمية الضغط، فاحتفظ بتنفيذ `compact()`
وفوّضها صراحة:

```ts
import {
  buildMemorySystemPromptAddition,
  delegateCompactionToRuntime,
} from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", () => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },

      return {
        messages,
        estimatedTokens: 0,
        systemPromptAddition: buildMemorySystemPromptAddition({
          availableTools: availableTools ?? new Set(),
          citationsMode,
        }),
      };

    async compact(params) {

    },
```

## إضافة قدرة جديدة

عندما تحتاج إضافة إلى سلوك لا يناسب واجهة البرمجة الحالية، فلا تتجاوز
نظام الإضافات عبر وصول خاص. أضف القدرة المفقودة.

التسلسل الموصى به:

1. تعريف العقد الأساسي
   قرر السلوك المشترك الذي يجب أن تملكه النواة: السياسة، وآلية التراجع، ودمج الإعدادات،
   ودورة الحياة، والدلالات المواجهة للقنوات، وشكل مساعد وقت التشغيل.
2. إضافة أسطح تسجيل/وقت تشغيل مكتوبة الأنواع
   وسّع `OpenClawPluginApi` و/أو `api.runtime` بأصغر
   سطح قدرة مكتوب الأنواع ومفيد.
3. توصيل مستهلكي النواة + القنوات/الميزات
   يجب أن تستهلك القنوات وإضافات الميزات القدرة الجديدة عبر النواة،
   لا عبر استيراد تنفيذ مورد مباشرة.
4. تسجيل تنفيذات المورد
   ثم تسجل إضافات المورد واجهاتها الخلفية مقابل القدرة.
5. إضافة تغطية للعقد
   أضف اختبارات حتى تبقى الملكية وشكل التسجيل واضحين مع الوقت.

هكذا يحافظ OpenClaw على وضوح رؤيته من دون أن يصبح مرمّزًا على
منظور مزود واحد. راجع [دفتر وصفات القدرات](/ar/plugins/architecture)
للحصول على قائمة ملفات عملية ومثال منجز.

### قائمة تحقق القدرة

عند إضافة قدرة جديدة، يجب عادةً أن يطال التنفيذ هذه
الأسطح معًا:

- أنواع العقود الأساسية في `src/<capability>/types.ts`
- المشغّل/مساعد وقت التشغيل الأساسي في `src/<capability>/runtime.ts`
- سطح تسجيل واجهة برمجة الإضافة في `src/plugins/types.ts`
- توصيل سجل الإضافات في `src/plugins/registry.ts`
- كشف وقت تشغيل الإضافة في `src/plugins/runtime/*` عندما تحتاج
  إضافات الميزات/القنوات إلى استهلاكه
- مساعدات الالتقاط/الاختبار في `src/test-utils/plugin-registration.ts`
- تأكيدات الملكية/العقد في `src/plugins/contracts/registry.ts`
- وثائق المشغل/الإضافات في `docs/`

إذا غاب أحد هذه الأسطح، فهذه غالبًا علامة على أن القدرة لم تُدمج
بالكامل بعد.

### قالب القدرة

الحد الأدنى من النمط:

```ts
// العقد الأساسي
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// واجهة برمجة الإضافة
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// مساعد وقت تشغيل مشترك لإضافات الميزات/القنوات
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

نمط اختبار العقد:

```ts
expect(findVideoGenerationProviderIdsForPlugin("openai")).toEqual(["openai"]);
```

وهذا يبقي القاعدة بسيطة:

- تملك النواة عقد القدرة + التنسيق
- تملك إضافات المورد تنفيذات المورد
- تستهلك إضافات الميزات/القنوات مساعدات وقت التشغيل
- تُبقي اختبارات العقود الملكية واضحة
