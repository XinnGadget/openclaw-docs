# Очередь команд (2026-01-16)

Мы сериализуем входящие запуски автоответа (для всех каналов) через небольшую очередь в процессе, чтобы предотвратить конфликты между несколькими запусками агентов, при этом сохраняя безопасный параллелизм между сессиями.

## Зачем это нужно

- Запуски автоответа могут быть ресурсоёмкими (вызовы LLM) и могут конфликтовать, если несколько входящих сообщений поступают почти одновременно.
- Сериализация позволяет избежать конкуренции за общие ресурсы (файлы сессий, логи, stdin CLI) и снижает вероятность достижения лимитов скорости на вышестоящем уровне.

## Как это работает

- FIFO-очередь с учётом полос (lane-aware) обрабатывает каждую полосу с настраиваемым ограничением параллелизма (по умолчанию для ненастроенных полос — 1; для основной полосы — 4, для субагента — 8).
- `runEmbeddedPiAgent` добавляет в очередь по **ключу сессии** (полоса `session:<key>`), чтобы гарантировать, что для каждой сессии активен только один запуск.
- Каждый запуск сессии затем помещается в **глобальную полосу** (`main` по умолчанию), поэтому общий параллелизм ограничивается параметром `agents.defaults.maxConcurrent`.
- Если включено подробное логирование, для заданий в очереди выводится краткое уведомление, если они ждали запуска более ~2 с.
- Индикаторы набора текста по-прежнему отправляются сразу при добавлении в очередь (если это поддерживается каналом), поэтому пользовательский опыт не меняется, пока мы ожидаем своей очереди.

## Режимы очереди (для каждого канала)

Входящие сообщения могут управлять текущим запуском, ожидать следующего хода или делать и то и другое:

- `steer`: немедленно внедрить в текущий запуск (отменяет ожидающие вызовы инструментов после следующей границы инструмента). Если потоковая передача не используется, возвращается к режиму followup.
- `followup`: добавить в очередь для следующего хода агента после завершения текущего запуска.
- `collect`: объединить все сообщения в очереди в **один** последующий ход (по умолчанию). Если сообщения направлены в разные каналы/потоки, они обрабатываются по отдельности, чтобы сохранить маршрутизацию.
- `steer-backlog` (также `steer+backlog`): управлять сейчас **и** сохранить сообщение для последующего хода.
- `interrupt` (устаревший): прервать активный запуск для этой сессии, затем обработать самое новое сообщение.
- `queue` (устаревший псевдоним): то же, что и `steer`.

Steer-backlog позволяет получить ответ после управляемого запуска, поэтому на потоковых поверхностях может показаться, что ответы дублируются. Предпочитайте `collect`/`steer`, если хотите получить один ответ на каждое входящее сообщение.

Отправьте `/queue collect` как отдельную команду (для сессии) или установите `messages.queue.byChannel.discord: "collect"`.

Значения по умолчанию (если не заданы в конфигурации):

- Все поверхности → `collect`

Настройте глобально или для каждого канала через `messages.queue`:

```json5
{
  messages: {
    queue: {
      mode: "collect",
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
      byChannel: { discord: "collect" },
    },
  },
}
```

## Параметры очереди

Параметры применяются к `followup`, `collect` и `steer-backlog` (а также к `steer`, если он возвращается к followup):

- `debounceMs`: ожидание затишья перед началом последующего хода (предотвращает многократное повторение "продолжить").
- `cap`: максимальное количество сообщений в очереди для сессии.
- `drop`: политика обработки переполнения (`old`, `new`, `summarize`).

Summarize сохраняет краткий список отброшенных сообщений в виде маркированного списка и внедряет его в качестве синтетического последующего запроса.

Значения по умолчанию: `debounceMs: 1000`, `cap: 20`, `drop: summarize`.

## Переопределения для сессии

- Отправьте `/queue <mode>` как отдельную команду, чтобы сохранить режим для текущей сессии.
- Параметры можно комбинировать: `/queue collect debounce:2s cap:25 drop:summarize`
- `/queue default` или `/queue reset` сбрасывает переопределение для сессии.

## Область применения и гарантии

- Применяется к запускам агентов автоответа для всех входящих каналов, использующих конвейер ответов шлюза (WhatsApp web, Telegram, Slack, Discord, Signal, iMessage, веб-чат и т. д.).
- Основная полоса (`main`) действует для всего процесса для входящих сообщений и основных пульсаций; установите `agents.defaults.maxConcurrent`, чтобы разрешить параллельную работу нескольких сессий.
- Могут существовать дополнительные полосы (например, `cron`, `subagent`), чтобы фоновые задания могли выполняться параллельно, не блокируя входящие ответы. Эти отсоединённые запуски отслеживаются как [фоновые задачи](/automation/tasks).
- Полосы для сессии гарантируют, что только один запуск агента взаимодействует с данной сессией в каждый момент времени.
- Нет внешних зависимостей или фоновых потоков; чистый TypeScript + промисы.

## Устранение неполадок

- Если команды кажутся зависшими, включите подробное логирование и ищите строки "queued for …ms", чтобы убедиться, что очередь обрабатывается.
- Если вам нужна глубина очереди, включите подробное логирование и следите за строками с временем обработки очереди.
