# Режим с повышенными привилегиями

Когда агент работает внутри песочницы, его команды `exec` ограничены средой песочницы. **Режим с повышенными привилегиями** позволяет агенту выйти за эти пределы и выполнять команды вне песочницы — с настраиваемыми этапами утверждения.

<Info>
  Режим с повышенными привилегиями изменяет поведение только в том случае, если агент находится в **песочнице**. Для агентов вне песочницы команды exec уже выполняются на хосте.
</Info>

## Директивы

Управляйте режимом с повышенными привилегиями для каждой сессии с помощью команд с косой чертой:

| Директива | Что делает |
| ---------------- | ---------------------------------------------------------------------- |
| `/elevated on` | Выполнять команды вне песочницы на настроенном пути хоста, сохранять этапы утверждения |
| `/elevated ask` | То же, что и `on` (псевдоним) |
| `/elevated full` | Выполнять команды вне песочницы на настроенном пути хоста и пропускать этапы утверждения |
| `/elevated off` | Вернуться к выполнению команд в пределах песочницы |

Также доступны в виде `/elev on|off|ask|full`.

Отправьте `/elevated` без аргументов, чтобы увидеть текущий уровень.

## Как это работает

<Steps>
  <Step title="Проверка доступности">
    Режим с повышенными привилегиями должен быть включён в конфигурации, а отправитель должен находиться в списке разрешённых:

    ```json5
    {
      tools: {
        elevated: {
          enabled: true,
          allowFrom: {
            discord: ["user-id-123"],
            whatsapp: ["+15555550123"],
          },
        },
      },
    }
    ```

  </Step>

  <Step title="Установка уровня">
    Отправьте сообщение, содержащее только директиву, чтобы задать значение по умолчанию для сессии:

    ```
    /elevated full
    ```

    Или используйте её встраиваемым способом (применяется только к этому сообщению):

    ```
    /elevated on run the deployment script
    ```

  </Step>

  <Step title="Команды выполняются вне песочницы">
    При активном режиме с повышенными привилегиями вызовы `exec` выходят за пределы песочницы. По умолчанию эффективным хостом является `gateway`, либо `node` — если в конфигурации или в сессии целью выполнения `exec` задан `node`. В режиме `full` этапы утверждения для `exec` пропускаются. В режимах `on`/`ask` по-прежнему применяются настроенные правила утверждения.
  </Step>
</Steps>

## Порядок разрешения

1. **Встраиваемая директива** в сообщении (применяется только к этому сообщению)
2. **Переопределение для сессии** (задаётся отправкой сообщения, содержащего только директиву)
3. **Глобальное значение по умолчанию** (`agents.defaults.elevatedDefault` в конфигурации)

## Доступность и списки разрешённых

- **Глобальный шлюз**: `tools.elevated.enabled` (должно быть `true`)
- **Список разрешённых отправителей**: `tools.elevated.allowFrom` с отдельными списками для каждого канала
- **Шлюз для отдельного агента**: `agents.list[].tools.elevated.enabled` (может только дополнительно ограничивать)
- **Список разрешённых для отдельного агента**: `agents.list[].tools.elevated.allowFrom` (отправитель должен соответствовать как глобальным, так и агент-специфичным условиям)
- **Резервный вариант для Discord**: если `tools.elevated.allowFrom.discord` не указан, в качестве запасного варианта используется `channels.discord.allowFrom`
- **Должны быть пройдены все шлюзы**; в противном случае режим с повышенными привилегиями считается недоступным

Форматы записей в списке разрешённых:

| Префикс | Соответствует |
| ----------------------- | ------------------------------- |
| (нет) | ID отправителя, E.164 или поле From |
| `name:` | Отображаемое имя отправителя |
| `username:` | Имя пользователя отправителя |
| `tag:` | Тег отправителя |
| `id:`, `from:`, `e164:` | Явное указание идентификатора |

## Что не контролируется режимом с повышенными привилегиями

- **Политика инструментов**: если выполнение `exec` запрещено политикой инструментов, режим с повышенными привилегиями не может это переопределить
- **Политика выбора хоста**: режим с повышенными привилегиями не превращает `auto` в свободное переопределение между хостами. Он использует настроенные правила цели выполнения `exec` для сессии, выбирая `node` только в том случае, если цель уже задана как `node`
- **Отделено от `/exec`**: директива `/exec` настраивает значения по умолчанию для `exec` в рамках сессии для авторизованных отправителей и не требует режима с повышенными привилегиями
