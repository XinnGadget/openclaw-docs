# Маршрутизация мультиагентов

Цель: реализовать несколько _изолированных_ агентов (отдельное рабочее пространство + `agentDir` + сессии), а также несколько учётных записей каналов (например, два WhatsApp) в одном запущенном шлюзе (Gateway). Входящие сообщения маршрутизируются к агенту через привязки (bindings).

## Что такое "один агент"?

**Агент** — это полностью автономный "интеллект", имеющий собственные:

- **Рабочее пространство** (файлы, AGENTS.md/SOUL.md/USER.md, локальные заметки, правила персоны).
- **Каталог состояния** (`agentDir`) для профилей аутентификации, реестра моделей и конфигурации, специфичной для агента.
- **Хранилище сессий** (история чатов + состояние маршрутизации) в каталоге `~/.openclaw/agents/<agentId>/sessions`.

Профили аутентификации относятся **к конкретному агенту**. Каждый агент считывает данные из своего файла:

```text
~/.openclaw/agents/<agentId>/agent/auth-profiles.json
```

`sessions_history` — это более безопасный способ получения данных между сессиями: он возвращает ограниченную и очищенную информацию, а не сырой дамп транскрипции. При получении данных помощником удаляются теги размышлений, разметка `<relevant-memories>`, XML-полезная нагрузка вызовов инструментов в виде обычного текста (включая `<tool_call>...</tool_call>`, `<function_call>...</function_call>`, `<tool_calls>...</tool_calls>`, `<function_calls>...</function_calls>` и усечённые блоки вызовов инструментов), упрощённая разметка вызовов инструментов, просочившиеся токены управления моделью в ASCII/полноширинном формате и некорректные XML-вызовы инструментов MiniMax до редактирования/усечения.

Основные учётные данные агента **не** передаются автоматически. Никогда не используйте `agentDir` для нескольких агентов (это приведёт к конфликтам аутентификации и сессий). Если необходимо поделиться учётными данными, скопируйте файл `auth-profiles.json` в `agentDir` другого агента.

Навыки загружаются из рабочего пространства каждого агента, а также из общих корневых каталогов, таких как `~/.openclaw/skills`, после чего фильтруются в соответствии со списком разрешённых навыков для конкретного агента (если он настроен). Используйте `agents.defaults.skills` для общего базового набора навыков и `agents.list[].skills` для замены навыков на уровне отдельных агентов. См. разделы [Навыки: для отдельных агентов и общие](/tools/skills#per-agent-vs-shared-skills) и [Навыки: списки разрешённых навыков для агентов](/tools/skills#agent-skill-allowlists).

Шлюз (Gateway) может размещать **одного агента** (по умолчанию) или **несколько агентов** одновременно.

**Примечание о рабочем пространстве:** рабочее пространство каждого агента является **текущим рабочим каталогом по умолчанию**, а не жёсткой песочницей. Относительные пути разрешаются внутри рабочего пространства, но абсолютные пути могут вести к другим местоположениям на хосте, если песочница не включена. См. раздел [Песочница](/gateway/sandboxing).

## Пути (краткая схема)

- Конфигурация: `~/.openclaw/openclaw.json` (или `OPENCLAW_CONFIG_PATH`)
- Каталог состояния: `~/.openclaw` (или `OPENCLAW_STATE_DIR`)
- Рабочее пространство: `~/.openclaw/workspace` (или `~/.openclaw/workspace-<agentId>`)
- Каталог агента: `~/.openclaw/agents/<agentId>/agent` (или `agents.list[].agentDir`)
- Сессии: `~/.openclaw/agents/<agentId>/sessions`

### Режим одного агента (по умолчанию)

Если не предпринимать никаких действий, OpenClaw запускает одного агента:

- `agentId` по умолчанию равен **`main`**.
- Сессии обозначаются как `agent:main:<mainKey>`.
- Рабочее пространство по умолчанию — `~/.openclaw/workspace` (или `~/.openclaw/workspace-<profile>`, если задана переменная `OPENCLAW_PROFILE`).
- Состояние по умолчанию — `~/.openclaw/agents/main/agent`.

## Помощник по работе с агентами

Используйте мастер агентов для добавления нового изолированного агента:

```bash
openclaw agents add work
```

Затем добавьте `bindings` (или позвольте мастеру сделать это), чтобы маршрутизировать входящие сообщения.

Проверьте результат с помощью команды:

```bash
openclaw agents list --bindings
```

```bash
openclaw channels login --channel whatsapp --account work
```

Добавьте агентов в раздел `agents.list`, учётные записи каналов — в `channels.<channel>.accounts`, а затем свяжите их с помощью `bindings` (примеры ниже).

  </Step>

  <Step title="Перезапустить и проверить">

```bash
openclaw gateway restart
openclaw agents list --bindings

```

## Несколько агентов = несколько человек, несколько личностей

При использовании **нескольких агентов** каждый `agentId` становится **полностью изолированной персоной**:

- **Разные номера телефонов/учётные записи** (для каждого `accountId` канала).
- **Разные личности** (файлы рабочего пространства для каждого агента, такие как `AGENTS.md` и `SOUL.md`).
- **Отдельная аутентификация и сессии** (отсутствие перекрёстных взаимодействий, если они явно не разрешены).

Это позволяет **нескольким пользователям** совместно использовать один сервер шлюза, сохраняя при этом изоляцию "интеллектов" ИИ и данных.

## Поиск памяти QMD между агентами

Если одному агенту необходимо выполнить поиск в транскрипциях сессий QMD другого агента, добавьте дополнительные коллекции в `agents.list[].memorySearch.qmd.extraCollections`. Используйте `agents.defaults.memorySearch.qmd.extraCollections` только в том случае, если каждый агент должен наследовать одни и те же общие коллекции транскрипций.

```json5
{
  agents: {
    defaults: {
      workspace: "~/workspaces/main",
      memorySearch: {
        qmd: {
          extraCollections: [{ path: "~/agents/family/sessions", name: "family-sessions" }],
        },
      },
    },
    list: [
      {
        id: "main",
        workspace: "~/workspaces/main",
        memorySearch: {
          qmd: {
            extraCollections: [{ path: "notes" }], // разрешается внутри рабочего пространства -> коллекция с именем "notes-main"
          },
        },
      },
      { id: "family", workspace: "~/workspaces/family" },
    ],
  },
  memory: {
    backend: "qmd",
    qmd: { includeDefaultMemory: false },
  },
}
```

Путь к дополнительной коллекции может быть общим для нескольких агентов, но имя коллекции остаётся явным, если путь находится за пределами рабочего пространства агента. Пути внутри рабочего пространства остаются в области действия агента, поэтому каждый агент сохраняет собственный набор для поиска транскрипций.

## Один номер WhatsApp, несколько человек (разделение личных сообщений)

Вы можете маршрутизировать **разные личные сообщения WhatsApp** к разным агентам, оставаясь при этом в рамках **одной учётной записи WhatsApp**. Сопоставляйте отправителя по E.164 (например, `+15551234567`) с `peer.kind: "direct"`. Ответы по-прежнему будут отправляться с одного и того же номера WhatsApp (без идентификации отправителя для каждого агента).
