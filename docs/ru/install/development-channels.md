# Каналы разработки

OpenClaw предоставляет три канала обновлений:

- **stable**: npm dist-tag `latest`. Рекомендуется для большинства пользователей.
- **beta**: npm dist-tag `beta`, если он актуален; если beta отсутствует или старше последней стабильной версии, процесс обновления переключается на `latest`.
- **dev**: подвижная головка ветки `main` (git). npm dist-tag: `dev` (при публикации).
  Ветка `main` предназначена для экспериментов и активной разработки. Она может содержать незавершённые функции или критические изменения. Не используйте её для производственных шлюзов.

Обычно мы сначала выпускаем стабильные сборки в **beta**, тестируем их там, а затем выполняем явный шаг продвижения, который перемещает проверенную сборку в `latest` без изменения номера версии. При необходимости сопровождающие также могут опубликовать стабильную версию напрямую в `latest`. Dist-теги являются первоисточником для установок через npm.

## Переключение каналов

```bash
openclaw update --channel stable
openclaw update --channel beta
openclaw update --channel dev
```

Параметр `--channel` сохраняет ваш выбор в конфигурации (`update.channel`) и согласовывает метод установки:

- **`stable`** (установка пакетов): обновление через npm dist-tag `latest`.
- **`beta`** (установка пакетов): предпочтение отдаётся npm dist-tag `beta`, но если `beta` отсутствует или старше текущего стабильного тега, происходит переключение на `latest`.
- **`stable`** (установка через git): извлекается последний стабильный git-тег.
- **`beta`** (установка через git): предпочтение отдаётся последнему бета-тегу git, но если бета-тег отсутствует или старше, происходит переключение на последний стабильный git-тег.
- **`dev`**: обеспечивает извлечение из git (по умолчанию `~/openclaw`, можно переопределить с помощью `OPENCLAW_GIT_DIR`), переключается на `main`, выполняет rebase относительно вышестоящего репозитория, собирает и устанавливает глобальный CLI из этого извлечения.

Совет: если вы хотите использовать stable и dev параллельно, сохраните два клона и укажите шлюзу на стабильный.

## Однократное указание версии или тега

Используйте `--tag`, чтобы указать конкретный dist-тег, версию или спецификацию пакета для однократного обновления **без** изменения сохранённого канала:

```bash
# Установить конкретную версию
openclaw update --tag 2026.4.1-beta.1

# Установить из бета-dist-тега (однократно, не сохраняется)
openclaw update --tag beta

# Установить из ветки main на GitHub (npm tarball)
openclaw update --tag main

# Установить конкретную спецификацию npm-пакета
openclaw update --tag openclaw@2026.4.1-beta.1
```

Примечания:

- `--tag` применяется **только к установкам пакетов (npm)**. Установки через git игнорируют этот параметр.
- Тег не сохраняется. При следующем выполнении `openclaw update` будет использоваться настроенные канал, как обычно.
- Защита от понижения версии: если целевая версия старше текущей, OpenClaw запросит подтверждение (можно пропустить с помощью `--yes`).
- `--channel beta` отличается от `--tag beta`: при использовании канала возможен переход на stable/latest, если beta отсутствует или старше, тогда как `--tag beta` нацелен на сырой dist-тег `beta` для этого единственного запуска.

## Пробный запуск

Предварительный просмотр того, что сделает `openclaw update`, без внесения изменений:

```bash
openclaw update --dry-run
openclaw update --channel beta --dry-run
openclaw update --tag 2026.4.1-beta.1 --dry-run
openclaw update --dry-run --json
```

Пробный запуск показывает действующий канал, целевую версию, запланированные действия и необходимость подтверждения понижения версии.

## Плагины и каналы

При переключении каналов с помощью `openclaw update` OpenClaw также синхронизирует источники плагинов:

- `dev` предпочитает встроенные плагины из git-извлечения.
- `stable` и `beta` восстанавливают пакеты плагинов, установленные через npm.
- Плагины, установленные через npm, обновляются после завершения обновления ядра.

## Проверка текущего состояния

```bash
openclaw update status
```

Показывает активный канал, тип установки (git или пакет), текущую версию и источник (конфигурация, git-тег, git-ветка или значение по умолчанию).

## Лучшие практики тегирования

- Тегируйте релизы, на которые должны переходить git-извлечения (`vYYYY.M.D` для stable, `vYYYY.M.D-beta.N` для beta).
- `vYYYY.M.D.beta.N` также распознаётся для совместимости, но предпочтительнее использовать `-beta.N`.
- Устаревшие теги `vYYYY.M.D-<patch>` по-прежнему распознаются как stable (не beta).
- Сохраняйте теги неизменяемыми: никогда не перемещайте и не используйте тег повторно.
- Dist-теги npm остаются первоисточником для установок через npm:
  - `latest` → stable;
  - `beta` → кандидат в релизы или стабильная сборка, выпущенная сначала в beta;
  - `dev` → снимок main (опционально).

## Доступность приложения для macOS

Бета-версии и версии dev **могут не** включать релиз приложения для macOS. Это допустимо:

- Git-тег и npm dist-тег всё равно могут быть опубликованы.
- Укажите "нет сборки для macOS для этой бета-версии" в примечаниях к релизу или в журнале изменений.
