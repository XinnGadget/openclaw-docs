# OpenClaw на Oracle Cloud (OCI)

## Цель

Запустить постоянный OpenClaw Gateway на тарифе **Always Free** ARM в Oracle Cloud.

Бесплатный тариф Oracle может хорошо подойти для OpenClaw (особенно если у вас уже есть учётная запись OCI), но у него есть свои ограничения:

- архитектура ARM (большинство компонентов работает, но некоторые бинарные файлы могут быть только для x86);
- ограниченная ёмкость и возможные сложности при регистрации.

## Предварительные требования

- учётная запись Oracle Cloud ([регистрация](https://www.oracle.com/cloud/free/)) — см. [руководство по регистрации в сообществе](https://gist.github.com/rssnyder/51e3cfedd730e7dd5f4a816143b25dbd), если возникнут проблемы;
- учётная запись Tailscale (бесплатно на [tailscale.com](https://tailscale.com));
- около 30 минут времени.

## 1) Создание экземпляра OCI

1. Войдите в [консоль Oracle Cloud](https://cloud.oracle.com/).
2. Перейдите в **Compute → Instances → Create Instance**.
3. Настройте:
   - **Имя:** `openclaw`;
   - **Образ:** Ubuntu 24.04 (aarch64);
   - **Форма:** `VM.Standard.A1.Flex` (Ampere ARM);
   - **OCPU:** 2 (или до 4);
   - **ОЗУ:** 12 ГБ (или до 24 ГБ);
   - **Загрузочный том:** 50 ГБ (до 200 ГБ бесплатно);
   - **SSH-ключ:** добавьте свой публичный ключ.
4. Нажмите **Create**.
5. Запишите публичный IP-адрес.

**Совет:** если создание экземпляра завершается ошибкой "Out of capacity" ("Недостаточно ресурсов"), попробуйте выбрать другой домен доступности или повторите попытку позже. Ёмкость бесплатного тарифа ограничена.

## 2) Подключение и обновление

```bash
# Подключение по публичному IP
ssh ubuntu@ВАШ_ПУБЛИЧНЫЙ_IP

# Обновление системы
sudo apt update && sudo apt upgrade -y
sudo apt install -y build-essential
```

**Примечание:** пакет `build-essential` необходим для компиляции некоторых зависимостей на ARM.

## 3) Настройка пользователя и имени хоста

```bash
# Установка имени хоста
sudo hostnamectl set-hostname openclaw

# Установка пароля для пользователя ubuntu
sudo passwd ubuntu

# Включение lingering (позволяет сервисам пользователя работать после выхода из системы)
sudo loginctl enable-linger ubuntu
```

## 4) Установка Tailscale

```bash
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up --ssh --hostname=openclaw
```

Это активирует Tailscale SSH, благодаря чему вы сможете подключаться через `ssh openclaw` с любого устройства в вашей tailnet — публичный IP не требуется.

Проверка:

```bash
tailscale status
```

**С этого момента подключайтесь через Tailscale:** `ssh ubuntu@openclaw` (или используйте IP-адрес Tailscale).

## 5) Установка OpenClaw

```bash
curl -fsSL https://openclaw.ai/install.sh | bash
source ~/.bashrc
```

Когда появится запрос "How do you want to hatch your bot?" ("Как вы хотите запустить своего бота?"), выберите **"Do this later"** ("Сделать позже").

> Примечание: если возникают проблемы со сборкой для ARM, сначала установите системные пакеты (например, `sudo apt install -y build-essential`), прежде чем обращаться к Homebrew.

## 6) Настройка шлюза (loopback + аутентификация по токену) и включение Tailscale Serve

Используйте аутентификацию по токену в качестве стандартной. Она предсказуема и позволяет избежать необходимости использовать флаги Control UI "insecure auth" ("небезопасная аутентификация").

```bash
# Оставьте шлюз приватным на виртуальной машине
openclaw config set gateway.bind loopback

# Требуйте аутентификацию для шлюза и Control UI
openclaw config set gateway.auth.mode token
openclaw doctor --generate-gateway-token

# Предоставьте доступ через Tailscale Serve (HTTPS + доступ через tailnet)
openclaw config set gateway.tailscale.mode serve
openclaw config set gateway.trustedProxies '["127.0.0.1"]'

systemctl --user restart openclaw-gateway.service
```

`gateway.trustedProxies=["127.0.0.1"]` здесь предназначен только для обработки переадресованного IP/локального клиента прокси Tailscale Serve. Это **не** `gateway.auth.mode: "trusted-proxy"`. В этой настройке маршруты просмотра Diff сохраняют поведение "fail-closed" ("закрыто при ошибке"): необработанные запросы просмотра с `127.0.0.1` без заголовков переадресованного прокси могут возвращать "Diff not found" ("Diff не найден"). Используйте `mode=file` / `mode=both` для вложений или намеренно включите удалённые просмотрщики и задайте `plugins.entries.diffs.config.viewerBaseUrl` (или передайте прокси `baseUrl`), если вам нужны общие ссылки на просмотрщики.

## 7) Проверка

```bash
# Проверка версии
openclaw --version

# Проверка состояния демона
systemctl --user status openclaw-gateway.service

# Проверка Tailscale Serve
tailscale serve status

# Тестирование локального ответа
curl http://localhost:18789
```

## 8) Ограничение безопасности VCN

Теперь, когда всё работает, ограничьте VCN, чтобы заблокировать весь трафик, кроме Tailscale. Виртуальная облачная сеть (VCN) Oracle действует как брандмауэр на границе сети — трафик блокируется до того, как достигнет вашего экземпляра.

1. Перейдите в **Networking → Virtual Cloud Networks** в консоли OCI.
2. Щёлкните по своему VCN → **Security Lists** → Default Security List.
3. **Удалите** все правила входящего трафика, кроме:
   - `0.0.0.0/0 UDP 41641` (Tailscale).
4. Оставьте правила исходящего трафика по умолчанию (разрешить весь исходящий трафик).

Это заблокирует SSH на порту 22, HTTP, HTTPS и всё остальное на границе сети. С этого момента вы сможете подключаться только через Tailscale.

---

## Доступ к Control UI

С любого устройства в вашей сети Tailscale:

```
https://openclaw.<имя_tailnet>.ts.net/
```

Замените `<имя_tailnet>` на имя вашей tailnet (его можно увидеть в `tailscale status`).

Туннель SSH не требуется. Tailscale предоставляет:

- шифрование HTTPS (автоматические сертификаты);
- аутентификацию через идентификатор Tailscale;
- доступ с любого устройства в вашей tailnet (ноутбук, телефон и т. д.).

---

## Безопасность: VCN + Tailscale (рекомендуемый базовый уровень)

При заблокированном VCN (открыт только UDP 41641) и шлюзе, привязанном к loopback, вы получаете надёжную многоуровневую защиту: публичный трафик блокируется на границе сети, а административный доступ осуществляется через вашу tailnet.

Такая настройка часто устраняет **необходимость** в дополнительных правилах брандмауэра на уровне хоста исключительно для предотвращения брутфорса SSH из Интернета — но вам всё равно следует обновлять ОС, запускать `openclaw security audit` и проверять, что вы случайно не слушаете на публичных интерфейсах.

| Традиционный шаг | Требуется? | Почему |
| ------------------ | ----------- | ---------------------------------------------------------------------------- |
| Брандмауэр UFW | Нет | VCN блокирует трафик до того, как он достигнет экземпляра |
