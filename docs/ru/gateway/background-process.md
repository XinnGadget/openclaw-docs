# Инструмент фонового exec и управления процессами

OpenClaw выполняет командные строки через инструмент `exec` и сохраняет долго выполняющиеся задачи в памяти. Инструмент `process` управляет этими фоновыми сессиями.

## Инструмент exec

Ключевые параметры:

- `command` (обязательно)
- `yieldMs` (по умолчанию 10 000): автоматический перевод в фоновый режим после указанной задержки (в миллисекундах)
- `background` (булев): немедленный перевод в фоновый режим
- `timeout` (в секундах, по умолчанию 1800): завершение процесса по истечении указанного времени
- `elevated` (булев): запуск вне песочницы, если включён/разрешён повышенный режим (`gateway` по умолчанию или `node`, если целевая система exec — `node`)
- Если требуется реальный TTY, установите `pty: true`.
- `workdir`, `env`

Поведение:

- При выполнении в переднем плане вывод возвращается напрямую.
- При переводе в фоновый режим (явном или по таймауту) инструмент возвращает `status: "running"` + `sessionId` и краткий хвост вывода.
- Вывод сохраняется в памяти до тех пор, пока сессия не будет опрошена или очищена.
- Если использование инструмента `process` запрещено, `exec` работает синхронно и игнорирует параметры `yieldMs`/`background`.
- Запущенные команды exec получают переменную `OPENCLAW_SHELL=exec` для правил оболочки/профиля с учётом контекста.
- Для долго выполняемой работы, которая начинается сейчас, запустите её один раз и полагайтесь на автоматическое пробуждение по завершении, если оно включено и команда выдаёт вывод или завершается с ошибкой.
- Если автоматическое пробуждение по завершении недоступно или вам нужно подтверждение успешного завершения команды, которая завершилась без вывода, используйте инструмент `process` для подтверждения завершения.
- Не имитируйте напоминания или отложенные последующие действия с помощью циклов `sleep` или повторного опроса; для будущей работы используйте cron.

## Мост для дочерних процессов

При запуске долго выполняющихся дочерних процессов вне инструментов exec/process (например, перезапуск CLI или вспомогательные средства шлюза) подключите вспомогательный компонент моста для дочерних процессов, чтобы сигналы завершения передавались, а слушатели отсоединялись при выходе/ошибке. Это позволяет избежать "осиротевших" процессов в systemd и обеспечивает согласованное поведение при завершении работы на разных платформах.

Переопределения переменных окружения:

- `PI_BASH_YIELD_MS`: задержка по умолчанию (в мс)
- `PI_BASH_MAX_OUTPUT_CHARS`: ограничение объёма вывода в памяти (в символах)
- `OPENCLAW_BASH_PENDING_MAX_OUTPUT_CHARS`: ограничение объёма ожидающего вывода stdout/stderr для каждого потока (в символах)
- `PI_BASH_JOB_TTL_MS`: время жизни завершённых сессий (в мс, ограничено диапазоном 1 мин — 3 ч)

Конфигурация (предпочтительный способ):

- `tools.exec.backgroundMs` (по умолчанию 10 000)
- `tools.exec.timeoutSec` (по умолчанию 1800)
- `tools.exec.cleanupMs` (по умолчанию 1 800 000)
- `tools.exec.notifyOnExit` (по умолчанию true): поставить в очередь системное событие + запросить сердцебиение при завершении фонового exec.
- `tools.exec.notifyOnExitEmptySuccess` (по умолчанию false): если true, также ставить в очередь события о завершении для успешно выполненных фоновых задач, которые не выдали вывод.

## Инструмент process

Действия:

- `list`: список выполняемых и завершённых сессий
- `poll`: получение нового вывода для сессии (также сообщает статус выхода)
- `log`: чтение агрегированного вывода (поддерживает `offset` + `limit`)
- `write`: отправка данных на стандартный ввод (`data`, опционально `eof`)
- `send-keys`: отправка явных ключевых токенов или байтов в сессию с поддержкой PTY
- `submit`: отправка клавиши Enter / возврата каретки в сессию с поддержкой PTY
- `paste`: отправка текста в буквальном виде, опционально в режиме вставки в квадратных скобках
- `kill`: завершение фоновой сессии
- `clear`: удаление завершённой сессии из памяти
- `remove`: завершение, если выполняется, или удаление, если завершена

Примечания:

- В памяти сохраняются только фоновые сессии.
- Сессии теряются при перезапуске процесса (нет сохранения на диске).
- Журналы сессий сохраняются в истории чата только в том случае, если вы выполнили `process poll/log` и результат инструмента был записан.
- Инструмент `process` ограничен областью действия агента: он видит только сессии, запущенные этим агентом.
- Используйте `poll` / `log` для получения статуса, журналов, подтверждения успешного завершения без вывода или подтверждения завершения, если автоматическое пробуждение по завершении недоступно.
- Используйте `write` / `send-keys` / `submit` / `paste` / `kill`, если требуется ввод или вмешательство.
- `process list` включает производное значение `name` (глагол команды + цель) для быстрого просмотра.
- `process log` использует построчное указание `offset`/`limit`.
- Если `offset` и `limit` не указаны, возвращается последние 200 строк и включается подсказка по разбиению на страницы.
- Если указан `offset`, а `limit` не указан, возвращается вывод от `offset` до конца (без ограничения в 200 строк).
- Опрос предназначен для получения статуса по требованию, а не для планирования циклов ожидания. Если работа должна быть выполнена позже, используйте cron.

## Примеры

Запуск долгой задачи и последующий опрос:

```json
{ "tool": "exec", "command": "sleep 5 && echo done", "yieldMs": 1000 }
```

```json
{ "tool": "process", "action": "poll", "sessionId": "<id>" }
```

Немедленный запуск в фоновом режиме:

```json
{ "tool": "exec", "command": "npm run build", "background": true }
```

Отправка данных на стандартный ввод:

```json
{ "tool": "process", "action": "write", "sessionId": "<id>", "data": "y\n" }
```

Отправка клавиш PTY:

```json
{ "tool": "process", "action": "send-keys", "sessionId": "<id>", "keys": ["C-c"] }
```

Отправка текущей строки:

```json
{ "tool": "process", "action": "submit", "sessionId": "<id>" }
```

Вставка текста в буквальном виде:

```json
{ "tool": "process", "action": "paste", "sessionId": "<id>", "text": "line1\nline2\n" }
```
