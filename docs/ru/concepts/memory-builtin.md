# Встроенный механизм памяти

Встроенный механизм — это стандартный бэкенд памяти. Он хранит индекс памяти в базе данных SQLite для каждого агента и не требует дополнительных зависимостей для начала работы.

## Что он предоставляет

- **Поиск по ключевым словам** с использованием полнотекстового индексирования FTS5 (оценка BM25).
- **Векторный поиск** с использованием эмбеддингов от любого поддерживаемого провайдера.
- **Гибридный поиск**, сочетающий оба метода для достижения наилучших результатов.
- **Поддержка CJK** с использованием триграммной токенизации для китайского, японского и корейского языков.
- **Ускорение sqlite-vec** для векторных запросов в базе данных (опционально).

## Начало работы

Если у вас есть API-ключ для OpenAI, Gemini, Voyage или Mistral, встроенный механизм автоматически обнаружит его и включит векторный поиск. Дополнительная настройка не требуется.

Чтобы явно указать провайдера:

```json5
{
  agents: {
    defaults: {
      memorySearch: {
        provider: "openai",
      },
    },
  },
}
```

Без провайдера эмбеддингов доступен только поиск по ключевым словам.

## Поддерживаемые провайдеры эмбеддингов

| Провайдер | ID | Автообнаружение | Примечания |
| -------- | --------- | ------------- | ----------------------------------- |
| OpenAI | `openai` | Да | По умолчанию: `text-embedding-3-small` |
| Gemini | `gemini` | Да | Поддерживает мультимодальность (изображение + аудио) |
| Voyage | `voyage` | Да | |
| Mistral | `mistral` | Да | |
| Ollama | `ollama` | Нет | Локальный, нужно указывать явно |
| Local | `local` | Да (первый) | Модель GGUF, загрузка ~0,6 ГБ |

Автообнаружение выбирает первого провайдера, API-ключ которого удаётся разрешить, в порядке, показанном в таблице. Чтобы переопределить выбор, задайте параметр `memorySearch.provider`.

## Как работает индексирование

OpenClaw разбивает файлы `MEMORY.md` и `memory/*.md` на фрагменты (~400 токенов с перекрытием в 80 токенов) и сохраняет их в базе данных SQLite для каждого агента.

- **Расположение индекса:** `~/.openclaw/memory/<agentId>.sqlite`
- **Отслеживание файлов:** изменения в файлах памяти запускают отложенное переиндексирование (1,5 с).
- **Автоматическое переиндексирование:** при изменении провайдера эмбеддингов, модели или конфигурации разбиения на фрагменты весь индекс перестраивается автоматически.
- **Переиндексирование по требованию:** `openclaw memory index --force`

<Info>
Вы также можете индексировать файлы Markdown вне рабочей области с помощью параметра `memorySearch.extraPaths`. См. [справочник по конфигурации](/reference/memory-config#additional-memory-paths).
</Info>

## Когда использовать

Встроенный механизм подойдёт большинству пользователей:

- Работает "из коробки" без дополнительных зависимостей.
- Хорошо справляется с поиском по ключевым словам и векторным поиском.
- Поддерживает всех провайдеров эмбеддингов.
- Гибридный поиск сочетает лучшие стороны обоих подходов к извлечению данных.

Рассмотрите переход на [QMD](/concepts/memory-qmd), если вам нужны переранжирование, расширение запросов или индексация каталогов вне рабочей области.

Рассмотрите [Honcho](/concepts/memory-honcho), если вам нужна память между сессиями с автоматическим моделированием пользователя.

## Устранение неполадок

**Поиск в памяти отключён?** Проверьте `openclaw memory status`. Если провайдер не обнаружен, укажите его явно или добавьте API-ключ.

**Устаревшие результаты?** Запустите `openclaw memory index --force`, чтобы перестроить индекс. В редких пограничных случаях наблюдатель может не заметить изменения.

**sqlite-vec не загружается?** OpenClaw автоматически переключается на вычисление косинусного сходства в процессе. Проверьте логи на наличие конкретной ошибки загрузки.
