# Жизненный цикл голосового оверлея (macOS)

Аудитория: разработчики приложений для macOS. Цель: обеспечить предсказуемое поведение голосового оверлея при перекрытии wake-word и push-to-talk.

## Текущее намерение

- Если оверлей уже виден (запущен через wake-word) и пользователь нажимает горячую клавишу, сессия горячей клавиши _принимает_ существующий текст, а не сбрасывает его. Оверлей остаётся активным, пока удерживается горячая клавиша. При отпускании клавиши: отправить текст, если он есть после обрезки, в противном случае закрыть оверлей.
- Wake-word по‑прежнему автоматически отправляет текст при тишине; push-to-talk отправляет текст сразу при отпускании клавиши.

## Реализовано (9 декабря 2025 г.)

- Сессии оверлея теперь содержат токен для каждого захвата (wake-word или push-to-talk). Обновления partial/final/send/dismiss/level отбрасываются, если токен не совпадает, что позволяет избежать устаревших колбэков.
- Push-to-talk принимает любой видимый текст оверлея в качестве префикса (то есть при нажатии горячей клавиши, когда активен оверлей wake-word, текст сохраняется и к нему добавляется новая речь). Ожидает до 1,5 с для получения окончательного транскрипта, после чего возвращается к текущему тексту.
- Логирование сигналов/оверлея выводится на уровне `info` в категориях `voicewake.overlay`, `voicewake.ptt` и `voicewake.chime` (начало сессии, partial, final, send, dismiss, причина сигнала).

## Следующие шаги

1. **VoiceSessionCoordinator (актор)**
   - В каждый момент времени управляет ровно одной `VoiceSession`.
   - API (на основе токенов): `beginWakeCapture`, `beginPushToTalk`, `updatePartial`, `endCapture`, `cancel`, `applyCooldown`.
   - Отбрасывает колбэки с устаревшими токенами (предотвращает повторное открытие оверлея старыми распознавателями).
2. **VoiceSession (модель)**
   - Поля: `token`, `source` (wakeWord|pushToTalk), зафиксированный/временный текст, флаги сигналов, таймеры (автоотправка, бездействие), `overlayMode` (display|editing|sending), крайний срок охлаждения.
3. **Привязка оверлея**
   - `VoiceSessionPublisher` (`ObservableObject`) отражает активную сессию в SwiftUI.
   - `VoiceWakeOverlayView` отображает содержимое только через publisher; напрямую не изменяет глобальные синглтоны.
   - Действия пользователя с оверлеем (`sendNow`, `dismiss`, `edit`) вызывают координатор с токеном сессии.
4. **Унифицированный путь отправки**
   - При `endCapture`: если обрезанный текст пуст → закрыть оверлей; в противном случае `performSend(session:)` (один раз воспроизводит сигнал отправки, пересылает текст, закрывает оверлей).
   - Push-to-talk: без задержки; wake-word: необязательная задержка для автоотправки.
   - После завершения push-to-talk применять короткий период охлаждения для wake-word, чтобы предотвратить немедленное повторное срабатывание.
5. **Логирование**
   - Координатор выводит логи уровня `.info` в подсистеме `ai.openclaw`, категории `voicewake.overlay` и `voicewake.chime`.
   - Ключевые события: `session_started`, `adopted_by_push_to_talk`, `partial`, `finalized`, `send`, `dismiss`, `cancel`, `cooldown`.

## Чек‑лист для отладки

- Выводите логи при воспроизведении проблемы с "зависающим" оверлеем:

  ```bash
  sudo log stream --predicate 'subsystem == "ai.openclaw" AND category CONTAINS "voicewake"' --level info --style compact
  ```

- Убедитесь, что активен только один токен сессии; устаревшие колбэки должны отбрасываться координатором.
- Проверьте, что при отпускании клавиши push-to-talk всегда вызывается `endCapture` с активным токеном; если текст пуст, ожидается `dismiss` без сигнала или отправки.

## Шаги миграции (рекомендуемые)

1. Добавить `VoiceSessionCoordinator`, `VoiceSession` и `VoiceSessionPublisher`.
2. Переработать `VoiceWakeRuntime`, чтобы создавать/обновлять/завершать сессии вместо прямого взаимодействия с `VoiceWakeOverlayController`.
3. Переработать `VoicePushToTalk`, чтобы принимать существующие сессии и вызывать `endCapture` при отпускании клавиши; применять период охлаждения для runtime.
4. Подключить `VoiceWakeOverlayController` к publisher; убрать прямые вызовы из runtime/PTT.
5. Добавить интеграционные тесты для принятия сессий, периода охлаждения и закрытия при пустом тексте.
