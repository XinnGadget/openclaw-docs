# WhatsApp (веб-канал)

Статус: готов к эксплуатации через WhatsApp Web (Baileys). Шлюз управляет связанными сессиями.

## Установка (по требованию)

- Онбординг (`openclaw onboard`) и команда `openclaw channels add --channel whatsapp` предложат установить плагин WhatsApp при первом выборе этого канала.
- Команда `openclaw channels login --channel whatsapp` также запускает процесс установки, если плагин ещё не установлен.
- Канал для разработчиков + git checkout: по умолчанию используется локальный путь к плагину.
- Стабильная/бета-версия: по умолчанию используется npm-пакет `@openclaw/whatsapp`.

Ручная установка остаётся доступной:

```bash
openclaw plugins install @openclaw/whatsapp
```

## Быстрая настройка

<Steps>
  <Step title="Настроить политику доступа к WhatsApp">

```json5
{
  channels: {
    whatsapp: {
      dmPolicy: "pairing",
      allowFrom: ["+15551234567"],
      groupPolicy: "allowlist",
      groupAllowFrom: ["+15551234567"],
    },
  },
}
```

  </Step>

  <Step title="Связать WhatsApp (QR-код)">

```bash
openclaw channels login --channel whatsapp
```

    Для конкретного аккаунта:

```bash
openclaw channels login --channel whatsapp --account work
```

  </Step>

  <Step title="Запустить шлюз">

```bash
openclaw gateway
```

  </Step>

  <Step title="Одобрить первый запрос на сопряжение (если используется режим сопряжения)">

```bash
openclaw pairing list whatsapp
openclaw pairing approve whatsapp <CODE>
```

    Запросы на сопряжение истекают через 1 час. Количество ожидающих запросов ограничено 3 на канал.

  </Step>
</Steps>

<Note>
OpenClaw рекомендует по возможности использовать WhatsApp на отдельном номере (метаданные канала и процесс настройки оптимизированы для такой конфигурации, но поддержка личных номеров также предусмотрена).
</Note>

## Модель выполнения

- Шлюз управляет сокетом WhatsApp и циклом переподключения.
- Для исходящих сообщений требуется активный слушатель WhatsApp для целевого аккаунта.
- Чаты статуса и рассылки игнорируются (`@status`, `@broadcast`).
- В прямых чатах используются правила сессий DM (`session.dmScope`; по умолчанию `main` объединяет DM в основную сессию агента).
- Групповые сессии изолированы (`agent:<agentId>:whatsapp:group:<jid>`).
- Транспорт WhatsApp Web учитывает стандартные переменные окружения прокси на хосте шлюза (`HTTPS_PROXY`, `HTTP_PROXY`, `NO_PROXY` / варианты в нижнем регистре). Предпочитайте настройку прокси на уровне хоста специфическим настройкам прокси WhatsApp для канала.

## Управление доступом и активация

<Tabs>
  <Tab title="Политика прямых сообщений">
    `channels.whatsapp.dmPolicy` управляет доступом к прямым чатам:

    - `pairing` (по умолчанию);
    - `allowlist`;
    - `open` (требует, чтобы `allowFrom` включал `"*"`);
    - `disabled`.

    `allowFrom` принимает номера в формате E.164 (нормализуются внутри системы).

    Переопределение для нескольких аккаунтов: `channels.whatsapp.accounts.<id>.dmPolicy` (и `allowFrom`) имеют приоритет над настройками канала по умолчанию для этого аккаунта.

    Детали поведения во время выполнения:

    - сопряжения сохраняются в хранилище разрешённых каналов и объединяются с настроенным `allowFrom`;
    - если список разрешённых не настроен, связанный личный номер разрешён по умолчанию;
    - исходящие DM `fromMe` никогда не сопрягаются автоматически.

  </Tab>

  <Tab title="Политика групп + списки разрешённых">
    Доступ к группам имеет два уровня:

    1. **Список разрешённых участников группы** (`channels.whatsapp.groups`):
       - если `groups` не указан, все группы допустимы;
       - если `groups` указан, он действует как список разрешённых групп (`"*"` разрешено).

    2. **Политика отправителей групп** (`channels.whatsapp.groupPolicy` + `groupAllowFrom`):
       - `open`: список разрешённых отправителей игнорируется;
       - `allowlist`: отправитель должен соответствовать `groupAllowFrom` (или `*`);
       - `disabled`: блокировка всех входящих сообщений в группах.

    Резервный вариант для списка разрешённых отправителей:

    - если `groupAllowFrom` не задан, во время выполнения используется `allowFrom`, если он доступен;
    - списки разрешённых отправителей проверяются до активации упоминаний/ответов.

    Примечание: если блок `channels.whatsapp` отсутствует вообще, резервная политика групп во время выполнения — `allowlist` (с предупреждением в логах), даже если задана `channels.defaults.groupPolicy`.

  </Tab>

  <Tab title="Упоминания + /активация">
    Для ответов в группах по умолчанию требуется упоминание.

    Обнаружение упоминаний включает:

    - явные упоминания бота в WhatsApp;
    - настроенные шаблоны регулярных выражений для упоминаний (`agents.list[].groupChat.mentionPatterns`, резервный вариант `messages.groupChat.mentionPatterns`);
    - неявное обнаружение ответов боту (отправитель ответа соответствует идентификатору бота).

    Примечание по безопасности:

    - цитата/ответ удовлетворяет условию упоминания, но **не** предоставляет авторизацию отправителя;
    - при `groupPolicy: "allowlist"` отправители, не входящие в список разрешённых, блокируются, даже если они отвечают на сообщение пользователя из списка разрешённых.

    Команда активации на уровне сессии:

    - `/activation mention`;
    - `/activation always`.

    `activation` обновляет состояние сессии (не глобальную конфигурацию). Доступ ограничен владельцем.

  </Tab>
</Tabs>

## Поведение с личным номером и самочатом

Когда связанный личный номер также присутствует в `allowFrom`, активируются механизмы защиты самочата WhatsApp:

- пропуск уведомлений о прочтении для сообщений самочата;
- игнорирование автоматического срабатывания по JID упоминания, которое в противном случае могло бы отправить уведомление самому себе;
- если `messages.responsePrefix` не задан, ответы в самочате по умолчанию используют `[{identity.name}]` или `[openclaw]`.
