# Генерация музыки

Инструмент `music_generate` позволяет агенту создавать музыку или аудио за счёт общей функциональности генерации музыки с настроенными провайдерами, такими как Google, MiniMax и ComfyUI с настроенным рабочим процессом.

Для сессий агента с поддержкой общих провайдеров OpenClaw запускает генерацию музыки как фоновую задачу, отслеживает её в реестре задач, а затем снова активирует агента, когда трек готов, — чтобы агент мог отправить готовый аудиофайл обратно в исходный канал.

<Note>
Встроенный общий инструмент отображается только при наличии хотя бы одного провайдера генерации музыки. Если вы не видите `music_generate` среди инструментов вашего агента, настройте `agents.defaults.musicGenerationModel` или задайте API-ключ провайдера.
</Note>

## Быстрый старт

### Генерация с поддержкой общего провайдера

1. Задайте API-ключ хотя бы для одного провайдера, например `GEMINI_API_KEY` или `MINIMAX_API_KEY`.
2. При необходимости задайте предпочтительную модель:

```json5
{
  agents: {
    defaults: {
      musicGenerationModel: {
        primary: "google/lyria-3-clip-preview",
      },
    },
  },
}
```

3. Попросите агента: "Сгенерируй энергичный synthpop-трек о ночной поездке по неоновому городу".

Агент автоматически вызывает `music_generate`. Разрешения для инструмента не требуются.

Для прямых синхронных контекстов без запуска агента с поддержкой сессии встроенный инструмент по-прежнему переключается на встроенную генерацию и возвращает конечный путь к медиафайлу в результате работы инструмента.

Примеры промтов:

```text
Сгенерируй кинематографичный фортепианный трек с мягкими струнными и без вокала.
```

```text
Сгенерируй энергичный chiptune-луп о запуске ракеты на рассвете.
```

### Генерация через рабочий процесс Comfy

Встроенный плагин `comfy` подключается к общему инструменту `music_generate` через реестр провайдеров генерации музыки.

1. Настройте `models.providers.comfy.music` с JSON-описанием рабочего процесса и узлами промта/вывода.
2. Если вы используете Comfy Cloud, задайте `COMFY_API_KEY` или `COMFY_CLOUD_API_KEY`.
3. Попросите агента сгенерировать музыку или вызовите инструмент напрямую.

Пример:

```text
/tool music_generate prompt="Тёплый эмбиент-синтезаторный луп с мягкой текстурой магнитной ленты"
```

## Поддержка встроенных общих провайдеров

| Провайдер | Модель по умолчанию | Входные данные для справки | Поддерживаемые параметры управления | API-ключ |
| -------- | ---------------------- | ---------------- | --------------------------------------------------------- | -------------------------------------- |
| ComfyUI  | `workflow`             | До 1 изображения | Музыка или аудио, определённые рабочим процессом | `COMFY_API_KEY`, `COMFY_CLOUD_API_KEY` |
| Google   | `lyria-3-clip-preview` | До 10 изображений | `lyrics`, `instrumental`, `format` | `GEMINI_API_KEY`, `GOOGLE_API_KEY` |
| MiniMax  | `music-2.5+`           | Нет | `lyrics`, `instrumental`, `durationSeconds`, `format=mp3` | `MINIMAX_API_KEY` |

### Матрица объявленных возможностей

Это явный контракт режима, используемый `music_generate`, тестами контрактов и общим живым проходом.

| Провайдер | `generate` | `edit` | Лимит на редактирование | Общие живые каналы |
| -------- | ---------- | ------ | ---------- | ------------------------------------------------------------------------- |
| ComfyUI  | Да | Да | 1 изображение | Не входит в общий проход; покрывается `extensions/comfy/comfy.live.test.ts` |
| Google   | Да | Да | 10 изображений | `generate`, `edit` |
| MiniMax  | Да | Нет | Нет | `generate` |

Используйте `action: "list"`, чтобы проверить доступные общие провайдеры и модели во время выполнения:

```text
/tool music_generate action=list
```

Используйте `action: "status"`, чтобы проверить активную задачу генерации музыки с поддержкой сессии:

```text
/tool music_generate action=status
```

Пример прямой генерации:

```text
/tool music_generate prompt="Мечтательный lo-fi hip hop с текстурой винила и лёгким дождём" instrumental=true
```

## Параметры встроенного инструмента

| Параметр | Тип | Описание |
| ----------------- | -------- | ------------------------------------------------------------------------------------------------- |
| `prompt` | string | Промт для генерации музыки (обязателен для `action: "generate"`) |
| `action` | string | `"generate"` (по умолчанию), `"status"` для текущей задачи сессии или `"list"` для проверки провайдеров |
| `model` | string | Переопределение провайдера/модели, например `google/lyria-3-pro-preview` или `comfy/workflow` |
| `lyrics` | string | Необязательные тексты песен, если провайдер поддерживает явный ввод текста |
| `instrumental` | boolean | Запрос вывода только инструментальной версии, если провайдер это поддерживает |
| `image` | string | Путь к одному справочному изображению или URL |
| `images` | string[] | Несколько справочных изображений (до 10) |
| `durationSeconds` | number | Целевая длительность в секундах, если провайдер поддерживает подсказки по длительности |
| `format` | string | Подсказка по формату вывода (`mp3` или `wav`), если провайдер это поддерживает |
| `filename` | string | Подсказка по имени выходного файла |

Не все провайдеры поддерживают все параметры. OpenClaw по-прежнему проверяет жёсткие ограничения, например количество входных данных, перед отправкой. Если провайдер поддерживает длительность, но использует максимальное значение меньше запрошенного, OpenClaw автоматически ограничивает длительность до ближайшего поддерживаемого значения. Действительно неподдерживаемые необязательные подсказки игнорируются с предупреждением, если выбранный провайдер или модель не могут их обработать.

Результаты работы инструмента сообщают о применённых настройках. Если OpenClaw ограничивает длительность при переключении на резервный провайдер, возвращаемое значение `durationSeconds` отражает отправленное значение, а `details.normalization.durationSeconds` показывает сопоставление запрошенного и применённого значений.

## Асинхронное поведение для пути с поддержкой общего провайдера

* Запуски агента с поддержкой сессии: `music_generate` создаёт фоновую задачу, немедленно возвращает ответ о запуске/задаче и отправляет готовый трек позже в последующем сообщении агента.
* Предотвращение дублирования: пока фоновая задача находится в состоянии `queued` или `running`, последующие вызовы `music_generate` в той же сессии возвращают статус задачи вместо запуска новой генерации.
* Поиск статуса: используйте `action: "status"`, чтобы проверить активную задачу генерации музыки с поддержкой сессии, не запуская новую.
* Отслеживание задач: используйте `openclaw tasks list` или `openclaw tasks show <taskId>`, чтобы проверить состояния очереди, выполнения и завершения генерации.
* Активация по завершении: OpenClaw внедряет внутреннее событие завершения обратно в ту же сессию, чтобы модель могла самостоятельно сформировать последующее сообщение для пользователя.
* Подсказка по промту: в последующих пользовательских/ручных шагах в той же сессии появляется небольшая подсказка во время выполнения задачи генерации музыки, чтобы модель не вызывала `music_generate` повторно вслепую.
* Откат при отсутствии сессии: прямые/локальные контексты без реальной сессии агента по-прежнему выполняются встроенно и возвращают конечный аудиорезультат в том же шаге.

### Жизненный цикл задачи

Каждый запрос `music_generate` проходит четыре состояния:

1. **queued** — задача создана, ожидает принятия провайдером.
2. **running** — провайдер обрабатывает запрос (обычно от 30 секунд до 3 минут, в зависимости от провайдера и длительности).
3. **succeeded** — трек готов; агент активируется и отправляет его в диалог.
4. **failed** — ошибка провайдера или тайм-аут; агент активируется с деталями ошибки.

Проверка статуса из CLI:

```bash
openclaw tasks list
openclaw tasks show <taskId>
openclaw tasks cancel <taskId>
```

Предотвращение дублирования: если задача генерации музыки уже находится в состоянии `queued` или `running` для текущей сессии, `music_generate` возвращает статус существующей задачи вместо запуска новой. Используйте `action: "status"`, чтобы явно проверить статус без запуска новой генерации.

## Настройка

### Выбор модели

```json5
{
  agents: {
    defaults: {
      musicGenerationModel: {
        primary: "google/lyria-3-clip-preview",
        fallbacks: ["minimax/music-2.5+"],
      },
    },
  },
}
```

### Порядок выбора провайдера

При генерации музыки OpenClaw пытается использовать провайдеры в следующем порядке:

1. Параметр `model` из вызова инструмента, если агент его указал.
2. `musicGenerationModel.primary` из конфигурации.
3. `musicGenerationModel.fallbacks` по порядку.
4. Автоматическое обнаружение с использованием только настроек провайдера с аутентификацией:
   * сначала текущий провайдер по умолчанию;
   * затем остальные зарегистрированные провайдеры генерации музыки в порядке идентификаторов провайдера.

Если провайдер выдаёт ошибку, автоматически пробуется следующий кандидат. Если все попытки завершились ошибкой, в сообщении об ошибке будут детали каждой попытки.

Задайте `agents.defaults.mediaGenerationAutoProviderFallback: false`, если хотите, чтобы генерация музыки использовала только явные записи `model`, `primary` и `fallbacks`.

## Примечания к провайдерам

* Google использует пакетную генерацию Lyria 3. Текущий встроенный поток поддерживает промт, необязательный текст песен и необязательные справочные изображения.
* MiniMax использует пакетную конечную точку `music_generation`. Текущий встроенный поток поддерживает промт, необязательные тексты песен, инструментальный режим, управление длительностью и вывод в формате mp3.
* Поддержка ComfyUI зависит от рабочего процесса и настроенного графа плюс сопоставления узлов для полей промта/вывода.

## Режимы возможностей провайдера

Общий контракт генерации музыки теперь поддерживает явные объявления режимов:

* `generate` для генерации только по промту;
* `edit`, если запрос включает одно или несколько справочных изображений.

Новые реализации провайдера должны отдавать предпочтение явным блокам режимов:

```typescript
capabilities: {
  generate: {
    maxTracks: 1,
    supportsLyrics: true,
    supportsFormat: true,
  },
  edit: {
    enabled: true,
    maxTracks: 1,
    maxInputImages: 1,
    supportsFormat: true,
  },
}
```

Устаревшие плоские поля, такие как `maxInputImages`, `supportsLyrics` и `supportsFormat`, недостаточны для объявления поддержки редактирования. Провайдеры должны явно объявлять `generate` и `edit`, чтобы живые тесты, тесты контрактов и общий инструмент `music_generate` могли детерминированно проверять поддержку режимов.

## Выбор правильного пути

* Используйте путь с поддержкой общего провайдера, если вам нужен выбор модели, откат на резервный провайдер и встроенный асинхронный поток задач/статусов.
* Используйте путь через плагин, например ComfyUI, если вам нужен пользовательский граф рабочего процесса или провайдер, который не входит в общую встроенную функциональность генерации музыки.
* Если вы отлаживаете поведение, специфичное для ComfyUI, см. [ComfyUI](/providers/comfy). Если вы отлаживаете поведение общего провайдера, начните с [Google (Gemini)](/providers/google) или [MiniMax](/providers/minimax).

## Живые тесты

Участие в живом покрытии для встроенных общих провайдеров:

```bash
OPENCLAW_LIVE_TEST=1 pnpm test:live -- extensions/music-generation-providers.live.test.ts
```

Обёртка репозитория:

```bash
pnpm test:live:media music
```

Этот живой файл загружает отсутствующие переменные окружения провайдера из `~/.profile`, по умолчанию отдаёт предпочтение API-ключам из среды выполнения перед сохранёнными профилями аутентификации и запускает покрытие `generate` и объявленного `edit`, если провайдер включает режим редактирования.

На текущий момент это означает:

* `google`: `generate` плюс `edit`;
* `minimax`: только `generate`;
* `comfy`: отдельное живое покрытие Comfy, не общий проход провайдера.

Участие в живом покрытии для встроенного музыкального пути ComfyUI:

```bash
OPENCLAW_LIVE_TEST=1 COMFY_LIVE_TEST=1 pnpm test:live -- extensions/comfy/comfy.live.test.ts
```

Живой файл Comfy также покрывает рабочие процессы с изображениями и видео, если эти разделы настроены.
