# LINE

LINE подключается к OpenClaw через LINE Messaging API. Плагин работает как приёмник вебхуков на шлюзе и использует токен доступа к каналу и секретный ключ канала для аутентификации.

Статус: плагин включён в комплект. Поддерживаются прямые сообщения, групповые чаты, медиафайлы, местоположения, Flex-сообщения, шаблонные сообщения и быстрые ответы. Реакции и потоки не поддерживаются.

## Плагин, включённый в комплект

В текущих версиях OpenClaw плагин LINE включён в комплект, поэтому для обычных упакованных сборок отдельная установка не требуется.

Если вы используете более старую сборку или пользовательскую установку, в которой отсутствует LINE, установите плагин вручную:

```bash
openclaw plugins install @openclaw/line
```

Локальная проверка (при запуске из репозитория Git):

```bash
openclaw plugins install ./path/to/local/line-plugin
```

## Настройка

1. Создайте учётную запись LINE Developers и откройте консоль:
   [https://developers.line.biz/console/](https://developers.line.biz/console/)
2. Создайте (или выберите) провайдера и добавьте канал **Messaging API**.
3. Скопируйте **токен доступа к каналу** и **секретный ключ канала** из настроек канала.
4. В настройках Messaging API включите **Use webhook**.
5. Установите URL вебхука для конечной точки вашего шлюза (требуется HTTPS):

```
https://gateway-host/line/webhook
```

Шлюз отвечает на проверку вебхука LINE (GET) и входящие события (POST).
Если вам нужен пользовательский путь, задайте `channels.line.webhookPath` или `channels.line.accounts.<id>.webhookPath` и соответствующим образом обновите URL.

Примечание по безопасности:

- Проверка подписи LINE зависит от тела запроса (HMAC для необработанного тела), поэтому OpenClaw применяет строгие ограничения на размер тела и тайм-аут перед проверкой.
- OpenClaw обрабатывает события вебхука из проверенных необработанных байтов запроса. Значения `req.body`, преобразованные промежуточным ПО вышестоящего уровня, игнорируются для обеспечения целостности подписи.

## Конфигурация

Минимальная конфигурация:

```json5
{
  channels: {
    line: {
      enabled: true,
      channelAccessToken: "LINE_CHANNEL_ACCESS_TOKEN",
      channelSecret: "LINE_CHANNEL_SECRET",
      dmPolicy: "pairing",
    },
  },
}
```

Переменные окружения (только для учётной записи по умолчанию):

- `LINE_CHANNEL_ACCESS_TOKEN`
- `LINE_CHANNEL_SECRET`

Файлы с токенами/секретными ключами:

```json5
{
  channels: {
    line: {
      tokenFile: "/path/to/line-token.txt",
      secretFile: "/path/to/line-secret.txt",
    },
  },
}
```

`tokenFile` и `secretFile` должны указывать на обычные файлы. Символические ссылки не допускаются.

Несколько учётных записей:

```json5
{
  channels: {
    line: {
      accounts: {
        marketing: {
          channelAccessToken: "...",
          channelSecret: "...",
          webhookPath: "/line/marketing",
        },
      },
    },
  },
}
```

## Контроль доступа

По умолчанию для прямых сообщений используется сопоставление. Неизвестные отправители получают код сопоставления, и их сообщения игнорируются до утверждения.

```bash
openclaw pairing list line
openclaw pairing approve line <CODE>
```

Списки разрешений и политики:

- `channels.line.dmPolicy`: `pairing | allowlist | open | disabled`
- `channels.line.allowFrom`: разрешённые идентификаторы пользователей LINE для прямых сообщений
- `channels.line.groupPolicy`: `allowlist | open | disabled`
- `channels.line.groupAllowFrom`: разрешённые идентификаторы пользователей LINE для групп
- Переопределения для отдельных групп: `channels.line.groups.<groupId>.allowFrom`
- Примечание о времени выполнения: если раздел `channels.line` полностью отсутствует, во время выполнения для проверок групп используется `groupPolicy="allowlist"` (даже если задан `channels.defaults.groupPolicy`).

Идентификаторы LINE чувствительны к регистру. Допустимые идентификаторы выглядят следующим образом:

- Пользователь: `U` + 32 шестнадцатеричных символа
- Группа: `C` + 32 шестнадцатеричных символа
- Комната: `R` + 32 шестнадцатеричных символа

## Поведение сообщений

- Текст разбивается на фрагменты по 5000 символов.
- Форматирование Markdown удаляется; блоки кода и таблицы по возможности преобразуются в Flex-карты.
- Потоковые ответы буферизуются; LINE получает полные фрагменты с анимацией загрузки, пока агент работает.
- Загрузка медиафайлов ограничена параметром `channels.line.mediaMaxMb` (по умолчанию — 10).

## Данные канала (расширенные сообщения)

Используйте `channelData.line` для отправки быстрых ответов, местоположений, Flex-карт или шаблонных сообщений.

```json5
{
  text: "Вот, пожалуйста",
  channelData: {
    line: {
      quickReplies: ["Статус", "Помощь"],
      location: {
        title: "Офис",
        address: "ул. Главная, 123",
        latitude: 35.681236,
        longitude: 139.767125,
      },
      flexMessage: {
        altText: "Карта статуса",
        contents: {
          /* Полезная нагрузка Flex */
        },
      },
      templateMessage: {
        type: "confirm",
        text: "Продолжить?",
        confirmLabel: "Да",
        confirmData: "yes",
        cancelLabel: "Нет",
        cancelData: "no",
      },
    },
  },
}
```

Плагин LINE также включает команду `/card` для предустановок Flex-сообщений:

```
/card info "Добро пожаловать" "Спасибо за присоединение!"
```

## Поддержка ACP

LINE поддерживает привязки разговоров ACP (Agent Communication Protocol):

- `/acp spawn <agent> --bind here` привязывает текущий чат LINE к сессии ACP без создания дочернего потока.
- Настроенные привязки ACP и активные сессии ACP, привязанные к разговору, работают в LINE так же, как и в других каналах разговоров.

Подробнее см. в разделе [ACP agents](/tools/acp-agents).

## Исходящие медиафайлы

Плагин LINE поддерживает отправку изображений, видео и аудиофайлов через инструмент сообщений агента. Медиафайлы отправляются через путь доставки, специфичный для LINE, с соответствующей обработкой предварительного просмотра и отслеживания:

- **Изображения**: отправляются как изображения LINE с автоматической генерацией предварительного просмотра.
- **Видео**: отправляются с явной обработкой предварительного просмотра и типа содержимого.
- **Аудио**: отправляются как аудиосообщения LINE.

При отправке общих медиафайлов используется существующий маршрут только для изображений, если путь, специфичный для LINE, недоступен.

## Устранение неполадок

- **Проверка вебхука не удаётся:** убедитесь, что URL вебхука использует HTTPS и что `channelSecret` соответствует значению в консоли LINE.
- **Нет входящих событий:** проверьте, что путь вебхука соответствует `channels.line.webhookPath` и что шлюз доступен из LINE.
- **Ошибки загрузки медиафайлов:** увеличьте значение `channels.line.mediaMaxMb`, если размер медиафайла превышает ограничение по умолчанию.
