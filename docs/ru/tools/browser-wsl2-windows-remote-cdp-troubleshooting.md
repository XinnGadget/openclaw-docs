# Устранение неполадок: WSL2 + Windows + удалённый Chrome CDP

В этом руководстве рассматриваются типичные конфигурации с разделённым хостом, в которых:

- OpenClaw Gateway работает внутри WSL2;
- Chrome работает в Windows;
- управление браузером должно пересекать границу WSL2/Windows.

Также в руководстве разбирается многоуровневый шаблон сбоев из [проблемы № 39369](https://github.com/openclaw/openclaw/issues/39369): одновременно может проявиться несколько независимых проблем, из-за чего сначала может показаться, что неисправен не тот уровень.

## Сначала выберите правильный режим работы браузера

Есть два допустимых варианта:

### Вариант 1: Сырой удалённый CDP из WSL2 в Windows

Используйте профиль удалённого браузера, который указывает из WSL2 на конечную точку CDP в Chrome для Windows.

Выберите этот вариант, если:

- Gateway остаётся внутри WSL2;
- Chrome работает в Windows;
- требуется, чтобы управление браузером пересекало границу WSL2/Windows.

### Вариант 2: Локальный для хоста Chrome MCP

Используйте `existing-session` / `user` только в том случае, если сам Gateway работает на том же хосте, что и Chrome.

Выберите этот вариант, если:

- OpenClaw и Chrome находятся на одном компьютере;
- вам нужно локальное состояние браузера с авторизацией;
- вам не требуется транспорт браузера между хостами;
- вам не нужны расширенные маршруты только для управляемого/сырого CDP, такие как `responsebody`, экспорт PDF, перехват загрузок или пакетные действия.

Для конфигурации WSL2 Gateway + Chrome для Windows предпочтительнее сырой удалённый CDP. Chrome MCP работает локально для хоста, а не как мост между WSL2 и Windows.

## Рабочая архитектура

Примерная конфигурация:

- WSL2 запускает Gateway на `127.0.0.1:18789`;
- Windows открывает контрольный UI в обычном браузере по адресу `http://127.0.0.1:18789/`;
- Chrome в Windows предоставляет конечную точку CDP на порту `9222`;
- WSL2 может получить доступ к этой конечной точке CDP в Windows;
- OpenClaw указывает профилю браузера адрес, доступный из WSL2.

## Почему эта конфигурация вызывает затруднения

Может возникнуть несколько перекрывающихся сбоев:

- WSL2 не может получить доступ к конечной точке CDP в Windows;
- контрольный UI открыт из небезопасного источника;
- `gateway.controlUi.allowedOrigins` не соответствует источнику страницы;
- отсутствует токен или пара;
- профиль браузера указывает на неверный адрес.

Из-за этого устранение проблемы на одном уровне может не убрать ошибку на другом уровне.

## Важное правило для контрольного UI

Когда UI открывается из Windows, используйте localhost для Windows, если у вас нет специально настроенного HTTPS.

Используйте:

`http://127.0.0.1:18789/`

Не используйте по умолчанию IP-адрес локальной сети для контрольного UI. Обычный HTTP на IP-адресе локальной сети или tailnet может вызвать поведение, связанное с небезопасным источником/авторизацией устройства, которое не имеет отношения к самому CDP. См. [Контрольный UI](/web/control-ui).

## Проверка по уровням

Работайте сверху вниз. Не пропускайте шаги.

### Уровень 1: Проверьте, что Chrome предоставляет CDP в Windows

Запустите Chrome в Windows с включённой удалённой отладкой:

```powershell
chrome.exe --remote-debugging-port=9222
```

Из Windows сначала проверьте сам Chrome:

```powershell
curl http://127.0.0.1:9222/json/version
curl http://127.0.0.1:9222/json/list
```

Если это не удаётся в Windows, проблема пока не в OpenClaw.

### Уровень 2: Проверьте, что WSL2 может получить доступ к этой конечной точке в Windows

Из WSL2 проверьте точный адрес, который планируете использовать в `cdpUrl`:

```bash
curl http://WINDOWS_HOST_OR_IP:9222/json/version
curl http://WINDOWS_HOST_OR_IP:9222/json/list
```

Хороший результат:

- `/json/version` возвращает JSON с метаданными Browser / Protocol-Version;
- `/json/list` возвращает JSON (пустой массив допустим, если нет открытых страниц).

Если это не удаётся:

- Windows ещё не предоставляет порт для WSL2;
- адрес неверный для стороны WSL2;
- отсутствует настройка брандмауэра / переадресации портов / локального проксирования.

Устраните это перед изменением конфигурации OpenClaw.

### Уровень 3: Настройте правильный профиль браузера

Для сырого удалённого CDP укажите OpenClaw адрес, доступный из WSL2:

```json5
{
  browser: {
    enabled: true,
    defaultProfile: "remote",
    profiles: {
      remote: {
        cdpUrl: "http://WINDOWS_HOST_OR_IP:9222",
        attachOnly: true,
        color: "#00AA00",
      },
    },
  },
}
```

Примечания:

- используйте адрес, доступный из WSL2, а не тот, который работает только в Windows;
- оставьте `attachOnly: true` для браузеров, управляемых извне;
- `cdpUrl` может быть `http://`, `https://`, `ws://` или `wss://`;
- используйте HTTP(S), если хотите, чтобы OpenClaw обнаружил `/json/version`;
- используйте WS(S) только в том случае, если поставщик браузера предоставляет прямой URL-адрес сокета DevTools;
- проверьте тот же URL с помощью `curl` перед тем, как ожидать успеха от OpenClaw.

### Уровень 4: Отдельно проверьте уровень контрольного UI

Откройте UI из Windows:

`http://127.0.0.1:18789/`

Затем проверьте:

- источник страницы соответствует тому, что ожидает `gateway.controlUi.allowedOrigins`;
- токенная авторизация или пара настроены правильно;
- вы не отлаживаете проблему авторизации контрольного UI, принимая её за проблему браузера.

Полезная страница:

- [Контрольный UI](/web/control-ui)

### Уровень 5: Проверьте сквозное управление браузером

Из WSL2:

```bash
openclaw browser open https://example.com --browser-profile remote
openclaw browser tabs --browser-profile remote
```

Хороший результат:

- вкладка открывается в Chrome для Windows;
- `openclaw browser tabs` возвращает целевой объект;
- последующие действия (`snapshot`, `screenshot`, `navigate`) работают из того же профиля.

## Распространённые вводящие в заблуждение ошибки

Рассматривайте каждое сообщение как подсказку, относящуюся к определённому уровню:

- `control-ui-insecure-auth`
  - проблема с источником UI / безопасным контекстом, а не с транспортом CDP;
- `token_missing`
  - проблема с настройкой авторизации;
- `pairing required`
  - проблема с утверждением устройства;
- `Remote CDP for profile "remote" is not reachable`
  - WSL2 не может получить доступ к настроенному `cdpUrl`;
- `Browser attachOnly is enabled and CDP websocket for profile "remote" is not reachable`
  - HTTP-конечная точка ответила, но WebSocket DevTools всё равно не удалось открыть;
- устаревший viewport / тёмный режим / локаль / переопределения офлайн после удалённого сеанса
  - выполните `openclaw browser stop --browser-profile remote`;
  - это закрывает активную сессию управления и освобождает состояние эмуляции Playwright/CDP без перезапуска Gateway или внешнего браузера;
- `gateway timeout after 1500ms`
  - часто это всё ещё проблема доступности CDP или медленная/недоступная удалённая конечная точка;
- `No Chrome tabs found for profile="user"`
  - выбран локальный профиль Chrome MCP, где нет доступных локальных вкладок.

## Быстрый чек-лист для первичной диагностики

1. Windows: работает ли `curl http://127.0.0.1:9222/json/version`?
2. WSL2: работает ли `curl http://WINDOWS_HOST_OR_IP:9222/json/version`?
3. Конфигурация OpenClaw: использует ли `browser.profiles.<name>.cdpUrl` именно этот адрес, доступный из WSL2?
4. Контрольный UI: вы открываете `http://127.0.0.1:18789/`, а не IP-адрес локальной сети?
5. Пытаетесь ли вы использовать `existing-session` между WSL2 и Windows вместо сырого удалённого CDP?

## Практический вывод

Конфигурация обычно работоспособна. Сложность заключается в том, что транспорт браузера, безопасность источника контрольного UI и токен/пара могут давать сбои независимо друг от друга, но с точки зрения пользователя эти сбои выглядят похоже.

Если сомневаетесь:

- сначала проверьте конечную точку Chrome в Windows локально;
- затем проверьте ту же конечную точку из WSL2;
- только после этого приступайте к отладке конфигурации OpenClaw или авторизации контрольного UI.
