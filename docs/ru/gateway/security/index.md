# Безопасность

<Warning>
**Модель доверия для персонального помощника:** в этих рекомендациях предполагается наличие одной доверенной границы оператора на шлюз (модель "один пользователь / персональный помощник").
OpenClaw **не** является средством обеспечения безопасности в условиях враждебной многопользовательской среды, где несколько антагонистичных пользователей делят один агент/шлюз.
Если вам нужна работа в условиях смешанного доверия или с антагонистичными пользователями, разделите границы доверия (отдельный шлюз + учётные данные, в идеале — отдельные пользователи ОС/хосты).
</Warning>

## Сначала область применения: модель безопасности персонального помощника

Рекомендации по безопасности OpenClaw предполагают развёртывание в режиме **персонального помощника**: одна доверенная граница оператора, потенциально множество агентов.

- Поддерживаемая модель безопасности: один пользователь/граница доверия на шлюз (предпочтительно — один пользователь ОС/хост/VPS на границу).
- Неподдерживаемая граница безопасности: один общий шлюз/агент, используемый взаимно недоверенными или антагонистичными пользователями.
- Если требуется изоляция антагонистичных пользователей, разделите границы доверия (отдельный шлюз + учётные данные, в идеале — отдельные пользователи ОС/хосты).
- Если несколько недоверенных пользователей могут отправлять сообщения агенту с включёнными инструментами, считайте, что они делят одну и ту же делегированную власть инструментов для этого агента.

На этой странице объясняется, как обеспечить защиту **в рамках этой модели**. Здесь не утверждается, что возможна изоляция в условиях враждебной многопользовательской среды на одном общем шлюзе.

## Быстрая проверка: `openclaw security audit`

См. также: [Формальная верификация (модели безопасности)](/security/formal-verification)

Выполняйте эту проверку регулярно (особенно после изменения конфигурации или раскрытия сетевых интерфейсов):

```bash
openclaw security audit
openclaw security audit --deep
openclaw security audit --fix
openclaw security audit --json
```

Команда `security audit --fix` намеренно ограничена: она преобразует общие политики открытых групп в белые списки, восстанавливает параметр `logging.redactSensitive: "tools"`, ужесточает права доступа к состоянию/конфигурации/включаемым файлам и использует сброс ACL Windows вместо POSIX `chmod` при работе в Windows.

Она отмечает распространённые ошибки (раскрытие аутентификации шлюза, раскрытие управления браузером, расширенные белые списки, права доступа к файловой системе, разрешительные одобрения выполнения, раскрытие инструментов в открытых каналах).

OpenClaw — это одновременно продукт и эксперимент: вы интегрируете поведение передовых моделей в реальные каналы обмена сообщениями и реальные инструменты. **Не существует "абсолютно безопасной" настройки.** Цель — осознанно подходить к следующим вопросам:

- кто может общаться с вашим ботом;
- где боту разрешено действовать;
- к чему бот может получить доступ.

Начните с минимально возможного уровня доступа, который всё ещё работает, а затем расширяйте его по мере накопления уверенности.

### Развёртывание и доверие к хосту

OpenClaw предполагает, что хост и граница конфигурации являются доверенными:

- Если кто‑то может изменить состояние/конфигурацию хоста шлюза (`~/.openclaw`, включая `openclaw.json`), считайте этого человека доверенным оператором.
- Использование одного шлюза для нескольких взаимно недоверенных/антагонистичных операторов **не рекомендуется**.
- Для команд со смешанным доверием разделите границы доверия с помощью отдельных шлюзов (или, как минимум, отдельных пользователей ОС/хостов).
- Рекомендуемая настройка по умолчанию: один пользователь на машину/хост (или VPS), один шлюз для этого пользователя и один или несколько агентов в этом шлюзе.
- Внутри одного экземпляра шлюза доступ аутентифицированного оператора является доверенной ролью плоскости управления, а не ролью арендатора для каждого пользователя.
- Идентификаторы сеансов (`sessionKey`, ID сеансов, метки) являются селекторами маршрутизации, а не токенами авторизации.
- Если несколько человек могут отправлять сообщения агенту с включёнными инструментами, каждый из них может управлять одним и тем же набором разрешений. Изоляция сеанса/памяти для каждого пользователя помогает обеспечить конфиденциальность, но не преобразует общий агент в авторизацию хоста для каждого пользователя.

### Общий рабочий пространство Slack: реальный риск

Если "каждый в Slack может отправлять сообщения боту", основной риск заключается в делегированной власти инструментов:

- любой разрешённый отправитель может инициировать вызовы инструментов (`exec`, браузер, сетевые/файловые инструменты) в рамках политики агента;
- внедрение подсказок/контента от одного отправителя может привести к действиям, влияющим на общее состояние, устройства или выходные данные;
- если у одного общего агента есть конфиденциальные учётные данные/файлы, любой разрешённый отправитель потенциально может инициировать утечку данных с помощью инструментов.

Используйте отдельные агенты/шлюзы с минимальным набором инструментов для рабочих процессов команды; храните агентов с персональными данными в частном порядке.

### Агент, общий для компании: допустимый шаблон

Это допустимо, если все, кто использует этого агента, находятся в одной границе доверия (например, команда одной компании), и агент строго ориентирован на бизнес.

- запустите его на выделенной машине/ВМ/контейнере;
- используйте выделенного пользователя ОС + выделенный браузер/профиль/учётные записи для этого времени выполнения;
- не подключайте это время выполнения к личным учётным записям Apple/Google или личным менеджерам паролей/профилям браузера.

Если вы смешиваете личные и корпоративные учётные записи на одном времени выполнения, вы стираете границы и увеличиваете риск утечки персональных данных.

## Концепция доверия к шлюзу и узлу

Рассматривайте шлюз и узел как одну область доверия оператора с разными ролями:

- **Шлюз** — это плоскость управления и поверхность политики (`gateway.auth`, политика инструментов, маршрутизация).
- **Узел** — это удалённая поверхность выполнения, связанная с этим шлюзом (команды, действия устройств, локальные возможности хоста).
- Вызывающий, аутентифицированный на шлюзе, считается доверенным в рамках шлюза. После сопряжения действия узла считаются действиями доверенного оператора на этом узле.
- `sessionKey` — это выбор маршрута/контекста, а не аутентификация для каждого пользователя.
- Одобрения выполнения (белый список + запрос) — это ограничения для намерений оператора, а не изоляция в условиях враждебной многопользовательской среды.
- Настройка OpenClaw по умолчанию для доверенных однопользовательских систем предполагает, что выполнение на хосте в `gateway`/`node` разрешено без запросов на одобрение (`security="full"`, `ask="off"`, если вы не ужесточите это). Эта настройка является намеренным UX‑решением, а не уязвимостью сама по себе.
- Одобрения выполнения привязывают точный контекст запроса и наилучшие усилия по прямым локальным файловым операндам; они не моделируют семантически каждый путь загрузчика среды выполнения/интерпретатора. Используйте песочницу и изоляцию хоста для надёжных границ.

Если вам нужна изоляция враждебных пользователей, разделите границы доверия по пользователям ОС/хостам и запустите отдельные шлюзы.

## Матрица границ доверия

Используйте её как быструю модель при оценке рисков:

| Граница или контроль | Что это означает | Распространённые ошибочные представления |
| ------------------- | ---------------- | ---------------------------------------- |
| `gateway.auth` (токен/пароль/доверенный прокси/аутентификация устройства) | Аутентифицирует вызывающие стороны для API шлюза | "Для обеспечения безопасности требуются подписи для каждого сообщения в каждом кадре" |
| `sessionKey` | Ключ маршрутизации для выбора контекста/сеанса | "Ключ сеанса — это граница аутентификации для каждого пользователя" |
| Ограничения для подсказок/контента | Снижают риск злоупотребления моделью | "Внедрение подсказок само по себе доказывает обход аутентификации" |
| `canvas.eval` / оценка в браузере | Намеренная возможность оператора при включении | "Любой примитив JS eval автоматически является уязвимостью в этой модели доверия" |
| Локальная TUI `!` оболочка | Явное выполнение, инициированное оператором локально | "Локальная команда оболочки — это удалённая инъекция" |
| Сопряжение узлов и команды узлов | Удалённое выполнение на сопряжённых устройствах на уровне оператора | "Управление удалённым устройством должно рассматриваться как доступ недоверенного пользователя по умолчанию" |

## Не являются уязвимостями по замыслу

Эти шаблоны часто упоминаются и обычно закрываются как "без действий", если не показан реальный обход границы:

- Цепочки только с внедрением подсказок без обхода политики/аутентификации/песочницы.
- Заявления, предполагающие враждебную многопользовательскую работу на одном общем хосте/конфигурации.
- Заявления, классифицирующие обычный доступ оператора к чтению (например, `sessions.list`/`sessions.preview`/`chat.
