# Логирование

В OpenClaw есть два основных источника логов:

- **Файловые логи** (в формате JSON lines), записываемые шлюзом (Gateway).
- **Вывод в консоль**, отображаемый в терминалах и в интерфейсе отладки шлюза (Gateway Debug UI).

Вкладка **Logs** интерфейса управления Control UI отслеживает файловый лог шлюза. На этой странице объясняется, где хранятся логи, как их читать и как настраивать уровни и форматы логов.

## Где хранятся логи

По умолчанию шлюз записывает циклический лог-файл по пути:

`/tmp/openclaw/openclaw-ГГГГ-ММ-ДД.log`

Дата указывается в локальном часовом поясе хоста шлюза.

Вы можете изменить этот путь в файле `~/.openclaw/openclaw.json`:

```json
{
  "logging": {
    "file": "/путь/к/openclaw.log"
  }
}
```

## Как читать логи

### CLI: отслеживание в реальном времени (рекомендуется)

Используйте CLI для отслеживания лог-файла шлюза через RPC:

```bash
openclaw logs --follow
```

Полезные опции:

- `--local-time`: отображение временных меток в вашем локальном часовом поясе;
- `--url <url>` / `--token <token>` / `--timeout <ms>`: стандартные флаги RPC шлюза;
- `--expect-final`: флаг ожидания окончательного ответа RPC с поддержкой агента (принимается через общий клиентский слой).

Режимы вывода:

- **Сеансы TTY**: красиво оформленные, цветные, структурированные строки логов.
- **Сеансы не TTY**: простой текст.
- `--json`: JSON с разделителями строк (одно событие лога на строку).
- `--plain`: принудительный вывод простого текста в сеансах TTY.
- `--no-color`: отключение цветов ANSI.

Если вы явно указываете `--url`, CLI не применяет автоматически конфигурацию или учётные данные из окружения — укажите `--token` самостоятельно, если целевой шлюз требует аутентификации.

В режиме JSON CLI выводит объекты с тегом `type`:

- `meta`: метаданные потока (файл, курсор, размер);
- `log`: разобранный элемент лога;
- `notice`: подсказки об усечении или ротации;
- `raw`: неразобранная строка лога.

Если локальный шлюз loopback запрашивает сопряжение, `openclaw logs` автоматически переключается на настроенное локальное лог-файл. Явные цели `--url` не используют этот запасной вариант.

Если шлюз недоступен, CLI выводит краткую подсказку запустить:

```bash
openclaw doctor
```

### Интерфейс управления Control UI (веб)

Вкладка **Logs** интерфейса управления Control UI отслеживает тот же файл с помощью `logs.tail`.
См. [/web/control-ui](/web/control-ui), чтобы узнать, как его открыть.

### Логи только для каналов

Чтобы отфильтровать активность канала (WhatsApp/Telegram и т. д.), используйте:

```bash
openclaw channels logs --channel whatsapp
```

## Форматы логов

### Файловые логи (JSONL)

Каждая строка в лог-файле представляет собой JSON-объект. CLI и интерфейс управления Control UI разбирают эти записи для отображения структурированного вывода (время, уровень, подсистема, сообщение).

### Вывод в консоль

Логи в консоли учитывают **TTY** и форматируются для удобства чтения:

- Префиксы подсистем (например, `gateway/channels/whatsapp`);
- Цветное выделение уровней (info/warn/error);
- Опциональный компактный режим или режим JSON.

Форматирование консоли управляется параметром `logging.consoleStyle`.

### Логи WebSocket шлюза

`openclaw gateway` также поддерживает логирование протокола WebSocket для трафика RPC:

- обычный режим: только важные результаты (ошибки, ошибки разбора, медленные вызовы);
- `--verbose`: весь трафик запросов/ответов;
- `--ws-log auto|compact|full`: выбор стиля подробного вывода;
- `--compact`: псевдоним для `--ws-log compact`.

Примеры:

```bash
openclaw gateway
openclaw gateway --verbose --ws-log compact
openclaw gateway --verbose --ws-log full
```

## Настройка логирования

Вся конфигурация логирования находится в разделе `logging` файла `~/.openclaw/openclaw.json`.

```json
{
  "logging": {
    "level": "info",
    "file": "/tmp/openclaw/openclaw-ГГГГ-ММ-ДД.log",
    "consoleLevel": "info",
    "consoleStyle": "pretty",
    "redactSensitive": "tools",
    "redactPatterns": ["sk-.*"]
  }
}
```

### Уровни логов

- `logging.level`: уровень **файловых логов** (JSONL).
- `logging.consoleLevel`: уровень детализации **консоли**.

Вы можете переопределить оба параметра с помощью переменной окружения **`OPENCLAW_LOG_LEVEL`** (например, `OPENCLAW_LOG_LEVEL=debug`). Переменная окружения имеет приоритет над файлом конфигурации, поэтому вы можете повысить детализацию для одного запуска без редактирования `openclaw.json`. Также вы можете использовать глобальный параметр CLI **`--log-level <level>`** (например, `openclaw --log-level debug gateway run`), который переопределяет переменную окружения для этой команды.

Параметр `--verbose` влияет только на вывод в консоль и детализацию логов WS; он не изменяет уровни файловых логов.

### Стили консоли

`logging.consoleStyle`:

- `pretty`: удобный для человека, цветной, с временными метками.
- `compact`: более компактный вывод (лучше для длительных сеансов).
- `json`: JSON в каждой строке (для обработчиков логов).

### Редактирование (скрытие чувствительной информации)

Сводные данные инструментов могут скрывать чувствительные токены перед выводом в консоль:

- `logging.redactSensitive`: `off` | `tools` (по умолчанию: `tools`);
- `logging.redactPatterns`: список регулярных выражений для переопределения набора по умолчанию.

Редактирование влияет **только на вывод в консоль** и не изменяет файловые логи.

## Диагностика + OpenTelemetry

Диагностические данные представляют собой структурированные, машиночитаемые события для запусков моделей **и** телеметрии потока сообщений (вебхуки, очереди, состояние сеанса). Они **не заменяют** логи; они предназначены для передачи метрик, трассировок и других данных экспортерам.

Диагностические события генерируются внутри процесса, но экспортеры подключаются только при включении диагностики и плагина экспортера.

### OpenTelemetry против OTLP

- **OpenTelemetry (OTel)**: модель данных + SDK для трассировок, метрик и логов.
- **OTLP**: протокол передачи данных, используемый для экспорта данных OTel в коллектор/бэкенд.
- OpenClaw осуществляет экспорт через **OTLP/HTTP (protobuf)**.

### Экспортируемые сигналы

- **Метрики**: счётчики + гистограммы (использование токенов, поток сообщений, очереди).
- **Трассировки**: диапазоны для использования модели + обработки вебхуков/сообщений.
- **Логи**: экспортируются через OTLP, когда включён параметр `diagnostics.otel.logs`. Объём логов может быть большим; учитывайте `logging.level` и фильтры экспортера.

### Каталог диагностических событий

Использование модели:

- `model.usage`: токены, стоимость, длительность, контекст, провайдер/модель/канал, идентификаторы сеанса.

Поток сообщений:

- `webhook.received`: поступление вебхука по каналу.
- `webhook.processed`: обработка вебхука + длительность.
- `webhook.error`: ошибки обработчика вебхуков.
- `message.queued`: сообщение поставлено в очередь на обработку.
- `message.processed`: результат + длительность + возможная ошибка.

Очереди и сеансы:

- `queue.lane.enqueue`: постановка в очередь полосы команд + глубина.
- `queue.lane.dequeue`: извлечение из очереди полосы команд + время ожидания.
- `session.state`: переход состояния сеанса + причина.
- `session.stuck`: предупреждение о зависшем сеансе + возраст.
- `run.attempt`: метаданные повторной попытки запуска.
- `diagnostic.heartbeat`: агрегированные счётчики (вебхуки/очереди/сеансы).

### Включение диагностики (без экспортера)

Используйте это, если вы хотите, чтобы диагностические события были доступны для плагинов или пользовательских приёмников:

```json
{
  "diagnostics": {
    "enabled": true
  }
}
```
