# Панель управления (Control UI)

Панель управления шлюза — это интерфейс управления (Control UI) в браузере, который по умолчанию доступен по пути `/` (можно переопределить с помощью `gateway.controlUi.basePath`).

Быстрый доступ (локальный шлюз):

- [http://127.0.0.1:18789/](http://127.0.0.1:18789/) (или [http://localhost:18789/](http://localhost:18789/))

Основные ссылки:

- [Control UI](/web/control-ui) — информация об использовании и возможностях интерфейса.
- [Tailscale](/gateway/tailscale) — автоматизация Serve/Funnel.
- [Web surfaces](/web) — режимы привязки и примечания по безопасности.

Аутентификация выполняется на этапе рукопожатия WebSocket через настроенную в шлюзе точку аутентификации:

- `connect.params.auth.token`
- `connect.params.auth.password`
- Заголовки идентификации Tailscale Serve, если `gateway.auth.allowTailscale: true`
- Заголовки идентификации доверенного прокси-сервера, если `gateway.auth.mode: "trusted-proxy"`

См. `gateway.auth` в разделе [Конфигурация шлюза](/gateway/configuration).

Примечание по безопасности: интерфейс управления (Control UI) — это **административная поверхность** (чат, конфигурация, утверждения exec). Не предоставляйте к нему публичный доступ. Интерфейс сохраняет токены URL панели управления в sessionStorage для текущей вкладки браузера и выбранного URL шлюза, а после загрузки удаляет их из URL. Рекомендуется использовать localhost, Tailscale Serve или SSH-туннель.

## Ускоренный способ (рекомендуется)

- После завершения начальной настройки CLI автоматически открывает панель управления и выводит чистую (без токенов) ссылку.
- В любой момент можно повторно открыть панель: `openclaw dashboard` (копирует ссылку, по возможности открывает браузер, выводит подсказку по SSH в случае headless-режима).
- Если интерфейс запрашивает аутентификацию с общим секретом, вставьте настроенные токен или пароль в настройки Control UI.

## Основы аутентификации (локальный и удалённый доступ)

- **Localhost**: откройте `http://127.0.0.1:18789/`.
- **Источник токена с общим секретом**: `gateway.auth.token` (или `OPENCLAW_GATEWAY_TOKEN`); команда `openclaw dashboard` может передать его через фрагмент URL для однократной начальной загрузки, а Control UI сохраняет его в sessionStorage для текущей вкладки браузера и выбранного URL шлюза (вместо localStorage).
- Если `gateway.auth.token` управляется через SecretRef, команда `openclaw dashboard` выводит/копирует/открывает URL без токена. Это позволяет избежать раскрытия токенов, управляемых извне, в журналах оболочки, истории буфера обмена или аргументах запуска браузера.
- Если `gateway.auth.token` настроен как SecretRef и не разрешён в текущей оболочке, команда `openclaw dashboard` всё равно выводит URL без токена, а также рекомендации по настройке аутентификации.
- **Пароль с общим секретом**: используйте настроенное значение `gateway.auth.password` (или `OPENCLAW_GATEWAY_PASSWORD`). Панель управления не сохраняет пароли после перезагрузки.
- **Режимы с идентификацией**: Tailscale Serve может выполнять аутентификацию Control UI/WebSocket через заголовки идентификации, если `gateway.auth.allowTailscale: true`, а обратный прокси-сервер с поддержкой идентификации (не loopback) может удовлетворять условию `gateway.auth.mode: "trusted-proxy"`. В этих режимах панели управления не требуется вставлять общий секрет для WebSocket.
- **Не localhost**: используйте Tailscale Serve, привязку с общим секретом (не loopback), обратный прокси-сервер с поддержкой идентификации и `gateway.auth.mode: "trusted-proxy"` или SSH-туннель. HTTP API по-прежнему используют аутентификацию с общим секретом, если вы намеренно не запустили private-ingress с `gateway.auth.mode: "none"` или HTTP-аутентификацию через доверенный прокси. См. [Web surfaces](/web).

## Если вы видите сообщение "unauthorized" / 1008

- Убедитесь, что шлюз доступен (локально: `openclaw status`; удалённо: SSH-туннель `ssh -N -L 18789:127.0.0.1:18789 user@host`, затем откройте `http://127.0.0.1:18789/`).
- При ошибке `AUTH_TOKEN_MISMATCH` клиенты могут выполнить одну доверенную повторную попытку с кэшированным токеном устройства, если шлюз возвращает подсказки для повторной попытки. При повторной попытке с кэшированным токеном повторно используются одобренные области действия токена; явные вызовы `deviceToken` / `scopes` сохраняют запрошенный набор областей действия. Если аутентификация по-прежнему завершается ошибкой после повторной попытки, устраните расхождение токенов вручную.
- За пределами этого пути повторной попытки приоритет аутентификации подключения следующий: сначала явный общий токен/пароль, затем явный `deviceToken`, затем сохранённый токен устройства, затем токен начальной загрузки.
- В асинхронном пути Tailscale Serve Control UI неудачные попытки для одного и того же `{scope, ip}` сериализуются до того, как ограничитель неудачной аутентификации зафиксирует их, поэтому вторая параллельная неудачная повторная попытка может уже показывать сообщение `retry later`.
- Для устранения расхождения токенов следуйте [чек-листу восстановления при расхождении токенов](/cli/devices#token-drift-recovery-checklist).
- Получите или укажите общий секрет с хоста шлюза:
  - Токен: `openclaw config get gateway.auth.token`
  - Пароль: определите настроенное значение `gateway.auth.password` или `OPENCLAW_GATEWAY_PASSWORD`
  - Токен, управляемый через SecretRef: определите внешнего поставщика секретов или экспортируйте `OPENCLAW_GATEWAY_TOKEN` в текущей оболочке, затем повторно запустите `openclaw dashboard`
  - Если общий секрет не настроен: `openclaw doctor --generate-gateway-token`
- В настройках панели управления вставьте токен или пароль в поле аутентификации, затем подключитесь.
- Выбор языка интерфейса находится в разделе **Overview -> Gateway Access -> Language**. Он является частью карточки доступа, а не раздела Appearance.
