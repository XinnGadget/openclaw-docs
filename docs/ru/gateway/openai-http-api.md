# OpenAI Chat Completions (HTTP)

Gateway в OpenClaw может предоставлять небольшую конечную точку Chat Completions, совместимую с OpenAI.

Эта конечная точка **по умолчанию отключена**. Сначала необходимо включить её в конфигурации.

- `POST /v1/chat/completions`
- Тот же порт, что и у Gateway (мультиплексирование WS + HTTP): `http://<gateway-host>:<port>/v1/chat/completions`

Когда HTTP-интерфейс Gateway, совместимый с OpenAI, включён, он также предоставляет:

- `GET /v1/models`
- `GET /v1/models/{id}`
- `POST /v1/embeddings`
- `POST /v1/responses`

Под капотом запросы выполняются как обычный запуск агента Gateway (тот же путь кода, что и у `openclaw agent`), поэтому маршрутизация, разрешения и конфигурация соответствуют вашему Gateway.

## Аутентификация

Используется конфигурация аутентификации Gateway.

Распространённые пути HTTP-аутентификации:

- аутентификация с общим секретом (`gateway.auth.mode="token"` или `"password"`):
  `Authorization: Bearer <token-or-password>`
- аутентификация с доверенной идентичностью через HTTP (`gateway.auth.mode="trusted-proxy"`):
  направление через настроенное прокси с учётом идентичности и предоставление ему возможности внедрить необходимые заголовки идентичности
- открытая аутентификация для частного входа (`gateway.auth.mode="none"`):
  заголовок аутентификации не требуется

Примечания:

- При `gateway.auth.mode="token"` используйте `gateway.auth.token` (или `OPENCLAW_GATEWAY_TOKEN`).
- При `gateway.auth.mode="password"` используйте `gateway.auth.password` (или `OPENCLAW_GATEWAY_PASSWORD`).
- При `gateway.auth.mode="trusted-proxy"` HTTP-запрос должен поступать от настроенного доверенного прокси-источника, не являющегося loopback; прокси на том же хосте с loopback не соответствуют этому режиму.
- Если настроена `gateway.auth.rateLimit` и происходит слишком много сбоев аутентификации, конечная точка возвращает `429` с `Retry-After`.

## Границы безопасности (важно)

Рассматривайте эту конечную точку как поверхность **полного доступа оператора** к экземпляру Gateway.

- Аутентификация по токену Bearer в данном случае не является моделью с узким охватом для каждого пользователя.
- Действительный токен/пароль Gateway для этой конечной точки следует рассматривать как учётные данные владельца/оператора.
- Запросы проходят через тот же путь агента плоскости управления, что и действия доверенного оператора.
- На этой конечной точке нет отдельной границы для не-владельцев/пользователей; как только вызывающий проходит аутентификацию Gateway, OpenClaw рассматривает его как доверенного оператора для этого Gateway.
- Для режимов аутентификации с общим секретом (`token` и `password`) конечная точка восстанавливает стандартные значения по умолчанию для полного оператора, даже если вызывающий отправляет более узкий заголовок `x-openclaw-scopes`.
- Режимы HTTP-аутентификации с доверенной идентичностью (например, аутентификация через доверенный прокси или `gateway.auth.mode="none"`) учитывают `x-openclaw-scopes`, если он присутствует, а в противном случае возвращаются к стандартному набору областей по умолчанию для оператора.
- Если политика целевого агента разрешает использование чувствительных инструментов, эта конечная точка может их использовать.
- Держите эту конечную точку только на loopback/tailnet/частном входе; не выставляйте её напрямую в публичный интернет.

Матрица аутентификации:

- `gateway.auth.mode="token"` или `"password"` + `Authorization: Bearer ...`
  - подтверждает владение общим секретом оператора Gateway
  - игнорирует более узкий `x-openclaw-scopes`
  - восстанавливает полный набор областей по умолчанию для оператора:
    `operator.admin`, `operator.approvals`, `operator.pairing`,
    `operator.read`, `operator.talk.secrets`, `operator.write`
  - рассматривает повороты чата на этой конечной точке как повороты от отправителя-владельца
- режимы HTTP-аутентификации с доверенной идентичностью (например, аутентификация через доверенный прокси или `gateway.auth.mode="none"` на частном входе)
  - аутентифицируют некоторую внешнюю доверенную идентичность или границу развёртывания
  - учитывают `x-openclaw-scopes`, если заголовок присутствует
  - возвращаются к стандартному набору областей по умолчанию для оператора, если заголовок отсутствует
  - теряют семантику владельца только в том случае, если вызывающий явно сужает области и исключает `operator.admin`

См. [Безопасность](/gateway/security) и [Удаленный доступ](/gateway/remote).

## Контракт модели, ориентированной на агента

OpenClaw рассматривает поле `model` в OpenAI как **цель агента**, а не как сырой идентификатор модели провайдера.

- `model: "openclaw"` направляет к настроенному агенту по умолчанию.
- `model: "openclaw/default"` также направляет к настроенному агенту по умолчанию.
- `model: "openclaw/<agentId>"` направляет к конкретному агенту.

Необязательные заголовки запроса:

- `x-openclaw-model: <provider/model-or-bare-id>` переопределяет бэкенд-модель для выбранного агента.
- `x-openclaw-agent-id: <agentId>` по-прежнему поддерживается как совместимость для переопределения.
- `x-openclaw-session-key: <sessionKey>` полностью контролирует маршрутизацию сессии.
- `x-openclaw-message-channel: <channel>` задаёт контекст синтетического входящего канала для подсказок и политик, учитывающих канал.

По-прежнему принимаются псевдонимы совместимости:

- `model: "openclaw:<agentId>"`
- `model: "agent:<agentId>"`

## Включение конечной точки

Установите `gateway.http.endpoints.chatCompletions.enabled` в `true`:

```json5
{
  gateway: {
    http: {
      endpoints: {
        chatCompletions: { enabled: true },
      },
    },
  },
}
```

## Отключение конечной точки

Установите `gateway.http.endpoints.chatCompletions.enabled` в `false`:

```json5
{
  gateway: {
    http: {
      endpoints: {
        chatCompletions: { enabled: false },
      },
    },
  },
}
```

## Поведение сессии

По умолчанию конечная точка **не сохраняет состояние между запросами** (новый ключ сессии генерируется при каждом вызове).

Если запрос включает строку `user` из OpenAI, Gateway выводит стабильный ключ сессии из неё, поэтому повторные вызовы могут использовать одну сессию агента.

## Почему эта поверхность важна

Это наиболее эффективный набор совместимости для самостоятельно размещённых фронтендов и инструментов:

- Большинство установок Open WebUI, LobeChat и LibreChat ожидают `/v1/models`.
- Многие системы RAG ожидают `/v1/embeddings`.
- Существующие клиенты чата OpenAI обычно могут начать с `/v1/chat/completions`.
- Всё больше клиентов, ориентированных на агентов, предпочитают `/v1/responses`.

  <Accordion title="Что возвращает `/v1/models`?">
    Список целей агентов OpenClaw.

    Возвращаемые идентификаторы — это записи `openclaw`, `openclaw/default` и `openclaw/<agentId>`.
    Используйте их напрямую как значения `model` в OpenAI.

  </Accordion>
  <Accordion title="Перечисляет ли `/v1/models` агентов или субагентов?">
    Он перечисляет цели агентов верхнего уровня, а не модели бэкенд-провайдера и не субагентов.

    Субагенты остаются внутренней топологией выполнения. Они не отображаются как псевдомодели.

  </Accordion>
  <Accordion title="Почему включён `openclaw/default`?">
    `openclaw/default` — это стабильный псевдоним для настроенного агента по умолчанию.

    Это означает, что клиенты могут продолжать использовать один предсказуемый идентификатор, даже если реальный идентификатор агента по умолчанию меняется между средами.

  </Accordion>
  <Accordion title="Как переопределить бэкенд-модель?">
    Используйте `x-openclaw-model`.

    Примеры:
    `x-openclaw-model: openai/gpt-5.4`
    `x-openclaw-model: gpt-5.4`

    Если вы опустите его, выбранный агент будет работать с выбранной по умолчанию моделью.
