# Инструмент Exec

Запускает командную оболочку в рабочей области. Поддерживает выполнение в переднем и фоновом режимах через `process`.
Если использование `process` запрещено, `exec` выполняется синхронно и игнорирует параметры `yieldMs`/`background`.
Фоновые сессии привязаны к агенту: `process` видит только сессии того же агента.

## Параметры

- `command` (обязательно)
- `workdir` (по умолчанию — текущий рабочий каталог)
- `env` (переопределения в формате "ключ‑значение")
- `yieldMs` (по умолчанию 10000): автоматический перевод в фоновый режим по истечении задержки
- `background` (булево): немедленный перевод в фоновый режим
- `timeout` (в секундах, по умолчанию 1800): завершение по истечении времени
- `pty` (булево): запуск в псевдотерминале, если доступно (CLI, работающие только в TTY, агенты кодирования, терминальные UI)
- `host` (`auto | sandbox | gateway | node`): место выполнения
- `security` (`deny | allowlist | full`): режим контроля для `gateway`/`node`
- `ask` (`off | on-miss | always`): запросы на подтверждение для `gateway`/`node`
- `node` (строка): идентификатор/имя узла для `host=node`
- `elevated` (булево): запрос на повышенный режим (выход из песочницы на заданный путь хоста); `security=full` принудительно применяется только если `elevated` разрешён до `full`

Примечания:

- По умолчанию для `host` установлено значение `auto`: песочница, если во время сессии активен рантайм песочницы, в противном случае — gateway.
- `auto` — это стратегия маршрутизации по умолчанию, а не подстановочный знак. Для отдельного вызова `host=node` разрешено из `auto`; для отдельного вызова `host=gateway` разрешено только если рантайм песочницы не активен.
- Без дополнительной конфигурации `host=auto` работает "из коробки": отсутствие песочницы означает, что будет использован `gateway`; активная песочница означает, что выполнение останется в песочнице.
- `elevated` позволяет выйти из песочницы на заданный путь хоста: по умолчанию используется `gateway`, или `node`, если задано `tools.exec.host=node` (или по умолчанию для сессии установлено `host=node`). Доступен только если для текущей сессии/провайдера разрешён повышенный доступ.
- Подтверждения для `gateway`/`node` контролируются файлом `~/.openclaw/exec-approvals.json`.
- Для `node` требуется сопряжённый узел (сопутствующее приложение или безголовый узел).
- Если доступно несколько узлов, задайте `exec.node` или `tools.exec.node`, чтобы выбрать один из них.
- `exec host=node` — единственный способ выполнения оболочки для узлов; устаревшая обёртка `nodes.run` удалена.
- На хостах, отличных от Windows, exec использует `SHELL`, если он задан; если `SHELL` равен `fish`, предпочтение отдаётся `bash` (или `sh`) из `PATH`, чтобы избежать несовместимых с fish скриптов, затем возвращается к `SHELL`, если ни один из них не найден.
- На хостах Windows exec предпочитает PowerShell 7 (`pwsh`) (Program Files, ProgramW6432, затем PATH), затем возвращается к Windows PowerShell 5.1.
- Выполнение на хосте (`gateway`/`node`) отклоняет `env.PATH` и переопределения загрузчика (`LD_*`/`DYLD_*`), чтобы предотвратить подмену бинарных файлов или внедрение кода.
- OpenClaw устанавливает `OPENCLAW_SHELL=exec` в среде запущенной команды (включая PTY и выполнение в песочнице), чтобы правила оболочки/профиля могли определить контекст exec-инструмента.
- Важно: песочница **отключена по умолчанию**. Если песочница отключена, неявный `host=auto` разрешается в `gateway`. Явный `host=sandbox` в таком случае завершится ошибкой, а не будет молча выполняться на хосте gateway. Включите песочницу или используйте `host=gateway` с подтверждениями.
- Предварительные проверки скриптов (на распространённые ошибки синтаксиса оболочки Python/Node) проверяют только файлы в пределах эффективной границы `workdir`. Если путь к скрипту находится за пределами `workdir`, предварительная проверка для этого файла пропускается.
- Для длительной работы, которая начинается сейчас, запустите её один раз и полагайтесь на автоматическое пробуждение по завершении, если оно включено и команда выдаёт вывод или завершается с ошибкой.
Используйте `process` для получения логов, статуса, ввода или вмешательства; не имитируйте планирование с помощью циклов ожидания, циклов тайм-аута или повторного опроса.
- Для работы, которая должна выполняться позже или по расписанию, используйте cron вместо шаблонов `exec` с задержками/ожиданиями.

## Конфигурация

- `tools.exec.notifyOnExit` (по умолчанию: true): если true, фоновые сессии exec ставят в очередь системное событие и запрашивают heartbeat при завершении.
- `tools.exec.approvalRunningNoticeMs` (по умолчанию: 10000): выводит одно уведомление "выполняется", если выполнение exec с подтверждением длится дольше указанного времени (0 отключает).
- `tools.exec.host` (по умолчанию: `auto`; разрешается в `sandbox`, если активен рантайм песочницы, в противном случае — `gateway`)
- `tools.exec.security` (по умолчанию: `deny` для песочницы, `full` для gateway + node, если не задано)
- `tools.exec.ask` (по умолчанию: `off`)
- Выполнение на хосте без подтверждений — значение по умолчанию для gateway + node. Если вы хотите использовать поведение с подтверждениями/списком разрешённых, ужесточите настройки как для `tools.exec.*`, так и для хоста `~/.openclaw/exec-approvals.json`; см. [Подтверждения exec](/tools/exec-approvals#no-approval-yolo-mode).
- YOLO определяется настройками политики хоста (`security=full`, `ask=off`), а не `host=auto`. Если вы хотите принудительно направить выполнение на gateway или node, задайте `tools.exec.host` или используйте `/exec host=...`.
- В режиме `security=full` плюс `ask=off` выполнение на хосте следует настроенной политике напрямую; дополнительный эвристический префильтр обфускации команд отсутствует.
- `tools.exec.node` (по умолчанию: не задано)
- `tools.exec.strictInlineEval` (по умолчанию: false): если true, формы встроенной оценки интерпретатора, такие как `python -c`, `node -e`, `ruby -e`, `perl -e`, `php -r`, `lua -e` и `osascript -e`, всегда требуют явного подтверждения. `allow-always` может по‑прежнему разрешать безобидные вызовы интерпретатора/скрипта, но формы inline-eval будут запрашивать подтверждение каждый раз.
- `tools.exec.pathPrepend`: список каталогов, которые нужно добавить в начало `PATH` для выполнения exec (только для gateway + sandbox).
- `tools.exec.safeBins`: безопасные бинарные файлы, работающие только с stdin, которые могут выполняться без явных записей в списке разрешённых. Для подробностей о поведении см. [Безопасные бинарные файлы](/tools/exec-approvals#safe-bins-stdin-only).
- `tools.exec.safeBinTrustedDirs`: дополнительные явно заданные каталоги, которым доверяется проверка путей для `safeBins`. Записи в `PATH` никогда не считаются доверенными автоматически. Встроенные значения по умолчанию — `/bin` и `/usr/bin`.
- `tools.exec.safeBinProfiles`: необязательная пользовательская политика argv для каждого безопасного бинарного файла (`minPositional`, `maxPositional`, `allowedValueFlags`, `deniedFlags`).

Пример:

```json5
{
  tools: {
    exec: {
      pathPrepend: ["~/bin", "/opt/oss/bin"],
    },
  },
}
```

### Обработка PATH

- `host=gateway`: объединяет `PATH` вашей оболочки входа в среду exec. Переопределения `env.PATH` отклоняются для выполнения на хосте. Сам демон по‑прежнему работает с минимальным `PATH`:
  - macOS: `/opt/homebrew/bin`, `/usr/local/bin`, `/usr/bin`, `/bin`
  - Linux: `/usr/local/bin`, `/usr/bin`, `/bin`
- `host=sandbox`: запускает `sh -lc` (оболочка входа) внутри контейнера, поэтому `/etc/profile` может сбросить `PATH`.
OpenClaw добавляет `env.PATH` после загрузки профиля через внутреннюю переменную окружения (без интерполяции оболочки); `tools.exec.pathPrepend` также применяется здесь.
