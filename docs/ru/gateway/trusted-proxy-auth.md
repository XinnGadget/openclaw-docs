# Аутентификация через доверенный прокси

> ⚠️ **Функция, связанная с безопасностью.** В этом режиме аутентификация полностью делегируется вашему обратному прокси-серверу. Неправильная конфигурация может привести к несанкционированному доступу к вашему шлюзу. Внимательно прочитайте эту страницу перед включением функции.

## Когда использовать

Используйте режим аутентификации `trusted-proxy`, если:

- Вы запускаете OpenClaw за **прокси-сервером с учётом идентификации пользователя** (Pomerium, Caddy + OAuth, nginx + oauth2-proxy, Traefik + forward auth).
- Ваш прокси-сервер выполняет всю аутентификацию и передаёт идентификацию пользователя через заголовки.
- Вы работаете в среде Kubernetes или в контейнерной среде, где прокси-сервер — единственный путь к шлюзу.
- Вы сталкиваетесь с ошибками WebSocket `1008 unauthorized`, поскольку браузеры не могут передавать токены в полезных данных WS.

## Когда НЕ использовать

- Если ваш прокси-сервер не выполняет аутентификацию пользователей (только завершает TLS или работает как балансировщик нагрузки).
- Если существует какой-либо путь к шлюзу, минуя прокси-сервер (дыры в брандмауэре, доступ через внутреннюю сеть).
- Если вы не уверены, корректно ли ваш прокси-сервер удаляет/перезаписывает пересылаемые заголовки.
- Если вам нужен только персональный доступ для одного пользователя (рассмотрите Tailscale Serve + loopback для более простой настройки).

## Как это работает

1. Ваш обратный прокси-сервер аутентифицирует пользователей (OAuth, OIDC, SAML и т. д.).
2. Прокси-сервер добавляет заголовок с идентификацией аутентифицированного пользователя (например, `x-forwarded-user: nick@example.com`).
3. OpenClaw проверяет, что запрос поступил от **доверенного IP-адреса прокси-сервера** (настроен в `gateway.trustedProxies`).
4. OpenClaw извлекает идентификацию пользователя из настроенного заголовка.
5. Если всё в порядке, запрос авторизуется.

## Поведение сопряжения с пользовательским интерфейсом управления

Когда `gateway.auth.mode = "trusted-proxy"` активен и запрос проходит проверку доверенного прокси-сервера, сеансы WebSocket пользовательского интерфейса управления могут подключаться без идентификации сопряжения устройства.

Последствия:

- Сопряжение больше не является основным барьером для доступа к пользовательскому интерфейсу управления в этом режиме.
- Политика аутентификации вашего прокси-сервера и `allowUsers` становятся эффективным средством контроля доступа.
- Убедитесь, что доступ к шлюзу ограничен только доверенными IP-адресами прокси-сервера (`gateway.trustedProxies` + брандмауэр).

## Конфигурация

```json5
{
  gateway: {
    // Аутентификация через доверенный прокси ожидает запросы от источника прокси, не являющегося loopback
    bind: "lan",

    // ВАЖНО: добавляйте сюда только IP-адреса вашего прокси
    trustedProxies: ["10.0.0.1", "172.17.0.1"],

    auth: {
      mode: "trusted-proxy",
      trustedProxy: {
        // Заголовок, содержащий идентификацию аутентифицированного пользователя (обязательно)
        userHeader: "x-forwarded-user",

        // Опционально: заголовки, которые ДОЛЖНЫ присутствовать (проверка прокси)
        requiredHeaders: ["x-forwarded-proto", "x-forwarded-host"],

        // Опционально: ограничение по конкретным пользователям (пусто = разрешить всех)
        allowUsers: ["nick@example.com", "admin@company.org"],
      },
    },
  },
}
```

Важное правило во время выполнения:

- Аутентификация через доверенный прокси отклоняет запросы с loopback-источника (`127.0.0.1`, `::1`, loopback CIDRs).
- Обратные прокси-серверы с loopback на одном хосте **не** удовлетворяют требованиям аутентификации через доверенный прокси.
- Для конфигураций с loopback-прокси на одном хосте используйте аутентификацию по токену/паролю или маршрутизируйте через не-loopback доверенный адрес прокси-сервера, который OpenClaw может проверить.
- Для развёртываний пользовательского интерфейса управления, не использующих loopback, по-прежнему требуется явное указание `gateway.controlUi.allowedOrigins`.

### Справочная информация по конфигурации

| Поле | Обязательно | Описание |
| --- | --- | --- |
| `gateway.trustedProxies` | Да | Массив IP-адресов прокси-серверов, которым можно доверять. Запросы от других IP-адресов отклоняются. |
| `gateway.auth.mode` | Да | Должно быть `"trusted-proxy"` |
| `gateway.auth.trustedProxy.userHeader` | Да | Название заголовка, содержащего идентификацию аутентифицированного пользователя |
| `gateway.auth.trustedProxy.requiredHeaders` | Нет | Дополнительные заголовки, которые должны присутствовать, чтобы запрос считался доверенным |
| `gateway.auth.trustedProxy.allowUsers` | Нет | Список разрешённых идентификаций пользователей. Пустое значение означает разрешение всем аутентифицированным пользователям. |

## Завершение TLS и HSTS

Используйте одну точку завершения TLS и применяйте HSTS там.

### Рекомендуемый шаблон: завершение TLS на прокси-сервере

Когда ваш обратный прокси-сервер обрабатывает HTTPS для `https://control.example.com`, установите `Strict-Transport-Security` на прокси-сервере для этого домена.

- Подходит для развёртываний, доступных из интернета.
- Позволяет хранить сертификат и политику усиления безопасности HTTP в одном месте.
- OpenClaw может оставаться на loopback HTTP за прокси-сервером.

Пример значения заголовка:

```text
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

### Завершение TLS на шлюзе

Если OpenClaw сам обслуживает HTTPS напрямую (без прокси-сервера, завершающего TLS), установите:

```json5
{
  gateway: {
    tls: { enabled: true },
    http: {
      securityHeaders: {
        strictTransportSecurity: "max-age=31536000; includeSubDomains",
      },
    },
  },
}
```

`strictTransportSecurity` принимает строковое значение заголовка или `false` для явного отключения.

### Рекомендации по развёртыванию

- Начните с короткого значения `max-age` (например, `max-age=300`) во время проверки трафика.
- Увеличьте до длительного значения (например, `max-age=31536000`) только после того, как будете уверены в стабильности.
- Добавляйте `includeSubDomains` только если все поддомены готовы к использованию HTTPS.
- Используйте `preload` только если вы намеренно соответствуете требованиям для предварительной загрузки для всего набора ваших доменов.
- Локальная разработка только с loopback не получает преимуществ от HSTS.

## Примеры настройки прокси-серверов

### Pomerium

Pomerium передаёт идентификацию в `x-pomerium-claim-email` (или других заголовках утверждений) и JWT в `x-pomerium-jwt-assertion`.

```json5
{
  gateway: {
    bind: "lan",
    trustedProxies: ["10.0.0.1"], // IP-адрес Pomerium
    auth: {
      mode: "trusted-proxy",
      trustedProxy: {
        userHeader: "x-pomerium-claim-email",
        requiredHeaders: ["x-pomerium-jwt-assertion"],
      },
    },
  },
}
```

Фрагмент конфигурации Pomerium:

```yaml
routes:
  - from: https://openclaw.example.com
    to: http://openclaw-gateway:18789
    policy:
      - allow:
          or:
            - email:
                is: nick@example.com
    pass_identity_headers: true
```
