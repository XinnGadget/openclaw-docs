# Навыки (OpenClaw)

OpenClaw использует папки навыков, совместимые с **[AgentSkills](https://agentskills.io), чтобы научить агента пользоваться инструментами. Каждый навык — это каталог, содержащий файл `SKILL.md` с YAML-заголовком и инструкциями. OpenClaw загружает **встроенные навыки** плюс необязательные локальные переопределения и фильтрует их во время загрузки на основе окружения, конфигурации и наличия исполняемых файлов.

## Расположение и приоритет

OpenClaw загружает навыки из следующих источников:

1. **Дополнительные папки навыков**: настраиваются с помощью `skills.load.extraDirs`.
2. **Встроенные навыки**: поставляются вместе с установкой (npm-пакет или OpenClaw.app).
3. **Управляемые/локальные навыки**: `~/.openclaw/skills`.
4. **Персональные навыки агента**: `~/.agents/skills`.
5. **Навыки проектного агента**: `<workspace>/.agents/skills`.
6. **Навыки рабочего пространства**: `<workspace>/skills`.

Если возникает конфликт имён навыков, приоритет следующий:

`<workspace>/skills` (наивысший) → `<workspace>/.agents/skills` → `~/.agents/skills` → `~/.openclaw/skills` → встроенные навыки → `skills.load.extraDirs` (наименьший).

## Навыки для отдельного агента vs общие навыки

В настройках с **несколькими агентами** у каждого агента есть собственное рабочее пространство. Это означает следующее:

- **Навыки для отдельного агента** находятся в `<workspace>/skills` и доступны только этому агенту.
- **Навыки проектного агента** находятся в `<workspace>/.agents/skills` и применяются к этому рабочему пространству до обычной папки `skills/` рабочего пространства.
- **Персональные навыки агента** находятся в `~/.agents/skills` и применяются ко всем рабочим пространствам на данном компьютере.
- **Общие навыки** находятся в `~/.openclaw/skills` (управляемые/локальные) и видны **всем агентам** на одном компьютере.
- **Общие папки** также можно добавить через `skills.load.extraDirs` (наименьший приоритет), если требуется общий набор навыков для нескольких агентов.

Если навык с одинаковым именем существует в нескольких местах, применяется обычный приоритет: сначала выигрывает рабочее пространство, затем навыки проектного агента, затем персональные навыки агента, затем управляемые/локальные, затем встроенные, затем дополнительные каталоги.

## Разрешённые списки навыков для агентов

**Расположение** навыка и его **видимость** — это отдельные механизмы управления.

- Расположение/приоритет определяет, какая копия навыка с одинаковым именем будет использована.
- Разрешённые списки для агентов определяют, какие видимые навыки агент может фактически использовать.

Используйте `agents.defaults.skills` для общей базовой настройки, затем переопределяйте для каждого агента с помощью `agents.list[].skills`:

```json5
{
  agents: {
    defaults: {
      skills: ["github", "weather"],
    },
    list: [
      { id: "writer" }, // наследует github, weather
      { id: "docs", skills: ["docs-search"] }, // заменяет значения по умолчанию
      { id: "locked-down", skills: [] }, // без навыков
    ],
  },
}
```

Правила:

- Опустите `agents.defaults.skills`, чтобы по умолчанию навыки не были ограничены.
- Опустите `agents.list[].skills`, чтобы наследовать `agents.defaults.skills`.
- Установите `agents.list[].skills: []`, чтобы отключить навыки.
- Непустой список `agents.list[].skills` — это окончательный набор навыков для данного агента; он не объединяется со значениями по умолчанию.

OpenClaw применяет эффективный набор навыков агента при построении промпта, обнаружении команд навыков через слеш-команды, синхронизации песочницы и создании снимков навыков.

## Плагины + навыки

Плагины могут поставлять собственные навыки, указывая каталоги `skills` в `openclaw.plugin.json` (пути относительно корня плагина). Навыки плагина загружаются, когда плагин включён. В настоящее время эти каталоги объединяются с тем же путём с низким приоритетом, что и `skills.load.extraDirs`, поэтому одноимённый встроенный, управляемый, агентский или рабочий навык переопределяет их.

Вы можете ограничить их с помощью `metadata.openclaw.requires.config` в записи конфигурации плагина. См. [Плагины](/tools/plugin) для обнаружения/настройки и [Инструменты](/tools) для описания инструментов, которым обучают эти навыки.

## ClawHub (установка + синхронизация)

ClawHub — это публичный реестр навыков для OpenClaw. Ознакомьтесь с ним на [https://clawhub.ai](https://clawhub.ai). Используйте встроенные команды `openclaw skills` для поиска/установки/обновления навыков или отдельный CLI `clawhub`, когда вам нужны рабочие процессы публикации/синхронизации.

Полное руководство: [ClawHub](/tools/clawhub).

Типичные сценарии:

- Установить навык в рабочее пространство:
  - `openclaw skills install <skill-slug>`
- Обновить все установленные навыки:
  - `openclaw skills update --all`
- Синхронизировать (сканировать + публиковать обновления):
  - `clawhub sync --all`

Встроенная команда `openclaw skills install` устанавливает в каталог `skills/` активного рабочего пространства. Отдельный CLI `clawhub` также устанавливает в `./skills` в текущем рабочем каталоге (или возвращается к настроенному рабочему пространству OpenClaw). OpenClaw обнаружит это как `<workspace>/skills` в следующем сеансе.

## Примечания по безопасности

- Относитесь к сторонним навыкам как к **ненадёжному коду**. Прочтите их перед включением.
- Предпочитайте запуск в песочнице для ненадёжных входных данных и рискованных инструментов. См. [Песочница](/gateway/sandboxing).
- Обнаружение навыков в рабочем пространстве и дополнительных каталогах принимает только корни навыков и файлы `SKILL.md`, чей разрешённый реальный путь остаётся внутри настроенного корня.
- Установка зависимостей навыков через шлюз (`skills.install`, адаптация и пользовательский интерфейс настроек навыков) запускает встроенный сканер опасного кода перед выполнением метаданных установщика. Нахождения уровня `critical` блокируются по умолчанию, если вызывающая сторона явно не устанавливает переопределение опасности; подозрительные находки только выдают предупреждение.
- `openclaw skills install <slug>` работает иначе: он загружает папку навыков ClawHub в рабочее пространство и не использует путь метаданных установщика, описанный выше.
- `skills.entries.*.env` и `skills.entries.*.apiKey` внедряют секреты в процесс **хоста** для данного хода агента (не в песочницу). Держите секреты вне промтов и логов.
- Для более широкой модели угроз и контрольных списков см. [Безопасность](/gateway/security).

## Формат (совместимый с AgentSkills + Pi)

`SKILL.md` должен содержать как минимум следующее:

```markdown
---
name: image-lab
description: Генерировать или редактировать изображения с помощью рабочего процесса, поддерживаемого провайдером
---
```

Примечания:

- Мы следуем спецификации AgentSkills для разметки/намерений.
- Парсер, используемый встроенным агентом, поддерживает только **однострочные** ключи заголовка.
- `metadata` должен быть **однострочным JSON-объектом**.
- Используйте `{baseDir}` в инструкциях для ссылки на путь к папке навыка.
- Необязательные ключи заголовка:
  - `homepage` — URL, отображаемый как "Веб-сайт" в пользовательском интерфейсе навыков macOS (также поддерживается через `metadata.openclaw.homepage`).
  - `user-invocable` — `true|false` (по умолчанию: `true`). Если `true`, навык предоставляется в виде слеш-команды для пользователя.
  - `disable-model-invocation` — `true|false` (по умолчанию: `false`). Если `true`, навык исключается из промта модели (но остаётся доступным через вызов пользователем).
  - `command-dispatch` — `tool` (необязательно). Если установлено значение `tool`, слеш-команда обходит модель и напрямую передаёт управление инструменту.
  - `command-tool` — имя инструмента, который нужно вызвать, если установлено `command-dispatch: tool`.
  - `command-arg-mode` — `raw` (по умолчанию). Для диспетчеризации инструментов передаёт строку необработанных аргументов инструменту (без основного парсинга).

    Инструмент вызывается с параметрами:
    `{ command: "<raw args>", commandName: "<slash command>", skillName: "<skill name>" }`.

## Ограничение доступа (фильтры во время загрузки)

OpenClaw **фильтрует навыки во время загрузки** с помощью `metadata` (однострочный JSON):

```markdown
---
name: image-lab
description: Генерировать или редактировать изображения с помощью рабочего процесса, поддерживаемого провайдером
metadata:
  {
    "openclaw":
      {
        "requires": { "bins": ["uv"], "env": ["GEMINI_API_KEY"], "config": ["browser.enabled"] },
        "primaryEnv": "GEMINI_API_KEY",
      },
  }
---
```

Поля в `metadata.openclaw`:

- `always: true` — всегда включать навык (пропускать другие ограничения).
- `emoji` — необязательный эмодзи, используемый в пользовательском интерфейсе навыков macOS.
- `homepage` — необязательный URL, отображаемый как "Веб-сайт" в пользовательском интерфейсе навыков macOS.
- `os` — необязательный список платформ (`darwin`, `linux`, `win32`). Если задано, навык доступен только на этих ОС.
- `requires.bins` — список; каждый элемент должен существовать в `PATH`.
- `requires.anyBins` — список; хотя бы один элемент должен существовать в `PATH`.
- `requires.env` — список; переменная окружения должна существовать **или** быть предоставлена в конфигурации.
- `requires.config` — список путей в `openclaw.json`, которые должны иметь истинное значение.
- `primaryEnv` — имя переменной окружения, связанной с `skills.entries.<name>.apiKey`.
- `install` — необязательный массив спецификаций установщика, используемых в пользовательском интерфейсе навыков macOS (brew/node/go/uv/download).

Примечание о песочнице:

- `requires.bins` проверяется на **хосте** во время загрузки навыка.
- Если агент работает в песочнице, исполняемый файл также должен существовать **внутри контейнера**. Установите его с помощью `agents.defaults.sandbox.docker.setupCommand` (или с помощью пользовательского образа). `setupCommand` выполняется один раз после создания контейнера.
- Установка пакетов также требует исходящего сетевого трафика, записываемой корневой файловой системы и пользователя root в песочнице.
- Пример: навык `summarize` (`skills/summarize/SKILL.md`) требует наличия CLI `summarize` в контейнере песочницы для выполнения там.

Пример установщика:

```markdown
---
name: gemini
description: Используйте Gemini CLI для помощи в кодировании и поиска в Google.
metadata:
  {
    "openclaw":
      {
        "emoji": "♊️",
        "requires": { "bins": ["gemini"] },
        "install":
          [
            {
              "id": "brew",
              "kind": "brew",
              "formula": "gemini-cli",
              "bins": ["gemini"],
              "label": "Установить Gemini CLI (brew)",
            },
          ],
      },
  }
---
```

Примечания:

- Если указано несколько установщиков, шлюз выбирает **один** предпочтительный вариант (brew, если доступен, в противном случае node).
- Если все установщики — `download`, OpenClaw выводит список всех записей, чтобы вы могли увидеть доступные артефакты.
- Спецификации установщика могут включать `os: ["darwin"|"linux"|"win32"]` для фильтрации вариантов по платформе.
- Установки Node учитывают `skills.install.nodeManager` в `openclaw.json` (по умолчанию: npm; варианты: npm/pnpm/yarn/bun). Это влияет только на **установку навыков**; среда выполнения шлюза должна оставаться Node (Bun не рекомендуется для WhatsApp/Telegram).
- Выбор установщика через шлюз основан на предпочтениях, а не только на Node: когда спецификации установки смешивают типы, OpenClaw предпочитает Homebrew, если `skills.install.preferBrew` включён и `brew` существует, затем `uv`, затем настроенный менеджер узлов, затем другие варианты, такие как `go` или `download`.
- Если каждая спецификация установки — `download`, OpenClaw отображает все варианты загрузки вместо выбора одного предпочтительного установщика.
- Установки Go: если `go` отсутствует, а `brew` доступен, шлюз сначала устанавливает Go через Homebrew и устанавливает `GOBIN` в `bin` Homebrew, когда это возможно.
- Установки через загрузку: `url` (обязательно), `archive` (`tar.gz` | `tar.bz2` | `zip`), `extract` (по умолчанию: автоматически при обнаружении архива), `stripComponents`, `targetDir` (по умолчанию: `~/.openclaw/tools/<skillKey>`).

Если `metadata.openclaw` отсутствует, навык всегда доступен (если не отключён в конфигурации или не заблокирован с помощью `skills.allowBundled` для встроенных навыков).

## Переопределения конфигурации (`~/.openclaw/openclaw.json`)

Встроенные/управляемые навыки можно включать/выключать и снабжать значениями окружения:

```json5
{
  skills: {
    entries: {
      "image-lab": {
        enabled: true,
        apiKey: { source: "env", provider: "default", id: "GEMINI_API_KEY" }, // или строковое значение
        env: {
          GEMINI_API_KEY: "GEMINI_KEY_HERE",
        },
        config: {
          endpoint: "https://example.invalid",
          model: "nano-pro",
        },
      },
      peekaboo: { enabled: true },
      sag: { enabled: false },
    },
  },
}
```

Примечание: если имя навыка содержит дефисы, заключите ключ в кавычки (JSON5 допускает ключи в кавычках).

Если вы хотите использовать генерацию/редактирование изображений внутри самого OpenClaw, используйте основной инструмент `image_generate` с `agents.defaults.imageGenerationModel` вместо встроенного навыка. Примеры навыков здесь предназначены для пользовательских или сторонних рабочих процессов.

Для встроенного анализа изображений используйте инструмент `image` с `agents.defaults.imageModel`. Для встроенной генерации/редактирования изображений используйте `image_generate` с `agents.defaults.imageGenerationModel`. Если вы выбираете `openai/*`, `google/*`, `fal/*` или другую модель изображений конкретного провайдера, добавьте также ключ авторизации/API этого провайдера.

Ключи конфигурации по умолчанию соответствуют **имени навыка**. Если навык определяет `metadata.openclaw.skillKey`, используйте этот ключ в `skills.entries`.

Правила:

- `enabled: false` отключает навык, даже если он встроен/установлен.
- `env`: внедряется **только если** переменная ещё не установлена в процессе.
- `apiKey`: удобство для навыков, которые объявляют `metadata.openclaw.primaryEnv`. Поддерживает строковое значение или объект SecretRef (`{ source, provider, id }`).
- `config`: необязательный набор пользовательских полей для каждого навыка; пользовательские ключи должны находиться здесь.
- `allowBundled`: необязательный разрешённый список только для **встроенных** навыков. Если задан, только встроенные навыки из списка доступны (управляемые/рабочие навыки не затрагиваются).

## Внедрение окружения (для каждого запуска агента)

Когда начинается запуск агента, OpenClaw:

1. Читает метаданные навыка.
2. Применяет любые `skills.entries.<key>.env` или `skills.entries.<key>.apiKey` к `process.env`.
3. Строит системный промпт с **подходящими** навыками.
4. Восстанавливает исходное окружение после завершения запуска.

Это **ограничено запуском агента**, а не глобальной средой оболочки.

## Снапшот сеанса (производительность)

OpenClaw создаёт снапшот подходящих навыков **при запуске сеанса** и повторно использует этот список для последующих ходов в том же сеансе. Изменения навыков или конфигурации вступают в силу в следующем новом сеансе.

Навыки также могут обновляться в середине сеанса, когда включён наблюдатель навыков или когда появляется новый подходящий удалённый узел (см. ниже). Думайте об этом как о **горячей перезагрузке**: обновлённый список подхватывается на следующем ходе агента.

Если разрешённый список навыков агента изменяется для этого сеанса, OpenClaw обновляет снапшот, чтобы видимые навыки оставались согласованными с текущим агентом.

## Удалённые узлы macOS (шлюз Linux)

Если шлюз работает на Linux, но подключён **узел macOS** с разрешённым `system.run` (безопасность утверждений Exec не установлена в `deny`), OpenClaw может считать навыки только для macOS подходящими, когда требуемые исполняемые файлы присутствуют на этом узле. Агент должен выполнять эти навыки с помощью инструмента `exec` с `host=node`.

Это зависит от того, что узел сообщает о поддержке команд, и от проверки бинарных файлов через `system.run`. Если узел macOS позже отключится, навыки останутся видимыми; вызовы могут завершиться ошибкой, пока узел не подключится снова.

## Наблюдатель навыков (автоматическое обновление)

По умолчанию OpenClaw отслеживает папки навыков и обновляет снапшот навыков, когда файлы `SKILL.md` изменяются. Настройте это в `skills.load`:

```json5
{
  skills: {
    load: {
      watch: true,
      watchDebounceMs: 250,
    },
  },
}
```

## Влияние на токены (список навыков)

Когда навыки подходят, OpenClaw внедряет компактный XML-список доступных навыков в системный промпт (через `formatSkillsForPrompt` в `pi-coding-agent`). Затраты детерминированы:

- **Базовые накладные расходы (только когда ≥1 навыка):** 195 символов.
- **На каждый навык:** 97 символов + длина XML-экранированных значений `<name>`, `<description>` и `<location>`.

Формула (символы):

```
total = 195 + Σ (97 + len(name_escaped) + len(description_escaped) + len(location_escaped))
```

Примечания:

- XML-экранирование преобразует `& < > " '` в сущности (`&amp;`, `&lt;` и т. д.), увеличивая длину.
- Количество токенов зависит от токенизатора модели. Грубая оценка в стиле OpenAI — ~4 символа/токен, то есть **97 символов ≈ 24 токена** на навык плюс длины ваших фактических полей.

## Жизненный цикл управляемых навыков

OpenClaw поставляет базовый набор навыков как **встроенные навыки** в составе установки (npm-пакет или OpenClaw.app). `~/.openclaw/skills` существует для локальных переопределений (например, закрепления/исправления навыка без изменения встроенной копии). Навыки рабочего пространства принадлежат пользователю и переопределяют оба варианта в случае конфликтов имён.
