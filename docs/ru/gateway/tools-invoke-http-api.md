# Вызов инструментов (HTTP)

Gateway в OpenClaw предоставляет простую HTTP-конечную точку для прямого вызова отдельного инструмента. Она всегда включена и использует аутентификацию Gateway вместе с политикой инструментов. Как и в случае с совместимой с OpenAI поверхностью `/v1/*`, аутентификация по принципу shared-secret (токен носителя) рассматривается как доступ доверенного оператора ко всему Gateway.

- `POST /tools/invoke`
- Тот же порт, что и у Gateway (мультиплексирование WS + HTTP): `http://<gateway-host>:<port>/tools/invoke`

Максимальный размер полезной нагрузки по умолчанию — 2 МБ.

## Аутентификация

Используется конфигурация аутентификации Gateway.

Распространённые пути HTTP-аутентификации:

- аутентификация по shared-secret (`gateway.auth.mode="token"` или `"password"`):
  `Authorization: Bearer <token-or-password>`
- доверенная HTTP-аутентификация с идентификацией (`gateway.auth.mode="trusted-proxy"`):
  направление через настроенный прокси с учётом идентификации и предоставление ему возможности вставить необходимые заголовки идентификации
- открытая аутентификация для частного входа (`gateway.auth.mode="none"`):
  заголовок аутентификации не требуется

Примечания:

- При `gateway.auth.mode="token"` используйте `gateway.auth.token` (или `OPENCLAW_GATEWAY_TOKEN`).
- При `gateway.auth.mode="password"` используйте `gateway.auth.password` (или `OPENCLAW_GATEWAY_PASSWORD`).
- При `gateway.auth.mode="trusted-proxy"` HTTP-запрос должен поступать от настроенного доверенного прокси-сервера, не являющегося loopback; прокси-серверы loopback на том же хосте не соответствуют этому режиму.
- Если настроена опция `gateway.auth.rateLimit` и происходит слишком много сбоев аутентификации, конечная точка возвращает код `429` с заголовком `Retry-After`.

## Границы безопасности (важно)

Рассматривайте эту конечную точку как поверхность **полного доступа оператора** к экземпляру Gateway.

- HTTP-аутентификация по принципу bearer здесь не является моделью с узким охватом для каждого пользователя.
- Действительный токен/пароль Gateway для этой конечной точки следует рассматривать как учётные данные владельца/оператора.
- Для режимов аутентификации по shared-secret (`token` и `password`) конечная точка восстанавливает стандартные настройки полного доступа оператора, даже если вызывающая сторона отправляет более узкий заголовок `x-openclaw-scopes`.
- Аутентификация по shared-secret также рассматривает прямые вызовы инструментов через эту конечную точку как действия отправителя-владельца.
- Режимы доверенной HTTP-аутентификации с идентификацией (например, аутентификация через доверенный прокси или `gateway.auth.mode="none"` на частном входе) учитывают `x-openclaw-scopes`, если этот заголовок присутствует, а в противном случае возвращаются к стандартному набору областей доступа оператора.
- Держите эту конечную точку только на loopback/tailnet/частном входе; не выставляйте её напрямую в публичный интернет.

Матрица аутентификации:

- `gateway.auth.mode="token"` или `"password"` + `Authorization: Bearer ...`
  - подтверждает владение общим секретом оператора Gateway;
  - игнорирует более узкий `x-openclaw-scopes`;
  - восстанавливает полный стандартный набор областей доступа оператора:
    `operator.admin`, `operator.approvals`, `operator.pairing`,
    `operator.read`, `operator.talk.secrets`, `operator.write`;
  - рассматривает прямые вызовы инструментов через эту конечную точку как действия отправителя-владельца.
- доверенные режимы HTTP-аутентификации с идентификацией (например, аутентификация через доверенный прокси или `gateway.auth.mode="none"` на частном входе)
  - аутентифицируют некоторую внешнюю доверенную идентичность или границу развёртывания;
  - учитывают `x-openclaw-scopes`, если заголовок присутствует;
  - возвращаются к стандартному набору областей доступа оператора, если заголовок отсутствует;
  - теряют семантику владельца только в том случае, если вызывающая сторона явно сужает области доступа и исключает `operator.admin`.

## Тело запроса

```json
{
  "tool": "sessions_list",
  "action": "json",
  "args": {},
  "sessionKey": "main",
  "dryRun": false
}
```

Поля:

- `tool` (строка, обязательно): название инструмента, который нужно вызвать.
- `action` (строка, необязательно): отображается в args, если схема инструмента поддерживает `action`, а полезная нагрузка args не содержит его.
- `args` (объект, необязательно): аргументы, специфичные для инструмента.
- `sessionKey` (строка, необязательно): целевой ключ сессии. Если он опущен или равен `"main"`, Gateway использует настроенный основной ключ сессии (учитывает `session.mainKey` и агента по умолчанию или `global` в глобальной области).
- `dryRun` (булево, необязательно): зарезервировано для будущего использования; в настоящее время игнорируется.

## Политика и поведение маршрутизации

Доступность инструментов фильтруется через ту же цепочку политик, которая используется агентами Gateway:

- `tools.profile` / `tools.byProvider.profile`
- `tools.allow` / `tools.byProvider.allow`
- `agents.<id>.tools.allow` / `agents.<id>.tools.byProvider.allow`
- групповые политики (если ключ сессии сопоставляется с группой или каналом)
- политика субагента (при вызове с ключом сессии субагента)

Если инструмент не разрешён политикой, конечная точка возвращает **404**.

Важные примечания о границах:

- Утверждения exec являются ограничителями для оператора, а не отдельной границей авторизации для этой HTTP-конечной точки. Если инструмент доступен здесь через аутентификацию Gateway + политику инструментов, `/tools/invoke` не добавляет дополнительного запроса на утверждение для каждого вызова.
- Не делитесь учётными данными bearer Gateway с ненадёжными вызывающими сторонами. Если вам нужно разделение по границам доверия, запустите отдельные Gateway (и, желательно, отдельные пользователи/хосты ОС).

Gateway HTTP также по умолчанию применяет жёсткий список запретов (даже если политика сессии разрешает инструмент):

- `exec` — прямое выполнение команд (поверхность RCE);
- `spawn` — произвольное создание дочерних процессов (поверхность RCE);
- `shell` — выполнение команд оболочки (поверхность RCE);
- `fs_write` — произвольное изменение файлов на хосте;
- `fs_delete` — произвольное удаление файлов на хосте;
- `fs_move` — произвольное перемещение/переименование файлов на хосте;
- `apply_patch` — применение патчей может перезаписать произвольные файлы;
- `sessions_spawn` — оркестрация сессий; удалённое создание агентов является RCE;
- `sessions_send` — внедрение сообщений между сессиями;
- `cron` — плоскость управления постоянной автоматизацией;
- `gateway` — плоскость управления Gateway; предотвращает перенастройку через HTTP;
- `nodes` — ретрансляция команд узла может достичь system.run на сопряжённых хостах;
- `whatsapp_login` — интерактивная настройка, требующая сканирования QR-кода в терминале; зависает на HTTP.

Вы можете настроить этот список запретов через `gateway.tools`:

```json5
{
  gateway: {
    tools: {
      // Дополнительные инструменты, которые нужно заблокировать через HTTP /tools/invoke
      deny: ["browser"],
      // Удаление инструментов из списка запретов по умолчанию
      allow: ["gateway"],
    },
  },
}
```

Чтобы помочь групповым политикам разрешить контекст, вы можете дополнительно установить:

- `x-openclaw-message-channel: <channel>` (например: `slack`, `telegram`);
- `x-openclaw-account-id: <accountId>` (если существует несколько учётных записей).

## Ответы

- `200` → `{ ok: true, result }`
- `400` → `{ ok: false, error: { type, message } }` (некорректный запрос или ошибка ввода инструмента)
- `401` → неавторизован
- `429` → ограничение частоты аутентификации (`Retry-After` установлен)
- `404` → инструмент недоступен (не найден или не включён в белый список)
- `405` → метод не разрешён
- `500` → `{ ok: false, error: { type, message } }` (неожиданная ошибка выполнения инструмента; сообщение очищено)
