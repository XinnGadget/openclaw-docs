# Сетевая модель

> Этот контент был объединён с разделом [Сеть](/network#core-model). Смотрите эту страницу для ознакомления с актуальным руководством.

Большинство операций проходит через Gateway (`openclaw gateway`) — единый долго работающий процесс, который управляет подключениями каналов и плоскостью управления WebSocket.

## Основные правила

- Рекомендуется использовать один Gateway на хост. Это единственный процесс, которому разрешено управлять сессией WhatsApp Web. Для спасательных ботов или строгой изоляции запустите несколько шлюзов с изолированными профилями и портами. См. [Несколько шлюзов](/gateway/multiple-gateways).
- Приоритет отдаётся loopback: по умолчанию Gateway WS использует `ws://127.0.0.1:18789`. Мастер создаёт аутентификацию на основе общего секрета (shared-secret) по умолчанию и обычно генерирует токен, даже для loopback. Для доступа не через loopback используйте допустимый путь аутентификации шлюза: аутентификацию по токену/паролю общего секрета или корректно настроенное развёртывание `trusted-proxy` для не-loopback. Настройки Tailnet/мобильных устройств обычно лучше работают через Tailscale Serve или другую конечную точку `wss://`, а не через необработанный tailnet `ws://`.
- Узлы (nodes) подключаются к Gateway WS через LAN, tailnet или SSH — в зависимости от необходимости. Устаревший TCP-мост был удалён.
- Хост canvas обслуживается HTTP-сервером Gateway на **том же порту**, что и Gateway (по умолчанию `18789`):
  - `/__openclaw__/canvas/`
  - `/__openclaw__/a2ui/`

  Когда `gateway.auth` настроен и Gateway выходит за пределы loopback, эти маршруты защищены аутентификацией Gateway. Клиенты узлов используют URL-адреса возможностей (capability URLs), привязанные к их активной WS-сессии. См. [Конфигурация Gateway](/gateway/configuration) (`canvasHost`, `gateway`).
- Для удалённого использования обычно применяется туннель SSH или VPN tailnet. См. [Удалённый доступ](/gateway/remote) и [Обнаружение](/gateway/discovery).
