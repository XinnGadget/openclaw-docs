# OpenShell

OpenShell — это управляемый бэкенд песочницы для OpenClaw. Вместо запуска контейнеров Docker локально OpenClaw делегирует управление жизненным циклом песочницы интерфейсу командной строки `openshell`, который подготавливает удалённые среды с выполнением команд через SSH.

Плагин OpenShell повторно использует тот же базовый транспорт SSH и мост удалённой файловой системы, что и общий [SSH-бэкенд](/gateway/sandboxing#ssh-backend). Он добавляет специфические для OpenShell операции жизненного цикла (`sandbox create/get/delete`, `sandbox ssh-config`) и дополнительный режим рабочей области `mirror`.

## Предварительные требования

- Установленный интерфейс командной строки `openshell` в переменной `PATH` (или заданный пользовательский путь через `plugins.entries.openshell.config.command`)
- Учётная запись OpenShell с доступом к песочнице
- OpenClaw Gateway, запущенный на хосте

## Быстрый старт

1. Включите плагин и задайте бэкенд песочницы:

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "all",
        backend: "openshell",
        scope: "session",
        workspaceAccess: "rw",
      },
    },
  },
  plugins: {
    entries: {
      openshell: {
        enabled: true,
        config: {
          from: "openclaw",
          mode: "remote",
        },
      },
    },
  },
}
```

2. Перезапустите Gateway. При следующем запуске агента OpenClaw создаст песочницу OpenShell и будет направлять выполнение инструментов через неё.

3. Проверьте:

```bash
openclaw sandbox list
openclaw sandbox explain
```

## Режимы рабочей области

Это наиболее важное решение при использовании OpenShell.

### `mirror`

Используйте `plugins.entries.openshell.config.mode: "mirror"`, если хотите, чтобы **локальная рабочая область оставалась основной**.

Поведение:

- Перед `exec` OpenClaw синхронизирует локальную рабочую область с песочницей OpenShell.
- После `exec` OpenClaw синхронизирует удалённую рабочую область обратно с локальной.
- Инструменты работы с файлами по-прежнему работают через мост песочницы, но локальная рабочая область остаётся источником истины между запусками.

Лучше всего подходит для:

- Вы редактируете файлы локально вне OpenClaw и хотите, чтобы эти изменения автоматически отображались в песочнице.
- Вы хотите, чтобы песочница OpenShell вела себя максимально похоже на бэкенд Docker.
- Вы хотите, чтобы рабочая область хоста отражала записи в песочнице после каждого запуска `exec`.

Компромисс: дополнительные затраты на синхронизацию до и после каждого `exec`.

### `remote`

Используйте `plugins.entries.openshell.config.mode: "remote"`, если хотите, чтобы **рабочая область OpenShell стала основной**.

Поведение:

- При первом создании песочницы OpenClaw один раз заполняет удалённую рабочую область из локальной.
- После этого операции `exec`, `read`, `write`, `edit` и `apply_patch` выполняются непосредственно в удалённой рабочей области OpenShell.
- OpenClaw **не** синхронизирует удалённые изменения обратно в локальную рабочую область.
- Чтение медиафайлов во время запроса по-прежнему работает, поскольку инструменты работы с файлами и медиа читают данные через мост песочницы.

Лучше всего подходит для:

- Песочница должна находиться преимущественно на удалённой стороне.
- Вы хотите снизить накладные расходы на синхронизацию при каждом запуске.
- Вы не хотите, чтобы локальные изменения на хосте незаметно перезаписывали состояние удалённой песочницы.

Важно: если вы редактируете файлы на хосте вне OpenClaw после первоначального заполнения, удалённая песочница **не** увидит эти изменения. Используйте `openclaw sandbox recreate`, чтобы повторно заполнить песочницу.

### Выбор режима

| | `mirror` | `remote` |
| --- | --- | --- |
| **Основная рабочая область** | Локальный хост | Удалённая OpenShell |
| **Направление синхронизации** | Двунаправленная (при каждом `exec`) | Однократное заполнение |
| **Накладные расходы при каждом запуске** | Выше (загрузка + выгрузка) | Ниже (прямые операции с удалённой областью) |
| **Видно ли локальные изменения?** | Да, при следующем `exec` | Нет, до повторного создания |
| **Лучше всего подходит для** | Рабочих процессов разработки | Долго работающих агентов, CI |

## Справочная информация по конфигурации

Вся конфигурация OpenShell находится в разделе `plugins.entries.openshell.config`:

| Ключ | Тип | Значение по умолчанию | Описание |
| --- | --- | --- | --- |
| `mode` | `"mirror"` или `"remote"` | `"mirror"` | Режим синхронизации рабочей области |
| `command` | `string` | `"openshell"` | Путь или имя интерфейса командной строки `openshell` |
| `from` | `string` | `"openclaw"` | Источник песочницы для первого создания |
| `gateway` | `string` | — | Имя шлюза OpenShell (`--gateway`) |
| `gatewayEndpoint` | `string` | — | URL-адрес конечной точки шлюза OpenShell (`--gateway-endpoint`) |
| `policy` | `string` | — | ID политики OpenShell для создания песочницы |
| `providers` | `string[]` | `[]` | Имена провайдеров, которые нужно подключить при создании песочницы |
| `gpu` | `boolean` | `false` | Запрос ресурсов GPU |
| `autoProviders` | `boolean` | `true` | Передача `--auto-providers` при создании песочницы |
| `remoteWorkspaceDir` | `string` | `"/sandbox"` | Основная записываемая рабочая область внутри песочницы |
| `remoteAgentWorkspaceDir` | `string` | `"/agent"` | Путь монтирования рабочей области агента (для доступа только для чтения) |
| `timeoutSeconds` | `number` | `120` | Таймаут для операций интерфейса командной строки `openshell` |

Настройки уровня песочницы (`mode`, `scope`, `workspaceAccess`) задаются в разделе `agents.defaults.sandbox`, как и для любого другого бэкенда. См. [Sandboxing](/gateway/sandboxing) для полной матрицы.

## Примеры

### Минимальная настройка для удалённого режима

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "all",
        backend: "openshell",
      },
    },
  },
  plugins: {
    entries: {
      openshell: {
        enabled: true,
        config: {
          from: "openclaw",
          mode: "remote",
        },
      },
    },
  },
}
```

### Режим зеркалирования с GPU

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "all",
        backend: "openshell",
        scope: "agent",
        workspaceAccess: "rw",
      },
    },
  },
  plugins: {
    entries: {
      openshell: {
        enabled: true,
        config: {
          from: "openclaw",
          mode: "mirror",
          gpu: true,
          providers: ["openai"],
          timeoutSeconds: 180,
        },
      },
    },
  },
}
```

### OpenShell для каждого агента с пользовательским шлюзом

```json5
{
  agents: {
    defaults: {
      sandbox: { mode: "off" },
    },
    list: [
      {
        id: "researcher",
        sandbox: {
          mode: "all",
          backend: "openshell",
          scope: "agent",
          workspaceAccess: "rw",
        },
      },
    ],
  },
  plugins: {
    entries: {
      openshell: {
        enabled: true,
        config: {
          from: "openclaw",
          mode: "remote",
          gateway: "lab",
          gatewayEndpoint: "https://lab.example",
          policy: "strict",
        },
      },
    },
  },
}
```

## Управление жизненным циклом

Песочницы OpenShell управляются через обычный интерфейс командной строки песочницы:

```bash
# Вывести список всех сред выполнения песочницы (Docker + OpenShell)
openclaw sandbox list

# Проверить действующую политику
openclaw sandbox explain

# Пересоздать (удаляет удалённую рабочую область, повторно заполняет при следующем использовании)
openclaw sandbox recreate --all
```
