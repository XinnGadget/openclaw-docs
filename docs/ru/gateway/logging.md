# Логирование

Для общего обзора (CLI + пользовательский интерфейс управления + конфигурация) см. [/logging](/logging).

В OpenClaw есть две "поверхности" логирования:

- **Вывод в консоль** (то, что вы видите в терминале / интерфейсе отладки).
- **Файловые логи** (строки JSON), записываемые логгером шлюза.

## Логгер на основе файлов

- Файл журнала с ротацией по умолчанию находится в каталоге `/tmp/openclaw/` (один файл в день): `openclaw-ГГГГ-ММ-ДД.log`.
  - Дата указывается в локальном часовом поясе хоста шлюза.
- Путь к файлу журнала и уровень логирования можно настроить через `~/.openclaw/openclaw.json`:
  - `logging.file`
  - `logging.level`

Формат файла — один объект JSON на строку.

Вкладка "Логи" в пользовательском интерфейсе управления отслеживает этот файл через шлюз (`logs.tail`).
CLI может делать то же самое:

```bash
openclaw logs --follow
```

**Подробный режим и уровни логирования**

- **Файловые логи** регулируются исключительно параметром `logging.level`.
- Параметр `--verbose` влияет только на **подробность вывода в консоль** (и стиль логов WS); он **не** повышает уровень логирования в файлах.
- Чтобы записать в файловые логи сведения, доступные только в подробном режиме, установите для `logging.level` значение `debug` или `trace`.

## Захват консоли

CLI перехватывает вызовы `console.log/info/warn/error/debug/trace` и записывает их в файловые логи, одновременно выводя в stdout/stderr.

Вы можете независимо настроить подробность вывода в консоль с помощью:

- `logging.consoleLevel` (по умолчанию `info`)
- `logging.consoleStyle` (`pretty` | `compact` | `json`)

## Редактирование сводок инструментов

Подробные сводки инструментов (например, `🛠️ Exec: ...`) могут маскировать конфиденциальные токены до того, как они попадут в поток консоли. Это относится **только к инструментам** и не влияет на файловые логи.

- `logging.redactSensitive`: `off` | `tools` (по умолчанию: `tools`)
- `logging.redactPatterns`: массив строк с регулярными выражениями (переопределяет значения по умолчанию)
  - Используйте необработанные строки с регулярными выражениями (автоматически `gi`) или `/pattern/flags`, если вам нужны пользовательские флаги.
  - Совпадения маскируются путём сохранения первых 6 и последних 4 символов (длина >= 18), в противном случае — `***`.
  - Значения по умолчанию охватывают распространённые назначения ключей, флаги CLI, поля JSON, заголовки bearer, блоки PEM и популярные префиксы токенов.

## Логи WebSocket шлюза

Шлюз выводит логи протокола WebSocket в двух режимах:

- **Обычный режим (без `--verbose`)**: выводятся только "интересные" результаты RPC:
  - ошибки (`ok=false`)
  - медленные вызовы (порог по умолчанию: `>= 50 мс`)
  - ошибки парсинга
- **Подробный режим (`--verbose`)**: выводятся все запросы и ответы WS.

### Стиль логов WS

`openclaw gateway` поддерживает переключатель стиля для каждого шлюза:

- `--ws-log auto` (по умолчанию): обычный режим оптимизирован; в подробном режиме используется компактный вывод
- `--ws-log compact`: компактный вывод (парные запросы/ответы) в подробном режиме
- `--ws-log full`: полный вывод для каждого кадра в подробном режиме
- `--compact`: псевдоним для `--ws-log compact`

Примеры:

```bash
# оптимизированный вывод (только ошибки/медленные вызовы)
openclaw gateway

# показать весь трафик WS (парные запросы/ответы)
openclaw gateway --verbose --ws-log compact

# показать весь трафик WS (полная метаинформация)
openclaw gateway --verbose --ws-log full
```

## Форматирование консоли (логирование подсистем)

Форматировщик консоли **учитывает TTY** и выводит согласованные строки с префиксами.
Логгеры подсистем группируют вывод, делая его удобным для просмотра.

Поведение:

- **Префиксы подсистем** в каждой строке (например, `[gateway]`, `[canvas]`, `[tailscale]`)
- **Цвета подсистем** (стабильные для каждой подсистемы) плюс цветовая разметка уровней
- **Цвет используется, если вывод идёт в TTY или среда выглядит как расширенная консоль** (`TERM`/`COLORTERM`/`TERM_PROGRAM`), учитывается `NO_COLOR`
- **Сокращённые префиксы подсистем**: удаляются начальные `gateway/` + `channels/`, сохраняются последние 2 сегмента (например, `whatsapp/outbound`)
- **Подлоггеры по подсистемам** (автоматический префикс + структурированное поле `{ subsystem }`)
- **`logRaw()`** для вывода QR/UX (без префикса и форматирования)
- **Стили консоли** (например, `pretty | compact | json`)
- **Уровень логирования в консоли** отдельно от уровня логирования в файле (в файле сохраняются все детали, если для `logging.level` установлено значение `debug`/`trace`)
- **Тела сообщений WhatsApp** записываются на уровне `debug` (чтобы увидеть их, используйте `--verbose`)

Это позволяет сохранить стабильность существующих файловых логов, одновременно делая интерактивный вывод удобным для просмотра.
