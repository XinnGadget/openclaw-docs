# Настройка

<Note>
Если вы настраиваете систему впервые, начните с [Начало работы](/start/getting-started).
Для деталей по адаптации см. [Адаптация (CLI)](/start/wizard).
</Note>

## Кратко

- **Настройка находится вне репозитория:** `~/.openclaw/workspace` (рабочая область) + `~/.openclaw/openclaw.json` (конфигурация).
- **Стабильный рабочий процесс:** установите приложение для macOS; позвольте ему запустить встроенный Gateway.
- **Рабочий процесс с самыми свежими наработками:** запустите Gateway самостоятельно через `pnpm gateway:watch`, затем позвольте приложению для macOS подключиться в локальном режиме.

## Предварительные требования (из исходного кода)

- Рекомендуется Node 24 (Node 22 LTS, в настоящее время `22.14+`, по-прежнему поддерживается).
- Предпочтительно использовать `pnpm` (или Bun, если вы намеренно используете [рабочий процесс Bun](/install/bun)).
- Docker (необязательно; только для контейнерной настройки/e2e — см. [Docker](/install/docker)).

## Стратегия настройки (чтобы обновления не причиняли вреда)

Если вы хотите, чтобы всё было "на 100 % подстроено под меня" _и_ чтобы обновления были простыми, храните свои настройки в:

- **Конфигурация:** `~/.openclaw/openclaw.json` (JSON/JSON5-подобный формат).
- **Рабочая область:** `~/.openclaw/workspace` (навыки, подсказки, воспоминания; сделайте из неё приватный git-репозиторий).

Однократно выполните начальную настройку:

```bash
openclaw setup
```

Изнутри этого репозитория используйте локальную точку входа CLI:

```bash
openclaw setup
```

Если у вас ещё нет глобальной установки, запустите её через `pnpm openclaw setup` (или `bun run openclaw setup`, если вы используете рабочий процесс Bun).

## Запуск Gateway из этого репозитория

После `pnpm build` вы можете запустить упакованный CLI напрямую:

```bash
node openclaw.mjs gateway --port 18789 --verbose
```

## Стабильный рабочий процесс (сначала приложение для macOS)

1. Установите и запустите **OpenClaw.app** (панель меню).
2. Выполните чек-лист адаптации/разрешений (подсказки TCC).
3. Убедитесь, что Gateway находится в режиме **Local** и запущен (приложение управляет им).
4. Свяжите каналы (например, WhatsApp):

```bash
openclaw channels login
```

5. Проверка работоспособности:

```bash
openclaw health
```

Если адаптация недоступна в вашей сборке:

- Запустите `openclaw setup`, затем `openclaw channels login`, затем вручную запустите Gateway (`openclaw gateway`).

## Рабочий процесс с самыми свежими наработками (Gateway в терминале)

Цель: работать над TypeScript Gateway, получить горячую перезагрузку, сохранить подключённый интерфейс приложения для macOS.

### 0) (необязательно) Запустите приложение для macOS из исходного кода

Если вы также хотите, чтобы приложение для macOS было с самыми свежими наработками:

```bash
./scripts/restart-mac.sh
```

### 1) Запустите Gateway для разработки

```bash
pnpm install
pnpm gateway:watch
```

`gateway:watch` запускает Gateway в режиме наблюдения и перезагружает его при соответствующих изменениях исходного кода, конфигурации и метаданных встроенных плагинов.

Если вы намеренно используете рабочий процесс Bun, эквивалентные команды будут следующими:

```bash
bun install
bun run gateway:watch
```

### 2) Укажите приложению для macOS на запущенный Gateway

В **OpenClaw.app**:

- Режим подключения: **Local**
  Приложение подключится к запущенному Gateway на настроенном порту.

### 3) Проверка

- Статус Gateway в приложении должен отображаться как **"Using existing gateway …"** ("Используется существующий Gateway…").
- Или через CLI:

```bash
openclaw health
```

### Распространённые ошибки

- **Неправильный порт:** по умолчанию Gateway WS использует `ws://127.0.0.1:18789`; убедитесь, что приложение и CLI используют один и тот же порт.
- **Где хранится состояние:**
  - Состояние канала/провайдера: `~/.openclaw/credentials/`
  - Профили аутентификации модели: `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`
  - Сессии: `~/.openclaw/agents/<agentId>/sessions/`
  - Логи: `/tmp/openclaw/`

## Карта хранения учётных данных

Используйте её при отладке аутентификации или при принятии решения о том, что нужно резервировать:

- **WhatsApp:** `~/.openclaw/credentials/whatsapp/<accountId>/creds.json`
- **Токен бота Telegram:** конфигурация/окружение или `channels.telegram.tokenFile` (только обычный файл; символические ссылки не принимаются)
- **Токен бота Discord:** конфигурация/окружение или SecretRef (провайдеры env/file/exec)
- **Токены Slack:** конфигурация/окружение (`channels.slack.*`)
- **Списки разрешённых подключений:**
  - `~/.openclaw/credentials/<channel>-allowFrom.json` (учётная запись по умолчанию)
  - `~/.openclaw/credentials/<channel>-<accountId>-allowFrom.json` (нестандартные учётные записи)
- **Профили аутентификации модели:** `~/.openclaw/agents/<agentId>/agent/auth-profiles.json`
- **Полезная нагрузка секретов на основе файлов (необязательно):** `~/.openclaw/secrets.json`
- **Импорт устаревшего OAuth:** `~/.openclaw/credentials/oauth.json`
  Подробнее: [Безопасность](/gateway/security#credential-storage-map).

## Обновление (без нарушения вашей настройки)

- Храните `~/.openclaw/workspace` и `~/.openclaw/` как "ваши данные"; не помещайте персональные подсказки/конфигурацию в репозиторий `openclaw`.
- Обновление исходного кода: `git pull` + шаг установки менеджера пакетов по вашему выбору (`pnpm install` по умолчанию; `bun install` для рабочего процесса Bun) + продолжайте использовать соответствующую команду `gateway:watch`.

## Linux (служба пользователя systemd)

В установках Linux используется служба **пользователя** systemd. По умолчанию systemd останавливает службы пользователя при выходе из системы/бездействии, что приводит к завершению работы Gateway. Процесс адаптации пытается включить для вас режим lingering (может запросить sudo). Если он всё ещё отключён, выполните:

```bash
sudo loginctl enable-linger $USER
```

Для серверов с постоянным доступом или для многопользовательских серверов рассмотрите возможность использования **системной** службы вместо службы пользователя (режим lingering не требуется). См. [Руководство по Gateway](/gateway) для заметок по systemd.
