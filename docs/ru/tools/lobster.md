# Лобстер

Лобстер — это оболочка рабочих процессов, которая позволяет OpenClaw выполнять последовательности инструментов из нескольких шагов как единую детерминированную операцию с явными контрольными точками утверждения.

Лобстер располагается на один уровень выше фоновой работы в отдельном режиме. Для оркестрации потоков над отдельными задачами см. [Task Flow](/automation/taskflow) (`openclaw tasks flow`). Для журнала активности задач см. [`openclaw tasks`](/automation/tasks).

## Хук

Ваш ассистент может создавать инструменты, которые управляют сами собой. Запросите рабочий процесс — и через 30 минут у вас будет интерфейс командной строки (CLI) плюс конвейеры, запускаемые одним вызовом. Лобстер — недостающий элемент: детерминированные конвейеры, явные утверждения и возобновляемое состояние.

## Зачем это нужно

Сегодня сложные рабочие процессы требуют множества последовательных вызовов инструментов. Каждый вызов расходует токены, а LLM должна координировать каждый шаг. Лобстер переносит эту оркестрацию в типизированную среду выполнения:

- **Один вызов вместо множества**: OpenClaw выполняет один вызов инструмента Лобстера и получает структурированный результат.
- **Встроенные утверждения**: побочные эффекты (отправка электронной почты, публикация комментария) приостанавливают рабочий процесс до явного утверждения.
- **Возможность возобновления**: приостановленные рабочие процессы возвращают токен; утвердите и возобновите выполнение без повторного запуска всего процесса.

## Почему DSL, а не обычные программы?

Лобстер намеренно компактен. Цель — не "новый язык", а предсказуемая спецификация конвейера, удобная для ИИ, с поддержкой утверждений и токенов возобновления на уровне основных возможностей.

- **Утверждение/возобновление встроено**: обычная программа может запросить подтверждение у человека, но не может _приостановить и возобновить_ работу с устойчивым токеном без самостоятельной реализации такой среды выполнения.
- **Детерминизм + возможность аудита**: конвейеры — это данные, поэтому их легко логировать, сравнивать, воспроизводить и проверять.
- **Ограниченная область для ИИ**: компактная грамматика плюс передача данных в формате JSON сокращают "творческие" пути выполнения кода и делают валидацию реалистичной.
- **Встроенные политики безопасности**: тайм-ауты, ограничения на объём вывода, проверки в песочнице и белые списки обеспечиваются средой выполнения, а не каждым скриптом.
- **По-прежнему программируемый**: каждый шаг может вызывать любой CLI или скрипт. Если вам нужен JS/TS, генерируйте файлы `.lobster` из кода.

## Как это работает

OpenClaw запускает рабочие процессы Лобстера **внутри процесса** с использованием встроенного раннера. Внешние подпроцессы CLI не создаются; движок рабочего процесса выполняется внутри процесса шлюза и напрямую возвращает JSON-конверт.

Если конвейер приостанавливается для утверждения, инструмент возвращает `resumeToken`, чтобы вы могли продолжить позже.

## Паттерн: небольшой CLI + передача данных в формате JSON + утверждения

Создавайте небольшие команды, работающие с JSON, а затем объединяйте их в один вызов Лобстера (приведённые ниже названия команд можно заменить на свои):

```bash
inbox list --json
inbox categorize --json
inbox apply --json
```

```json
{
  "action": "run",
  "pipeline": "exec --json --shell 'inbox list --json' | exec --stdin json --shell 'inbox categorize --json' | exec --stdin json --shell 'inbox apply --json' | approve --preview-from-stdin --limit 5 --prompt 'Apply changes?'",
  "timeoutMs": 30000
}
```

Если конвейер запрашивает утверждение, возобновите выполнение с помощью токена:

```json
{
  "action": "resume",
  "token": "<resumeToken>",
  "approve": true
}
```

ИИ запускает рабочий процесс, Лобстер выполняет шаги. Точки утверждения делают побочные эффекты явными и поддающимися аудиту.

Пример: преобразование входных элементов в вызовы инструментов:

```bash
gog.gmail.search --query 'newer_than:1d' \
  | openclaw.invoke --tool message --action send --each --item-key message --args-json '{"provider":"telegram","to":"..."}'
```

## Шаги LLM только с JSON (llm-task)

Для рабочих процессов, требующих **структурированного шага LLM**, включите дополнительный инструмент-плагин `llm-task` и вызывайте его из Лобстера. Это сохраняет детерминированность рабочего процесса, позволяя при этом классифицировать, обобщать и составлять черновики с помощью модели.

Включите инструмент:

```json
{
  "plugins": {
    "entries": {
      "llm-task": { "enabled": true }
    }
  },
  "agents": {
    "list": [
      {
        "id": "main",
        "tools": { "allow": ["llm-task"] }
      }
    ]
  }
}
```

Используйте его в конвейере:

```lobster
openclaw.invoke --tool llm-task --action json --args-json '{
  "prompt": "Given the input email, return intent and draft.",
  "thinking": "low",
  "input": { "subject": "Hello", "body": "Can you help?" },
  "schema": {
    "type": "object",
    "properties": {
      "intent": { "type": "string" },
      "draft": { "type": "string" }
    },
    "required": ["intent", "draft"],
    "additionalProperties": false
  }
}'
```

Подробнее о настройках и параметрах конфигурации см. в разделе [LLM Task](/tools/llm-task).

## Файлы рабочих процессов (.lobster)

Лобстер может запускать YAML/JSON-файлы рабочих процессов с полями `name`, `args`, `steps`, `env`, `condition` и `approval`. В вызовах инструментов OpenClaw задайте для `pipeline` путь к файлу.

```yaml
name: inbox-triage
args:
  tag:
    default: "family"
steps:
  - id: collect
    command: inbox list --json
  - id: categorize
    command: inbox categorize --json
    stdin: $collect.stdout
  - id: approve
    command: inbox apply --approve
    stdin: $categorize.stdout
    approval: required
  - id: execute
    command: inbox apply --execute
    stdin: $categorize.stdout
    condition: $approve.approved
```

Примечания:

- `stdin: $step.stdout` и `stdin: $step.json` передают выходные данные предыдущего шага.
- `condition` (или `when`) может управлять выполнением шагов на основе `$step.approved`.

## Установка Лобстера

Встроенные рабочие процессы Лобстера выполняются внутри процесса; отдельный бинарный файл `lobster` не требуется. Встроенный раннер поставляется с плагином Лобстера.

Если вам нужен автономный CLI Лобстера для разработки или внешних конвейеров, установите его из [репозитория Лобстера](https://github.com/openclaw/lobster) и убедитесь, что `lobster` добавлен в `PATH`.

## Включение инструмента

Лобстер — **опциональный** инструмент-плагин (по умолчанию не включён).

Рекомендуемый способ (добавочный, безопасный):

```json
{
  "tools": {
    "alsoAllow": ["lobster"]
  }
}
```

Или для конкретного агента:

```json
{
  "agents": {
    "list": [
      {
        "id": "main",
        "tools": {
          "alsoAllow": ["lobster"]
        }
      }
    ]
  }
}
```

Избегайте использования `tools.allow: ["lobster"]`, если вы не планируете работать в режиме строгого белого списка.

Примечание: белые списки подключаются для опциональных плагинов. Если в вашем белом списке указаны только инструменты-плагины (например, `lobster`), OpenClaw оставляет основные инструменты включёнными. Чтобы ограничить основные инструменты, включите нужные основные инструменты или группы в белый список.

## Пример: сортировка электронной почты

Без Лобстера:

```
Пользователь: "Проверь мою электронную почту и составь ответы"
→ openclaw вызывает gmail.list
→ LLM обобщает данные
→ Пользователь: "Составь ответы для #2 и #5"
→ LLM составляет черновики
→ Пользователь: "Отправь #2"
→ openclaw вызывает gmail.send
(повторяется ежедневно, без сохранения информации о том, что было отсортировано)
```

С Лобстером:

```json
{
  "action": "run",
  "pipeline": "email.triage --limit 20",
  "timeoutMs": 30000
}
```

Возвращает JSON-конверт (сокращённо):

```json
{
  "ok": true,
  "status": "needs_approval",
  "output": [{ "summary": "5 need replies, 2 need action" }],
  "requiresApproval": {
    "type": "approval_request",
    "prompt": "Send 2 draft replies?",
    "items": [],
    "resumeToken": "..."
  }
}
```

Пользователь утверждает → возобновление:

```json
{
  "action": "resume",
  "token": "<resumeToken>",
  "approve": true
}
```

Один рабочий процесс. Детерминированный. Безопасный.

## Параметры инструмента

### `run`

Запуск конвейера в режиме инструмента.

```json
{
  "action": "run",
  "pipeline": "gog.gmail.search --query 'newer_than:1d' | email.triage",
  "cwd": "workspace",
  "timeoutMs": 30000,
  "maxStdoutBytes": 512000
}
```

Запуск файла рабочего процесса с аргументами:

```json
{
  "action": "run",
  "pipeline": "/path/to/inbox-triage.lobster",
  "argsJson": "{\"tag\":\"family\"}"
}
```

### `resume`

Продолжение приостановленного рабочего процесса после утверждения.

```json
{
  "action": "resume",
  "token": "<resumeToken>",
  "approve": true
}
```

### Необязательные входные данные

- `cwd`: относительная рабочая директория для конвейера (должна находиться в пределах рабочей директории шлюза).
- `timeoutMs`: прерывание рабочего процесса, если он превышает заданную продолжительность (по умолчанию: 20 000).
- `maxStdoutBytes`: прерывание рабочего процесса, если объём вывода превышает заданный размер (по умолчанию: 512 000).
- `argsJson`: строка JSON, передаваемая в `lobster run --args-json` (только для файлов рабочих процессов).

## JSON-конверт вывода

Лобстер возвращает JSON-конверт с одним из трёх статусов:

- `ok` → успешно завершено;
- `needs_approval` → приостановлено; для возобновления требуется `requiresApproval.resumeToken`;
- `cancelled` → явно отклонено или отменено.

Инструмент предоставляет конверт как в `content` (отформатированный JSON), так и в `details` (сырой объект).

## Утверждения

Если присутствует `requiresApproval`, изучите запрос и примите решение:

- `approve: true` → возобновить и продолжить побочные эффекты;
- `approve: false` → отменить и завершить рабочий процесс.

Используйте `approve --preview-from-stdin --limit N`, чтобы прикрепить предварительный просмотр JSON к запросам на утверждение без использования пользовательских конструкций jq/heredoc. Токены возобновления теперь компактны: Лобстер хранит состояние возобновления рабочего процесса в своём каталоге состояний и возвращает компактный ключ токена.

## OpenProse

OpenProse хорошо сочетается с Лобстером: используйте `/prose` для оркестрации подготовки с участием нескольких агентов, затем запустите конвейер Лобстера для детерминированных утверждений. Если программе Prose нужен Лобстер, разрешите инструмент `lobster` для субагентов через `tools.subagents.tools`. См. [OpenProse](/prose).

## Безопасность

- **Только локальное выполнение внутри процесса** — рабочие процессы выполняются внутри процесса шлюза; плагин сам по себе не выполняет сетевых вызовов.
- **Отсутствие секретов** — Лобстер не управляет OAuth; он вызывает инструменты OpenClaw, которые это делают.
- **Учёт песочницы** — отключено, если контекст инструмента находится в песочнице.
- **Усиленная защита** — тайм-ауты и ограничения на объём вывода обеспечиваются встроенным раннером.

## Устранение неполадок

- **`lobster timed out`** → увеличьте `timeoutMs` или разделите длинный конвейер.
- **`lobster output exceeded maxStdoutBytes`** → увеличьте `maxStdoutBytes` или сократите объём вывода.
- **`lobster returned invalid JSON`** → убедитесь, что конвейер выполняется в режиме инструмента и выводит только JSON.
- **`lobster failed`** → проверьте журналы шлюза для получения подробной информации об ошибках встроенного раннера.

## Кейс: рабочие процессы сообщества

Один публичный пример: CLI "второго мозга" + конвейеры Лобстера, управляющие тремя хранилищами Markdown (личное, для партнёра, общее). CLI выводит JSON для статистики, списков входящих данных и сканирования устаревших элементов; Лобстер объединяет эти команды в рабочие процессы вроде `weekly-review`, `inbox-triage`, `memory-consolidation` и `shared-task-sync`, каждый с точками утверждения. ИИ выполняет оценку (категоризацию), когда это возможно, и переходит к детерминированным правилам, когда это невозможно.
