# Аутентификация (поставщики моделей)

<Note>
На этой странице рассматривается аутентификация **поставщиков моделей** (API-ключи, OAuth, повторное использование Claude CLI и токен настройки Anthropic). Для аутентификации **подключения к шлюзу** (токен, пароль, доверенный прокси) см. [Конфигурацию](/gateway/configuration) и [Аутентификацию через доверенный прокси](/gateway/trusted-proxy-auth).
</Note>

OpenClaw поддерживает OAuth и API-ключи для поставщиков моделей. Для постоянно работающих хостов шлюза API-ключи обычно являются наиболее предсказуемым вариантом. Потоки подписки/OAuth также поддерживаются, если они соответствуют модели учётной записи вашего поставщика.

См. [/concepts/oauth](/concepts/oauth) для полного описания потока OAuth и структуры хранения.
Для аутентификации на основе SecretRef (`env`/`file`/`exec` поставщики) см. [Управление секретами](/gateway/secrets).
Для правил соответствия учётных данных/кодов причин, используемых командой `models status --probe`, см. [Семантику учётных данных аутентификации](/auth-credential-semantics).

## Рекомендуемая настройка (API-ключ, любой поставщик)

Если вы используете долго работающий шлюз, начните с API-ключа для выбранного вами поставщика.
Для Anthropic в частности аутентификация через API-ключ по-прежнему является наиболее предсказуемой настройкой сервера, но OpenClaw также поддерживает повторное использование локального входа в Claude CLI.

1. Создайте API-ключ в консоли вашего поставщика.
2. Разместите его на **хосте шлюза** (на машине, где выполняется `openclaw gateway`).

```bash
export <PROVIDER>_API_KEY="..."
openclaw models status
```

3. Если шлюз работает под управлением systemd/launchd, предпочтительно разместить ключ в `~/.openclaw/.env`, чтобы демон мог его прочитать:

```bash
cat >> ~/.openclaw/.env <<'EOF'
<PROVIDER>_API_KEY=...
EOF
```

Затем перезапустите демон (или перезапустите процесс шлюза) и проверьте снова:

```bash
openclaw models status
openclaw doctor
```

Если вы не хотите самостоятельно управлять переменными окружения, процесс подключения может сохранить API-ключи для использования демоном: `openclaw onboard`.

См. [Справку](/help) для подробностей о наследовании переменных окружения (`env.shellEnv`, `~/.openclaw/.env`, systemd/launchd).

## Anthropic: совместимость Claude CLI и токена

Аутентификация через токен настройки Anthropic по-прежнему доступна в OpenClaw как поддерживаемый путь использования токена. Сотрудники Anthropic сообщили нам, что использование Claude CLI в стиле OpenClaw снова разрешено, поэтому OpenClaw рассматривает повторное использование Claude CLI и использование `claude -p` как допустимые для этой интеграции, если Anthropic не опубликует новую политику. Когда повторное использование Claude CLI доступно на хосте, это теперь предпочтительный путь.

Для долго работающих хостов шлюза API-ключ Anthropic по-прежнему является наиболее предсказуемой настройкой. Если вы хотите повторно использовать существующий вход в Claude на том же хосте, используйте путь Anthropic Claude CLI при подключении/настройке.

Ввод токена вручную (любой поставщик; записывает `auth-profiles.json` и обновляет конфигурацию):

```bash
openclaw models auth paste-token --provider openrouter
```

Ссылки на профили аутентификации также поддерживаются для статических учётных данных:

- Учётные данные `api_key` могут использовать `keyRef: { source, provider, id }`
- Учётные данные `token` могут использовать `tokenRef: { source, provider, id }`
- Профили в режиме OAuth не поддерживают учётные данные SecretRef; если `auth.profiles.<id>.mode` установлен в `"oauth"`, ввод `keyRef`/`tokenRef`, поддерживаемый SecretRef, для этого профиля отклоняется.

Проверка, удобная для автоматизации (возвращает `1`, если срок действия истёк/данные отсутствуют, `2`, если срок действия истекает):

```bash
openclaw models status --check
```

Пробы аутентификации в реальном времени:

```bash
openclaw models status --probe
```

Примечания:

- Строки пробы могут поступать из профилей аутентификации, учётных данных окружения или `models.json`.
- Если явный `auth.order.<provider>` исключает сохранённый профиль, проба сообщает `excluded_by_auth_order` для этого профиля вместо того, чтобы пытаться его использовать.
- Если аутентификация существует, но OpenClaw не может определить кандидата модели для пробы для этого поставщика, проба сообщает `status: no_model`.
- Охлаждение при ограничении частоты может быть привязано к модели. Профиль, находящийся в режиме охлаждения для одной модели, всё ещё может использоваться для родственной модели у того же поставщика.

Дополнительные скрипты операций (systemd/Termux) описаны здесь:
[Скрипты мониторинга аутентификации](/help/scripts#auth-monitoring-scripts)

## Примечание по Anthropic

Бэкенд Anthropic `claude-cli` снова поддерживается.

- Сотрудники Anthropic сообщили нам, что этот путь интеграции с OpenClaw снова разрешён.
- Поэтому OpenClaw рассматривает повторное использование Claude CLI и использование `claude -p` как допустимые для запусков с поддержкой Anthropic, если Anthropic не опубликует новую политику.
- API-ключи Anthropic остаются наиболее предсказуемым выбором для долго работающих хостов шлюза и явного контроля биллинга на стороне сервера.

## Проверка статуса аутентификации модели

```bash
openclaw models status
openclaw doctor
```

## Поведение при ротации API-ключей (шлюз)

Некоторые поставщики поддерживают повторную отправку запроса с альтернативными ключами, когда вызов API достигает ограничения частоты поставщика.

- Порядок приоритетов:
  - `OPENCLAW_LIVE_<PROVIDER>_KEY` (единое переопределение)
  - `<PROVIDER>_API_KEYS`
  - `<PROVIDER>_API_KEY`
  - `<PROVIDER>_API_KEY_*`
- Для поставщиков Google также используется `GOOGLE_API_KEY` в качестве дополнительного запасного варианта.
- Один и тот же список ключей дедуплицируется перед использованием.
- OpenClaw повторяет попытку с следующим ключом только для ошибок, связанных с ограничением частоты (например, `429`, `rate_limit`, `quota`, `resource exhausted`, `Too many concurrent requests`, `ThrottlingException`, `concurrency limit reached` или `workers_ai ... quota limit exceeded`).
- Ошибки, не связанные с ограничением частоты, не повторяются с альтернативными ключами.
- Если все ключи не сработали, возвращается окончательная ошибка из последней попытки.

## Управление тем, какие учётные данные используются

### Для сеанса (команда чата)

Используйте `/model <alias-or-id>@<profileId>`, чтобы закрепить конкретные учётные данные поставщика для текущего сеанса (примеры идентификаторов профилей: `anthropic:default`, `anthropic:work`).

Используйте `/model` (или `/model list`) для компактного выбора; используйте `/model status` для полного просмотра (кандидаты + следующий профиль аутентификации, а также детали конечной точки поставщика, если они настроены).

### Для агента (переопределение через CLI)

Установите явное переопределение порядка профилей аутентификации для агента (сохраняется в `auth-state.json` этого агента):

```bash
openclaw models auth order get --provider anthropic
openclaw models auth order set --provider anthropic anthropic:default
openclaw models auth order clear --provider anthropic
```

Используйте `--agent <id>`, чтобы указать конкретного агента; опустите этот параметр, чтобы использовать агента по умолчанию.
При отладке проблем с порядком `openclaw models status --probe` отображает исключённые сохранённые профили как `excluded_by_auth_order` вместо того, чтобы молча их пропускать.
При отладке проблем с охлаждением помните, что охлаждение при ограничении частоты может быть привязано к идентификатору одной модели, а не ко всему профилю поставщика.

## Устранение неполадок

### "Учётные данные не найдены"

Если профиль Anthropic отсутствует, настройте API-ключ Anthropic на **хосте шлюза** или настройте путь токена настройки Anthropic, затем проверьте снова:

```bash
openclaw models status
```

### Токены истекают/истекли

Запустите `openclaw models status`, чтобы подтвердить, какой профиль истекает. Если профиль токена Anthropic отсутствует или истёк, обновите его через токен настройки или перейдите на API-ключ Anthropic.
