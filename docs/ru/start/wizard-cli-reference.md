# Справочник по настройке CLI

Эта страница — полное руководство по `openclaw onboard`.
Для краткого руководства см. [Подключение (CLI)](/start/wizard).

## Что делает мастер настройки

Локальный режим (по умолчанию) проводит вас через следующие этапы:

- Настройка модели и авторизации (OAuth для подписки OpenAI Code, CLI или API-ключ Anthropic Claude, а также варианты MiniMax, GLM, Ollama, Moonshot, StepFun и AI Gateway)
- Выбор расположения рабочей области и начальных файлов
- Настройка шлюза (порт, привязка, авторизация, tailscale)
- Каналы и провайдеры (Telegram, WhatsApp, Discord, Google Chat, Mattermost, Signal, BlueBubbles и другие встроенные плагины каналов)
- Установка демона (LaunchAgent, пользовательский модуль systemd или нативная задача планировщика Windows с резервным вариантом папки автозагрузки)
- Проверка работоспособности
- Настройка навыков

Удаленный режим настраивает эту машину для подключения к шлюзу на другом устройстве.
Он не устанавливает и не изменяет ничего на удалённом хосте.

## Детали локального процесса

<Steps>
  <Step title="Обнаружение существующей конфигурации">
    - Если файл `~/.openclaw/openclaw.json` существует, выберите "Сохранить", "Изменить" или "Сбросить".
    - Повторный запуск мастера настройки не удаляет данные, если вы явно не выбрали "Сбросить" (или не передали флаг `--reset`).
    - По умолчанию флаг CLI `--reset` сбрасывает `config+creds+sessions`; используйте `--reset-scope full`, чтобы также удалить рабочую область.
    - Если конфигурация недействительна или содержит устаревшие ключи, мастер настройки останавливается и просит запустить `openclaw doctor` перед продолжением.
    - При сбросе используется `trash`, доступны следующие области:
      - Только конфигурация
      - Конфигурация + учётные данные + сессии
      - Полный сброс (также удаляет рабочую область)
  </Step>
  <Step title="Модель и авторизация">
    - Полная матрица опций представлена в разделе [Опции авторизации и моделей](#auth-and-model-options).
  </Step>
  <Step title="Рабочая область">
    - По умолчанию — `~/.openclaw/workspace` (можно изменить).
    - Заполняет файлы рабочей области, необходимые для начальной настройки.
    - Структура рабочей области: [Рабочая область агента](/concepts/agent-workspace).
  </Step>
  <Step title="Шлюз">
    - Запрашивает порт, привязку, режим авторизации и параметры tailscale.
    - Рекомендуется оставить включённой авторизацию по токену даже для loopback, чтобы локальные клиенты WS проходили аутентификацию.
    - В режиме токенов интерактивная настройка предлагает:
      - **Сгенерировать/сохранить токен в виде обычного текста** (по умолчанию)
      - **Использовать SecretRef** (по желанию)
    - В режиме паролей интерактивная настройка также поддерживает хранение в виде обычного текста или SecretRef.
    - Неинтерактивный путь к SecretRef токена: `--gateway-token-ref-env <ENV_VAR>`.
      - Требуется непустая переменная окружения в среде процесса подключения.
      - Нельзя комбинировать с `--gateway-token`.
    - Отключайте авторизацию только в том случае, если полностью доверяете всем локальным процессам.
    - Для привязок, не являющихся loopback, авторизация всё равно требуется.
  </Step>
  <Step title="Каналы">
    - [WhatsApp](/channels/whatsapp): необязательный вход через QR-код
    - [Telegram](/channels/telegram): токен бота
    - [Discord](/channels/discord): токен бота
    - [Google Chat](/channels/googlechat): JSON учётной записи службы + аудитория вебхука
    - [Mattermost](/channels/mattermost): токен бота + базовый URL
    - [Signal](/channels/signal): необязательная установка `signal-cli` + настройка учётной записи
    - [BlueBubbles](/channels/bluebubbles): рекомендуется для iMessage; URL сервера + пароль + вебхук
    - [iMessage](/channels/imessage): устаревший путь CLI `imsg` + доступ к БД
    - Безопасность DM: по умолчанию используется сопряжение. Первое DM отправляет код; подтвердите через `openclaw pairing approve <channel> <code>` или используйте списки разрешений.
  </Step>
  <Step title="Установка демона">
    - macOS: LaunchAgent
      - Требуется сеанс вошедшего в систему пользователя; для безголового режима используйте пользовательский LaunchDaemon (не поставляется).
    - Linux и Windows через WSL2: пользовательский модуль systemd
      - Мастер настройки пытается выполнить `loginctl enable-linger <user>`, чтобы шлюз оставался активным после выхода из системы.
      - Может запросить sudo (записывает в `/var/lib/systemd/linger`); сначала пытается без sudo.
    - Нативная Windows: сначала задача планировщика
      - Если создание задачи отклонено, OpenClaw использует элемент входа в папку автозагрузки для каждого пользователя и немедленно запускает шлюз.
      - Задачи планировщика остаются предпочтительным вариантом, поскольку они обеспечивают лучший статус супервизора.
    - Выбор среды выполнения: Node (рекомендуется; требуется для WhatsApp и Telegram). Bun не рекомендуется.
  </Step>
  <Step title="Проверка работоспособности">
    - Запускает шлюз (при необходимости) и выполняет `openclaw health`.
    - `openclaw status --deep` добавляет проверку работоспособности шлюза в вывод статуса, включая проверки каналов, если они поддерживаются.
  </Step>
  <Step title="Навыки">
    - Считывает доступные навыки и проверяет требования.
    - Позволяет выбрать менеджер узлов: npm, pnpm или bun.
    - Устанавливает необязательные зависимости (некоторые используют Homebrew на macOS).
  </Step>
  <Step title="Завершение">
    - Краткое изложение и следующие шаги, включая варианты приложений для iOS, Android и macOS.
  </Step>
</Steps>

<Note>
Если графический интерфейс не обнаружен, мастер настройки выводит инструкции по перенаправлению портов SSH для Control UI вместо открытия браузера.
Если ресурсы Control UI отсутствуют, мастер настройки пытается их собрать; резервный вариант — `pnpm ui:build` (автоматически устанавливает зависимости UI).
</Note>

## Детали удалённого режима

Удалённый режим настраивает эту машину для подключения к шлюзу на другом устройстве.

<Info>
Удалённый режим не устанавливает и не изменяет ничего на удалённом хосте.
</Info>

Что вы настраиваете:

- URL удалённого шлюза (`ws://...`)
- Токен, если для удалённого шлюза требуется авторизация (рекомендуется)

<Note>
- Если шлюз доступен только через loopback, используйте туннелирование SSH или tailnet.
- Подсказки для обнаружения:
  - macOS: Bonjour (`dns-sd`)
  - Linux: Avahi (`avahi-browse`)
</Note>

## Опции авторизации и моделей

<AccordionGroup>
  <Accordion title="API-ключ Anthropic">
    Использует `ANTHROPIC_API_KEY`, если он есть, или запрашивает ключ, затем сохраняет его для использования демоном.
  </Accordion>
  <Accordion title="Подписка OpenAI Code (повторное использование Codex CLI)">
    Если файл `~/.codex/auth.json` существует, мастер настройки может его использовать.
    Учётные данные Codex CLI, повторно используемые мастером настройки, остаются под управлением Codex CLI; по истечении срока действия OpenClaw сначала повторно считывает этот источник и, если провайдер может обновить его, записывает обновлённые учётные данные обратно в хранилище Codex, не беря на себя управление.
  </Accordion>
  <Accordion title="Подписка OpenAI Code (OAuth)">
    Процесс через браузер; вставьте `code#state`.

    Устанавливает `agents.defaults.model` в `openai-codex/gpt-5.4`, если модель не задана или имеет значение `openai/*`.

  </Accordion>
  <Accordion title="API-ключ OpenAI">
    Использует `OPENAI_API_KEY`, если он есть, или запрашивает ключ, затем сохраняет учётные данные в профилях авторизации.

    Устанавливает `agents.defaults.model` в `openai/gpt-5.4`, если модель не задана, имеет значение `openai/*` или `openai-codex/*`.

  </Accordion>
  <Accordion title="API-ключ xAI (Grok)">
    Запрашивает `XAI_API_KEY` и настраивает xAI в качестве провайдера моделей.
  </Accordion>
  <Accordion title="OpenCode">
    Запрашивает `OPENCODE_API_KEY` (или `OPENCODE_ZEN_API_KEY`) и позволяет выбрать каталог Zen или Go.
    URL настройки: [opencode.ai/auth](https://op
