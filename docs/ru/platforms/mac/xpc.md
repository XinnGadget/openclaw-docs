# Архитектура IPC в macOS для OpenClaw

**Текущая модель:** локальный Unix-сокет соединяет **службу хоста узла** с **приложением macOS** для утверждения выполнения команд + `system.run`. Существует отладочная CLI-утилита `openclaw-mac` для проверки обнаружения и подключения; действия агента по‑прежнему проходят через Gateway WebSocket и `node.invoke`. Для автоматизации UI используется PeekabooBridge.

## Цели

- Единый экземпляр GUI‑приложения, который выполняет всю работу, связанную с TCC (уведомления, запись экрана, микрофон, речь, AppleScript).
- Минимальный набор инструментов для автоматизации: Gateway + команды узла, а также PeekabooBridge для автоматизации UI.
- Предсказуемые разрешения: всегда один и тот же подписанный идентификатор пакета, запускаемый через launchd, чтобы разрешения TCC сохранялись.

## Как это работает

### Gateway + транспорт узла

- Приложение запускает Gateway (в локальном режиме) и подключается к нему как узел.
- Действия агента выполняются через `node.invoke` (например, `system.run`, `system.notify`, `canvas.*`).

### Служба узла + IPC приложения

- Безголовая служба хоста узла подключается к Gateway WebSocket.
- Запросы `system.run` пересылаются в приложение macOS через локальный Unix‑сокет.
- Приложение выполняет команду в контексте UI, при необходимости выводит запрос и возвращает результат.

Диаграмма (SCI):

```
Agent -> Gateway -> Node Service (WS)
                      |  IPC (UDS + token + HMAC + TTL)
                      v
                  Mac App (UI + TCC + system.run)
```

### PeekabooBridge (автоматизация UI)

- Для автоматизации UI используется отдельный Unix‑сокет с именем `bridge.sock` и протокол JSON PeekabooBridge.
- Порядок предпочтений хоста (на стороне клиента): Peekaboo.app → Claude.app → OpenClaw.app → локальное выполнение.
- Безопасность: хосты моста должны иметь разрешённый TeamID; для отладки предусмотрен обход с одинаковым UID, который контролируется переменной `PEEKABOO_ALLOW_UNSIGNED_SOCKET_CLIENTS=1` (соглашение Peekaboo).
- Подробнее см.: [Использование PeekabooBridge](/platforms/mac/peekaboo).

## Рабочие процессы

- Перезапуск/пересборка: `SIGN_IDENTITY="Apple Development: <Developer Name> (<TEAMID>)" scripts/restart-mac.sh`
  - Завершает работающие экземпляры.
  - Сборка и упаковка на Swift.
  - Запись/начальная настройка/запуск LaunchAgent.
- Единый экземпляр: приложение завершается на раннем этапе, если уже запущен другой экземпляр с тем же идентификатором пакета.

## Примечания по усилению безопасности

- Предпочитать требование совпадения TeamID для всех привилегированных компонентов.
- PeekabooBridge: `PEEKABOO_ALLOW_UNSIGNED_SOCKET_CLIENTS=1` (только для отладки) может разрешать вызывающим сторонам с тем же UID для локальной разработки.
- Вся коммуникация остаётся локальной; сетевые сокеты не раскрываются.
- Запросы TCC исходят только из пакета GUI‑приложения; необходимо сохранять стабильный подписанный идентификатор пакета при пересборке.
- Усиление IPC: режим сокета `0600`, токен, проверка peer‑UID, вызов/ответ с HMAC, короткий TTL.
