# Memory Wiki

`memory-wiki` — это встроенный плагин, который преобразует устойчивую память в хранилище скомпилированных знаний.

Он **не** заменяет плагин активной памяти. Плагин активной памяти по-прежнему отвечает за извлечение, продвижение, индексирование и "сны". `memory-wiki` работает параллельно с ним и компилирует устойчивые знания в удобную для навигации вики с детерминированными страницами, структурированными утверждениями, указанием источников, панелями мониторинга и машиночитаемыми дайджестами.

Используйте его, если хотите, чтобы память работала скорее как поддерживаемый слой знаний, а не как набор файлов Markdown.

## Что он добавляет

- Отдельное хранилище вики с детерминированной структурой страниц
- Структурированные метаданные утверждений и доказательств, а не только текст
- Указание источников, уровня уверенности, противоречий и открытых вопросов на уровне страниц
- Скомпилированные дайджесты для агентов и сред выполнения
- Инструменты поиска/получения/применения/проверки в стиле вики
- Опциональный режим моста, который импортирует публичные артефакты из плагина активной памяти
- Опциональный режим рендеринга, совместимый с Obsidian, и интеграция с CLI

## Как он сочетается с памятью

Представьте разделение следующим образом:

| Уровень | Отвечает за |
| --- | --- |
| Плагин активной памяти (`memory-core`, QMD, Honcho и т. д.) | Извлечение, семантический поиск, продвижение, "сны", среда выполнения памяти |
| `memory-wiki` | Скомпилированные страницы вики, синтезы с указанием источников, панели мониторинга, поиск/получение/применение в стиле вики |

Если плагин активной памяти предоставляет общие артефакты извлечения, OpenClaw может искать в обоих слоях за один проход с помощью `memory_search corpus=all`.

Когда вам нужен специфичный для вики ранжирование, указание источников или прямой доступ к страницам, используйте инструменты, встроенные в вики.

## Режимы хранилища

`memory-wiki` поддерживает три режима хранилища:

### `isolated`

Собственное хранилище, собственные источники, отсутствие зависимости от `memory-core`.

Используйте этот режим, если хотите, чтобы вики была самостоятельным хранилищем тщательно отобранных знаний.

### `bridge`

Читает публичные артефакты памяти и события памяти из плагина активной памяти через общедоступные интерфейсы SDK плагина.

Используйте этот режим, если хотите, чтобы вики компилировала и организовывала экспортированные артефакты плагина памяти, не затрагивая его внутренние приватные компоненты.

Режим моста может индексировать:
- экспортированные артефакты памяти
- отчёты о "снах"
- ежедневные заметки
- корневые файлы памяти
- журналы событий памяти

### `unsafe-local`

Явный механизм выхода на том же компьютере для локальных приватных путей.

Этот режим намеренно экспериментальный и не переносимый. Используйте его только в том случае, если вы понимаете границы доверия и вам конкретно нужен доступ к локальной файловой системе, который не может обеспечить режим моста.

## Структура хранилища

Плагин инициализирует хранилище следующим образом:

```text
<vault>/
  AGENTS.md
  WIKI.md
  index.md
  inbox.md
  entities/
  concepts/
  syntheses/
  sources/
  reports/
  _attachments/
  _views/
  .openclaw-wiki/
```

Управляемый контент остаётся внутри сгенерированных блоков. Блоки заметок пользователя сохраняются.

Основные группы страниц:
- `sources/` — для импортированного исходного материала и страниц, поддерживаемых режимом моста
- `entities/` — для устойчивых сущностей: людей, систем, проектов и объектов
- `concepts/` — для идей, абстракций, шаблонов и политик
- `syntheses/` — для скомпилированных сводок и поддерживаемых сводных данных
- `reports/` — для сгенерированных панелей мониторинга

## Структурированные утверждения и доказательства

Страницы могут содержать структурированные метаданные `claims` (утверждений), а не только произвольный текст.

Каждое утверждение может включать:
- `id`
- `text`
- `status`
- `confidence`
- `evidence[]`
- `updatedAt`

Записи доказательств могут включать:
- `sourceId`
- `path`
- `lines`
- `weight`
- `note`
- `updatedAt`

Именно это позволяет вики работать скорее как слой убеждений, а не как пассивный набор заметок. Утверждения можно отслеживать, оценивать, оспаривать и сопоставлять с источниками.

## Конвейер компиляции

Этап компиляции считывает страницы вики, нормализует сводки и выдаёт стабильные машиночитаемые артефакты в следующих директориях:
- `.openclaw-wiki/cache/agent-digest.json`
- `.openclaw-wiki/cache/claims.jsonl`

Эти дайджесты существуют для того, чтобы агентам и коду среды выполнения не приходилось извлекать данные из страниц Markdown.

Скомпилированный вывод также используется для:
- первичного индексирования вики для потоков поиска/получения
- поиска страниц по идентификатору утверждения
- компактных дополнений к промтам
- генерации отчётов и панелей мониторинга

## Панели мониторинга и отчёты о состоянии

Когда включена опция `render.createDashboards`, на этапе компиляции создаются панели мониторинга в директории `reports/`.

Встроенные отчёты включают:
- `reports/open-questions.md`
- `reports/contradictions.md`
- `reports/low-confidence.md`
- `reports/claim-health.md`
- `reports/stale-pages.md`

Эти отчёты отслеживают такие вещи, как:
- кластеры заметок с противоречиями
- кластеры конкурирующих утверждений
- утверждения без структурированных доказательств
- страницы и утверждения с низким уровнем уверенности
- устаревшие данные или данные с неизвестной актуальностью
- страницы с нерешёнными вопросами

## Поиск и извлечение

`memory-wiki` поддерживает два механизма поиска:
- `shared`: использование общего потока поиска памяти, если он доступен
- `local`: локальный поиск в вики

Также поддерживаются три корпуса:
- `wiki`
- `memory`
- `all`

Важное поведение:
- `wiki_search` и `wiki_get` по возможности используют скомпилированные дайджесты в качестве первого этапа
- идентификаторы утверждений могут быть сопоставлены со страницей, которой они принадлежат
- оспариваемые/устаревшие/актуальные утверждения влияют на ранжирование
- метки источников могут сохраняться в результатах

Практическое правило:
- используйте `memory_search corpus=all` для одного широкого прохода извлечения
- используйте `wiki_search` + `wiki_get`, если вам важно ранжирование, специфичное для вики, указание источников или структура убеждений на уровне страниц

## Инструменты для агентов

Плагин регистрирует следующие инструменты:
- `wiki_status`
- `wiki_search`
- `wiki_get`
- `wiki_apply`
- `wiki_lint`

Их функции:
- `wiki_status`: текущий режим хранилища, состояние, доступность Obsidian CLI
- `wiki_search`: поиск страниц вики и, при настройке, общих корпусов памяти
- `wiki_get`: чтение страницы вики по идентификатору/пути или возврат к общему корпусу памяти
- `wiki_apply`: узконаправленные изменения синтезов/метаданных без редактирования страниц в свободном формате
- `wiki_lint`: структурные проверки, пробелы в указании источников, противоречия, открытые вопросы

Плагин также регистрирует необязательное дополнение к корпусу памяти, поэтому общие `memory_search` и `memory_get` могут обращаться к вики, если плагин активной памяти поддерживает выбор корпуса.

## Поведение промтов и контекста

Когда включена опция `context.includeCompiledDigestPrompt`, разделы промтов памяти дополняют компактным скомпилированным снимком из `agent-digest.json`.

Этот снимок намеренно небольшой и содержит только важную информацию:
- только топовые страницы
- только топовые утверждения
- количество противоречий
- количество вопросов
- квалификаторы уровня уверенности и актуальности

Эта функция включена по запросу, поскольку она изменяет форму промта и в основном полезна для механизмов контекста или устаревшей сборки промтов, которые явно используют дополнения к памяти.
