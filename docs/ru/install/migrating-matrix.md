# Миграция Matrix

На этой странице описано обновление с предыдущего публичного плагина `matrix` до текущей реализации.

Для большинства пользователей обновление выполняется на месте:

- плагин остаётся `@openclaw/matrix`;
- канал остаётся `matrix`;
- ваша конфигурация остаётся в `channels.matrix`;
- кэшированные учётные данные остаются в `~/.openclaw/credentials/matrix/`;
- состояние во время выполнения остаётся в `~/.openclaw/matrix/`.

Вам не нужно переименовывать ключи конфигурации или переустанавливать плагин под новым именем.

## Что миграция выполняет автоматически

Когда шлюз запускается и когда вы выполняете команду [`openclaw doctor --fix`](/gateway/doctor), OpenClaw пытается автоматически восстановить старое состояние Matrix.
Перед тем как какие-либо действия по миграции Matrix изменят состояние на диске, OpenClaw создаёт или повторно использует целевой снимок для восстановления.

При использовании `openclaw update` точный триггер зависит от способа установки OpenClaw:

- при установке из исходного кода во время процесса обновления выполняется `openclaw doctor --fix`, после чего по умолчанию происходит перезапуск шлюза;
- при установке через менеджер пакетов обновляется пакет, выполняется неинтерактивный проход `doctor`, после чего полагается на перезапуск шлюза по умолчанию, чтобы завершить миграцию Matrix при запуске;
- если вы используете `openclaw update --no-restart`, миграция Matrix, поддерживаемая запуском, откладывается до тех пор, пока вы позже не выполните `openclaw doctor --fix` и не перезапустите шлюз.

Автоматическая миграция включает:

- создание или повторное использование снимка перед миграцией в `~/Backups/openclaw-migrations/`;
- повторное использование ваших кэшированных учётных данных Matrix;
- сохранение того же выбора учётной записи и конфигурации `channels.matrix`;
- перемещение самого старого плоского хранилища синхронизации Matrix в текущее расположение, привязанное к учётной записи;
- перемещение самого старого плоского хранилища криптографии Matrix в текущее расположение, привязанное к учётной записи, если целевая учётная запись может быть безопасно определена;
- извлечение ранее сохранённого ключа расшифровки резервной копии ключей комнат Matrix из старого хранилища криптографии Rust, если этот ключ существует локально;
- повторное использование наиболее полного существующего корневого хранилища хэшей токенов для той же учётной записи Matrix, сервера и пользователя, когда позже изменяется токен доступа;
- сканирование соседних корневых хранилищ хэшей токенов на наличие ожидающих метаданных восстановления зашифрованного состояния, когда токен доступа Matrix изменился, но идентификатор учётной записи/устройства остался прежним;
- восстановление резервных копий ключей комнат в новое хранилище криптографии при следующем запуске Matrix.

Детали снимка:

- OpenClaw записывает маркерный файл в `~/.openclaw/matrix/migration-snapshot.json` после успешного создания снимка, чтобы последующие запуски и операции восстановления могли повторно использовать тот же архив.
- Эти автоматические снимки миграции Matrix резервируют только конфигурацию и состояние (`includeWorkspace: false`).
- Если в Matrix есть только предупреждения о состоянии миграции, например, потому что `userId` или `accessToken` всё ещё отсутствуют, OpenClaw пока не создаёт снимок, поскольку никакие изменения в Matrix не могут быть выполнены.
- Если этап создания снимка завершается ошибкой, OpenClaw пропускает миграцию Matrix для этого запуска вместо того, чтобы изменять состояние без точки восстановления.

О обновлениях для нескольких учётных записей:

- самое старое плоское хранилище Matrix (`~/.openclaw/matrix/bot-storage.json` и `~/.openclaw/matrix/crypto/`) было создано в рамках однохранилищной схемы, поэтому OpenClaw может перенести его только в одну разрешённую целевую учётную запись Matrix;
- уже существующие хранилища Matrix, привязанные к учётной записи, обнаруживаются и подготавливаются для каждой настроенной учётной записи Matrix.

## Что миграция не может выполнить автоматически

Предыдущий публичный плагин Matrix **не** создавал автоматически резервные копии ключей комнат Matrix. Он сохранял локальное криптографическое состояние и запрашивал проверку устройства, но не гарантировал, что ваши ключи комнат были сохранены на сервере.

Это означает, что некоторые зашифрованные установки могут быть перенесены лишь частично.

OpenClaw не может автоматически восстановить:

- локальные ключи комнат, которые никогда не были сохранены в резервную копию;
- зашифрованное состояние, когда целевая учётная запись Matrix ещё не может быть определена, поскольку `homeserver`, `userId` или `accessToken` всё ещё недоступны;
- автоматическую миграцию одного общего плоского хранилища Matrix, когда настроено несколько учётных записей Matrix, но `channels.matrix.defaultAccount` не задано;
- установки плагина по пользовательскому пути, которые привязаны к пути репозитория вместо стандартного пакета Matrix;
- отсутствующий ключ восстановления, когда в старом хранилище были сохранены ключи, но ключ расшифровки не был сохранён локально.

Текущий объём предупреждений:

- установки плагина Matrix по пользовательскому пути отображаются как при запуске шлюза, так и при выполнении `openclaw doctor`.

Если в вашей старой установке была зашифрованная история, которая существовала только локально и никогда не сохранялась в резервную копию, некоторые старые зашифрованные сообщения могут остаться нечитаемыми после обновления.

## Рекомендуемый порядок обновления

1. Обновите OpenClaw и плагин Matrix обычным образом.
   Предпочитайте простой `openclaw update` без `--no-restart`, чтобы запуск мог немедленно завершить миграцию Matrix.
2. Выполните:

   ```bash
   openclaw doctor --fix
   ```

   Если для Matrix есть действия по миграции, `doctor` сначала создаст или повторно использует снимок перед миграцией и выведет путь к архиву.

3. Запустите или перезапустите шлюз.
4. Проверьте текущее состояние проверки и резервного копирования:

   ```bash
   openclaw matrix verify status
   openclaw matrix verify backup status
   ```

5. Если OpenClaw сообщает, что требуется ключ восстановления, выполните:

   ```bash
   openclaw matrix verify backup restore --recovery-key "<ваш-ключ-восстановления>"
   ```

6. Если это устройство всё ещё не проверено, выполните:

   ```bash
   openclaw matrix verify device "<ваш-ключ-восстановления>"
   ```

7. Если вы намеренно отказываетесь от невосстановимой старой истории и хотите создать новую базовую линию резервного копирования для будущих сообщений, выполните:

   ```bash
   openclaw matrix verify backup reset --yes
   ```

8. Если на сервере ещё нет резервной копии ключей, создайте её для будущих восстановлений:

   ```bash
   openclaw matrix verify bootstrap
   ```

## Как работает зашифрованная миграция

Зашифрованная миграция — это двухэтапный процесс:

1. При запуске или выполнении `openclaw doctor --fix` создаётся или повторно используется снимок перед миграцией, если зашифрованная миграция возможна.
2. При запуске или выполнении `openclaw doctor --fix` проверяется старое хранилище криптографии Matrix через активную установку плагина Matrix.
3. Если ключ расшифровки резервной копии найден, OpenClaw записывает его в новый поток ключей восстановления и помечает восстановление ключей комнат как ожидающее.
4. При следующем запуске Matrix OpenClaw автоматически восстанавливает сохранённые ключи комнат в новое хранилище криптографии.

Если старое хранилище сообщает о ключах комнат, которые никогда не были сохранены в резервную копию, OpenClaw выдаёт предупреждение вместо того, чтобы имитировать успешное восстановление.

## Распространённые сообщения и их значение

### Сообщения об обновлении и обнаружении

`Matrix plugin upgraded in place.`

- Значение: старое состояние Matrix на диске было обнаружено и перенесено в текущий формат.
- Что делать: ничего, если в том же выводе нет предупреждений.

`Matrix migration snapshot created before applying Matrix upgrades.`

- Значение: OpenClaw создал архив восстановления перед изменением состояния Matrix.
- Что делать: сохраните выведенный путь к архиву до тех пор, пока не подтвердите успешность миграции.

`Matrix migration snapshot reused before applying Matrix upgrades.`

- Значение: OpenClaw нашёл существующий маркер снимка миграции Matrix и повторно использовал этот архив вместо создания дубликата резервной копии.
- Что делать: сохраните выведенный путь к архиву до тех пор, пока не подтвердите успешность миграции.

`Legacy Matrix state detected at ... but channels.matrix is not configured yet.`

- Значение: старое состояние Matrix существует, но OpenClaw не может сопоставить его с текущей учётной записью Matrix, поскольку Matrix не настроен.
- Что делать: настройте `channels.matrix`, затем повторно выполните `openclaw doctor --fix` или перезапустите шлюз.

`Legacy Matrix state detected at ... but the new account-scoped target could not be resolved yet (need homeserver, userId, and access token for channels.matrix...).`

- Значение: OpenClaw обнаружил старое состояние, но пока не может определить точную текущую учётную запись/устройство.
- Что делать: запустите шлюз один раз с рабочим входом в Matrix или повторно выполните `openclaw doctor --fix` после того, как появятся кэшированные учётные данные.

`Legacy Matrix state detected at ... but multiple Matrix accounts are configured and channels.matrix.defaultAccount is not set.`

- Значение: OpenClaw обнаружил одно общее плоское хранилище
