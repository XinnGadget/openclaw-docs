# Откат между моделями

OpenClaw обрабатывает сбои в два этапа:

1. **Ротация профиля аутентификации** в рамках текущего провайдера.
2. **Откат к другой модели** — переход к следующей модели в `agents.defaults.model.fallbacks`.

В этом документе описаны правила выполнения и данные, на которых они основаны.

## Поток выполнения

При обычном текстовом запросе OpenClaw оценивает кандидатов в следующем порядке:

1. Текущая выбранная модель сеанса.
2. Настроенные `agents.defaults.model.fallbacks` в указанном порядке.
3. Настроенная основная модель в конце, если запуск начался с переопределения.

Для каждого кандидата OpenClaw пытается выполнить откат профиля аутентификации, прежде чем переходить к следующему кандидату-модели.

Общая последовательность действий:

1. Определить активную модель сеанса и предпочтительный профиль аутентификации.
2. Сформировать цепочку кандидатов-моделей.
3. Попробовать текущего провайдера с учётом правил ротации профилей аутентификации и периодов охлаждения.
4. Если у провайдера исчерпаны возможности из-за ошибки, требующей отказоустойчивости, перейти к следующему кандидату-модели.
5. Сохранить выбранное переопределение для отката до начала повторной попытки, чтобы другие читатели сеанса видели тот же провайдер/модель, которые собирается использовать исполнитель.
6. Если кандидат для отката не сработал, откатить только поля переопределения сеанса, принадлежащие откату, если они всё ещё соответствуют этому неудачному кандидату.
7. Если все кандидаты не сработали, выдать ошибку `FallbackSummaryError` с деталями по каждой попытке и самым ранним сроком окончания периода охлаждения, если он известен.

Это намеренно более узконаправленный подход, чем "сохранить и восстановить весь сеанс". Исполнитель ответа сохраняет только те поля выбора модели, которые относятся к откату:

- `providerOverride`
- `modelOverride`
- `authProfileOverride`
- `authProfileOverrideSource`
- `authProfileOverrideCompactionCount`

Это предотвращает перезапись более новых несвязанных изменений сеанса при неудачной повторной попытке отката, таких как ручные изменения через `/model` или обновления ротации сеанса, произошедшие во время выполнения попытки.

## Хранение данных аутентификации (ключи + OAuth)

OpenClaw использует **профили аутентификации** как для API-ключей, так и для токенов OAuth.

- Секреты хранятся в `~/.openclaw/agents/<agentId>/agent/auth-profiles.json` (устаревший вариант: `~/.openclaw/agent/auth-profiles.json`).
- Состояние маршрутизации аутентификации во время выполнения хранится в `~/.openclaw/agents/<agentId>/agent/auth-state.json`.
- Конфигурация `auth.profiles` / `auth.order` содержит **только метаданные и настройки маршрутизации** (без секретов).
- Устаревший файл OAuth только для импорта: `~/.openclaw/credentials/oauth.json` (импортируется в `auth-profiles.json` при первом использовании).

Подробнее: [/concepts/oauth](/concepts/oauth)

Типы учётных данных:

- `type: "api_key"` → `{ provider, key }`
- `type: "oauth"` → `{ provider, access, refresh, expires, email? }` (+ `projectId`/`enterpriseUrl` для некоторых провайдеров)

## Идентификаторы профилей

При входе через OAuth создаются отдельные профили, что позволяет сосуществовать нескольким учётным записям.

- По умолчанию: `provider:default`, если адрес электронной почты недоступен.
- OAuth с адресом электронной почты: `provider:<email>` (например, `google-antigravity:user@gmail.com`).

Профили хранятся в `~/.openclaw/agents/<agentId>/agent/auth-profiles.json` в разделе `profiles`.

## Порядок ротации

Если у провайдера несколько профилей, OpenClaw выбирает порядок следующим образом:

1. **Явная конфигурация**: `auth.order[provider]` (если задана).
2. **Настроенные профили**: `auth.profiles`, отфильтрованные по провайдеру.
3. **Сохранённые профили**: записи в `auth-profiles.json` для провайдера.

Если явный порядок не задан, OpenClaw использует циклический порядок (round-robin):

- **Основной ключ**: тип профиля (**OAuth перед API-ключами**).
- **Вторичный ключ**: `usageStats.lastUsed` (сначала самые старые в рамках каждого типа).
- **Профили в периоде охлаждения/отключённые** перемещаются в конец, упорядоченные по самому раннему сроку окончания периода.

### Привязка сеанса (с учётом кэширования)

OpenClaw **привязывает выбранный профиль аутентификации к сеансу**, чтобы поддерживать актуальность кэша провайдера. Ротация не выполняется при каждом запросе. Привязанный профиль используется повторно до тех пор, пока:

- сеанс не будет сброшен (`/new` / `/reset`);
- не завершится уплотнение (увеличивается счётчик уплотнения);
- профиль не окажется в периоде охлаждения/отключён.

Ручной выбор через `/model …@<profileId>` устанавливает **пользовательское переопределение** для этого сеанса и не подвергается автоматической ротации до начала нового сеанса.

Автоматически привязанные профили (выбранные маршрутизатором сеанса) рассматриваются как **предпочтение**: они используются в первую очередь, но OpenClaw может перейти к другому профилю при ограничениях по скорости/тайм-аутах. Пользовательски привязанные профили остаются закреплёнными за этим профилем; если он не сработал и настроены откаты между моделями, OpenClaw переходит к следующей модели вместо переключения профилей.

### Почему OAuth может "казаться потерянным"

Если у вас есть и профиль OAuth, и профиль с API-ключом для одного провайдера, циклический порядок может переключаться между ними в разных сообщениях, если они не привязаны. Чтобы принудительно использовать один профиль:

- привяжите его с помощью `auth.order[provider] = ["provider:profileId"]`;
- используйте переопределение для сеанса через `/model …` с переопределением профиля (если это поддерживается вашим пользовательским интерфейсом/чатом).

## Периоды охлаждения

Если профиль не работает из-за ошибок аутентификации/ограничений по скорости (или тайм-аута, который выглядит как ограничение по скорости), OpenClaw помечает его как находящийся в периоде охлаждения и переходит к следующему профилю.

Этот "пул" ограничений по скорости шире, чем просто `429`: он также включает сообщения провайдера, такие как `Too many concurrent requests`, `ThrottlingException`, `concurrency limit reached`, `workers_ai ... quota limit exceeded`, `throttled`, `resource exhausted`, а также периодические ограничения по окну использования, такие как `weekly/monthly limit reached`.

Ошибки формата/некорректного запроса (например, ошибки валидации идентификатора вызова инструмента Cloud Code Assist) считаются требующими отказоустойчивости и используют те же периоды охлаждения.

Ошибки причины остановки, совместимые с OpenAI, такие как `Unhandled stop reason: error`, `stop reason: error` и `reason: error`, классифицируются как сигналы тайм-аута/отказоустойчивости.

Общий текст сервера, специфичный для провайдера, также может попасть в этот "пул" тайм-аутов, если источник соответствует известному временному шаблону. Например, в Anthropic простой текст `An unknown error occurred` и JSON-payload `api_error` с временным текстом сервера, таким как `internal server error`, `unknown error, 520`, `upstream error` или `backend error`, рассматриваются как требующие отказоустойчивости тайм-ауты. Общий текст вышестоящего уровня OpenRouter, такой как простой `Provider returned error`, также рассматривается как тайм-аут только в том случае, если контекст провайдера действительно является OpenRouter. Общий внутренний текст отката, такой как `LLM request failed with an unknown error.`, остаётся консервативным и сам по себе не запускает откат.

Периоды охлаждения, связанные с ограничениями по скорости, также могут быть привязаны к модели:

- OpenClaw записывает `cooldownModel` для сбоев из-за ограничений по скорости, когда известен идентификатор модели, в которой произошёл сбой.
- Модель-"сестра" на том же провайдере всё ещё может быть использована, если период охлаждения привязан к другой модели.
- Окна оплаты/отключения блокируют весь профиль для всех моделей.

Периоды охлаждения используют экспоненциальное увеличение времени ожидания:

- 1 минута;
- 5 минут;
- 25 минут;
- 1 час (максимум).
