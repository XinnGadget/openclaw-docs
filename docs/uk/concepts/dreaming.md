# Dreaming

Dreaming — це фонова система ущільнення пам’яті в `memory-core`.
Вона допомагає OpenClaw переносити сильні короткострокові сигнали в довготривалу пам’ять, водночас
зберігаючи процес зрозумілим і придатним для перевірки.

Dreaming є **опційною** функцією й за замовчуванням вимкнена.

## Що записує Dreaming

Dreaming зберігає два типи виводу:

- **Машинний стан** у `memory/.dreams/` (сховище recall, сигнали фаз, контрольні точки ingest, блокування).
- **Людинозрозумілий вивід** у `DREAMS.md` (або наявному `dreams.md`) і необов’язкові файли звітів фаз у `memory/dreaming/<phase>/YYYY-MM-DD.md`.

Довгострокове просування, як і раніше, записується лише в `MEMORY.md`.

## Модель фаз

Dreaming використовує три взаємодіючі фази:

| Фаза | Призначення | Довготривалий запис |
| ----- | ----------- | ------------------- |
| Light | Сортування та підготовка нещодавнього короткострокового матеріалу | Ні |
| Deep  | Оцінювання та просування довготривалих кандидатів | Так (`MEMORY.md`) |
| REM   | Осмислення тем і повторюваних ідей | Ні |

Ці фази є внутрішніми деталями реалізації, а не окремими користувацькими
«режимами».

### Фаза Light

Фаза Light поглинає нещодавні щоденні сигнали пам’яті та сліди recall, усуває дублікати
й готує рядки-кандидати.

- Читає з короткострокового стану recall, нещодавніх щоденних файлів пам’яті та відредагованих транскриптів сесій, якщо вони доступні.
- Записує керований блок `## Light Sleep`, якщо сховище підтримує вбудований вивід.
- Фіксує сигнали підсилення для подальшого ранжування Deep.
- Ніколи не записує в `MEMORY.md`.

### Фаза Deep

Фаза Deep вирішує, що стане довготривалою пам’яттю.

- Ранжує кандидатів за допомогою зваженого оцінювання та порогових перевірок.
- Вимагає проходження `minScore`, `minRecallCount` і `minUniqueQueries`.
- Перед записом повторно підтягує фрагменти з актуальних щоденних файлів, тож застарілі або видалені фрагменти пропускаються.
- Додає просунуті записи до `MEMORY.md`.
- Записує підсумок `## Deep Sleep` у `DREAMS.md` і за потреби записує `memory/dreaming/deep/YYYY-MM-DD.md`.

### Фаза REM

Фаза REM витягує шаблони й рефлексивні сигнали.

- Формує підсумки тем і рефлексій із нещодавніх короткострокових слідів.
- Записує керований блок `## REM Sleep`, якщо сховище підтримує вбудований вивід.
- Фіксує сигнали підсилення REM, які використовуються під час ранжування Deep.
- Ніколи не записує в `MEMORY.md`.

## Поглинання транскриптів сесій

Dreaming може поглинати відредаговані транскрипти сесій до корпусу Dreaming. Якщо
транскрипти доступні, вони подаються у фазу Light разом зі щоденними
сигналами пам’яті та слідами recall. Особистий і чутливий вміст редагується
перед поглинанням.

## Щоденник сновидінь

Dreaming також веде наративний **Щоденник сновидінь** у `DREAMS.md`.
Після того як для кожної фази накопичується достатньо матеріалу, `memory-core` виконує фоновий
хід субагента в режимі best-effort (із використанням стандартної runtime-моделі) і додає короткий запис до щоденника.

Цей щоденник призначений для читання людиною в інтерфейсі Dreams, а не є джерелом для просування.
Артефакти щоденника/звітів, згенеровані Dreaming, виключаються з короткострокового
просування. До просування в
`MEMORY.md` можуть потрапити лише обґрунтовані фрагменти пам’яті.

Також існує обґрунтований канал історичного backfill для перевірки та відновлення:

- `memory rem-harness --path ... --grounded` показує попередній перегляд обґрунтованого виводу щоденника з історичних нотаток `YYYY-MM-DD.md`.
- `memory rem-backfill --path ...` записує оборотні обґрунтовані записи щоденника в `DREAMS.md`.
- `memory rem-backfill --path ... --stage-short-term` додає обґрунтовані довготривалі кандидати до того самого короткострокового сховища доказів, яке вже використовує звичайна фаза Deep.
- `memory rem-backfill --rollback` і `--rollback-short-term` видаляють ці артефакти backfill зі staging без впливу на звичайні записи щоденника або живий короткостроковий recall.

Control UI надає той самий потік backfill/reset для щоденника, щоб ви могли переглянути
результати в сцені Dreams, перш ніж вирішувати, чи заслуговують обґрунтовані кандидати
на просування. Сцена також показує окрему обґрунтовану смугу, щоб ви могли бачити,
які короткострокові записи у staging надійшли з історичного відтворення, які просунуті
елементи були зумовлені grounded-режимом, і очищати лише обґрунтовані записи у staging
без впливу на звичайний живий короткостроковий стан.

## Сигнали ранжування Deep

Ранжування Deep використовує шість базових зважених сигналів і підсилення фаз:

| Сигнал | Вага | Опис |
| ------ | ---- | ---- |
| Частота | 0.24 | Скільки короткострокових сигналів накопичив запис |
| Релевантність | 0.30 | Середня якість отримання для запису |
| Різноманітність запитів | 0.15 | Окремі контексти запитів/днів, у яких він з’являвся |
| Актуальність | 0.15 | Оцінка свіжості з часовим згасанням |
| Ущільнення | 0.10 | Сила повторення протягом кількох днів |
| Концептуальна насиченість | 0.06 | Щільність тегів концептів із фрагмента/шляху |

Попадання у фази Light і REM додають невелике підсилення з часовим згасанням з
`memory/.dreams/phase-signals.json`.

## Планування

Коли функцію ввімкнено, `memory-core` автоматично керує одним завданням Cron для повного
циклу Dreaming. Кожен цикл запускає фази по порядку: light -> REM -> deep.

Стандартна поведінка розкладу:

| Налаштування | За замовчуванням |
| ------------ | ---------------- |
| `dreaming.frequency` | `0 3 * * *` |

## Швидкий старт

Увімкніть Dreaming:

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true
          }
        }
      }
    }
  }
}
```

Увімкніть Dreaming із власною частотою циклу:

```json
{
  "plugins": {
    "entries": {
      "memory-core": {
        "config": {
          "dreaming": {
            "enabled": true,
            "timezone": "America/Los_Angeles",
            "frequency": "0 */6 * * *"
          }
        }
      }
    }
  }
}
```

```
/dreaming status
/dreaming on
/dreaming off
/dreaming help
```

## Робочий процес CLI

Використовуйте просування через CLI для попереднього перегляду або ручного застосування:

```bash
openclaw memory promote
openclaw memory promote --apply
openclaw memory promote --limit 5
openclaw memory status --deep
```

Ручна команда `memory promote` за замовчуванням використовує пороги фази Deep, якщо їх не перевизначено
прапорцями CLI.

Пояснити, чому конкретний кандидат буде або не буде просунутий:

```bash
openclaw memory promote-explain "router vlan"
openclaw memory promote-explain "router vlan" --json
```

Переглянути відображення REM, кандидатні істини та результат просування Deep без
запису будь-чого:

```bash
openclaw memory rem-harness
openclaw memory rem-harness --json
```

## Ключові значення за замовчуванням

Усі налаштування розташовані в `plugins.entries.memory-core.config.dreaming`.

| Ключ | За замовчуванням |
| ---- | ---------------- |
| `enabled` | `false` |
| `frequency` | `0 3 * * *` |

Політика фаз, пороги та поведінка сховища є внутрішніми деталями
реалізації (не користувацькою конфігурацією).

Повний список ключів див. у [довіднику з конфігурації Memory](/uk/reference/memory-config#dreaming).

## Інтерфейс Dreams

Коли функцію ввімкнено, вкладка Gateway **Dreams** показує:

- поточний стан увімкнення Dreaming
- статус на рівні фаз і наявність керованого циклу
- кількість короткострокових, обґрунтованих, сигнальних і просунутих сьогодні записів
- час наступного запланованого запуску
- окрему обґрунтовану смугу Scene для записів історичного відтворення у staging
- розгортаний читач Щоденника сновидінь, що працює на основі `doctor.memory.dreamDiary`
