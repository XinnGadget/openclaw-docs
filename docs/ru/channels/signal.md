# Signal (signal-cli)

Статус: интеграция с внешним CLI. Шлюз взаимодействует с `signal-cli` через HTTP JSON-RPC + SSE.

## Предварительные требования

- OpenClaw установлен на вашем сервере (ниже приведён процесс для Linux, протестирован на Ubuntu 24).
- `signal-cli` доступен на хосте, где работает шлюз.
- Номер телефона, способный принимать одно SMS для верификации (для пути регистрации через SMS).
- Доступ к браузеру для прохождения капчи Signal (`signalcaptchas.org`) во время регистрации.

## Быстрая настройка (для начинающих)

1. Используйте **отдельный номер Signal** для бота (рекомендуется).
2. Установите `signal-cli` (если вы используете сборку для JVM, потребуется Java).
3. Выберите один из путей настройки:
   - **Путь A (QR-ссылка):** выполните `signal-cli link -n "OpenClaw"` и отсканируйте QR-код в Signal.
   - **Путь B (регистрация через SMS):** зарегистрируйте выделенный номер с помощью капчи и SMS-верификации.
4. Настройте OpenClaw и перезапустите шлюз.
5. Отправьте первое прямое сообщение и подтвердите сопряжение (`openclaw pairing approve signal <CODE>`).

Минимальная конфигурация:

```json5
{
  channels: {
    signal: {
      enabled: true,
      account: "+15551234567",
      cliPath: "signal-cli",
      dmPolicy: "pairing",
      allowFrom: ["+15557654321"],
    },
  },
}
```

Справочник по полям:

| Поле | Описание |
| ----------- | ------------------------------------------------- |
| `account` | Номер телефона бота в формате E.164 (`+15551234567`) |
| `cliPath` | Путь к `signal-cli` (`signal-cli`, если он находится в `PATH`) |
| `dmPolicy` | Политика доступа к прямым сообщениям (`pairing` рекомендуется) |
| `allowFrom` | Номера телефонов или значения `uuid:<id>`, которым разрешено отправлять прямые сообщения |

## Что это такое

- Канал Signal через `signal-cli` (не встроенная библиотека libsignal).
- Детерминированная маршрутизация: ответы всегда возвращаются в Signal.
- Прямые сообщения используют основную сессию агента; группы изолированы (`agent:<agentId>:signal:group:<groupId>`).

## Запись конфигурации

По умолчанию Signal разрешено записывать обновления конфигурации, инициированные командой `/config set|unset` (требуется `commands.config: true`).

Чтобы отключить:

```json5
{
  channels: { signal: { configWrites: false } },
}
```

## Модель номера (важно)

- Шлюз подключается к **устройству Signal** (учётная запись `signal-cli`).
- Если вы запускаете бота на **своей личной учётной записи Signal**, он будет игнорировать ваши собственные сообщения (защита от зацикливания).
- Для сценария "я отправляю сообщение боту, а он отвечает" используйте **отдельный номер для бота**.

## Путь настройки A: привязка существующей учётной записи Signal (QR)

1. Установите `signal-cli` (сборка для JVM или нативная сборка).
2. Привяжите учётную запись бота:
   - выполните `signal-cli link -n "OpenClaw"`, затем отсканируйте QR-код в Signal.
3. Настройте Signal и запустите шлюз.

Пример:

```json5
{
  channels: {
    signal: {
      enabled: true,
      account: "+15551234567",
      cliPath: "signal-cli",
      dmPolicy: "pairing",
      allowFrom: ["+15557654321"],
    },
  },
}
```

Поддержка нескольких учётных записей: используйте `channels.signal.accounts` с конфигурацией для каждой учётной записи и необязательным полем `name`. См. [`gateway/configuration`](/gateway/configuration-reference#multi-account-all-channels) для общего шаблона.

## Путь настройки B: регистрация выделенного номера для бота (SMS, Linux)

Используйте этот путь, если вы хотите выделить отдельный номер для бота, а не привязывать существующую учётную запись приложения Signal.

1. Получите номер, способный принимать SMS (или голосовую верификацию для стационарных телефонов).
   - Используйте выделенный номер для бота, чтобы избежать конфликтов учётных записей/сессий.
2. Установите `signal-cli` на хосте шлюза:

```bash
VERSION=$(curl -Ls -o /dev/null -w %{url_effective} https://github.com/AsamK/signal-cli/releases/latest | sed -e 's/^.*\/v//')
curl -L -O "https://github.com/AsamK/signal-cli/releases/download/v${VERSION}/signal-cli-${VERSION}-Linux-native.tar.gz"
sudo tar xf "signal-cli-${VERSION}-Linux-native.tar.gz" -C /opt
sudo ln -sf /opt/signal-cli /usr/local/bin/
signal-cli --version
```

Если вы используете сборку для JVM (`signal-cli-${VERSION}.tar.gz`), сначала установите JRE 25+.
Поддерживайте `signal-cli` в актуальном состоянии; в официальных примечаниях указано, что старые версии могут перестать работать из-за изменений в API серверов Signal.

3. Зарегистрируйте и подтвердите номер:

```bash
signal-cli -a +<BOT_PHONE_NUMBER> register
```

Если требуется капча:

1. Откройте `https://signalcaptchas.org/registration/generate.html`.
2. Пройдите капчу, скопируйте ссылку `signalcaptcha://...` из раздела "Open Signal".
3. Выполните команду с того же внешнего IP-адреса, что и сессия браузера (по возможности).
4. Немедленно повторите регистрацию (токены капчи быстро истекают):

```bash
signal-cli -a +<BOT_PHONE_NUMBER> register --captcha '<SIGNALCAPTCHA_URL>'
signal-cli -a +<BOT_PHONE_NUMBER> verify <VERIFICATION_CODE>
```

4. Настройте OpenClaw, перезапустите шлюз, проверьте канал:

```bash
# Если вы запускаете шлюз как пользовательскую службу systemd:
systemctl --user restart openclaw-gateway.service

# Затем проверьте:
openclaw doctor
openclaw channels status --probe
```

5. Сопрягите отправителя прямых сообщений:
   - Отправьте любое сообщение на номер бота.
   - Подтвердите код на сервере: `openclaw pairing approve signal <PAIRING_CODE>`.
   - Сохраните номер бота как контакт в своём телефоне, чтобы избежать метки "Неизвестный контакт".

Важно: регистрация учётной записи по номеру телефона с помощью `signal-cli` может привести к аннулированию основной сессии приложения Signal для этого номера. Рекомендуется использовать выделенный номер для бота или режим привязки через QR-код, если вам нужно сохранить текущую настройку приложения для телефона.

Ссылки на официальные материалы:

- README для `signal-cli`: `https://github.com/AsamK/signal-cli`
- Процесс прохождения капчи: `https://github.com/AsamK/signal-cli/wiki/Registration-with-captcha`
- Процесс привязки: `https://github.com/AsamK/signal-cli/wiki/Linking-other-devices-(Provisioning)`

## Режим внешнего демона (httpUrl)

Если вы хотите самостоятельно управлять `signal-cli` (медленный холодный старт JVM, инициализация контейнера или общие процессоры), запустите демон отдельно и укажите OpenClaw на него:

```json5
{
  channels: {
    signal: {
      httpUrl: "http://127.0.0.1:8080",
      autoStart: false,
    },
  },
}
```

Это позволит избежать автоматического запуска и ожидания запуска внутри OpenClaw. Для медленного запуска при автоматическом запуске установите `channels.signal.startupTimeoutMs`.

## Контроль доступа (прямые сообщения + группы)

Прямые сообщения:

- По умолчанию: `channels.signal.dmPolicy = "pairing"`.
- Неизвестные отправители получают код сопряжения; сообщения игнорируются до утверждения (коды истекают через 1 час).
- Утверждение через:
  - `openclaw pairing list signal`
  - `openclaw pairing approve signal <CODE>`
- Сопряжение — это обмен токенами по умолчанию для прямых сообщений Signal. Подробности: [Сопряжение](/channels/pairing)
- Отправители только по UUID (из `sourceUuid`) сохраняются как `uuid:<id>` в `channels.signal.allowFrom`.
