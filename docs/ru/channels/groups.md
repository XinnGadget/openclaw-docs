# Групповые чаты

OpenClaw обеспечивает единообразное поведение групповых чатов на следующих платформах: Discord, iMessage, Matrix, Microsoft Teams, Signal, Slack, Telegram, WhatsApp, Zalo.

## Краткое введение для начинающих (2 минуты)

OpenClaw "работает" в ваших учётных записях мессенджеров. Отдельного пользователя-бота в WhatsApp не существует.
Если **вы** состоите в группе, OpenClaw может видеть эту группу и отвечать в ней.

Поведение по умолчанию:

- Доступ к группам ограничен (`groupPolicy: "allowlist"`).
- Для ответов требуется упоминание, если вы явно не отключили фильтрацию по упоминаниям.

Перевод: отправители из списка разрешённых могут активировать OpenClaw, упомянув его.

> Кратко
>
> - **Доступ к личным сообщениям** контролируется параметром `*.allowFrom`.
> - **Доступ к группам** контролируется параметрами `*.groupPolicy` + списки разрешённых (`*.groups`, `*.groupAllowFrom`).
> - **Активация ответа** контролируется фильтрацией по упоминаниям (`requireMention`, `/activation`).

Краткий алгоритм обработки группового сообщения:

```
groupPolicy? disabled -> отбросить
groupPolicy? allowlist -> группа разрешена? нет -> отбросить
requireMention? да -> есть упоминание? нет -> сохранить только для контекста
иначе -> ответить
```

## Видимость контекста и списки разрешённых

В обеспечении безопасности групповых чатов задействованы два разных механизма:

- **Авторизация активации**: кто может активировать агента (`groupPolicy`, `groups`, `groupAllowFrom`, списки разрешённых для конкретного канала).
- **Видимость контекста**: какой дополнительный контекст передаётся модели (текст ответа, цитаты, история переписки, пересланные метаданные).

По умолчанию OpenClaw придерживается обычного поведения чата и сохраняет контекст в основном в том виде, в котором он получен. Это означает, что списки разрешённых в первую очередь определяют, кто может инициировать действия, а не служат универсальным фильтром для каждой цитируемой или исторической выдержки.

Текущее поведение зависит от канала:

- Некоторые каналы уже применяют фильтрацию отправителей для дополнительного контекста в определённых сценариях (например, заполнение потоков в Slack, поиск ответов/потоков в Matrix).
- Другие каналы по-прежнему передают контекст цитат/ответов/пересылок в том виде, в котором он получен.

Планируемые улучшения безопасности:

- `contextVisibility: "all"` (по умолчанию) сохраняет текущее поведение — контекст передаётся в том виде, в котором получен.
- `contextVisibility: "allowlist"` фильтрует дополнительный контекст, оставляя только сообщения от отправителей из списка разрешённых.
- `contextVisibility: "allowlist_quote"` — это `allowlist` плюс одно явное исключение для цитат/ответов.

Пока эта модель повышения безопасности не будет последовательно реализована на всех каналах, возможны различия в поведении.

Если вы хотите…

| Цель | Что настроить |
| --- | --- |
| Разрешить все группы, но отвечать только на упоминания | `groups: { "*": { requireMention: true } }` |
| Отключить все ответы в группах | `groupPolicy: "disabled"` |
| Разрешить только определённые группы | `groups: { "<group-id>": { ... } }` (без ключа `"*"`) |
| Только вы можете активировать агента в группах | `groupPolicy: "allowlist"`, `groupAllowFrom: ["+1555..."]` |

## Ключи сессий

- Для групповых сессий используются ключи сессий вида `agent:<agentId>:<channel>:group:<id>` (для комнат/каналов — `agent:<agentId>:<channel>:channel:<id>`).
- В темах форумов Telegram к идентификатору группы добавляется `:topic:<threadId>`, чтобы у каждой темы была своя сессия.
- Для прямых чатов используется основная сессия (или сессия для каждого отправителя, если настроено).
- Для групповых сессий сердцебиения (heartbeats) пропускаются.

## Паттерн: личные ЛС + публичные группы (один агент)

Да — это работает хорошо, если ваш "личный" трафик — это **личные сообщения (ЛС)**, а "публичный" трафик — это **группы**.

Почему: в режиме одного агента ЛС обычно попадают в **основную** сессию (`agent:main:main`), а группы всегда используют **неосновные** ключи сессий (`agent:main:<channel>:group:<id>`). Если вы включите песочницу с `mode: "non-main"`, групповые сессии будут выполняться в Docker, а ваша основная сессия ЛС останется на хосте.

Это даёт вам один "мозг" агента (общее рабочее пространство + память), но два режима выполнения:

- **ЛС**: полный набор инструментов (на хосте).
- **Группы**: песочница + ограниченный набор инструментов (Docker).

> Если вам нужны действительно отдельные рабочие пространства/персоны ("личное" и "публичное" никогда не должны смешиваться), используйте второго агента + привязки. См. [Маршрутизация между несколькими агентами](/concepts/multi-agent).

Пример (ЛС на хосте, группы в песочнице + только инструменты для обмена сообщениями):

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "non-main", // группы/каналы — не основные -> в песочнице
        scope: "session", // максимальная изоляция (один контейнер на группу/канал)
        workspaceAccess: "none",
      },
    },
  },
  tools: {
    sandbox: {
      tools: {
        // Если allow не пуст, всё остальное блокируется (deny имеет приоритет).
        allow: ["group:messaging", "group:sessions"],
        deny: ["group:runtime", "group:fs", "group:ui", "nodes", "cron", "gateway"],
      },
    },
  },
}
```

Хотите, чтобы "группы могли видеть только папку X" вместо "нет доступа к хосту"? Оставьте `workspaceAccess: "none"` и смонтируйте в песочницу только пути из списка разрешённых:

```json5
{
  agents: {
    defaults: {
      sandbox: {
        mode: "non-main",
        scope: "session",
        workspaceAccess: "none",
        docker: {
          binds: [
            // hostPath:containerPath:mode
            "/home/user/FriendsShared:/data:ro",
          ],
        },
      },
    },
  },
}
```

## Метки отображения

- В пользовательском интерфейсе метки используют `displayName`, если он доступен, в формате `<channel>:<token>`.
- `#room` зарезервировано для комнат/каналов; для групповых чатов используется `g-<slug>` (строчные буквы, пробелы заменяются на `-`, сохраняются символы `#@+._-`).
