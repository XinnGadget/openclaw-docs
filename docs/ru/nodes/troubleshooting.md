# Устранение неполадок узлов

Используйте эту страницу, если узел виден в статусе, но инструменты узла не работают.

## Последовательность команд

```bash
openclaw status
openclaw gateway status
openclaw logs --follow
openclaw doctor
openclaw channels status --probe
```

Затем выполните проверки, специфичные для узла:

```bash
openclaw nodes status
openclaw nodes describe --node <idOrNameOrIp>
openclaw approvals get --node <idOrNameOrIp>
```

Признаки исправной работы:

- Узел подключён и сопряжён для роли `node`.
- В выводе команды `nodes describe` присутствует возможность, которую вы вызываете.
- Утверждения exec показывают ожидаемый режим/белый список.

## Требования к режиму переднего плана

`canvas.*`, `camera.*` и `screen.*` работают только в режиме переднего плана на узлах iOS/Android.

Быстрая проверка и исправление:

```bash
openclaw nodes describe --node <idOrNameOrIp>
openclaw nodes canvas snapshot --node <idOrNameOrIp>
openclaw logs --follow
```

Если вы видите `NODE_BACKGROUND_UNAVAILABLE`, переведите приложение узла в режим переднего плана и повторите попытку.

## Матрица разрешений

| Возможность | iOS | Android | Приложение узла macOS | Типичный код ошибки |
| --- | --- | --- | --- | --- |
| `camera.snap`, `camera.clip` | Камера (+ микрофон для аудио в clip) | Камера (+ микрофон для аудио в clip) | Камера (+ микрофон для аудио в clip) | `*_PERMISSION_REQUIRED` |
| `screen.record` | Запись экрана (+ микрофон необязательно) | Запрос на захват экрана (+ микрофон необязательно) | Запись экрана | `*_PERMISSION_REQUIRED` |
| `location.get` | "Во время использования" или "Всегда" (зависит от режима) | Местоположение в режиме переднего/фонового плана в зависимости от режима | Разрешение на доступ к местоположению | `LOCATION_PERMISSION_REQUIRED` |
| `system.run` | н/д (путь к хосту узла) | н/д (путь к хосту узла) | Требуются утверждения exec | `SYSTEM_RUN_DENIED` |

## Сопряжение и утверждения (pairing versus approvals)

Это разные уровни контроля:

1. **Сопряжение устройства**: может ли этот узел подключиться к шлюзу?
2. **Политика команд узла шлюза**: разрешён ли идентификатор RPC-команды параметрами `gateway.nodes.allowCommands` / `denyCommands` и настройками платформы по умолчанию?
3. **Утверждения exec**: может ли этот узел выполнить конкретную команду оболочки локально?

Быстрые проверки:

```bash
openclaw devices list
openclaw nodes status
openclaw approvals get --node <idOrNameOrIp>
openclaw approvals allowlist add --node <idOrNameOrIp> "/usr/bin/uname"
```

Если сопряжение отсутствует, сначала утвердите устройство узла.
Если в выводе `nodes describe` отсутствует команда, проверьте политику команд узла шлюза и то, действительно ли узел объявил эту команду при подключении.
Если сопряжение в порядке, но `system.run` завершается ошибкой, исправьте утверждения/белый список exec на этом узле.

Сопряжение узла — это уровень идентификации/доверия, а не утверждение для каждой команды. Для `system.run` политика для каждого узла хранится в файле утверждений exec этого узла (`openclaw approvals get --node ...`), а не в записи сопряжения шлюза.

Для запусков с поддержкой утверждений `host=node` шлюз также привязывает выполнение к подготовленному каноническому `systemRunPlan`. Если последующий вызывающий изменяет команду/текущий рабочий каталог (cwd) или метаданные сеанса до того, как утверждённый запуск будет перенаправлен, шлюз отклоняет запуск как несоответствие утверждений вместо того, чтобы доверять отредактированной полезной нагрузке.

## Распространённые коды ошибок узлов

- `NODE_BACKGROUND_UNAVAILABLE` → приложение находится в фоновом режиме; переведите его в режим переднего плана.
- `CAMERA_DISABLED` → переключатель камеры отключён в настройках узла.
- `*_PERMISSION_REQUIRED` → отсутствует/отклонено разрешение ОС.
- `LOCATION_DISABLED` → режим определения местоположения отключён.
- `LOCATION_PERMISSION_REQUIRED` → запрошенный режим определения местоположения не предоставлен.
- `LOCATION_BACKGROUND_UNAVAILABLE` → приложение находится в фоновом режиме, но есть только разрешение "Во время использования".
- `SYSTEM_RUN_DENIED: approval required` → запрос exec требует явного утверждения.
- `SYSTEM_RUN_DENIED: allowlist miss` → команда заблокирована режимом белого списка.
  На хостах узлов Windows формы оболочки-обёртки, такие как `cmd.exe /c ...`, рассматриваются как пропуски белого списка в режиме белого списка, если они не утверждены через поток запросов.

## Цикл быстрого восстановления

```bash
openclaw nodes status
openclaw nodes describe --node <idOrNameOrIp>
openclaw approvals get --node <idOrNameOrIp>
openclaw logs --follow
```

Если проблема не устранена:

- Повторно утвердите сопряжение устройства.
- Повторно откройте приложение узла (в режиме переднего плана).
- Повторно предоставьте разрешения ОС.
- Пересоздайте/настройте политику утверждений exec.
