# Политика выпуска

У OpenClaw есть три публичных канала выпуска:

- stable: версии с тегами, которые по умолчанию публикуются в npm `beta`, либо в npm `latest` — если это явно запрошено;
- beta: предварительные версии с тегами, которые публикуются в npm `beta`;
- dev: текущая версия ветки `main`.

## Наименование версий

- Версия стабильного выпуска: `ГГГГ.М.Д`
  - Тег Git: `vГГГГ.М.Д`
- Версия исправления для стабильной версии: `ГГГГ.М.Д-N`
  - Тег Git: `vГГГГ.М.Д-N`
- Предварительная версия (beta): `ГГГГ.М.Д-beta.N`
  - Тег Git: `vГГГГ.М.Д-beta.N`
- Не дополняйте нулями месяц или день.
- `latest` означает текущий продвигаемый стабильный релиз в npm.
- `beta` означает текущую целевую версию для установки beta.
- Стабильные версии и версии исправлений по умолчанию публикуются в npm `beta`; операторы выпуска могут явно указать `latest` либо позднее продвинуть проверенную бета-версию.
- Каждый выпуск OpenClaw включает пакет npm и приложение для macOS.

## Периодичность выпуска

- Сначала выпускаются бета-версии.
- Стабильные версии выпускаются только после проверки последней бета-версии.
- Подробная процедура выпуска, утверждения, учётные данные и заметки по восстановлению доступны только сопровождающим.

## Предварительная проверка перед выпуском

- Выполните `pnpm build && pnpm ui:build` перед `pnpm release:check`, чтобы обеспечить наличие ожидаемых артефактов выпуска `dist/*` и пакета Control UI для этапа проверки упаковки.
- Выполните `pnpm release:check` перед каждым выпуском с тегом.
- Предварительная проверка npm для основной ветки также выполняет команду `OPENCLAW_LIVE_TEST=1 OPENCLAW_LIVE_CACHE_TEST=1 pnpm test:live:cache` перед упаковкой tar-архива, используя секреты рабочего процесса `OPENAI_API_KEY` и `ANTHROPIC_API_KEY`.
- Выполните `RELEASE_TAG=vГГГГ.М.Д node --import tsx scripts/openclaw-npm-release-check.ts` (или соответствующий тег для бета-версии/версии исправления) перед утверждением.
- После публикации в npm выполните `node --import tsx scripts/openclaw-npm-postpublish-verify.ts ГГГГ.М.Д` (или соответствующую бета-версию/версию исправления), чтобы проверить путь установки опубликованного реестра в новом временном префиксе.
- Автоматизация выпуска для сопровождающих теперь использует схему "предварительная проверка, затем продвижение":
  - реальная публикация в npm должна пройти успешную проверку `preflight_run_id` в npm;
  - стабильные выпуски в npm по умолчанию направляются в `beta`;
  - стабильные выпуски в npm могут быть явно направлены в `latest` через входные данные рабочего процесса;
  - продвижение стабильных выпусков в npm из `beta` в `latest` по-прежнему доступно в качестве явного ручного режима в доверенном рабочем процессе "OpenClaw NPM Release";
  - для этого режима продвижения по-прежнему требуется действительный `NPM_TOKEN` в среде `npm-release`, поскольку управление тегами `npm dist-tag` осуществляется отдельно от доверенной публикации;
  - публичный "macOS Release" предназначен только для проверки;
  - реальная частная публикация для macOS должна пройти успешную частную проверку `preflight_run_id` и `validate_run_id`;
  - реальные пути публикации продвигают подготовленные артефакты, не перестраивая их заново.
- Для версий исправлений стабильных выпусков (например, `ГГГГ.М.Д-N`) средство проверки после публикации также проверяет путь обновления в временном префиксе с `ГГГГ.М.Д` до `ГГГГ.М.Д-N`, чтобы исправления выпуска не оставляли незаметно старые глобальные установки на базовой стабильной версии.
- Предварительная проверка выпуска в npm завершается ошибкой, если tar-архив не содержит `dist/control-ui/index.html` и непустой полезной нагрузки `dist/control-ui/assets/`, чтобы мы больше не отправляли пустую панель управления в браузере.
- Если работа по выпуску затрагивала планирование CI, манифесты времени выполнения расширений или быстрые тестовые матрицы, перегенерируйте и проверьте выходные данные матрицы рабочего процесса `checks-fast-extensions`, принадлежащие планировщику, из `.github/workflows/ci.yml` перед утверждением, чтобы в заметках о выпуске не описывалась устаревшая структура CI.
- Готовность стабильного выпуска для macOS также включает поверхности обновления:
  - в выпуске GitHub должны быть упакованы файлы `.zip`, `.dmg` и `.dSYM.zip`;
  - `appcast.xml` в ветке `main` должен указывать на новый стабильный zip-архив после публикации;
  - упакованное приложение должно иметь идентификатор пакета без отладки, непустой URL-адрес ленты Sparkle и `CFBundleVersion`, равный или превышающий канонический минимальный уровень сборки Sparkle для этой версии выпуска.

## Входные данные рабочего процесса npm

"OpenClaw NPM Release" принимает следующие управляемые оператором входные данные:

- `tag`: обязательный тег выпуска, например `v2026.4.2`, `v2026.4.2-1` или `v2026.4.2-beta.1`;
- `preflight_only`: `true` для проверки/сборки/упаковки, `false` для реального пути публикации;
- `preflight_run_id`: обязателен для реального пути публикации, чтобы рабочий процесс использовал подготовленный tar-архив из успешного запуска предварительной проверки;
- `npm_dist_tag`: тег цели npm для пути публикации; по умолчанию — `beta`;
- `promote_beta_to_latest`: `true` для пропуска публикации и перемещения уже опубликованной стабильной бета-версии в `latest`.

Правила:

- Теги стабильных версий и версий исправлений могут публиковаться как в `beta`, так и в `latest`.
- Теги предварительных бета-версий могут публиковаться только в `beta`.
- Реальный путь публикации должен использовать тот же `npm_dist_tag`, что и во время предварительной проверки; рабочий процесс проверяет эти метаданные перед продолжением публикации.
- Режим продвижения должен использовать стабильный тег или тег версии исправления, `preflight_only=false`, пустой `preflight_run_id` и `npm_dist_tag=beta`.
- Для режима продвижения также требуется действительный `NPM_TOKEN` в среде `npm-release`, поскольку для `npm dist-tag add` по-прежнему требуется обычная аутентификация npm.

## Последовательность выпуска стабильной версии в npm

При выпуске стабильной версии в npm:

1. Запустите "OpenClaw NPM Release" с `preflight_only=true`.
2. Выберите `npm_dist_tag=beta` для обычного потока "сначала бета", либо `latest` — если вы намеренно хотите выполнить прямую публикацию стабильной версии.
3. Сохраните успешный `preflight_run_id`.
4. Снова запустите "OpenClaw NPM Release" с `preflight_only=false`, тем же `tag`, тем же `npm_dist_tag` и сохранённым `preflight_run_id`.
5. Если выпуск попал в `beta`, запустите "OpenClaw NPM Release" позже с тем же стабильным `tag`, `promote_beta_to_latest=true`, `preflight_only=false`, пустым `preflight_run_id` и `npm_dist_tag=beta`, если вы хотите переместить опубликованную сборку в `latest`.

Для режима продвижения по-прежнему требуется утверждение среды `npm-release` и действительный `NPM_TOKEN` в этой среде.

Это позволяет задокументировать и сделать видимым для операторов как путь прямой публикации, так и путь продвижения "сначала бета".
