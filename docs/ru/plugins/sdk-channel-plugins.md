# Создание плагинов каналов

Это руководство описывает процесс создания плагина канала, который подключает OpenClaw к платформе обмена сообщениями. К концу работы у вас будет готовый канал с защитой личных сообщений (DM), согласованием, потоковой обработкой ответов и отправкой исходящих сообщений.

<Info>
  Если вы ещё не создавали плагины для OpenClaw, сначала прочитайте раздел [Начало работы](/plugins/building-plugins), чтобы ознакомиться с базовой структурой пакета и настройкой манифеста.
</Info>

## Как работают плагины каналов

Плагинам каналов не нужны собственные инструменты отправки, редактирования и реакций. В ядре OpenClaw есть один общий инструмент `message`. Ваш плагин отвечает за:

- **Конфигурацию** — разрешение учётных записей и мастер настройки;
- **Безопасность** — политику DM и списки разрешений;
- **Согласование** — процесс утверждения DM;
- **Грамматику сессии** — сопоставление идентификаторов разговоров, специфичных для провайдера, с базовыми чатами, идентификаторами потоков и резервными вариантами родительских элементов;
- **Исходящие сообщения** — отправку текста, медиа и опросов на платформу;
- **Потоковую обработку** — организацию потоков ответов.

Ядро отвечает за общий инструмент сообщений, подключение подсказок, внешнюю форму ключа сессии, общий учёт `:thread:` и диспетчеризацию.

Если ваша платформа хранит дополнительные данные в идентификаторах разговоров, обрабатывайте их в плагине с помощью `messaging.resolveSessionConversation(...)`. Это стандартный хук для сопоставления `rawId` с базовым идентификатором разговора, необязательным идентификатором потока, явным `baseConversationId` и любыми `parentConversationCandidates`. Когда вы возвращаете `parentConversationCandidates`, упорядочивайте их от самого узкого родительского элемента к самому широкому/базовому разговору.

Встроенные плагины, которым требуется такая же обработка до запуска реестра каналов, могут также предоставить файл верхнего уровня `session-key-api.ts` с соответствующим экспортом `resolveSessionConversation(...)`. Ядро использует этот безопасный при загрузке интерфейс только тогда, когда реестр плагинов во время выполнения ещё недоступен.

`messaging.resolveParentConversationCandidates(...)` остаётся доступным как резервный вариант для обеспечения совместимости, если плагину нужны только резервные варианты родительских элементов поверх общего/сырого идентификатора. Если оба хука существуют, ядро сначала использует `resolveSessionConversation(...).parentConversationCandidates` и обращается к `resolveParentConversationCandidates(...)` только в том случае, если канонический хук их не предоставляет.

## Утверждения и возможности канала

Большинству плагинов каналов не требуется код, специфичный для утверждений.

- Ядро отвечает за команду `/approve` в рамках одного чата, общие полезные нагрузки кнопок утверждения и общую доставку резервных вариантов.
- Предпочитайте использовать один объект `approvalCapability` в плагине канала, если каналу требуется поведение, специфичное для утверждений.
- `ChannelPlugin.approvals` удалён. Поместите данные о доставке, нативной реализации, рендеринге и аутентификации утверждений в `approvalCapability`.
- `plugin.auth` предназначен только для входа и выхода из системы; ядро больше не считывает хуки аутентификации утверждений из этого объекта.
- `approvalCapability.authorizeActorAction` и `approvalCapability.getActionAvailabilityState` — это канонические точки входа для аутентификации утверждений.
- Используйте `approvalCapability.getActionAvailabilityState` для проверки доступности аутентификации утверждений в рамках одного чата.
- Если ваш канал поддерживает нативные утверждения exec, используйте `approvalCapability.getExecInitiatingSurfaceState` для состояния инициирующей поверхности/нативного клиента, если оно отличается от аутентификации утверждений в рамках одного чата. Ядро использует этот специфичный для exec хук, чтобы различать состояния `enabled` и `disabled`, определять, поддерживает ли инициирующий канал нативные утверждения exec, и включать канал в рекомендации по резервному варианту для нативного клиента. Для типичного случая используется `createApproverRestrictedNativeApprovalCapability(...)`.
- Используйте `outbound.shouldSuppressLocalPayloadPrompt` или `outbound.beforeDeliverPayload` для поведения жизненного цикла полезных нагрузок, специфичного для канала, например, для скрытия повторяющихся локальных подсказок об утверждениях или отправки индикаторов набора текста перед доставкой.
- Используйте `approvalCapability.delivery` только для маршрутизации нативных утверждений или подавления резервных вариантов.
- Используйте `approvalCapability.nativeRuntime` для данных о нативных утверждениях, принадлежащих каналу. Отложите загрузку на горячих точках входа в канал с помощью `createLazyChannelApprovalNativeRuntimeAdapter(...)`, который может импортировать ваш модуль во время выполнения по требованию, позволяя ядру собрать жизненный цикл утверждения.
- Используйте `approvalCapability.render` только в том случае, если каналу действительно нужны пользовательские полезные нагрузки утверждений вместо общего рендерера.
- Используйте `approvalCapability.describeExecApprovalSetup`, если канал хочет, чтобы ответ по пути отключения объяснял точные настройки конфигурации, необходимые для включения нативных утверждений exec. Хук получает `{ channel, channelLabel, accountId }`; каналы с именованными учётными записями должны отображать пути, привязанные к учётной записи, например `channels.<channel>.accounts.<id>.execApprovals.*`, вместо настроек верхнего уровня по умолчанию.
- Если канал может вывести стабильные идентификаторы DM, похожие на владельца, из существующей конфигурации, используйте `createResolvedApproverActionAuthAdapter` из `openclaw/plugin-sdk/approval-runtime`, чтобы ограничить команду `/approve` в рамках одного чата без добавления специфичной для утверждений логики в ядро.
- Если каналу требуется доставка нативных утверждений, сосредоточьте код канала на нормализации целевых объектов, а также на данных о транспортировке и представлении. Используйте `createChannelExecApprovalProfile`, `createChannelNativeOriginTargetResolver`, `createChannelApproverDmTargetResolver` и `createApproverRestrictedNativeApprovalCapability` из `openclaw/plugin-sdk/approval-runtime`. Поместите данные, специфичные для канала, в `approvalCapability.nativeRuntime`, предпочтительно через `createChannelApprovalNativeRuntimeAdapter(...)` или `createLazyChannelApprovalNativeRuntimeAdapter(...)`, чтобы ядро могло собрать обработчик и управлять фильтрацией запросов, маршрутизацией, дедупликацией, истечением срока действия, подпиской на шлюз и уведомлениями о перенаправлении. `nativeRuntime` разделён на несколько более мелких точек входа:
  - `availability` — настроена ли учётная запись и должен ли запрос быть обработан;
  - `presentation` — преобразование общей модели представления утверждений в ожидающие, разрешённые или истёкшие нативные полезные нагрузки или окончательные действия;
  - `transport` — подготовка целевых объектов, а также отправка, обновление и удаление нативных сообщений об утверждениях;
  - `interactions` — необязательные хуки привязки/отвязки/очистки действий для нативных кнопок или реакций;
  - `observe` — необязательные хуки диагностики доставки.
- Если каналу требуются объекты, принадлежащие времени выполнения, такие как клиент, токен, приложение Bolt или приёмник вебхуков, зарегистрируйте их через `openclaw/plugin-sdk/channel-runtime-context`. Общий реестр контекста времени выполнения позволяет ядру загружать обработчики, управляемые возможностями, из состояния запуска канала без добавления специфичных для утверждений связующих компонентов.
- Используйте низкоуровневые `createChannelApprovalHandler` или `createChannelNativeApprovalRuntime` только в том случае, если точка входа, управляемая возможностями, ещё недостаточно выразительна.
- Каналы с нативными утверждениями должны передавать `accountId` и `approvalKind` через эти вспомогательные функции. `accountId` позволяет ограничить политику утверждений для нескольких учётных записей нужной учётной записью бота, а `approvalKind` позволяет каналу различать поведение утверждений exec и плагинов без жёстко заданных ветвлений в ядре.
- Теперь ядро также отвечает за уведомления о перенаправлении утверждений. Плагины каналов не должны отправлять собственные сообщения о том, что "утверждение было отправлено в DM или другой канал" из `createChannelNativeApprovalRuntime`; вместо этого предоставьте точные данные о происхождении и маршрутизации DM утверждающего через общие вспомогательные функции возможностей утверждений и позвольте ядру агрегировать фактические доставки перед отправкой любого уведомления обратно в инициирующий чат.
- Сохраняйте тип идентификатора доставленного утверждения от начала до конца. Нативные клиенты не должны угадывать или переписывать маршрутизацию утверждений exec и плагинов на основе локального состояния канала.
- Различные типы утверждений могут намеренно предоставлять разные нативные интерфейсы.
  Текущие встроенные примеры:
  - Slack сохраняет нативную маршрутизацию утверждений доступной как для идентификаторов exec, так и для идентификаторов плагинов.
  - Matrix сохраняет одинаковую нативную маршрутизацию DM/канала и UX реакций для утверждений exec и плагинов, при этом позволяя аутентификации различаться в зависимости от типа утверждения.
- `createApproverRestrictedNativeApprovalAdapter` всё ещё существует как оболочка для обеспечения совместимости, но в новом коде следует предпочитать построитель возможностей и предоставлять `approvalCapability` в плагине.
