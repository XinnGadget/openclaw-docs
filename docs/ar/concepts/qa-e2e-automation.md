# أتمتة QA الشاملة من البداية إلى النهاية

تهدف حزمة QA الخاصة إلى اختبار OpenClaw بطريقة أكثر واقعية وبهيئة أقرب إلى القنوات، مقارنة بما يمكن لاختبار وحدة واحد أن يقدمه.

الأجزاء الحالية:

- `extensions/qa-channel`: قناة رسائل اصطناعية تتضمن أسطح الرسائل الخاصة DM، والقنوات، والخيوط، والتفاعلات، والتحرير، والحذف.
- `extensions/qa-lab`: واجهة مستخدم لتصحيح الأخطاء وناقل QA لمراقبة النص الحواري، وحقن الرسائل الواردة، وتصدير تقرير Markdown.
- `qa/`: أصول تمهيدية مدعومة بالمستودع لمهمة البداية وسيناريوهات QA الأساسية.

تدفق عمل مشغّل QA الحالي هو موقع QA بلوحتين:

- اليسار: لوحة معلومات Gateway (Control UI) مع الوكيل.
- اليمين: QA Lab، ويعرض النص الحواري بأسلوب يشبه Slack وخطة السيناريو.

شغّله باستخدام:

```bash
pnpm qa:lab:up
```

يبني هذا موقع QA، ويبدأ مسار Gateway المدعوم بـ Docker، ويعرض صفحة QA Lab حيث يمكن لمشغّل أو حلقة أتمتة أن يكلّف الوكيل بمهمة QA، ويراقب سلوك القناة الفعلي، ويسجل ما نجح وما فشل وما بقي معطّلًا.

للحصول على تكرار أسرع في واجهة QA Lab من دون إعادة بناء صورة Docker في كل مرة، ابدأ الحزمة مع حزمة QA Lab مركّبة عبر bind mount:

```bash
pnpm openclaw qa docker-build-image
pnpm qa:lab:build
pnpm qa:lab:up:fast
pnpm qa:lab:watch
```

يبقي `qa:lab:up:fast` خدمات Docker على صورة مبنية مسبقًا ويربط `extensions/qa-lab/web/dist` داخل حاوية `qa-lab` عبر bind mount. ويعيد `qa:lab:watch` بناء تلك الحزمة عند التغيير، ويعيد المتصفح التحميل تلقائيًا عندما يتغير تجزئة أصول QA Lab.

لتشغيل مسار smoke حقيقي النقل لـ Matrix، شغّل:

```bash
pnpm openclaw qa matrix
```

يوفّر هذا المسار خادم Tuwunel منزليًا مؤقتًا داخل Docker، ويسجل مستخدمين مؤقتين للسائق، وSUT، والمراقب، وينشئ غرفة خاصة واحدة، ثم يشغّل Plugin Matrix الحقيقي داخل عملية فرعية QA gateway. ويحافظ مسار النقل الحي على ضبط العملية الفرعية محصورًا في النقل قيد الاختبار، بحيث يعمل Matrix من دون `qa-channel` في ضبط العملية الفرعية.

لتشغيل مسار smoke حقيقي النقل لـ Telegram، شغّل:

```bash
pnpm openclaw qa telegram
```

يستهدف هذا المسار مجموعة Telegram خاصة حقيقية واحدة بدلًا من توفير خادم مؤقت. ويتطلب `OPENCLAW_QA_TELEGRAM_GROUP_ID` و`OPENCLAW_QA_TELEGRAM_DRIVER_BOT_TOKEN` و`OPENCLAW_QA_TELEGRAM_SUT_BOT_TOKEN`، بالإضافة إلى روبوتين مختلفين داخل المجموعة الخاصة نفسها. ويجب أن يمتلك روبوت SUT اسم مستخدم على Telegram، وتعمل المراقبة بين الروبوتات على أفضل نحو عندما يكون لدى كلا الروبوتين وضع Bot-to-Bot Communication Mode مفعّلًا في `@BotFather`.

تتشارك مسارات النقل الحية الآن عقدًا أصغر واحدًا بدلًا من أن يخترع كل منها شكل قائمة السيناريوهات الخاصة به.

يبقى `qa-channel` حزمة سلوك المنتج الاصطناعية الواسعة، وليس جزءًا من مصفوفة تغطية النقل الحي.

| المسار | Canary | تقييد الإشارة | حظر قائمة السماح | رد من المستوى الأعلى | استئناف بعد إعادة التشغيل | متابعة الخيط | عزل الخيط | مراقبة التفاعل | أمر المساعدة |
| ------ | ------ | -------------- | ----------------- | --------------------- | -------------------------- | ------------ | ---------- | --------------- | ------------ |
| Matrix   | x      | x              | x                 | x                     | x                          | x            | x          | x               |              |
| Telegram | x      |                |                   |                       |                            |              |            |                 | x            |

يبقي هذا `qa-channel` كحزمة سلوك المنتج الواسعة، بينما تتشارك Matrix وTelegram ووسائط النقل الحية المستقبلية قائمة فحص صريحة واحدة لعقد النقل.

لتشغيل مسار Linux VM مؤقت من دون إدخال Docker في مسار QA، شغّل:

```bash
pnpm openclaw qa suite --runner multipass --scenario channel-chat-baseline
```

يقوم هذا بتشغيل ضيف Multipass جديد، وتثبيت التبعيات، وبناء OpenClaw داخل الضيف، وتشغيل `qa suite`، ثم نسخ تقرير QA العادي والملخص مرة أخرى إلى `.artifacts/qa-e2e/...` على المضيف.
ويعيد استخدام السلوك نفسه لاختيار السيناريو كما في `qa suite` على المضيف.
تنفّذ عمليات suite على المضيف وMultipass عدة سيناريوهات محددة بالتوازي مع عمّال Gateway معزولين افتراضيًا، حتى 64 عاملًا أو عدد السيناريوهات المحددة. استخدم `--concurrency <count>` لضبط عدد العمّال، أو `--concurrency 1` للتنفيذ التسلسلي.
تُمرِّر التشغيلات الحية مدخلات مصادقة QA المدعومة والعملية للضيف: مفاتيح المزوّد المعتمدة على env، ومسار ضبط مزوّد QA الحي، و`CODEX_HOME` عند وجوده. أبقِ `--output-dir` تحت جذر المستودع حتى يتمكن الضيف من الكتابة مرة أخرى عبر مساحة العمل المركّبة.

## البذور المدعومة بالمستودع

توجد الأصول التمهيدية في `qa/`:

- `qa/scenarios/index.md`
- `qa/scenarios/*.md`

هذه موجودة عمدًا في git بحيث تكون خطة QA مرئية لكل من البشر والوكيل.

يجب أن يبقى `qa-lab` مشغّل Markdown عامًا. كل ملف Markdown خاص بسيناريو هو المصدر المرجعي لتشغيل اختبار واحد، ويجب أن يعرّف:

- بيانات السيناريو الوصفية
- مراجع المستندات والشفرة
- متطلبات Plugin الاختيارية
- تصحيح ضبط Gateway اختياري
- `qa-flow` القابل للتنفيذ

يجب أن تبقى القائمة الأساسية واسعة بما يكفي لتغطية:

- محادثات DM والقنوات
- سلوك الخيوط
- دورة حياة إجراءات الرسائل
- استدعاءات Cron
- استرجاع الذاكرة
- تبديل النماذج
- تمرير المهمة إلى وكيل فرعي
- قراءة المستودع وقراءة المستندات
- مهمة بناء صغيرة واحدة مثل Lobster Invaders

## محولات النقل

يمتلك `qa-lab` طبقة نقل عامة لسيناريوهات QA المكتوبة بـ Markdown.
`qa-channel` هو أول محول على هذه الطبقة، لكن الهدف التصميمي أوسع:
يجب أن تُوصَل القنوات المستقبلية، سواء كانت حقيقية أو اصطناعية، إلى مشغّل suite نفسه بدلًا من إضافة مشغّل QA خاص بالنقل.

على مستوى البنية، يكون التقسيم كالتالي:

- يمتلك `qa-lab` تنفيذ السيناريو العام، وتزامن العمّال، وكتابة الأصول الناتجة، وإعداد التقارير.
- يمتلك محول النقل ضبط Gateway، والجاهزية، ومراقبة الوارد والصادر، وإجراءات النقل، وحالة النقل المطبّعة.
- تحدد ملفات السيناريو Markdown تحت `qa/scenarios/` تشغيل الاختبار؛ ويوفر `qa-lab` سطح وقت التشغيل القابل لإعادة الاستخدام الذي ينفذها.

يوجد دليل الاعتماد الموجه للمشرفين لمحولات القنوات الجديدة في
[Testing](/ar/help/testing#adding-a-channel-to-qa).

## إعداد التقارير

يصدّر `qa-lab` تقرير بروتوكول بصيغة Markdown من المخطط الزمني للناقل المرصود.
ويجب أن يجيب التقرير عن:

- ما الذي نجح
- ما الذي فشل
- ما الذي بقي معطّلًا
- ما سيناريوهات المتابعة التي تستحق الإضافة

لفحوصات الطابع والأسلوب، شغّل السيناريو نفسه عبر عدة مراجع نماذج حية
واكتب تقرير Markdown مُقيَّمًا:

```bash
pnpm openclaw qa character-eval \
  --model openai/gpt-5.4,thinking=xhigh \
  --model openai/gpt-5.2,thinking=xhigh \
  --model openai/gpt-5,thinking=xhigh \
  --model anthropic/claude-opus-4-6,thinking=high \
  --model anthropic/claude-sonnet-4-6,thinking=high \
  --model zai/glm-5.1,thinking=high \
  --model moonshot/kimi-k2.5,thinking=high \
  --model google/gemini-3.1-pro-preview,thinking=high \
  --judge-model openai/gpt-5.4,thinking=xhigh,fast \
  --judge-model anthropic/claude-opus-4-6,thinking=high \
  --blind-judge-models \
  --concurrency 16 \
  --judge-concurrency 16
```

يشغّل هذا الأمر عمليات فرعية محلية لـ QA gateway، وليس Docker. ويجب أن تضبط سيناريوهات تقييم الطابع الشخصية عبر `SOUL.md`، ثم تُجري أدوار مستخدم عادية مثل الدردشة، والمساعدة في مساحة العمل، والمهام الصغيرة على الملفات. ويجب ألا يتم إخبار النموذج المرشح بأنه قيد التقييم. ويحافظ الأمر على كل نص حواري كامل، ويسجّل إحصاءات التشغيل الأساسية، ثم يطلب من نماذج التحكيم في الوضع السريع مع استدلال `xhigh` ترتيب التشغيلات حسب الطبيعية والطابع والفكاهة.
استخدم `--blind-judge-models` عند مقارنة المزوّدين: إذ يظل موجّه التحكيم يتلقى كل نص حواري وحالة تشغيل، لكن تُستبدل مراجع المرشحين بتسميات محايدة مثل `candidate-01`؛ ثم يربط التقرير التصنيفات مرة أخرى بالمراجع الفعلية بعد التحليل.
تستخدم تشغيلات المرشحين افتراضيًا مستوى تفكير `high`، مع `xhigh` لنماذج OpenAI التي تدعمه. ويمكنك تجاوز مرشح محدد ضمنيًا باستخدام
`--model provider/model,thinking=<level>`. ولا يزال `--thinking <level>` يضبط قيمة احتياطية عامة، كما يُحتفظ بالصيغة الأقدم `--model-thinking <provider/model=level>` للتوافق.
تستخدم مراجع المرشحين الخاصة بـ OpenAI الوضع السريع افتراضيًا بحيث تُستخدم المعالجة ذات الأولوية عندما يدعم المزوّد ذلك. أضف `,fast` أو `,no-fast` أو `,fast=false` ضمن المرجع عندما يحتاج مرشح واحد أو حكم واحد إلى تجاوز هذا السلوك. مرّر `--fast` فقط عندما تريد فرض الوضع السريع لكل نماذج المرشحين. وتُسجَّل مدد تشغيل المرشحين والحكّام في التقرير لتحليل القياس، لكن موجّهات الحكّام تنص صراحة على عدم الترتيب حسب السرعة.
تستخدم كل من تشغيلات نماذج المرشحين والحكّام تزامنًا افتراضيًا مقداره 16. خفّض
`--concurrency` أو `--judge-concurrency` عندما تجعل حدود المزوّد أو ضغط Gateway المحلي التشغيل مشوشًا أكثر من اللازم.
عند عدم تمرير أي `--model` للمرشحين، يستخدم تقييم الطابع افتراضيًا
`openai/gpt-5.4` و`openai/gpt-5.2` و`openai/gpt-5` و`anthropic/claude-opus-4-6` و`anthropic/claude-sonnet-4-6` و`zai/glm-5.1` و`moonshot/kimi-k2.5` و`google/gemini-3.1-pro-preview`.
وعند عدم تمرير أي `--judge-model`، تكون نماذج التحكيم الافتراضية هي
`openai/gpt-5.4,thinking=xhigh,fast` و
`anthropic/claude-opus-4-6,thinking=high`.
