## Быстрая настройка

<Steps>
  <Step title="Создать токен бота в BotFather">
    Откройте Telegram и начните чат с **@BotFather** (убедитесь, что имя пользователя именно `@BotFather`).

    Выполните команду `/newbot`, следуйте инструкциям и сохраните токен.
  </Step>

  <Step title="Настроить токен и политику личных сообщений">

```json5
{
  channels: {
    telegram: {
      enabled: true,
      botToken: "123:abc",
      dmPolicy: "pairing",
      groups: { "*": { requireMention: true } },
    },
  },
}
```

    Резервное значение из окружения: `TELEGRAM_BOT_TOKEN=...` (только для учётной записи по умолчанию).
    Telegram **не** использует `openclaw channels login telegram`; настройте токен в конфигурации/окружении, затем запустите шлюз.
  </Step>

  <Step title="Запустить шлюз и подтвердить первое личное сообщение">

```bash
openclaw gateway
openclaw pairing list telegram
openclaw pairing approve telegram <CODE>
```

    Коды сопряжения истекают через 1 час.
  </Step>

  <Step title="Добавить бота в группу">
    Добавьте бота в свою группу, затем настройте `channels.telegram.groups` и `groupPolicy` в соответствии с вашей моделью доступа.
  </Step>
</Steps>

<Note>
Порядок разрешения токенов учитывает учётную запись. На практике значения конфигурации имеют приоритет над резервными значениями из окружения, а `TELEGRAM_BOT_TOKEN` применяется только к учётной записи по умолчанию.
</Note>

## Настройки со стороны Telegram

<AccordionGroup>
  <Accordion title="Режим конфиденциальности и видимость группы">
    Боты Telegram по умолчанию работают в **режиме конфиденциальности**, который ограничивает объём получаемых ими сообщений из групп.

    Если бот должен видеть все сообщения группы, выполните одно из следующих действий:

    - отключите режим конфиденциальности через `/setprivacy`;
    - сделайте бота администратором группы.

    При изменении режима конфиденциальности удалите бота из каждой группы и добавьте его заново, чтобы Telegram применил изменения.
  </Accordion>

  <Accordion title="Права доступа в группе">
    Статус администратора управляется в настройках группы Telegram.

    Боты-администраторы получают все сообщения группы, что полезно для постоянного поведения в группе.
  </Accordion>

  <Accordion title="Полезные переключатели BotFather">

    - `/setjoingroups` — разрешить или запретить добавление в группы;
    - `/setprivacy` — настроить поведение видимости в группах.
  </Accordion>
</AccordionGroup>

## Контроль доступа и активация

<Tabs>
  <Tab title="Политика личных сообщений">
    `channels.telegram.dmPolicy` управляет доступом к личным сообщениям:

    - `pairing` (по умолчанию);
    - `allowlist` (требуется хотя бы один ID отправителя в `allowFrom`);
    - `open` (требуется, чтобы `allowFrom` содержал `"*"`);
    - `disabled`.

    `channels.telegram.allowFrom` принимает числовые ID пользователей Telegram. Префиксы `telegram:` / `tg:` принимаются и нормализуются.
    `dmPolicy: "allowlist"` с пустым `allowFrom` блокирует все личные сообщения и отклоняется при проверке конфигурации.
    При регистрации принимается ввод `@username` и преобразуется в числовые ID.
    Если вы обновились и ваша конфигурация содержит записи `@username` в списке разрешённых, выполните `openclaw doctor --fix`, чтобы преобразовать их (по возможности; требуется токен бота Telegram).
    Если вы ранее полагались на файлы списка разрешённых в хранилище сопряжения, `openclaw doctor --fix` может восстановить записи в `channels.telegram.allowFrom` в потоках списка разрешённых (например, когда `dmPolicy: "allowlist"` не имеет явных ID).

    Для ботов с одним владельцем предпочтительнее использовать `dmPolicy: "allowlist"` с явными числовыми ID в `allowFrom`, чтобы политика доступа сохранялась в конфигурации (вместо зависимости от предыдущих утверждений сопряжения).

    Распространённая ошибка: утверждение сопряжения для личных сообщений не означает "этот отправитель авторизован везде".
    Сопряжение предоставляет доступ только к личным сообщениям. Авторизация отправителя в группе по-прежнему осуществляется через явные списки разрешённых в конфигурации.
    Если вы хотите, чтобы "я был авторизован один раз, и личные сообщения и команды в группе работали", добавьте свой числовой ID пользователя Telegram в `channels.telegram.allowFrom`.

    ### Как узнать свой ID пользователя Telegram

    Безопасный способ (без сторонних ботов):

    1. Отправьте личное сообщение своему боту.
    2. Выполните `openclaw logs --follow`.
    3. Прочитайте `from.id`.

    Официальный метод Bot API:

```bash
curl "https://api.telegram.org/bot<bot_token>/getUpdates"
```

    Сторонний метод (менее конфиденциальный): `@userinfobot` или `@getidsbot`.
  </Tab>

  <Tab title="Политика групп и списки разрешённых">
    Применяются два контроля:

    1. **Какие группы разрешены** (`channels.telegram.groups`):
       - нет конфигурации `groups`:
         - при `groupPolicy: "open"`: любая группа может пройти проверку ID группы;
         - при `groupPolicy: "allowlist"` (по умолчанию): группы заблокированы, пока вы не добавите записи `groups` (или `"*"`);
       - конфигурация `groups`: действует как список разрешённых (явные ID или `"*"`).

    2. **Какие отправители разрешены в группах** (`channels.telegram.groupPolicy`):
       - `open`;
       - `allowlist` (по умолчанию);
       - `disabled`.

    `groupAllowFrom` используется для фильтрации отправителей в группах. Если не задано, Telegram использует резерв `allowFrom`.
    Записи `groupAllowFrom` должны быть числовыми ID пользователей Telegram (`telegram:` / `tg:` префиксы нормализуются).
    Не добавляйте ID чатов групп или супергрупп Telegram в `groupAllowFrom`. Отрицательные ID чатов относятся к `channels.telegram.groups`.
    Нечисловые записи игнорируются при авторизации отправителя.
    Граница безопасности (`2026.2.25+`): авторизация отправителя в группе **не** наследует утверждения из хранилища сопряжения для личных сообщений.
    Сопряжение остаётся только для личных сообщений. Для групп настройте `groupAllowFrom` или `allowFrom` для каждой группы/темы.
    Если `groupAllowFrom` не задано, Telegram использует резерв `allowFrom` из конфигурации, а не из хранилища сопряжения.
    Практический шаблон для ботов с одним владельцем: добавьте свой ID пользователя в `channels.telegram.allowFrom`, оставьте `groupAllowFrom` незаданным и разрешите целевые группы в `channels.telegram.groups`.
    Примечание о времени выполнения: если `channels.telegram` полностью отсутствует, по умолчанию используется `groupPolicy="allowlist"` с отказом в доступе, если `channels.defaults.groupPolicy` явно не задано.

    Пример: разрешить любому участнику в одной конкретной группе:

```json5
{
  channels: {
    telegram: {
      groups: {
        "-1001234567890": {
          groupPolicy: "open",
          requireMention: false,
        },
      },
    },
  },
}
```

    Пример: разрешить только определённым пользователям в одной конкретной группе:

```json5
{
  channels: {
    telegram: {
      groups: {
        "-1001234567890": {
          requireMention: true,
          allowFrom: ["8734062810", "745123456"],
        },
      },
    },
  },
}
```
