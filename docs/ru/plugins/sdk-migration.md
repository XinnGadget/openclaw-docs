# Миграция на Plugin SDK

В OpenClaw произошёл переход от широкого слоя обратной совместимости к современной архитектуре плагинов с целенаправленными, документированными импортами. Если ваш плагин был создан до появления новой архитектуры, это руководство поможет вам выполнить миграцию.

## Что меняется

Старая система плагинов предоставляла две широкие возможности, позволяющие плагинам импортировать всё необходимое из единой точки входа:

- **`openclaw/plugin-sdk/compat`** — единый импорт, который повторно экспортировал десятки вспомогательных инструментов. Он был введён для обеспечения работы старых плагинов на основе хуков во время разработки новой архитектуры плагинов.
- **`openclaw/extension-api`** — мост, который давал плагинам прямой доступ к вспомогательным инструментам на стороне хоста, таким как встроенный раннер агентов.

Обе возможности теперь **устарели**. Они по-прежнему работают во время выполнения, но новые плагины не должны их использовать, а существующие плагины следует мигрировать до следующего крупного релиза, в котором они будут удалены.

<Warning>
Слой обратной совместимости будет удалён в одном из будущих крупных релизов. Плагины, которые по-прежнему используют эти возможности, перестанут работать после этого.
</Warning>

## Почему произошли изменения

Старый подход вызывал следующие проблемы:

- **Медленный запуск** — импорт одного вспомогательного инструмента загружал десятки несвязанных модулей.
- **Циклические зависимости** — широкие повторные экспорта облегчали создание циклов импорта.
- **Неясность API** — невозможно было определить, какие экспорта являются стабильными, а какие — внутренними.

Современный Plugin SDK решает эти проблемы: каждый путь импорта (`openclaw/plugin-sdk/\<subpath\>`) представляет собой небольшой автономный модуль с чёткой целью и документированным контрактом.

Устаревшие удобные интерфейсы для встроенных каналов также были удалены. Импорты типа `openclaw/plugin-sdk/slack`, `openclaw/plugin-sdk/discord`, `openclaw/plugin-sdk/signal`, `openclaw/plugin-sdk/whatsapp`, специализированные вспомогательные интерфейсы для каналов и `openclaw/plugin-sdk/telegram-core` были частными сокращениями в монорепозитории, а не стабильными контрактами для плагинов. Вместо них следует использовать узкие общие подпути SDK. Во встроенном рабочем пространстве плагина храните вспомогательные инструменты, принадлежащие провайдеру, в файлах `api.ts` или `runtime-api.ts` этого плагина.

Примеры встроенных провайдеров:

- Anthropic хранит вспомогательные инструменты для потоков, специфичные для Claude, в своих файлах `api.ts` / `contract-api.ts`.
- OpenAI хранит построители провайдеров, вспомогательные инструменты для моделей по умолчанию и построители провайдеров для реального времени в своём файле `api.ts`.
- OpenRouter хранит построитель провайдера и вспомогательные инструменты для настройки/конфигурации в своём файле `api.ts`.

## Как выполнить миграцию

<Steps>
  <Step title="Миграция обработчиков, поддерживающих утверждения, к фактам возможностей">
    Плагины каналов, поддерживающие утверждения, теперь предоставляют собственное поведение утверждений через `approvalCapability.nativeRuntime` вместе с общим реестром контекста выполнения.

    Ключевые изменения:

    - Замените `approvalCapability.handler.loadRuntime(...)` на `approvalCapability.nativeRuntime`.
    - Перенесите аутентификацию/доставку, специфичные для утверждений, с устаревших `plugin.auth` / `plugin.approvals` на `approvalCapability`.
    - `ChannelPlugin.approvals` был удалён из публичного контракта канала-плагина; перенесите поля доставки/нативного/рендеринга на `approvalCapability`.
    - `plugin.auth` остаётся только для потоков входа/выхода из канала; хуки аутентификации утверждений там больше не считываются ядром.
    - Регистрируйте объекты выполнения, принадлежащие каналу (например, клиенты, токены или приложения Bolt), через `openclaw/plugin-sdk/channel-runtime-context`.
    - Не отправляйте уведомления о перенаправлении, принадлежащие плагину, из нативных обработчиков утверждений; теперь ядро отвечает за уведомления о перенаправлении на основе фактических результатов доставки.
    - При передаче `channelRuntime` в `createChannelManager(...)` предоставьте реальный интерфейс `createPluginRuntime().channel`. Частичные заглушки будут отклонены.

    См. `/plugins/sdk-channel-plugins` для ознакомления с текущей структурой возможностей утверждений.

  </Step>

  <Step title="Проверка поведения отката для оболочки Windows">
    Если ваш плагин использует `openclaw/plugin-sdk/windows-spawn`, неразрешённые оболочки Windows `.cmd`/`.bat` теперь завершаются с ошибкой, если вы явно не передадите `allowShellFallback: true`.

    ```typescript
    // До
    const program = applyWindowsSpawnProgramPolicy({ candidate });

    // После
    const program = applyWindowsSpawnProgramPolicy({
      candidate,
      // Устанавливайте это только для доверенных вызывающих, которые намеренно
      // принимают откат через оболочку.
      allowShellFallback: true,
    });
    ```

    Если ваш вызывающий не полагается намеренно на откат через оболочку, не устанавливайте `allowShellFallback` и обрабатывайте возникающую ошибку.

  </Step>

  <Step title="Поиск устаревших импортов">
    Найдите в своём плагине импорты из устаревших источников:

    ```bash
    grep -r "plugin-sdk/compat" my-plugin/
    grep -r "openclaw/extension-api" my-plugin/
    ```

  </Step>

  <Step title="Замена на целенаправленные импорты">
    Каждый экспорт из старого источника соответствует определённому современному пути импорта:

    ```typescript
    // До (устаревший слой обратной совместимости)
    import {
      createChannelReplyPipeline,
      createPluginRuntimeStore,
      resolveControlCommandGate,
    } from "openclaw/plugin-sdk/compat";

    // После (современные целенаправленные импорты)
    import { createChannelReplyPipeline } from "openclaw/plugin-sdk/channel-reply-pipeline";
    import { createPluginRuntimeStore } from "openclaw/plugin-sdk/runtime-store";
    import { resolveControlCommandGate } from "openclaw/plugin-sdk/command-auth";
    ```

    Для вспомогательных инструментов на стороне хоста используйте внедряемый рантайм плагина вместо прямого импорта:

    ```typescript
    // До (устаревший мост extension-api)
    import { runEmbeddedPiAgent } from "openclaw/extension-api";
    const result = await runEmbeddedPiAgent({ sessionId, prompt });

    // После (внедряемый рантайм)
    const result = await api.runtime.agent.runEmbeddedPiAgent({ sessionId, prompt });
    ```

    Тот же шаблон применяется к другим устаревшим вспомогательным инструментам моста:

    | Старый импорт | Современный эквивалент |
    | --- | --- |
    | `resolveAgentDir` | `api.runtime.agent.resolveAgentDir` |
    | `resolveAgentWorkspaceDir` | `api.runtime.agent.resolveAgentWorkspaceDir` |
