# Суб-агенты

Суб-агенты — это фоновые сеансы работы агентов, запускаемые из существующего сеанса работы агента. Они выполняются в собственном сеансе (`agent:<agentId>:subagent:<uuid>`) и по завершении **передают** свой результат обратно в канал чата-запросчика. Каждый сеанс работы суб-агента отслеживается как [фоновая задача](/automation/tasks).

## Косая черта (slash-команда)

Используйте `/subagents`, чтобы проверять или управлять сеансами работы суб-агентов для **текущего сеанса**:

- `/subagents list`
- `/subagents kill <id|#|all>`
- `/subagents log <id|#> [limit] [tools]`
- `/subagents info <id|#>`
- `/subagents send <id|#> <message>`
- `/subagents steer <id|#> <message>`
- `/subagents spawn <agentId> <task> [--model <model>] [--thinking <level>]`

Управление привязкой к потоку:

Эти команды работают на каналах, поддерживающих постоянные привязки к потоку. См. раздел **Каналы, поддерживающие потоки** ниже.

- `/focus <subagent-label|session-key|session-id|session-label>`
- `/unfocus`
- `/agents`
- `/session idle <duration|off>`
- `/session max-age <duration|off>`

`/subagents info` отображает метаданные сеанса (статус, временные метки, идентификатор сеанса, путь к транскрипту, очистку).
Используйте `sessions_history` для ограниченного представления с фильтрацией по безопасности; проверяйте путь к транскрипту на диске, если вам нужен полный необработанный транскрипт.

### Поведение при запуске

`/subagents spawn` запускает фоновый суб-агент как пользовательскую команду, а не как внутреннее реле, и отправляет одно окончательное уведомление о завершении обратно в канал чата-запросчика по завершении сеанса.

- Команда запуска не блокирует выполнение: она немедленно возвращает идентификатор сеанса.
- По завершении суб-агент передаёт сводное сообщение/результат обратно в канал чата-запросчика.
- Завершение осуществляется по принципу push-уведомлений. После запуска не опрашивайте `/subagents list`, `sessions_list` или `sessions_history` в цикле только для того, чтобы дождаться завершения; проверяйте статус только по требованию для отладки или вмешательства.
- По завершении OpenClaw по возможности закрывает отслеживаемые вкладки браузера/процессы, открытые в рамках сеанса работы суб-агента, прежде чем продолжить процесс очистки после передачи результата.
- Для ручного запуска доставка устойчива:
  - OpenClaw сначала пытается выполнить прямую доставку через `agent` с устойчивым ключом идемпотентности.
  - Если прямая доставка не удаётся, выполняется переход на маршрутизацию через очередь.
  - Если маршрутизация через очередь по-прежнему недоступна, передача результата повторяется с короткой экспоненциальной задержкой перед окончательным отказом.
- Доставка завершения сохраняет разрешённый маршрут запросчика:
  - маршруты завершения, привязанные к потоку или разговору, имеют приоритет при наличии
  - если источник завершения предоставляет только канал, OpenClaw заполняет отсутствующий целевой/учётный параметр из разрешённого маршрута сеанса запросчика (`lastChannel` / `lastTo` / `lastAccountId`), чтобы прямая доставка по-прежнему работала
- Передача завершения в сеанс запросчика представляет собой сгенерированный во время выполнения внутренний контекст (не текст, созданный пользователем) и включает:
  - `Result` (последний видимый текст ответа `assistant`, в противном случае — очищенный последний текст инструмента/результата инструмента)
  - `Status` (`completed successfully` / `failed` / `timed out` / `unknown`)
  - компактные статистические данные о времени выполнения/токенах
  - инструкцию по доставке, сообщающую агенту-запросчику переписать ответ обычным голосом помощника (а не пересылать необработанные внутренние метаданные)
- Параметры `--model` и `--thinking` переопределяют значения по умолчанию для этого конкретного сеанса.
- Используйте `info`/`log`, чтобы проверить детали и вывод после завершения.
- `/subagents spawn` работает в однократном режиме (`mode: "run"`). Для постоянных сеансов, привязанных к потоку, используйте `sessions_spawn` с `thread: true` и `mode: "session"`.
- Для сеансов ACP-обвязки (Codex, Claude Code, Gemini CLI) используйте `sessions_spawn` с `runtime: "acp"` и см. [ACP-агенты](/tools/acp-agents).

Основные цели:

- Параллелизация работы по "исследованиям / длительным задачам / медленным инструментам" без блокировки основного сеанса.
- По умолчанию суб-агенты изолированы (разделение сеансов + опциональная песочница).
- Свести к минимуму вероятность неправильного использования инструментов: по умолчанию суб-агенты **не** получают инструменты сеанса.
- Поддержка настраиваемой глубины вложенности для шаблонов оркестрации.

Примечание о затратах: каждый суб-агент имеет **собственный** контекст и использование токенов. Для тяжёлых или повторяющихся задач установите более дешёвую модель для суб-агентов, оставив основной агент на модели более высокого качества.
Вы можете настроить это через `agents.defaults.subagents.model` или переопределение для каждого агента.

## Инструмент

Используйте `sessions_spawn`:

- Запускает сеанс работы суб-агента (`deliver: false`, глобальный канал: `subagent`)
- Затем выполняет шаг передачи результата и отправляет ответ с результатом в канал чата-запросчика
- Модель по умолчанию: наследуется от вызывающего, если не задано `agents.defaults.subagents.model` (или для каждого агента `agents.list[].subagents.model`); явно указанный `sessions_spawn.model` имеет приоритет.
- Уровень мышления по умолчанию: наследуется от вызывающего, если не задано `agents.defaults.subagents.thinking` (или для каждого агента `agents.list[].subagents.thinking`); явно указанный `sessions_spawn.thinking` имеет приоритет.
- Таймаут сеанса по умолчанию: если `sessions_spawn.runTimeoutSeconds` опущен, OpenClaw использует `agents.defaults.subagents.runTimeoutSeconds`, если задано; в противном случае используется `0` (без таймаута).

Параметры инструмента:

- `task` (обязательно)
- `label?` (необязательно)
- `agentId?` (необязательно; запуск под другим идентификатором агента, если разрешено)
- `model?` (необязательно; переопределяет модель суб-агента; недопустимые значения пропускаются, и суб-агент запускается на модели по умолчанию с предупреждением в результате инструмента)
- `thinking?` (необязательно; переопределяет уровень мышления для сеанса работы суб-агента)
- `runTimeoutSeconds?` (по умолчанию `agents.defaults.subagents.runTimeoutSeconds`, если задано, иначе `0`; при установке сеанс работы суб-агента прерывается через N секунд)
- `thread?` (по умолчанию `false`; при `true` запрашивает привязку потока канала для этого сеанса работы суб-агента)
- `mode?` (`run|session`)
  - по умолчанию `run`
  - если `thread: true` и `mode` не указан, по умолчанию используется `session`
  - `mode: "session"` требует `thread: true`
- `cleanup?` (`delete|keep`, по умолчанию `keep`)
- `sandbox?` (`inherit|require`, по умолчанию `inherit`; `require` отклоняет запуск, если целевая дочерняя среда выполнения не находится в песочнице)
- `sessions_spawn` **не** принимает параметры доставки через канал (`target`, `channel`, `to`, `threadId`, `replyTo`, `transport`). Для доставки используйте `message`/`sessions_send` из запущенного сеанса.

## Сеансы, привязанные к потоку

Когда привязки к потоку включены для канала, суб-агент может оставаться привязанным к потоку, чтобы последующие сообщения пользователя в этом потоке продолжали направляться в тот же сеанс работы суб-агента.

### Каналы, поддерживающие потоки

- Discord (в настоящее время единственный поддерживаемый канал): поддерживает постоянные сеансы суб-агентов, привязанные к потоку (`sessions_spawn` с `thread: true`), ручное управление потоками (`/focus`, `/unfocus`, `/agents`, `/session idle`, `/session max-age`), а также ключи адаптера `channels.discord.threadBindings.enabled`, `channels.discord.threadBindings.idleHours`, `channels.discord.threadBindings.maxAgeHours` и `channels.discord.threadBindings.spawnSubagentSessions`.

Краткий процесс:

1. Запуск с помощью `sessions_spawn` с использованием `thread: true` (и, при необходимости, `mode: "session"`).
2. OpenClaw создаёт или привязывает поток к цели сеанса в активном канале.
3. Ответы и последующие сообщения в этом потоке направляются в привязанный сеанс.
4. Используйте `/session idle`, чтобы проверить/обновить автоматическое отключение при бездействии, и `/session max-age`, чтобы управлять жёстким ограничением.
5. Используйте `/unfocus`, чтобы отсоединиться вручную.

Ручное управление:

- `/focus <target>` привязывает текущий поток (или создаёт его) к цели суб-агента/сеанса.
- `/unfocus` удаляет привязку для текущего привязанного потока.
- `/agents` выводит список активных сеансов и состояния привязки (`thread:<id>` или `unbound`).
- `/session idle` и `/session max-age` работают только для привязанных потоков с фокусом.

Переключатели конфигурации:

- Глобальное значение по умолчанию: `session.threadBindings.enabled`, `session.threadBindings.idleHours`, `session.threadBindings.maxAgeHours`
- Переопределение для канала и ключи автоматической привязки создаются для конкретного адаптера. См. раздел **Каналы, поддерживающие потоки** выше.

См. [Справочник по конфигурации](/gateway/configuration-reference) и [Slash-команды](/tools/slash-commands) для получения актуальных сведений об адаптере.

Список разрешений:

- `agents.list[].subagents.allowAgents`: список идентификаторов агентов, на которые можно нацелиться через `agentId` (`["*"]` для разрешения любых). По умолчанию: только агент-запросчик.
- `agents.defaults.subagents.allowAgents`: список разрешённых целевых агентов по умолчанию, используемый, когда агент-запросчик не устанавливает собственный `subagents.allowAgents`.
- Защита наследования песочницы: если сеанс запросчика находится в песочнице, `sessions_spawn` отклоняет цели, которые будут выполняться без песочницы.
- `agents.defaults.subagents.requireAgentId` / `agents.list[].subagents.requireAgentId`: при значении `true` блокирует вызовы `sessions_spawn`, в которых отсутствует `agentId` (принудительно выбирает явный профиль). По умолчанию: `false`.

Обнаружение:

- Используйте `agents_list`, чтобы увидеть, какие идентификаторы агентов в настоящее время разрешены для `sessions_spawn`.

Автоархивация:

- Сеансы суб-агентов автоматически архивируются через `agents.defaults.subagents.archiveAfterMinutes` (по умолчанию: 60).
- При архивации используется `sessions.delete` и переименовывается транскрипт в `*.deleted.<timestamp>` (в той же папке).
- `cleanup: "delete"` архивирует сразу после передачи результата (при этом транскрипт сохраняется через переименование).
- Автоархивация выполняется по возможности; ожидающие таймеры теряются при перезапуске шлюза.
- `runTimeoutSeconds` **не** выполняет автоархивацию; он только останавливает сеанс. Сеанс остаётся до автоархивации.
- Автоархивация применяется одинаково к сеансам глубины 1 и глубины 2.
- Очистка браузера отделена от очистки архива: отслеживаемые вкладки браузера/процессы по возможности закрываются по завершении сеанса, даже если запись о сеансе/транскрипте сохраняется.

## Вложенные суб-агенты

По умолчанию суб-агенты не могут запускать собственные суб-агенты (`maxSpawnDepth: 1`). Вы можете включить один уровень вложенности, установив `maxSpawnDepth: 2`, что позволяет использовать **шаблон оркестратора**: основной → суб-агент-оркестратор → рабочий суб-суб-агент.

### Как включить

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // разрешить суб-агентам запускать дочерние элементы (по умолчанию: 1)
        maxChildrenPerAgent: 5, // максимальное количество активных дочерних элементов на сеанс агента (по умолчанию: 5)
        maxConcurrent: 8, // глобальное ограничение параллелизма (по умолчанию: 8)
        runTimeoutSeconds: 900, // таймаут по умолчанию для sessions_spawn, если не указан (0 = без таймаута)
      },
    },
  },
}
```

### Уровни глубины

| Глубина | Форма ключа сеанса | Роль | Может запускать? |
| ----- | -------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0 | `agent:<id>:main` | Основной агент | Всегда |
| 1 | `agent:<id>:subagent:<uuid>` | Суб-агент (оркестратор, если разрешена глубина 2) | Только если `maxSpawnDepth >= 2` |
| 2 | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | Суб-суб-агент (рабочий узел) | Никогда |

### Цепочка передачи результатов

Результаты передаются вверх по цепочке:

1. Рабочий узел глубины 2 завершает работу → передаёт результат своему родителю (оркестратору глубины 1)
2. Оркестратор глубины 1 получает результат, синтезирует результаты, завершает работу → передаёт результат основному агенту
3. Основной агент получает результат и доставляет его пользователю

Каждый уровень видит только результаты от своих непосредственных дочерних элементов.

Рекомендации по эксплуатации:

- Запускайте дочернюю работу один раз и ждите событий завершения вместо создания циклов опроса вокруг `sessions_list`, `sessions_history`, `/subagents list` или команд `exec` sleep.
- Если событие завершения дочернего элемента поступает после того, как вы уже отправили окончательный ответ, правильным последующим действием является точный тихий токен `NO_REPLY` / `no_reply`.

### Политика инструментов по глубине

- Роль и область управления записываются в метаданные сеанса во время запуска. Это предотвращает случайное восстановление привилегий оркестратора для плоских или восстановленных ключей сеанса.
- **Глубина 1 (оркестратор, когда `maxSpawnDepth >= 2`)**: получает `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history`, чтобы управлять своими дочерними элементами. Другие инструменты сеанса/системы остаются запрещёнными.
- **Глубина 1 (листовой, когда `maxSpawnDepth == 1`)**: нет инструментов сеанса (текущее поведение по умолчанию).
- **Глубина 2 (рабочий узел)**: нет инструментов сеанса — `sessions_spawn` всегда запрещён на глубине 2. Не может запускать дальнейших дочерних элементов.

### Лимит запуска для каждого агента

Каждый сеанс агента (на любой глубине) может иметь не более `maxChildrenPerAgent` (по умолчанию: 5) активных дочерних элементов одновременно. Это предотвращает неконтролируемое разветвление от одного оркестратора.

### Каскадное прекращение

Прекращение работы оркестратора глубины 1 автоматически останавливает всех его дочерних элементов глубины 2:

- `/stop` в основном чате останавливает все агенты глубины 1 и каскадно распространяется на их дочерние элементы глубины 2.
- `/subagents kill <id>` останавливает конкретный суб-агент и каскадно распространяется на его дочерние элементы.
- `/subagents kill all` останавливает все суб-агенты для запросчика и каскадно распространяется.

## Аутентификация

Аутентификация суб-агента определяется по **идентификатору агента**, а не по типу сеанса:

- Ключ сеанса суб-агента — `agent:<agentId>:subagent:<uuid>`.
- Хранилище аутентификации загружается из `agentDir` этого агента.
- Профили аутентификации основного агента объединяются в качестве **резервного варианта**; профили агентов переопределяют основные профили в случае конфликтов.

Примечание: объединение является аддитивным, поэтому основные профили всегда доступны в качестве резервных. Полностью изолированная аутентификация для каждого агента пока не поддерживается.

## Передача результата (Announce)

Суб-агенты передают результаты через шаг передачи результата:

- Шаг передачи результата выполняется внутри сеанса суб-агента (не в сеансе запросчика).
- Если суб-агент отвечает точно `ANNOUNCE_SKIP`, ничего не публикуется.
- Если последний текст помощника — это точный тихий токен `NO_REPLY` / `no_reply`, вывод передачи результата подавляется, даже если ранее существовал видимый прогресс.
- В противном случае доставка зависит от глубины запросчика:
  - сеансы запросчика верхнего уровня используют последующий вызов `agent` с внешней доставкой (`deliver=true`)
  - вложенные сеансы суб-агентов-запросчиков получают внутреннюю последующую инъекцию (`deliver=false`), чтобы оркестратор мог синтезировать результаты дочерних элементов в сеансе
  - если вложенный сеанс суб-агента-запросчика отсутствует, OpenClaw переходит к запросчику этого сеанса, если доступно
- Для сеансов запросчика верхнего уровня прямая доставка в режиме завершения сначала разрешает любой привязанный маршрут разговора/потока и переопределение хука, затем заполняет отсутствующие поля целевого канала из сохранённого маршрута сеанса запросчика. Это сохраняет завершения в правильном чате/теме, даже когда источник завершения идентифицирует только канал.
- Агрегация результатов дочерних элементов ограничена текущим сеансом запросчика при построении вложенных результатов завершения, предотвращая утечку устаревших выходных данных дочерних элементов из предыдущего сеанса в текущую передачу результата.
- Ответы передачи результата сохраняют маршрутизацию по потоку/теме, если она доступна на адаптерах каналов.
- Контекст передачи результата нормализуется до стабильного внутреннего блока событий:
  - источник (`subagent` или `cron`)
  - ключ/идентификатор дочернего сеанса
  - тип передачи результата + метка задачи
  - строка статуса, полученная из результата выполнения во время выполнения (`success`, `error`, `timeout` или `unknown`)
