# حلقة الوكيل (OpenClaw)

الحلقة الوكيلة هي التشغيل الكامل "الحقيقي" للوكيل: الاستقبال → تجميع السياق → استدلال النموذج →
تنفيذ الأدوات → بث الردود → الاستمرارية. وهي المسار المرجعي الذي يحول الرسالة
إلى إجراءات ورد نهائي، مع الحفاظ على اتساق حالة الجلسة.

في OpenClaw، تكون الحلقة تشغيلًا واحدًا ومتسلسلًا لكل جلسة، ويصدر أحداث دورة الحياة وأحداث التدفق
أثناء تفكير النموذج، واستدعائه للأدوات، وبثه للمخرجات. يشرح هذا المستند كيفية توصيل هذه الحلقة الفعلية من البداية إلى النهاية.

## نقاط الدخول

- Gateway RPC: ‏`agent` و `agent.wait`.
- CLI: الأمر `agent`.

## كيف يعمل (نظرة عامة)

1. يتحقق RPC ‏`agent` من صحة المعلمات، ويحل الجلسة (`sessionKey/sessionId`)، ويحفظ بيانات تعريف الجلسة، ويعيد `{ runId, acceptedAt }` فورًا.
2. يشغّل `agentCommand` الوكيل:
   - يحل القيم الافتراضية للنموذج و`thinking`/`verbose`
   - يحمّل لقطة Skills
   - يستدعي `runEmbeddedPiAgent` (بيئة تشغيل pi-agent-core)
   - يصدر **نهاية/خطأ دورة الحياة** إذا لم تصدر الحلقة المضمّنة أيًا منهما
3. `runEmbeddedPiAgent`:
   - يُسلسل عمليات التشغيل عبر طوابير لكل جلسة وطوابير عامة
   - يحل النموذج وملف المصادقة الشخصي ويبني جلسة Pi
   - يشترك في أحداث Pi ويبث فروق المساعد/الأداة
   - يفرض مهلة زمنية -> يجهض التشغيل إذا تم تجاوزها
   - يعيد الحمولات الوصفية وبيانات الاستخدام
4. يقوم `subscribeEmbeddedPiSession` بربط أحداث pi-agent-core بتدفق `agent` في OpenClaw:
   - أحداث الأداة => `stream: "tool"`
   - فروق المساعد => `stream: "assistant"`
   - أحداث دورة الحياة => `stream: "lifecycle"` (`phase: "start" | "end" | "error"`)
5. يستخدم `agent.wait` الدالة `waitForAgentRun`:
   - ينتظر **نهاية/خطأ دورة الحياة** لـ `runId`
   - يعيد `{ status: ok|error|timeout, startedAt, endedAt, error? }`

## الاصطفاف + التزامن

- تُسلسل عمليات التشغيل لكل مفتاح جلسة (مسار الجلسة) ويمكن اختياريًا تمريرها عبر مسار عام.
- يمنع ذلك تعارضات الأدوات/الجلسات ويحافظ على اتساق سجل الجلسة.
- يمكن لقنوات المراسلة اختيار أوضاع الاصطفاف (collect/steer/followup) التي تغذي نظام المسارات هذا.
  راجع [طابور الأوامر](/ar/concepts/queue).

## إعداد الجلسة + مساحة العمل

- تُحل مساحة العمل وتُنشأ؛ وقد تعيد عمليات التشغيل المعزولة توجيهها إلى جذر مساحة عمل معزول.
- يتم تحميل Skills (أو إعادة استخدامها من لقطة) وحقنها في البيئة والموجّه.
- تُحل ملفات bootstrap/context وتُحقن في تقرير موجّه النظام.
- يُكتسب قفل كتابة للجلسة؛ ويتم فتح `SessionManager` وتجهيزه قبل البث.

## تجميع الموجّه + موجّه النظام

- يُبنى موجّه النظام من الموجّه الأساسي لـ OpenClaw، وموجّه Skills، وسياق bootstrap، وعمليات التجاوز الخاصة بكل تشغيل.
- تُفرض حدود خاصة بالنموذج ورموز الحجز الخاصة بالضغط.
- راجع [موجّه النظام](/ar/concepts/system-prompt) لمعرفة ما يراه النموذج.

## نقاط الربط (حيث يمكنك الاعتراض)

يحتوي OpenClaw على نظامي hooks:

- **الخطافات الداخلية** (خطافات Gateway): سكربتات قائمة على الأحداث للأوامر وأحداث دورة الحياة.
- **خطافات الإضافات**: نقاط توسعة داخل دورة حياة الوكيل/الأداة ومسار Gateway.

### الخطافات الداخلية (خطافات Gateway)

- **`agent:bootstrap`**: يعمل أثناء بناء ملفات bootstrap قبل الانتهاء من موجّه النظام.
  استخدمه لإضافة/إزالة ملفات سياق bootstrap.
- **خطافات الأوامر**: ‏`/new` و`/reset` و`/stop` وأحداث أوامر أخرى (راجع مستند Hooks).

راجع [Hooks](/ar/automation/hooks) للإعداد والأمثلة.

### خطافات الإضافات (دورة حياة الوكيل + Gateway)

تعمل هذه داخل حلقة الوكيل أو مسار Gateway:

- **`before_model_resolve`**: يعمل قبل الجلسة (من دون `messages`) لتجاوز المزود/النموذج بشكل حتمي قبل حل النموذج.
- **`before_prompt_build`**: يعمل بعد تحميل الجلسة (مع `messages`) لحقن `prependContext` أو `systemPrompt` أو `prependSystemContext` أو `appendSystemContext` قبل إرسال الموجّه. استخدم `prependContext` للنص الديناميكي الخاص بكل دور، واستخدم حقول سياق النظام للإرشادات الثابتة التي ينبغي أن تبقى في مساحة موجّه النظام.
- **`before_agent_start`**: خطاف توافق قديم قد يعمل في أي من المرحلتين؛ ويفضل استخدام الخطافات الصريحة أعلاه.
- **`before_agent_reply`**: يعمل بعد الإجراءات المضمنة وقبل استدعاء LLM، ما يتيح للإضافة الاستحواذ على الدور وإرجاع رد تركيبي أو إسكات الدور بالكامل.
- **`agent_end`**: يفحص قائمة الرسائل النهائية وبيانات تعريف التشغيل بعد الاكتمال.
- **`before_compaction` / `after_compaction`**: يراقب أو يعلّق دورات الضغط.
- **`before_tool_call` / `after_tool_call`**: يعترض معلمات/نتائج الأدوات.
- **`before_install`**: يفحص نتائج الفحص المضمنة ويمكنه اختياريًا حظر تثبيت Skill أو إضافة.
- **`tool_result_persist`**: يحوّل نتائج الأدوات بشكل متزامن قبل كتابتها في سجل الجلسة.
- **`message_received` / `message_sending` / `message_sent`**: خطافات الرسائل الواردة والصادرة.
- **`session_start` / `session_end`**: حدود دورة حياة الجلسة.
- **`gateway_start` / `gateway_stop`**: أحداث دورة حياة Gateway.

قواعد قرارات الخطافات لحواجز الخروج/الأدوات:

- `before_tool_call`: القيمة `{ block: true }` نهائية وتوقف المعالجات ذات الأولوية الأدنى.
- `before_tool_call`: القيمة `{ block: false }` بلا تأثير ولا تلغي حظرًا سابقًا.
- `before_install`: القيمة `{ block: true }` نهائية وتوقف المعالجات ذات الأولوية الأدنى.
- `before_install`: القيمة `{ block: false }` بلا تأثير ولا تلغي حظرًا سابقًا.
- `message_sending`: القيمة `{ cancel: true }` نهائية وتوقف المعالجات ذات الأولوية الأدنى.
- `message_sending`: القيمة `{ cancel: false }` بلا تأثير ولا تلغي إلغاءً سابقًا.

راجع [خطافات الإضافات](/ar/plugins/architecture#provider-runtime-hooks) لمعرفة واجهة برمجة الخطافات وتفاصيل التسجيل.

## البث + الردود الجزئية

- تُبث فروق المساعد من pi-agent-core وتُصدر كأحداث `assistant`.
- يمكن لبث الكتل إصدار ردود جزئية إما عند `text_end` أو `message_end`.
- يمكن إصدار بث الاستدلال كتدفق منفصل أو كردود كتلية.
- راجع [البث](/ar/concepts/streaming) لمعرفة سلوك التقطيع وردود الكتل.

## تنفيذ الأدوات + أدوات المراسلة

- تُصدر أحداث بدء/تحديث/انتهاء الأداة على تدفق `tool`.
- تُنظف نتائج الأدوات من حيث الحجم وحمولات الصور قبل التسجيل/الإصدار.
- يتم تتبع عمليات الإرسال الخاصة بأدوات المراسلة لمنع تأكيدات المساعد المكررة.

## تشكيل الرد + المنع

- تُجمع الحمولات النهائية من:
  - نص المساعد (والاستدلال الاختياري)
  - ملخصات الأدوات المضمنة (عند تفعيل verbose وعند السماح بذلك)
  - نص خطأ المساعد عند حدوث خطأ في النموذج
- تتم تصفية الرمز الصامت الدقيق `NO_REPLY` / `no_reply` من
  الحمولات الصادرة.
- تُزال تكرارات أدوات المراسلة من قائمة الحمولات النهائية.
- إذا لم تبق أي حمولات قابلة للعرض وحدث خطأ في أداة، فسيُصدر رد احتياطي لخطأ الأداة
  (ما لم تكن أداة مراسلة قد أرسلت بالفعل ردًا مرئيًا للمستخدم).

## الضغط + إعادة المحاولة

- يصدر الضغط التلقائي أحداث تدفق `compaction` ويمكن أن يؤدي إلى إعادة المحاولة.
- عند إعادة المحاولة، تتم إعادة تعيين المخازن المؤقتة الموجودة في الذاكرة وملخصات الأدوات لتجنب المخرجات المكررة.
- راجع [الضغط](/ar/concepts/compaction) لمعرفة مسار الضغط.

## تدفقات الأحداث (حاليًا)

- `lifecycle`: يصدره `subscribeEmbeddedPiSession` (وكحل احتياطي بواسطة `agentCommand`)
- `assistant`: فروق مبثوثة من pi-agent-core
- `tool`: أحداث أدوات مبثوثة من pi-agent-core

## التعامل مع قناة الدردشة

- تُخزَّن فروق المساعد مؤقتًا في رسائل دردشة `delta`.
- يُصدر `final` للدردشة عند **نهاية/خطأ دورة الحياة**.

## المهلات الزمنية

- القيمة الافتراضية لـ `agent.wait`: ‏30 ثانية (للاِنتظار فقط). تتجاوزها المعلمة `timeoutMs`.
- وقت تشغيل الوكيل: القيمة الافتراضية لـ `agents.defaults.timeoutSeconds` هي 172800 ثانية (48 ساعة)؛ ويتم فرضها في مؤقت الإجهاض ضمن `runEmbeddedPiAgent`.
- مهلة خمول LLM: تؤدي `agents.defaults.llm.idleTimeoutSeconds` إلى إجهاض طلب النموذج عندما لا تصل أي أجزاء استجابة قبل انتهاء نافذة الخمول. اضبطها صراحةً للنماذج المحلية البطيئة أو لمزودي الاستدلال/استدعاء الأدوات؛ واضبطها على 0 لتعطيلها. إذا لم يتم ضبطها، يستخدم OpenClaw القيمة `agents.defaults.timeoutSeconds` عند تكوينها، وإلا فيستخدم 60 ثانية. تعطل عمليات التشغيل المشغَّلة عبر Cron التي لا تحتوي على مهلة صريحة لـ LLM أو الوكيل مراقب الخمول وتعتمد على المهلة الخارجية لـ Cron.

## أين يمكن أن تنتهي الأمور مبكرًا

- مهلة الوكيل (إجهاض)
- `AbortSignal` (إلغاء)
- انقطاع Gateway أو مهلة RPC
- مهلة `agent.wait` (للاِنتظار فقط، ولا توقف الوكيل)
