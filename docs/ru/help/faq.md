# FAQ

Быстрые ответы и углубленное устранение неполадок для реальных сценариев (локальная разработка, VPS, мульти-агентные системы, OAuth/API ключи, отказоустойчивость моделей). Для диагностики во время выполнения см. [Устранение неполадок](https://www.google.com/search?q=/gateway/troubleshooting). Полный справочник по конфигурации доступен в разделе [Конфигурация](https://www.google.com/search?q=/gateway/configuration).

## Первые 60 секунд, если что-то сломалось

1.  **Быстрая проверка статуса**

    ```bash
    openclaw status
    ```

    Краткая локальная сводка: ОС + обновления, доступность шлюза/сервисов, агенты/сессии, конфигурация провайдеров + проблемы во время выполнения (если шлюз доступен).

2.  **Отчет для вставки (безопасно для публикации)**

    ```bash
    openclaw status --all
    ```

    Диагностика в режиме "только чтение" с хвостом логов (токены скрыты).

3.  **Состояние демона + портов**

    ```bash
    openclaw gateway status
    ```

    Показывает состояние супервизора в рантайме против доступности RPC, целевой URL проверки и то, какой конфиг, скорее всего, использовал сервис.

4.  **Глубокая проверка**

    ```bash
    openclaw status --deep
    ```

    Запускает живую проверку работоспособности шлюза, включая проверку каналов (требуется доступный шлюз). См. [Здоровье](https://www.google.com/search?q=/gateway/health).

5.  **Просмотр последних логов**

    ```bash
    openclaw logs --follow
    ```

    Если RPC недоступен, используйте:

    ```bash
    tail -f "$(ls -t /tmp/openclaw/openclaw-*.log | head -1)"
    ```

    Логи файлов отделены от логов сервиса; см. [Логирование](https://www.google.com/search?q=/logging) и [Устранение неполадок](https://www.google.com/search?q=/gateway/troubleshooting).

6.  **Запуск "доктора" (исправление)**

    ```bash
    openclaw doctor
    ```

    Исправляет/мигрирует конфигурацию и состояние + запускает проверки здоровья. См. [Doctor](https://www.google.com/search?q=/gateway/doctor).

7.  **Снапшот шлюза**

    ```bash
    openclaw health --json
    openclaw health --verbose   # показывает целевой URL + путь к конфигу при ошибках
    ```

    Запрашивает у запущенного шлюза полный снапшот (только через WS). См. [Здоровье](https://www.google.com/search?q=/gateway/health).

<Accordion title="Хартбит постоянно пропускается. Что значат причины пропуска (skip reasons)?"\>
Распространенные причины пропуска хартбита:

```
- `quiet-hours`: вне настроенного окна активных часов.
- `empty-heartbeat-file`: `HEARTBEAT.md` существует, но содержит только пустые строки или заголовки-заглушки.
- `no-tasks-due`: режим задач в `HEARTBEAT.md` активен, но интервалы выполнения задач еще не наступили.
- `alerts-disabled`: вся видимость хартбита отключена (параметры `showOk`, `showAlerts` и `useIndicator` выключены).

В режиме задач временные метки обновляются только после завершения реального прогона хартбита. Пропущенные прогоны не помечают задачи как выполненные.

Документация: [Heartbeat](/gateway/heartbeat), [Автоматизация и задачи](/automation).
```

<Accordion title="Как авторизовать дашборд на localhost против удаленного доступа?"\>
**Localhost (на той же машине):**

```
- Откройте `http://127.0.0.1:18789/`.
- Если запрашивается авторизация по общему секрету (shared-secret), вставьте настроенный токен или пароль в настройки Control UI.
- Источник токена: `gateway.auth.token` (или `OPENCLAW_GATEWAY_TOKEN`).
- Источник пароля: `gateway.auth.password` (или `OPENCLAW_GATEWAY_PASSWORD`).
- Если общий секрет еще не настроен, создайте токен с помощью `openclaw doctor --generate-gateway-token`.

**Не на localhost:**

- **Tailscale Serve** (рекомендуется): оставьте привязку к loopback, запустите `openclaw gateway --tailscale serve`, откройте `https://<magicdns>/`. Если `gateway.auth.allowTailscale` имеет значение `true`, заголовки идентификации удовлетворяют авторизации Control UI/WebSocket (не нужно вставлять секрет, подразумевается доверенный хост шлюза); HTTP API все равно требуют авторизации по общему секрету, если вы намеренно не используете `none` для приватного ингресса или авторизацию через доверенный прокси (trusted-proxy).
- **Привязка к Tailnet**: запустите `openclaw gateway --bind tailnet --token "<token>"` (или настройте авторизацию по паролю), откройте `http://<tailscale-ip>:18789/`, затем вставьте соответствующий общий секрет в настройки дашборда.
- **Identity-aware reverse proxy**: держите Gateway за доверенным прокси (не loopback), настройте `gateway.auth.mode: "trusted-proxy"`, затем откройте URL прокси.
- **SSH туннель**: `ssh -N -L 18789:127.0.0.1:18789 user@host`, затем откройте `http://127.0.0.1:18789/`. Авторизация по общему секрету по-прежнему применяется через туннель.

См. [Дашборд](/web/dashboard) и [Веб-интерфейсы](/web) для подробностей о режимах привязки и авторизации.
```

<Accordion title="Почему существует два конфига для подтверждения выполнения (exec approval) в чате?"\>
Они управляют разными уровнями:

```
- `approvals.exec`: пересылает запросы на подтверждение в чаты.
- `channels.<channel>.execApprovals`: заставляет конкретный канал работать как нативный клиент для подтверждения выполнения.

Политика выполнения на хосте по-прежнему остается реальным барьером. Конфигурация чата управляет только тем, где появляются запросы и как на них можно ответить.

В большинстве случаев вам **не** нужны оба:

- Если чат уже поддерживает команды и ответы, подтверждение через тот же чат работает по общему пути через `/approve`.
- Если поддерживаемый нативный канал может безопасно определить подтверждающих, OpenClaw теперь автоматически включает нативные подтверждения сначала в DM, если `channels.<channel>.execApprovals.enabled` не задан или равен `"auto"`.
- При наличии нативных карточек/кнопок подтверждения этот интерфейс является основным; агент должен включать ручную команду `/approve` только если результат инструмента говорит о недоступности чат-подтверждений.
- Используйте `approvals.exec` только когда запросы должны дублироваться в другие чаты или специальные операционные комнаты.

Кратко: пересылка нужна для маршрутизации, конфигурация нативного клиента — для более богатого UX в конкретном канале.
См. [Подтверждение выполнения (Exec Approvals)](/tools/exec-approvals).
```

````
openclaw logs --follow
```

3. Если все еще висит, запустите:

```bash
openclaw doctor
```

Если шлюз удаленный, убедитесь, что туннель или соединение Tailscale активно и UI направлен на правильный шлюз. См. [Удаленный доступ](/gateway/remote).
````

<Accordion title="Можно ли перенести настройки на новую машину (Mac mini) без повторного онбординга?"\>
Да. Скопируйте **директорию состояния** и **рабочее пространство**, затем один раз запустите Doctor. Это сохранит вашего бота "в точности таким же" (память, история сессий, авторизация, состояние каналов), если вы скопируете **оба** расположения:


<Accordion title="Разница между stable и beta"\>
**Stable** и **beta** — это **теги npm (dist-tags)**, а не разные ветки кода:

```
- `latest` = стабильная версия.
- `beta` = ранняя сборка для тестирования.

Обычно стабильный релиз сначала попадает в `beta`, а затем переносится в `latest`. Поэтому после продвижения версии оба тега могут указывать на один и тот же билд.
```

</Accordion\>

<Accordion title="При установке на Windows пишет git not found или openclaw не распознано"\>
Две частые проблемы Windows:

```
**1) Ошибка npm spawn git / git not found**
- Установите **Git for Windows** и убедитесь, что `git` добавлен в PATH.

- Папка бинарных файлов npm не в PATH. Проверьте путь: `npm config get prefix`.

Для максимальной стабильности на Windows рекомендуется использовать **WSL2**.
```

</Accordion\>

<Accordion title="Вывод команд в Windows показывает искаженный текст (иероглифы)"\>
Это обычно связано с несоответствием кодировок в нативной оболочке Windows.
Быстрое решение в PowerShell:

````
```powershell
chcp 65001
[Console]::InputEncoding = [System.Text.UTF8Encoding]::new($false)
[Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false)
$OutputEncoding = [System.Text.UTF8Encoding]::new($false)
```
Затем перезапустите Gateway.
````

</Accordion\>

</Accordion\>

<Accordion title="Как установить OpenClaw на Linux / VPS?"\>
Любой Linux VPS подойдет. Установите на сервер, затем используйте SSH или Tailscale для доступа к шлюзу.
Инструкции: [Linux](https://www.google.com/search?q=/platforms/linux), [VPS hosting](https://www.google.com/search?q=/vps).

</Accordion\>

<Accordion title="Можно ли попросить OpenClaw обновить самого себя?"\>
Коротко: **возможно, но не рекомендуется**. Процесс обновления может перезапустить шлюз (что оборвет сессию) и потребовать подтверждений. Надежнее запускать обновления из терминала.

```
Команды: `openclaw update`, `openclaw update --channel stable|beta|dev`.
```

</Accordion\>

<Accordion title="Что на самом деле делает онбординг?"\>
`openclaw onboard` — это рекомендуемый путь настройки. Он проводит вас через:
\- Настройку моделей и авторизации (OAuth, API ключи).
\- Выбор расположения рабочего пространства.
\- Настройки шлюза (порт, Tailscale).
\- Подключение каналов (WhatsApp, Telegram, Discord и др.).
\- Установку демона (автозапуск).
\- Проверки здоровья и выбор скиллов.

</Accordion\>

<Accordion title="Нужна ли подписка Claude или OpenAI для запуска?"\>
Нет. Можно использовать **API-ключи** (с оплатой по факту) или **локальные модели**, чтобы данные не покидали устройство.
OpenClaw также поддерживает повторное использование авторизации Claude CLI.

</Accordion\>

<Accordion title="Почему я вижу ошибку HTTP 429 rate_limit_error от Anthropic?"\>
Это означает, что ваша **квота или лимит запросов** исчерпаны. Подождите сброса окна или повысьте лимиты в консоли провайдера.
Совет: настройте **резервную модель (fallback model)**, чтобы OpenClaw мог продолжать отвечать через другого провайдера.

</Accordion\>

<Accordion title="Поддерживается ли AWS Bedrock?"\>
Да. В OpenClaw есть встроенный провайдер **Amazon Bedrock (Converse)**. При наличии переменных окружения AWS шлюз может автоматически обнаружить доступные модели.

</Accordion\>

<Accordion title="Как работает авторизация Codex?"\>
OpenClaw поддерживает **OpenAI Code (Codex)** через OAuth (вход в ChatGPT). Онбординг может запустить этот процесс и установить модель по умолчанию.

</Accordion\>

<Accordion title="Подходит ли локальная модель для повседневного общения?"\>
Обычно нет. OpenClaw требует большого контекста и высокого уровня безопасности; маленькие модели часто ошибаются. Если необходимо, используйте **самые крупные** доступные локальные сборки (через LM Studio). Маленькие/квантованные модели повышают риск промпт-инъекций.

</Accordion\>

<Accordion title="Нужно ли покупать Mac Mini для установки?"\>
Нет. OpenClaw работает на macOS, Linux и Windows (через WSL2). Mac mini удобен как всегда включенный хост, но VPS или Raspberry Pi тоже подходят.
Mac нужен только для **macOS-специфичных инструментов** (например, iMessage через BlueBubbles) или запуска команд непосредственно в macOS.

</Accordion\>

<Accordion title="Можно ли использовать Bun?"\>
Bun **не рекомендуется**. Замечены ошибки в рантайме, особенно с WhatsApp и Telegram. Используйте **Node.js** для стабильной работы.

</Accordion\>

<Accordion title="Telegram: что писать в allowFrom?"\>
`channels.telegram.allowFrom` — это **числовой ID пользователя Telegram**. Это НЕ имя пользователя (username). Чтобы узнать свой ID, напишите боту и проверьте логи (`openclaw logs --follow`) или используйте сторонних ботов типа `@userinfobot`.

</Accordion\>

<Accordion title="Могут ли несколько человек использовать один номер WhatsApp с разными экземплярами OpenClaw?"\>
Да, через **мульти-агентную маршрутизацию**. Привяжите DM каждого отправителя к отдельному `agentId`, чтобы у каждого было свое рабочее пространство и история сессий. Ответы будут приходить с одного аккаунта WhatsApp.

</Accordion\>

<Accordion title="Разница между установкой из git и из npm?"\>
\- **Git (Hackable):** полный исходный код, можно редактировать, лучше для контрибьюторов.
\- **npm:** глобальная установка CLI, без репозитория, лучше для простого использования.
Вы можете переключаться между ними с помощью `openclaw doctor`.

</Accordion\>
</AccordionGroup\>

## Что такое OpenClaw?

<AccordionGroup\>
<Accordion title="Что такое OpenClaw в одном абзаце?"\>
OpenClaw — это персональный ИИ-ассистент, который вы запускаете на своих устройствах. Он отвечает в мессенджерах, которыми вы уже пользуетесь (WhatsApp, Telegram, Slack и др.), поддерживает голос и живой холст (Canvas). **Gateway** — это постоянно работающий узел управления; ассистент — это конечный продукт.
</Accordion\>

<Accordion title="Ценностное предложение"\>
OpenClaw — это не просто "обертка над Claude". Это **локальная панель управления**, позволяющая запускать мощного ассистента на **вашем собственном оборудовании**, с сохранением сессий, памяти и инструментов, не передавая контроль над рабочими процессами облачным SaaS-провайдерам.

```
Основные моменты:
- **Ваши устройства, ваши данные:** рабочее пространство и история сессий остаются локальными.
- **Реальные каналы:** мессенджеры, а не просто веб-песочница.
- **Независимость от моделей:** поддержка Anthropic, OpenAI, локальных моделей и др. с переключением при сбоях.
- **Мульти-агентность:** разные агенты для разных задач или каналов.
```

</Accordion\>

<Accordion title="Преимущества перед Claude Code для веб-разработки?"\>
OpenClaw — это **координатор и ассистент**, а не замена IDE. Используйте Claude Code для быстрого цикла написания кода в репозитории. Используйте OpenClaw, когда вам нужна долгосрочная память, доступ с разных устройств и оркестрация инструментов (браузер, файлы, расписание).

</Accordion\>
</AccordionGroup\>

## Скиллы и автоматизация

<AccordionGroup\>
<Accordion title="Как настраивать скиллы, не загрязняя репозиторий?"\>
Используйте управляемые переопределения. Поместите свои изменения в `~/.openclaw/skills/&lt;name&gt;/SKILL.md`. OpenClaw ищет скиллы в определенном порядке приоритетности, где локальные переопределения побеждают встроенные.

</Accordion\>

<Accordion title="Бот зависает при тяжелой работе. Как это разгрузить?"\>
Используйте **субагентов** для длительных или параллельных задач. Субагенты работают в собственных сессиях и возвращают результат в основной чат, не блокируя его. Попросите бота "spawn a sub-agent for this task".

</Accordion\>

<Accordion title="Cron или напоминания не срабатывают. Что проверить?"\>
Cron работает внутри процесса Gateway. Если шлюз выключен, задачи не выполнятся.
Проверьте: включен ли cron (`cron.enabled`), не установлена ли переменная `OPENCLAW_SKIP_CRON`, и настройки часового пояса.

</Accordion\>

<Accordion title="Можно ли запускать macOS-скиллы с Linux?"\>
Напрямую — нет. Но есть три варианта:
\- **А:** Запустить Gateway на Mac.
\- **Б:** Использовать macOS-узел (node). Gateway на Linux вызывает инструменты на Mac через WebSocket.
\- **В:** Проксировать бинарники через SSH (для продвинутых).

</Accordion\>

<Accordion title="Как использовать мой существующий профиль Chrome с OpenClaw?"\>
Используйте встроенный профиль браузера `user`, который подключается через Chrome DevTools MCP. Это работает локально на хосте; если шлюз удаленный, используйте узлы (nodes).

</Accordion\>
</AccordionGroup\>

## Песочница и память

<AccordionGroup\>
<Accordion title="Как работает память?"\>
Память OpenClaw — это просто Markdown-файлы в рабочем пространстве агента:
\- Ежедневные заметки в `memory/YYYY-MM-DD.md`.
\- Долгосрочные заметки в `MEMORY.md`.
Бот делает "флеш-память" перед архивацией сессии, чтобы сохранить важные данные.

</Accordion\>

<Accordion title="Память постоянно что-то забывает. Как закрепить знания?"\>
Попросите бота **записать факт в память**. Долгосрочные данные должны быть в `MEMORY.md`. Если он все равно забывает, убедитесь, что при перезапуске используется то же самое рабочее пространство.

</Accordion\>

<Accordion title="Нужен ли API-ключ OpenAI для семантического поиска по памяти?"\>
Только если вы используете эмбеддинги OpenAI. OpenClaw поддерживает **OpenAI, Gemini, Voyage, Mistral, Ollama или локальные** модели эмбеддингов. Для полной локальности установите `memorySearch.provider = &quot;local&quot;`.

</Accordion\>
</AccordionGroup\>

## Где что лежит на диске

<AccordionGroup\>
<Accordion title="Все ли данные сохраняются локально?"\>
OpenClaw сохраняет **состояние локально** (сессии, память, конфиги), но сообщения, которые вы отправляете ИИ-провайдерам или в мессенджеры, проходят через их серверы. Использование локальных моделей сводит передачу данных к минимуму.

</Accordion\>

<Accordion title="Где OpenClaw хранит свои данные?"\>
Все находится в `$OPENCLAW_STATE_DIR` (обычно `~/.openclaw`):
\- `openclaw.json`: основной конфиг.
\- `agents/`: состояние агентов и сессии.
\- `credentials/`: данные каналов (например, сессия WhatsApp).
\- `workspace/`: ваши MD-файлы памяти и скиллы.

</Accordion\>

<Accordion title="Рекомендуемая стратегия бэкапа"\>
Храните ваше **рабочее пространство агента** (workspace) в **приватном** git-репозитории. Это сохранит "разум" вашего ассистента. **Не** коммитьте папку `~/.openclaw`, так как там содержатся секреты и токены.

</Accordion\>
</AccordionGroup\>

## Основы конфигурации

<AccordionGroup\>
<Accordion title="Нужно ли перезагружать шлюз после изменения конфига?"\>
Шлюз поддерживает горячую перезагрузку (`hybrid` режим по умолчанию): безопасные изменения применяются сразу, для критических происходит автоматический перезапуск.

</Accordion\>

<Accordion title="Как включить веб-поиск?"\>
Запустите `openclaw configure --section web`. Поддерживаются Brave, Google Gemini, Perplexity, Tavily и другие. Для работы большинства из них требуется соответствующий API-ключ.

</Accordion\>

<Accordion title="config.apply стер мой конфиг. Как восстановить?"\>
`config.apply` заменяет **весь конфиг целиком**. Если вы отправили неполный объект, остальные части удаляются. Восстановите из бэкапа или используйте `openclaw doctor` для базового восстановления. Для частичных правок используйте `openclaw config set` или `config.patch` через RPC.

</Accordion\>
</AccordionGroup\>

## Удаленные шлюзы и узлы

<AccordionGroup\>
<Accordion title="Как агент может получить доступ к моему компьютеру, если шлюз удаленный?"\>
Подключите ваш компьютер как **узел (node)**. Шлюз на VPS сможет вызывать инструменты `node.*` (экран, камера, запуск команд) на вашем локальном компьютере через WebSocket-соединение.

</Accordion\>

<Accordion title="Могут ли два экземпляра OpenClaw общаться друг с другом?"\>
Да. Самый простой способ — через общий канал чата (Telegram/WhatsApp). Один бот пишет другому, тот отвечает. Также можно использовать CLI-мост для отправки сообщений на другой шлюз.

</Accordion\>
</AccordionGroup\>

## Переменные окружения

<AccordionGroup\>
<Accordion title="Как OpenClaw загружает переменные окружения?"\>
OpenClaw читает переменные из родительского процесса, а также загружает:
\- `.env` из текущей директории.
\- Глобальный `.env` из `~/.openclaw/.env`.
Вы также можете прописать переменные прямо в блоке `env` файла `openclaw.json`.

</Accordion\>
</AccordionGroup\>

## Сессии и мульти-чаты

<AccordionGroup\>
<Accordion title="Как начать новый разговор?"\>
Отправьте `/new` или `/reset` отдельным сообщением. Это создаст новый ID сессии.

</Accordion\>

<Accordion title="Почему бот не отвечает в группе?"\>
Обычно из-за того, что включена фильтрация по упоминаниям (mention gating). Вам нужно либо тегнуть бота, либо добавить группу в белый список в конфиге.

</Accordion\>
</AccordionGroup\>

## Модели: выбор и переключение

<AccordionGroup\>
<Accordion title="Какую модель вы рекомендуете?"\>
Используйте самые мощные модели последнего поколения для задач с использованием инструментов. Для простого чата подойдут более дешевые модели. Помните: слабые модели более уязвимы к промпт-инъекциям.

</Accordion\>

<Accordion title="Как переключать модели на лету?"\>
Используйте команду `/model` в чате:
\- `/model sonnet`
\- `/model gpt`
\- `/model list` (покажет список доступных с номерами)
\- `/model 3` (выбор по номеру)

</Accordion\>

<Accordion title="Можно ли использовать свои модели (Ollama)?"\>
Да. Ollama — самый простой путь. Установите Ollama, скачайте модель и при онбординге выберите Ollama в качестве провайдера.

</Accordion\>
</AccordionGroup\>

## Ошибки "All models failed" и переключение при сбоях

<AccordionGroup\>
<Accordion title="Как работает переключение при сбое (failover)?"\>
Сначала OpenClaw пробует другие профили авторизации того же провайдера. Если не помогает — переключается на следующую модель из списка `fallbacks` в конфиге. Для ошибок типа "rate limit" применяются кулдауны.

</Accordion\>
</AccordionGroup\>

## Безопасность и контроль доступа

<AccordionGroup\>
<Accordion title="Безопасно ли открывать DMs для всех?"\>
По умолчанию в OpenClaw включен **режим сопряжения (pairing)**. Неизвестные отправители получают код сопряжения, и бот не обрабатывает их сообщения, пока вы не одобрите их командой `openclaw pairing approve`. Это минимизирует риски.

</Accordion\>

<Accordion title="Опасны ли промпт-инъекции, если я единственный пользователь?"\>
Да. Если ассистент читает внешний контент (веб-страницы, письма, файлы), в них могут быть инструкции, нацеленные на перехват управления моделью. Всегда используйте песочницу (sandboxing) для инструментов и будьте осторожны при обработке недоверенных данных.

</Accordion\>
</AccordionGroup\>

`stop`, `abort`, `wait`, `interrupt`, `please stop`.
