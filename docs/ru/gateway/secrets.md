# Управление секретами

OpenClaw поддерживает добавление SecretRefs, благодаря чему поддерживаемые учётные данные не нужно хранить в виде обычного текста в конфигурации.

Хранение в виде обычного текста по-прежнему поддерживается. Использование SecretRefs — опционально для каждого учётного данных.

## Цели и модель выполнения

Секреты преобразуются в снимок в памяти во время выполнения.

- Разрешение выполняется активно во время активации, а не лениво при запросах.
- При запуске происходит быстрый сбой, если эффективно активный SecretRef не может быть разрешён.
- При перезагрузке используется атомарная замена: либо полный успех, либо сохранение последнего известного рабочего снимка.
- Нарушения политики SecretRef (например, профили аутентификации в режиме OAuth в сочетании с вводом SecretRef) приводят к сбою активации до замены во время выполнения.
- Запросы во время выполнения считывают данные только из активного снимка в памяти.
- После первой успешной активации/загрузки конфигурации пути кода во время выполнения продолжают считывать этот активный снимок в памяти до тех пор, пока успешная перезагрузка не заменит его.
- Пути исходящей доставки также считывают данные из этого активного снимка (например, доставка ответов/потоков в Discord и отправка действий в Telegram); они не выполняют повторное разрешение SecretRefs при каждой отправке.

Это позволяет исключить сбои поставщика секретов из критически важных путей запросов.

## Фильтрация активной поверхности

SecretRefs проверяются только на эффективно активных поверхностях.

- Включённые поверхности: неразрешённые ссылки блокируют запуск/перезагрузку.
- Неактивные поверхности: неразрешённые ссылки не блокируют запуск/перезагрузку.
- Для неактивных ссылок выдаются некритические диагностические сообщения с кодом `SECRETS_REF_IGNORED_INACTIVE_SURFACE`.

Примеры неактивных поверхностей:

- Отключённые записи каналов/учётных записей.
- Учётные данные верхнего уровня канала, которые не наследуются ни одной включённой учётной записью.
- Отключённые поверхности инструментов/функций.
- Ключи поставщика веб-поиска, которые не выбраны параметром `tools.web.search.provider`.
  В автоматическом режиме (если поставщик не задан) ключи проверяются по приоритету для автоматического определения поставщика до тех пор, пока один из них не будет разрешён.
  После выбора ключи поставщиков, которые не были выбраны, считаются неактивными до момента их выбора.
- Материалы аутентификации SSH для песочницы (`agents.defaults.sandbox.ssh.identityData`, `certificateData`, `knownHostsData`, а также переопределения для каждого агента) активны только тогда, когда эффективный бэкенд песочницы — `ssh` для агента по умолчанию или включённого агента.
- SecretRefs `gateway.remote.token` / `gateway.remote.password` активны, если выполняется одно из следующих условий:
  - `gateway.mode=remote`
  - Настроен `gateway.remote.url`
  - `gateway.tailscale.mode` имеет значение `serve` или `funnel`
  - В локальном режиме без этих удалённых поверхностей:
    - `gateway.remote.token` активен, когда аутентификация по токену может быть использована и не настроен токен env/auth.
    - `gateway.remote.password` активен только тогда, когда аутентификация по паролю может быть использована и не настроен пароль env/auth.
- SecretRef `gateway.auth.token` неактивен для разрешения аутентификации при запуске, если задан `OPENCLAW_GATEWAY_TOKEN`, поскольку ввод токена из окружения имеет приоритет для этого времени выполнения.

## Диагностика поверхности аутентификации шлюза

Когда SecretRef настроен для `gateway.auth.token`, `gateway.auth.password`, `gateway.remote.token` или `gateway.remote.password`, при запуске/перезагрузке шлюза явно регистрируется состояние поверхности:

- `active`: SecretRef является частью эффективной поверхности аутентификации и должен быть разрешён.
- `inactive`: SecretRef игнорируется для этого времени выполнения, поскольку другая поверхность аутентификации имеет приоритет или удалённая аутентификация отключена/неактивна.

Эти записи регистрируются с кодом `SECRETS_GATEWAY_AUTH_SURFACE` и включают причину, использованную политикой активной поверхности, чтобы вы могли увидеть, почему учётные данные были признаны активными или неактивными.

## Предварительная проверка при подключении

Когда подключение выполняется в интерактивном режиме и вы выбираете хранение SecretRef, OpenClaw выполняет предварительную проверку перед сохранением:

- Ссылки на окружение: проверяется имя переменной окружения и подтверждается, что во время настройки видно непустое значение.
- Ссылки на провайдера (`file` или `exec`): проверяется выбор провайдера, разрешается `id` и проверяется тип разрешённого значения.
- Путь повторного использования при быстром старте: если `gateway.auth.token` уже является SecretRef, подключение разрешает его перед начальной загрузкой проб/панели мониторинга (для ссылок `env`, `file` и `exec`), используя тот же механизм быстрого сбоя.

Если проверка завершается сбоем, подключение отображает ошибку и позволяет повторить попытку.

## Контракт SecretRef

Используйте одну и ту же структуру объекта везде:

```json5
{ source: "env" | "file" | "exec", provider: "default", id: "..." }
```

### `source: "env"`

```json5
{ source: "env", provider: "default", id: "OPENAI_API_KEY" }
```

Проверка:

- `provider` должен соответствовать шаблону `^[a-z][a-z0-9_-]{0,63}$`
- `id` должен соответствовать шаблону `^[A-Z][A-Z0-9_]{0,127}$`

### `source: "file"`

```json5
{ source: "file", provider: "filemain", id: "/providers/openai/apiKey" }
```

Проверка:

- `provider` должен соответствовать шаблону `^[a-z][a-z0-9_-]{0,63}$`
- `id` должен быть абсолютным указателем JSON (`/...`)
- Экранирование по RFC6901 в сегментах: `~` => `~0`, `/` => `~1`

### `source: "exec"`

```json5
{ source: "exec", provider: "vault", id: "providers/openai/apiKey" }
```

Проверка:

- `provider` должен соответствовать шаблону `^[a-z][a-z0-9_-]{0,63}$`
- `id` должен соответствовать шаблону `^[A-Za-z0-9][A-Za-z0-9._:/-]{0,255}$`
- `id` не должен содержать `.` или `..` в качестве сегментов пути, разделённых слешами (например, `a/../b` будет отклонён)

## Конфигурация провайдера

Определите провайдеров в разделе `secrets.providers`:

```json5
{
  secrets: {
    providers: {
      default: { source: "env" },
      filemain: {
        source: "file",
        path: "~/.openclaw/secrets.json",
        mode: "json", // или "singleValue"
      },
      vault: {
        source: "exec",
        command: "/usr/local/bin/openclaw-vault-resolver",
        args: ["--profile", "prod"],
        passEnv: ["PATH", "VAULT_ADDR"],
        jsonOnly: true,
      },
    },
    defaults: {
      env: "default",
      file: "filemain",
      exec: "vault",
    },
    resolution: {
      maxProviderConcurrency: 4,
      maxRefsPerProvider: 512,
      maxBatchBytes: 262144,
    },
  },
}
```

### Провайдер окружения

- Опциональный белый список через `allowlist`.
- Отсутствие или пустое значение в окружении приводит к сбою разрешения.

### Провайдер файлов

- Считывает локальный файл по пути `path`.
- `mode: "json"` ожидает объект JSON и разрешает `id` как указатель.
- `mode: "singleValue"` ожидает идентификатор ссылки `"value"` и возвращает содержимое файла.
- Путь должен пройти проверку прав собственности и разрешений.
- Примечание для Windows (режим отказа): если проверка ACL недоступна для пути, разрешение завершается сбоем. Для доверенных путей можно установить `allowInsecurePath: true` для этого провайдера, чтобы обойти проверку безопасности пути.

### Провайдер exec

- Запускает указанный абсолютный путь к бинарному файлу, без оболочки.
- По умолчанию `command` должен указывать на обычный файл (не на символическую ссылку).
- Установите `allowSymlinkCommand: true`, чтобы разрешить пути
