## نموذج الإمكانات العام

الإمكانات هي النموذج العام لـ **plugin أصلية** داخل OpenClaw. كل
plugin أصلية في OpenClaw تسجل نفسها في نوع قدرة واحد أو أكثر:

| القدرة                | طريقة التسجيل                                   | أمثلة على plugins                    |
| --------------------- | ------------------------------------------------ | ------------------------------------ |
| استدلال النص          | `api.registerProvider(...)`                      | `openai`، `anthropic`                |
| خلفية استدلال CLI     | `api.registerCliBackend(...)`                    | `openai`، `anthropic`                |
| الكلام                | `api.registerSpeechProvider(...)`                | `elevenlabs`، `microsoft`            |
| النسخ الفوري          | `api.registerRealtimeTranscriptionProvider(...)` | `openai`                             |
| الصوت الفوري          | `api.registerRealtimeVoiceProvider(...)`         | `openai`                             |
| فهم الوسائط           | `api.registerMediaUnderstandingProvider(...)`    | `openai`، `google`                   |
| توليد الصور           | `api.registerImageGenerationProvider(...)`       | `openai`، `google`، `fal`، `minimax` |
| توليد الموسيقى        | `api.registerMusicGenerationProvider(...)`       | `google`، `minimax`                  |
| توليد الفيديو         | `api.registerVideoGenerationProvider(...)`       | `qwen`                               |
| جلب الويب             | `api.registerWebFetchProvider(...)`              | `firecrawl`                          |
| بحث الويب             | `api.registerWebSearchProvider(...)`             | `google`                             |
| القناة / المراسلة     | `api.registerChannel(...)`                       | `msteams`، `matrix`                  |

تُعد plugin التي تسجل صفرًا من الإمكانات لكنها توفر خطافات أو أدوات أو
خدمات **plugin قديمة تعتمد على الخطافات فقط**. لا يزال هذا النمط مدعومًا بالكامل.

### موقف التوافق الخارجي

لقد تم اعتماد نموذج الإمكانات في النواة ويُستخدم اليوم من قبل plugins
الأصلية/المضمّنة، لكن توافق plugins الخارجية لا يزال يحتاج إلى معيار أكثر
صرامة من مجرد "تم تصديره، إذن فهو ثابت."

الإرشادات الحالية:

- **plugins الخارجية الحالية:** حافظ على عمل التكاملات المستندة إلى الخطافات؛
  واعتبر ذلك خط الأساس للتوافق
- **plugins الأصلية/المضمّنة الجديدة:** فضّل التسجيل الصريح للإمكانات بدلًا من
  الوصولات الخاصة بالمورّد أو التصميمات الجديدة التي تعتمد على الخطافات فقط
- **plugins الخارجية التي تعتمد تسجيل الإمكانات:** هذا مسموح، لكن تعامل مع
  أسطح المساعدة الخاصة بالإمكانات على أنها قابلة للتطور ما لم توضح المستندات صراحةً أن
  العقد مستقر

القاعدة العملية:

- واجهات برمجة تطبيقات تسجيل الإمكانات هي الاتجاه المقصود
- تبقى الخطافات القديمة هي المسار الأكثر أمانًا لتجنب الكسر بالنسبة إلى plugins الخارجية أثناء
  مرحلة الانتقال
- ليست كل المسارات الفرعية المصدّرة متساوية؛ فافضّل العقد الضيق الموثّق،
  وليس صادرات المساعدة العرضية

### أشكال plugin

يصنّف OpenClaw كل plugin محمّلة إلى شكل بناءً على سلوك التسجيل
الفعلي لها (وليس فقط البيانات الوصفية الثابتة):

- **plain-capability** -- تسجل نوع قدرة واحدًا بالضبط (مثلًا plugin
  موفر فقط مثل `mistral`)
- **hybrid-capability** -- تسجل أنواع قدرات متعددة (مثلًا
  `openai` يمتلك استدلال النص والكلام وفهم الوسائط وتوليد
  الصور)
- **hook-only** -- تسجل خطافات فقط (مكتوبة الأنواع أو مخصصة)، بلا إمكانات
  أو أدوات أو أوامر أو خدمات
- **non-capability** -- تسجل أدوات أو أوامر أو خدمات أو مسارات، لكن بلا
  إمكانات

استخدم `openclaw plugins inspect <id>` لرؤية شكل plugin وتفصيل
إمكاناتها. راجع [مرجع CLI](/cli/plugins#inspect) للتفاصيل.

### الخطافات القديمة

لا يزال الخطاف `before_agent_start` مدعومًا كمسار توافق للـ plugins
التي تعتمد على الخطافات فقط. ولا تزال plugins واقعية قديمة تعتمد عليه.

الاتجاه:

- حافظ على عمله
- وثّقه على أنه قديم
- فضّل `before_model_resolve` في أعمال تجاوز النموذج/الموفر
- فضّل `before_prompt_build` في أعمال تعديل prompt
- أزله فقط بعد انخفاض الاستخدام الفعلي ووجود تغطية fixtures تثبت أمان الترحيل

### إشارات التوافق

عند تشغيل `openclaw doctor` أو `openclaw plugins inspect <id>`، قد ترى
إحدى هذه التسميات:

| الإشارة                    | المعنى                                                       |
| -------------------------- | ------------------------------------------------------------ |
| **config valid**           | يتم تحليل الإعداد بنجاح ويتم حل plugins                      |
| **compatibility advisory** | تستخدم plugin نمطًا مدعومًا لكنه أقدم (مثل `hook-only`)     |
| **legacy warning**         | تستخدم plugin الخطاف `before_agent_start`، وهو مهمل          |
| **hard error**             | الإعداد غير صالح أو فشل تحميل plugin                         |

لن يؤدي كل من `hook-only` و`before_agent_start` إلى كسر plugin الخاصة بك اليوم --
فـ `hook-only` مجرد تنبيه استشاري، و`before_agent_start` يطلق تحذيرًا فقط. تظهر
هذه الإشارات أيضًا في `openclaw status --all` و`openclaw plugins doctor`.

## نظرة عامة على البنية

يحتوي نظام plugins في OpenClaw على أربع طبقات:

1. **البيان + الاكتشاف**
   يعثر OpenClaw على plugins المرشحة من المسارات المضبوطة، وجذور workspace،
   وجذور الامتدادات العامة، والامتدادات المضمّنة. يقرأ الاكتشاف ملفات
   `openclaw.plugin.json` الأصلية إضافة إلى بيانات الحزم المتوافقة المدعومة أولًا.
2. **التمكين + التحقق**
   تحدد النواة ما إذا كانت plugin المكتشفة ممكّنة أو معطّلة أو محظورة أو
   مختارة لفتحة حصرية مثل الذاكرة.
3. **التحميل في وقت التشغيل**
   يتم تحميل plugins الأصلية لـ OpenClaw داخل العملية عبر jiti وتسجيل
   الإمكانات في سجل مركزي. ويتم تطبيع الحزم المتوافقة إلى
   سجلات في السجل بدون استيراد شيفرة وقت التشغيل.
4. **استهلاك الأسطح**
   يقرأ باقي OpenClaw السجل لكشف الأدوات والقنوات وإعداد
   الموفر والخطافات ومسارات HTTP وأوامر CLI والخدمات.

وبالنسبة إلى CLI الخاص بالـ plugin تحديدًا، ينقسم اكتشاف الأمر الجذري إلى مرحلتين:

- تأتي بيانات وقت التحليل من `registerCli(..., { descriptors: [...] })`
- ويمكن أن تبقى وحدة CLI الحقيقية الخاصة بالـ plugin كسولة وتُسجَّل عند أول استدعاء

وهذا يُبقي شيفرة CLI المملوكة للـ plugin داخل plugin مع السماح لـ OpenClaw
بحجز أسماء الأوامر الجذرية قبل التحليل.

الحد التصميمي المهم:

- يجب أن يعمل الاكتشاف + التحقق من الإعداد من **البيان/بيانات schema**
  بدون تنفيذ شيفرة plugin
- بينما يأتي سلوك وقت التشغيل الأصلي من مسار `register(api)` الخاص بوحدة plugin

يتيح هذا الفصل لـ OpenClaw التحقق من الإعداد، وشرح plugins المفقودة/المعطّلة،
وبناء تلميحات UI/schema قبل تنشيط وقت التشغيل الكامل.

### Plugins القنوات وأداة الرسائل المشتركة

لا تحتاج Plugins القنوات إلى تسجيل أداة send/edit/react منفصلة
لإجراءات الدردشة العادية. يحتفظ OpenClaw بأداة `message` مشتركة واحدة في النواة،
في حين تمتلك Plugins القنوات الاكتشاف والتنفيذ الخاصين بالقناة وراءها.

الحد الحالي هو:

- تمتلك النواة مضيف أداة `message` المشتركة، وربط prompt، ومسك
  الجلسة/السلسلة، وتوجيه التنفيذ
- تمتلك Plugins القنوات اكتشاف الإجراءات ذات النطاق، واكتشاف الإمكانات، وأي
  أجزاء schema خاصة بالقناة
- تمتلك Plugins القنوات قواعد محادثة الجلسة الخاصة بالموفر، مثل
  كيفية ترميز معرّفات المحادثات لمعرّفات السلاسل أو وراثتها من المحادثات الأصلية
- تنفذ Plugins القنوات الإجراء النهائي من خلال محول الإجراءات الخاص بها

بالنسبة إلى Plugins القنوات، فإن سطح SDK هو
`ChannelMessageActionAdapter.describeMessageTool(...)`. تسمح مكالمة الاكتشاف
الموحدة هذه للـ plugin بإرجاع إجراءاتها المرئية وإمكاناتها ومساهمات schema
معًا حتى لا تنحرف هذه الأجزاء عن بعضها.

تمرر النواة نطاق وقت التشغيل إلى خطوة الاكتشاف هذه. وتشمل الحقول المهمة:

- `accountId`
- `currentChannelId`
- `currentThreadTs`
- `currentMessageId`
- `sessionKey`
- `sessionId`
- `agentId`
- قيمة `requesterSenderId` الواردة الموثوقة

وهذا مهم للـ plugins الحساسة للسياق. إذ يمكن للقناة إخفاء أو إظهار
إجراءات الرسائل بناءً على الحساب النشط أو الغرفة/السلسلة/الرسالة الحالية أو
هوية الطالب الموثوقة، بدون ترميز فروع خاصة بالقناة داخل أداة `message`
في النواة.

ولهذا السبب تظل تغييرات التوجيه في embedded-runner عملًا خاصًا بالـ plugin:
فالمشغّل مسؤول عن تمرير هوية الدردشة/الجلسة الحالية إلى حد
اكتشاف plugin حتى تعرض أداة `message` المشتركة السطح الصحيح المملوك للقناة
في الدورة الحالية.

أما بالنسبة إلى مساعدات التنفيذ المملوكة للقناة، فيجب على plugins المضمّنة إبقاء
وقت تشغيل التنفيذ داخل وحدات الامتداد المملوكة لها. لم تعد النواة تمتلك
أوقات تشغيل إجراءات الرسائل الخاصة بـ Discord أو Slack أو Telegram أو WhatsApp
ضمن `src/agents/tools`.
كما أننا لا ننشر مسارات فرعية منفصلة `plugin-sdk/*-action-runtime`، ويجب على plugins
المضمّنة استيراد شيفرة وقت التشغيل المحلية الخاصة بها مباشرة من
وحدات الامتداد المملوكة لها.

وينطبق الحد نفسه عمومًا على أسطح SDK المسماة باسم الموفر: يجب ألّا
تستورد النواة أشرطة راحة خاصة بالقنوات مثل Slack أو Discord أو Signal أو
WhatsApp أو امتدادات مشابهة. إذا احتاجت النواة إلى سلوك ما، فإما أن تستهلك
شريط `api.ts` / `runtime-api.ts` الخاص بالـ plugin المضمّنة نفسها أو
ترفع الحاجة إلى قدرة عامة ضيقة في SDK المشتركة.

وبالنسبة إلى استطلاعات الرأي تحديدًا، يوجد مساران للتنفيذ:

- `outbound.sendPoll` هو خط الأساس المشترك للقنوات التي تناسب نموذج
  الاستطلاع الشائع
- `actions.handleAction("poll")` هو المسار المفضل لدلالات الاستطلاع الخاصة بالقناة
  أو لمعاملات الاستطلاع الإضافية

تؤجل النواة الآن تحليل الاستطلاع المشترك إلى ما بعد رفض إرسال الاستطلاع عبر plugin
للإجراء، حتى تتمكن معالجات الاستطلاع المملوكة للـ plugin من قبول
حقول استطلاع خاصة بالقناة دون أن يمنعها محلل الاستطلاع العام أولًا.

راجع [خط أنابيب التحميل](#load-pipeline) لمعرفة التسلسل الكامل عند بدء التشغيل.

## نموذج ملكية الإمكانات

يتعامل OpenClaw مع plugin الأصلية على أنها حد الملكية لـ **شركة** أو
**ميزة**، وليس مجرد حاوية لتكاملات غير مترابطة.

وهذا يعني:

- ينبغي عادةً أن تمتلك plugin الشركة جميع الأسطح التي يواجهها OpenClaw من تلك
  الشركة
- ينبغي عادةً أن تمتلك plugin الميزة السطح الكامل للميزة التي تقدمها
- يجب أن تستهلك القنوات إمكانات النواة المشتركة بدلًا من إعادة تنفيذ
  سلوك الموفر بشكل مخصص

أمثلة:

- تمتلك plugin المضمّنة `openai` سلوك موفر نماذج OpenAI وسلوك
  OpenAI للكلام + الصوت الفوري + فهم الوسائط + توليد الصور
- تمتلك plugin المضمّنة `elevenlabs` سلوك الكلام الخاص بـ ElevenLabs
- تمتلك plugin المضمّنة `microsoft` سلوك الكلام الخاص بـ Microsoft
- تمتلك plugin المضمّنة `google` سلوك موفر نماذج Google إضافة إلى
  سلوك Google لفهم الوسائط + توليد الصور + بحث الويب
- تمتلك plugin المضمّنة `firecrawl` سلوك جلب الويب الخاص بـ Firecrawl
- تمتلك plugins المضمّنة `minimax` و`mistral` و`moonshot` و`zai`
  خلفيات فهم الوسائط الخاصة بها
- تمتلك plugin `qwen` المضمّنة سلوك موفر النص في Qwen إضافة إلى
  سلوك فهم الوسائط وتوليد الفيديو
- أما plugin `voice-call` فهي plugin ميزة: فهي تمتلك نقل المكالمات والأدوات،
  وCLI، والمسارات، وجسر تدفق وسائط Twilio، لكنها تستهلك
  إمكانات الكلام المشتركة إضافة إلى النسخ الفوري والصوت الفوري بدلًا من
  استيراد plugins المورّدين مباشرة

الحالة النهائية المقصودة هي:

- يعيش OpenAI في plugin واحدة حتى لو امتد عبر نماذج النص والكلام والصور
  وربما الفيديو مستقبلًا
- ويمكن لمورّد آخر أن يفعل الشيء نفسه لسطحه الخاص
- لا تهتم القنوات بأي plugin مورّد تمتلك الموفر؛ فهي تستهلك عقد
  الإمكانات المشتركة التي تكشفها النواة

وهذا هو التمييز الأساسي:

- **plugin** = حد الملكية
- **capability** = عقد النواة الذي يمكن أن تنفذه أو تستهلكه عدة plugins

لذا إذا أضاف OpenClaw مجالًا جديدًا مثل الفيديو، فالسؤال الأول ليس
"أي موفر يجب أن يرمّز معالجة الفيديو بشكل صلب؟" بل السؤال الأول هو "ما
عقد قدرة الفيديو في النواة؟" وبمجرد وجود هذا العقد، يمكن للـ plugins الخاصة بالمورّدين
التسجيل فيه، ويمكن للـ plugins الخاصة بالقنوات/الميزات استهلاكه.

إذا لم تكن القدرة موجودة بعد، فعادةً ما تكون الخطوة الصحيحة هي:

1. تعريف القدرة المفقودة في النواة
2. كشفها بطريقة مكتوبة الأنواع عبر واجهة برمجة plugin / وقت التشغيل
3. ربط القنوات/الميزات بهذه القدرة
4. والسماح للـ plugins الخاصة بالمورّدين بتسجيل التنفيذات

وهذا يُبقي الملكية صريحة مع تجنب سلوك في النواة يعتمد على
مورّد واحد أو مسار خاص بـ plugin واحدة.

### تدرج الإمكانات

استخدم هذا النموذج الذهني عند تقرير مكان الشيفرة:

- **طبقة قدرات النواة**: التنسيق المشترك، والسياسة، والرجوع الاحتياطي، وقواعد دمج
  الإعداد، ودلالات التسليم، والعقود المكتوبة
- **طبقة plugin المورّد**: واجهات API الخاصة بالمورّد، والمصادقة، وفهارس النماذج، وتوليف
  الكلام، وتوليد الصور، وخلفيات الفيديو مستقبلًا، ونقاط نهاية الاستخدام
- **طبقة plugin القناة/الميزة**: تكاملات Slack/Discord/voice-call/... التي
  تستهلك إمكانات النواة وتعرضها على سطح ما

على سبيل المثال، يتبع TTS هذا الشكل:

- تمتلك النواة سياسة TTS وقت الرد، وترتيب الرجوع، والتفضيلات، وتسليم القنوات
- وتملك `openai` و`elevenlabs` و`microsoft` تنفيذات التوليف
- وتستهلك `voice-call` مساعد وقت تشغيل TTS الخاص بالاتصالات الهاتفية

وينبغي تفضيل هذا النمط أيضًا للإمكانات المستقبلية.

### مثال على plugin شركة متعددة الإمكانات

يجب أن تبدو plugin الشركة متماسكة من الخارج. إذا كان لدى OpenClaw عقود مشتركة
للنماذج والكلام والنسخ الفوري والصوت الفوري وفهم الوسائط
وتوليد الصور وتوليد الفيديو وجلب الويب وبحث الويب،
فيمكن لمورّد امتلاك جميع أسطحه في مكان واحد:

```ts
import type { OpenClawPluginDefinition } from "openclaw/plugin-sdk/plugin-entry";
import {
  describeImageWithModel,
  transcribeOpenAiCompatibleAudio,
} from "openclaw/plugin-sdk/media-understanding";

const plugin: OpenClawPluginDefinition = {
  id: "exampleai",
  name: "ExampleAI",
  register(api) {
    api.registerProvider({
      id: "exampleai",
      // auth/model catalog/runtime hooks
    });

    api.registerSpeechProvider({
      id: "exampleai",
      // vendor speech config — implement the SpeechProviderPlugin interface directly
    });

    api.registerMediaUnderstandingProvider({
      id: "exampleai",
      capabilities: ["image", "audio", "video"],
      async describeImage(req) {
        return describeImageWithModel({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
      async transcribeAudio(req) {
        return transcribeOpenAiCompatibleAudio({
          provider: "exampleai",
          model: req.model,
          input: req.input,
        });
      },
    });

    api.registerWebSearchProvider(
      createPluginBackedWebSearchProvider({
        id: "exampleai-search",
        // credential + fetch logic
      }),
    );
  },
};

export default plugin;
```

ما يهم ليس أسماء المساعدات بالضبط. بل الشكل هو المهم:

- plugin واحدة تمتلك سطح المورّد
- لا تزال النواة تمتلك عقود الإمكانات
- تستهلك القنوات وplugins الميزات مساعدات `api.runtime.*`، لا شيفرة المورّد
- ويمكن لاختبارات العقد التحقق من أن plugin سجّلت الإمكانات التي
  تدعي امتلاكها

### مثال على القدرة: فهم الفيديو

يعامل OpenClaw بالفعل فهم الصور/الصوت/الفيديو كقدرة مشتركة واحدة.
وينطبق نموذج الملكية نفسه هنا:

1. تعرّف النواة عقد فهم الوسائط
2. تسجل plugins الخاصة بالمورّدين `describeImage` و`transcribeAudio` و
   `describeVideo` حسب الاقتضاء
3. تستهلك القنوات وplugins الميزات سلوك النواة المشترك بدلًا من
   الربط المباشر بشيفرة المورّد

وهذا يتجنب ترسيخ افتراضات الفيديو الخاصة بموفّر واحد داخل النواة. فالـ plugin تمتلك
سطح المورّد؛ بينما تمتلك النواة عقد القدرة وسلوك الرجوع الاحتياطي.

كما أن توليد الفيديو يتبع التسلسل نفسه بالفعل: فالنواة تمتلك عقد
القدرة المكتوب ومساعد وقت التشغيل، بينما تسجّل plugins الخاصة بالمورّدين
تنفيذات `api.registerVideoGenerationProvider(...)` له.

هل تحتاج إلى قائمة إطلاق عملية؟ راجع
[Capability Cookbook](/ar/plugins/architecture).

## العقود وفرضها

سطح واجهة برمجة plugin مكتوب الأنواع ومركزي عمدًا في
`OpenClawPluginApi`. يعرّف هذا العقد نقاط التسجيل المدعومة
ومساعدات وقت التشغيل التي يمكن أن تعتمد عليها plugin.

لماذا هذا مهم:

- يحصل مؤلفو plugins على معيار داخلي واحد مستقر
- يمكن للنواة رفض الملكية المكررة، مثل تسجيل pluginَين لمعرّف الموفر نفسه
- يمكن لبدء التشغيل عرض تشخيصات قابلة للتنفيذ للتسجيلات غير الصحيحة
- يمكن لاختبارات العقد فرض ملكية plugins المضمّنة ومنع الانحراف الصامت

هناك طبقتان من الفرض:

1. **فرض التسجيل في وقت التشغيل**
   يتحقق سجل plugins من التسجيلات أثناء تحميل plugins. أمثلة:
   معرّفات الموفّر المكررة، ومعرّفات موفر الكلام المكررة، والتسجيلات
   غير الصحيحة تنتج تشخيصات للـ plugin بدلًا من سلوك غير معرّف.
2. **اختبارات العقد**
   يتم التقاط plugins المضمّنة في سجلات العقود أثناء تشغيل الاختبارات حتى
   يتمكن OpenClaw من فرض الملكية صراحةً. ويُستخدم هذا اليوم لموفري النماذج،
   وموفري الكلام، وموفري بحث الويب، وملكية التسجيل المضمّن.

والأثر العملي هو أن OpenClaw يعرف مسبقًا أي plugin تمتلك أي
سطح. وهذا يتيح للنواة والقنوات التركيب بسلاسة لأن الملكية مُعلنة
ومكتوبة وقابلة للاختبار بدلًا من أن تكون ضمنية.

### ما الذي ينتمي إلى عقد

عقود plugins الجيدة تكون:

- مكتوبة الأنواع
- صغيرة
- خاصة بقدرة محددة
- مملوكة للنواة
- قابلة لإعادة الاستخدام من عدة plugins
- قابلة للاستهلاك من القنوات/الميزات بدون معرفة بالمورّد

أما عقود plugins السيئة فهي:

- سياسة خاصة بمورّد مخفية داخل النواة
- منافذ هروب خاصة بـ plugin واحدة تتجاوز السجل
- شيفرة قناة تصل مباشرة إلى تنفيذ مورّد
- كائنات وقت تشغيل مخصصة ليست جزءًا من `OpenClawPluginApi` أو
  `api.runtime`

عند الشك، ارفع مستوى التجريد: عرّف القدرة أولًا، ثم اسمح للـ plugins
بالارتباط بها.

## نموذج التنفيذ

تعمل plugins الأصلية لـ OpenClaw **داخل العملية** مع Gateway. وهي غير
معزولة. وتملك plugin الأصلية المحمّلة حد ثقة على مستوى العملية مماثلًا
لشيفرة النواة.

الآثار:

- يمكن لـ plugin الأصلية تسجيل أدوات ومعالجات شبكات وخطافات وخدمات
- يمكن لخلل في plugin أصلية أن يعطل gateway أو يزعزع استقرارها
- plugin أصلية خبيثة تعادل تنفيذ شيفرة عشوائية داخل
  عملية OpenClaw

الحزم المتوافقة أكثر أمانًا افتراضيًا لأن OpenClaw يعاملها حاليًا
كحِزم بيانات وصفية/محتوى. وفي الإصدارات الحالية، يعني ذلك في الغالب
Skills المضمّنة.

استخدم قوائم السماح ومسارات التثبيت/التحميل الصريحة للـ plugins غير المضمّنة. وتعامل مع
plugins الـ workspace كشيفرة وقت تطوير، لا كافتراضات إنتاج.

وبالنسبة إلى أسماء حزم workspace المضمّنة، أبقِ معرّف plugin مثبتًا في
اسم npm: `@openclaw/<id>` افتراضيًا، أو لاحقة نوعية معتمدة مثل
`-provider` أو `-plugin` أو `-speech` أو `-sandbox` أو `-media-understanding` عندما
تكشف الحزمة عمدًا دور plugin أضيق.

ملاحظة ثقة مهمة:

- تثق `plugins.allow` في **معرّفات plugins**، لا في مصدرها.
- إن plugin workspace لها المعرّف نفسه الذي لplugin مضمّنة تحجب
  النسخة المضمّنة عمدًا عندما تكون plugin workspace هذه مفعلة/مدرجة في قائمة السماح.
- وهذا سلوك طبيعي ومفيد للتطوير المحلي واختبار الترقيعات والإصلاحات العاجلة.

## حد التصدير

يصدّر OpenClaw الإمكانات، لا وسائل راحة التنفيذ.

أبقِ تسجيل الإمكانات عامًا. وقلّص صادرات المساعدات غير التعاقدية:

- المسارات الفرعية الخاصة بمساعدات plugins المضمّنة
- المسارات الفرعية الخاصة ببنية وقت التشغيل غير المقصود أن تكون API عامة
- مساعدات الراحة الخاصة بالمورّد
- مساعدات الإعداد/التهيئة التي تُعد تفاصيل تنفيذ

لا تزال بعض المسارات الفرعية المساعدة للـ plugins المضمّنة موجودة في خريطة صادرات
SDK المُولَّدة من أجل التوافق وصيانة plugins المضمّنة. من الأمثلة الحالية
`plugin-sdk/feishu`، و`plugin-sdk/feishu-setup`، و`plugin-sdk/zalo`،
و`plugin-sdk/zalo-setup`، والعديد من حدود `plugin-sdk/matrix*`. تعامل مع هذه
على أنها صادرات محجوزة كتفاصيل تنفيذ، لا على أنها نمط SDK موصى به
للـ plugins الخارجية الجديدة.

## خط أنابيب التحميل

عند بدء التشغيل، يقوم OpenClaw تقريبًا بما يلي:

1. اكتشاف جذور plugins المرشحة
2. قراءة البيانات الأصلية أو بيانات الحزم المتوافقة وبيانات الحزمة
3. رفض المرشحين غير الآمنين
4. تطبيع إعداد plugin (`plugins.enabled`، و`allow`، و`deny`، و`entries`،
   و`slots`، و`load.paths`)
5. تحديد التمكين لكل مرشح
6. تحميل الوحدات الأصلية المفعلة عبر jiti
7. استدعاء خطافات `register(api)` الأصلية (أو `activate(api)` — وهو اسم قديم بديل) وجمع التسجيلات في سجل plugin
8. كشف السجل إلى أسطح الأوامر/وقت التشغيل

<Note>
`activate` اسم قديم بديل لـ `register` — يحل المحمّل أيهما موجود (`def.register ?? def.activate`) ويستدعيه في النقطة نفسها. تستخدم جميع plugins المضمّنة `register`؛ ويفضل استخدام `register` في plugins الجديدة.
</Note>

تحدث بوابات الأمان **قبل** تنفيذ وقت التشغيل. يتم حظر المرشحين
عندما يخرج الإدخال من جذر plugin، أو يكون المسار قابلاً للكتابة من الجميع،
أو تبدو ملكية المسار مشبوهة للـ plugins غير المضمّنة.

### سلوك البيان أولًا

البيان هو مصدر الحقيقة في مستوى التحكم. يستخدمه OpenClaw من أجل:

- تحديد plugin
- اكتشاف القنوات/Skills/schema الإعداد المعلنة أو إمكانات الحزمة
- التحقق من `plugins.entries.<id>.config`
- تعزيز تسميات/عناصر نائبة لـ Control UI
- عرض بيانات التثبيت/الفهرس

وبالنسبة إلى plugins الأصلية، فإن وحدة وقت التشغيل هي جزء مستوى البيانات. فهي تسجل
السلوك الفعلي مثل الخطافات أو الأدوات أو الأوامر أو تدفقات الموفر.

### ما الذي يخزنه المحمّل مؤقتًا

يحتفظ OpenClaw بذاكرات تخزين مؤقت قصيرة داخل العملية من أجل:

- نتائج الاكتشاف
- بيانات سجل البيان
- سجلات plugins المحمّلة

تقلل هذه الذاكرات من أعباء بدء التشغيل المتقطع وتكرار الأوامر.
ومن الآمن اعتبارها ذواكر أداء قصيرة العمر، لا تخزينًا دائمًا.

ملاحظة أداء:

- اضبط `OPENCLAW_DISABLE_PLUGIN_DISCOVERY_CACHE=1` أو
  `OPENCLAW_DISABLE_PLUGIN_MANIFEST_CACHE=1` لتعطيل هذه الذواكر المؤقتة.
- ويمكنك ضبط نوافذ التخزين المؤقت بواسطة `OPENCLAW_PLUGIN_DISCOVERY_CACHE_MS` و
  `OPENCLAW_PLUGIN_MANIFEST_CACHE_MS`.

## نموذج السجل

لا تعدّل plugins المحمّلة مباشرة متغيرات عشوائية عامة في النواة. بل تسجل نفسها في
سجل plugins مركزي.

يتتبع السجل:

- سجلات plugins (الهوية، والمصدر، والمنشأ، والحالة، والتشخيصات)
- الأدوات
- الخطافات القديمة والخطافات المكتوبة
- القنوات
- الموفّرين
- معالجات Gateway RPC
- مسارات HTTP
- مسجلات CLI
- الخدمات الخلفية
- الأوامر المملوكة للـ plugin

ثم تقرأ ميزات النواة من هذا السجل بدلًا من التحدث مباشرة إلى وحدات
plugins. وهذا يُبقي التحميل في اتجاه واحد:

- وحدة plugin -> التسجيل في السجل
- وقت تشغيل النواة -> استهلاك السجل

وهذا الفصل مهم لقابلية الصيانة. فهو يعني أن معظم أسطح النواة تحتاج فقط
إلى نقطة تكامل واحدة: "اقرأ السجل"، لا "ضع حالة خاصة لكل وحدة plugin".

## نداءات ربط المحادثة

يمكن للـ plugins التي تربط محادثة أن تتفاعل عند حل الموافقة.

استخدم `api.onConversationBindingResolved(...)` لتلقي نداء بعد الموافقة على
طلب الربط أو رفضه:

```ts
export default {
  id: "my-plugin",
  register(api) {
    api.onConversationBindingResolved(async (event) => {
      if (event.status === "approved") {
        // A binding now exists for this plugin + conversation.
        console.log(event.binding?.conversationId);
        return;
      }

      // The request was denied; clear any local pending state.
      console.log(event.request.conversation.conversationId);
    });
  },
};
```

حقول حمولة النداء:

- `status`: إما `"approved"` أو `"denied"`
- `decision`: إما `"allow-once"` أو `"allow-always"` أو `"deny"`
- `binding`: الربط المحلول للطلبات المعتمدة
- `request`: ملخص الطلب الأصلي، وتلميح الفصل، ومعرّف المرسل، وبيانات
  المحادثة الوصفية

هذا النداء مخصّص للإشعار فقط. ولا يغيّر من يُسمح له بربط
محادثة، ويعمل بعد انتهاء معالجة الموافقة في النواة.

## خطافات وقت تشغيل الموفّر

تملك Plugins الموفر الآن طبقتين:

- بيانات وصفية في البيان: `providerAuthEnvVars` للبحث الرخيص عن مصادقة الموفر عبر env
  قبل تحميل وقت التشغيل، و`channelEnvVars` للبحث الرخيص عن env/إعداد القناة
  قبل تحميل وقت التشغيل، إضافة إلى `providerAuthChoices` لعرض
  تسميات onboarding/اختيار المصادقة وبيانات أعلام CLI البسيطة قبل تحميل وقت التشغيل
- خطافات وقت الإعداد: `catalog` / الاسم القديم `discovery` إضافة إلى `applyConfigDefaults`
- خطافات وقت التشغيل: `normalizeModelId`، و`normalizeTransport`،
  و`normalizeConfig`،
  و`applyNativeStreamingUsageCompat`، و`resolveConfigApiKey`،
  و`resolveSyntheticAuth`، و`resolveExternalAuthProfiles`،
  و`shouldDeferSyntheticProfileAuth`،
  و`resolveDynamicModel`، و`prepareDynamicModel`، و`normalizeResolvedModel`،
  و`contributeResolvedModelCompat`، و`capabilities`،
  و`normalizeToolSchemas`، و`inspectToolSchemas`,
  و`resolveReasoningOutputMode`، و`prepareExtraParams`، و`createStreamFn`,
  و`wrapStreamFn`، و`resolveTransportTurnState`,
  و`resolveWebSocketSessionPolicy`، و`formatApiKey`، و`refreshOAuth`,
  و`buildAuthDoctorHint`، و`matchesContextOverflowError`,
  و`classifyFailoverReason`، و`isCacheTtlEligible`,
  و`buildMissingAuthMessage`، و`suppressBuiltInModel`، و`augmentModelCatalog`,
  و`isBinaryThinking`، و`supportsXHighThinking`,
  و`resolveDefaultThinkingLevel`، و`isModernModelRef`، و`prepareRuntimeAuth`,
  و`resolveUsageAuth`، و`fetchUsageSnapshot`، و`createEmbeddingProvider`,
  و`buildReplayPolicy`,
  و`sanitizeReplayHistory`، و`validateReplayTurns`، و`onModelSelected`

لا يزال OpenClaw يمتلك الحلقة العامة للوكيل، والرجوع الاحتياطي، ومعالجة
النصوص، وسياسة الأدوات. وهذه الخطافات هي سطح الامتداد للسلوك الخاص بالمورّد دون
الحاجة إلى نقل استدلال مخصص كامل.

استخدم `providerAuthEnvVars` في البيان عندما يكون لدى الموفر بيانات اعتماد
مستندة إلى env يجب أن تراها مسارات auth/status/model-picker العامة بدون
تحميل وقت تشغيل plugin. واستخدم `providerAuthChoices` في البيان عندما
تحتاج أسطح onboarding/اختيار المصادقة في CLI إلى معرفة معرّف اختيار الموفر،
وتسميات المجموعة، وربط المصادقة البسيط بعلم واحد دون تحميل وقت تشغيل الموفر.
وأبقِ `envVars` في وقت تشغيل الموفر للتلميحات الموجهة للمشغّل، مثل تسميات
onboarding أو متغيرات إعداد OAuth client-id/client-secret.

واستخدم `channelEnvVars` في البيان عندما تكون لدى القناة مصادقة أو إعداد
مدفوع بـ env يجب أن تراه بدائل shell-env العامة، وفحوصات الإعداد/الحالة،
أو مطالبات الإعداد دون تحميل وقت تشغيل القناة.

### ترتيب الخطافات واستخدامها

بالنسبة إلى Plugins النماذج/الموفّر، يستدعي OpenClaw الخطافات تقريبًا بهذا الترتيب.
ويُعد عمود "متى يُستخدم" دليل القرار السريع.

| #   | الخطاف                            | ما الذي يفعله                                                                                                   | متى يُستخدم                                                                                                                                 |
| --- | --------------------------------- | -------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | `catalog`                         | نشر إعداد الموفر إلى `models.providers` أثناء توليد `models.json`                                              | عندما يمتلك الموفر فهرسًا أو إعدادات افتراضية لـ base URL                                                                                  |
| 2   | `applyConfigDefaults`             | تطبيق الإعدادات الافتراضية العامة المملوكة للموفر أثناء تشكيل الإعداد                                          | عندما تعتمد القيم الافتراضية على وضع المصادقة أو env أو دلالات عائلة نماذج الموفر                                                          |
| --  | _(بحث النماذج المضمنة)_           | يحاول OpenClaw أولًا مسار السجل/الفهرس العادي                                                                  | _(ليس خطاف plugin)_                                                                                                                         |
| 3   | `normalizeModelId`                | تطبيع الأسماء البديلة القديمة أو المعاينة لمعرّف النموذج قبل البحث                                              | عندما يمتلك الموفر تنظيف الأسماء البديلة قبل الحل القانوني للنموذج                                                                          |
| 4   | `normalizeTransport`              | تطبيع `api` / `baseUrl` لعائلة الموفر قبل التجميع العام للنموذج                                               | عندما يمتلك الموفر تنظيف النقل لمعرّفات موفّر مخصصة ضمن عائلة النقل نفسها                                                                  |
| 5   | `normalizeConfig`                 | تطبيع `models.providers.<id>` قبل حل وقت التشغيل/الموفر                                                        | عندما يحتاج الموفر إلى تنظيف إعداد يجب أن يعيش مع plugin؛ وتؤمّن مساعدات Google المضمّنة أيضًا تنظيفًا احتياطيًا لإدخالات Google المدعومة |

تتحقق `normalizeModelId` و`normalizeTransport` و`normalizeConfig` أولًا من
plugin الموفّر المطابق، ثم تنتقل إلى plugins الموفّر الأخرى القادرة على الخطافات
حتى تقوم إحداها فعلًا بتغيير معرّف النموذج أو النقل/الإعداد. وهذا يُبقي
الـ shims الخاصة بالأسماء البديلة/التوافقية تعمل بدون أن يضطر المستدعي إلى معرفة
أي plugin مضمّنة تمتلك إعادة الكتابة. وإذا لم يُعِد أي خطاف موفّر كتابة
إدخال إعداد مدعوم من عائلة Google، فإن مُطبّع إعداد Google المضمّن
لا يزال يطبّق هذا التنظيف التوافقي.

إذا كان الموفر يحتاج إلى بروتوكول سلكي مخصص بالكامل أو منفّذ طلبات مخصص،
فهذه فئة مختلفة من الامتداد. وهذه الخطافات مخصصة لسلوك الموفر
الذي لا يزال يعمل ضمن حلقة الاستدلال العادية في OpenClaw.

### مثال على موفر

```ts
api.registerProvider({
  id: "example-proxy",
  label: "Example Proxy",
  auth: [],
  catalog: {
    order: "simple",
    run: async (ctx) => {
      const apiKey = ctx.resolveProviderApiKey("example-proxy").apiKey;
      if (!apiKey) {
        return null;
      }
      return {
        provider: {
          baseUrl: "https://proxy.example.com/v1",
          apiKey,
          api: "openai-completions",
          models: [{ id: "auto", name: "Auto" }],
        },
      };
    },
  },
  resolveDynamicModel: (ctx) => ({
    id: ctx.modelId,
    name: ctx.modelId,
    provider: "example-proxy",
    api: "openai-completions",
    baseUrl: "https://proxy.example.com/v1",
    reasoning: false,
    input: ["text"],
    cost: { input: 0, output: 0, cacheRead: 0, cacheWrite: 0 },
    contextWindow: 128000,
    maxTokens: 8192,
  }),
  prepareRuntimeAuth: async (ctx) => {
    const exchanged = await exchangeToken(ctx.apiKey);
    return {
      apiKey: exchanged.token,
      baseUrl: exchanged.baseUrl,
      expiresAt: exchanged.expiresAt,
    };
  },
  resolveUsageAuth: async (ctx) => {
    const auth = await ctx.resolveOAuthToken();
    return auth ? { token: auth.token } : null;
  },
  fetchUsageSnapshot: async (ctx) => {
    return await fetchExampleProxyUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn);
  },
});
```

### أمثلة مضمّنة

- تستخدم Anthropic الخطافات `resolveDynamicModel` و`capabilities` و`buildAuthDoctorHint`،
  و`resolveUsageAuth`، و`fetchUsageSnapshot`، و`isCacheTtlEligible`،
  و`resolveDefaultThinkingLevel`، و`applyConfigDefaults`، و`isModernModelRef`،
  و`wrapStreamFn` لأنها تمتلك توافق Claude 4.6 المستقبلي،
  وتلميحات عائلة الموفر، وإرشاد إصلاح المصادقة، وتكامل نقطة نهاية الاستخدام،
  وأهلية prompt-cache، والقيم الافتراضية للإعداد المرتبطة بالمصادقة، وسياسة
  التفكير الافتراضي/التكيفي لـ Claude، وتشكيل البث الخاص بـ Anthropic من أجل
  headers التجريبية و`/fast` / `serviceTier` و`context1m`.
- تبقى مساعدات البث الخاصة بـ Claude في Anthropic حاليًا ضمن
  الحد العام المملوك للـ plugin نفسها `api.ts` / `contract-api.ts`.
  ويصدّر سطح الحزمة هذا `wrapAnthropicProviderStream`، و`resolveAnthropicBetas`،
  و`resolveAnthropicFastMode`، و`resolveAnthropicServiceTier`، ومنشئي
  المغلفات الأدنى مستوى لـ Anthropic بدلًا من توسيع SDK العامة حول قواعد
  headers التجريبية لموفر واحد.
- تستخدم OpenAI الخطافات `resolveDynamicModel`، و`normalizeResolvedModel`،
  و`capabilities` إضافة إلى `buildMissingAuthMessage`، و`suppressBuiltInModel`،
  و`augmentModelCatalog`، و`supportsXHighThinking`، و`isModernModelRef`

  `openai-completions` إلى `openai-responses`، وتلميحات المصادقة الخاصة بـ Codex،
  وإخفاء Spark، وصفوف قائمة OpenAI الصناعية، وسياسة التفكير /
  النموذج المباشر لـ GPT-5؛ وتمتلك عائلة البث `openai-responses-defaults` المغلفات
  الأصلية المشتركة لـ OpenAI Responses من أجل headers الإسناد،
  و`/fast`/`serviceTier`، وإسهاب النص، وبحث الويب الأصلي لـ Codex،
  وتشكيل حمولة reasoning-compat، وإدارة سياق Responses.
- تستخدم OpenRouter `catalog` إضافة إلى `resolveDynamicModel` و
  `prepareDynamicModel` لأن الموفر تمريري وقد يعرض معرّفات نماذج جديدة قبل
  تحديث الفهرس الثابت في OpenClaw؛ كما تستخدم
  `capabilities` و`wrapStreamFn` و`isCacheTtlEligible` لإبقاء
  headers الطلب والبيانات الوصفية للتوجيه وترقيعات reasoning
  وسياسة prompt-cache الخاصة بالموفر خارج النواة. وتأتي سياسة replay الخاصة بها من
  عائلة `passthrough-gemini`، بينما تمتلك عائلة البث `openrouter-thinking`

  والقيم الافتراضية لـ `tool_stream`، وتجربة التفكير الثنائية، ومطابقة
  النماذج الحديثة، وكلًا من مصادقة الاستخدام وجلب الحصة؛ وتُبقي عائلة البث
  `tool-stream-default-on` مغلف `tool_stream` الافتراضي المفعل خارج
  الربط اليدوي لكل موفر.
- تستخدم xAI `normalizeResolvedModel`، و`normalizeTransport`،
  و`contributeResolvedModelCompat`، و`prepareExtraParams`، و`wrapStreamFn`،
  و`resolveSyntheticAuth`، و`resolveDynamicModel`، و`isModernModelRef`
  لأنها تمتلك تطبيع النقل الأصلي لـ xAI Responses، وإعادة كتابة
  الأسماء البديلة لـ Grok في الوضع السريع، والقيمة الافتراضية `tool_stream`، وتنظيف
  strict-tool / reasoning-payload، وإعادة استخدام المصادقة الاحتياطية لأدوات
  مملوكة للـ plugin، وحل نماذج Grok المتوافق مستقبلًا، وترقيعات توافق
  مملوكة للموفر مثل ملف schema الأدوات الخاص بـ xAI، والكلمات المفتاحية غير
  المدعومة في schema، و`web_search` الأصلي، وفك ترميز
  معاملات استدعاء الأداة المشفرة بكيانات HTML.
- تستخدم Mistral وOpenCode Zen وOpenCode Go `capabilities` فقط لإبقاء
  خصائص النصوص/الأدوات خارج النواة.
- يستخدم موفرو الحزم المضمّنون المعتمدون على `catalog` فقط مثل `byteplus`،
  و`cloudflare-ai-gateway`، و`huggingface`، و`kimi-coding`، و`nvidia`، و`qianfan`,
  و`synthetic`، و`together`، و`venice`، و`vercel-ai-gateway`، و`volcengine`
  الخطاف `catalog` فقط.
- تستخدم Qwen `catalog` لموفر النص لديها إضافة إلى تسجيلات
  فهم الوسائط وتوليد الفيديو المشتركة لأسطحها متعددة الوسائط.
- تستخدم MiniMax وXiaomi `catalog` إضافة إلى خطافات الاستخدام لأن
  سلوك `/usage` لديهما مملوك للـ plugin رغم أن الاستدلال لا يزال يعمل عبر
  وسائل النقل المشتركة.

## مساعدات وقت التشغيل

يمكن للـ plugins الوصول إلى مساعدات أساسية مختارة عبر `api.runtime`. وبالنسبة إلى TTS:

```ts
const clip = await api.runtime.tts.textToSpeech({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const result = await api.runtime.tts.textToSpeechTelephony({
  text: "Hello from OpenClaw",
  cfg: api.config,
});

const voices = await api.runtime.tts.listVoices({
  provider: "elevenlabs",
  cfg: api.config,
});
```

ملاحظات:

- يعيد `textToSpeech` حمولة خرج TTS العادية من النواة لأسطح
  الملفات/المذكرات الصوتية.
- يستخدم إعداد النواة `messages.tts` واختيار الموفر.
- يعيد PCM audio buffer + sample rate. ويجب على plugins إعادة التعيين/الترميز من أجل الموفّرين.
- `listVoices` اختيارية لكل موفر. استخدمها لمنتقيات الأصوات المملوكة للمورّد أو لتدفقات الإعداد.
- يمكن أن تتضمن قوائم الأصوات بيانات وصفية أغنى مثل locale والجنس وعلامات الشخصية لمنتقيات واعية بالموفر.
- يدعم OpenAI وElevenLabs الاتصالات الهاتفية اليوم. ولا يدعمها Microsoft.

يمكن للـ plugins أيضًا تسجيل موفري الكلام عبر `api.registerSpeechProvider(...)`.

```ts
api.registerSpeechProvider({
  id: "acme-speech",
  label: "Acme Speech",
  isConfigured: ({ config }) => Boolean(config.messages?.tts),
  synthesize: async (req) => {
    return {
      audioBuffer: Buffer.from([]),
      outputFormat: "mp3",
      fileExtension: ".mp3",
      voiceCompatible: false,
    };
  },
});
```

ملاحظات:

- أبقِ سياسة TTS والرجوع الاحتياطي وتسليم الرد داخل النواة.
- استخدم موفري الكلام لسلوك التوليف المملوك للمورّد.
- يُطبَّع إدخال Microsoft القديم `edge` إلى معرّف الموفر `microsoft`.
- نموذج الملكية المفضل موجه للشركة: يمكن لplugin مورّد واحدة أن تمتلك
  موفري النص والكلام والصور والوسائط المستقبلية مع إضافة OpenClaw لعقود هذه الإمكانات.

وبالنسبة إلى فهم الصور/الصوت/الفيديو، تسجل plugins موفر
فهم وسائط واحدًا مكتوبًا بدل حاوية key/value عامة:

```ts
api.registerMediaUnderstandingProvider({
  id: "google",
  capabilities: ["image", "audio", "video"],
  describeImage: async (req) => ({ text: "..." }),
  transcribeAudio: async (req) => ({ text: "..." }),
  describeVideo: async (req) => ({ text: "..." }),
});
```

ملاحظات:

- أبقِ التنسيق والرجوع الاحتياطي والإعداد وربط القنوات في النواة.
- وأبقِ سلوك المورّد في plugin الموفر.
- يجب أن يبقى التوسع الإضافي مكتوبًا: طرق اختيارية جديدة، وحقول نتائج
  اختيارية جديدة، وإمكانات اختيارية جديدة.
- كما أن توليد الفيديو يتبع النمط نفسه بالفعل:
  - تمتلك النواة عقد القدرة ومساعد وقت التشغيل
  - وتسجل plugins الخاصة بالمورّدين `api.registerVideoGenerationProvider(...)`
  - وتستهلك plugins الميزات/القنوات `api.runtime.videoGeneration.*`

وبالنسبة إلى مساعدات وقت تشغيل فهم الوسائط، يمكن للـ plugins استدعاء:

```ts
const image = await api.runtime.mediaUnderstanding.describeImageFile({
  filePath: "/tmp/inbound-photo.jpg",
  cfg: api.config,
  agentDir: "/tmp/agent",
});

const video = await api.runtime.mediaUnderstanding.describeVideoFile({
  filePath: "/tmp/inbound-video.mp4",
  cfg: api.config,
});
```

أما بالنسبة إلى نسخ الصوت، فيمكن للـ plugins استخدام وقت تشغيل فهم الوسائط
أو الاسم القديم STT:

```ts
const { text } = await api.runtime.mediaUnderstanding.transcribeAudioFile({
  filePath: "/tmp/inbound-audio.ogg",
  cfg: api.config,
  // Optional when MIME cannot be inferred reliably:
  mime: "audio/ogg",
});
```

ملاحظات:

- يُعد `api.runtime.mediaUnderstanding.*` السطح المشترك المفضل لـ
  فهم الصور/الصوت/الفيديو.
- يستخدم إعداد النواة الخاص بالصوت في فهم الوسائط (`tools.media.audio`) وترتيب الرجوع الاحتياطي للموفر.
- يعيد `{ text: undefined }` عند عدم إنتاج خرج نسخ (مثلًا عند تخطي الإدخال/عدم دعمه).
- يبقى `api.runtime.stt.transcribeAudioFile(...)` اسمًا بديلًا للتوافق.

يمكن للـ plugins أيضًا إطلاق تشغيلات subagent في الخلفية عبر `api.runtime.subagent`:

```ts
const result = await api.runtime.subagent.run({
  sessionKey: "agent:main:subagent:search-helper",
  message: "Expand this query into focused follow-up searches.",
  provider: "openai",
  model: "gpt-4.1-mini",
  deliver: false,
});
```

ملاحظات:

- الحقلان `provider` و`model` تجاوزات اختيارية لكل تشغيل، وليسا تغييرات دائمة للجلسة.
- لا يحترم OpenClaw حقلي التجاوز هذين إلا للمستدعين الموثوقين.
- بالنسبة إلى التشغيلات الاحتياطية المملوكة للـ plugin، يجب على المشغلين تفعيل ذلك صراحة عبر `plugins.entries.<id>.subagent.allowModelOverride: true`.
- استخدم `plugins.entries.<id>.subagent.allowedModels` لتقييد plugins الموثوقة إلى أهداف قانونية محددة من `provider/model`، أو `"*"` للسماح الصريح بأي هدف.
- لا تزال تشغيلات subagent الخاصة بالـ plugins غير الموثوقة تعمل، لكن طلبات التجاوز تُرفض بدلًا من الرجوع بصمت.

أما بالنسبة إلى بحث الويب، فيمكن للـ plugins استهلاك مساعد وقت التشغيل المشترك بدلًا من
الوصول إلى ربط أداة الوكيل:

```ts
const providers = api.runtime.webSearch.listProviders({
  config: api.config,
});

const result = await api.runtime.webSearch.search({
  config: api.config,
  args: {
    query: "OpenClaw plugin runtime helpers",
    count: 5,
  },
});
```

ويمكن للـ plugins أيضًا تسجيل موفري بحث الويب عبر
`api.registerWebSearchProvider(...)`.

ملاحظات:

- أبقِ اختيار الموفر، وحل بيانات الاعتماد، ودلالات الطلب المشتركة في النواة.
- استخدم موفري بحث الويب لوسائل النقل الخاصة بالمورّد.
- يُعد `api.runtime.webSearch.*` السطح المشترك المفضل لplugins الميزات/القنوات التي تحتاج إلى سلوك البحث دون الاعتماد على مغلف أداة الوكيل.

## مسارات Gateway HTTP

يمكن للـ plugins كشف نقاط نهاية HTTP بواسطة `api.registerHttpRoute(...)`.

```ts
api.registerHttpRoute({
  path: "/acme/webhook",
  auth: "plugin",
  match: "exact",
  handler: async (_req, res) => {
    res.statusCode = 200;
    res.end("ok");
    return true;
  },
});
```

حقول المسار:

- `path`: مسار route ضمن خادم HTTP الخاص بـ gateway.
- `auth`: مطلوب. استخدم `"gateway"` لطلب مصادقة gateway العادية، أو `"plugin"` للمصادقة/التحقق من webhook المُدار بواسطة plugin.
- `match`: اختياري. إما `"exact"` (الافتراضي) أو `"prefix"`.
- `replaceExisting`: اختياري. يسمح للـ plugin نفسها باستبدال تسجيل مسارها الحالي.
- `handler`: أعد `true` عندما يكون المسار قد عالج الطلب.

ملاحظات:

- تمت إزالة `api.registerHttpHandler(...)` وستسبب خطأ في تحميل plugin. استخدم `api.registerHttpRoute(...)` بدلًا منها.
- يجب على مسارات plugins التصريح بـ `auth` صراحة.
- تُرفض تعارضات `path + match` المطابقة تمامًا ما لم تكن `replaceExisting: true`، ولا يمكن لplugin أن تستبدل مسار plugin أخرى.
- تُرفض المسارات المتداخلة ذات مستويات `auth` المختلفة. أبقِ سلاسل السقوط `exact`/`prefix` ضمن مستوى المصادقة نفسه فقط.
- لا تتلقى المسارات ذات `auth: "plugin"` نطاقات وقت تشغيل operator تلقائيًا. فهي مخصصة لـ webhooks/التحقق من التوقيع الذي تديره plugin، لا لاستدعاءات مساعد Gateway ذات الامتيازات.
- تعمل المسارات ذات `auth: "gateway"` داخل نطاق وقت تشغيل لطلب Gateway، لكن هذا النطاق محافظ عمدًا:
  - مصادقة bearer بالسر المشترك (`gateway.auth.mode = "token"` / `"password"`) تُبقي نطاقات وقت تشغيل مسار plugin مثبتة على `operator.write`، حتى لو أرسل المستدعي `x-openclaw-scopes`
  - أوضاع HTTP الموثوقة الحاملة للهوية (مثل `trusted-proxy` أو `gateway.auth.mode = "none"` على ingress خاص) لا تحترم `x-openclaw-scopes` إلا عند وجود هذا header صراحة
  - إذا غاب `x-openclaw-scopes` في طلبات مسار plugin الحاملة للهوية، يعود نطاق وقت التشغيل إلى `operator.write`
- القاعدة العملية: لا تفترض أن مسار plugin المصادق عبر gateway هو سطح إدارة ضمني. إذا كان مسارك يحتاج إلى سلوك إداري فقط، فاطلب وضع مصادقة حاملًا للهوية ووثّق عقد header الصريح `x-openclaw-scopes`.

## مسارات استيراد Plugin SDK

استخدم المسارات الفرعية لـ SDK بدل استيراد `openclaw/plugin-sdk` الأحادي عند
تأليف plugins:

- `openclaw/plugin-sdk/plugin-entry` من أجل عناصر تسجيل plugin الأساسية.
- `openclaw/plugin-sdk/core` من أجل العقد المشترك العام المواجه للـ plugin.
- `openclaw/plugin-sdk/config-schema` من أجل
  تصدير Zod schema الجذرية لـ `openclaw.json` (`OpenClawSchema`).
- العناصر الثابتة الخاصة بالقنوات مثل `openclaw/plugin-sdk/channel-setup`،
  و`openclaw/plugin-sdk/setup-runtime`،
  و`openclaw/plugin-sdk/setup-adapter-runtime`،
  و`openclaw/plugin-sdk/setup-tools`،
  و`openclaw/plugin-sdk/channel-pairing`،
  و`openclaw/plugin-sdk/channel-contract`،
  و`openclaw/plugin-sdk/channel-feedback`،
  و`openclaw/plugin-sdk/channel-inbound`،
  و`openclaw/plugin-sdk/channel-lifecycle`،
  و`openclaw/plugin-sdk/channel-reply-pipeline`،
  و`openclaw/plugin-sdk/command-auth`،
  و`openclaw/plugin-sdk/secret-input`، و
  `openclaw/plugin-sdk/webhook-ingress` من أجل ربط
  الإعداد/المصادقة/الرد/webhook المشترك. ويُعد `channel-inbound` الموطن المشترك لإلغاء
  الارتداد، ومطابقة الإشارة، ومساعدات سياسة الإشارة الواردة، وتنسيق المغلف،
  ومساعدات سياق المغلف الوارد.
  ويُعد `channel-setup` حد الإعداد الاختياري الضيق.
  ويُعد `setup-runtime` سطح إعداد آمنًا لوقت التشغيل يُستخدم بواسطة `setupEntry` /
  بدء التشغيل المؤجل، بما في ذلك محولات ترقيع الإعداد الآمنة عند الاستيراد.
  ويُعد `setup-adapter-runtime` حد محول إعداد الحساب الواعي بالـ env.
  ويُعد `setup-tools` حدًا صغيرًا لمساعدات CLI/archive/docs (`formatCliCommand`,
  و`detectBinary`، و`extractArchive`، و`resolveBrewExecutable`، و`formatDocsLink`,

- المسارات الخاصة بالمجال مثل `openclaw/plugin-sdk/channel-config-helpers`،
  و`openclaw/plugin-sdk/allow-from`،
  و`openclaw/plugin-sdk/channel-config-schema`,
  و`openclaw/plugin-sdk/telegram-command-config`,
  و`openclaw/plugin-sdk/channel-policy`,
  و`openclaw/plugin-sdk/approval-runtime`,
  و`openclaw/plugin-sdk/config-runtime`,
  و`openclaw/plugin-sdk/infra-runtime`,
  و`openclaw/plugin-sdk/agent-runtime`,
  و`openclaw/plugin-sdk/lazy-runtime`,
  و`openclaw/plugin-sdk/reply-history`,
  و`openclaw/plugin-sdk/routing`,
  و`openclaw/plugin-sdk/status-helpers`,
  و`openclaw/plugin-sdk/text-runtime`,
  و`openclaw/plugin-sdk/runtime-store`، و
  `openclaw/plugin-sdk/directory-runtime` من أجل مساعدات وقت التشغيل/الإعداد المشتركة.
  ويُعد `telegram-command-config` الحد العام الضيق لتطبيع/التحقق من
  الأوامر المخصصة في Telegram ويبقى متاحًا حتى إذا كان سطح عقد Telegram المضمّن
  غير متاح مؤقتًا.
  ويُعد `text-runtime` الحد المشترك للنص/markdown/التسجيل، بما في ذلك
  إزالة النصوص المرئية للمساعد، ومساعدات عرض/chunking في markdown، ومساعدات الإخفاء،
  ومساعدات directive-tag، وأدوات النص الآمن.
- يجب أن تفضّل حدود القنوات الخاصة بالموافقة عقد `approvalCapability`
  واحدًا على plugin. ثم تقرأ النواة مصادقة الموافقة وتسليمها وعرضها وتوجيهها
  الأصلي عبر هذه القدرة الواحدة بدل خلط سلوك الموافقة في حقول plugin غير ذات الصلة.
- إن `openclaw/plugin-sdk/channel-runtime` مهمل ويظل فقط كشريحة توافق
  للـ plugins الأقدم. ويجب على الشيفرة الجديدة استيراد العناصر العامة الأضيق بدلًا منه،
  ويجب على شيفرة المستودع ألا تضيف استيرادات جديدة من هذه الشريحة.
- تبقى أساسيات الامتدادات المضمّنة خاصة. ويجب على plugins الخارجية استخدام
  المسارات الفرعية `openclaw/plugin-sdk/*` فقط. ويمكن لشيفرة/اختبارات النواة في OpenClaw استخدام
  نقاط الدخول العامة في المستودع تحت جذر حزمة plugin مثل `index.js`، و`api.js`،
  و`runtime-api.js`، و`setup-entry.js`، وملفات ضيقة النطاق مثل
  `login-qr-api.js`. لا تستورد أبدًا `src/*` التابعة لحزمة plugin من النواة أو من امتداد آخر.
- تقسيم نقطة الدخول في المستودع:
  `<plugin-package-root>/api.js` هو شريط المساعدات/الأنواع،
  و`<plugin-package-root>/runtime-api.js` هو شريط وقت التشغيل فقط،
  و`<plugin-package-root>/index.js` هو مدخل plugin المضمّنة،
  و`<plugin-package-root>/setup-entry.js` هو مدخل plugin الخاص بالإعداد.
- أمثلة الموفّرين المضمّنة الحالية:
  - تستخدم Anthropic `api.js` / `contract-api.js` لمساعدات بث Claude مثل
    `wrapAnthropicProviderStream`، ومساعدات beta-header، وتحليل `service_tier`.
  - تستخدم OpenAI `api.js` لبناة الموفر، ومساعدات النموذج الافتراضي، وبناة
    الموفّر الفوري.
  - تستخدم OpenRouter `api.js` لباني الموفر الخاص بها إضافة إلى مساعدات
    onboarding/config، في حين لا يزال بإمكان `register.runtime.js`
    إعادة تصدير المساعدات العامة `plugin-sdk/provider-stream` للاستخدام المحلي في المستودع.
- تفضّل نقاط الدخول العامة المحمّلة عبر facade لقطة إعداد وقت التشغيل النشطة
  عند وجودها، ثم تعود إلى ملف الإعداد المحلول على القرص عندما
  لا يكون OpenClaw قد بدأ بعد في تقديم لقطة وقت تشغيل.
- تبقى العناصر العامة المشتركة هي عقد SDK العامة المفضلة. ولا تزال توجد
  مجموعة توافقية صغيرة ومحجوزة من حدود المساعدات ذات العلامات التجارية للقنوات المضمّنة.
  تعامل مع هذه على أنها حدود للصيانة/التوافق للـ plugins المضمّنة، لا كأهداف استيراد
  جديدة للجهات الخارجية؛ ويجب أن تهبط العقود المشتركة الجديدة بين القنوات
  على مسارات `plugin-sdk/*` العامة أو على الأشرطة المحلية `api.js` /
  `runtime-api.js` الخاصة بالـ plugin.

ملاحظة توافق:

- تجنّب الشريط الجذري `openclaw/plugin-sdk` في الشيفرة الجديدة.
- وفضّل أولًا العناصر الثابتة الضيقة. فالمسارات الفرعية الأحدث الخاصة بالإعداد/الاقتران/الرد/
  feedback/contract/inbound/threading/command/secret-input/webhook/infra/
  allowlist/status/message-tool هي العقد المقصود للأعمال الجديدة على plugins
  المضمّنة والخارجية.
  ويجب أن ينتمي تحليل/مطابقة الأهداف إلى `openclaw/plugin-sdk/channel-targets`.
  كما يجب أن تنتمي بوابات إجراءات الرسائل ومساعدات معرّف رسالة التفاعل إلى
  `openclaw/plugin-sdk/channel-actions`.
- لا تُعد أشرطة المساعدات الخاصة بالامتدادات المضمّنة مستقرة افتراضيًا. إذا كانت
  المساعدة مطلوبة فقط لامتداد مضمّن، فأبقها خلف الحد المحلي `api.js` أو `runtime-api.js`
  الخاص بذلك الامتداد بدل ترقيتها إلى `openclaw/plugin-sdk/<extension>`.
- يجب أن تكون حدود المساعدات المشتركة الجديدة عامة لا موسومة باسم قناة. فتحليل
  الأهداف المشتركة ينتمي إلى `openclaw/plugin-sdk/channel-targets`؛ بينما تبقى
  أساسيات القنوات الخاصة وراء الحد المحلي `api.js` أو `runtime-api.js`
  المملوك للـ plugin.
- توجد مسارات فرعية خاصة بالإمكانات مثل `image-generation`,
  و`media-understanding`، و`speech` لأن plugins الأصلية/المضمّنة تستخدمها اليوم.
  ولا يعني وجودها بحد ذاته أن كل مساعد مُصدَّر هو عقد خارجي
  طويل الأمد ومجمّد.

## Schemas أداة الرسائل

يجب أن تمتلك plugins مساهمات schema الخاصة بـ `describeMessageTool(...)`
والخاصة بالقناة. وأبقِ الحقول الخاصة بالمورّد في plugin، لا في النواة المشتركة.

وبالنسبة إلى أجزاء schema المحمولة المشتركة، أعد استخدام المساعدات العامة المصدّرة عبر
`openclaw/plugin-sdk/channel-actions`:

- `createMessageToolButtonsSchema()` لحِمولات شبكات الأزرار
- `createMessageToolCardSchema()` لحِمولات البطاقات المنظمة

إذا كان شكل schema لا معنى له إلا لموفر واحد، فعرّفه في
مصدر تلك plugin نفسها بدل ترقيته إلى SDK المشتركة.

## حل أهداف القنوات

يجب أن تمتلك Plugins القنوات دلالات الأهداف الخاصة بالقناة. أبقِ
المضيف المشترك للرسائل الصادرة عامًا، واستخدم سطح محول المراسلة لقواعد
الموفر:

- يقرر `messaging.inferTargetChatType({ to })` ما إذا كان ينبغي معاملة هدف مطبّع
  على أنه `direct` أو `group` أو `channel` قبل البحث في الدليل.
- يحدد `messaging.targetResolver.looksLikeId(raw, normalized)` للنواة ما إذا كان ينبغي
  تخطي البحث مباشرة إلى حل شبيه بالمعرّف بدلًا من البحث في الدليل.
- تُعد `messaging.targetResolver.resolveTarget(...)` آلية الرجوع الخاصة بالـ plugin عندما
  تحتاج النواة إلى حل نهائي مملوك للموفر بعد التطبيع أو بعد فشل الدليل.
- وتمتلك `messaging.resolveOutboundSessionRoute(...)` إنشاء مسار الجلسة الصادر
  الخاص بالموفر بعد حل الهدف.

التقسيم الموصى به:

- استخدم `inferTargetChatType` لقرارات الفئة التي يجب أن تحدث قبل
  البحث في الأقران/المجموعات.
- واستخدم `looksLikeId` لفحوص "تعامل مع هذا على أنه معرّف هدف أصلي/صريح".
- واستخدم `resolveTarget` كرجوع خاص بالتطبيع عند الموفر، لا
  لبحث واسع في الدليل.
- وأبقِ المعرّفات الأصلية للمورّد مثل معرّفات الدردشة ومعرّفات السلاسل وJIDs
  والمقابض ومعرّفات الغرف داخل قيم `target` أو المعاملات الخاصة بالمورّد، لا في
  حقول SDK العامة.

## الأدلة المدعومة بالإعداد

يجب أن تُبقي plugins التي تستمد إدخالات الدليل من الإعداد هذا المنطق في
plugin نفسها، وأن تعيد استخدام المساعدات المشتركة من
`openclaw/plugin-sdk/directory-runtime`.

استخدم هذا عندما تحتاج قناة إلى أقران/مجموعات مدعومة بالإعداد مثل:

- أقران الرسائل المباشرة المدفوعون بقائمة السماح
- خرائط القنوات/المجموعات المضبوطة
- بدائل دليل ثابتة خاصة بالحساب

تعالج المساعدات المشتركة في `directory-runtime` العمليات العامة فقط:

- ترشيح الاستعلام
- تطبيق الحد
- إزالة التكرار/التطبيع
- بناء `ChannelDirectoryEntry[]`

أما فحص الحسابات الخاصة بالقناة وتطبيع المعرّفات فيجب أن يبقيا داخل
تنفيذ plugin.

## فهارس الموفّرين

يمكن Plugins الموفّر تعريف فهارس النماذج للاستدلال بواسطة
`registerProvider({ catalog: { run(...) { ... } } })`.

تُعيد `catalog.run(...)` الشكل نفسه الذي يكتبه OpenClaw إلى
`models.providers`:

- `{ provider }` لإدخال موفر واحد
- `{ providers }` لعدة إدخالات موفرين

استخدم `catalog` عندما تمتلك plugin معرّفات نماذج خاصة بالموفر، أو إعدادات
افتراضية لـ base URL، أو بيانات وصفية للنموذج محكومة بالمصادقة.

تتحكم `catalog.order` في توقيت دمج فهرس plugin نسبةً إلى الموفّرين
الضمنيين المضمّنين في OpenClaw:

- `simple`: موفرو API-key أو env البسطاء
- `profile`: الموفرون الذين يظهرون عند وجود ملفات مصادقة
- `paired`: الموفرون الذين يصطنعون عدة إدخالات موفّرين مرتبطة
- `late`: التمريرة الأخيرة بعد الموفّرين الضمنيين الآخرين

يربح الموفّر المتأخر عند تعارض المفاتيح، لذا يمكن للـ plugins عمدًا تجاوز
إدخال موفّر مضمّن له معرّف الموفّر نفسه.

التوافق:

- لا يزال `discovery` يعمل كاسم قديم بديل
- إذا تم تسجيل كل من `catalog` و`discovery`، يستخدم OpenClaw `catalog`

## فحص القنوات للقراءة فقط

إذا كانت plugin الخاصة بك تسجل قناة، فافضّل تنفيذ
`plugin.config.inspectAccount(cfg, accountId)` إلى جانب `resolveAccount(...)`.

السبب:

- `resolveAccount(...)` هو مسار وقت التشغيل. ويُسمح له بافتراض أن بيانات الاعتماد
  قد تحققت بالكامل، ويمكنه الفشل سريعًا عند غياب الأسرار المطلوبة.
- لا ينبغي لمسارات الأوامر المخصصة للقراءة فقط مثل `openclaw status`، و`openclaw status --all`،
  و`openclaw channels status`، و`openclaw channels resolve`، وتدفقات doctor/config
  للإصلاح أن تحتاج إلى تفعيل بيانات اعتماد وقت التشغيل فقط من أجل
  وصف الإعداد.

السلوك الموصى به لـ `inspectAccount(...)`:

- أعد فقط حالة وصفية للحساب.
- حافظ على `enabled` و`configured`.
- أدرج حقول مصدر/حالة بيانات الاعتماد عند الصلة، مثل:
  - `tokenSource`، `tokenStatus`
  - `botTokenSource`، `botTokenStatus`
  - `appTokenSource`، `appTokenStatus`
  - `signingSecretSource`، `signingSecretStatus`
- لا تحتاج إلى إعادة قيم الرموز الخام فقط للإبلاغ عن
  التوفر للقراءة فقط. يكفي إرجاع `tokenStatus: "available"` (مع
  حقل المصدر المطابق) لأوامر على نمط الحالة.
- استخدم `configured_unavailable` عندما تكون بيانات الاعتماد مضبوطة عبر SecretRef لكنها
  غير متاحة في مسار الأمر الحالي.

وهذا يتيح للأوامر المخصصة للقراءة فقط الإبلاغ عن "مضبوطة لكنها غير متاحة في
مسار هذا الأمر" بدل التعطل أو الإبلاغ الخاطئ عن أن الحساب غير مضبوط.

## حِزم الحِزم

يمكن أن يحتوي دليل plugin على `package.json` يتضمن `openclaw.extensions`:

```json
{
  "name": "my-pack",
  "openclaw": {
    "extensions": ["./src/safety.ts", "./src/tools.ts"],
    "setupEntry": "./src/setup-entry.ts"
  }
}
```

يصبح كل إدخال plugin. وإذا كانت الحزمة تسرد عدة امتدادات، يصبح معرّف plugin
`name/<fileBase>`.

إذا كانت plugin الخاصة بك تستورد تبعيات npm، فثبّتها في ذلك الدليل حتى
يكون `node_modules` متاحًا (`npm install` / `pnpm install`).

حاجز أمان: يجب أن يبقى كل إدخال `openclaw.extensions` داخل دليل plugin
بعد حل الروابط الرمزية. وتُرفض الإدخالات التي تهرب من دليل الحزمة.

ملاحظة أمان: يقوم `openclaw plugins install` بتثبيت تبعيات plugin بواسطة
`npm install --omit=dev --ignore-scripts` (بدون lifecycle scripts وبدون تبعيات تطوير وقت التشغيل). حافظ على أشجار تبعيات plugin
"JS/TS خالصة" وتجنب الحزم التي تتطلب بناءً عبر `postinstall`.

اختياريًا: يمكن أن يشير `openclaw.setupEntry` إلى وحدة خفيفة مخصصة للإعداد فقط.
وعندما يحتاج OpenClaw إلى أسطح إعداد لـ channel plugin معطّلة، أو
عندما تكون channel plugin مفعّلة لكنها غير مضبوطة بعد، فإنه يحمّل `setupEntry`
بدل مدخل plugin الكامل. وهذا يبقي بدء التشغيل والإعداد أخف
عندما يكون المدخل الرئيسي للـ plugin يوصل أيضًا أدوات أو خطافات أو شيفرة
أخرى مخصصة لوقت التشغيل فقط.

اختياريًا: يمكن لـ
`openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen`
إدخال channel plugin في مسار `setupEntry` نفسه خلال مرحلة بدء التشغيل
قبل الاستماع في gateway، حتى عندما تكون القناة مضبوطة بالفعل.

استخدم هذا فقط عندما يغطي `setupEntry` بالكامل سطح بدء التشغيل الذي يجب أن
يوجد قبل أن تبدأ gateway في الاستماع. وعمليًا، يعني هذا أن مدخل الإعداد
يجب أن يسجل كل قدرة مملوكة للقناة يعتمد عليها بدء التشغيل، مثل:

- تسجيل القناة نفسها
- أي مسارات HTTP يجب أن تكون متاحة قبل أن تبدأ gateway في الاستماع
- أي أساليب Gateway أو أدوات أو خدمات يجب أن توجد خلال هذه النافذة نفسها

إذا كان المدخل الكامل لا يزال يمتلك أي قدرة مطلوبة عند بدء التشغيل، فلا تفعّل
هذا العلم. أبقِ plugin على السلوك الافتراضي ودع OpenClaw يحمّل
المدخل الكامل أثناء بدء التشغيل.

يمكن للقنوات المضمّنة أيضًا نشر مساعدات سطح عقد للإعداد فقط يمكن للنواة
الرجوع إليها قبل تحميل وقت تشغيل القناة الكامل. وسطح الترقية الحالي للإعداد هو:

- `singleAccountKeysToMove`
- `namedAccountPromotionKeys`
- `resolveSingleAccountPromotionTarget(...)`

تستخدم النواة هذا السطح عندما تحتاج إلى ترقية إعداد قناة قديم أحادي الحساب
إلى `channels.<id>.accounts.*` بدون تحميل مدخل plugin الكامل.
وتُعد Matrix المثال المضمّن الحالي: فهي تنقل فقط مفاتيح المصادقة/التمهيد إلى
حساب مُرقّى مسمى عندما تكون الحسابات المسماة موجودة بالفعل، ويمكنها
الحفاظ على مفتاح حساب افتراضي غير قانوني مضبوط بدل إنشاء
`accounts.default` دائمًا.

تُبقي محولات ترقيع الإعداد هذه اكتشاف سطح العقد المضمّن كسولًا. فيبقى
وقت الاستيراد خفيفًا؛ ولا يُحمّل سطح الترقية إلا عند أول استخدام بدل
إعادة الدخول إلى بدء تشغيل القناة المضمّنة عند استيراد الوحدة.

عندما تتضمن أسطح بدء التشغيل هذه أساليب Gateway RPC، فأبقها على
بادئة خاصة بالـ plugin. تبقى مساحات أسماء الإدارة الأساسية (`config.*`,
`exec.approvals.*`, `wizard.*`, `update.*`) محجوزة وتحل دائمًا
إلى `operator.admin` حتى إذا طلبت plugin نطاقًا أضيق.

مثال:

```json
{
  "name": "@scope/my-channel",
  "openclaw": {
    "extensions": ["./index.ts"],
    "setupEntry": "./setup-entry.ts",
    "startup": {
      "deferConfiguredChannelFullLoadUntilAfterListen": true
    }
  }
}
```

### بيانات فهرس القناة الوصفية

يمكن Plugins القنوات الإعلان عن بيانات وصفية للإعداد/الاكتشاف عبر `openclaw.channel` و
تلميحات التثبيت عبر `openclaw.install`. وهذا يُبقي بيانات الفهرس في النواة خالية من البيانات الصلبة.

مثال:

```json
{
  "name": "@openclaw/nextcloud-talk",
  "openclaw": {
    "extensions": ["./index.ts"],
    "channel": {
      "id": "nextcloud-talk",
      "label": "Nextcloud Talk",
      "selectionLabel": "Nextcloud Talk (self-hosted)",
      "docsPath": "/channels/nextcloud-talk",
      "docsLabel": "nextcloud-talk",
      "blurb": "Self-hosted chat via Nextcloud Talk webhook bots.",
      "order": 65,
      "aliases": ["nc-talk", "nc"]
    },
    "install": {
      "npmSpec": "@openclaw/nextcloud-talk",
      "localPath": "<bundled-plugin-local-path>",
      "defaultChoice": "npm"
    }
  }
}
```

حقول `openclaw.channel` المفيدة إضافة إلى المثال الأدنى:

- `detailLabel`: تسمية ثانوية لأسطح الفهرس/الحالة الأكثر غنى
- `docsLabel`: تجاوز نص الرابط الخاص بالمستندات
- `preferOver`: معرّفات plugins/قنوات أقل أولوية يجب أن يتفوق عليها إدخال الفهرس هذا
- `selectionDocsPrefix`، و`selectionDocsOmitLabel`، و`selectionExtras`: أدوات تحكم نصية لأسطح الاختيار
- `markdownCapable`: تضع علامة على القناة بأنها قادرة على markdown من أجل قرارات التنسيق الصادر
- `exposure.configured`: إخفاء القناة من أسطح سرد القنوات المضبوطة عند تعيينها إلى `false`
- `exposure.setup`: إخفاء القناة من منتقيات الإعداد/التهيئة التفاعلية عند تعيينها إلى `false`
- `exposure.docs`: وسم القناة على أنها داخلية/خاصة لأسطح التنقل في المستندات
- `showConfigured` / `showInSetup`: أسماء قديمة بديلة لا تزال مقبولة للتوافق؛ يفضّل `exposure`
- `quickstartAllowFrom`: يدرج القناة ضمن تدفق `allowFrom` القياسي للبدء السريع
- `forceAccountBinding`: يفرض ربط الحساب صراحةً حتى عند وجود حساب واحد فقط
- `preferSessionLookupForAnnounceTarget`: يفضّل البحث في الجلسة عند حل أهداف الإعلانات

يمكن لـ OpenClaw أيضًا دمج **فهارس قنوات خارجية** (مثلًا، تصدير سجل MPM).
ضع ملف JSON في أحد المواقع التالية:

- `~/.openclaw/mpm/plugins.json`
- `~/.openclaw/mpm/catalog.json`
- `~/.openclaw/plugins/catalog.json`

أو وجّه `OPENCLAW_PLUGIN_CATALOG_PATHS` (أو `OPENCLAW_MPM_CATALOG_PATHS`) إلى
ملف JSON واحد أو أكثر (مفصولة بفاصلة/فاصلة منقوطة/`PATH`). ويجب أن يحتوي كل ملف
على `{ "entries": [ { "name": "@scope/pkg", "openclaw": { "channel": {...}, "install": {...} } } ] }`. كما يقبل المحلل
القيمتين `"packages"` أو `"plugins"` كأسماء قديمة بديلة للمفتاح `"entries"`.

## Plugins محرك السياق

تمتلك Plugins محرك السياق تنسيق سياق الجلسة من أجل الإدخال والتجميع
والضغط. سجّلها من plugin الخاصة بك عبر
`api.registerContextEngine(id, factory)`، ثم اختر المحرك النشط بواسطة
`plugins.slots.contextEngine`.

استخدم هذا عندما تحتاج plugin الخاصة بك إلى استبدال خط أنابيب السياق الافتراضي أو
توسيعه بدل مجرد إضافة بحث ذاكرة أو خطافات.

```ts
export default function (api) {
  api.registerContextEngine("lossless-claw", () => ({
    info: { id: "lossless-claw", name: "Lossless Claw", ownsCompaction: true },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages }) {
      return { messages, estimatedTokens: 0 };
    },
    async compact() {
      return { ok: true, compacted: false };
    },
  }));
}
```

إذا كان المحرك **لا** يمتلك خوارزمية الضغط، فأبقِ `compact()`
منفذة وفوّضها صراحةً:

```ts
import { delegateCompactionToRuntime } from "openclaw/plugin-sdk/core";

export default function (api) {
  api.registerContextEngine("my-memory-engine", () => ({
    info: {
      id: "my-memory-engine",
      name: "My Memory Engine",
      ownsCompaction: false,
    },
    async ingest() {
      return { ingested: true };
    },
    async assemble({ messages }) {
      return { messages, estimatedTokens: 0 };
    },
    async compact(params) {
      return await delegateCompactionToRuntime(params);
    },
  }));
}
```

## إضافة قدرة جديدة

عندما تحتاج plugin إلى سلوك لا يناسب API الحالية، فلا تتجاوز
نظام plugins عبر وصول خاص. أضف القدرة المفقودة.

التسلسل الموصى به:

1. تعريف عقد النواة
   حدّد السلوك المشترك الذي يجب أن تمتلكه النواة: السياسة، والرجوع الاحتياطي، ودمج الإعداد،
   ودورة الحياة، والدلالات المواجهة للقنوات، وشكل مساعد وقت التشغيل.
2. إضافة أسطح تسجيل/وقت تشغيل مكتوبة للـ plugins
   وسّع `OpenClawPluginApi` و/أو `api.runtime` بأصغر
   سطح قدرة مكتوب مفيد.
3. ربط مستهلكي النواة + القنوات/الميزات
   يجب أن تستهلك القنوات وplugins الميزات القدرة الجديدة عبر النواة،
   لا عبر استيراد تنفيذ مورّد مباشرة.
4. تسجيل تنفيذات المورّدين
   ثم تسجل plugins الخاصة بالمورّدين خلفياتها مقابل هذه القدرة.
5. إضافة تغطية للعقد
   أضف اختبارات حتى تبقى الملكية وشكل التسجيل صريحين مع مرور الوقت.

هكذا يحافظ OpenClaw على توجهه الواضح دون أن يصبح مقيّدًا بنظرة
موفّر واحد للعالم. راجع [Capability Cookbook](/ar/plugins/architecture)
للحصول على قائمة ملفات عملية ومثال تطبيقي.

### قائمة التحقق من القدرة

عند إضافة قدرة جديدة، ينبغي عادةً أن يلمس التنفيذ هذه
الأسطح معًا:

- أنواع عقد النواة في `src/<capability>/types.ts`
- مساعد المشغّل/وقت التشغيل في النواة في `src/<capability>/runtime.ts`
- سطح تسجيل API الخاص بالـ plugin في `src/plugins/types.ts`
- ربط سجل plugin في `src/plugins/registry.ts`
- كشف وقت تشغيل plugin في `src/plugins/runtime/*` عندما تحتاج
  plugins الميزات/القنوات إلى استهلاكه
- مساعدات الالتقاط/الاختبار في `src/test-utils/plugin-registration.ts`
- تأكيدات الملكية/العقد في `src/plugins/contracts/registry.ts`
- مستندات المشغّل/الـ plugin في `docs/`

إذا كان أحد هذه الأسطح مفقودًا، فغالبًا ما يكون ذلك إشارة إلى أن القدرة
لم تُدمج بالكامل بعد.

### قالب القدرة

النمط الأدنى:

```ts
// core contract
export type VideoGenerationProviderPlugin = {
  id: string;
  label: string;
  generateVideo: (req: VideoGenerationRequest) => Promise<VideoGenerationResult>;
};

// plugin API
api.registerVideoGenerationProvider({
  id: "openai",
  label: "OpenAI",
  async generateVideo(req) {
    return await generateOpenAiVideo(req);
  },
});

// shared runtime helper for feature/channel plugins
const clip = await api.runtime.videoGeneration.generate({
  prompt: "Show the robot walking through the lab.",
  cfg,
});
```

نمط اختبار العقد:

```ts
expect(findVideoGenerationProviderIdsForPlugin("openai")).toEqual(["openai"]);
```

وهذا يُبقي القاعدة بسيطة:

- تمتلك النواة عقد القدرة + التنسيق
- وتمتلك plugins الخاصة بالمورّدين تنفيذات المورّد
- وتستهلك plugins الميزات/القنوات مساعدات وقت التشغيل
- وتحافظ اختبارات العقد على صراحة الملكية
