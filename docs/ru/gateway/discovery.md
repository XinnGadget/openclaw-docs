# Обнаружение и транспортные механизмы

В OpenClaw есть две отдельные задачи, которые на первый взгляд похожи:

1. **Удалённое управление оператором**: приложение в строке меню macOS управляет шлюзом, работающим на другом устройстве.
2. **Сопряжение узлов**: устройства на iOS/Android (и будущие узлы) находят шлюз и безопасно сопрягаются с ним.

Цель проектирования — оставить всё обнаружение и рекламу в сети в **узле-шлюзе** (`openclaw gateway`), а клиенты (приложение для macOS, iOS) сделать потребителями данных.

## Термины

- **Шлюз (Gateway)**: единственный долго работающий процесс шлюза, который управляет состоянием (сессиями, сопряжением, реестром узлов) и запускает каналы. В большинстве конфигураций используется один шлюз на хост; возможны изолированные конфигурации с несколькими шлюзами.
- **Gateway WS (плоскость управления, control plane)**: endpoint WebSocket по умолчанию на `127.0.0.1:18789`; может быть привязан к локальной сети/tailnet через `gateway.bind`.
- **Прямой транспорт WS (Direct WS transport)**: endpoint Gateway WS, доступный в локальной сети/tailnet (без SSH).
- **Транспорт через SSH (SSH transport, запасной вариант)**: удалённое управление путём переадресации `127.0.0.1:18789` через SSH.
- **Устаревший TCP-мост (Legacy TCP bridge, удалён)**: старый транспорт для узлов (см. [Протокол моста](/gateway/bridge-protocol)); больше не рекламируется для обнаружения и не входит в текущие сборки.

Детали протоколов:

- [Протокол шлюза](/gateway/protocol)
- [Протокол моста (устаревший)](/gateway/bridge-protocol)

## Почему мы сохраняем и "прямое подключение", и SSH

- **Direct WS** — наилучший пользовательский опыт в пределах одной сети и tailnet:
  - автоматическое обнаружение в локальной сети через Bonjour;
  - токены сопряжения + ACL, управляемые шлюзом;
  - не требуется доступ к оболочке; поверхность протокола может оставаться компактной и поддающейся аудиту.
- **SSH** остаётся универсальным запасным вариантом:
  - работает везде, где есть доступ по SSH (даже в несвязанных сетях);
  - устойчив к проблемам с multicast/mDNS;
  - не требует открытия новых входящих портов, кроме SSH.

## Источники обнаружения (как клиенты узнают, где находится шлюз)

### 1) Обнаружение через Bonjour / DNS-SD

Multicast Bonjour работает по принципу "best-effort" и не пересекает границы сетей. OpenClaw также может просматривать тот же маячок шлюза через настроенную широковещательную область DNS-SD, поэтому обнаружение может охватывать:

- `local.` в пределах одной локальной сети;
- настроенную одноадресную область DNS-SD для обнаружения в разных сетях.

Направление работы:

- **Шлюз** рекламирует свой endpoint WS через Bonjour.
- Клиенты просматривают и отображают список "выберите шлюз", затем сохраняют выбранный endpoint.

Устранение неполадок и детали маячка: [Bonjour](/gateway/bonjour).

#### Детали маячка службы

- Типы служб:
  - `_openclaw-gw._tcp` (маячок транспорта шлюза).
- Ключи TXT (несекретные):
  - `role=gateway`;
  - `transport=gateway`;
  - `displayName=<удобное имя>` (имя для отображения, задаваемое оператором);
  - `lanHost=<hostname>.local`;
  - `gatewayPort=18789` (Gateway WS + HTTP);
  - `gatewayTls=1` (только при включённом TLS);
  - `gatewayTlsSha256=<sha256>` (только при включённом TLS и наличии отпечатка);
  - `canvasPort=<port>` (порт хоста canvas; в настоящее время совпадает с `gatewayPort`, если хост canvas включён);
  - `tailnetDns=<magicdns>` (опциональная подсказка; определяется автоматически при наличии Tailscale);
  - `sshPort=<port>` (только в полном режиме mDNS; в широковещательной области DNS-SD может отсутствовать, в этом случае порт SSH по умолчанию остаётся `22`);
  - `cliPath=<path>` (только в полном режиме mDNS; в широковещательной области DNS-SD всё ещё записывается как подсказка для удалённой установки).

Примечания по безопасности:

- Записи TXT в Bonjour/mDNS **не аутентифицированы**. Клиенты должны рассматривать значения TXT только как подсказки для пользовательского интерфейса.
- При маршрутизации (хост/порт) следует отдавать предпочтение **разрешённому endpoint службы** (SRV + A/AAAA), а не указанным в TXT `lanHost`, `tailnetDns` или `gatewayPort`.
- При закреплении TLS никогда не следует разрешать, чтобы указанный `gatewayTlsSha256` переопределял ранее сохранённый отпечаток.
- Узлы на iOS/Android должны требовать явного подтверждения "доверить этот отпечаток" перед сохранением первого отпечатка (внеполосная проверка), если выбранный маршрут является безопасным/на основе TLS.

Отключение/переопределение:

- `OPENCLAW_DISABLE_BONJOUR=1` отключает рекламу.
- `gateway.bind` в `~/.openclaw/openclaw.json` управляет режимом привязки шлюза.
- `OPENCLAW_SSH_PORT` переопределяет порт SSH, указанный при выдаче `sshPort`.
- `OPENCLAW_TAILNET_DNS` публикует подсказку `tailnetDns` (MagicDNS).
- `OPENCLAW_CLI_PATH` переопределяет указанный путь к CLI.

### 2) Tailnet (межсетевое обнаружение)

Для конфигураций в стиле London/Vienna Bonjour не поможет. Рекомендуемая цель для "прямого" подключения:

- Имя MagicDNS Tailscale (предпочтительно) или стабильный IP-адрес tailnet.

Если шлюз может определить, что он работает в Tailscale, он публикует `tailnetDns` в качестве дополнительной подсказки для клиентов (включая широковещательные маячки).

Приложение для macOS теперь предпочитает имена MagicDNS сырым IP-адресам Tailscale для обнаружения шлюза. Это повышает надёжность при изменении IP-адресов tailnet (например, после перезапуска узла или переназначения CGNAT), поскольку имена MagicDNS автоматически разрешаются в текущий IP-адрес.

Для сопряжения мобильных узлов подсказки обнаружения не ослабляют безопасность транспорта в tailnet/публичных маршрутах:

- iOS/Android по-прежнему требуют безопасного пути первого подключения в tailnet/публичной сети (`wss://` или Tailscale Serve/Funnel).
- Обнаруженный сырой IP-адрес tailnet — это подсказка для маршрутизации, а не разрешение на использование незашифрованного удалённого `ws://`.
- Прямое подключение в частной локальной сети через `ws://` по-прежнему поддерживается.
- Если вам нужен самый простой путь Tailscale для мобильных узлов, используйте Tailscale Serve, чтобы обнаружение и код настройки разрешались в один и тот же защищённый endpoint MagicDNS.

### 3) Ручное подключение / цель через SSH

Когда нет прямого маршрута (или прямое подключение отключено), клиенты всегда могут подключиться через SSH, переадресовав порт шлюза на loopback.

См. [Удалённый доступ](/gateway/remote).

## Выбор транспорта (политика клиента)

Рекомендуемое поведение клиента:

1. Если настроен и доступен сопряжённый прямой endpoint, используйте его.
2. В противном случае, если обнаружение находит шлюз в `local.` или в настроенной широковещательной области, предложите вариант "Использовать этот шлюз" в одно касание и сохраните его как прямой endpoint.
3. В противном случае, если настроен DNS/IP tailnet, попробуйте прямое подключение.
   Для мобильных узлов в tailnet/публичных маршрутах прямое подключение означает защищённый endpoint, а не незашифрованный удалённый `ws://`.
4. В противном случае перейдите к SSH.

## Сопряжение и аутентификация (прямой транспорт)

Шлюз является источником достоверной информации о допуске узлов/клиентов.

- Запросы на сопряжение создаются, одобряются или отклоняются на шлюзе (см. [Сопряжение шлюза](/gateway/pairing)).
- Шлюз обеспечивает:
  - аутентификацию (токен / пара ключей);
  - области действия/ACL (шлюз не является простым прокси для каждого метода);
  - ограничения по частоте запросов.

## Обязанности компонентов

- **Шлюз (Gateway)**: рекламирует маячки обнаружения, принимает решения о сопряжении и размещает endpoint WS.
- **Приложение для macOS**: помогает выбрать шлюз, отображает подсказки для сопряжения и использует SSH только как запасной вариант.
- **Узлы на iOS/Android**: просматривают Bonjour для удобства и подключаются к сопряжённому Gateway WS.
