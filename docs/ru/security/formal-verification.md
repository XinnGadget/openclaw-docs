# Формальная верификация (модели безопасности)

На этой странице представлены **формальные модели безопасности** OpenClaw (на данный момент — TLA+/TLC; в дальнейшем — по необходимости).

> Примечание: некоторые старые ссылки могут относиться к предыдущему названию проекта.

**Цель (ориентир):** предоставить проверяемый машиной аргумент того, что OpenClaw соблюдает заданную политику безопасности (авторизацию, изоляцию сессий, контроль инструментов и защиту от неправильной конфигурации) при явных допущениях.

**Что это (на данный момент):** исполняемый набор тестов на регрессию безопасности, ориентированный на действия злоумышленника:

- Для каждого утверждения существует исполняемая проверка модели в конечном пространстве состояний.
- Для многих утверждений имеется соответствующая **негативная модель**, которая генерирует трассировку контрпримера для реалистичного класса ошибок.

**Чего это пока нет:** доказательства того, что "OpenClaw безопасен во всех отношениях" или что полная реализация на TypeScript корректна.

## Где находятся модели

Модели поддерживаются в отдельном репозитории: [vignesh07/openclaw-formal-models](https://github.com/vignesh07/openclaw-formal-models).

## Важные оговорки

- Это **модели**, а не полная реализация на TypeScript. Возможно расхождение между моделью и кодом.
- Результаты ограничены пространством состояний, исследованным TLC; статус "зелёный" не означает безопасность за пределами допущений и границ, заданных в модели.
- Некоторые утверждения опираются на явные допущения относительно окружения (например, корректное развёртывание, корректные входные данные конфигурации).

## Воспроизведение результатов

На данный момент результаты воспроизводятся путём клонирования репозитория с моделями локально и запуска TLC (см. ниже). В будущей версии может быть реализовано:

- Запуск моделей в CI с публичными артефактами (трассировки контрпримеров, журналы выполнения)
- Размещённый рабочий процесс "запустить эту модель" для небольших проверок с ограниченными параметрами

Начало работы:

```bash
git clone https://github.com/vignesh07/openclaw-formal-models
cd openclaw-formal-models

# Требуется Java 11+ (TLC работает на JVM).
# В репозитории есть зафиксированный файл `tla2tools.jar` (инструменты TLA+) и предоставлены `bin/tlc` + цели Make.

make <target>
```

### Открытие шлюза и неправильная конфигурация открытого шлюза

**Утверждение:** привязка за пределами loopback без аутентификации может сделать возможным удалённое компрометацию / увеличить уязвимость; токены/пароли блокируют неавторизованных злоумышленников (согласно допущениям модели).

- Зелёные запуски:
  - `make gateway-exposure-v2`
  - `make gateway-exposure-v2-protected`
- Красные (ожидаемые):
  - `make gateway-exposure-v2-negative`

См. также: `docs/gateway-exposure-matrix.md` в репозитории моделей.

### Конвейер выполнения на узле (наиболее рискованная возможность)

**Утверждение:** для `exec host=node` требуется (а) белый список команд узла плюс объявленные команды и (б) активное одобрение при настройке; одобрения токенизируются для предотвращения повтора (в модели).

- Зелёные запуски:
  - `make nodes-pipeline`
  - `make approvals-token`
- Красные (ожидаемые):
  - `make nodes-pipeline-negative`
  - `make approvals-token-negative`

### Хранилище сопряжения (контроль DM)

**Утверждение:** запросы на сопряжение учитывают TTL и ограничения на количество ожидающих запросов.

- Зелёные запуски:
  - `make pairing`
  - `make pairing-cap`
- Красные (ожидаемые):
  - `make pairing-negative`
  - `make pairing-cap-negative`

### Контроль входящего трафика (упоминания + обход команд управления)

**Утверждение:** в групповых контекстах, требующих упоминания, неавторизованная "команда управления" не может обойти контроль упоминаний.

- Зелёный:
  - `make ingress-gating`
- Красный (ожидаемый):
  - `make ingress-gating-negative`

### Изоляция маршрутизации/ключа сессии

**Утверждение:** DM от разных участников не объединяются в одну сессию, если это явно не указано в настройках/конфигурации.

- Зелёный:
  - `make routing-isolation`
- Красный (ожидаемый):
  - `make routing-isolation-negative`

## v1++: дополнительные ограниченные модели (параллелизм, повторные попытки, корректность трассировки)

Это последующие модели, которые повышают точность в отношении реальных сценариев сбоев (неатомарные обновления, повторные попытки и распространение сообщений).

### Параллелизм/идемпотентность хранилища сопряжения

**Утверждение:** хранилище сопряжения должно обеспечивать соблюдение `MaxPending` и идемпотентность даже при чередовании операций (т. е. "проверка перед записью" должна быть атомарной/заблокированной; обновление не должно создавать дубликаты).

Что это означает:

- При параллельных запросах нельзя превысить `MaxPending` для канала.
- Повторные запросы/обновления для одного и того же `(канал, отправитель)` не должны создавать дубликаты активных ожидающих записей.

- Зелёные запуски:
  - `make pairing-race` (атомарная/заблокированная проверка ограничения)
  - `make pairing-idempotency`
  - `make pairing-refresh`
  - `make pairing-refresh-race`
- Красные (ожидаемые):
  - `make pairing-race-negative` (неатомарное начало/завершение гонки за ограничение)
  - `make pairing-idempotency-negative`
  - `make pairing-refresh-negative`
  - `make pairing-refresh-race-negative`

### Корреляция/идемпотентность трассировки входящего трафика

**Утверждение:** приём должен сохранять корреляцию трассировки при распространении сообщений и быть идемпотентным при повторных попытках провайдера.

Что это означает:

- Когда одно внешнее событие превращается в несколько внутренних сообщений, каждая часть сохраняет ту же идентичность трассировки/события.
- Повторные попытки не приводят к двойной обработке.
- Если идентификаторы событий провайдера отсутствуют, дедупликация использует безопасный ключ (например, идентификатор трассировки), чтобы избежать потери отдельных событий.

- Зелёный:
  - `make ingress-trace`
  - `make ingress-trace2`
  - `make ingress-idempotency`
  - `make ingress-dedupe-fallback`
- Красный (ожидаемый):
  - `make ingress-trace-negative`
  - `make ingress-trace2-negative`
  - `make ingress-idempotency-negative`
  - `make ingress-dedupe-fallback-negative`

### Приоритет dmScope при маршрутизации + identityLinks

**Утверждение:** маршрутизация должна по умолчанию изолировать сессии DM и объединять сессии только при явной настройке (приоритет канала + ссылки на идентичность).

Что это означает:

- Переопределения dmScope для конкретного канала должны иметь приоритет над глобальными настройками по умолчанию.
- identityLinks должны объединять сессии только в явно указанных связанных группах, а не между несвязанными участниками.

- Зелёный:
  - `make routing-precedence`
  - `make routing-identitylinks`
- Красный (ожидаемый):
  - `make routing-precedence-negative`
  - `make routing-identitylinks-negative`
