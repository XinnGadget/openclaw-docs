# Рефакторинг QA

Статус: базовая миграция выполнена.

## Цель

Перевести OpenClaw QA с модели с разделённым определением на единый источник истины:

- метаданные сценария;
- промты, отправляемые модели;
- настройка и демонтаж;
- логика каркаса;
- утверждения и критерии успеха;
- артефакты и подсказки для отчётов.

Желаемое конечное состояние — универсальный каркас QA, который загружает мощные файлы с определением сценариев вместо жёсткой проработки большей части поведения в TypeScript.

## Текущее состояние

Основной источник истины сейчас находится в `qa/scenarios/index.md` плюс по одному файлу на сценарий в `qa/scenarios/*.md`.

Реализовано:

- `qa/scenarios/index.md`:
  - канонические метаданные пакета QA;
  - идентификатор оператора;
  - стартовая миссия.
- `qa/scenarios/*.md`:
  - по одному Markdown-файлу на сценарий;
  - метаданные сценария;
  - привязки обработчиков;
  - конфигурация выполнения для конкретного сценария.
- `extensions/qa-lab/src/scenario-catalog.ts`:
  - парсер пакета Markdown + валидация через zod.
- `extensions/qa-lab/src/qa-agent-bootstrap.ts`:
  - отрисовка плана из пакета Markdown.
- `extensions/qa-lab/src/qa-agent-workspace.ts`:
  - сгенерированные файлы совместимости плюс `QA_SCENARIOS.md`.
- `extensions/qa-lab/src/suite.ts`:
  - выбор исполняемых сценариев через привязки обработчиков, определённые в Markdown.
- Протокол шины QA + UI:
  - универсальные встроенные вложения для отрисовки изображений, видео, аудио и файлов.

Оставшиеся разделяемые компоненты:

- `extensions/qa-lab/src/suite.ts`:
  - по-прежнему содержит большую часть исполняемой логики пользовательских обработчиков.
- `extensions/qa-lab/src/report.ts`:
  - по-прежнему формирует структуру отчёта на основе выходных данных во время выполнения.

Таким образом, разделение источника истины устранено, но выполнение по-прежнему в основном опирается на обработчики, а не является полностью декларативным.

## Как выглядит реальная поверхность сценария

Анализ текущего набора сценариев показывает несколько отчётливых классов сценариев.

### Простое взаимодействие

- базовый канал;
- базовый DM;
- последовательное продолжение в потоке;
- переключение модели;
- подтверждение выполнения;
- реакция/редактирование/удаление.

### Настройка и изменение во время выполнения

- отключение навыка через патч конфигурации;
- применение конфигурации, перезапуск, пробуждение;
- переключение возможностей при перезапуске конфигурации;
- проверка дрейфа инвентаря во время выполнения.

### Утверждения для файловой системы и репозитория

- отчёт об обнаружении исходных кодов/документов;
- сборка Lobster Invaders;
- поиск сгенерированного артефакта изображения.

### Оркестрация памяти

- вызов памяти;
- инструменты памяти в контексте канала;
- откат при сбое памяти;
- ранжирование памяти сеанса;
- изоляция памяти потока;
- "сны" памяти.

### Интеграция инструментов и плагинов

- вызов инструментов плагина MCP;
- видимость навыка;
- горячая установка навыка;
- нативная генерация изображений;
- круговой обход изображения;
- понимание изображения из вложения.

### Многоэтапные и многоакторные сценарии

- передача управления субагенту;
- синтез разветвления субагентов;
- потоки восстановления после перезапуска.

Эти категории важны, поскольку они определяют требования к DSL. Плоского списка "промт + ожидаемый текст" недостаточно.

## Направление работы

### Единый источник истины

Использовать `qa/scenarios/index.md` плюс `qa/scenarios/*.md` в качестве исходного источника истины.

Пакет должен оставаться:

- читаемым человеком при проверке;
- пригодным для машинного парсинга;
- достаточно богатым, чтобы обеспечивать:
  - выполнение набора сценариев;
  - инициализацию рабочей области QA;
  - метаданные UI QA Lab;
  - промты для документации/обнаружения;
  - генерацию отчётов.

### Предпочтительный формат создания

Использовать Markdown в качестве формата верхнего уровня со структурированным YAML внутри.

Рекомендуемая структура:

- YAML-заголовок (frontmatter):
  - id;
  - title;
  - surface;
  - tags;
  - ссылки на документацию (docs refs);
  - ссылки на код (code refs);
  - переопределения модели/провайдера;
  - предварительные условия (prerequisites).
- текстовые разделы:
  - цель (objective);
  - заметки (notes);
  - подсказки по отладке (debugging hints).
- блоки YAML в ограждении (fenced YAML blocks):
  - setup;
  - steps;
  - assertions;
  - cleanup.

Это даёт:

- лучшую читаемость PR по сравнению с гигантским JSON;
- более богатый контекст по сравнению с чистым YAML;
- строгий парсинг и валидацию через zod.

Сырой JSON допустим только в качестве промежуточной сгенерированной формы.

## Предлагаемая структура файла сценария

Пример:

````md
---
id: image-generation-roundtrip
title: Круговой обход генерации изображения
surface: image
tags: [media, image, roundtrip]
models:
  primary: openai/gpt-5.4
requires:
  tools: [image_generate]
  plugins: [openai, qa-channel]
docsRefs:
  - docs/help/testing.md
  - docs/concepts/model-providers.md
codeRefs:
  - extensions/qa-lab/src/suite.ts
  - src/gateway/chat-attachments.ts
---

# Цель

Проверить, что сгенерированный медиафайл повторно прикрепляется на следующем этапе.

# Настройка

```yaml scenario.setup
- action: config.patch
  patch:
    agents:
      defaults:
        imageGenerationModel:
          primary: openai/gpt-image-1
- action: session.create
  key: agent:qa:image-roundtrip
```

# Шаги

```yaml scenario.steps
- action: agent.send
  session: agent:qa:image-roundtrip
  message: |
    Проверка генерации изображения: сгенерируйте изображение маяка для QA и кратко опишите его одним предложением.
- action: artifact.capture
  kind: generated-image
  promptSnippet: Проверка генерации изображения
  saveAs: lighthouseImage
- action: agent.send
  session: agent:qa:image-roundtrip
  message: |
    Проверка кругового обхода изображения: опишите прикреплённое изображение маяка одним коротким предложением.
  attachments:
    - fromArtifact: lighthouseImage
```

# Ожидаемые результаты

```yaml scenario.expect
- assert: outbound.textIncludes
  value: lighthouse
- assert: requestLog.matches
  where:
    promptIncludes: Проверка кругового обхода изображения
  imageInputCountGte: 1
- assert: artifact.exists
  ref: lighthouseImage
```
````

## Возможности раннера, которые должен охватывать DSL

Исходя из текущего набора сценариев, универсальному раннеру требуется больше, чем просто выполнение промтов.

### Действия по настройке среды

- `bus.reset`;
- `gateway.waitHealthy`;
- `channel.waitReady`;
- `session.create`;
- `thread.create`;
- `workspace.writeSkill`.

### Действия на этапе работы агента

- `agent.send`;
- `agent.wait`;
- `bus.injectInbound`;
- `bus.injectOutbound`.

### Действия по настройке и во время выполнения

- `config.get`;
- `config.patch`;
- `config.apply`;
- `gateway.restart`;
- `tools.effective`;
- `skills.status`.

### Действия с файлами и артефактами

- `file.write`;
- `file.read`;
- `file.delete`;
- `file.touchTime`;
- `artifact.captureGeneratedImage`;
- `artifact.capturePath`.

### Действия с памятью и cron

- `memory.indexForce`;
- `memory.searchCli`;
- `doctor.memory.status`;
- `cron.list`;
- `cron.run`;
- `cron.waitCompletion`;
- `sessionTranscript.write`.

### Действия MCP

- `mcp.callTool`.

### Утверждения

- `outbound.textIncludes`;
- `outbound.inThread`;
- `outbound.notInRoot`;
- `tool.called`;
- `tool.notPresent`;
- `skill.visible`;
- `skill.disabled`;
- `file.contains`;
- `memory.contains`;
- `requestLog.matches`;
- `sessionStore.matches`;
- `cron.managedPresent`;
