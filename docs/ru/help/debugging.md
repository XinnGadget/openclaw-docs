# Отладка

На этой странице описаны вспомогательные инструменты для отладки потокового вывода, особенно в случаях, когда провайдер включает рассуждения в обычный текст.

## Переопределения конфигурации во время выполнения

Используйте команду `/debug` в чате, чтобы задать переопределения конфигурации **только для времени выполнения** (в памяти, а не на диске).

Команда `/debug` по умолчанию отключена; включите её с помощью `commands.debug: true`.
Это удобно, когда нужно переключать неочевидные настройки без редактирования файла `openclaw.json`.

Примеры:

```
/debug show
/debug set messages.responsePrefix="[openclaw]"
/debug unset messages.responsePrefix
/debug reset
```

`/debug reset` удаляет все переопределения и возвращает конфигурацию, сохранённую на диске.

## Режим наблюдения Gateway

Для быстрой итеративной разработки запустите Gateway под файловым наблюдателем:

```bash
pnpm gateway:watch
```

Это соответствует команде:

```bash
node scripts/watch-node.mjs gateway --force
```

Наблюдатель перезапускает Gateway при изменении файлов, важных для сборки, в каталоге `src/`, файлов исходного кода расширений, файлов `package.json` расширений и метаданных `openclaw.plugin.json`, а также файлов `tsconfig.json`, `package.json` и `tsdown.config.ts`.

При изменении метаданных расширения Gateway перезапускается без принудительной перестройки `tsdown`; при изменении исходного кода и конфигурационных файлов сначала выполняется перестройка `dist`.

Добавьте любые флаги CLI Gateway после `gateway:watch` — они будут передаваться при каждом перезапуске. Повторный запуск той же команды наблюдения для того же репозитория и набора флагов теперь заменяет старый наблюдатель, а не оставляет дублирующиеся родительские процессы наблюдателя.

## Профиль разработки + Gateway для разработки (--dev)

Используйте профиль разработки, чтобы изолировать состояние и создать безопасную одноразовую среду для отладки. Существует **два** флага `--dev`:

- **Глобальный `--dev` (профиль):** изолирует состояние в каталоге `~/.openclaw-dev` и устанавливает порт Gateway по умолчанию равным `19001` (производные порты смещаются соответственно).
- **`gateway --dev`:** указывает Gateway автоматически создать конфигурацию и рабочее пространство по умолчанию, если они отсутствуют (и пропустить BOOTSTRAP.md).

Рекомендуемый порядок действий (профиль разработки + начальная настройка для разработки):

```bash
pnpm gateway:dev
OPENCLAW_PROFILE=dev openclaw tui
```

Если у вас ещё нет глобальной установки, запустите CLI с помощью `pnpm openclaw ...`.

Что происходит:

1. **Изоляция профиля** (глобальный `--dev`):
   - `OPENCLAW_PROFILE=dev`
   - `OPENCLAW_STATE_DIR=~/.openclaw-dev`
   - `OPENCLAW_CONFIG_PATH=~/.openclaw-dev/openclaw.json`
   - `OPENCLAW_GATEWAY_PORT=19001` (порты браузера/canvas смещаются соответственно)

2. **Начальная настройка для разработки** (`gateway --dev`):
   - Записывает минимальную конфигурацию, если она отсутствует (`gateway.mode=local`, привязка к loopback).
   - Устанавливает `agent.workspace` в рабочее пространство для разработки.
   - Устанавливает `agent.skipBootstrap=true` (без BOOTSTRAP.md).
   - Заполняет файлы рабочего пространства, если они отсутствуют: `AGENTS.md`, `SOUL.md`, `TOOLS.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`.
   - Идентификация по умолчанию: **C3-PO** (протокольный дроид).
   - Пропускает провайдеров каналов в режиме разработки (`OPENCLAW_SKIP_CHANNELS=1`).

Порядок действий для сброса (новый старт):

```bash
pnpm gateway:dev:reset
```

Примечание: `--dev` — это **глобальный** флаг профиля, который может игнорироваться некоторыми запускающими программами.
Если вам нужно явно указать его, используйте форму переменной окружения:

```bash
OPENCLAW_PROFILE=dev openclaw gateway --dev --reset
```

Флаг `--reset` удаляет конфигурацию, учётные данные, сессии и рабочее пространство для разработки (с помощью `trash`, а не `rm`), а затем воссоздаёт стандартную настройку для разработки.

Совет: если уже запущен Gateway не для разработки (через launchd/systemd), сначала остановите его:

```bash
openclaw gateway stop
```

## Логирование необработанного потока (OpenClaw)

OpenClaw может вести логи **необработанного потока ассистента** до любой фильтрации или форматирования.
Это лучший способ проверить, поступают ли рассуждения в виде дельт обычного текста (или в виде отдельных блоков рассуждений).

Включите эту функцию через CLI:

```bash
pnpm gateway:watch --raw-stream
```

Необязательное переопределение пути:

```bash
pnpm gateway:watch --raw-stream --raw-stream-path ~/.openclaw/logs/raw-stream.jsonl
```

Эквивалентные переменные окружения:

```bash
OPENCLAW_RAW_STREAM=1
OPENCLAW_RAW_STREAM_PATH=~/.openclaw/logs/raw-stream.jsonl
```

Файл по умолчанию:

`~/.openclaw/logs/raw-stream.jsonl`

## Логирование необработанных чанков (pi-mono)

Чтобы зафиксировать **необработанные чанки, совместимые с OpenAI**, до их разбора на блоки, pi-mono предоставляет отдельный логгер:

```bash
PI_RAW_STREAM=1
```

Необязательный путь:

```bash
PI_RAW_STREAM_PATH=~/.pi-mono/logs/raw-openai-completions.jsonl
```

Файл по умолчанию:

`~/.pi-mono/logs/raw-openai-completions.jsonl`

> Примечание: это выводится только процессами, использующими провайдер `openai-completions` в pi-mono.

## Примечания по безопасности

- Логи необработанного потока могут содержать полные промты, вывод инструментов и пользовательские данные.
- Храните логи локально и удаляйте их после отладки.
- Если вы делитесь логами, предварительно удалите секретные данные и персональную информацию.
