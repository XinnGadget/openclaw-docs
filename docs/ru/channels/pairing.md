# Сопряжение

"Сопряжение" (*pairing*) — это этап явного **утверждения владельцем** в OpenClaw.
Он используется в двух случаях:

1. **Сопряжение DM** (кто может общаться с ботом)
2. **Сопряжение узлов** (какие устройства/узлы могут присоединиться к сети шлюза)

Контекст безопасности: [Безопасность](/gateway/security)

## 1) Сопряжение DM (доступ к входящим чатам)

Когда для канала настроена политика DM `pairing`, неизвестные отправители получают короткий код, а их сообщение **не обрабатывается** до тех пор, пока вы его не утвердите.

Политики DM по умолчанию описаны в: [Безопасность](/gateway/security)

Коды сопряжения:

- 8 символов, прописные буквы, без неоднозначных символов (`0O1I`).
- **Истекают через 1 час**. Бот отправляет сообщение о сопряжении только при создании нового запроса (примерно раз в час для каждого отправителя).
- Запросы на сопряжение DM в ожидании ограничены **3 запросами на канал** по умолчанию; дополнительные запросы игнорируются, пока один из них не истечёт или не будет утверждён.

### Утверждение отправителя

```bash
openclaw pairing list telegram
openclaw pairing approve telegram <КОД>
```

Поддерживаемые каналы: `bluebubbles`, `discord`, `feishu`, `googlechat`, `imessage`, `irc`, `line`, `matrix`, `mattermost`, `msteams`, `nextcloud-talk`, `nostr`, `openclaw-weixin`, `signal`, `slack`, `synology-chat`, `telegram`, `twitch`, `whatsapp`, `zalo`, `zalouser`.

### Где хранится состояние

Хранится в каталоге `~/.openclaw/credentials/`:

- Запросы в ожидании: `<канал>-pairing.json`
- Список утверждённых разрешений:
  - Учётная запись по умолчанию: `<канал>-allowFrom.json`
  - Нестандартная учётная запись: `<канал>-<идентификаторУчётнойЗаписи>-allowFrom.json`

Поведение в зависимости от учётной записи:

- Нестандартные учётные записи читают/записывают только свой ограниченный файл списка разрешений.
- Учётная запись по умолчанию использует файл списка разрешений, ограниченный каналом.

Относитесь к этим файлам как к конфиденциальным (они контролируют доступ к вашему ассистенту).

Важно: этот хранилище предназначено для доступа к DM. Авторизация в группах осуществляется отдельно.
Утверждение кода сопряжения DM не даёт отправителю автоматически права выполнять команды в группах или управлять ботом в группах. Для доступа к группам настройте явные списки разрешений для групп в канале (например, `groupAllowFrom`, `groups` или переопределения для каждой группы/темы в зависимости от канала).

## 2) Сопряжение устройств узлов (iOS/Android/macOS/безголовые узлы)

Узлы подключаются к шлюзу в качестве **устройств** с `role: node`. Шлюз создаёт запрос на сопряжение устройства, который необходимо утвердить.

### Сопряжение через Telegram (рекомендуется для iOS)

Если вы используете плагин `device-pair`, вы можете выполнить первичное сопряжение устройства полностью через Telegram:

1. В Telegram отправьте боту сообщение: `/pair`
2. Бот ответит двумя сообщениями: сообщением с инструкцией и отдельным сообщением с **кодом настройки** (его легко скопировать и вставить в Telegram).
3. На вашем телефоне откройте приложение OpenClaw для iOS → Настройки → Шлюз.
4. Вставьте код настройки и подключитесь.
5. Вернитесь в Telegram: `/pair pending` (просмотрите идентификаторы запросов, роль и области действия), затем утвердите.

Код настройки представляет собой JSON-полезную нагрузку в кодировке base64, которая содержит:

- `url`: URL WebSocket шлюза (`ws://...` или `wss://...`)
- `bootstrapToken`: краткосрочный токен начальной загрузки для одного устройства, используемый для первоначального рукопожатия при сопряжении

Этот токен начальной загрузки содержит встроенный профиль начальной загрузки для сопряжения:

- основной передаваемый токен `node` остаётся с `scopes: []`
- любой передаваемый токен `operator` остаётся ограниченным списком разрешений начальной загрузки: `operator.approvals`, `operator.read`, `operator.talk.secrets`, `operator.write`
- проверки области действия начальной загрузки имеют префикс роли, а не единый пул областей:
  записи области действия оператора удовлетворяют только запросам оператора, а роли, не являющиеся операторами, должны запрашивать области действия с префиксом своей роли

Относитесь к коду настройки как к паролю, пока он действителен.

### Утверждение устройства узла

```bash
openclaw devices list
openclaw devices approve <идентификаторЗапроса>
openclaw devices reject <идентификаторЗапроса>
```

Если одно и то же устройство повторяет попытку с другими данными аутентификации (например, с другой ролью/областями действия/открытым ключом), предыдущий ожидающий запрос заменяется, и создаётся новый `requestId`.

### Хранение состояния сопряжения узлов

Хранится в каталоге `~/.openclaw/devices/`:

- `pending.json` (краткосрочное хранение; ожидающие запросы истекают)
- `paired.json` (сопряжённые устройства + токены)

### Примечания

- Устаревший API `node.pair.*` (CLI: `openclaw nodes pending|approve|reject|rename`) представляет собой отдельное хранилище сопряжения, принадлежащее шлюзу. Узлам WS по-прежнему требуется сопряжение устройств.
- Запись о сопряжении является надёжным источником информации об утверждённых ролях. Активные токены устройств остаются ограниченными утверждённым набором ролей; случайная запись токена вне утверждённых ролей не создаёт нового доступа.
