# Автоматизация QA E2E

Частный стек QA предназначен для тестирования OpenClaw в более реалистичных условиях, имитирующих работу канала, — в отличие от одиночного модульного теста.

Текущие компоненты:

- `extensions/qa-channel`: синтетический канал сообщений с поддержкой личных сообщений (DM), каналов, потоков, реакций, редактирования и удаления.
- `extensions/qa-lab`: пользовательский интерфейс отладчика и шина QA для наблюдения за транскриптом, внедрения входящих сообщений и экспорта отчёта в формате Markdown.
- `qa/`: ресурсы с предопределёнными данными в репозитории для начальной задачи и базовых сценариев QA.

Текущий процесс работы оператора QA представляет собой сайт QA с двумя панелями:

- Левая: панель управления Gateway (Control UI) с агентом.
- Правая: QA Lab, отображающая транскрипцию в стиле Slack и план сценария.

Запустите его с помощью команды:

```bash
pnpm qa:lab:up
```

Эта команда собирает сайт QA, запускает канал Gateway на базе Docker и открывает страницу QA Lab, где оператор или цикл автоматизации может задать агенту задачу QA, наблюдать за реальным поведением канала и фиксировать, что сработало, что не сработало и что осталось заблокированным.

Для более быстрой итерации пользовательского интерфейса QA Lab без перестройки образа Docker каждый раз запустите стек с подключённым бандлом QA Lab:

```bash
pnpm openclaw qa docker-build-image
pnpm qa:lab:build
pnpm qa:lab:up:fast
pnpm qa:lab:watch
```

Команда `qa:lab:up:fast` сохраняет сервисы Docker на предварительно собранном образе и подключает `extensions/qa-lab/web/dist` в контейнер `qa-lab`. Команда `qa:lab:watch` перестраивает этот бандл при изменениях, а браузер автоматически перезагружается при изменении хэша ресурса QA Lab.

## Ресурсы с предопределёнными данными в репозитории

Ресурсы с предопределёнными данными находятся в `qa/`:

- `qa/scenarios/index.md`
- `qa/scenarios/*.md`

Они намеренно размещены в Git, чтобы план QA был доступен как людям, так и агенту. Базовый список должен быть достаточно широким, чтобы охватить:

- Личные сообщения (DM) и чат в канале
- Поведение потоков
- Жизненный цикл действий с сообщениями
- Обратные вызовы cron
- Восстановление памяти
- Переключение моделей
- Передача управления субагенту
- Чтение репозитория и документации
- Одну небольшую задачу сборки, например Lobster Invaders

## Отчётность

`qa-lab` экспортирует отчёт по протоколу в формате Markdown на основе наблюдаемой временной шкалы шины. Отчёт должен отвечать на следующие вопросы:

- Что сработало
- Что не сработало
- Что осталось заблокированным
- Какие последующие сценарии стоит добавить

Для проверки характера и стиля запустите один и тот же сценарий на нескольких актуальных версиях моделей и составьте оценочный отчёт в формате Markdown:

```bash
pnpm openclaw qa character-eval \
  --model openai/gpt-5.4,thinking=xhigh \
  --model openai/gpt-5.2,thinking=xhigh \
  --model openai/gpt-5,thinking=xhigh \
  --model anthropic/claude-opus-4-6,thinking=high \
  --model anthropic/claude-sonnet-4-6,thinking=high \
  --model zai/glm-5.1,thinking=high \
  --model moonshot/kimi-k2.5,thinking=high \
  --model google/gemini-3.1-pro-preview,thinking=high \
  --judge-model openai/gpt-5.4,thinking=xhigh,fast \
  --judge-model anthropic/claude-opus-4-6,thinking=high \
  --blind-judge-models \
  --concurrency 16 \
  --judge-concurrency 16
```

Эта команда запускает локальные дочерние процессы шлюза QA, а не Docker. Сценарии оценки характера должны задавать персону через `SOUL.md`, а затем выполнять обычные пользовательские действия, такие как чат, помощь в рабочей области и небольшие задачи с файлами. Кандидатной модели не следует сообщать, что она проходит оценку. Команда сохраняет полную транскрипцию, записывает основные статистические данные о выполнении, а затем просит модели-судьи в быстром режиме с рассуждением `xhigh` ранжировать выполнения по естественности, атмосфере и юмору.

Используйте `--blind-judge-models` при сравнении провайдеров: запрос к судье по-прежнему получает каждую транскрипцию и статус выполнения, но ссылки на кандидатные модели заменяются нейтральными метками, такими как `candidate-01`; отчёт сопоставляет рейтинги с реальными ссылками после анализа.

По умолчанию для кандидатных выполнений используется уровень мышления `high`, а для моделей OpenAI, поддерживающих этот режим, — `xhigh`. Переопределите конкретный кандидатный параметр встроенным способом с помощью `--model provider/model,thinking=<level>`. Параметр `--thinking <level>` по-прежнему задаёт глобальный запасной вариант, а более старый формат `--model-thinking <provider/model=level>` сохранён для совместимости.

Для кандидатных ссылок OpenAI по умолчанию используется быстрый режим, чтобы задействовать приоритетную обработку там, где это поддерживает провайдер. Добавьте `,fast`, `,no-fast` или `,fast=false` встроенным способом, если для отдельного кандидата или судьи требуется переопределение. Передавайте `--fast` только в том случае, если вы хотите принудительно включить быстрый режим для каждой кандидатной модели.

Длительность выполнения кандидатных и судейских моделей записывается в отчёт для анализа бенчмарков, но в запросах к судьям явно указано не ранжировать по скорости.

По умолчанию для выполнения кандидатных и судейских моделей установлено значение параллелизма 16. Уменьшите значения `--concurrency` или `--judge-concurrency`, если ограничения провайдера или нагрузка на локальный шлюз делают выполнение слишком "шумным".

Если параметр `--model` для кандидата не передан, оценка характера по умолчанию использует следующие модели: `openai/gpt-5.4`, `openai/gpt-5.2`, `openai/gpt-5`, `anthropic/claude-opus-4-6`, `anthropic/claude-sonnet-4-6`, `zai/glm-5.1`, `moonshot/kimi-k2.5` и `google/gemini-3.1-pro-preview`.

Если параметр `--judge-model` не передан, судьи по умолчанию используют следующие модели: `openai/gpt-5.4,thinking=xhigh,fast` и `anthropic/claude-opus-4-6,thinking=high`.
