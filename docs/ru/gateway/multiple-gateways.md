# Несколько шлюзов (на одном хосте)

В большинстве случаев следует использовать один шлюз, поскольку единственный шлюз способен обрабатывать несколько соединений для обмена сообщениями и агентов. Если требуется более строгая изоляция или резервирование (например, для резервного бота), запустите отдельные шлюзы с изолированными профилями и портами.

## Чек-лист по изоляции (обязательно)

- `OPENCLAW_CONFIG_PATH` — файл конфигурации для каждого экземпляра;
- `OPENCLAW_STATE_DIR` — сессии, учётные данные, кэши для каждого экземпляра;
- `agents.defaults.workspace` — корневая директория рабочей области для каждого экземпляра;
- `gateway.port` (или `--port`) — уникальный для каждого экземпляра;
- Производные порты (браузер/canvas) не должны пересекаться.

Если эти параметры будут общими, возникнут конфликты конфигураций и портов.

## Рекомендация: профили (`--profile`)

Профили автоматически задают область видимости для `OPENCLAW_STATE_DIR` и `OPENCLAW_CONFIG_PATH`, а также добавляют суффикс к именам сервисов.

```bash
# основной
openclaw --profile main setup
openclaw --profile main gateway --port 18789

# резервный
openclaw --profile rescue setup
openclaw --profile rescue gateway --port 19001
```

Сервисы для каждого профиля:

```bash
openclaw --profile main gateway install
openclaw --profile rescue gateway install
```

## Руководство по резервному боту

Запустите второй шлюз на том же хосте со своими:

- профилем/конфигурацией;
- директорией состояния;
- рабочей областью;
- базовым портом (плюс производные порты).

Это позволит изолировать резервный бот от основного, чтобы он мог выполнять отладку или применять изменения конфигурации, если основной бот недоступен.

Распределение портов: оставляйте зазор как минимум в 20 портов между базовыми портами, чтобы производные порты браузера/canvas/CDP никогда не пересекались.

### Как установить (резервный бот)

```bash
# Основной бот (существующий или новый, без параметра --profile)
# Работает на порту 18789 + порты Chrome CDC/Canvas/…
openclaw onboard
openclaw gateway install

# Резервный бот (изолированный профиль + порты)
openclaw --profile rescue onboard
# Примечания:
# - имя рабочей области по умолчанию будет дополнено суффиксом -rescue;
# - порт должен быть как минимум 18789 + 20 портов,
#   лучше выбрать совершенно другой базовый порт, например, 19789;
# - остальная часть процесса подключения такая же, как обычно.

# Чтобы установить сервис (если это не произошло автоматически во время настройки)
openclaw --profile rescue gateway install
```

## Сопоставление портов (производные)

Базовый порт = `gateway.port` (или `OPENCLAW_GATEWAY_PORT` / `--port`).

- порт сервиса управления браузером = базовый порт + 2 (только loopback);
- хост canvas обслуживается на HTTP-сервере шлюза (на том же порту, что и `gateway.port`);
- порты CDP профиля браузера автоматически выделяются в диапазоне от `browser.controlPort + 9` до `+ 108`.

Если вы переопределяете какие-либо из этих параметров в конфигурации или окружении, убедитесь, что они уникальны для каждого экземпляра.

## Примечания по браузеру/CDP (распространённая ошибка)

- **Не** фиксируйте `browser.cdpUrl` на одних и тех же значениях для нескольких экземпляров.
- Каждому экземпляру требуется свой порт управления браузером и диапазон CDP (вычисляется на основе порта шлюза).
- Если вам нужны явные порты CDP, задайте `browser.profiles.<name>.cdpPort` для каждого экземпляра.
- Для удалённого Chrome используйте `browser.profiles.<name>.cdpUrl` (для каждого профиля, для каждого экземпляра).

## Пример ручной настройки окружения

```bash
OPENCLAW_CONFIG_PATH=~/.openclaw/main.json \
OPENCLAW_STATE_DIR=~/.openclaw-main \
openclaw gateway --port 18789

OPENCLAW_CONFIG_PATH=~/.openclaw/rescue.json \
OPENCLAW_STATE_DIR=~/.openclaw-rescue \
openclaw gateway --port 19001
```

## Быстрые проверки

```bash
openclaw --profile main gateway status --deep
openclaw --profile rescue gateway status --deep
openclaw --profile rescue gateway probe
openclaw --profile main status
openclaw --profile rescue status
openclaw --profile rescue browser status
```

Интерпретация:

- `gateway status --deep` помогает обнаружить устаревшие сервисы launchd/systemd/schtasks из предыдущих установок.
- Текст предупреждения `gateway probe`, например, `multiple reachable gateways detected`, ожидается только в том случае, если вы намеренно запускаете более одного изолированного шлюза.
