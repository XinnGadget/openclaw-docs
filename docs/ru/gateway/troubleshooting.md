# Устранение неполадок шлюза

Эта страница содержит подробное руководство.
Начните с [/help/troubleshooting](/help/troubleshooting), если сначала хотите выполнить быструю первичную диагностику.

## Последовательность команд

Выполните следующие команды в указанном порядке:

```bash
openclaw status
openclaw gateway status
openclaw logs --follow
openclaw doctor
openclaw channels status --probe
```

Ожидаемые признаки исправной работы:

- `openclaw gateway status` показывает `Runtime: running` и `RPC probe: ok`.
- `openclaw doctor` не сообщает о блокирующих проблемах с конфигурацией или сервисами.
- `openclaw channels status --probe` отображает актуальный статус транспорта для каждого аккаунта, а также (если поддерживается) результаты проверки/аудита, например `works` или `audit ok`.

## Ошибка Anthropic 429: требуется дополнительный объём использования для длинного контекста

Используйте этот раздел, если в логах/ошибках присутствует:
`HTTP 429: rate_limit_error: Extra usage is required for long context requests`.

```bash
openclaw logs --follow
openclaw models status
openclaw config get agents.defaults.models
```

Обратите внимание на следующее:

- Выбранная модель Anthropic Opus/Sonnet имеет параметр `params.context1m: true`.
- Текущие учётные данные Anthropic не позволяют использовать длинный контекст.
- Запросы завершаются ошибкой только в длинных сессиях/запусках модели, требующих пути 1M beta.

Варианты решения:

1. Отключите `context1m` для этой модели, чтобы вернуться к обычному окну контекста.
2. Используйте учётные данные Anthropic, позволяющие отправлять запросы с длинным контекстом, или переключитесь на API-ключ Anthropic.
3. Настройте резервные модели, чтобы выполнение продолжалось, когда запросы Anthropic с длинным контекстом отклоняются.

## Локальный бэкенд, совместимый с OpenAI, проходит прямые проверки, но выполнение агента завершается ошибкой

Используйте этот раздел, если:

- `curl ... /v1/models` работает
- простые прямые запросы `/v1/chat/completions` работают
- выполнение модели в OpenClaw завершается ошибкой только при обычных ходах агента

```bash
curl http://127.0.0.1:1234/v1/models
curl http://127.0.0.1:1234/v1/chat/completions \
  -H 'content-type: application/json' \
  -d '{"model":"<id>","messages":[{"role":"user","content":"hi"}],"stream":false}'
openclaw infer model run --model <provider/model> --prompt "hi" --json
openclaw logs --follow
```

Обратите внимание на следующее:

- простые прямые запросы выполняются успешно, но выполнение в OpenClaw завершается ошибкой только при более длинных промтах
- ошибки бэкенда о том, что `messages[].content` должен быть строкой
- сбои бэкенда, возникающие только при большем количестве токенов в промте или при полном промте времени выполнения агента

Типичные признаки:

- `messages[...].content: invalid type: sequence, expected a string` → бэкенд отклоняет структурированные части содержимого Chat Completions. Решение: установите `models.providers.<provider>.models[].compat.requiresStringContent: true`.
- простые прямые запросы выполняются успешно, но выполнение агента в OpenClaw завершается ошибкой из-за сбоев бэкенда/модели (например, Gemma в некоторых сборках `inferrs`) → транспорт OpenClaw, вероятно, уже настроен правильно; бэкенд не справляется с формой промта во время выполнения агента.
- ошибки уменьшаются после отключения инструментов, но не исчезают полностью → схемы инструментов частично создавали нагрузку, но оставшаяся проблема всё ещё связана с пропускной способностью модели/сервера или ошибкой бэкенда.

Варианты решения:

1. Установите `compat.requiresStringContent: true` для бэкендов Chat Completions, работающих только со строками.
2. Установите `compat.supportsTools: false` для моделей/бэкендов, которые не могут надёжно обрабатывать схему инструментов OpenClaw.
3. Снижайте нагрузку на промт, где это возможно: уменьшите размер начальной загрузки рабочей области, сократите историю сессии, используйте более лёгкую локальную модель или бэкенд с лучшей поддержкой длинного контекста.
4. Если простые прямые запросы продолжают выполняться успешно, а ходы агента в OpenClaw по-прежнему завершаются сбоем внутри бэкенда, считайте это ограничением вышестоящего сервера/модели и отправьте отчёт о проблеме с принятой формой полезной нагрузки.

## Нет ответов

Если каналы работают, но никто не отвечает, проверьте маршрутизацию и политику перед повторным подключением чего-либо.

```bash
openclaw status
openclaw channels status --probe
openclaw pairing list --channel <channel> [--account <id>]
openclaw config get channels
openclaw logs --follow
```

Обратите внимание на следующее:

- Ожидание подтверждения сопряжения для отправителей прямых сообщений.
- Ограничение упоминаний в группах (`requireMention`, `mentionPatterns`).
- Несоответствия в списках разрешённых каналов/групп.

Типичные признаки:

- `drop guild message (mention required` → сообщение в группе игнорируется до упоминания.
- `pairing request` → отправителю требуется одобрение.
- `blocked` / `allowlist` → отправитель/канал был отфильтрован согласно политике.

Связанные материалы:

## Подключение к панели управления (UI)

Если панель управления/UI не подключается, проверьте URL, режим аутентификации и предположения о защищённом контексте.

```bash
openclaw gateway status
openclaw status
openclaw logs --follow
openclaw doctor
openclaw gateway status --json
```

Обратите внимание на следующее:

- Правильный URL для проверки и URL панели управления.
- Несоответствие режима аутентификации/токена между клиентом и шлюзом.
- Использование HTTP там, где требуется идентификация устройства.

Типичные признаки:

- `device identity required` → незащищённый контекст или отсутствие аутентификации устройства.
- `origin not allowed` → `Origin` браузера не входит в `gateway.controlUi.allowedOrigins` (или вы подключаетесь из источника браузера, не являющегося loopback, без явного списка разрешённых).
- `device nonce required` / `device nonce mismatch` → клиент не завершает процесс аутентификации устройства на основе вызова (`connect.challenge` + `device.nonce`).
- `device signature invalid` / `device signature expired` → клиент подписал неправильную полезную нагрузку (или устаревшую временную метку) для текущего рукопожатия.
- `AUTH_TOKEN_MISMATCH` с `canRetryWithDeviceToken=true` → клиент может выполнить одну доверенную повторную попытку с кэшированным токеном устройства.
- Повторная попытка с кэшированным токеном использует набор областей, сохранённый вместе с сопряжённым токеном устройства. Явные вызовы `deviceToken` / `scopes` сохраняют запрошенный набор областей.
- Вне этого пути повторных попыток приоритет аутентификации подключения следующий: явный общий токен/пароль, затем явный `deviceToken`, затем сохранённый токен устройства, затем токен начальной загрузки.
- В асинхронном пути Tailscale Serve Control UI неудачные попытки для одного и того же `{scope, ip}` сериализуются до того, как ограничитель зафиксирует сбой. Две неудачные параллельные повторные попытки от одного и того же клиента могут привести к появлению сообщения `retry later` при второй попытке вместо двух простых несоответствий.
- `too many failed authentication attempts (retry later)` от клиента с источником loopback в браузере → повторные сбои от одного и того же нормализованного `Origin` временно заблокированы; другой источник localhost использует отдельный пул.
- повторяющееся `unauthorized` после повторной попытки → расхождение общего токена/токена устройства; обновите конфигурацию токена и при необходимости повторно утвердите/сбросьте токен устройства.
- `gateway connect failed:` → неправильный хост/порт/URL.
