# Рабочее пространство агента

Рабочее пространство — это "дом" агента. Это единственная рабочая директория, которая используется для инструментов работы с файлами и для контекста рабочего пространства. Обращайтесь с ним как с приватной информацией и считайте его памятью.

Оно отделено от директории `~/.openclaw/`, в которой хранятся конфигурация, учётные данные и сессии.

**Важно:** рабочее пространство — это **текущая рабочая директория по умолчанию**, а не жёсткая "песочница". Инструменты разрешают относительные пути относительно рабочего пространства, но абсолютные пути по-прежнему могут вести к другим местам на хосте, если "песочница" не включена. Если вам нужна изоляция, используйте [`agents.defaults.sandbox`](/gateway/sandboxing) (и/или конфигурацию "песочницы" для конкретного агента).

Когда "песочница" включена и `workspaceAccess` не имеет значения `"rw"`, инструменты работают внутри рабочего пространства "песочницы" в директории `~/.openclaw/sandboxes`, а не в рабочем пространстве хоста.

## Расположение по умолчанию

- По умолчанию: `~/.openclaw/workspace`.
- Если переменная `OPENCLAW_PROFILE` задана и не равна `"default"`, то по умолчанию используется `~/.openclaw/workspace-<profile>`.
- Можно переопределить в `~/.openclaw/openclaw.json`:

```json5
{
  agent: {
    workspace: "~/.openclaw/workspace",
  },
}
```

Команды `openclaw onboard`, `openclaw configure` или `openclaw setup` создадут рабочее пространство и заполнят его начальными файлами, если их нет.

При копировании начальных данных в "песочницу" принимаются только обычные файлы из рабочего пространства; символические и жёсткие ссылки, указывающие за пределы исходного рабочего пространства, игнорируются.

Если вы уже сами управляете файлами рабочего пространства, вы можете отключить создание начальных файлов:

```json5
{ agent: { skipBootstrap: true } }
```

## Дополнительные папки рабочего пространства

В старых установках могла быть создана папка `~/openclaw`. Наличие нескольких директорий рабочего пространства может привести к путанице с аутентификацией или расхождению состояний, поскольку одновременно активно только одно рабочее пространство.

**Рекомендация:** используйте одно активное рабочее пространство. Если вы больше не используете дополнительные папки, архивируйте их или переместите в "Корзину" (например, `trash ~/openclaw`). Если вы намеренно сохраняете несколько рабочих пространств, убедитесь, что `agents.defaults.workspace` указывает на активное.

Команда `openclaw doctor` выдаёт предупреждение, если обнаруживает дополнительные директории рабочего пространства.

## Карта файлов рабочего пространства (значение каждого файла)

Ниже перечислены стандартные файлы, которые OpenClaw ожидает найти в рабочем пространстве:

- `AGENTS.md`
  - Инструкции по работе с агентом и использованию памяти.
  - Загружается в начале каждой сессии.
  - Хорошее место для правил, приоритетов и деталей о том, "как себя вести".

- `SOUL.md`
  - Персона, тон и границы.
  - Загружается в каждой сессии.
  - Руководство: [SOUL.md Personality Guide](/concepts/soul).

- `USER.md`
  - Информация о пользователе и о том, как к нему обращаться.
  - Загружается в каждой сессии.

- `IDENTITY.md`
  - Имя агента, его "атмосфера" и эмодзи.
  - Создаётся/обновляется во время начальной настройки.

- `TOOLS.md`
  - Заметки о локальных инструментах и соглашениях.
  - Не влияет на доступность инструментов; это только руководство.

- `HEARTBEAT.md`
  - Необязательный краткий чек-лист для запусков "heartbeat".
  - Старайтесь делать его коротким, чтобы избежать расхода токенов.

- `BOOT.md`
  - Необязательный чек-лист запуска, выполняемый при перезапуске шлюза, когда включены внутренние хуки.
  - Старайтесь делать его коротким; используйте инструмент сообщений для исходящих отправлений.

- `BOOTSTRAP.md`
  - Одноразовый ритуал первого запуска.
  - Создаётся только для нового рабочего пространства.
  - Удалите его после завершения ритуала.

- `memory/YYYY-MM-DD.md`
  - Ежедневный журнал памяти (один файл в день).
  - Рекомендуется читать текущий и вчерашний файл при начале сессии.

- `MEMORY.md` (необязательно)
  - Отредактированная долгосрочная память.
  - Загружается только в основной приватной сессии (не в общих/групповых контекстах).

См. [Memory](/concepts/memory) для ознакомления с рабочим процессом и автоматическим очищением памяти.

- `skills/` (необязательно)
  - Навыки, специфичные для рабочего пространства.
  - Имеет наивысший приоритет для данного рабочего пространства.
  - Переопределяет навыки проектного агента, личные навыки агента, управляемые навыки, встроенные навыки и `skills.load.extraDirs` в случае совпадения имён.

- `canvas/` (необязательно)
  - Файлы интерфейса Canvas UI для отображения узлов (например, `canvas/index.html`).

Если какой-либо начальный файл отсутствует, OpenClaw вставляет в сессию маркер "отсутствует файл" и продолжает работу. Большие начальные файлы при вставке усекаются; пределы можно настроить с помощью `agents.defaults.bootstrapMaxChars` (по умолчанию: 20 000) и `agents.defaults.bootstrapTotalMaxChars` (по умолчанию: 150 000).

Команда `openclaw setup` может воссоздать отсутствующие файлы по умолчанию без перезаписи существующих файлов.

## Что НЕ находится в рабочем пространстве

Эти элементы находятся в директории `~/.openclaw/` и **не должны** помещаться в репозиторий рабочего пространства:

- `~/.openclaw/openclaw.json` (конфигурация);
- `~/.openclaw/agents/<agentId>/agent/auth-profiles.json` (профили аутентификации модели: OAuth + API-ключи);
- `~/.openclaw/credentials/` (состояние каналов/провайдеров плюс данные импорта OAuth старой версии);
- `~/.openclaw/agents/<agentId>/sessions/` (транскрипты сессий + метаданные);
- `~/.openclaw/skills/` (управляемые навыки).

Если вам нужно перенести сессии или конфигурацию, скопируйте их отдельно и не включайте в систему контроля версий.

## Резервное копирование в Git (рекомендуется, приватное)

Относитесь к рабочему пространству как к приватной памяти. Поместите его в **приватный** репозиторий Git, чтобы обеспечить резервное копирование и возможность восстановления.

Выполните следующие шаги на машине, где работает шлюз (именно там находится рабочее пространство).

### 1) Инициализация репозитория

Если Git установлен, новые рабочие пространства инициализируются автоматически. Если это рабочее пространство ещё не является репозиторием, выполните:

```bash
cd ~/.openclaw/workspace
git init
git add AGENTS.md SOUL.md TOOLS.md IDENTITY.md USER.md HEARTBEAT.md memory/
git commit -m "Add agent workspace"
```

### 2) Добавление приватного удалённого репозитория (варианты для начинающих)

Вариант А: веб-интерфейс GitHub

1. Создайте новый **приватный** репозиторий на GitHub.
2. Не инициализируйте его с README (чтобы избежать конфликтов при слиянии).
3. Скопируйте HTTPS-адрес удалённого репозитория.
4. Добавьте удалённый репозиторий и отправьте данные:

```bash
git branch -M main
git remote add origin <https-url>
git push -u origin main
```

Вариант Б: GitHub CLI (`gh`)

```bash
gh auth login
gh repo create openclaw-workspace --private --source . --remote origin --push
```

Вариант В: веб-интерфейс GitLab

1. Создайте новый **приватный** репозиторий на GitLab.
2. Не инициализируйте его с README (чтобы избежать конфликтов при слиянии).
3. Скопируйте HTTPS-адрес удалённого репозитория.
4. Добавьте удалённый репозиторий и отправьте данные:

```bash
git branch -M main
git remote add origin <https-url>
git push -u origin main
```

### 3) Регулярные обновления

```bash
git status
git add .
git commit -m "Update memory"
git push
```

## Не коммитите секреты

Даже в приватном репозитории избегайте хранения секретов в рабочем пространстве:

- API-ключей, токенов OAuth, паролей или других учётных данных.
- Чего-либо из директории `~/.openclaw/`.
- Сырых дампов чатов или конфиденциальных вложений.

Если вам необходимо хранить ссылки на конфиденциальные данные, используйте плейсхолдеры и храните настоящие секреты в другом месте (в менеджере паролей, переменных окружения
