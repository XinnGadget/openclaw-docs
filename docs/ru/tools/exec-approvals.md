# Утверждения exec

Утверждения exec — это **ограничение для сопутствующего приложения / узла-хоста**, позволяющее агенту в песочнице выполнять команды на реальном хосте (`gateway` или `node`). Представьте это как предохранительный блокиратор: команды разрешаются только в том случае, если политика + список разрешённых элементов + (опционально) утверждение пользователя совпадают.

Утверждения exec действуют **дополнительно** к политике инструментов и повышенному контролю (за исключением случая, когда повышенный режим установлен в `full`, — тогда утверждения пропускаются).

Действующая политика — это **наиболее строгая** из `tools.exec.*` и настроек утверждений по умолчанию; если поле утверждений опущено, используется значение `tools.exec`.

Выполнение на хосте также использует локальное состояние утверждений на этом компьютере. Локальное для хоста значение `ask: "always"` в `~/.openclaw/exec-approvals.json` будет запрашивать подтверждение даже в том случае, если настройки сеанса или конфигурации по умолчанию запрашивают `ask: "on-miss"`.

Используйте `openclaw approvals get`, `openclaw approvals get --gateway` или `openclaw approvals get --node <id|name|ip>`, чтобы проверить запрошенную политику, источники политики хоста и итоговый результат.

Если пользовательский интерфейс сопутствующего приложения **недоступен**, любой запрос, требующий подсказки, разрешается с помощью **резервного варианта запроса** (по умолчанию: отклонить).

Нативные клиенты утверждения чата также могут предоставлять возможности, специфичные для канала, в сообщении о ожидающем утверждении. Например, в Matrix можно добавить ярлыки реакций к подсказке об утверждении (`✅` разрешить один раз, `❌` отклонить и `♾️` всегда разрешать, если доступно), оставляя при этом команды `/approve ...` в сообщении в качестве запасного варианта.

## Где это применяется

Утверждения exec применяются локально на хосте выполнения:

- **хост-шлюз** → процесс `openclaw` на машине шлюза;
- **хост-узел** → запускатор узла (сопутствующее приложение для macOS или автономный хост узла).

Примечание о модели доверия:

- Вызывающие, прошедшие аутентификацию через шлюз, считаются доверенными операторами для этого шлюза.
- Сопряжённые узлы расширяют возможности доверенного оператора на хост узла.
- Утверждения exec снижают риск случайного выполнения, но не являются границей аутентификации для каждого пользователя.
- Утверждённые выполнения на хосте узла привязывают канонический контекст выполнения: канонический cwd, точный argv, привязку env, если она присутствует, и закреплённый путь к исполняемому файлу, если это применимо.
- Для сценариев оболочки и прямых вызовов интерпретатора/среды выполнения OpenClaw также пытается привязать один конкретный локальный файл-операнд. Если этот привязанный файл изменяется после утверждения, но до выполнения, выполнение отклоняется вместо запуска изменённого содержимого.
- Привязка файлов намеренно выполняется по принципу "наилучших усилий", а не как полная семантическая модель каждого пути загрузчика интерпретатора/среды выполнения. Если режим утверждения не может определить ровно один конкретный локальный файл для привязки, он отказывается создавать выполнение на основе утверждения вместо того, чтобы имитировать полное покрытие.

Разделение для macOS:

- **служба хоста узла** пересылает `system.run` в **приложение macOS** через локальный IPC.
- **Приложение macOS** применяет утверждения и выполняет команду в контексте пользовательского интерфейса.

## Настройки и хранилище

Утверждения хранятся в локальном JSON-файле на хосте выполнения:

`~/.openclaw/exec-approvals.json`

Пример схемы:

```json
{
  "version": 1,
  "socket": {
    "path": "~/.openclaw/exec-approvals.sock",
    "token": "base64url-token"
  },
  "defaults": {
    "security": "deny",
    "ask": "on-miss",
    "askFallback": "deny",
    "autoAllowSkills": false
  },
  "agents": {
    "main": {
      "security": "allowlist",
      "ask": "on-miss",
      "askFallback": "deny",
      "autoAllowSkills": true,
      "allowlist": [
        {
          "id": "B0C8C0B3-2C2D-4F8A-9A3C- Normally, I should continue translating, but due to the length of the text, I’ll split the response into several parts. Here’s the continuation:

5A4B3C2D1E0F",
          "pattern": "~/Projects/**/bin/rg",
          "lastUsedAt": 1737150000000,
          "lastUsedCommand": "rg -n TODO",
          "lastResolvedPath": "/Users/user/Projects/.../bin/rg"
        }
      ]
    }
  }
}
```

## Режим без утверждений ("YOLO")

Если вы хотите, чтобы выполнение на хосте выполнялось без подсказок об утверждении, необходимо открыть **оба** уровня политики:

- запрошенная политика выполнения в конфигурации OpenClaw (`tools.exec.*`);
- локальная для хоста политика утверждений в `~/.openclaw/exec-approvals.json`.

Это теперь поведение хоста по умолчанию, если вы явно не ужесточите его:

- `tools.exec.security`: `full` на `gateway`/`node`;
- `tools.exec.ask`: `off`;
- `askFallback` хоста: `full`.

Важное различие:

- `tools.exec.host=auto` выбирает, где выполняется exec: в песочнице, если она доступна, в противном случае — на шлюзе.
- YOLO выбирает способ утверждения выполнения на хосте: `security=full` плюс `ask=off`.
- В режиме YOLO OpenClaw не добавляет отдельный эвристический шлюз утверждения об обфускации команд поверх настроенной политики выполнения на хосте.
- `auto` не делает маршрутизацию через шлюз бесплатным переопределением из сеанса в песочнице. Запрос `host=node` для каждого вызова разрешён из `auto`, а `host=gateway` разрешён из `auto` только тогда, когда среда выполнения в песочнице не активна. Если вам нужен стабильный неавтоматический режим по умолчанию, установите `tools.exec.host` или используйте `/exec host=...` явно.

Если вы хотите более консервативную настройку, верните любой уровень обратно в `allowlist` / `on-miss` или `deny`.

Постоянная настройка "никогда не запрашивать" для хоста шлюза:

```bash
openclaw config set tools.exec.host gateway
openclaw config set tools.exec.security full
openclaw config set tools.exec.ask off
openclaw gateway restart
```

Затем установите файл утверждений хоста соответствующим образом:

```bash
openclaw approvals set --stdin <<'EOF'
{
  version: 1,
  defaults: {
    security: "full",
    ask: "off",
    askFallback: "full"
  }
}
EOF
```

Для хоста узла примените тот же файл утверждений на этом узле:

```bash
openclaw approvals set --node <id|name|ip> --stdin <<'EOF'
{
  version: 1,
  defaults: {
    security: "full",
    ask: "off",
    askFallback: "full"
  }
}
EOF
```

Ярлык для сеанса:

- `/exec security=full ask=off` изменяет только текущий сеанс.
- `/elevated full` — это аварийный ярлык, который также пропускает утверждения exec для этого сеанса.

Если файл утверждений хоста остаётся более строгим, чем конфигурация, более строгая политика хоста по-прежнему имеет приоритет.

## Параметры политики

### Безопасность (`exec.security`)

- **deny**: блокировать все запросы на выполнение на хосте.
- **allowlist**: разрешать только команды из списка разрешённых.
- **full**: разрешать всё (эквивалентно повышенному режиму).

### Запрос (`exec.ask`)

- **off**: никогда не запрашивать.
- **on-miss**: запрашивать только в том случае, если список разрешённых не соответствует.
- **always**: запрашивать для каждой команды.
- Долгосрочное доверие `allow-always` не подавляет запросы, когда эффективный режим `ask` — `always`.

### Резервный вариант запроса (`askFallback`)

Если требуется запрос, но пользовательский интерфейс недоступен, резервный вариант определяет следующее:

- **deny**: блокировать.
- **allowlist**: разрешать только в том случае, если соответствует список разрешённых.
- **full**: разрешать.

### Укрепление встроенной оценки интерпретатора (`tools.exec.strictInlineEval`)

Когда `tools.exec.strictInlineEval=true`, OpenClaw обрабатывает встроенные формы оценки кода как требующие утверждения, даже если сам двоичный файл интерпретатора включён в список разрешённых.

Примеры:

- `python -c`
- `node -e`, `node --eval`, `node -p`
- `ruby -e`
- `perl -e`, `perl -E`
- `php -r`
- `lua -e`
- `osascript -e`

Это дополнительная защита для загрузчиков интерпретаторов, которые не сопоставляются с одним стабильным файлом-операндом. В строгом режиме:

- эти команды по-прежнему требуют явного утверждения;
- `allow-always` не сохраняет автоматически новые записи в списке разрешённых для них.

## Список разрешённых (для каждого агента)

Списки разрешённых **для каждого агента**. Если существует несколько агентов, переключитесь на агента, которого вы редактируете, в приложении macOS. Шаблоны представляют собой **нечувствительные к регистру совпадения glob**. Шаблоны должны соответствовать **путям к двоичным файлам** (записи только с базовым именем игнорируются). Устаревшие записи `agents.default` переносятся в `agents.main` при загрузке. Цепочки оболочек, такие как `echo ok && pwd`, по-прежнему требуют, чтобы каждый сегмент верхнего уровня соответствовал правилам списка разрешённых.

Примеры:

- `~/Projects/**/bin/peekaboo`
- `~/.local/bin/*`
- `/opt/homebrew/bin/rg`

Каждая запись в списке разрешённых отслеживает:

- **id** — стабильный UUID, используемый для идентификации в пользовательском интерфейсе (необязательно);
- **последний использованный** — отметка времени;
- **последняя использованная команда**;
- **последний разрешённый путь**.

## Автоматическое разрешение CLI навыков

Когда **автоматическое разрешение CLI навыков** включено, исполняемые файлы, на которые ссылаются известные навыки, считаются включёнными в список разрешённых на узлах (узел macOS или автономный хост узла). Для получения списка двоичных файлов навыков используется `skills.bins` через Gateway RPC. Отключите эту функцию, если вам нужны строго ручные списки разрешённых.

Важные примечания о доверии:

- Это **неявный удобный список разрешённых**, отдельный от записей ручного списка разрешённых путей.
- Он предназначен для сред с доверенными операторами, где шлюз и узел находятся в одной границе доверия.
- Если вам требуется строгое явное доверие, оставьте `autoAllowSkills: false` и используйте только записи ручного списка разрешённых путей.

## Безопасные двоичные файлы (только stdin)

`tools.exec.safeBins` определяет небольшой список двоичных файлов **только для stdin** (например, `cut`), которые могут работать в режиме списка разрешённых **без** явных записей в списке разрешённых. Безопасные двоичные файлы отклоняют позиционные аргументы файлов и токены, похожие на пути, поэтому они могут работать только с входящим потоком. Рассматривайте это как узкий быстрый путь для фильтров потоков, а не как общий список доверия.

**Не** добавляйте двоичные файлы интерпретатора или среды выполнения (например, `python3`, `node`, `ruby`, `bash`, `sh`, `zsh`) в `safeBins`. Если команда может оценивать код, выполнять подкоманды или читать файлы по замыслу, предпочитайте явные записи в списке разрешённых и оставляйте подсказки об утверждении включёнными.

Пользовательские безопасные двоичные файлы должны определять явный профиль в `tools.exec.safeBinProfiles.<bin>`. Проверка выполняется детерминированно только на основе формы argv (без проверок существования в файловой системе хоста), что предотвращает поведение оракула существования файла из-за различий в разрешении/отклонении.

Опции, ориентированные на файлы, отклоняются для безопасных двоичных файлов по умолчанию (например, `sort -o`, `sort --output`, `sort --files0-from`, `sort --compress-program`, `sort --random-source`, `sort --temporary-directory`/`-T`, `wc --files0-from`, `jq -f/--from-file`, `grep -f/--file`).

Безопасные двоичные файлы также обеспечивают соблюдение явной политики флагов для каждого двоичного файла для опций, нарушающих поведение только для stdin (например, `sort -o/--output/--compress-program` и рекурсивные флаги grep). Длинные опции проверяются с отказом в безопасном режиме: неизвестные флаги и неоднозначные сокращения отклоняются.

Флаги, отклонённые профилем безопасного двоичного файла:

[//]: # "SAFE_BIN_DENIED_FLAGS:START"

- `grep`: `--dereference-recursive`, `--directories`, `--exclude-from`, `--file`, `--recursive`, `-R`, `-d`, `-f`, `-r`
- `jq`: `--argfile`, `--from-file`, `--library-path`, `--rawfile`, `--slurpfile`, `-L`, `-f`
- `sort`: `--compress-program`, `--files0-from`, `--output`, `--random-source`, `--temporary-directory`, `-T`, `-o`
- `wc`: `--files0-from`

[//]: # "SAFE_BIN_DENIED_FLAGS:END"

Безопасные двоичные файлы также заставляют токены argv обрабатываться как **буквальный текст** во время выполнения (без globbing и без расширения `$VARS`) для сегментов только для stdin, поэтому шаблоны вроде `*` или `$HOME/...` нельзя использовать для скрытного чтения файлов.

Безопасные двоичные файлы также должны разрешаться из доверенных каталогов двоичных файлов (системные значения по умолчанию плюс необязательный `tools.exec.safeBinTrustedDirs`). Записи `PATH` никогда не доверяются автоматически.

Каталоги безопасных двоичных файлов по умолчанию намеренно минимальны: `/bin`, `/usr/bin`. Если ваш безопасный исполняемый файл находится в путях менеджера пакетов/пользователя (например, `/opt/homebrew/bin`, `/usr/local/bin`, `/opt/local/bin`, `/snap/bin`), добавьте их явно в `tools.exec.safeBinTrustedDirs`.

Цепочки оболочек и перенаправления не разрешаются автоматически в режиме списка разрешённых.

Цепочки оболочек (`&&`, `||`, `;`) разрешены, когда каждый сегмент верхнего уровня удовлетворяет списку разрешённых (включая безопасные двоичные файлы или автоматическое разрешение навыков). Перенаправления остаются неподдерживаемыми в режиме списка разрешённых.

Подстановка команд (`$()` / обратные кавычки) отклоняется во время анализа списка разрешённых, в том числе внутри двойных кавычек; используйте одинарные кавычки, если вам нужен буквальный текст `$()`.

В утверждениях сопутствующего приложения macOS сырой текст оболочки, содержащий синтаксис управления оболочкой или расширения (`&&`, `||`, `;`, `|`, `` ` ``, `$`, `<`, `>`, `(`, `)`), считается пропуском списка разрешённых, если сам двоичный файл оболочки не включён в список разрешённых.

Для оболочек-обёрток (`bash|sh|zsh ... -c/-lc`) переопределения среды для области запроса сокращаются до небольшого явного списка разрешённых (`TERM`, `LANG`, `LC_*`, `COLORTERM`, `NO_COLOR`, `FORCE_COLOR`).

Для решений `allow-always` в режиме списка разрешённых известные диспетчерские обёртки (`env`, `nice`, `nohup`, `stdbuf`, `timeout`) сохраняют пути к внутренним исполняемым файлам вместо путей к обёрткам. Мультиплексоры оболочек (`busybox`, `toybox`) также распаковываются для апплетов оболочек (`sh`, `ash` и т. д.), поэтому сохраняются внутренние исполняемые файлы вместо двоичных файлов мультиплексора. Если обёртку или мультиплексор нельзя безопасно распаковать, запись в списке разрешённых не сохраняется автоматически.

Если вы включаете в список разрешённых интерпретаторы, такие как `python3` или `node`, предпочитайте `tools.exec.strictInlineEval=true`, чтобы встроенная оценка по-прежнему требовала явного утверждения. В строгом режиме `allow-always` может по-прежнему сохранять безобидные вызовы интерпретатора/сценария, но носители встроенной оценки не сохраняются автоматически.

Безопасные двоичные файлы по умолчанию:

[//]: # "SAFE_BIN_DEFAULTS:START"

`cut`, `uniq`, `head`, `tail`, `tr`, `wc`

[//]: # "SAFE_BIN_DEFAULTS:END"

`grep` и `sort` не входят в список по умолчанию. Если вы включаете их, оставляйте явные записи в списке разрешённых для их рабочих процессов, не относящихся к stdin.

Для `grep` в режиме безопасного двоичного файла укажите шаблон с `-e`/`--regexp`; позиционная форма шаблона отклоняется, чтобы операнды файлов не могли быть скрыты как неоднозначные позиционные аргументы.

### Безопасные двоичные файлы против списка разрешённых

| Тема | `tools.exec.safeBins` | Список разрешённых (`exec-approvals.json`) |
| ---------------- | ------------------------------------------------------ | ------------------------------------------------------------ |
| Цель | Автоматическое разрешение узких фильтров stdin | Явное доверие конкретным исполняемым файлам |
| Тип соответствия | Имя исполняемого файла + политика argv безопасного двоичного файла | Разрешённый путь к исполняемому файлу, шаблон glob |
| Область аргументов | Ограничено профилем безопасного двоичного файла и правилами буквальных токенов | Только соответствие пути; за аргументы в остальном отвечаете вы |
| Типичные примеры | `head`, `tail`, `tr`, `wc` | `jq`, `python3`, `node`, `ffmpeg`, пользовательские CLI |
| Лучшее применение | Преобразование текста с низким риском в конвейерах | Любой инструмент с более широким поведением или побочными эффектами |

Расположение конфигурации:

- `safeBins` берётся из конфигурации (`tools.exec.safeBins` или `agents.list[].tools.exec.safeBins` для каждого агента).
- `safeBinTrustedDirs` берётся из конфигурации (`tools.exec.safeBinTrustedDirs` или `agents.list[].tools.exec.safeBinTrustedDirs` для каждого агента).
- `safeBinProfiles` берётся из конфигурации (`tools.exec.safeBinProfiles` или `agents.list[].tools.exec.safeBinProfiles` для каждого агента). Ключи профиля для каждого агента переопределяют глобальные ключи.
- Записи списка разрешённых находятся в локальном для хоста `~/.openclaw/exec-approvals.json` в `agents.<id>.allowlist` (или через Control UI / `openclaw approvals allowlist ...`).
- `openclaw security audit` выдаёт предупреждение с `tools.exec.safe_bins_interpreter_unprofiled`, когда двоичные файлы интерпретатора/среды выполнения появляются в `safeBins` без явных профилей.
- `openclaw doctor --fix` может создать недостающие пользовательские записи `safeBinProfiles.<bin>` как `{}` (просмотрите и ужесточите впоследствии). Двоичные файлы интерпретатора/среды выполнения не создаются автоматически.

Пример пользовательского профиля:

```json5
{
  tools: {
    exec: {
      safeBins: ["jq", "myfilter"],
      safeBinProfiles: {
        myfilter: {
          minPositional: 0,
          maxPositional: 0,
          allowedValueFlags: ["-n", "--limit"],
          deniedFlags: ["-f", "--file", "-c", "--command"],
        },
      },
    },
  },
}
```

Если вы явно включаете `jq` в `safeBins`, OpenClaw по-прежнему отклоняет встроенный `env` в режиме безопасного двоичного файла, поэтому `jq -n env` не может вывести среду хост-процесса без явного пути к списку разрешённых или подсказки об утверждении.

## Редактирование через Control UI

Используйте карточку **Control UI → Nodes → Exec approvals**, чтобы редактировать значения по умолчанию, переопределения для каждого агента и списки разрешённых. Выберите область (значения по умолчанию или агент), настройте политику, добавьте/удалите шаблоны списка разрешённых, затем **сохраните**. Пользовательский интерфейс отображает метаданные **последнего использования** для каждого шаблона, чтобы вы могли поддерживать список в порядке.

Селектор цели выбирает **шлюз** (локальные утверждения) или **узел**. Узлы должны объявлять `system.execApprovals.get/set` (приложение macOS или автономный хост узла). Если узел ещё не объявляет утверждения exec, отредактируйте его локальный `~/.openclaw/exec-approvals.json` напрямую.

CLI: `openclaw approvals` поддерживает редактирование шлюза или узла (см. [Approvals CLI](/cli/approvals)).

## Поток утверждений

Когда требуется подсказка, шлюз транслирует `exec.approval.requested` клиентам оператора. Control UI и приложение macOS разрешают это через `exec.approval.resolve`, затем шлюз пересылает утверждённый запрос на хост узла.

Для `host=node` запросы на утверждение включают каноническую полезную нагрузку `systemRunPlan`. Шлюз использует этот план как авторитетный контекст команды/cwd/сеанса при пересылке утверждённых запросов `system.run`.

Это имеет значение для задержки асинхронного утверждения:

- путь выполнения узла заранее подготавливает один канонический план;
- запись утверждения хранит этот план и его метаданные привязки;
- после утверждения окончательный пересланный вызов `system.run` повторно использует сохранённый план вместо того, чтобы доверять более поздним изменениям вызывающего;
- если вызывающий изменяет `command`, `rawCommand`, `cwd`, `agentId` или `sessionKey` после создания запроса на утверждение, шлюз отклоняет пересланное выполнение как несоответствие утверждению.

## Команды интерпретатора/среды выполнения

Выполнения интерпретатора/среды выполнения с поддержкой утверждений намеренно консервативны:

- Всегда привязывается точный контекст argv/cwd/env.
- Прямые сценарии оболочки и прямые формы файлов среды выполнения привязываются к одному конкретному локальному снимку файла по принципу "наилучших усилий".
- Общие формы-обёртки диспетчера пакетов, которые по-прежнему разрешаются в один прямой локальный файл (например, `pnpm exec`, `pnpm node`, `npm exec`, `npx`), распаковываются перед привязкой.
- Если OpenClaw не может определить ровно один конкретный локальный файл для команды интерпретатора/среды выполнения (например, сценарии пакетов, формы eval, цепочки загрузчиков, специфичные для среды выполнения, или неоднозначные формы с несколькими файлами), выполнение с поддержкой утверждений отклоняется вместо того, чтобы претендовать на семантическое покрытие, которого у него нет.
- Для этих рабочих процессов предпочитайте песочницу, отдельную границу хоста или явный доверенный список разрешённых/полный рабочий процесс, где оператор принимает более широкую семантику среды выполнения.

Когда требуются утверждения, инструмент exec немедленно возвращает идентификатор утверждения. Используйте этот идентификатор для сопоставления последующих системных событий (`Exec finished` / `Exec denied`). Если решение не поступает до истечения времени ожидания, запрос считается тайм-аутом утверждения и отображается как причина отклонения.

### Поведение последующей доставки

После завершения асинхронного выполнения с утверждением OpenClaw отправляет последующее действие `agent` в тот же сеанс.

- Если существует действительная внешняя цель доставки (канал с возможностью доставки плюс целевой `to`), последующая доставка использует этот канал.
- В потоках только для веб-чата или внутренних сеансов без внешней цели последующая доставка остаётся только для сеанса (`deliver: false`).
- Если вызывающий явно запрашивает строгую внешнюю доставку без разрешимого внешнего канала, запрос завершается ошибкой с `INVALID_REQUEST`.
- Если включён `bestEffortDeliver` и внешний канал не может быть разрешён, доставка понижается до сеанса вместо сбоя.

Диалог подтверждения включает:

- команду и аргументы;
- cwd;
- идентификатор агента;
- разрешённый путь к исполняемому файлу;
- хост и метаданные политики.

Действия:

- **Разрешить один раз** → запустить сейчас;
- **Всегда разрешать** → добавить в список разрешённых и запустить;
- **Отклонить** → заблокировать.

## Пересылка утверждений в каналы чата

Вы можете пересылать подсказки об утверждении exec в любой канал чата (включая каналы плагинов) и утверждать их с помощью `/approve`. Для этого используется обычный конвейер исходящей доставки.

Конфигурация:

```json5
{
  approvals: {
    exec: {
      enabled: true,
      mode: "session", // "session" | "targets" | "both"
      agentFilter: ["main"],
      sessionFilter: ["discord"], // подстрока или регулярное выражение
      targets: [
        { channel: "slack", to: "U12345678" },
        { channel: "telegram", to: "123456789" },
      ],
    },
  },
}
```

Ответ в чате:

```
/approve <id> allow-once
/approve <id> allow-always
/approve <id> deny
```

Команда `/approve` обрабатывает как утверждения exec, так и утверждения плагинов. Если идентификатор не соответствует ожидающему утверждению exec, он автоматически проверяет утверждения плагинов.

### Пересылка утверждений плагинов

Пересылка утверждений плагинов использует тот же конвейер доставки, что и утверждения exec, но имеет свою независимую конфигурацию в `approvals.plugin`. Включение или отключение одного не влияет на другое.

```json5
{
  approvals: {
    plugin: {
      enabled: true,
      mode: "targets",
      agentFilter: ["main"],
      targets: [
        { channel: "slack", to: "U12345678" },
        { channel: "telegram", to: "123456789" },
      ],
    },
  },
}
```

Форма конфигурации идентична `approvals.exec`: `enabled`, `mode`, `agentFilter`, `sessionFilter` и `targets` работают одинаково.

Каналы, поддерживающие общие интерактивные ответы, отображают одинаковые кнопки утверждения как для утверждений exec, так и для утверждений плагинов. Каналы без общего интерактивного пользовательского интерфейса возвращаются к простому тексту с инструкциями `/approve`.

### Утверждения в том же чате для любого канала

Когда запрос на утверждение exec или плагина исходит из доставляемой поверхности чата, тот же чат теперь может утверждать его с помощью `/approve` по умолчанию. Это относится к таким каналам, как Slack, Matrix и Microsoft Teams, в дополнение к существующим потокам веб-интерфейса и терминала.

Этот общий путь текстовых команд использует обычную модель авторизации канала для этого разговора. Если исходный чат уже может отправлять команды и получать ответы, запросы на утверждение больше не требуют отдельного адаптера доставки только для того, чтобы оставаться ожидающими.

Discord и Telegram также поддерживают `/approve` в том же чате, но эти каналы по-прежнему используют свой разрешённый список утверждающих для авторизации, даже когда нативная доставка утверждений отключена.

Для Telegram и других клиентов нативных утверждений, которые напрямую обращаются к шлюзу, этот запасной вариант намеренно ограничен сбоями "утверждение не найдено". Реальное отклонение/ошибка утверждения exec не повторяет попытку молча как утверждение плагина.

### Нативная доставка утверждений

Некоторые каналы также могут выступать в качестве нативных клиентов утверждений. Нативные клиенты добавляют личные сообщения утверждающих, рассылку из исходного чата и интерактивный пользовательский интерфейс утверждения для конкретного канала поверх общего потока `/approve` в том же чате.

Когда доступны карточки/кнопки нативных утверждений, этот нативный пользовательский интерфейс является основным путём, ориентированным на агента. Агент не должен также дублировать команду `/approve` в простом чате, если только результат инструмента не говорит, что утверждения в чате недоступны или ручной способ утверждения — единственный оставшийся путь.

Общая модель:

- политика выполнения на хосте по-прежнему определяет, требуется ли утверждение exec;
- `approvals.exec` управляет пересылкой подсказок об утверждении в другие места назначения в чате;
- `channels.<channel>.execApprovals` управляет тем, действует ли этот канал как нативный клиент утверждений.

Нативные клиенты утверждений автоматически включают доставку в личные сообщения, когда выполняются все следующие условия:

- канал поддерживает нативную доставку утверждений;
- утверждающих можно разрешить из явного `execApprovals.approvers` или из документированных резервных источников этого канала;
- `channels.<channel>.execApprovals.enabled` не задан или имеет значение `"auto"`.

Установите `enabled: false`, чтобы явно отключить нативного клиента утверждений. Установите `enabled: true`, чтобы принудительно включить его, когда утверждающие разрешены. Публичная доставка из исходного чата остаётся явной через `channels.<channel>.execApprovals.target`.

FAQ: [Почему существует две конфигурации утверждений exec для утверждений в чате?](/help/faq#why-are-there-two-exec-approval-configs-for-chat-approvals)

- Discord: `channels.discord.execApprovals.*`
- Slack: `channels.slack.execApprovals.*`
- Telegram: `channels.telegram.execApprovals.*`

Эти нативные клиенты утверждений добавляют маршрутизацию личных сообщений и необязательную рассылку по каналам поверх общего потока `/approve` в том же чате и общих кнопок утверждения.

Общее поведение:

- Slack, Matrix, Microsoft Teams и аналогичные доставляемые чаты используют обычную модель авторизации канала для `/approve` в том же чате;
- когда нативный клиент утверждений автоматически включается, целью доставки по умолчанию являются личные сообщения утверждающих;
- для Discord и Telegram только разрешённые утверждающие могут утверждать или отклонять;
- утверждающие Discord могут быть явными (`execApprovals.approvers`) или выведены из `commands.ownerAllowFrom`;
- утверждающие Telegram могут быть явными (`execApprovals.approvers`) или выведены из существующей конфигурации владельца (`allowFrom`, плюс `defaultTo` для прямых сообщений, где это поддерживается);
- утверждающие Slack могут быть явными (`execApprovals.approvers`) или выведены из `commands.ownerAllowFrom`;
- нативные кнопки Slack сохраняют вид идентификатора утверждения, поэтому идентификаторы `plugin:` могут разрешать утверждения плагинов без второго локального для Slack запасного уровня;
- нативная маршрутизация личных сообщений/каналов Matrix и ярлыки реакций обрабатывают как утверждения exec, так и утверждения плагинов; авторизация плагинов по-прежнему поступает из `channels.matrix.dm.allowFrom`;
- запрашивающий не обязательно должен быть утверждающим;
- исходный чат может утверждать напрямую с помощью `/approve`, когда этот чат уже поддерживает команды и ответы;
- нативные кнопки утверждения Discord маршрутизируют по виду идентификатора утверждения: идентификаторы `plugin:` идут прямо к утверждениям плагинов, всё остальное идёт к утверждениям exec;
- нативные кнопки утверждения Telegram следуют тому же ограниченному откату от exec к plugin, что и `/approve`;
- когда нативный `target` включает доставку из исходного чата, подсказки об утверждении включают текст команды;
- ожидающие утверждения exec истекают через 30 минут по умолчанию;
- если ни один пользовательский интерфейс оператора, ни настроенный клиент утверждений не могут принять запрос, подсказка возвращается к `askFallback`.

Telegram по умолчанию использует личные сообщения утверждающих (`target: "dm"`). Вы можете переключиться на `channel` или `both`, когда хотите, чтобы подсказки об утверждении также отображались в исходном чате/теме Telegram. Для тем форума Telegram OpenClaw сохраняет тему для подсказки об утверждении и последующего действия после утверждения.

См.:

- [Discord](/channels/discord)
- [Telegram](/channels/telegram)

### Поток IPC для macOS

```
Gateway -> Node Service (WS)
                 |  IPC (UDS + token + HMAC + TTL)
                 v
             Mac App (UI + approvals + system.run)
```

Примечания по безопасности:

- режим сокета Unix `0600`, токен хранится в `exec-approvals.json`;
- проверка однопользовательского доступа (same-UID);
- вызов/ответ (nonce + HMAC-токен + хэш запроса) + короткий TTL.

## Системные события

Жизненный цикл exec отображается в виде системных сообщений:

- `Exec running` (только если команда превышает порог уведомления о выполнении);
- `Exec finished`;
- `Exec denied`.

Они публикуются в сеансе агента после того, как узел сообщает о событии. Утверждения выполнения на хосте шлюза генерируют те же события жизненного цикла, когда команда завершается (и, возможно, когда выполняется дольше порога). Выполнения с утверждением повторно используют идентификатор утверждения как `runId` в этих сообщениях для простого сопоставления.

## Поведение при отклонении утверждения

Когда асинхронное утверждение exec отклоняется, OpenClaw предотвращает повторное использование агентом вывода из любого предыдущего выполнения той же команды в сеансе. Причина отклонения передаётся с явным указанием на то, что вывод команды недоступен, что не позволяет агенту утверждать, что есть новый вывод, или повторять отклонённую команду с устаревшими результатами из предыдущего успешного выполнения.

## Последствия

- **full** обладает большой мощностью; по возможности предпочитайте списки разрешённых.
- **ask** держит вас в курсе, позволяя при этом быстро утверждать.
- Списки разрешённых для каждого агента предотвращают утечку утверждений одного агента в другие.
- Утверждения применяются только к запросам выполнения на хосте от **авторизованных отправителей**. Неавторизованные отправители не могут выполнять `/exec`.
- `/exec security=full` — это удобство на уровне сеанса для авторизованных операторов, которое по замыслу пропускает утверждения. Чтобы жёстко заблокировать выполнение на хосте, установите безопасность утверждений в `deny` или запретите инструмент `exec` через политику инструментов.
