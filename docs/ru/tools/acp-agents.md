# Агенты ACP

Сессии [протокола клиента агента (Agent Client Protocol, ACP)](https://agentclientprotocol.com/) позволяют OpenClaw запускать внешние упряжи кодирования (например, Pi, Claude Code, Codex, Cursor, Copilot, OpenClaw ACP, OpenCode, Gemini CLI и другие поддерживаемые упряжи ACPX) через плагин бэкенда ACP.

Если вы попросите OpenClaw на обычном языке "запустить это в Codex" или "запустить Claude Code в потоке", OpenClaw должен направить этот запрос в ACP runtime (а не в среду выполнения нативного субагента). Каждое создание сессии ACP отслеживается как [фоновая задача](/automation/tasks).

Если вы хотите, чтобы Codex или Claude Code подключались как внешний клиент MCP напрямую к существующим диалогам каналов OpenClaw, используйте вместо ACP команду [`openclaw mcp serve`](/cli/mcp).

## Какую страницу мне выбрать?

Есть три похожих раздела, которые легко спутать:

| Вы хотите… | Используйте | Примечания |
|------------|------------|------------|
| Запустить Codex, Claude Code, Gemini CLI или другую внешнюю упряжь _через_ OpenClaw | Эту страницу: агенты ACP | Сессии, привязанные к чату, `/acp spawn`, `sessions_spawn({ runtime: "acp" })`, фоновые задачи, элементы управления средой выполнения |
| Предоставить сессию OpenClaw Gateway _в качестве_ сервера ACP для редактора или клиента | [`openclaw acp`](/cli/acp) | Режим моста. IDE/клиент взаимодействует с OpenClaw по протоколу ACP через stdio/WebSocket |
| Использовать локальный CLI ИИ в качестве резервной текстовой модели | [Бэкенды CLI](/gateway/cli-backends) | Не ACP. Нет инструментов OpenClaw, нет элементов управления ACP, нет среды выполнения упряжи |

## Работает ли это "из коробки"?

Обычно да.

* В новых установках встроенный плагин среды выполнения `acpx` по умолчанию включён.
* Встроенный плагин `acpx` отдаёт предпочтение закреплённому в плагине бинарному файлу `acpx`.
* При запуске OpenClaw проверяет этот бинарный файл и при необходимости самостоятельно его восстанавливает.
* Для быстрой проверки готовности начните с `/acp doctor`.

Что может произойти при первом использовании:

* Адаптер целевой упряжи может быть загружен по требованию с помощью `npx` при первом использовании этой упряжи.
* На хосте должны быть настроены аутентификационные данные поставщика для этой упряжи.
* Если на хосте нет доступа к npm/сети, загрузка адаптеров при первом запуске может завершиться ошибкой, пока кэши не будут предварительно заполнены или адаптер не будет установлен другим способом.

Примеры:

* `/acp spawn codex`: OpenClaw должен быть готов к загрузке `acpx`, но адаптеру Codex ACP может потребоваться загрузка при первом запуске.
* `/acp spawn claude`: то же самое для адаптера Claude ACP, плюс аутентификация на стороне Claude на этом хосте.

## Быстрый рабочий процесс для оператора

Используйте это, если вам нужен практический справочник по командам `/acp`:

1. Создайте сессию:
   * `/acp spawn codex --bind here`
   * `/acp spawn codex --mode persistent --thread auto`
2. Работайте в привязанном диалоге или потоке (либо явно укажите ключ этой сессии).
3. Проверьте состояние среды выполнения:
   * `/acp status`
4. Настройте параметры среды выполнения по мере необходимости:
   * `/acp model <provider/model>`
   * `/acp permissions <profile>`
   * `/acp timeout <seconds>`
5. Направьте активную сессию без замены контекста:
   * `/acp steer tighten logging and continue`
6. Завершите работу:
   * `/acp cancel` (остановить текущий ход), или
   * `/acp close` (закрыть сессию и удалить привязки)

## Быстрый старт для пользователей

Примеры естественных запросов:

* "Привяжите этот канал Discord к Codex".
* "Запустите постоянную сессию Codex в потоке здесь и сохраняйте фокус".
* "Запустите это как одноразовую сессию Claude Code ACP и суммируйте результат".
* "Привяжите этот чат iMessage к Codex и сохраняйте последующие запросы в том же рабочем пространстве".
* "Используйте Gemini CLI для этой задачи в потоке, затем сохраняйте последующие запросы в том же потоке".

Что должен сделать OpenClaw:

1. Выбрать `runtime: "acp"`.
2. Определить целевую упряжь (`agentId`, например `codex`).
3. Если запрошена привязка к текущему диалогу и активный канал её поддерживает, привязать сессию ACP к этому диалогу.
4. В противном случае, если запрошена привязка к потоку и текущий канал её поддерживает, привязать сессию ACP к потоку.
5. Направлять последующие привязанные сообщения в ту же сессию ACP до тех пор, пока она не потеряет фокус, не будет закрыта или не истечёт срок её действия.

## ACP против субагентов

Используйте ACP, если вам нужна внешняя среда выполнения упряжи. Используйте субагентов, если вам нужны делегированные запуски, встроенные в OpenClaw.

| Область | Сессия ACP | Запуск субагента |
|--------|-----------|------------------|
| Среда выполнения | Плагин бэкенда ACP (например, acpx) | Встроенная среда выполнения субагентов OpenClaw |
| Ключ сессии | `agent:<agentId>:acp:<uuid>` | `agent:<agentId>:subagent:<uuid>` |
| Основные команды | `/acp ...` | `/subagents ...` |
| Инструмент создания | `sessions_spawn` с `runtime:"acp"` | `sessions_spawn` (среда выполнения по умолчанию) |

См. также [Субагенты](/tools/subagents).

## Как ACP запускает Claude Code

Для Claude Code через ACP стек выглядит следующим образом:

1. Плоскость управления сессией ACP в OpenClaw.
2. Встроенный плагин среды выполнения `acpx`.
3. Адаптер Claude ACP.
4. Механизмы выполнения/сессии на стороне Claude.

Важное различие:

* ACP Claude — это сессия упряжи с элементами управления ACP, возобновлением сессии, отслеживанием фоновых задач и необязательной привязкой к диалогу/потоку.
* Бэкенды CLI — это отдельные локальные резервные среды выполнения только для текста. См. [Бэкенды CLI](/gateway/cli-backends).

Для операторов практическое правило:

* Если вам нужны `/acp spawn`, привязываемые сессии, элементы управления средой выполнения или постоянная работа с упряжью — используйте ACP.
* Если вам нужен простой локальный текстовый резерв через сырой CLI — используйте бэкенды CLI.

## Привязанные сессии

### Привязки к текущему диалогу

Используйте `/acp spawn <harness> --bind here`, если хотите, чтобы текущий диалог стал устойчивым рабочим пространством ACP без создания дочернего потока.

Поведение:

* OpenClaw сохраняет управление транспортом канала, аутентификацией, безопасностью и доставкой.
* Текущий диалог привязывается к ключу созданной сессии ACP.
* Последующие сообщения в этом диалоге направляются в ту же сессию ACP.
* Команды `/new` и `/reset` сбрасывают ту же привязанную сессию ACP на месте.
* Команда `/acp close` закрывает сессию и удаляет привязку к текущему диалогу.

Что это означает на практике:

* `--bind here` сохраняет тот же интерфейс чата. В Discord текущий канал остаётся текущим каналом.
* `--bind here` всё равно может создать новую сессию ACP, если вы запускаете новую работу. Привязка привязывает эту сессию к текущему диалогу.
* `--bind here` сам по себе не создаёт дочерний поток Discord или тему Telegram.
* Среда выполнения ACP всё равно может иметь свой собственный рабочий каталог (`cwd`) или управляемое бэкендом рабочее пространство на диске. Это рабочее пространство среды выполнения отделено от интерфейса чата и не подразумевает создание нового потока обмена сообщениями.
* Если вы запускаете работу для другого агента ACP и не передаёте `--cwd`, OpenClaw по умолчанию наследует рабочее пространство **целевого агента**, а не запрашивающего.
* Если унаследованный путь к рабочему пространству отсутствует (`ENOENT`/`ENOTDIR`), OpenClaw вместо этого переходит к рабочему каталогу по умолчанию для бэкенда, а не повторно использует неправильное дерево.
* Если унаследованное рабочее пространство существует, но к нему нет доступа (например, `EACCES`), команда spawn возвращает реальную ошибку доступа вместо того, чтобы игнорировать `cwd`.

Ментальная модель:

* интерфейс чата: где люди продолжают общаться (`канал Discord`, `тема Telegram`, `чат iMessage`);
* сессия ACP: устойчивое состояние среды выполнения Codex/Claude/Gemini, к которому OpenClaw направляет запросы;
* дочерний поток/тема: необязательный дополнительный интерфейс обмена сообщениями, создаваемый только с помощью `--thread ...`;
* рабочее пространство среды выполнения: расположение в файловой системе, где выполняется упряжь (`cwd`, проверка репозитория, рабочее пространство бэкенда).

Примеры:

* `/acp spawn codex --bind here`: сохранить этот чат, запустить или подключить сессию Codex ACP и направлять будущие сообщения сюда в эту сессию;
* `/acp spawn codex --thread auto`: OpenClaw может создать дочерний поток/тему и привязать к ней сессию ACP;
* `/acp spawn codex --bind here --cwd /workspace/repo`: та же привязка чата, что и выше, но Codex работает в `/workspace/repo`.

Поддержка привязки к текущему диалогу:

* Каналы чата/сообщений, которые объявляют поддержку привязки к текущему диалогу, могут использовать `--bind here` через общий путь привязки диалога.
* Каналы с пользовательской семантикой потоков/тем могут по-прежнему предоставлять канонизацию, специфичную для канала, за тем же общим интерфейсом.
* `--bind here` всегда означает "привязать текущий диалог на месте".
* Общие привязки к текущему диалогу используют общее хранилище привязок OpenClaw и сохраняются после обычных перезапусков шлюза.

Примечания:

* `--bind here` и `--thread ...` взаимоисключающие в `/acp spawn`.
* В Discord `--bind here` привязывает текущий канал или поток на месте. `spawnAcpSessions` требуется только тогда, когда OpenClaw нужно создать дочерний поток для `--thread auto|here`.
* Если активный канал не поддерживает привязки ACP к текущему диалогу, OpenClaw возвращает чёткое сообщение о неподдерживаемой функции.
* Вопросы о `resume` и "новой сессии" относятся к вопросам о сессии ACP, а не к вопросам о канале. Вы можете повторно использовать или заменить состояние среды выполнения без изменения текущего интерфейса чата.

### Сессии, привязанные к потокам

Когда для адаптера канала включены привязки к потокам, сессии ACP могут быть привязаны к потокам:

* OpenClaw привязывает поток к целевой сессии ACP.
* Последующие сообщения в этом потоке направляются в привязанную сессию ACP.
* Вывод ACP доставляется обратно в тот же поток.
* Потеря фокуса, закрытие, архивация, истечение времени бездействия или максимального возраста удаляет привязку.

Поддержка привязки к потокам зависит от адаптера. Если активный адаптер канала не поддерживает привязки к потокам, OpenClaw возвращает чёткое сообщение о неподдерживаемой/недоступной функции.

Необходимые флаги функций для ACP, привязанного к потокам:

* `acp.enabled=true`;
* `acp.dispatch.enabled` включён по умолчанию (установите `false`, чтобы приостановить отправку ACP);
* Флаг создания потоков ACP для адаптера канала включён (зависит от адаптера):
    * Discord: `channels.discord.threadBindings.spawnAcpSessions=true`;
    * Telegram: `channels.telegram.threadBindings.spawnAcpSessions=true`.

### Каналы с поддержкой потоков

* Любой адаптер канала, который предоставляет возможность привязки сессии/потока.
* Текущая встроенная поддержка:
    * потоки/каналы Discord;
    * темы Telegram (темы форума в группах/супергруппах и темы личных сообщений).
* Каналы плагинов могут добавить поддержку через тот же интерфейс привязки.

## Настройки для конкретных каналов

Для неэфемерных рабочих процессов настройте постоянные привязки ACP в записях верхнего уровня `bindings[]`.

### Модель привязки

* `bindings[].type="acp"` обозначает постоянную привязку диалога ACP.
* `bindings[].match` идентифицирует целевой диалог:
    * канал или поток Discord: `match.channel="discord"` + `match.peer.id="<channelOrThreadId>"`;
    * тема форума Telegram: `match.channel="telegram"` + `match.peer.id="<chatId>:topic:<topicId>"`;
    * чат BlueBubbles (личные сообщения/групповой чат): `match.channel="bluebubbles"` + `match.peer.id="<handle|chat_id:*|chat_guid:*|chat_identifier:*>"`. Для стабильных групповых привязок предпочтительно использовать `chat_id:*` или `chat_identifier:*`.
    * чат iMessage (личные сообщения/групповой чат): `match.channel="imessage"` + `match.peer.id="<handle|chat_id:*|chat_guid:*|chat_identifier:*>"`. Для стабильных групповых привязок предпочтительно использовать `chat_id:*`.
* `bindings[].agentId` — идентификатор агента OpenClaw, которому принадлежит привязка.
* Необязательные переопределения ACP находятся в `bindings[].acp`:
    * `mode` (`persistent` или `oneshot`);
    * `label`;
    * `cwd`;
    * `backend`.

### Настройки среды выполнения по умолчанию для агента

Используйте `agents.list[].runtime`, чтобы определить настройки ACP по умолчанию один раз для каждого агента:

* `agents.list[].runtime.type="acp"`;
* `agents.list[].runtime.acp.agent` (идентификатор упряжи, например `codex` или `claude`);
* `agents.list[].runtime.acp.backend`;
* `agents.list[].runtime.acp.mode`;
* `agents.list[].runtime.acp.cwd`.

Приоритет переопределений для привязанных сессий ACP:

1. `bindings[].acp.*`;
2. `agents.list[].runtime.acp.*`;
3. глобальные настройки ACP по умолчанию (например, `acp.backend`).

Пример:

```json5
{
  agents: {
    list: [
      {
        id: "codex",
        runtime: {
          type: "acp",
          acp: {
            agent: "codex",
            backend: "acpx",
            mode: "persistent",
            cwd: "/workspace/openclaw",
          },
        },
      },
      {
        id: "claude",
        runtime: {
          type: "acp",
          acp: { agent: "claude", backend: "acpx", mode: "persistent" },
        },
      },
    ],
  },
  bindings: [
    {
      type: "acp",
      agentId: "codex",
      match: {
        channel: "discord",
        accountId: "default",
        peer: { kind: "channel", id: "222222222222222222" },
      },
      acp: { label: "codex-main" },
    },
    {
      type: "acp",
      agentId: "claude",
      match: {
        channel: "telegram",
        accountId: "default",
        peer: { kind: "group", id: "-1001234567890:topic:42" },
      },
      acp: { cwd: "/workspace/repo-b" },
    },
    {
      type: "route",
      agentId: "main",
      match: { channel: "discord", accountId: "default" },
    },
    {
      type: "route",
      agentId: "main",
      match: { channel: "telegram", accountId: "default" },
    },
  ],
  channels: {
    discord: {
      guilds: {
        "111111111111111111": {
          channels: {
            "222222222222222222": { requireMention: false },
          },
        },
      },
    },
    telegram: {
      groups: {
        "-1001234567890": {
          topics: { "42": { requireMention: false } },
        },
      },
    },
  },
}
```

Поведение:

* OpenClaw гарантирует, что сконфигурированная сессия ACP существует перед использованием.
* Сообщения в этом канале или теме направляются в сконфигурированную сессию ACP.
* В привязанных диалогах команды `/new` и `/reset` сбрасывают тот же ключ сессии ACP на месте.
* Временные привязки среды выполнения (например, созданные потоками фокусировки) по-прежнему применяются там, где они присутствуют.
* Для межагентных запусков ACP без явного `cwd` OpenClaw наследует рабочее пространство целевого агента из конфигурации агента.
* Отсутствующие унаследованные пути к рабочему пространству переходят к рабочему каталогу по умолчанию для бэкенда; ошибки доступа к существующим путям возвращаются как ошибки запуска.

## Запуск сессий ACP (интерфейсы)

### Из `sessions_spawn`

Используйте `runtime: "acp"`, чтобы запустить сессию ACP из хода агента или вызова инструмента.

```json
{
  "task": "Откройте репозиторий и суммируйте неудачные тесты",
  "runtime": "acp",
  "agentId": "codex",
  "thread": true,
  "mode": "session"
}
```

Примечания:

* По умолчанию `runtime` имеет значение `subagent`, поэтому для сессий ACP явно укажите `runtime: "acp"`.
* Если `agentId` опущен, OpenClaw использует `acp.defaultAgent`, если он настроен.
* Для `mode: "session"` требуется `thread: true`, чтобы сохранить постоянный привязанный диалог.

Детали интерфейса:

* `task` (обязательно): начальный промпт, отправляемый в сессию ACP.
* `runtime` (обязательно для ACP): должно быть `"acp"`.
* `agentId` (необязательно): идентификатор целевой упряжи ACP. Если задан, возвращается к `acp.defaultAgent`.
* `thread` (необязательно, по умолчанию `false`): запрос потока привязки, где это поддерживается.
* `mode` (необязательно): `run` (одноразовый) или `session` (постоянный).
    * по умолчанию — `run`;
    * если `thread: true` и режим опущен, OpenClaw может по умолчанию использовать постоянное поведение в соответствии с путём среды выполнения;
    * для `mode: "session"` требуется `thread: true`.
* `cwd` (необязательно): запрошенный рабочий каталог среды выполнения (проверяется политикой бэкенда/среды выполнения). Если опущен, создание ACP наследует рабочее пространство целевого агента, если оно настроено; отсутствующие унаследованные пути переходят к настройкам по умолчанию для бэкенда, а реальные ошибки доступа возвращаются.
* `label` (необязательно): метка для оператора, используемая в тексте сессии/баннера.
* `resumeSessionId` (необязательно): возобновить существующую сессию ACP вместо создания новой. Агент воспроизводит историю своего диалога через `session/load`. Требуется `runtime: "acp"`.
* `streamTo` (необязательно): `"parent"` передаёт сводки прогресса начального запуска ACP обратно в сессию запрашивающего в виде системных событий.
    * Когда доступно, принятые ответы включают `streamLogPath`, указывающий на JSONL-журнал, ограниченный сессией (`<sessionId>.acp-stream.jsonl`), который вы можете отслеживать для получения полной истории ретрансляции.

### Возобновление существующей сессии

Используйте `resumeSessionId`, чтобы продолжить предыдущую сессию ACP вместо запуска новой. Агент воспроизводит историю своего диалога через `session/load`, поэтому он продолжает работу с полным контекстом того, что было раньше.

```json
{
  "task": "Продолжайте с того места, где остановились — исправьте оставшиеся сбои тестов",
  "runtime": "acp",
  "agentId": "codex",
  "resumeSessionId": "<previous-session-id>"
}
```

Распространённые варианты использования:

* Передать сессию Codex с ноутбука на телефон — попросите агента продолжить с того места, где вы остановились.
* Продолжить сеанс кодирования, который вы начали интерактивно в CLI, теперь безголовым способом через вашего агента.
* Возобновить работу, прерванную перезапуском шлюза или истечением времени бездействия.

Примечания:

* `resumeSessionId` требует `runtime: "acp"` — возвращает ошибку, если используется со средой выполнения субагента.
* `resumeSessionId` восстанавливает историю диалога ACP на вышестоящем уровне; `thread` и `mode` по-прежнему применяются нормально к новой сессии OpenClaw, которую вы создаёте, поэтому для `mode: "session"` по-прежнему требуется `thread: true`.
* Целевой агент должен поддерживать `session/load` (Codex и Claude Code поддерживают).
* Если идентификатор сессии не найден, создание завершается ошибкой с чётким сообщением — без автоматического перехода к новой сессии.

### Дымовой тест для оператора

Используйте это после развёртывания шлюза, когда вам нужна быстрая проверка работоспособности ACP spawn в режиме реального времени, а не просто прохождение модульных тестов.

Рекомендуемый этап:

1. Проверьте версию/коммит развёрнутого шлюза на целевом хосте.
2. Убедитесь, что развёрнутый исходный код включает принятие цепочки ACP в `src/gateway/sessions-patch.ts` (`subagent:* or acp:* sessions`).
3. Откройте временную сессию моста ACPX с живым агентом (например, `razor(main)` на `jpclawhq`).
4. Попросите этого агента вызвать `sessions_spawn` со следующими параметрами:
    * `runtime: "acp"`;
    * `agentId: "codex"`;
    * `mode: "run"`;
    * задача: `Reply with exactly LIVE-ACP-SPAWN-OK`.
5. Убедитесь, что агент сообщает:
    * `accepted=yes`;
    * реальный `childSessionKey`;
    * отсутствие ошибки валидатора.
6. Удалите временную сессию моста ACPX.

Пример промпта для живого агента:

```text
Используйте инструмент sessions_spawn сейчас с runtime: "acp", agentId: "codex" и mode: "run".
Установите задачу: "Reply with exactly LIVE-ACP-SPAWN-OK".
Затем сообщите только: accepted=<yes/no>; childSessionKey=<value or none>; error=<exact text or none>.
```

Примечания:

* Оставьте этот дымовой тест на `mode: "run"`, если вы намеренно не тестируете постоянные сессии ACP, привязанные к потокам.
* Не требуйте `streamTo: "parent"` для базового этапа. Этот путь зависит от возможностей запрашивающего/сессии и является отдельной проверкой интеграции.
* Рассматривайте тестирование `mode: "session"`, привязанного к потоку, как второй, более сложный этап интеграции из реального потока Discord или темы Telegram.

## Совместимость с песочницей

Сессии ACP в настоящее время выполняются в среде выполнения хоста, а не внутри песочницы OpenClaw.

Текущие ограничения:

* Если сессия запрашивающего находится в песочнице, создание ACP блокируется как для `sessions_spawn({ runtime: "acp" })`, так и для `/acp spawn`.
    * Ошибка: `Сессии в песочнице не могут создавать сессии ACP, потому что runtime="acp" выполняется на хосте. Используйте runtime="subagent" из сессий в песочнице.`
* `sessions_spawn` с `runtime: "acp"` не поддерживает `sandbox: "require"`.
    * Ошибка: `sessions_spawn sandbox="require" не поддерживается для runtime="acp", потому что сессии ACP выполняются вне песочницы. Используйте runtime="subagent" или sandbox="inherit".`

Используйте `runtime: "subagent"`, когда вам нужно выполнение с принудительным применением песочницы.

### Из команды `/acp`

Используйте `/acp spawn` для явного управления оператором из чата, когда это необходимо.

```text
/acp spawn codex --mode persistent --thread auto
/acp spawn codex --mode oneshot --thread off
/acp spawn codex --bind here
/acp spawn codex --thread here
```

Ключевые флаги:

* `--mode persistent|oneshot`;
* `--bind here|off`;
* `--thread auto|here|off`;
* `--cwd <absolute-path>`;
* `--label <name>`.

См. [Косые команды](/tools/slash-commands).

## Разрешение целевого объекта сессии

Большинство действий `/acp` принимают необязательный целевой объект сессии (`session-key`, `session-id` или `session-label`).

Порядок разрешения:

1. Явный аргумент цели (или `--session` для `/acp steer`):
    * пытается использовать ключ;
    * затем UUID-подобный идентификатор сессии;
    * затем метку.
2. Привязка к текущему потоку (если этот диалог/поток привязан к сессии ACP).
3. Резервная сессия запрашивающего.

Привязки к текущему диалогу и потоку участвуют в шаге 2.

Если цель не определена, OpenClaw возвращает чёткую ошибку (`Unable to resolve session target: ...`).

## Режимы привязки при создании

`/acp spawn` поддерживает `--bind here|off`.

| Режим | Поведение |
|------|-----------|
| `here` | Привязать текущий активный диалог на месте; завершить с ошибкой, если активный диалог отсутствует. |
| `off` | Не создавать привязку к текущему диалогу. |

Примечания:

* `--bind here` — самый простой путь оператора для "сделать этот канал или чат поддерживаемым Codex".
* `--bind here` не создаёт дочерний поток.
* `--bind here` доступен только на каналах, которые поддерживают привязку к текущему диалогу.
* `--bind` и `--thread` нельзя комбинировать в одном вызове `/acp spawn`.

## Режимы создания потоков

`/acp spawn` поддерживает `--thread auto|here|off`.

| Режим | Поведение |
|------|-----------|
| `auto` | В активном потоке: привязать этот поток. Вне потока: создать/привязать дочерний поток, если это поддерживается. |
| `here` | Требовать текущий активный поток; завершить с ошибкой, если его нет. |
| `off` | Без привязки. Сессия запускается без привязки. |

Примечания:

* На поверхностях без привязки к потокам поведение по умолчанию фактически равно `off`.
* Создание привязки к потоку требует поддержки политики канала:
    * Discord: `channels.discord.threadBindings.spawnAcpSessions=true`;
    * Telegram: `channels.telegram.threadBindings.spawnAcpSessions=true`.
* Используйте `--bind here`, если хотите закрепить текущий диалог без создания дочернего потока.

## Элементы управления ACP

Доступное семейство команд:

* `/acp spawn`;
* `/acp cancel`;
* `/acp steer`;
* `/acp close`;
* `/acp status`;
* `/acp set-mode`;
* `/acp set`;
* `/acp cwd`;
* `/acp permissions`;
* `/acp timeout`;
* `/acp model`;
* `/acp reset-options`;
* `/acp sessions`;
* `/acp doctor`;
* `/acp install`.

`/acp status` показывает эффективные параметры среды выполнения и, когда доступно, идентификаторы сессии как на уровне среды выполнения, так и на уровне бэкенда.

Некоторые элементы управления зависят от возможностей бэкенда. Если бэкенд не поддерживает элемент управления, OpenClaw возвращает чёткую ошибку о неподдерживаемом элементе управления.

## Сборник команд ACP

| Команда | Что делает | Пример |
|--------|-----------|--------|
| `/acp spawn` | Создать сессию ACP; необязательная текущая привязка или привязка к потоку. | `/acp spawn codex --bind here --cwd /repo` |
| `/acp cancel` | Отменить текущий ход для целевой сессии. | `/acp cancel agent:codex:acp:<uuid>` |
| `/acp steer` | Отправить инструкцию управления в запущенную сессию. | `/acp steer --session support inbox prioritize failing tests` |
| `/acp close` | Закрыть сессию и отменить привязки к потокам. | `/acp close` |
| `/acp status` | Показать бэкенд, режим, состояние, параметры среды выполнения, возможности. | `/acp status` |
| `/acp set-mode` | Установить режим среды выполнения для целевой сессии. | `/acp set-mode plan` |
| `/acp set` | Общая запись параметра конфигурации среды выполнения. | `/acp set model openai/gpt-5.4` |
| `/acp cwd` | Установить переопределение рабочего каталога среды выполнения. | `/acp cwd /Users/user/Projects/repo` |
| `/acp permissions` | Установить профиль политики утверждения. | `/acp permissions strict` |
| `/acp timeout` | Установить тайм-аут среды выполнения (в секундах). | `/acp timeout 120` |
| `/acp model` | Установить переопределение модели среды выполнения. | `/acp model anthropic/claude-opus-4-6` |
| `/acp reset-options` | Удалить переопределения параметров среды выполнения для сессии. | `/acp reset-options` |
| `/acp sessions` | Вывести список недавних сессий ACP из хранилища. | `/acp sessions` |
| `/acp doctor` | Проверка работоспособности бэкенда, возможностей, действий по устранению неполадок. | `/acp doctor` |
| `/acp install` | Вывести детерминированные шаги установки и включения. | `/acp install` |

`/acp sessions` считывает хранилище для текущей привязанной или запрашивающей сессии. Команды, принимающие токены `session-key`, `session-id` или `session-label`, разрешают цели через обнаружение сессий шлюза, включая пользовательские корни `session.store` для каждого агента.

## Сопоставление параметров среды выполнения

У `/acp` есть удобные команды и общий установщик.

Эквивалентные операции:

* `/acp model <id>` сопоставляется с ключом конфигурации среды выполнения `model`.
* `/acp permissions <profile>` сопоставляется с ключом конфигурации среды выполнения `approval_policy`.
* `/acp timeout <seconds>` сопоставляется с ключом конфигурации среды выполнения `timeout`.
* `/acp cwd <path>` напрямую обновляет переопределение рабочего каталога среды выполнения.
* `/acp set <key> <value>` — общий путь.
    * Особый случай: `key=cwd` использует путь переопределения рабочего каталога.
* `/acp reset-options` удаляет все переопределения параметров среды выполнения для целевой сессии.

## Необходимая конфигурация

Базовая конфигурация ACP:

```json5
{
  acp: {
    enabled: true,
    // Необязательно. По умолчанию true; установите false, чтобы приостановить отправку ACP, сохраняя элементы управления /acp.
    dispatch: { enabled: true },
    backend: "acpx",
    defaultAgent: "codex",
    allowedAgents: [
      "claude",
      "codex",
      "copilot",
      "cursor",
      "droid",
      "gemini",
      "iflow",
      "kilocode",
      "kimi",
      "kiro",
      "openclaw",
      "opencode",
      "pi",
      "qwen",
    ],
    maxConcurrentSessions: 8,
    stream: {
      coalesceIdleMs: 300,
      maxChunkChars: 1200,
    },
    runtime: {
      ttlMinutes: 120,
    },
  },
}
```

Конфигурация привязки к потокам зависит от адаптера канала. Пример для Discord:

```json5
{
  session: {
    threadBindings: {
      enabled: true,
      idleHours: 24,
      maxAgeHours: 0,
    },
  },
  channels: {
    discord: {
      threadBindings: {
        enabled: true,
        spawnAcpSessions: true,
      },
    },
  },
}
```

Если создание ACP, привязанного к потоку, не работает, сначала проверьте флаг функции адаптера:

* Discord: `channels.discord.threadBindings.spawnAcpSessions=true`.

Привязки к текущему диалогу не требуют создания дочернего потока. Они требуют активного контекста диалога и адаптера канала, который предоставляет привязки ACP к диалогам.

См. [Справочник по конфигурации](/gateway/configuration-reference).

## Настройка плагина для бэкенда acpx

Новые установки по умолчанию включают встроенный плагин среды выполнения `acpx`, поэтому ACP обычно работает без ручного шага установки плагина.

Начните с:

```text
/acp doctor
```

Если вы отключили `acpx`, запретили его через `plugins.allow` / `plugins.deny` или хотите переключиться на локальную рабочую копию для разработки, используйте явный путь к плагину:

```bash
openclaw plugins install acpx
openclaw config set plugins.entries.acpx.enabled true
```

Локальная установка рабочей копии во время разработки:

```bash
openclaw plugins install ./path/to/local/acpx-plugin
```

Затем проверьте работоспособность бэкенда:

```text
/acp doctor
```

### Конфигурация команды и версии acpx

По умолчанию встроенный плагин бэкенда acpx (`acpx`) использует закреплённый в плагине бинарный файл:

1. Команда по умолчанию — плагин-локальный `node_modules/.bin/acpx` внутри пакета плагина ACPX.
2. Ожидаемая версия по умолчанию соответствует закреплению расширения.
3. При запуске бэкенд ACP сразу регистрируется как неготовый.
4. Фоновая задача проверки проверяет `acpx --version`.
5. Если плагин-локальный бинарный файл отсутствует или не соответствует, выполняется:
   `npm install --omit=dev --no-save acpx@<pinned>` и повторная проверка.

Вы можете переопределить команду/версию в конфигурации плагина:

```json
{
  "
